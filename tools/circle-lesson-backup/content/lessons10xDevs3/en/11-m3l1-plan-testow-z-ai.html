You deliver faster. But how do you know it still works?

Somewhat professional, but still a bit theatrical.

In this lesson, we will build something more durable than individual tests. We will start with a test plan, a classic QA practice that will serve as a strategic guide for the entire quality assurance process in the project.

But beware - we are not reinventing the wheel. If you know what needs to be protected and don't want to introduce new variations into your workflow, the cycle research-plan-implement you know from the previous module is more than enough.

A new skill simply moves up one layer, as if a virtual quality engineer joined your team. It helps determine what to test first, why exactly that, and in what order to deliver the missing elements.

Test plan in the AI era

The classic test plan is associated with a document that is created alongside the team's work. Someone describes the scope, environments, test types, roles, and schedule. With a good agent, the same strategic documentation is created in a few minutes during an ordinary terminal session.

This is what /10x-test-plan is for.

This is a new skill that creates and maintains context/foundation/test-plan.md - a document that combines the testing strategy and the current state of its implementation. The skill reads it at each run, checks what is already on the disk, and suggests the next step.

A small disclaimer at the start: if your project is just a PRD and a few notes without code, the skill will stop with a message that you need to come back after /10x-shape and /10x-prd. The test plan feeds on what already exists in the repo.

Gates before production

This plan will also introduce the concept of quality gates. Conceptually, these are checkpoints that a change must pass through before we consider it ready for the next stage of work.

Such a gate can be very simple: lint, typecheck, unit tests, critical flow test, review in a Pull Request, pre-commit, or a check in CI. It's about a simple rule – the change doesn't move forward just because the agent finished writing the code and said it's ready.

In AI programming this is especially important, because the agent can quickly generate a lot of seemingly sensible code. Speed is an advantage, but it also increases the risk that you will skip the moment of reflection: whether this change fits the project contract, whether it doesn't break a critical scenario, or whether it doesn't cover up a problem with a nice diff.

Quality gates intentionally complicate the direct path from the agent's response to production. This is a mechanism that slows down the bad reflex: "it works locally, looks good, let's ship it." We want the code to first receive signals from tools, tests, and reviews, and only then reach the users.

Note that /10x-test-plan is not a separate testing engine. It is an orchestration layer on top of the cycle you already know from the second module.

The skill can be repeated at different stages of the work. You type /10x-test-plan for the first time, the agent conducts analysis, interview, and generates the first plan. It suggests implementing the first change by redirecting the flow to the classic /10x-new <change-id>. You can clear the session with /clear or start a new thread through the UI, come back after two days, type /10x-test-plan again, and continue. The agent looks at the disk again, sees that you already have change.md but lack research.md, and says: "time for /10x-research".

You don't need to remember where you left off. The files are the memory of the rollout.

Our assistant's work does not involve a single long session in which the agent devises a strategy, does research, plans, writes tests, and along the way loses half of the agreements. We have a persistent document, small verifiable steps, and a clean context at each subsequent stage.

Risks first

The worst way to start working on quality sounds like this:

Write tests for this file.

The agent does exactly what you ask. It looks for the file, finds the functions, adds tests to the visible paths, and often ends up where it is easiest. After an hour, you have dozens of tests and all are green. A false sense of security, while the real risks may be somewhere completely different.

Another approach? Start with the risks and sensitive areas of the project from the perspective of business value.

/10x-test-plan starts right from them, based on contextual documents:

  • context/foundation/prd.md - user, flow, requirements, non-goals,
  • context/foundation/roadmap.md - status of functional changes,
  • context/archive/*/plan.md - state of archived initiatives,
  • context/foundation/tech-stack.md - stack and constraints,
  • AGENTS.md or CLAUDE.md - work rules in the repo,
  • the existing test configuration, if the project already has one.

The whole shows a fuller picture of the project and what you should really watch out for:

The GIT history is one of the strongest signals – changes focus where the project is still alive. But before the skill starts scanning, it will suggest a scope: "I detected packages/api/src, packages/course-content/src, excluded dist, coverage, fixtures. Do you accept or should I add a workspace I missed?" It's worth pausing here. A scan stretched over the lockfile or snapshots will drown the real signal in churn that no one writes manually.

Note the last data source.

The skill conducts a brief interview with you, uncovering what you are really afraid to break, where you have been burned before, which area you change without certainty, and what you don't want to waste your testing budget on. This is not a soft addition to the "real" analysis. Your experience is often the best signal of daily risk.

The interview will adapt to your project. Are you working on a new service where there isn't a single test yet? The agent will skip the question "what do you feel is under-tested?" - because the answer "everything" adds nothing. Do you have a handful of tests in one package while the rest of the repo is bare? The question will change to "where is the biggest gap that worries you?".

The last interview question is phrased differently from the rest: "what would you NOT want the test budget to go towards?". The answer goes to a separate exclusion section in the plan. It seems trivial, but six months later, when someone new on the team asks "maybe we add snapshots of marketing pages?", you have the decision documented with the reason.

The result is a risk map - several scenarios evaluated in two dimensions:

  • impact, meaning how painful the failure is for the user and the business,
  • probability, that is how likely it is that this failure will occur.

To ensure this assessment is not a lottery, it is worth sticking to a simple scale. A three-level division - high, medium, low - with a one-sentence definition, which you can always refer back to, is completely sufficient:

  • Rating: High | Impact: user loses access, data, or money; publicly visible failure | Probability: area changes weekly or has caused issues here before
  • Rating: Medium | Impact: function works worse, workaround exists, some users feel the effect | Probability: code is occasionally modified, has been a source of errors
  • Rating: Low | Impact: cosmetic, easy to revert, no data impact | Probability: code is stable, rarely touched

Priority is simply a combination of both axes. You protect high impact × high probability scenarios first, and last (or not at all) those low × low. The extremes are the most interesting: a failure with huge impact but negligible probability (e.g., a global cloud provider outage) is usually not worth testing – it’s cheaper to handle it with observability and alerting than with a test scenario. On the other hand, a small but daily error can erode more trust than a spectacular disaster once a year.

The scale is intentionally coarse. The goal is not to calculate the risk with accuracy to the second decimal place, but to ensure that two people looking at the same row agree whether it is the "top" or the "bottom" of the table.

An axis easily forgotten: security

Interviews and documents do a great job of catching functional risks - "paying user gets a 403," "import loses records." It is much harder for risks that no one reports as a feature to emerge on their own, because these are abuse scenarios, not regular usage.

Therefore, when building a risk map, consciously add a separate question: what will happen if someone uses this against the intention? A few categories worth considering regardless of whether they came up in the interview:

  • Authorization and access - can user A reach user B's resource by swapping the identifier in the URL (IDOR)? Does the endpoint check not only "are you logged in" but also "does this belong to you"?
  • User input - does untrusted data go into a query, template, or command without validation (injection)? Does the server validate the same as the client, or does it blindly trust the front-end?
  • Secrets and sensitive data - does a key, token, or personal data leak into logs, error messages, or the front-end bundle?
  • Resource abuse - can someone bypass the rate-limit, order an expensive operation in a loop, or send mass magic links?

These are still failure scenarios, so they end up on the same map and are subject to the same impact × likelihood scale. The difference is that the agent rarely suggests them on its own – the "happy path" does not include an attacker. If your project has authorization, payments, or accepts anything from the user, the absence of even one security row on the risk map is usually not a sign that it is safe. It is a sign that no one asked about it.

Signal, not diagnosis

This is the most important boundary of the new skill.

/10x-test-plan reads the repository for clues, not to confirm a specific threat. It might say: "this area has changed frequently," "PRD says this flow is critical," "the user fears regression in auth," "the project has too few tests."

It should not say: "the error is located in src/auth/session.ts:42".

What's more - it usually won't allow it even when you ask. Enter a risk like "no retry in session.ts" and you'll get a response saying "that describes the implementation, not the failure scenario. Rephrase it to what the user will actually experience." The filename in the source column will also disappear - it will be replaced by what truly raised the risk: the interview question number, the line from the PRD, or the directory with the highest churn.

The specifics in the code are only looked at by /10x-research, already in the concrete phase of the plan's rollout. Research will trace the real code, find user inputs, API boundaries, validation, and the cheapest test layer that will catch a given failure. Otherwise, the agent pretends to have knowledge it does not have.

That's exactly why we don't need to invent a new method for implementing tests. Once you know which risk is important, the normal cycle works very well:

  1. /10x-research checks where the risk truly exists
  2. /10x-plan turns research into small work phases.
  3. /10x-implement delivers one phase at a time.
  4. The final phase updates the test-plan.md on how to add similar tests from now on.

/10x-test-plan does not replace research, planning, or implementation. It sets their order and ensures they don’t start with a convenient file, but with an important risk.

Sometimes, research turns out to be smarter than the plan and that's OK too. When /10x-research finishes analyzing the code and discovers that the failure resides in a different directory than the hot-spot indicated, on the next run of /10x-test-plan, it will ask you: "research suggested a fix for §2, add it now or leave it for the next revision?". The plan can accept a correction from the layer that knows the code better.

Green test that protects nothing

There is one trap that agents writing tests fall into very easily, and which can undo all the effort put into the risk map. It is worth knowing it before you start delivering the first phase (we will address this in the next lesson).

Imagine that you ask the agent for a test for a discount-calculating function. The agent reads the code, sees that the function returns 0.15, and writes:

expect(calculateDiscount(order)).toBe(0.15)

The test passes. CI shows green. However, this test does not verify whether the discount is correct - it checks whether the code does what it already does. If there is a bug in the function, the test has just cemented that bug. From now on, every attempt to fix it will "break the test," even though the test is wrong.

This is the oracle problem: to write a meaningful test, you need to know the expected result from an independent source - from requirements, contracts, specifications, common sense - not from the implementation itself that you are testing. When a human writes a test, they draw on that knowledge from memory. An agent by default mainly has code at hand and is very likely to take it as the oracle.

This is the most important risk of the whole idea of "let AI write tests." A test that just mirrors the implementation is worse than having no test at all – it gives a false sense of security and makes later corrections more difficult.

How to defend against this:

  • Give the agent space to interpret the assumptions. Instead of "write tests for this function," say "the discount is 10% above 200 PLN and 15% above 500 PLN – write tests for these thresholds." The expected result then comes from you, not from the code.
  • Read assertions, not just the coverage bar. When reviewing the generated test, ask about each assertion: "where did this value come from?". If the answer is "because the function returns that," then it is not a test.
  • Check if the test can fail. Break the production code for a moment and see if the test breaks. If it’s still green, the assertion isn’t checking anything. We will develop this intuition in the next lesson.
  • Beware of working under the pressure of the green. When you tell the agent to "fix it until the tests pass," the easiest way is often to adjust the test to the code rather than the code to the requirements. Watch the direction.

/10x-test-plan sets this defense system-wide. The risk map describes the behavior that is to be protected ("the paying user sees their content"), rather than "function X returns Y." Thanks to this, when in the planning and implementation phase an agent is already writing a specific test, they receive an oracle from the perspective of value for the user - not from the line of code they are currently viewing.

Cookbook instead of memory in the head

test-plan.md has one more feature that pays off on its own over time.

The document grows as successive phases are completed. Each finished phase adds specific rules to it – where to keep tests, how to name them, which patterns to base them on. It's like a personal notebook of what the agent learns along the way.

We are talking about the ## 6. Cookbook Patterns section, which is initially not filled with any details:

But don't worry - this is intentional. The skill should not pretend to know local testing patterns before research and implementation show how the project really works.

If you go through /10x-test-plan to the end, the last step of each phase is to update the cookbook. After completing the first phase, our TBD changes into something like this:

### 6.2 Adding an integration test

The next agent - yours or from the team - no longer has to guess where the integration tests go in this specific project. They open the cookbook and have the pattern, path, and command.

This is the moment when the test plan ceases to be abstract. It becomes an instruction for subsequent agents and for you from the future.

Without a cookbook, every new change starts with guessing the local test culture.

Returns to the test plan don't hurt

Executing the test plan may require repeatedly clearing sessions, going through cycles of research, planning, and multi-phase implementation. How not to lose sight of progress in QA?

Type /10x-test-plan --status. Instead of work, you’ll get a short table: which phase is closed, which is in progress, and what is the next command to run. Just enough for you to come back after two weeks without starting by reading chat history.

The plan will also become outdated over time. The stack changes, AI tools receive new versions, the team realizes that some risk is no longer an issue or that a new one has emerged. Then /10x-test-plan --refresh will open a fresh round of needs analysis. Changes go through the same cycle as the initial rollout, so you have an audit of what and when changed.

Run the first test plan

Run /10x-test-plan in your project repository.

After completing the first run, review the agent's actions:

  1. Have the context sources been correctly classified?
  2. Does the baseline state of the tests sound fair (greenfield usually just means no tests)?
  3. Was the GIT history useful?

If something doesn't match, correct it now by directly indicating to the agent what it missed or overinterpreted. An incorrect scope from the start will ruin the priorities of the entire rollout.

Obtain a Risk Map

After preliminary analysis, the skill will generate a risk map. Check a few elements:

  1. Are these risks truly significant for your project, and not describing unlikely scenarios (e.g., GitHub company closure)?
  2. Which source documents most strongly influenced the final shape of the entire risk map?
  3. Does the risk description not overly focus on the implementation and describing low-level code?
  4. Do you agree with the assessment of impact and likelihood? If not, suggest the necessary corrections to the agent.
  5. Is there at least one abuse scenario on the map (access, user input, secrets, rate-limiting) if the project has authorization or payments?

Remember - if the agent skipped a critical path, add it manually or ask for a reassessment. If the line reads like "no tests for module X," turn it into a user failure scenario.

Do not proceed further just because the table looks elegant.

Check project fit

In context/foundation/test-plan.md find the sections Stack and Quality Gates. Analyze whether the test plan adequately describes the working realities in this project:

  • whether the proposed types of tests are well mapped to the risks
  • whether using AI will positively affect quality
  • whether quality gates (hooks, triggers) are not redundant

This is not a competition for the largest number of modern tools – the created plan must be understandable to you and not cause burdens during further work. Over time, you can edit it and adjust the initial assumptions.

Risk-based testing, without AI

Risk-based prioritization is not an idea from the LLM era. In classical QA, the concept of risk-based testing has long existed: you first protect areas with the highest combination of impact and failure probability.

AI changes the way this rule is operationalized, not the rule itself.

Previously, the risk map was prepared by a person, tester, or team. Now an agent can read the PRD, roadmap, and change archive faster, then propose candidates. However, it still requires your correction because it does not know the full business context.

The greatest benefit of AI is therefore not in "generating a list of tests." It lies in the fact that you move more quickly from design artifacts to a discussion about real failure scenarios.

This is especially important for a solo developer. You don't have a separate QA team, so you need to prioritize. The risk map is your segregation mechanism.

Signal versus Knowledge

In the old approach to testing, the agent very easily mixed two levels of work - business analysis and focusing on specific technical causes:

  • "this area looks risky",
  • "this particular file is the point of failure".

The first may result from the PRD, roadmap, git history, or your concern. The second requires research in the code. One does not necessarily have to result from the other.

/10x-test-plan intentionally keeps the risk map at the signal level. This way, it does not lock the research into a false hypothesis. If the scan of the most frequently changed locations shows src/lib/, it is information about frequent changes. Not proof that the failure resides in src/lib/something.ts.

This distinction saves time in subsequent phases. /10x-research can then do its job: trace data flow, check existing tests, find the real system boundary, and suggest the cheapest test. If it discovers that the risk was poorly formulated, the plan can be corrected instead of continuing to test a fictitious problem.

A good test plan does not pretend to be a code expert. It is a good assignment for research.
