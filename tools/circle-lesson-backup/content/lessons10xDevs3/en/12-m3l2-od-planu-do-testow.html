IMAGE — https://images.przeprogramowani.pl/images/2026/05/iH0cat3ZHNaDxN8PpJ6noQk.jpeg

Before you write your first test with the agent, it's worth being aware of one thing: writing tests is precisely the area of working with AI where it is easiest to be deceived.

The start always looks great. You ask the model for tests for selected functions, you get a new batch of code, a green report, and coverage that jumps up. It’s easy to believe that it works.

The problem is that testing simple functions on artificial examples does not effectively help manage risk in a project. Moreover, the effectiveness of agents in writing tests clearly deteriorates if your functions are not as typical as those the models have seen in the training data.

A random example. In the paper Benchmarking LLMs for Unit Test Generation from Real-World Functions (Huang et al., ACM TOSEM 2026), the researchers first measured the quality of generated tests on the popular TestEval benchmark — and it looked great. Then they replaced simple functions with real, complex cases from real projects and repeated the measurement. Coverage dropped by about half: from ~92% to ~45% of lines and from ~82% to ~30% of branches.

This repeats in many other areas of AI programming - the result from a benchmark does not automatically transfer to your project.

Why does this happen? Because the hardest part of the test is not generating the code, but deciding which test will actually impact the quality of the entire project. This is the oracle problem introduced in the previous lesson - and it is here, not in the syntax, that most of the errors in tests written by LLMs lie.

In m3-l1 you saw it on the function calculating the discount, where the agent took the expected result directly from the implementation. Now the same trap on a real function - one from a flashcard application that calculates the interval to the next card review:

// grade: 0 = don't remember, 1 = weakly, 2 = well
function getNextInterval(prevInterval: number, grade: number): number {
  return prevInterval * grade;
}

At first glance, one thing is clear: the result is prevInterval * grade. The model will therefore write the assertion directly on this formula - getNextInterval(10, 2) should return 20. The test will pass green.

What is not visible in the code? The business rule and the bug. When a student does not remember the card (grade equal to 0), the interval should reset to one day - not drop to zero, because zero means the card will never come back for study. The function has a bug here, but the model won't see it: it will only see that for grade equal to 0 the output is 0, and it will record such an assertion.

The model reflected what the code does. What the code should do could not be inferred from the implementation itself - and that is precisely the essence of the test.

Hence two conclusions upon which this entire lesson stands:

  • A high test coverage rate is not proof of protection. Studies consistently show that code coverage poorly correlates with actual bug detection. The green bar only means that the line was executed – not that the assertion made any sense.
  • Naive prompting is not enough. Tests that truly protect the code result from risk analysis, proper context, and careful attention to their optimal number. Neither too few nor too many.

Fortunately, we already have a solid foundation. In the previous lesson, we obtained the Test Plan - a quality contract for the project that includes a risk map, describes phased implementation improvements, and organizes the cycle /10x-new → /10x-research → /10x-plan → /10x-implement through which we carry out changes. That was an important step, although the map itself is just the beginning.

Vibe Testing, or how not to introduce tests

A naive approach to testing may initially look like this when working with agents:

Write tests for the auth module.

The problem is that such tests often check the shape of the current implementation rather than the behavior important to the user, which arises, for example, from the documentation.

See a simplified example:

it("authenticates user", async () => {
  const result = await authenticate(input);

  expect(result).toEqual(await authenticate(input));
});

This looks like a test, but it hardly protects anything. The assertion asks the function what it returned, and then checks if it returned what it returned.

In practice, tests generated by the best AI models are usually less caricatured, but the mechanism remains the same. This can be described using three classes of problems:

  • Anti-pattern: Implementation mirror | What it looks like: The test checks an internal call, a private detail, or a result calculated with the same logic as the whole module. | What should happen: The test describes observable behavior and the expected outcome of a business scenario.
  • Anti-pattern: Happy paths | What it looks like: The agent creates easy tests on valid data. | What should happen: At least one edge case arising from risk appears.
  • Anti-pattern: Missing edge cases | What it looks like: There is no test with null, empty data, environment error, or library incompatibility. | What should happen: The test captures a case that could realistically damage the user experience.

Hence two specific requirements for the entire process: give the agent concrete input (description, data, or scenario) related to the risk (otherwise the test will not touch the critical path) and verify the assertion separately (because it is the most common weak point).

How the Test Plan Disarms These Pitfalls

Our context/foundation/test-plan.md and the chain /10x-new → /10x-research → /10x-plan → /10x-implement were created precisely to defuse each of these traps before the agent writes the first assertion. Let's go through them one by one.

"The agent will generate tests for each file" → risk instead of file. The Test Plan is not a list of files to cover, but a risk map. The risks section explains which business scenario must not fail and why – including an estimate of impact, likelihood, and source. The decision on "what to test" is therefore already made, based on what truly matters rather than what is easiest to generate.

Oracle problem → research and description of expected behavior. Remember that from the implementation alone it was not possible to read what the code should do? The plan supplements this. For each risk, it holds not only a description of the threat but also what behavior would prove protection (this is your oracle) and which convenient assumption the agent should challenge - for example, "successful login does not immediately mean that the user sees the correct page content."

Coverage as evidence → the cost × signal principle. The plan is based on one rule: for each risk, we look for the cheapest test that provides a real signal. We do not promote jumping to e2e just because "it's safer," and we do not chase coverage percentage. Coverage ceases to be the goal—the goal is a signal for a specific risk. We will show later how this signal differs from coverage itself.

Three anti-patterns → built-in protection in handoff. When a phase reaches /10x-plan, the handoff suggests upfront that every test stage should specify: what behavior it asserts, which regression it catches, where it takes context from research.md, that it has at least one edge case, and which anti-pattern it avoids.

Pay attention to the suggested parameters of each of the 10x-research and 10x-plan skills – working with the 10x-test-plan skill will allow you to use them in the most effective way.

Case study: from risk to integration testing

To ensure this is not just theory, let's track a real process of securing existing code based on risk from context/foundation/test-plan.md. The example will be 10xCards, a flashcard application.

When the user reviews the generated cards and accepts some of them, the save operation (save) promotes the accepted drafts to the real deck. So we take into account the risk that this save might corrupt the deck.

We start with context/foundation/test-plan.md and end with working tests, this time mostly integration tests. This is exactly the same loop you can perform in your tasks.

Starting point: one risk, zero tests on the write path

We will focus on one of the risks related to data recording that was not covered by any test:

  • Risk #2 - session save corrupts the deck: drafts in pending/rejected state end up in the deck along with accepted ones, or a partial save on error leaves the deck in an inconsistent state. The business contract is simple: only accepted cards can be added to the deck.

The entire save path - the saveSession() function in src/lib/save-session.ts, the save endpoint, and the neighboring draft and flashcard endpoints - initially had zero tests. The project already had a hermetic set of tests from an earlier phase, but not a single test touching the real database.

What the research discovered

Before the first test prompt was issued, /10x-research started. And again the most important phrase of the entire process appeared: research changed the task's goal, although in a different way than one might expect.

It turned out that one risk has two completely different faces:

  • “pending/rejected are promoted” is in practice a well-secured path. The record filters drafts through .eq("state","accepted"), and the data sent to the database is created exclusively from this filtered set. The database itself enforces states through a CHECK constraint. To break this, someone would have to change the filter itself. This is not a hidden bug, but an invariant to defend.
  • "partial commit on error" is a real and untested risk. Here research uncovered a crucial fact: the save is not atomic. saveSession() performs three independent Supabase operations in sequence: upsert to flashcards, upsert to review_states, and then delete from flashcard_drafts. The Supabase JS client does not wrap this for us into a single transaction. An error in the middle of the sequence leaves a partial state.

And one more inconvenient detail: these partial failures are not fixed today. When the review_states commit fails, the function logs orphan_review_state and still returns ok: true. When the draft deletion fails, it logs orphan_drafts and also returns ok: true. There is no rollback.

This changed the task in a way that is easy to overlook: we do not write a test for a rollback that does not exist. The decision whether to enforce atomic recording through a single transaction (e.g., Postgres RPC) is a product decision. The research noted it as an open risk but did not pretend that there is protection in the code that does not exist.

The task of the phase was therefore different: to prove the rules that are already secured, and to pin down the current, consciously tolerated partial failure contract. This way, a future change will not silently break it.

Research also selected a testing strategy – and here a division into two layers appeared, each chosen based on the cheapest signal:

  • Integration tests on local Supabase - for rules that honestly cannot be simulated with a mock: only approved drafts go to flashcards, exactly one review_states row is created for the promoted card, all session drafts disappear after saving, and the second save is idempotent (unique constraint does not create duplicates). This required new infrastructure: a client with a service-role key, a fresh account, and a fresh session for each test. The service-role key bypasses RLS, but in this scenario, it is acceptable because saveSession filters by explicit accountId anyway.
  • Hermetic tests with a client stub - for partial failures where the real database cannot be easily forced to fail in the middle of the sequence. A configurable client stub allows you to make a specific operation return an error while the rest succeed. Then the assertion attaches the behavior: “review_states failed → log orphan_review_state and ok: true”.

In the end, the research produced a map of the “cheapest test that captures a given piece of risk” - even before the first prompt.

From research to plan

The implementation of tests has moved from the research phase to the planning phase. The 10x-plan skill divided the work into four phases, arranged in a deliberate order - according to dependencies:

  • Phase: 2.1 Integration environment + promotion rule | What it delivers: new set of helpers for local Supabase + test “only accepted promotes” | Why here: without this, writing the rest of the integration tests is impossible
  • Phase: 2.2 Remaining DB rules | What it delivers: pairing review_states, draft cleanup, idempotency | Why here: relies on the environment from 2.1
  • Phase: 2.3 Encapsulated stubs + endpoint | What it delivers: branches of partial saves, early terminations, HTTP status mapping | Why here: does not require database, lands in default npm test
  • Phase: 2.4 Cookbook + gate correction | What it delivers: filled patterns in test-plan.md, integration gate marked as ad hoc | Why here: durable knowledge transfer to subsequent phases

The order corresponded to the real dependencies in the project. First, it was necessary to build the missing integration environment and immediately test the most important rule on it so that Phase 2.1 would provide proof, not just scaffolding. Only then did the subsequent database rules, cheap hermetic tests without a database, and the consolidation of patterns for the following phases make sense.

Note - after setting up the test environment, it is worth updating AGENTS.md / CLAUDE.md describing how we test code in this project.

Where this aligns with good testing principles

When implementing tests based on the risk map, we ensured several good testing practices:

  • Risk, not file. The starting point was Risk #2 from the map, not "let's cover save-session.ts". Moreover, the research split this risk into a part that was already well secured and a part that was actually uncovered. The tests went where it really hurt.
  • Oracle from research, not from implementation. The fact that the write is non-atomic and that partial failures end with ok: true could not be guessed from the shape of the function alone. The assessment was: full promotion or a countable, logged partial state - never silent deck corruption.
  • Honest contract instead of the ideal one. We did not write a test for rollback because there is no rollback. We documented the current behavior and noted the RPC call as an open product risk. The test ensures what the code does today, and will catch the moment when someone changes it without a conscious decision.
  • Cost × signal instead of chasing coverage. Integration on local Supabase appeared only where a mock would lie: unique constraint, cascades, real SQL. A client with a service key replaced user login because authorization through RLS would be a separate phase. The client stub handled failures that the real database would not cause in a simple way. The integration gateway was ad hoc, not on every commit, so as not to force the project to run Supabase with every small change.
  • Behavioral assertion. The rule “only accepted promotes” does not check how the query is constructed. The test seeds a mixed set of drafts (accepted + pending + rejected) and checks the observable effect: only accepted cards land in flashcards, and after saving, every session draft disappears. Such a test catches a regression that is not yet present: a loosened state filter or a narrowed delete.
  • Assertion verification by intentional breaking. Each phase has a manual step like “invert the result of one branch of saveSession and check if exactly that one endpoint test fails” or “narrow down draft deletions and check if the assertion »all drafts disappear« actually fails.” This is a simplified version of mutation tests developed in Deep Dive, but run manually: if after breaking the code the test is still green, the assertion protects nothing.

Notice that no step said "generate tests for save-session.ts". Each one said: this piece of risk, this hypothesis, this cheapest test - integration or hermetic. This is the difference between a test from an agent and the quality assurance process.

Optional skill for new functionalities: /10x-tdd

You will get the skill by downloading the artifact package for this lesson:

npx @przeprogramowani/10x-cli@latest get m3l2

This package provides the skill: /10x-tdd. Below we describe its assumptions and how it works.

So far, our executor for all kinds of plans has been the /10x-implement skill. This is a good default path: we have an approved plan.md, go through the successive phases, and deliver the change along with tests.

However, there is a situation where it is worth tightening the process. If the planning phase involves writing new code first and then adding tests, you can run the optional skill /10x-tdd, which reverses this process.

This is not a separate workflow and is not a mandatory quality gate for every change. /10x-tdd reads the same file context/changes/<change-id>/plan.md, uses the same ## Progress section, and implements the same phases as /10x-implement. It only changes the order of work:

RED -> GREEN -> REFACTOR

First, the agent writes a test that must fail (turn red). Then they add the minimal code to make it pass (turn green). Finally, they tidy up the solution without changing its behavior (note that this practice does not require a test plan and can be used for implementing tests on the fly).

IMAGE — https://images.przeprogramowani.pl/images/2026/05/rn0cap6XMLLY7M8Plb7fcQ.jpeg

In classic TDD, this is a well-known cycle. When working with an agent, it has one more function: it pauses the model at the assertion before it starts producing the implementation.

Instead of writing:

Implement the promotion of accepted drafts to the deck according to the XYZ phase, then cover it with unit tests.

with the 10x-tdd skill you follow the path:

First, implement a test that demonstrates that the rule only promotes drafts with the `accepted` status,
and that `pending`/`rejected` drafts never make it into the deck. Then move on to implementing the requirements for the XYZ phase.

The agent must first reveal what it considers as evidence of correct behavior. Only then does it get the right to write code.

That is exactly why /10x-tdd is a noteworthy alternative. Here, we are constantly struggling with tests that only look good: they have a green report, cause coverage to increase, and an assertion that does not protect against risk. A red test at the beginning forces us to ask earlier about the source of truth:

  • what exactly should happen,
  • what result do we see from the outside,
  • what should break if someone breaches the protection,
  • whether the assertion describes behavior or just repeats the implementation.

Exactly. If you can't answer these questions, then you don't yet have enough material for a good test.

You can therefore stick to a simple rule:

If in a given phase of the plan you can name the first red test in one sentence, consider /10x-tdd. If you cannot, stick with /10x-implement or go back to research.

Good starting conditions sound like this, for example:

  • "promotes only drafts in the accepted state, and pending/rejected never reach the deck",
  • "returns ok:true and logs orphan_review_state when the upsert of the review state fails during save",
  • "returns 401 when the user does not have access to the course",
  • "resets the repetition interval to one day when the rating is 0".

Each of them describes an observable outcome. It does not tell the agent how to name the function, where to place the mock, or which private method to call. It states what behavior should be defended.

/10x-tdd works particularly well for:

  • validators, parsers, and data transformations,
  • business logic with clear input and output,
  • API contracts: status, response body, auth, gating, error handling,
  • reducers, state machines, and flag calculations,
  • bugfixes, where you first want to reproduce the bug,
  • regression guards for risks from test-plan.md.

There are also cases where it is not worth forcing it:

  • pure scaffolding of directories and files,
  • configuration of tools, runners, CI/CD, deployment, or environment variables,
  • documentation and content changes,
  • UI cosmetics without a meaningful path for automatic assertion,
  • thin wiring, where the test would only rewrite the implementation,
  • spike, in which you are still searching for the correct contract.

There is one more condition: /10x-tdd assumes that the testing environment already exists. The skill does not install the runner, set up the entire configuration, or build the CI from scratch. If the repository does not yet have tests, first deliver the bootstrap. Only then does the test-first pattern start to be useful.

In practice, you can mix both modes within a single plan:

/10x-implement <change-id> phase 1    # test environment
/10x-tdd <change-id> phase 2               # wrapper contract
/10x-tdd <change-id> phase 3               # API contract
/10x-implement <change-id> phase 4   # cookbook and plan synchronisation

It works because both skills save progress in the same place: ## Progress in plan.md. You don’t create a second task list. You don’t split the change into two parallel processes. You simply choose the appropriate mode for the specific phase.

The greatest value of /10x-tdd is that the agent defines and allows you to evaluate the assertion before writing the code. You can then stop at the red test and ask: does it really capture the risk? Does it not mirror the implementation? Does it have the necessary edge conditions? Will it fail if we intentionally break the code?

This is the moment when the test ceases to be an addition to the implementation. It becomes a conversation about behavior before the implementation even exists.

What if you absolutely want to test by files

This whole lesson encourages you to start from risk, not from a file. However, there are situations where you really have one specific file and want to cover it with tests right away - without the entire cycle of /10x-research → /10x-plan → /10x-implement. This approach does not utilize the full potential of the context and business knowledge gathered so far, but if you choose it, at least don’t let it slide into Vibe Testing and prompts like write me more tests.

The raw prompt below (without any skill) enforces the same safeguards mentioned above: an oracle from sources instead of implementation, behavioral assertions, edge cases resulting from risk, and the right to ask you for clarification when something is unclear.

Replace the placeholders with paths to key files (in brownfield projects instead of PRD you can use another contextual document, and instead of TECH_STACK you can point to package.json or another list of installed libraries):

And then paste the fragment in the same prompt:

Follow these three steps:

1) Determine the expected behaviour from the sources, not from the implementation itself: read the TECH_STACK
and the PRD, and on that basis decide which aspects of the file’s behaviour are relevant to
the user and the business. List them before implementing the tests. Do not assume that what
the code is currently doing is what it should be doing – if the PRD or tech stack does not
clearly indicate the correct behaviour (the oracle problem), STOP and
ask me questions, rather than guessing and copying the result from the implementation.

2) For every significant behaviour, write a behavioural test: the assertion should
check the observable outcome of the scenario, not internal calls, private
details or a result calculated using the same logic as the code under test. Include at least
the edge cases arising from risks (e.g. `null`, empty data, dependency error,
invalid input). Keep the number of tests to a minimum – avoid having several almost
identical copies checking the same thing; each test should catch a different regression.

3) Write a summary of the changes – describe in tabular form which regression each test catches and what
change in the code it would detect. If you cannot specify this for a particular section,

(Note this prompt assumes that the initial test environment setup already exists - if not, start with 10x-research and introduce the necessary libraries or test runners).

If after the result you still are not sure whether these tests actually protect the code, proceed to the Deep Dive - this is exactly the moment when mutation testing will answer the question "does the assertion catch anything," which the green report itself will not resolve.

Why coverage is not enough as a gate

This entire lesson stems from one problem - the agent gives you a green report and increasing coverage, but you don't know if those tests actually protect anything. Coverage won't settle that, because it answers a different question.

Coverage indicates whether a given line, branch, or function was executed during tests. It's a useful signal, but very limited. You can execute a line of code without verifying anything important. You might also hit the correct function but have an assertion so weak that any error passes. This is exactly the kind of test written by an agent that mirrored the implementation instead of the behavior — the line executed, coverage increased, but the assertion protects nothing.

So you need a separate gateway that checks not "whether the code executed," but "whether the test would notice if a change were introduced." This is exactly what mutation testing does.

Mutation testing as a quality gate

The idea is simple: since we don't trust the assertions written by the agent, let's deliberately break the production code and see if the tests notice it at all.

Mutation testing tools work according to a similar scheme - regardless of the language:

  1. They parse the production code and for each place that can be changed, they generate a mutant - a copy of the code with one small, intentional flaw. Typical mutations include replacing > with >=, inverting a condition, swapping a logical value to true/false, removing part of an expression, or changing an arithmetic operator.
  2. Run tests for each mutant (in practice only those that touch the changed line - we will return to this topic when discussing performance).
  3. Classify the result. If at least one test fails for a given mutant, the mutant is killed, meaning the tests detected the fault. If all tests still pass, the mutant survived, providing concrete proof that no assertion covers that fragment.
  4. Count the mutation score - details of the score showing the test status below.

IMAGE — https://images.przeprogramowani.pl/images/2026/05/1X0caqPGHNXvnsEPuPCEqAE.jpeg

Mutant states that you can see in reports:

  • Status: Killed | Meaning: The test fails when changed - this is what we want. | Counts towards score: yes (detected)
  • Status: Survived | Meaning: Tests passed despite the change - weak or missing assertion. | Counts towards score: yes (undetected)
  • Status: No coverage | Meaning: No test even executed this code. | Counts towards score: yes (undetected)
  • Status: Timeout | Meaning: The mutation is probably too complex; tests did not finish. | Counts towards score: yes (counted as killed)
  • Status: Runtime/Compile error | Meaning: The mutant produced invalid or crashing code. | Counts towards score: no (excluded)
  • Status: Ignored | Meaning: The mutation is deliberately filtered out in the configuration. | Counts towards score: no (excluded)

Mutation score - based on the Stryker library - can be calculated as follows:

mutation score = (killed + timeout) / (killed + timeout + survived + no coverage) × 100%

How to read this in practice? In the numerator are the mutants that your tests actually caught - killed (the test failed due to the change) plus timeout (the mutation caused the tests to loop, which Stryker counts as a detection). In the denominator is the entire pool of mutants that could have been detected: those caught plus those that slipped through - survived (the test passed despite the change) and no coverage (no test touched that code). Excluded mutants (compile error, ignored) are not included in the equation at all, so they neither lower nor raise the result. In other words: the denominator is "how many were realistically catchable," the numerator is "how many we caught." A result of 80% means that out of five meaningful mutations your tests would let one pass without blinking - and that one is the signal where the assertion is blind.

Remember - the higher the mutation score, the fewer changes in the code your tests let go unnoticed - which means it is harder to break something without triggering a red report.

Does this testing method mean that code coverage should be completely forgotten? No - these methods answer different questions - coverage finds dead zones ("there is no test here at all"), and mutation score finds weak assertions ("there is a test here, but it's blind"). For AI-based tests, the second signal is more important because the agent rarely leaves code completely uncovered – it leaves code covered only seemingly.

Stryker step by step

In the JS/TS ecosystem, the standard tool for mutation testing is Stryker - getting started at this link.

1. Installation and initialization. Stryker has an interactive wizard that will detect the test runner for you, add plugins, and save the configuration file:

npx stryker init

The wizard will append dependencies to package.json (including @stryker-mutator/core and a runner) and create stryker.config.json.

2. Configuration for Vitest. The minimal but complete file looks like this:

{
  "$schema": "./node_modules/@stryker-mutator/core/schema/stryker-schema.json",
  "testRunner": "vitest",
  "plugins": ["@stryker-mutator/vitest-runner"],
  "mutate": [
    "src/**/*.ts",
    "!src/**/*.spec.ts",
    "!src/**/*.test.ts"
  ],
  "reporters": ["progress", "clear-text", "html"],
  "concurrency": 4,
  "thresholds": { "high": 80, "low": 60, "break": null }
}

What is important here:

  • testRunner + plugins - Stryker runs tests using your real runner (besides Vitest, it supports Jest, Mocha, Karma, among others).
  • mutate - list of production files to mutate (never test files). Here you narrow the gate to what you really want to protect.
  • reporters - clear-text prints mutants directly in the terminal, html generates a clickable report with highlighting of each mutation in the code. Additional options such as json and markdown may be useful if the results are to be interpreted by agents.
  • concurrency - how many mutants to run in parallel; mutation testing is inherently slow, so this directly affects the duration.
  • thresholds - high/low control the report colors; break is the threshold below which the build fails. Leave null as long as you treat this as a selective gate, not a hard gate in CI.

Because mutation testing is still not a standard part of most web projects, add a brief instruction to the agent in CLAUDE.md or AGENTS.md. Let it know that there is an additional quality gate in this repo and that it should not treat it like ordinary coverage.

Example:

## Mutation testing

Repo uses Stryker for selective mutation testing on risk-critical modules.
Run it only for code covered by the current change or a risk from test-plan.md,
prefer narrowed scope with --mutate "path/to/file.ts:start-end", and do not chase
100% mutation score. Survived mutants should be reviewed one by one: add an
assertion only when the mutant represents a user-visible or business-relevant bug.

3. Execution:

npx stryker run

4. Reading the report. After completing the first series of tests, open the generated HTML report and look for mutants in the Survived state. For each one, you will see the exact line, the applied mutator, and the diff before/after, for example:

Mutant survived: src/srs.ts: line 3:10
Mutator: ArithmeticOperator
-   return prevInterval * grade;
+   return prevInterval + grade;

A surviving mutant is proof that no test would catch this change. It is also a concrete task: look for ways to "kill it" through test updates (e.g. better assertions).

5. Narrowing the scope. The first full run on a larger project is slow. Two ways to keep it under control:

  • Mutation range - mutate only a file fragment by specifying the line range: --mutate "src/srs.ts:1-40". Ideal when you want to target one critical module from your context/foundation/test-plan.md.
  • Incremental mode - --incremental (or "incremental": true) saves the result to reports/stryker-incremental.json and on subsequent runs recalculates only the changed files. This makes it possible to actually run Stryker before merging, not just once a quarter.

The loop in which all this lives is short:

  1. you have a risk test from context/foundation/test-plan.md,
  2. you run the tests and see the coverage,
  3. you run Stryker on a narrowed, critical module,
  4. you open the report and select a live mutant (code change not detected by tests),
  5. you ask the agent for a test that will kill this mutation without mirroring the implementation,
  6. you run Stryker again and look at the mutation score (potentially - an improvement in the result).

Stryker and the 10x-Workflow

Stryker does not replace any of the skills. It is an additional quality control loop for tests written with the agent.

You can run it at the end of the entire plan, when all phases are ready. You can also do it between phases, especially when a given phase was supposed to deliver an important regression guard.

In practice, a simple instruction is sufficient:

Run Stryker on the introduced tests in the completed phase.

The Stryker report then becomes material for discussion with the agent. You show a live mutant and ask what behavior it reveals, whether the current test truly protects against the risk, and whether it's worth adding an assertion.

How do you assess the quality of the tests in relation to Stryker's results?

This way, you don't judge the work solely by the green npm test. You continuously deepen your understanding of the tests and the quality of what is being created.

What Stryker found after two sessions with the Test Plan

In our project, we ran Stryker on 10xCards after two real sessions with the Test Plan: Phase 1 (flashcard generation + privacy) and Phase 2 (session saving from drafts). There were 50 green tests on the table.

The result clearly shows the problem from this lesson: coverage reassures, but about 38% of deliberate errors would go unnoticed. The code is covered, but not fully protected by assertion.

Mutation score by file (set of unit tests):

  • Source file: src/pages/api/sessions/[sessionId]/save.ts | Mutation score: 87.5% | killed / survived: 35 / 5 | Verdict: Good - equivalent mutations survived (response cosmetics)
  • Source file: src/lib/save-session.ts | Mutation score: 69.5% | killed / survived: 66 / 29 | Verdict: Branch gaps (see integration disclaimer)
  • Source file: src/pages/api/generate.ts | Mutation score: 66.2% | killed / survived: 45 / 20 (+3 no-cov) | Verdict: Assertions too shallow
  • Source file: src/lib/openrouter.ts | Mutation score: 53.6% | killed / survived: 74 / 62 | Verdict: Most survived mutations are prompt content

The most interesting thing here is not any number, but three specific observations.

1. Stryker helps to separate working tests from apparent ones. We discovered a problem in tests that looked like extra protection but did not detect any mutation. A good example is the "Risk #7" block in generate.test.ts: 6 almost identical tests (timeout, http, parse, unexpected, persistence, happy). Each copied the main test setup and added one assertSentinelAbsent().

The intention was correct: the privacy guard is supposed to ensure that we never log pasted text. The problem is that no code mutation caused the sentinel to leak, so all 6 tests killed 0 mutants. The same contract can be maintained by one parameterized test (it.each([...])) or the general assumption that the risk of input data leakage does not lie in OpenRouter's kind of error.

The lesson is simple: a test can have a meaningful intention and still be redundant in form. If six copies check the same absence of a leak and none provide additional insight, then one well-named assertion is enough.

2. Weak assertions are visible by what survives. Sometimes a test exists, runs the correct piece of code, and still does not protect the behavior. The Stryker report shows this very clearly: a mutation that changes an important decision in the logic survives, and the tests are still green.

In our case, this concerned three types of vulnerabilities. First, the test only checked the general shape of the error, but not the details that distinguish proper validation handling from a simple fallback. Second, the tests covered the write operation but did not sufficiently enforce the condition "create replay state only when a card has actually been created." Third, write idempotency was an important invariant, but some unit tests failed to detect changes in the behavior of upsert.

This is a practical cue for a conversation with an agent: you don’t ask “why is the coverage low?” but rather “what decision in the logic changed so that the test should have failed, but didn’t?”. Only from this does a sensible assertion fix emerge.

3. Not every survived mutant deserves a new test. The Stryker report is not a to-do list for automatic checking off. It is a list of questions: does this change really break behavior that matters to the user or the business?

In our run, many mutations concerned the text of the system prompt, the response header, or a small shape of a technical object. It would be possible to write tests that check the exact wording of the prompt or a specific detail of the response, but that would be a mirror of the implementation. Such a test breaks with every normal refactoring and says little about the product quality.

That's why you consciously leave some survived mutations. Not because you ignore the report, but because after analysis you conclude: "this change does not represent a bug we want to defend against." This is precisely the difference between using Stryker as a tool for thinking and chasing a mutation score of 100%.

A practical takeaway from the whole test? The agent rarely leaves code uncovered. More often, it leaves it covered but unprotected. Stryker shows this difference with numbers and a specific mutant, instead of a green bar that by itself guarantees little.

How Stryker Manages Mutants

The first encounter with mutation testing may raise concerns about the performance of the entire process. Since each mutant is a separate version of the code, does Stryker run the entire test suite once for every mutant? No – a specific optimization makes mutation testing feasible.

Before Stryker starts mutating, it performs one dry run with coverage analysis in perTest mode - mapping which test executes which lines. Then, for a given mutant, it runs only those tests that touch the changed line. A test that never executes this code has no chance to kill it anyway. In practice, for a real module, this results in on the order of several to a dozen tests per mutant, not the entire suite.

The conclusion is important for intuition about cost: time does not come from restarting everything, but from the number of mutants. There can be hundreds of these per file. That is why the main speed control is concurrency (how many mutants to calculate in parallel) and narrowing the scope with --mutate, rather than cutting the number of tests.

One pitfall when narrowing down: if you specify --mutate on a file that has no tests, Stryker will return a "No tests were executed" error. The Vitest runner looks for tests related to the mutated file - target mutation on modules that actually have tests.

Mutation score is not the goal - live mutants matter

The most common beginner mistake: treating the mutation score like coverage and chasing it to 100%. This is a trap, because to kill every mutant, you’ll start writing tests asserting random details – exactly those implementation mirrors that this entire lesson warns against. The race for the score produces fragile tests.

The result is only a hint. The real product is living mutants, and each of them requires a separate testing decision. For each living mutant, ask one question: "Would a real bug of this kind hurt the user?". Hence three categories:

  1. Real bug → add an assertion. The mutation changes behavior that should break something, but it doesn't break anything. This is the only condition where you actually act – for example, replacing >= with > or || with && in error handling.
  2. Equivalent mutant → ignore. The "bug" does not change observable behavior (the value is overwritten anyway, a different message in the log). No test can kill it. Stryker does not know this - you do.
  3. Real change, but not worth pinning → consciously ignore. The behavior has changed, but an assertion on it would be fragile or worthless.

Where It Pays Off

Mutation testing in large projects can indeed be time-consuming, so don’t treat it like a linter to run after every commit. It’s a selective gate that you trigger where potential live mutants most likely require your action. Sensible places:

  • critical domain logic,
  • high-impact code from context/foundation/test-plan.md,
  • a module that changes frequently,
  • an area where coverage looks good, but you still don't trust the tests (typical with agent tests),
  • moment review before an important merge.

Counterparts in other ecosystems include Stryker.NET (C#), Stryker4s (Scala), PIT (Java/JVM), mutmut (Python), cargo-mutants (Rust), and mull (C/C++). The terms are the same: mutant, killed/survived, mutation score.
