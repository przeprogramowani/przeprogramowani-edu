In the previous lesson, you completed the first stream: plan approved, phases closed, commits ready, ## Progress looks clean.

And now comes a very tempting moment: "if it works, then merge".

Exactly. The code may work locally, tests may pass (we will add them in module 3), and the change may still not be ready to merge.

The agent might have solved the correct problem in the wrong way, silently expanded the scope, bypassed the project convention, or added a test that checks exactly the same wrong assumption it just generated.

Sound familiar? In the prework, we talked about the "approve without defense" mode: AI delivers the result, you click accept, but you don’t gain the confidence in your mind that you understand the consequences of the change.

We will not be teaching classic team code review or test planning here — that will come later. Here you learn solo review of code generated by an agent: fast, systematic, and sharp enough not to miss a risky change just because it looks neat.

The code works, but is it ready?

After the previous lesson, you have three things that are a good starting point for review:

  • implemented slice,
  • plan.md as a reference of intent,
  • ## Progress with completed phases and commits.

AI code review without a plan quickly turns into reading the diff "by feel." You look at the change, see a lot of correct TypeScript, a few sensible names, some test, and think: okay, it’s probably fine.

The thing is, AI can write code that looks reasonable even when it solves the wrong problem.

The proper question during a review is: does this change deliver the plan according to the requirements?

This is what /10x-impl-review is for — an organized review of the implementation regarding the plan, scope, project rules, and success criteria. It helps catch many issues before they reach production, but the final merge decision still belongs to you.

Scorecard instead of intuition

/10x-impl-review organizes review through six dimensions:

  • Dimension: Plan Adherence | Question you ask: Does the implementation follow the approved plan?
  • Dimension: Scope Discipline | Question you ask: Did the agent avoid adding changes outside the scope?
  • Dimension: Safety & Quality | Question you ask: Does the change avoid introducing security, data, or stability risks?
  • Dimension: Architecture | Question you ask: Does the solution fit the project's architecture?
  • Dimension: Pattern Consistency | Question you ask: Does the code use local patterns instead of inventing new ones?
  • Dimension: Success Criteria | Question you ask: Are the success criteria truly met?

Scorecard protects you from a common mistake in agent code review. You focus on what is easy to see instead of what can really hurt.

The variable name is easy to notice.

It is harder to notice that the agent added a new API response format that does not match the rest of the application.

It is easy to notice the missing comment.

It is harder to notice that the test passes because it checks the implementation, not the required behavior.

It's easy to notice the file that the agent changed.

It is harder to notice a file that the agent should not have touched.

That is why the scorecard works like an attention filter. First, you look at deviations from the plan, scope, security, architecture, and patterns. Only then do you focus on the details.

Results are not a "fix everything" list

After the review, you will receive findings organized by importance. And here lies a trap: the desire to fix everything.

The agent reports five remarks, so you instinctively say: "fix everything." It sounds responsible, but in practice, it often means unnecessary work on things that do not significantly impact the quality of the change.

That is why we analyze findings through two axes:

  • severity — how serious the problem is,
  • impact — how far the change reaches and how many decisions the fix requires.

Severity tells you whether the problem is critical, warning, or informational in nature.

Impact, in turn, indicates whether the fix is small and local, or touches on architecture, product, data, contract, or the further plan.

These are different things.

A finding can have high severity and low impact: for example, a missing validation in one handler that can be fixed locally. Such an issue is usually fixed immediately.

A finding may have lower severity but high impact: for example, a suggestion to change the name of a status stored in the database. It looks like tidying up an enum, but it affects the frontend, backend, historical data, and documentation.

Here you don't click "fix" out of habit.

The severity × impact matrix is a heuristic, not a rigid rule:

  • Severity / Impact: Critical | Low: Fix now | Medium: Fix carefully | High: Stop and narrow the fix
  • Severity / Impact: Warning | Low: Fix or skip | Medium: Decision depends on context | High: Often requires planning
  • Severity / Impact: Observation | Low: Usually skip | Medium: Leave a note or rule (/10x-lesson) | High: Investigate if the issue affects multiple places

The most important change in your behavior is simple: not every finding needs to be fixed during the review.

Sometimes the best decision is fix now. Sometimes fix differently, because you have a better idea than the agent. Sometimes skip. And sometimes record as lesson — so the agent won't repeat the same pattern.

After running /10x-impl-review you have two ways to work with the results. You can proceed directly to the analysis, or first save the report, review it at your own pace, and only then return to the review by calling /10x-impl-review @path-to-report.

We recommend the second approach, especially for larger changes. You read the report without pressure, build a picture of the whole, and only then do you enter triage.

Triage: finding after finding

The review only ends when you go through the findings and make a decision for each of them.

IMAGE — https://images.przeprogramowani.pl/images/2026/05/iUUTapiMOdupkdUPpbqkwQI.jpeg

We basically have four types of reactions. It's simpler than it looks at first.

1. Fix it now.

Classic fix. The finding is real, significant, and the fix fits within the scope of the change. The agent fixes the problem in place, and you do a quick re-review after the fix.

An example from 10xCards: review detects that the endpoint saving the accepted flashcard bypasses the existing authorization mechanism. This is not a topic for later. It must be fixed before you click merge.

2. Fix it differently.

The agent proposed a fix, but you see a better solution. Maybe the fix is technically correct, but it breaks the local pattern. Maybe there is a simpler way.

Fix differently opens a discussion: you describe your approach, the agent implements it, and you verify the result.

3. Don't fix it now.

Skip fits observations with low impact: the name could be better, but it does not violate conventions, does not increase risk, and does not block the merge.

If the problem is more serious but you consciously postpone it, describe the justification using the “Other” option — this is equivalent to accept risk. If the finding is simply incorrect, enter dismiss. Both cases leave a trace in the report.

4. Build project memory.

Some findings are not one-time errors, but a signal that the agent does not yet know the local project rule.

In the lesson about project rules (M1L4) you learned the mechanism of saving repetitive patterns to lessons.md. Review is a natural moment when these patterns become apparent.

A good review sometimes ends with a new entry in lessons.md. Not to produce documentation for the sake of documentation, but so that the next run of /10x-plan or /10x-implement starts with better context. The more patterns you record, the fewer of the same findings return in subsequent reviews.

If review detects a recurring pattern, you use Record as lesson. The rule goes to context/foundation/lessons.md, and subsequent workflow runs can load it as a prior lesson of the project.

Example: "new server actions must use the existing auth guard pattern, instead of manually checking userId in each handler".

This is not a one-time fix. It is a rule that would change several previous decisions and will return in subsequent slices.

After saving the rule, the agent will ask if you want to apply the fix to the code immediately. You can say yes, then the finding is fixed and saved as a rule. Or no, the rule goes to lessons.md, but the code remains unchanged. It depends on whether the fix fits within the current slice and how many changes you are already making in this PR.

How to read the agent diff

Code review generated by an agent differs from code review written manually. The agent can very quickly generate a consistently looking diff, whose intention has diverged from the plan.

Therefore, read the diff in this order.

1. Scope.

First, check if the agent changed only what was implied by the plan. If the slice concerned saving accepted flashcards, and the diff touches routing, layout, and deploy configuration, stop.

Check if the implementation solves exactly the slice you planned. AI tends to solve a nearby problem: a bit bigger, a bit more general, a bit more "nice".

Maybe there is a reason. But it is a reason that needs to be understood.

2. Risk boundaries.

Then you look at the places that most often cause pain after a merge: authorization, data saving, migrations, external APIs, environment configuration.

"Looks good" is not enough here. You need to know that the code works, is secure, and follows the project guidelines.

3. Local patterns.

Check if the code uses existing helpers, names, error formats, folder structures, and testing rules. The agent often comes up with a technically correct solution that does not fit the project.

This is exactly the Pattern Consistency from the scorecard.

4. Tests.

At the end, check the tests, but not in the "are there any" mode. Check if the test would actually catch the error you want to avoid in the future.

A test written by an agent is also code. A green result is information, but it is not proof that the change makes sense and meets the assumptions from the plan.

We will focus on tests in the next module, but it is worth keeping them in mind during the review stage.

Verdict before merge

At the end of the review you need a final decision.

In practice, you can think of three variants:

  • Approve - findings are fixed, rejected, or consciously accepted,
  • Request changes - the change is close, but requires fixes before merging,
  • Block - the problem is so serious that it is necessary to return to the plan and/or initial assumptions.

/10x-impl-review helps prepare the report and organize the findings, but the verdict is up to you.

You decide whether the change is in line with the plan, design, and standards that are meant to last longer than a single slice.

This basic cycle - plan, implementation, review, triage — closes a single change. In the next lesson, the same workflow will face a more difficult case: a slice, where the choice of library affects the data model, UI, and business logic simultaneously. You will learn new skills that will help you handle this.

📚 Additional materials

  • Google Engineering Practices: What to look for in a code review — a stable reminder that a review covers design, functionality, complexity, tests, names, comments, style, and documentation.
  • Expectations, Outcomes, and Challenges of Modern Code Review — Bird, Bacchelli (Microsoft Research / IEEE ICSE 2013). Older but still useful study showing that review is also about understanding the change and knowledge transfer, not just catching defects.
  • Do Users Write More Insecure Code with AI Assistants? — Perry, Srivastava, Kumar, Boneh (CCS 2023). A study on the security of code created with AI assistance. Good as a warning against false confidence, not as a universal benchmark for all agents.
  • Asleep at the Keyboard? Assessing the Security of GitHub Copilot's Code Contributions — Pearce et al. (IEEE S&P 2022). A historical signal that seemingly correct AI suggestions require review from a security perspective.
  • Claude Code Review documentation — analysis of the code review system in Claude Code, it's worth seeing what is similar and what works differently compared to /10x-impl-review.
  • SWE-PRBench: Benchmarking AI Code Review Quality Against Pull Request Feedback — Kumar (2026). A benchmark showing that AI review still falls short of an expert, and that context strategy matters more than its quantity.
