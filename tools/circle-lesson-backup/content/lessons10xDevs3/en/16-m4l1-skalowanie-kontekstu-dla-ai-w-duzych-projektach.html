IMAGE — https://images.przeprogramowani.pl/lessons/m4-l1/assets/cover-en.jpg

In the Agent Onboarding lesson (M1L4) we implemented the agent for one project: a concise AGENTS.md file of 50-100 lines was created. It worked because the project had not yet grown to a scale where one file can no longer contain all the conventions and procedures of the entire system.

Welcome to the fourth module. Here we work with large and legacy projects, meaning those that have several modules, their own subsystems, and a history that no one fully remembers anymore.

And here the first reflex fails. "Since AGENTS.md works, I'll just keep appending more instructions to it" sounds reasonable, but only up to a point. At some point, the file becomes so large (300+ lines) that the agent starts losing track of the most important rules in it.

The second reflex is the opposite and just as costly: just in case, you create instruction files in every folder and take on the burden of managing this distributed context architecture before the project truly needs it. You then pay the price for maintaining a structure that yields little benefit.

So we start this entire module not with tasks, but with the foundation. First, we will discuss the context architecture that scales with the project, and only then will we tackle challenges: code mapping, feature analysis, refactoring, and legacy modernization.

This is actually the consistent rhythm of this module. Each lesson starts with a specific pain point that you feel with large or legacy code, and only then does the technique that addresses this pain come in. Challenge lessons, from the second one onwards, follow the same scheme: lens (what we focus on), tools and data (which allow extracting and transforming information from the repository), and report element (what remains from this work). This lesson is the foundation for the entire module: its lens is the context architecture, but the first report element will only be created in the second lesson.

The question in this lesson is not "what should the agent know?". It is: what should the agent know right now?

The economy of context at large scale

Let's start with the mechanism, because without it the rest of the lesson would be just a set of rules to memorize.

In the prework [3.1], the concept of Maximum Effective Context Window was introduced: the context window is a finite resource, and its usefulness decreases the more you load into it. The model has a limited attention budget at its disposal, not an unlimited storage.

Despite the growing popularity of models with an official one-million-token window, this upper limit is significantly overestimated compared to the effective window. Roughly: the strongest models maintain quality up to about 250 thousand tokens, while most cheaper alternatives noticeably degrade already around 100 thousand. Treat these numbers as directional, as degradation depends on the model and type of task.

In the onboarding lesson (M1L4), we looked at this from the perspective of a single file. Now we are applying the same principle higher up, at the level of file and directory architecture in a large project. The more context you hold "constantly," the less attention is left for the current task, and the agent's work quality gradually decreases.

The same principle will soon resonate in the code: since the attention is finite, the agent (and you) will not read the entire repo at once. This is progressive disclosure of scale: a theme that will recur throughout the module as we consciously start to read increasingly narrower fragments of code. But we will return to this in the following lessons.

What exactly breaks when a single instruction file grows endlessly? The OpenAI team in the harness engineering article names four failure modes of a monolithic context:

  • Pushing the task out - the more fixed instructions there are, the less space and attention remain for what you are doing here and now.
  • Dilutes the tips - when everything is important, nothing is important, and the key rule gets lost among dozens of side details.
  • Rot - rules become outdated faster than anyone reviews them, and the file turns into a collection of obsolete policies.
  • Complicates verification - the longer the file, the harder it is to check whether the agent is following it at all.

This is not theory detached from practice. Anthropic also warns that a cluttered CLAUDE.md causes Claude to ignore your real instructions. Not because it rejects them, but because they get lost in the noise.

This is also described by two studies that you will find at the end of the lesson in the Deep Dive section. For now, you need one conclusion from them: excessive, unnecessary context can not only fail to help but also increase the cost and worsen the result. This was specifically measured on the OpenAI Codex agent for small PRs, so treat these measurements as directional rather than as a universal guarantee for every tool.

Before you rebuild anything, start with the hygiene of what you already have: cut out the rules that are not essential to make room for those that actually protect the agent from repeating the same mistakes.

When hygiene alone is no longer enough, it's time to move some content to where the agent will reach for it only when it really needs it.

Architecture: lean root + context/

Since we can't maintain one large file indefinitely, what should a good structure look like?

Let's start with the division into three types of context that we use when working with agents:

  • conventions (how we work in this project, not directly inferred from the code) → AGENTS.md / CLAUDE.md plus files with rules matched to the technology in the rules/ folder,
  • references (PRD, plans, research, decisions) → context/,
  • procedures (repeatable steps) → skills, prompts.

After modules 1-3 you already know that each of these elements plays an important role in programming with agents. In this module, you will generate code maps, feature analyses, and refactoring plans – all these artifacts will go into context/.

The starting point is one change in thinking: the root AGENTS.md is a table of contents, not an encyclopedia. It holds what concerns the entire project and is constantly needed (conventions, the most important commands, pointers to the rest), not all the knowledge about the project.

IMAGE — https://images.przeprogramowani.pl/images/2026/06/SHslaqqwN6SBkdUP7fDq2A8.jpeg

The rest, that is PRD, plans, research, and decisions, ends up in a separate layer: the context/ directory, which serves as the system of record.

The agent reaches there just-in-time: sometimes on its own, when the task requires it, but more often because we provide it with a reference to a specific file, knowing that it will be needed in the current work.

Here is the division of roles: the agent performs, and we are responsible for providing the context needed for the current task. Auxiliary mechanisms, such as automatic activation of skills or file reading, only partially relieve us of this duty.

IMAGE — https://images.przeprogramowani.pl/images/2026/06/YHslaqmkL5_knsEPy6De0AE.jpeg

Where to start?

When we start working with AI on any project, we don't need to immediately create an elaborate context structure. One CLAUDE.md / AGENTS.md file serving as the project map and a context/ folder for all in-depth documentation is enough.

This is how it looks in 10xCards, our course application at the MVP stage. Its CLAUDE.md has 82 lines, and the sections speak for themselves: Project, Commands, Architecture, Conventions, Reference. No PRD, no plans, just what is needed constantly and globally.

All in-depth knowledge is stored in context/:

  • context/foundation/ - durable foundations: prd.md, roadmap.md, tech-stack.md, test-plan.md,
  • context/changes/<id>/ - individual changes along with their plans and research,
  • context/archive/ - completed changes go there so they don't clutter current work.

AI workflow procedures in the project are stored in folders with skills and prompts.

How does it work in practice? In short, since you already know the skill mechanics from the first module:

  • /10x-init creates the context/ skeleton,
  • /10x-agents-md generates a lean root that refers to context/ instead of pulling it into itself,
  • /10x-rule-review checks whether the root hasn't swollen and whether the rules deserve their place.

This is the first, and for most projects the only necessary, stage of evolution: one root AGENTS.md + one centralized context/ folder.

We start precisely with centralization, because as long as the project allows it, keeping the entire context in one place simply makes work easier.

Adding structure closer to the code, such as a separate AGENTS.md, and eventually even a separate context/ per module, are next steps. You reach for them when the project truly needs it, not prematurely. We'll return to this later, so for now, don't rush.

Maturity Ladder and Escalation Signals

Since we are starting with centralized context management, the question arises: when should we even add structure? This is where the maturity ladder comes in.

Let's clarify the most important thing right away: this is a ladder you climb on demand, not a checklist of steps to tick off one by one.

Rungs look roughly like this:

  1. root AGENTS.md + folder context/ - the starting point of every project,
  2. AGENTS.md per module + an index of these files in the main AGENTS.md - when a module is complex and has its own specific patterns and conventions: key for the agent operating in its area, and irrelevant for other modules. The second reason is a module maintained by a dedicated team, independent from the rest of the monorepository,
  3. own context/ in the module - when the module requires a dedicated PRD, roadmap, and other artifacts because it is led by a separate team or the work on it is intense enough that the main context/ is no longer sufficient.

IMAGE — https://images.przeprogramowani.pl/images/2026/06/e3slao2hE6LOnsEPiLzb6Q0.jpeg

However, the most important are not the steps, but the signals that justify moving higher:

  • main AGENTS.md grows and becomes unreadable (Anthropic suggests an approximate goal of about 200 lines per file, we recommend splitting above 300 lines),
  • the agent repeatedly loses the context of the specific module and makes mistakes, despite your corrections and instructions in the main AGENTS.md,
  • with every task in the module, you give the agent more and more references to its documentation or you lack a dedicated PRD or roadmap there because you carry out a lot of complex work,
  • the module gains a separate deploy or owner, that is, a real boundary of responsibility.

Notice what the real decision rule is here. Not the number of folders or the feeling that "my application is large".

You add structure where the situation requires it: you know this from the agent's failures in specific tasks or from the real line of responsibility between teams.

Calibration: your MVP vs a larger repo

Since the structure is supposed to grow based on signals, where is your stopping point? A good point of reference is the repositories of projects developed using AI.

Let's start with the 10xDevs course projects, including your project developed from the first module. Projects at the MVP stage have only a few modules with limited complexity and interdependencies. For this scale, the answer is simple and somewhat against the engineering instinct: stay at the first level for as long as possible. One AGENTS.md + a centralized context/ is more than enough, and creating per-folder files would be paying for a structure that yields no benefit.

Many of you also work in larger corporate projects, so we analyzed several well-known repositories to see how they approach context scaling.

What does "good" look like at a larger scale?

  • cloudflare/workers-sdk is a pattern to learn from: main, concise AGENTS.md, explicit index of descendant AGENTS.md files and descendant AGENTS.md files that only add local context and refer back to the root (AGENTS.md in the module says: "Wrangler-specific context only. See root AGENTS.md for monorepo conventions.").
  • open-mercato shows what the hard variant looks like: over 400-line root, an extensive instruction router, and a large number of AGENTS.md per module.
  • openai/codex is, in turn, minimal nesting: a 286-line root and only one child AGENTS.md, even though the repo has dozens of modules. Context scaling is not a convention there. And a nice detail at the end: this is the same OpenAI that, in the mentioned text about harness engineering, praises lean, about 100-line root as a table of contents, and yet in its public repo keeps an elaborate, 286-line file.

And the most important calibration warning: these numbers are a derivative of the adopted strategy, not the goal. There is no "right" number of instruction files you should aim for. It all depends on how you work with AI and the challenges you face.

One caveat, so you don't draw the wrong conclusions: here we are looking at the layout of these repositories, not whether it actually improves the agent's performance, because no one has measured effectiveness. Probably, over time we will see comprehensive studies on how decentralized context architecture affects the efficiency of agents in large repositories, but currently we rely only on company case studies and anecdotal evidence.

So do not transfer patterns from cloudflare/workers-sdk or open-mercato directly to your MVP or work project. Stick to a centralized context until you see problems arising from it. You will reach for nested context files only when the agent sends signals known to you from the ladder.

How to extract context for complex modules

You have already absorbed the "simple by default" rule. Your project stays at the first level and that is a good decision.

But what if you have a repository with several complex, old modules that no one fully understands, each with its own specific conventions, and you feel that a single root is no longer enough?

The ladder told you when to go up. Now we will show how, without prematurely complicating the context architecture.

When does a module get its own AGENTS.md? It's the second rung of the ladder. Only when it has conventions specific and detailed enough that they won't fit in the root, which is already large anyway (roughly above ~300 lines, but that's a feeling, not a threshold).

If this condition is met, you create a file AGENTS.md in the module and move the conventions of that module into it. In the file, you note that the rest of the general rules reside in the root AGENTS.md, and in the root you add an index with a list of child AGENTS.md files.

This is best seen in a live example from cloudflare/workers-sdk. The root holds the entire monorepo convention and ends with an explicit list of children:

# AGENTS.md (root) — ~245 lines, conventions of the entire monorepo
...
## Packages with their own AGENTS.md for deeper context
- packages/wrangler/AGENTS.md  — CLI architecture, command structure, test patterns
- packages/miniflare/AGENTS.md — Worker simulation

And the descendant AGENTS.md in the module opens with an inheritance line and adds only what cannot be inferred from the root:

# packages/wrangler/AGENTS.md — ~31–91 lines
Wrangler-specific context only. See root AGENTS.md for monorepo conventions.

## Gotchas
- Entry point is `src/cli.ts`, NOT `src/index.ts`
- No `console.*` — use the `logger`
- No global `fetch` — use undici

## Anti-Patterns
- ...

Note the division of work. The root says "this is how we work across the entire repo and where to look for details," and the child file says "I am only responsible for the Wrangler, the rest I inherit from the root." The child AGENTS.md does not duplicate the monorepo conventions, but only adds its specific nuances – it assumes the presence of context from the root and copies nothing from it.

You can view both files in the repository: root AGENTS.md and packages/wrangler/AGENTS.md.

When does a module get its own context/? This is the third rung of the ladder. It happens when it requires a dedicated PRD, roadmap, and other artifacts that can no longer be reasonably maintained in the main context/.

This makes sense in two situations:

  1. The module is maintained and developed by a dedicated team that works in a monorepository alongside other teams.
  2. The module is complex enough, and the work on it intensive enough, that a separate context/ helps organize the work and speeds up its pace.

Only then is it worth initializing context/ via /10x-init inside the given module. In the module's AGENTS.md, you then follow the same rule as in the root: leave information that the dedicated documentation resides in the module's context/, with a brief description of the structure, but without duplicating its content.

The rule that keeps this in check is simple: there is always one context/ in the root, and you add your own context/ in a module only when its absence starts to really bother you.

Do not assume context/ in every module "just in case". This is the same premature structure we warned against in the case of AGENTS.md.

The decision rule is the same: divide according to the boundary of ownership, taking into account complexity and real needs, not based on the number of folders.

IMAGE — https://images.przeprogramowani.pl/images/2026/06/l3slaumaJeT9nsEP7uT_6A0.jpeg

For the module rules to really work

Extracting the context is only half the success. There are two places where it is easy to lose the benefits of this operation.

The first is the location from which you launch the agent, and this may be counterintuitive. The directory from which you start the session determines which instruction files will automatically be included in the context.

In Claude Code, when you start from the root of the repo, only the root CLAUDE.md is loaded initially. The module file is loaded lazily, only when the agent starts working within that module. Importantly, this dynamic loading operates with varying levels of reliability: the agent may skip module rules if we do not explicitly provide it with a reference.

When you start an agent from within a module, its rules file plus all parent files are always loaded, and this is completely predictable. (However, there are differences between tools; we break down the full mechanics in the Deep Dive.)

The conclusion is simple. If the module's rules are critical and must work, you have two options: run the agent from inside the module or explicitly pass the file reference if you are working from the root.

IMAGE — https://images.przeprogramowani.pl/images/2026/06/tHslauffLKiEnsEPyZCmqAw.jpeg

The second place is maintenance. When a module gets its own files, you take on their care, because otherwise in six months they will start causing harm with outdated instructions.

Two techniques realistically slow down the degradation of rules in the repository:

  • Do not put things that change quickly into a file - a slowly changing file should not hold rapidly changing facts. Therefore, it is not worth creating per-module files at an early stage of the module's life when changes occur frequently and are extensive.
  • Do periodic context review - for example once a month or quarter, combining /10x-rule-review with your own critical analysis of individual instructions.

Remember to expand the context architecture consciously: one module at a time, in response to a signal, and not just because "it's expected to have a structure at work".

You already understand the context architecture and know how to scale it in a monorepo. How to handle multirepo, that is scaling context when working across multiple repositories, is discussed in the Deep Dive.

The same attention budget that determines context management also dictates how we perform tasks and navigate code in large projects. Just as we cannot read the entire documentation, we certainly cannot load the entire repository code into the context window.

However, we can extract key information point-by-point with an agent: understand the entire project at a high level or thoroughly examine the operation of a selected module. Based on this, we will carry out refactoring and modernization of individual modules, even if the repository contains hundreds of thousands of lines of code in thousands of files.

How tools really load files

Let's start with one mental model, because it protects you from a whole series of mistakes: tools load instruction files through extension, not replacement. The file closest to the task adds to the root, it does not delete it. If you assume that "the closest file wins and replaces the rest" — that's not how it works.

Now for the specifics, because the devil is in the details of every tool.

Claude Code goes from the root of the file system down to your working directory and concatenates the encountered files (the closer one is read at the end, so it appends rather than overwrites). Files in subdirectories load lazily, only when the agent reaches that part of the tree. Imports @path expand at the start of the session and, importantly, do not save tokens - they simply paste the content.

It's easy to check this on your own. Start a session in a monorepo subdirectory, and the entire chain of files from the root to your directory will be loaded at startup. Start it from the root itself, and the files from the subdirectories will only be loaded upon the first access of a given module.

Codex does it differently, and this is the best example of "my tool doesn't load it the way I think". First, it detects the repo root by going up to the git directory, and then loads files downwards, from the root to your directory, merging them into a single document built once at startup. There is no lazy loading, and the file is not read again during the session.

There is also a catch that is easy to forget: the total size of these files has a limit (project_doc_max_bytes), and once exceeded, the content is silently truncated. The limit is on the order of several tens of kilobytes and varies between versions (you may encounter 32 KiB and 64 KiB), so don't get attached to a specific number, just check it for your version. And one more important thing: this limit is shared for all concatenated files, not counted separately for each file.

Cursor breaks this down into four types of rules: always active, intelligently selected based on the task description, pinned to files through globs, and called manually. This allows linking rules with a theme, not necessarily with a directory.

Copilot, on the other hand, merges rather than selects: the rule matched by applyTo and the general rule of the repo operate simultaneously. This is again the same principle of additivity.

Finally, a subtlety that surprises when switching between tools. A common file name does not guarantee common behavior. The agents.md specification long only said that "the nearest file takes precedence," which was ambiguous (merge or replace?). Only the proposed version 1.1 clarifies that the local AGENTS.md extends parent files instead of replacing them. This is still a proposal, not an approved standard. And one more thing: among popular tools, Claude Code is the only one that does not read AGENTS.md (it uses CLAUDE.md).

Multi-repo: challenges

Everything we have described so far works within a single repository. Nesting files close to the code, gluing them from the root to the working directory, the entire context/: all of this ends exactly at the repository boundary. An agent working in repo A by default does not see the code or conventions of repo B.

A monorepo encapsulates complexity in the file tree, where tools help you because they see all the code and context in one place.

Multirepo (also known as polyrepo) shifts it to distribution and merging of context across multiple repositories. To do this, you need your own infrastructure: a package, CLI, or MCP server.

We must therefore reckon with a higher cost of implementation and maintenance. This is exactly the same kind of friction you know from sharing code between services: there you add infrastructure for synchronization, communication, and recovery from failures, and here, similarly, for distribution, synchronization, and maintenance of the shared context.

Here arises a natural reflex that you probably feel: to build one central "meta-context" or a separate agentic repo that aggregates knowledge about all services and dependencies among them. Unfortunately, such a central meta-context quickly becomes outdated because it tries to describe many independently changing services at once.

When we want to share context between several repositories, the problem is divided into three layers:

Layer 1: universal context for all repos. What should apply everywhere: shared skills, prompts, and universal instructions for AI (company conventions, security policies). We want every agent, regardless of which repository they work in, to have access to the same consistent set of tools and rules.

Layer 2: distribution of common context to each repo. We do not want to duplicate the universal context from layer 1 by copy-pasting. The problem is not in duplication itself, but in maintaining the currency of copies scattered across repositories.

If our approach does not provide automatic updates and synchronization, sooner or later the context in individual repositories will start to diverge, and manually reconciling the differences will become costly.

The solution is a single source of truth in a dedicated repository of type company/ai-toolkit combined with a distribution and update mechanism.

In the dedicated repository, we save all shared rules, skills, prompts, and configuration files.

Next, we need to provide a mechanism for distributing and synchronizing this context to the repositories that will use it.

In practice, you will encounter three popular mechanisms:

  1. package in the company registry - e.g. npm package @przeprogramowani/ai-toolkit,
  2. CLI - e.g. 10x-cli, which retrieves context via API from the source of truth repository (przeprogramowani/10x-toolkit),
  3. MCP server - providing the same context through an API from the source of truth repository.

The order is not accidental: the list goes from the simplest to the most complex implementation.

Just remember that each repo will still need its own committed context files according to the rules we described in the main part of the lesson.

The above solutions only provide us with the distribution of the universal context (rules, instructions, skills, prompts) and its update.

Layer 3: cross-repo visibility at runtime. Universal rules and distributed files still do not allow the agent in repo A to understand the workings of repo B, which it interacts with. Here, we need a way to obtain the current specification of the other system components.

You can generate such a specification through an LLM, but this is another dedicated infrastructure that brings additional complexity, costs, and risks.

Therefore, it is worth reaching for proven methods of automatically sharing specifications, which we used even before the times of AI. Three examples:

  1. Automatically generated API contracts (e.g., OpenAPI/Swagger for REST, GraphQL SDL, Protobuf definitions for gRPC, AsyncAPI for event-driven architectures)
  2. Automatically shared test specifications for UI and API (e.g., as a package)
  3. Automatically generated code documentation (e.g., TypeDoc)

All these methods, when automated in CI/CD in a way that ensures freshness, provide other services with reliable information about how a given module works and do so more cheaply and reliably than artifacts generated by LLMs.

Tools like Repomix, which pack the entire repository into a single file for AI, should be treated as a temporary snapshot (when an agent needs to look into repo B just once), rather than a permanent contractual infrastructure: instead of a narrow, explicit specification, you put the entire code into the context and pay a high token cost for it (both when generating the snapshot and when reading it).

These three layers are stacked on top of each other: each solves a different problem and none replaces the others.

IMAGE — https://images.przeprogramowani.pl/images/2026/06/5nslasroOqSBkdUPzZvooQ4.jpeg

Due to this additional overhead of complexity and infrastructure, we encourage avoiding premature breaking down of monoliths into microservices. This advice was repeated even before the era of AI, and working with agents gives us further reasons to approach this decision with great caution.

And here we draw the line for this lesson. You already know the shape of the solution, but the practical construction of this shared infrastructure for working in a multirepo (Shared AI Registry: skills, commands, and rules for the team) is broken down in a separate lesson (M5L3). This is a real problem only for companies working with multiple repositories.

What research says about AGENTS.md

Let's go back for a moment to the conclusion from the section on context economy. It is supported by two separate studies, both on the OpenAI Codex agent and for small PRs, so consider their numbers as directional.

In the first well-utilized, concise AGENTS.md it was associated with a lower median task completion time (about 28%) and fewer generated tokens (about 17%). This is an argument for efficiency, not a promise of improved accuracy for every agent.

The second looks from the opposite perspective: excess, unnecessary context could worsen the outcome — task performance dropped by over 20%, while the cost increased. The key culprit is redundancy and unnecessary requirements, not the file length itself. Hence the discipline: the instruction file should describe the minimum of what is necessary.
