Wprowadzenie

W poprzedniej lekcji poznałeś dwa uniwersalne narzędzia, dzięki którym twoja aplikacja oparta o Reacta i TypeScript pobierze dane z zewnętrznego serwera. Wspomniana uniwersalność Fetch API i axiosa to jednocześnie zaleta i wada. Z jednej strony, możemy ich używać niezależnie od ulubionego frameworka i zyskujemy łatwy transfer wiedzy, ale z drugiej strony piszemy kod nadmiarowy, który nie zawsze pasuje do funkcyjnego charakteru komponentów Reacta.

W tej lekcji wprowadzimy bardziej zaawansowane rozwiązania, które tworzone są stricte pod najpopularniejszy obecnie framework frontendowy. Zapewniają one również wysokie wsparcia dla potencjału TypeScriptu, dzięki czemu kod i dane stają się łatwiejsze w obsłudze.

Pobieranie danych i zarządzanie stanem

swr (https://swr.vercel.app/)

SWR to rozwijana przez Vercela biblioteka do pobierania i zarządzania danymi w aplikacjach React, która opiera się na strategii "Stale-While-Revalidate". Umożliwia prostą obsługę asynchronicznego pobierania danych, cachingu i automatycznego odświeżania w tle, poprawiając UX i wydajność aplikacji.

Zalety

  • Automatyczne cachowanie i deduplikacja zapytań – dane są przechowywane w globalnym cache, co pozwala unikać powielania zapytań o ten sam zasób.

  • Odświeżanie w tle w strategii “stale-while-revalidate”, co może mieć pozytywny wpływ na doświadczenia użytkownika.

  • Jeden główny hook (useSWR) do obsługi większości scenariuszy.

  • Wsparcie dla TypeScriptu – generyki umożliwiają łatwe typowanie danych i błędów.

Wady

  • Niektóre scenariusze i strategie zarządzania danymi mogą wymagać większego wysiłku niż w przypadku Fetch API i Axiosa

React Query (https://tanstack.com/query/v5)

React Query to biblioteka do obsługi kompleksowych scenariuszy związanych z asynchronicznym zarządzaniem stanem aplikacji. Wprowadza abstrakcję nad żądaniami HTTP i dostarcza zaawansowane funkcje sterowania cachem, synchronizacją danych i automatycznym odświeżaniem treści na stronie.

Zalety

  • Integracja z TypeScriptem – obsługuje generyki, które typują zarówno dane zwracane, jak i parametry zapytań.

  • Hierarchiczne klucze cache

  • Rozbudowana konfiguracja obsługi przechowywania danych w cache

  • Automatyczne ponawianie zapytań

  • Wsparcie dla anulowania zapytań z wykorzystaniem AbortControllera

Wady

  • Wymaga nauki nowych koncepcji dotyczących stanu i cache (np. query keys, stale state).

  • Podobnie jak swr, dodaje nową warstwę abstrakcji, co może być zbędnym narzutem w prostych aplikacjach.

Podsumowanie

Poznałeś już cztery popularne podejścia do wymiany danych w aplikacjach opartych o Reacta i TypeScript. Każde z nich sprawdzi się w określonych scenariuszach - Fetch API / axios zapewnią uniwersalne API w projektach o różnorodnym stacku technicznym, natomiast swr / React Query spełnią oczekiwania wymagających użytkowników Reacta i TypeScriptu.

Patrząc po trendach, to swr jest obecnie na fali wznoszącej, ale trudno nam określać jedno z przedstawionych tutaj rozwiązań jako jednoznacznego lidera - oba rozwiązania dostarczają zadowalający poziom integracji z systemem typów, złożone strategie obsługi danych oraz ich optymalizację. Każde z nich sprawdzi się w wymagającym projekcie i każde zebrało już znaczną liczbę fanów na GitHubie (30k gwiazdek dla swr, 42k dla React Query).
