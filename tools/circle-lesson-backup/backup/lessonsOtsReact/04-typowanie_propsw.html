W tej sekcji omówimy szczegóły i podejścia do typowania propsów w komponentach React z TypeScriptem, porównując dwa główne podejścia: bezpośrednie typowanie oraz wykorzystanie wbudowanego typu React.FC. Wyjaśnimy różnice między tymi podejściami, wskażemy ich zalety oraz naszą rekomendację: z czego korzystać.

Po ukończeniu tej sekcji będziesz w stanie:

  • Typować propsy komponentów React z wykorzystaniem bezpośredniego typowania oraz React.FC

  • Wybrać odpowiednie podejście do typowania w zależności od specyfiki projektu

Typowanie propsów z React.FC

React.FC (skrót od React.FunctionComponent) jest wbudowanym typem TypeScript dostępnym w bibliotece React, który pozwala na szybkie typowanie komponentów funkcyjnych. Dzięki React.FC, komponent automatycznie posiada typ ReactNode dla zwracanej wartości oraz obsługę domyślnej właściwości children.

// Definicja typu propsów
type ButtonProps = {
  label: string;
  onClick: () => void;
};

// Typowanie komponentu z React.FC
const Button: React.FC<ButtonProps> = ({ label, onClick }) => (
  <button onClick={onClick}>{label}</button>
);

// Dodanie właściwości statycznych z automatycznym typowaniem
Button.defaultProps = {
  label: 'Click Me',
};

Zalety React.FC:

  1. Podpowiedzi dla statycznych właściwości komponentu
    Dzięki React.FC, TypeScript automatycznie rozpoznaje statyczne właściwości komponentów, takie jak defaultProps czy displayName. Jako że z tych właściwości korzysta się sporadycznie, nie jest to istotna zaleta.

// Bezpośrednie typowanie - brak autocompletiona
const Button = (props: ButtonProps) => <button {...props} />;

Button.defaultProps = { // TypeScript nie podpowiada
  variant: 'primary'
};

// React.FC - działający autocomplete
const Button: React.FC<ButtonProps> = (props) => <button {...props} />;

Button.defaultProps = { // TypeScript podpowiada dostępne propsy
  variant: 'primary'
};

  1. Elastyczne typowanie zwracanych wartości
    Od TypeScript 5.1/React 18, React.FC poprawnie obsługuje zwracanie różnych wartości, takich jak undefined, string czy number, bez powodowania błędów typowania. Tak samo działa to dla bezpośredniego typowania, które omówimy zaraz.

  2. Brak wymuszonych children
    Przed TypeScript 5.1/React 18, React.FC również automatycznie dodawał children do typu propsów, co mogło prowadzić do błędów, jeśli komponent ich nie przyjmował. Teraz ta właściwość została usunięta, co pozwala na precyzyjniejsze typowanie komponentów bez obawy o niechciane children. Tak samo działa to dla bezpośredniego typowania.

Bezpośrednie typowanie propsów

Bezpośrednie typowanie propsów pozwala na precyzyjne określenie struktury i typów wymaganych w komponencie React bez dodatkowego narzutu wynikającego ze stosowania React.FC. W tym podejściu definiujemy typy propsów, a następnie stosujemy je bezpośrednio w parametrze komponentu.

// Definicja typu propsów
type SelectProps = {
  value: string;
  onChange: (value: string) => void;
};

// Bezpośrednie typowanie propsów w komponencie
const Select = ({ value, onChange }: SelectProps) => {
  return <input value={value} onChange={(e) => onChange(e.target.value)} />;
};

Zalety bezpośredniego typowania propsów:

  1. Czytelność i prostota
    Bezpośrednie typowanie zapewnia intuicyjną i czytelną strukturę. Łatwo jest zrozumieć, jakie propsy przyjmuje komponent, bez dodatkowych obiektów lub właściwości domyślnie dodawanych przez React.FC. Komponent jest jasno typowany, co zwiększa przejrzystość kodu dla innych programistów.

  2. Lepsza obsługa generyków
    Przy tworzeniu #Komponenty generyczne bezpośrednie typowanie jest bardziej naturalne i wymaga mniej skomplikowanej składni niż React.FC, co poprawia czytelność typowania.

    // Bezpośrednie typowanie - naturalna składnia
function Select<T,>(props: SelectProps<T>) {
  return <select {...props} />;
}

// React.FC - bardziej złożona składnia
const Select: React.FC<SelectProps<T,>> = <T>(props) => {
  return <select {...props} />;
};

  3. Unikanie nadmiarowego typowania
    Bezpośrednie typowanie nie dodaje dodatkowych właściwości, takich jak children czy domyślny typ zwracanej wartości. Dzięki temu jest bardziej efektywne pod względem wydajności edytora (różnica jest zauważalna w dużych projektach) i zmniejsza szansę na przypadkowe dodanie nadmiarowych danych do komponentu.

Z czego korzystać?

Dla wersji TypeScript 5.1/React 18, React.FC jest akceptowalnym sposobem typowania komponentów, pozbawionym wcześniejszych problemów, takich jak domyślnie dołączane children i błędy przy zwracaniu typów innych niż ReactElement.

Mimo to, rekomendujemy bezpośrednie typowanie propsów w większości przypadków, ze względu na następujące korzyści:

  1. Prostsza składnia i łatwość refaktoryzacji- bezpośrednie typowanie wymaga mniej dodatkowych konstrukcji i jest łatwiejsze do refaktoryzacji. Dzięki temu to rozwiązanie jest bardziej zrozumiałe dla programistów, którzy dopiero zaczynają pracę z TypeScriptem.

  2. Większa elastyczność dla komponentów generycznych - bezpośrednie typowanie lepiej obsługuje komponenty generyczne, eliminując potrzebę stosowania złożonej składni, która bywa nieczytelna przy użyciu React.FC.

Jednak warto mieć na uwadze, że React.FC można stosować bez obaw o błędy czy problemy. Stąd jeżeli pracujesz nad projektem z wersją React 18/TypeScript 5.1+ i spotkasz tam React.FC nie warto poświęcać czasu na refaktoryzację do bezpośredniego typowania propsów.

Jednak w nowych projektach, ze względu na elastyczność i prostotę kodu, bezpośrednie typowanie propsów jest rozsądniejszym podejściem.

Podsumowanie

W kolejnej lekcji skupimy się w typowaniu propsa children. Jest to kluczowa cecha, która wpływa na elastyczność i reużywalność komponentów. Omówimy różne sposoby definiowania typów children, w tym scenariusze, gdy children są elementami Reacta, funkcjami lub strukturami danych.
