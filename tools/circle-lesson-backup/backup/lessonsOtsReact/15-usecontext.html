W tej lekcji omówimy, jak tworzyć i typować kontekst, jak bezpiecznie konsumować go w komponentach oraz jakie są najlepsze praktyki w zapewnieniu spójności typów w całej aplikacji.

Jak bezpiecznie stosować Context API?

Kontekst (Context API) pozwala na przesyłanie danych (np. aktualny motyw, dane użytkownika) przez strukturę komponentów bez konieczności jawnego przekazywania propsów na każdym poziomie. W tej lekcji zaprezentujemy dwie strategie pracy z Context API w React i TypeScript:

  • Zapewniona wartość domyślna – sytuacja, w której kontekst zawsze ma sensowną domyślną wartość, eliminując konieczność obsługi błędów.

  • Jawny null jako wartość domyślna – strategia wymagająca obsługi pustego stanu, ale oferująca większą elastyczność i kontrolę nad typowaniem.

Zarządzanie trybem ciemnym/jasnym z wartością domyślną

W tym przykładzie zaimplementujemy system przełączania między trybem jasnym a ciemnym, gdzie możemy zapewnić sensowną wartość domyślną.

Definicja typu i kontekstu 

type Theme = "light" | "dark";
interface ThemeContextType {
  theme: Theme;
  toggleTheme: () => void;
}
const ThemeContext = createContext<ThemeContextType>({
  theme: "dark", // Domyślny motyw
  toggleTheme: () => {
    console.warn("toggleTheme called outside of ThemeProvider");
  },
});

  • Theme: Prosty unia typów reprezentujących dwa możliwe stany trybu: jasny i ciemny.

  • ThemeContextType: Definiuje strukturę obiektu kontekstu, zawierającą bieżący motyw (theme) i funkcję (toggleTheme), która pozwala na zmianę motywu.

  • createContext: Tworzymy kontekst z domyślną wartością “dark”. toggleTheme domyślnie loguje warning, ponieważ nie mamy mechanizmu zmiany stanu - zaimplementujemy go dopiero na poziomie providera. Dzięki temu mamy sensowną wartość domyślną na wypadek użycia kontekstu motywu poza providerem.

Tworzenie providera kontekstu

const ThemeProvider = ({ children }: { children: ReactNode }) => {
  const [theme, setTheme] = useState<Theme>("light");

  const toggleTheme = () => {
    setTheme((prev) => (prev === "light" ? "dark" : "light"));
  };

  return (
    <ThemeContext.Provider value={{ theme, toggleTheme }}>
      {children}
    </ThemeContext.Provider>
  );
};

  • ThemeContext.Provider: Dostarcza wartość kontekstu do komponentów podrzędnych, które jej potrzebują.

    • Stan zarządzany przez useState: Przechowujemy bieżący motyw w stanie komponentu ThemeProvider.

    • toggleTheme: Funkcja zmienia wartość stanu na przeciwną (light → dark i odwrotnie).

Konsumpcja kontekstu

const ThemedButton = () => {
  const { theme, toggleTheme } = useContext(ThemeContext);

  return (
    <button
      onClick={toggleTheme}
      style={{
        backgroundColor: theme === "light" ? "#fff" : "#333",
        color: theme === "light" ? "#000" : "#fff",
      }}
    >
      Switch to {theme === "light" ? "dark" : "light"} mode
    </button>
  );
};

  • Komponent ThemedButton: Odczytuje wartość kontekstu i dynamicznie ustawia styl oraz działanie przycisku.

Zalety tego podejścia

  1. Brak undefined jako domyślnej wartości:

    • Zapewnia, że komponenty zawsze mają dostęp do działającej wartości kontekstu.

    • Eliminuje potrzebę sprawdzania obecności Providera (if (!context)).

  2. Bezpieczeństwo typu:

    • Domyślna wartość kontekstu jest w pełni typowana (ThemeContextType).

    • Funkcja toggleTheme zawsze istnieje, co eliminuje potencjalne błędy runtime.

  3. Elastyczność i ostrzeżenia:

    • Wywołanie toggleTheme poza Providerem generuje ostrzeżenie, co pomaga w debugowaniu.

Zarządzanie stanem użytkownika z wartością startową null

Nie każda aplikacja może zagwarantować obecność danych w kontekście. W przypadku, gdy dane użytkownika są opcjonalne (np. użytkownik może być niezalogowany), stosujemy jawne użycie null jako wartości domyślnej w kontekście.

Taka strategia wymusza obsługę brakującej wartości, co zwiększa elastyczność i bezpieczeństwo aplikacji, szczególnie w bardziej złożonych przypadkach. W tej sekcji pokażemy, jak prawidłowo zaimplementować i używać takiego kontekstu z zachowaniem wygody typowania.

Definicja typu i kontekstu

interface User {
  id: string;
  username: string;
}

interface UserContextType {
  user: User | null;
  login: (username: string, password: string) => Promise<void>;
  logout: () => void;
}

const UserContext = createContext<UserContextType | null>(null);

  • User: Definiuje strukturę danych użytkownika (id i nazwę).

  • UserContext: Domyślna wartość to null, co oznacza brak użytkownika w domyślnym stanie aplikacji. Zapewnia to większą elastyczność w scenariuszach, gdzie użytkownik może być niezalogowany.

Provider: zarządzanie stanem użytkownika

const UserProvider = ({children}: {children: ReactNode}) => {
  const [user, setUser] = useState<User | null>(null);

  // Uproszczone logowanie na potrzeby przykładu
  const login = async (username: string, password: string) => {
    // Symulacja zapytania do API
    const fetchedUser = await new Promise<User>((resolve) =>
      setTimeout(() => resolve({id: "1", username}), 1000)
    );
    setUser(fetchedUser);
  };

  const logout = () => {
    setUser(null);
  };

  return (
    <UserContext.Provider value={{user, login, logout}}>
      {children}
    </UserContext.Provider>
  );
};

const useUser = () => {
  const context = useContext(UserContext);
  if (!context) {
    throw new Error("useUser must be used within a UserProvider");
  }
  return context;
};

  • Provider: Dostarcza aktualne dane użytkownika do komponentów podrzędnych.

    • login: Symuluje żądanie do API, które zwraca dane użytkownika. Po sukcesie ustawia stan użytkownika.

    • logout: Ustawia user na null, symulując wylogowanie.

  • Hook useUser: Zapewnia bezpieczny dostęp do kontekstu użytkownika. Wyrzuca błąd, jeśli jest używany poza Providerem.

Konsumpcja kontekstu

import React, { useState } from "react";
import { useUser } from "./UserContext";

const UserInfo = () => {
  const {user, login, logout} = useUser();
  const [username, setUsername] = useState("");
  const [password, setPassword] = useState("");
  const [loading, setLoading] = useState(false);

  if (user) {
    return (
      <div>
        <h1>Welcome, {user.username}!</h1>
        <button onClick={logout}>Logout</button>
      </div>
    );
  }

  const handleLogin = async () => {
    setLoading(true);
    try {
      await login(username, password);
    } catch (err) {
      console.error("Login failed", err);
    } finally {
      setLoading(false);
    }
  };

  return (
    <div>
      <h1>Please log in</h1>
      <input
        type='text'
        placeholder='Username'
        value={username}
        onChange={(e) => setUsername(e.target.value)}
      />
      <input
        type='password'
        placeholder='Password'
        value={password}
        onChange={(e) => setPassword(e.target.value)}
      />
      <button onClick={handleLogin} disabled={loading}>
        {loading ? "Logging in..." : "Login"}
      </button>
    </div>
  );
};

  • Obsługa stanu user: Wyświetla dane zalogowanego użytkownika lub formularz logowania, jeżeli user jeszcze się nie zalogował.

  • handleLogin: Wywołuje metodę login z kontekstu i zarządza stanem ładowania.

W ten sposób obsłużyliśmy bardziej złożony przypadek:

  • customowy hook chroni nas przed użyciem kontekstu poza Providerem

  • warunkowy render w komponencie konsumującym pozwala obsłużyć wartość null w sposób bezpieczny, efektywny i dostosowany do potrzeb aplikacji.

Najlepsze praktyki

  1. Unikaj wartości domyślnych wrażliwych na błędy
    Jeśli kontekst może być używany bez dostawcy, zawsze sprawdzaj obecność wartości w runtime.

  2. Twórz dedykowane hooki
    Dedykowane hooki, takie jak useUser, pozwalają centralizować logikę weryfikacji kontekstu i upraszczają użycie w komponentach.

  3. Typowanie bez kompromisów
    Nigdy nie używaj any dla wartości kontekstu. Preferuj dobrze zdefiniowane interfejsy.

Podsumowanie

  • Zapewniona wartość domyślna to świetny wybór w prostych scenariuszach, gdzie zawsze jest dostępna sensowna wartość, np. tryb motywu.

  • Jawna wartość null oferuje elastyczność w bardziej złożonych aplikacjach, np. obsługa niezalogowanego użytkownika.

Obie strategie mają swoje miejsce w kontekście aplikacji React, a wybór zależy od wymagań konkretnego projektu.
