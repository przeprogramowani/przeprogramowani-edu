Hook useEffect pozwala na wykonywanie efektów ubocznych w komponentach React. Jest typowany za pomocą dwóch głównych parametrów: effect i deps, które są zdefiniowane w TypeScript jako EffectCallback i DependencyList. Nie ma potrzeby dodawania typów bezpośrednio do useEffect, ponieważ TypeScript automatycznie typuje wewnętrzne elementy tego hooka.

Jako że nie mamy czego się uczyć w zakresie samodzielnego typowania, przeanalizujemy sobie wewnętrzne typy useEffect, aby lepiej zrozumieć jak TypeScript radzi sobie z obsługą tego hooka.

Składnia useEffect w TypeScript

W pliku typów React (np. node_modules/@types/react/index.d.ts) useEffect ma następującą definicję:

function useEffect(effect: EffectCallback, deps?: DependencyList): void;

Typowanie useEffect składa się z dwóch kluczowych elementów:

  1. EffectCallback: Funkcja zawierająca logikę efektu

  2. DependencyList: Lista zależności

DependencyList: Typ dla listy zależności

DependencyList to tablica typu ReadonlyArray<unknown>, co oznacza, że jest to tablica tylko do odczytu, mogąca przechowywać dowolne typy danych (unknown). Dzięki temu każdy typ wartości może zostać przekazany do tablicy zależności useEffect.

type DependencyList = ReadonlyArray<unknown>;

Przykład: lista zależności może przyjmować wartości typu string, number, boolean, itp. Ponieważ jest to tablica tylko do odczytu, nie można jej modyfikować bezpośrednio.

EffectCallback: Typ dla funkcji efektu

Typ EffectCallback definiuje funkcję efektu, która:

  • Nie przyjmuje żadnych argumentów

  • Może zwracać void (co w JavaScript oznacza brak jawnego zwrócenia wartości) lub tzw. funkcję czyszczącą, zwaną destruktorem (Destructor)

type EffectCallback = () => (void | Destructor);

TypeScript zadba o wychwytywanie nieoczekiwanych wartości zwrotnych, co prowadzi do kodu zgodnego z API Reacta. Warto jednak, żebyś miał na uwadze pułapkę związaną ze zwracaniem wartości przez funkcje przekazywane do useEffect.

Przykład poniżej ilustruje częsty błąd: sytuację, gdy funkcja efektu zwraca wartość, co jest sprzeczne z oczekiwanym przez React zwrotem void lub funkcji czyszczącej.

function TimedEffect(props: { delayMs: number }) {
  const { delayMs } = props;

  useEffect(
    () =>
      setTimeout(() => {
        /* wykonaj operację */
      }, delayMs),
    [delayMs]
  );
  // Błąd! `setTimeout` zwraca ID timera jako liczbę.
  // Ponieważ ciało funkcji strzałkowej nie jest otoczone klamrami,
  // wynik `setTimeout` staje się niejawnie zwracany.
  return null;
}

W powyższym przykładzie, ponieważ funkcja w useEffect nie jest otoczona klamrami {}, wynik setTimeout (ID timera typu number) zostaje zwrócony. React oczekuje jednak, że efekt zwróci void lub funkcję czyszczącą, więc pojawia się niezgodność typów.

Aby rozwiązać problem, musimy upewnić się, że useEffect zwraca wyłącznie void lub że nie zwraca żadnej wartości. Możemy to osiągnąć, używając klamer {} w funkcji strzałkowej, co sprawia, że setTimeout nie jest niejawnie zwracany.

function TimedEffect(props: { delayMs: number }) {
  const { delayMs } = props;

  useEffect(() => {
    setTimeout(() => {
      /* wykonaj operację */
    }, delayMs);
  }, [delayMs]);
  // Poprawione: dodanie klamer sprawia, że setTimeout nie jest niejawnie zwracany
  return null;
}

Destructor: Funkcja czyszcząca

Typ Destructor definiuje funkcję czyszczącą (cleanup), która:

  • Jest wywoływana po zakończeniu działania efektu, najczęściej wykorzystuje się ją do usuwania event listenerów i/lub przesłanie logów

  • Zwraca void (lub obiekt { [UNDEFINED_VOID_ONLY]: never }, ale o tym więcej poniżej)

type Destructor = () => void | { [UNDEFINED_VOID_ONLY]: never };

Korzystając z useEffect z zasady zwracamy void, co w JavaScript oznacza niejawne zwrócenie wartości undefined przez funkcję pozbawioną słowa kluczowego return.

useEffect(() => {
    // Funkcja, która będzie wywoływana przy każdej zmianie wielkości okna przeglądarki
    const handleResize = () => {
      console.log(`Rozmiar okna: ${window.innerWidth} x ${window.innerHeight}`);
    };

    // Dodajemy nasłuchiwanie na zdarzenie 'resize' przy zamontowaniu komponentu
    window.addEventListener('resize', handleResize);

    // Funkcja czyszcząca, usuwająca nasłuchiwanie przy odmontowaniu komponentu
    return () => {
      window.removeEventListener('resize', handleResize); 
      // Brak jawnego zwrócenia wartości oznacza, że zwracamy `void`
    };
}, []); // Pusta lista zależności, więc efekt wykona się tylko raz

Ale o co chodzi z tym { [UNDEFINED_VOID_ONLY]: never }?

Zagadkowe UNDEFINED_VOID_ONLY

Typ { [UNDEFINED_VOID_ONLY]: never } jest stosowany zamiast undefined w funkcji czyszczącej (Destructor) hooka useEffect. 

declare const UNDEFINED_VOID_ONLY: unique symbol;

Pozwala zapewnić ścisłe sprawdzanie typu void, na co nie pozwoliłaby prosta unia void | undefined. Choć return undefined jest akceptowane przez TypeScript jako zgodne z void, zastosowanie UNDEFINED_VOID_ONLY:

  1. Wskazuje intencję: Konstrukcja ta przypomina, że jedynym akceptowalnym zwrotem jest void, co zwiększa przejrzystość kodu.

  2. Zapobiega przypadkowemu zwracaniu innych wartości: { [UNDEFINED_VOID_ONLY]: never } tworzy typ, którego nie da się osiągnąć lub przypisać przypadkowo, zmniejszając ryzyko błędów.

  3. Poprawia spójność typów: Ta konstrukcja wspiera narzędzia i autouzupełnianie, ułatwiając przewidywalne sprawdzanie void jako poprawnego zwrotu.

Dzięki temu UNDEFINED_VOID_ONLY zwiększa precyzję typów, nawet jeśli TypeScript technicznie dopuszcza return undefined.

Podsumowanie lekcji

W tej lekcji przeanalizowaliśmy szczegółowo typy wewnętrzne używane przez useEffect, w tym EffectCallback oraz DependencyList. Oto najważniejsze punkty, które omówiliśmy:

  • Składnia useEffect w TypeScript: Dowiedzieliśmy się, że TypeScript automatycznie typuje useEffect, dzięki czemu nie jest wymagane ręczne określanie typów dla efektu oraz jego zależności.

  • DependencyList: Poznaliśmy typ DependencyList, który jest zdefiniowany jako ReadonlyArray<unknown>. Oznacza to, że tablica zależności jest tylko do odczytu i akceptuje dowolne typy danych, co umożliwia dynamiczne określenie zależności efektu.

  • EffectCallback: Zbadaliśmy typ EffectCallback, który reprezentuje funkcję efektu nieprzyjmującą argumentów, mogącą zwracać void lub funkcję czyszczącą Destructor. Przeanalizowaliśmy częsty błąd z niejawnym zwracaniem callbacka, który powoduje konflikt typów.

  • Destructor i { [UNDEFINED_VOID_ONLY]: never }: Zrozumieliśmy rolę typu { [UNDEFINED_VOID_ONLY]: never } w wymuszaniu ścisłego typowania void w destruktorze, co zapobiega przypadkowemu zwracaniu innych wartości i poprawia czytelność kodu.

Dzięki tej wiedzy mamy lepsze zrozumienie, jak TypeScript obsługuje hook useEffect, co pozwala na efektywne zarządzanie efektami ubocznymi i pisanie bardziej bezpiecznego kodu.
