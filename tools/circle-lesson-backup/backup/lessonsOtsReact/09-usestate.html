useState

Hook useState to fundament zarządzania stanem w aplikacjach React. Gdy używamy go z TypeScriptem, odpowiednie typowanie pozwala osiągnąć większe bezpieczeństwo kodu i zapobiegać błędom.

W tej lekcji przeanalizujemy, jak typować useState zarówno w prostych przypadkach, jak i w bardziej złożonych scenariuszach, takich jak unie dyskryminowane, opcjonalny stan oraz przeciążenia funkcji.

Podstawowe typowanie w useState

W wielu przypadkach TypeScript automatycznie rozpoznaje typy na podstawie wartości początkowej. Działa to dobrze, gdy inicjalizujemy stan prostymi wartościami:

const [count, setCount] = useState(0); // count: number
const [name, setName] = useState("Alice"); // name: string

W powyższych przykładach TypeScript rozpoznaje, że count to number, a name to string, więc nie trzeba jawnie definiować ich typów.

Gdy typowanie jawne jest konieczne

Jednakże, jeśli stan może mieć więcej niż jeden typ, np. string | null, trzeba jawnie podać ten typ, by uniknąć błędów:

const [data, setData] = useState<string | null>(null);

W ten sposób TypeScript wymaga, aby każda nowa wartość stanu była zgodna z typem string | null.

Typowanie stanu opcjonalnego

W przypadku, gdy nie podajemy wartości początkowej do useState, TypeScript domyślnie uznaje stan za undefined:

const [data, setData] = useState<Data | undefined>();

Jeśli chcemy, by stan początkowy przyjął wartość null, należy określić typ jako Data | null:

const [data, setData] = useState<Data | null>(null);

To czy stosujemy undefined czy null jako stan początkowy jest zależne od wzorców przyjętych w danym projekcie.

undefined może być mniej czytelne niż null, ponieważ nie sygnalizuje jednoznacznie, że wartość jest celowo nieustawiona. W JS undefined jest również używane jako wartość domyślnej dla brakujących kluczy obiektów lub niezainicjowanych zmiennych. To wszystko bywa mylące. Stąd zalecamy stosowanie null, ale traktuj to proszę to jako luźną rekomendację a nie “święte prawo”.

Porównywanie obiektów i nadmiarowe właściwości

Podczas pracy z useState, jeśli stan komponentu jest obiektem, React umożliwia jego aktualizację na dwa sposoby:

  1. Poprzez przekazanie nowego obiektu do setState.

  2. Poprzez przekazanie funkcji, która zwraca nowy obiekt na podstawie poprzedniego stanu (tzw. functional update).

Przeanalizujmy jak te sposoby współgrają z TypeScriptem. Rozważmy komponent, który ma stan TagState zdefiniowany jako obiekt. Stan zawiera wybrane tagi i listę wszystkich tagów:

interface TagState {
  tagSelected: number | null;
  tags: { id: number; value: string }[];
}

const [state, setState] = useState<TagState>({ tagSelected: null, tags: [] });

Standardowa aktualizacja poprzez przekazanie nowego obiektu

Jeśli aktualizujemy stan, przekazując do setState nowy obiekt, TypeScript precyzyjnie sprawdza, czy obiekt spełnia wymagania typu TagState. Każda nadmiarowa właściwość w tym obiekcie spowoduje błąd:

// Aktualizacja stanu poprzez przekazanie nowego obiektu (działa poprawnie)
setState({ tagSelected: 2, tags: [{ id: 1, value: "JavaScript" }] });

// Próba dodania nadmiarowej właściwości spowoduje błąd TypeScript
setState({ tagSelected: 2, tags: [], extraProp: "Nieprawidłowa właściwość" });
// Error: Object literal may only specify known properties, and 'extraProp' does not exist in type 'SetStateAction<TagState>'

TypeScript poprawnie zgłasza błąd, gdy przekazujemy nadmiarową właściwość (extraProp). Dzięki temu zachowujemy bezpieczeństwo typów i unikamy nieprawidłowych aktualizacji.

Problematyczna sytuacja: aktualizacja za pomocą funkcji

Jeśli użyjemy funkcji, aby zaktualizować stan na podstawie jego poprzedniej wartości, TypeScript domyślnie nie wykryje nadmiarowych właściwości.

Wynika to z faktu, że w przypadku funkcji TypeScript stosuje tzw. porównanie kontrawariantne, co pozwala na większą elastyczność, ale nie chroni przed nadmiarowymi właściwościami.

// Aktualizacja stanu za pomocą funkcji - nadmiarowa właściwość nie wywoła błędu
setState(prevState => ({
  ...prevState,
  extraProp: "Nadmiarowa właściwość", // TypeScript nie zgłasza błędu
}));

W powyższym przykładzie TypeScript nie wykryje, że extraProp jest nadmiarową właściwością, mimo że nie należy do TagState. Może to prowadzić do problemów, jeśli extraProp zostanie wykorzystane w logice aplikacji lub renderowaniu, mimo że nie jest częścią typu TagState.

Rozwiązanie: Określenie typu zwracanej wartości w funkcji aktualizującej stan

Aby upewnić się, że TypeScript sprawdzi nadmiarowe właściwości przy aktualizacji za pomocą funkcji, należy jawnie określić typ zwracany przez tę funkcję:

setState((prevState): TagState => ({
  ...prevState,
  extraProp: "Nadmiarowa właściwość", // TypeScript zgłasza błąd
}));
// Error: Object literal may only specify known properties, and 'extraProp' does not exist in type 'TagState'.

Określając typ zwracany przez funkcję jako TagState, wymuszamy pełną zgodność obiektu wynikowego z typem TagState. TypeScript zgłosi błąd dla każdej nadmiarowej właściwości, co zapobiega błędom w kodzie i utrzymuje spójność typów.

Podsumowanie lekcji

W tej lekcji przeanalizowaliśmy, jak skutecznie typować stan w React z wykorzystaniem hooka useState w TypeScript. Oto kluczowe punkty, które omówiliśmy:

  • Podstawowe typowanie w useState: Dowiedzieliśmy się, że TypeScript automatycznie rozpoznaje typ stanu na podstawie wartości początkowej. Dzięki temu możemy używać useState z prostymi wartościami, takimi jak number czy string, bez konieczności jawnego określania typu.

  • Typowanie opcjonalnego stanu: Poznaliśmy sposoby definiowania opcjonalnego stanu, gdzie wartość początkowo może być undefined lub null. Omówiliśmy także, kiedy i dlaczego warto stosować null zamiast undefined dla lepszej czytelności.

  • Obsługa obiektów w stanie: Pracowaliśmy z bardziej złożonym stanem przechowującym obiekty. W tym kontekście przeanalizowaliśmy, jak React i TypeScript współpracują przy aktualizacjach obiektów, w tym jak uniknąć problemów związanych z nadmiarowymi właściwościami.

  • Rozwiązywanie problemów z nadmiarowymi właściwościami w funkcjonalnych aktualizacjach: Zrozumieliśmy, że TypeScript nie zawsze zgłasza nadmiarowe właściwości, gdy używamy funkcji do aktualizacji stanu. Nauczyliśmy się, jak rozwiązać ten problem, jawnie określając typ zwracany przez funkcję aktualizującą, co zapewnia zgodność typu i bezpieczeństwo kodu.

Dzięki zdobytej wiedzy potrafisz teraz efektywnie zarządzać stanem komponentów z useState, niezależnie od tego, czy pracujesz z typami prostymi, złożonymi, czy opcjonalnymi, co pozwala na tworzenie bardziej czytelnych i bezpiecznych aplikacji w TypeScripcie.
