Komponenty generyczne w React wraz z TypeScript umożliwiają elastyczne dostosowywanie się do różnych typów danych, co czyni je bardziej uniwersalnymi i wielokrotnie użytecznymi.

W tej lekcji przeanalizujemy praktyczne przykłady tworzenia komponentów generycznych, wykorzystania typów domyślnych, zagnieżdżania generyków oraz ich zastosowania w komponentach klasowych.

Cele lekcji

Po ukończeniu tej lekcji będziesz w stanie:

  • Tworzyć komponenty generyczne z wykorzystaniem TypeScript.

  • Wykorzystywać techniki rozwiązywania konfliktów składniowych w JSX/TSX.

  • Stosować generyki w zagnieżdżonych strukturach komponentów.

  • Używać generyki w komponentach klasowych.

Teoria

Czym są komponenty generyczne?

Komponenty generyczne pozwalają na dynamiczne definiowanie typów używanych w komponencie. Dzięki temu możliwe jest tworzenie uniwersalnych komponentów, które mogą obsługiwać różne struktury danych. TypeScript wspiera ten mechanizm poprzez parametry generyczne, które można definiować na poziomie interfejsu propsów.

Przykład podstawowy: Generyczny komponent Table

Rozpocznijmy od stworzenia komponentu Table, który renderuje listę obiektów dowolnego typu. Dodamy parametr generyczny TRow, który będzie reprezentował typ danych dla każdego wiersza.

interface TableProps<TRow> {
  rows: TRow[];
  renderRow: (row: TRow) => React.ReactNode;
}

export const Table = <TRow,>({ rows, renderRow }: TableProps<TRow>) => (
  <table>
    <tbody>
      {rows.map((row, index) => (
        <tr key={index}>{renderRow(row)}</tr>
      ))}
    </tbody>
  </table>
);

W przykładzie powyżej TableProps<TRow> definiuje strukturę propsów komponentu Table, który przyjmuje dowolny typ TRow. Umożliwia to przekazanie różnych typów danych do komponentu:

type Product = {
  id: number;
  name: string;
  price: number;
};

const App = () => {
  const products: Product[] = [
    { id: 1, name: 'Laptop', price: 1200 },
    { id: 2, name: 'Phone', price: 700 },
  ];

  return (
    <Table
      rows={products}
      renderRow={(product) => (
        <>
          <td>{product.id}</td>
          <td>{product.name}</td>
          <td>{product.price}</td>
        </>
      )}
    />
  );
};

Rozwiązywanie konfliktów składniowych JSX dla generyków

Przy pracy z generykami w JSX/TSX, TypeScript może mylnie interpretować nawiasy kątowe jako elementy JSX, co prowadzi do błędów parsowania. Aby uniknąć tego problemu, można zastosować kilka technik:

  1. Dodanie przecinka po parametrze typu (<T,>).

  2. Użycie extends unknown dla typu generycznego (<T extends unknown>).

// Użycie przecinka dla rozróżnienia od elementu JSX
const Table = <T,>({ rows }: TableProps<T>) => { /* ... */ }

// Alternatywnie
const Table = <T extends unknown>({ rows }: TableProps<T>) => { /* ... */ }

Rekomendujemy stosowanie <T,>, ponieważ ta składnia jest krótsza i bardziej jednoznaczna. Użycie extends unknown może wprowadzać niepotrzebne komplikacje, zwłaszcza dla początkujących.

Czy trzeba przekazywać typy generyczne, czy wystarczy ich użycie w propsach?

W większości przypadków TypeScript potrafi samodzielnie wywnioskować typ generyczny na podstawie przekazanych propsów, co oznacza, że jawne przekazanie typu, np. List<Product>, jest zbędne. TypeScript automatycznie rozpoznaje typy danych zdefiniowanych w propsach i przypisuje odpowiedni typ do parametru generycznego.

interface ListProps<T> {
  items: T[];
  renderItem: (item: T) => React.ReactNode;
}

const List = <T,>({ items, renderItem }: ListProps<T>) => (
  <ul>
    {items.map((item, index) => (
      <li key={index}>{renderItem(item)}</li>
    ))}
  </ul>
);

const App = () => {
  const products = [
    { id: 1, name: 'Laptop' },
    { id: 2, name: 'Phone' },
  ];

  return (
    // TypeScript automatycznie rozpoznaje typ `T` jako { id: number; name: string }
    <List
      items={products}
      renderItem={(product) => <span>{product.name}</span>}
    />
  );
};

Kiedy warto jawnie podać typ generyczny?

Chociaż wnioskowanie typu zazwyczaj działa dobrze, istnieją sytuacje, gdy jawne określenie typu generycznego jest korzystne:

  • Złożone struktury danych: Gdy dane są bardzo złożone, na przykład zawierają opcjonalne pola lub zagnieżdżone typy, TypeScript może potrzebować dodatkowych wskazówek, aby jednoznacznie rozpoznać typ.

  • Nietypowe konwersje lub użycie typów: Jeśli typ propsów zawiera wartość, która mogłaby pasować do różnych typów, TypeScript może zgłosić błąd. W takim przypadku jawne określenie typu rozwiązuje problem.

Jednak takie sytuacje występują rzadko, dlatego w większości przypadków przekazywanie typu jest niekonieczne.

Przekazywanie typów generycznych do komponentów podrzędnych

Komponenty kontenerowe mogą wykorzystywać generyki i przekazywać je dalej do komponentów podrzędnych, co umożliwia pełną kompatybilność typów w całym drzewie komponentów.

interface ListProps<T> {
  items: T[];
  renderItem: (item: T) => React.ReactNode;
}

const List = <T,>({ items, renderItem }: ListProps<T>) => (
  <ul>
    {items.map((item, index) => (
      <li key={index}>{renderItem(item)}</li>
    ))}
  </ul>
);

interface ContainerProps<T> {
  items: T[];
}

const Container = <T,>({ items }: ContainerProps<T>) => (
  <List items={items} renderItem={(item: T) => <span>{JSON.stringify(item)}</span>} />
);

W tym przykładzie Container przyjmuje dowolny typ T i przekazuje go do komponentu List, aby zachować zgodność:

interface Item {
  id: number;
  description: string;
}

const App = () => {
  const items: Item[] = [
    { id: 1, description: 'Item 1' },
    { id: 2, description: 'Item 2' },
  ];

  return (
    <Container items={items} />
  );
};

Zagnieżdżone typy generyczne

Komponenty generyczne stają się szczególnie przydatne, gdy pracujemy ze złożonymi, zagnieżdżonymi strukturami danych, np. hierarchicznymi listami lub drzewami. Zagnieżdżając typy, możemy dokładniej kontrolować, jak typy propagują się w strukturach danych.

interface TreeNode<T> {
  value: T;
  children?: TreeNode<T>[];
}

interface TreeProps<T> {
  nodes: TreeNode<T>[];
  renderNode: (node: TreeNode<T>) => React.ReactNode;
}

export const Tree = <T,>({ nodes, renderNode }: TreeProps<T>) => (
  <ul>
    {nodes.map((node, index) => (
      <li key={index}>
        {renderNode(node)}
        {node.children && <Tree nodes={node.children} renderNode={renderNode} />}
      </li>
    ))}
  </ul>
);

Powyższy komponent Tree jest w stanie wyświetlać dowolne drzewo zagnieżdżonych danych, zachowując pełną kontrolę nad typowaniem na każdym poziomie hierarchii.

type Category = {
  name: string;
};

const App = () => {
  const categoryTree: TreeNode<Category>[] = [
    {
      value: { name: 'Electronics' },
      children: [
        { value: { name: 'Laptops' } },
        { value: { name: 'Phones' } },
      ],
    },
    {
      value: { name: 'Furniture' },
      children: [
        { value: { name: 'Tables' } },
        { value: { name: 'Chairs' } },
      ],
    },
  ];

  return (
    <Tree
      nodes={categoryTree}
      renderNode={(node) => <span>{node.value.name}</span>}
    />
  );
};

Komponenty klasowe z typami generycznymi

W przypadku komponentów klasowych, generyki działają podobnie jak w funkcjonalnych komponentach, lecz wymagają deklaracji typu generycznego zaraz po nazwie klasy i przed dziedziczeniem z extends​.

interface TableProps<TRow> {
  rows: TRow[];
  renderRow: (row: TRow) => React.ReactNode;
}

class Table<TRow,> extends React.Component<TableProps<TRow>> {
  render() {
    const { rows, renderRow } = this.props;
    return (
      <table>
        <tbody>
          {rows.map((row, index) => (
            <tr key={index}>{renderRow(row)}</tr>
          ))}
        </tbody>
      </table>
    );
  }
}

Komponent Table z powyższego przykładu jest klasą generyczną, w której możemy dokładnie typować dane przy użyciu TRow. Działa analogicznie do pierwszego przykładu komponentu funkcyjnego.

Alternatywa dla generyków: unie dyskryminowane

Unie dyskryminowane to technika pozwalająca na obsługę kilku wariantów typu bez stosowania generyków, co zwiększa czytelność kodu, gdy typ jest ograniczony do kilku konkretnych przypadków.

Unie dyskryminowane składają się z:

  • Unii typów, która zawiera kilka możliwych struktur, w kontekście Reacta będą to zazwyczaj propsy.

  • Właściwości dyskryminującej (np. variant, type), która pozwala odróżnić jeden wariant od drugiego.

Zamiast używać generyków, które mogą prowadzić do bardziej skomplikowanej struktury, unie dyskryminowane są użyteczne, gdy komponent ma kilka ustalonych stanów.

Przykład: Modal z unią dyskryminowaną

Rozważmy komponent Modal, który może występować w dwóch wariantach:

  1. Z przyciskiem (with-button), w którym oczekiwany jest buttonLabel i onButtonClick.

  2. Bez przycisku (without-button), gdzie te właściwości są zbędne.

interface ModalWithButton {
  variant: "with-button";
  isOpen: boolean;
  buttonLabel: string;
  onButtonClick: () => void;
}

interface ModalWithoutButton {
  variant: "without-button";
  isOpen: boolean;
}

type ModalProps = ModalWithButton | ModalWithoutButton;

const Modal = (props: ModalProps) => {
  return (
    <div style={{ display: props.isOpen ? "block" : "none" }}>
      {props.variant === "with-button" && (
        <button onClick={props.onButtonClick}>{props.buttonLabel}</button>
      )}
      {props.variant === "without-button" && <p>Modal bez przycisku</p>}
    </div>
  );
};

Przykład użycia:

const App = () => {
  return (
    <div>
      <Modal
        variant="with-button"
        isOpen={true}
        buttonLabel="Kliknij mnie"
        onButtonClick={() => alert("Kliknięto przycisk!")}
      />
      <Modal variant="without-button" isOpen={true} />
    </div>
  );
};

Unie dyskryminowane oferują kilka korzyści:

  • Prosta obsługa różnych wariantów: Umożliwiają łatwe definiowanie komponentów lub funkcji z różnymi zestawami wymaganych i opcjonalnych pól w zależności od wariantu.

  • Bezpieczeństwo typowania: Dzięki unii TypeScript zapewnia, że tylko właściwe atrybuty są dostępne dla danego wariantu, co minimalizuje błędy.

Zalecenia: generyki vs unie dyskryminowane

Stosuj unie dyskryminowane, gdy masz kilka ustalonych wariantów komponentu, a każdy z nich wymaga innego zestawu propsów. Są szczególnie użyteczne w komponentach z różnymi trybami wyświetlania lub akcjami.

Generyki natomiast są bardziej odpowiednie, gdy komponent powinien być elastyczny w obsłudze różnych typów danych, które mogą się zmieniać, na przykład w komponentach renderujących listy lub pracujących z dynamicznymi strukturami danych.

Podsumowując:

  • Unie dyskryminowane są idealne do tworzenia komponentów z ograniczoną liczbą, ale różnorodnymi wariantami.

  • Generyki zapewniają elastyczność dla komponentów, które nie muszą obsługiwać z góry określonych wariantów, ale mogą przyjmować różne typy danych.

Podsumowanie

W tej lekcji nauczyliśmy się tworzyć komponenty generyczne w React z TypeScriptem, co umożliwia tworzenie bardziej elastycznych i wielokrotnie użytecznych komponentów obsługujących różne typy danych. Omówiliśmy następujące kluczowe punkty:

  • Tworzenie generycznych komponentów: Dowiedzieliśmy się, jak używać generyków do dynamicznego definiowania typów w komponentach, co pozwala na lepsze dopasowanie komponentu do danych, które przyjmuje.

  • Rozwiązywanie konfliktów składniowych JSX: Omówiliśmy sposoby na uniknięcie problemów z parsowaniem generyków w JSX, szczególnie przy użyciu przecinka po parametrze typu.

  • Automatyczne wnioskowanie typów: Zrozumieliśmy, że TypeScript zazwyczaj potrafi automatycznie wywnioskować typ generyczny na podstawie przekazanych propsów, co często eliminuje konieczność ręcznego określania typów w JSX.

  • Zagnieżdżone struktury generyczne: Przykłady takie jak Tree pokazały, jak stosować generyki w złożonych, hierarchicznych strukturach danych, co ułatwia pracę z danymi o różnych poziomach zagnieżdżenia.

  • Generyki w komponentach klasowych: Zobaczyliśmy, jak zastosować generyki w komponentach klasowych, zachowując pełną kontrolę nad typowaniem danych.

  • Alternatywa dla generyków – unie dyskryminowane: Poznaliśmy unie dyskryminowane jako wygodne rozwiązanie do obsługi komponentów z ograniczoną liczbą, ale różnorodnymi wariantów, co zwiększa czytelność kodu w sytuacjach, gdy liczba typów jest ograniczona i można precyzyjnie określić strukturę danych.

Dzięki tym umiejętnościom możesz tworzyć komponenty, które są zarówno bardziej elastyczne, jak i bezpieczne pod względem typów, co znacząco zwiększa ich użyteczność i niezawodność w różnych scenariuszach.
