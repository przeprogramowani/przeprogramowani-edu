Plik .cursorrules - dostosowanie AI do Twoich standardów

Cursor oferuje możliwość konfiguracji zachowania AI poprzez plik .cursorrules. To bardzo przydatna funkcja, która pozwala dostosować generowany kod do preferencji naszego zespołu. Przykładowo możemy poprzez .cursorrules zalecić AI stosowanie praktyk, które rekomendowaliśmy w Opanuj TypeScript:

// Przykładowa zawartość pliku .cursorrules
You are an experienced TypeScript and React developer. When generating code:

- Always use direct prop typing instead of React.FC
- Use function components
- Use proper TypeScript types instead of 'any'
- Follow project standards:
  - Use interface for public APIs
  - Use type for internal types
  - Use proper naming conventions (PascalCase for components, camelCase for functions)

Dzięki takiej konfiguracji, generowany kod będzie automatycznie zgodny z naszymi standardami, na przykład:

// Preferowany sposób typowania komponentów
interface ButtonProps {
  label: string;
  onClick: () => void;
}

function Button({ label, onClick }: ButtonProps) {
  return <button onClick={onClick}>{label}</button>;
}

// Zamiast:
const Button: React.FC<ButtonProps> = ({ label, onClick }) => {
  return <button onClick={onClick}>{label}</button>;
};

Dobre praktyki współpracy z AI

Niezależnie od tego, z jakim edytorem pracujesz, praktyki współpracy z obecną generacją modeli językowych pozostają takie same. Poniżej przedstawiamy te aspekty, o których wielu programistów nie pamięta lub wobec których ma błędne oczekiwania:

1. Kontekst jest najważniejszy

Wielu programistów wyobraża sobie, że AI / LLMy to algorytmy do przewidywania myśli, intencji i oczekiwań. Nic bardziej mylnego - skuteczna współpraca z AI bardzo mocno opiera się na przekazywaniu właściwego kontekstu i uzupełnianiu domysłów.

Współczesne edytory takie jak Cursor pomagają zarządzać tym problemem poprzez automatyczne indeksowanie kodu, ale nawet one mają braki w przewidywaniu twoich intencji.

Aby uzyskać najlepsze wyniki, jasno określ swoje potrzeby. Wskazuj, jaki język programowania, framework czy styl kodu jest wymagany. Im bardziej szczegółowy kontekst podasz, tym bardziej trafne odpowiedzi otrzymasz. Aby nie marnować czasu, korzystać z plików typu .cursorrules gdzie jasno wyrazisz swoje oczekiwania (np. na poziomie całego projektu).

2. Iteracje i cierpliwość

Zdarza się, że nawet precyzyjnie określony kontekst nie przekłada się na uzyskiwanie właściwych odpowiedzi z modelu językowego. Szczególnie widać to w bardziej zaawansowanych zadaniach, które wymagają przedstawienia problemu z wielu stron i poprawiania ruchów AI.

Z tego płynie wniosek dotyczący odpowiedniego nastawienia względem modeli - podobnie jak w przypadku kolaboracji z człowiekiem, przygotuj się na wymianę myśli i iteracyjne rozwiązywanie problemów. Korzystaj nie tylko z generowania kodu, ale dostępnego Czatu, gdzie przeprowadzisz burzę myśli na temat danego zadania. Odwołuj się do zewnętrznych źródeł danych i pytaj o alternatywne pomysły na dalsze kroki.

Pomoże to nie tylko w uzyskaniu lepszych wyników, ale również poszerzy twoje zrozumienie problemu i pozwoli uniknąć myślenia tunelowego.

3. Jeden strzał czy wiele kroków?

Do niedawna, uniwersalną rekomendacją na takiej liście było rozbijanie dużych problemów na mniejsze składowe - modele językowe łatwo gubiły kontekst i halucynowały. Dzisiaj trzeba podchodzić do tego bardziej uważnie, bo na rynku mamy dwa rodzaje systemów opartych o LLMy:

  • klasyczne modele językowe (np. GPT-4o, Claude 3.5 Sonnet, LLaMa 3)

  • systemy wzbogacone o tzw. Chain of Thought (np. OpenAI o1, DeepSeek R1, Qwen with Questions)

Pracując z klasycznymi modelami językowymi, porada o “jedzeniu słonia po kawałku” nadal obowiązuje.

Zamiast złożonego problemu, który może powodować błędne rozumowanie, warto rozpocząć od określenia kontekstu i zapytania modelu o to, jak krok po kroku przejść dane zadanie. Następnie, iterując i poszerzając kontekst, można rozwiązać pierwszy, drugi a później trzeci problem. Tak naprawdę nie różni się od tego, jak na codzień rozwiązujemy wyzwania programistyczne.

W przypadku systemów takich jak OpenAI o1, społeczność coraz częściej rekomenduje tworzenie precyzyjnych i bardziej złożonych promptów, które model przeanalizuje i uporządkuje po swojej stronie. Wynika to z rozszerzonego trybu wnioskowania opartego o praktykę Chain of Thought. Model taki jak o1, po otrzymaniu naszego polecenia, najpierw sam buduje przepis na jego rozwiązanie, a następnie krok po kroku przechodzi przez wszystkie zaproponowane etapy. Stąd nacisk na pierwsze polecenie wysokiej jakości, nawet przy złożonych warunkach i kryteriach.

Warto dostosować tryb wydawania poleceń do sposobu działania i rozumowania modelu.

4. Komendy, błędy, stack trace - jest na to sposób

Dyskusja o kontekście, poleceniach, realizowaniu zadań krok po kroku i precyzyjnym języku może czasami sugerować, że LLMy są dobre wyłącznie w tym, co sami rozumiemy. W praktyce okazuje się, że pokazują one swój potencjał nawet tam, gdzie nie wiemy jak ruszyć dalej.

Najczęstszym przykładem takich scenariuszy jest analiza błędów. Kiedy nie wiem co jest powodem problemu, wklejenie surowego stack trace’u może wystarczyć, aby model językowy wskazał nam problematyczne obszary naszego kodu. Podobnie w przypadku większych zmian wygenerowanych przez Composera (agenta) - jeśli pierwsza modyfikacja nie działa, nie musimy sprzątać po AI, a po prostu wskazać na błąd np. z Dev Toolsów - zwykle całość natychmiast ruszy do przodu.

Należy pamiętać, że nie mówimy tutaj o magicznym zgadywaniu “co dalej”, ale wnioskowaniu na podstawie kontekstu i danych treningowych. W końcu na forach, mediach społecznościowych i stronach typu Stack Overflow pojawiło się mnóstwo błędów, które rozwiązywali programiści. Jego skuteczność w tym obszarze będzie skorelowana z popularnością technologii, której dotyczyły takie dyskusje naprawcze.

Teraz, na podstawie tej wiedzy, model językowy może być dla ciebie skutecznym blocker-killerem.

5. Techniki Prompt Engineeringu

Od czasu boomu na ChataGPT, opublikowanych zostało wiele ustandaryzowanych technik skutecznej komunikacji z modelami.

Techniki takie jak promptowanie z rolą, wspomniany wcześniej Chain-of-Thought, ReAct czy Panel of Experts mogą się okazać pomocne w przeniesieniu konwersacji na nowy poziom.

Badania pokazują, że modele językowe całkiem nieźle radzą sobie z oceną swoich własnych odpowiedzi, co może być jednym ze sposobów na uzyskanie lepszych wyników. Należy jednak o tej możliwości pamiętać i wracać do rekomendowanych przez nas iteracji, zamiast zamykania konwersacji po uzyskaniu pierwszej niepoprawnej odpowiedzi.

Więcej o praktykach PE przeczytasz tutaj, a także w naszym szkoleniu Cursora.

AI w projektach opartych o TypeScript

W drugiej części lekcji zaprezentujemy ci scenariusze dotyczące języka TypeScript, w których modele językowe są najbardziej użyteczne.

Dlaczego TypeScript jest świetny do pracy z AI?

TypeScript, ze swoim rozbudowanym systemem typów, okazuje się być idealnym językiem do współpracy z modelami AI. Jego ekspresywność i precyzja w definiowaniu struktur danych sprawia, że modele językowe znacznie lepiej "rozumieją" intencje programisty i generują dokładniejszy kod. Oto dlaczego:

  1. Precyzyjne kontrakty - TypeScript pozwala jasno zdefiniować oczekiwania wobec kodu poprzez interfejsy i typy. To sprawia, że AI ma więcej kontekstu i generuje bardziej przewidywalny kod.

  2. Mniej miejsca na błędy - Statyczne typowanie eliminuje całą klasę błędów, które mogłyby powstać w JavaScript. AI może skupić się na logice biznesowej zamiast na obsłudze edge case'ów związanych z dynamicznym typowaniem.

  3. Łatwiejsza refaktoryzacja - Dzięki systemowi typów, AI może bezpieczniej proponować zmiany w kodzie, mając pewność, że nie złamie istniejących kontraktów.

Te cechy sprawiają, że modele AI popełniają znacznie mniej błędów pracując z TypeScript niż z czystym JavaScript. Szczególnie widoczne jest to w bardziej złożonych scenariuszach, jak praca z zaawansowanymi wzorcami React czy generowanie typów dla zewnętrznych bibliotek.

Wyjaśnianie złożonych typów

Po ukończeniu naszego szkolenia będziesz potrafił budować typy złożone wykorzystujące wiele nowoczesnych elementów TypeScriptu. Może się jednak zdarzyć, że odkryjesz obcy typ, wprowadzony do projektu przez inną osobę. Co gorsza, może to być typ bez dokumentacji w TS Doc.

W takiej sytuacji warto poprosić o pomoc AI - chociażby w udokumentowaniu danej konstrukcji i dodaniu przykładów, które pomogą tobie (oraz innym programistom) zrozumieć ten nowy typ (poniżej komentarz wygenerowany przez Claude 3.5 Sonnet):

/**
 * Makes all properties in T optional recursively
 * For nested objects, their properties also become optional
 * For non-object types, they just become optional
 *
 * Example:
 * interface Person {
 *   name: string;
 *   address: {
 *     street: string;
 *     city: string;
 *   }
 * }
 *
 * DeepPartial<Person> allows:
 * {
 *   name?: string;
 *   address?: {
 *     street?: string;
 *     city?: string;
 *   }
 * }
 */
type DeepPartial<T> = {
  [P in keyof T]?: T[P] extends object ? DeepPartial<T[P]> : T[P];
};

Tooling
