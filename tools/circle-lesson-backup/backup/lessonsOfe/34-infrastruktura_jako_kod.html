Infrastructure as Code (IaC)

Do tej pory wszystkie chmurowe modyfikacje wykonywaliśmy bezpośrednio na AWS - czy to manualnie, czy z pomocą AWS CLI. Taki sposób pracy sprawdza się w mniejszych projektach, ale założę się, że już po kilku dniach trudno byłoby ci odpowiedzieć jak właściwie skonfigurowany jest Cloudfront lub bucket na S3.

Wyobraź sobie, że do naszej konfiguracji dodajemy teraz nie jeden, ale dwadzieścia bucketów, pięć dystrybucji CloudFronta, cztery domeny, dedykowane maszyny wirtualne, system kolejek i powiadomień.

Klikanie po konsoli byłoby nie tylko czasochłonne, ale również podatne na błędy i różnice w konfiguracji pomiędzy obiektami tej samej klasy, a także wrażliwe na ręczne modyfikacje. Infrastruktura zmieniałaby się losowo, bez kontroli nad wersjami i jej stanem.

Nietrudno się domyślić, że chcemy ci zaproponować nieco inne podejście do zarządzania bardziej złożonymi projektami chmurowymi. Będzie ono zgodne z filozofią Infrastructure as Code, gdzie naszą infrastrukturę opisujemy przez deklaratywny język zasobów i trzymamy go w repozytorium, zaraz obok właściwego kodu aplikacji.

Do budowania infrastruktury w tym modelu wykorzystamy Terraforma, który będzie stanowił warstwę middleware pomiędzy opisem usług (w repozytorium) a faktyczną infrastrukturą na AWS:

W takim modelu rozwoju infrastruktury uzyskamy wszystkie korzyści systemu kontroli wersji, a opis tworzonych (i usuwanych) usług będzie przechowywany w formie czytelnego kodu. Wykorzystując tzw. “providery”, Terraform będzie w stanie porównywać zadeklarowany stan z plików oraz stan zdalny, wdrażając niezbędne korekty poprzez API danej chmury (np. AWS).

👉 Komunikacja przez API oznacza, że Terraform korzysta z danych dostępowych utworzonego przez ciebie użytkownika IAM. Jego uprawnienia wpływają na to, co Terraform może w twoim imieniu robić na poziomie samej chmury. Ich zakres powinien dotyczyć wyłącznie tego, czym faktycznie zarządza TF.

Pamiętaj, że, Terraform - co zobaczymy w kolejnych filmach - nie zwalnia nas z potrzeby znajomości konkretnej chmury, którą wykorzystujemy w projekcie. Jest to narzędzie do zarządzania zasobami, ale przeznaczenie tych zasobów musimy wciąż odkrywać na własną rękę.

👉 Warto wiedzieć, że kilka miesięcy temu, na skutek zmian licencyjnych, Terraform stał się przedmiotem gorących dyskusji w środowisku. HashiCorp, firma stojąca z tym rozwiązaniem, przeprowadziła zmianę licencji przechodząc na tzw. “Business Source License”. Co to oznacza? Terraform pozostaje rozwiązaniem z otwartym kodem źródłowym, przy jednoczesnym zakazie konkurencji i budowania podobnych modeli biznesowych przez użytkowników. Szczegóły znajdziesz w FAQ.

W odpowiedzi na zmiany licencji, na rynku pojawił się otwarty fork o nazwie OpenTofu, który realizuje te same zadania co Terraform i deklaruje pełną zgodność z jego API.

Biorąc pod uwagę ilość dostępnej w internecie dokumentacji i dyskusji dot. Terraforma, a także kontekstu szkolenia, gdzie wspólnie budujemy rozwiązania w kontekście edukacyjnym, a nie komercyjnym, postanowiliśmy pozostać przy Terraformie. Po ukończeniu szkolenia możesz zapoznać się z OpenTofu.

Pierwsze kroki z Terraformem

Rozpocznijmy pracę z Terraformem od jego instalacji - https://developer.hashicorp.com/terraform/install

W zależności od systemu operacyjnego, możesz wykorzystać klasyczne instalatory, pliki binarne, albo managery paczek takie jak homebrew dla macOS czy apt dla Linuxa.

Poprawna instalacja powinna zakończyć się możliwością sprawdzenia wersji w konsoli:

$ terraform -v
Terraform v1.7.5
on darwin_arm64

Jeśli masz już zainstalowanego Terraforma, upewnij się, że jego wersja to co najmniej 1.5.0 - w dalszej części lekcji będziemy korzystać z rozszerzenia, które zostało wprowadzona właśnie w tej wersji. Najlepiej będzie, jeśli wspólnie będziemy używać wersji 1.7.5.

Zarządzanie zasobami

Po instalacji Terraforma, utwórz folder “terraform” na głównym poziomie projektu, a w nim umieść plik “main.tf” z następującą zawartością:

terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.40.0"
    }
  }

  required_version = ">= 1.5.0"
}

provider "aws" {
  region = "eu-central-1"
}

Ta konfiguracja określa trzy parametry:

  • po pierwsze, wskazujemy, że będziemy pracować z tzw. “providerem aws”, czyli modułem służącym do zarządzania tą konkretną chmurą - będzie on miał wersję 5.40.0 lub wyższą

  • po drugie, wymagamy Terraforma w wersji wyższej lub równej 1.5.0

  • po trzecie, nasza komunikacja z AWS powinna się odbywać w ramach regionu “eu-central-1”

Wykonaj teraz polecenie “init” wewnątrz nowego folderu i sprawdź, czy provider AWS zainicjalizował się w folderze “terraform/.terraform”:

cd terraform
terraform init

Folder “.terraform” możesz od razu dodać do .gitignore, ponieważ będzie on odtwarzany niezależnie, przez każdego użytkownika tego repozytorium. W razie problemów czekamy na sygnał - #Praktyka.

Backend zdalny

Utrzymywanie stanu Terraforma na poziomie repozytorium jest wystarczające dla małych projektów i programistów rozwijających infrastrukturę w trybie solo. Pomaga nam zrozumieć jak działa Terraform i czym jest stan tego narzędzia, ale nie wspiera środowisk zdalnych i wielu aktorów modyfikujących konfigurację z wielu miejsc.

Wystarczy tylko, że infrastrukturę chcemy modyfikować jednocześnie my i runner Github Actions, no i mamy problem:

  1. Programista modyfikuje zasób w repozytorium dodając nowy “resource” - zapomina o “terraform apply”

  2. Terraform poprzez Github Actions zauważa zmianę, wdraża ją na AWS, zmienia stan lokalny

  3. Zmiana zostaje na poziomie scenariusza GHA, nie wraca do repozytorium, nie pojawia się nowy commit

  4. Programista ma problem - infrastruktura się zmieniła, ale jej opis w kodzie o tym nie mówi

Aby rozwiązać ten problem, w projektach Terraform bardzo często wprowadzamy tzw. remote state, albo zdalny backend, który utrzymywany jest na współdzielonej infrastrukturze:

Aby wprowadzić zdalny backend, zmodyfikujmy plik main.tf dodając do niego sekcję “backend” - mówi ona, że stan powinien być teraz przechowywany na S3 oraz z wykorzystaniem DynamoDB:

terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.40.0"
    }
  }

  required_version = ">= 1.5.0"

  backend "s3" {
    bucket         = "ofe-terraform-state-bucket"
    key            = "state/terraform.tfstate"
    region         = "eu-central-1"
    encrypt        = true
    dynamodb_table = "terraform-state-lock"
  }
}

Potrzebujemy dwóch elementów - miejsca na przechowywanie pliku terraform.state (np. AWS S3) oraz bazy, w której przechowywany będzie tzw. “lock”, czyli flaga zabraniająca modyfikacji stanu z wielu miejsc w tym samym momencie (możemy ją utrzymywać np. w AWS DynamoDB).

Oba te elementy możemy utworzyć ręcznie - w S3 dodajmy nowy bucket z dodatkowo włączoną opcją Versioning:

Dzięki takiej konfiguracji zarówno ty, jak i inni programiści w zespole oraz Github Actions możecie pracować na tym samym stanie infrastruktury, a efekty polecenia “terraform apply” będą synchronizowane dla wszystkich.

Mając zdefiniowany stan zdalny, możesz albo zaimportować do niego istniejące zasoby, albo pracować z nowym projektem, dodając kolejno niezbędne elementy twojej preferowanej chmury.

👉 Para S3 + DynamoDB może być również utworzona jako resource Terraforma. W takim wypadku najpierw tworzysz zasoby zapisując je do stanu lokalnego, następnie usuwasz stan lokalny, a na końcu dodajesz sekcję backend do pliku main.tf i korzystasz z nich jako źródeł stanu.

👨‍💻 Ćwiczenia praktyczne

W zależności od tego, czy wykonywałeś kroki z tej z lekcji na bieżąco, czy dopiero zaczynasz pracę, możesz wybrać jedno z poniższych zadań:

  • Nowe środowisko: Przeprowadź konfigurację S3 + CloudFront z wykorzystaniem Terraforma, aby umożliwić deployment na nowe środowisko przez GitHub Actions:

    • Skonfiguruj zdalny backend dla Terraforma

    • Dodaj nowy bucket S3 z opcjami dotyczącymi hostowania strony

      • Wykorzystaj zasoby związane z S3:

        • aws_s3_bucket

        • aws_s3_bucket_ownership_controls

        • aws_s3_bucket_public_access_block

        • aws_s3_bucket_acl

        • aws_s3_bucket_policy

        • aws_s3_bucket_website_configuration

      • Do dodania polityki przez zasób “aws_s3_bucket_policy” możesz potrzebować nowych uprawnień. Znajdziesz je w folderze examples/module3/lesson3/inline-policy.json

    • Dodaj powiązany z nim CDN - Cloudfront - wykorzystaj zasób aws_cloudfront_distribution.

    • Spraw, aby GitHub actions był w stanie wdrażać twój projekt na nowe środowisko

  • CDN dla obrazków: Zbuduj nowy fragment infrastruktury, który pozwoli bardziej efektywnie serwować obrazki dla użytkowników aplikacji full-stackowej.

    • W tym scenariuszu zakładamy, że aplikacja frontendowa ma już zapewniony hosting, ale na produkcji obrazki powinny być serwowane poprzez CloudFronta.

    • Aby to osiągnąć, potrzebujesz zbudować:

      • Nowy fragment infrastruktury - S3 + Cloudfront - na serwowanie obrazków

      • Nowy scenariusz GHA, który zbuduje aplikację i wrzuci na S3 określone pliki (np. jpg + png)
