Wprowadzenie

GitHub Actions - jak czytamy na stronie głównej - to rozwiązanie do automatyzacji wszystkiego, co jest związane z oprogramowaniem. Dzięki tej usłudze możemy budować, testować i wdrażać nasze aplikacje w sposób ciągły, systematycznie zbliżając się do pełnego modelu CI/CD (tej tematyce przyjrzymy się bliżej w kolejnej lekcji).

Github Actions (GHA) wprowadza ustandaryzowany model opisywania i uruchamiania procesów, które mogą realizować najróżniejsze scenariusze. Dzięki ścisłej integracji z istniejącymi repozytoriami GitHuba, wdrożenie GHA jest stosunkowo proste i nie wymaga od nas konfiguracji dedykowanych serwerów.

A jak wygląda cennik? Dla publicznych repozytoriów i na standardowych instancjach serwerów Githuba korzystamy z tej usługi zupełnie za darmo, a w przypadku repozytoriów prywatnych domyślnie uzyskujemy 2000 minut na nasze zadania i 500 MB miejsca na dysku (np. na sam projekt lub cache). Wchodząc na plany płatne i przekraczając poniższe limity, zaczynamy płacić zgodnie z dedykowanym cennikiem.

Terminologia

Aby po świecie GHA poruszać się gładko i sprawnie, zapoznaj się z krótkim opisem najważniejszych pojęć.

Workflow (w lekcji zamiennie - Scenariusz) - Proces automatyzacji składający się z jednego lub wielu zadań. Workflow jest opisywany przy pomocy pliku YAML, przechowywanego w folderze .github/workflows na poziomie repozytorium.

Trigger - Zdarzenie uruchamiające dany workflow. Może to być event związany z aktywnością programisty, zmiana w repozytorium, modyfikacja Pull Requesta, lub jeszcze coś innego. Dany workflow może mieć zdefiniowany więcej niż jeden trigger.

Job - Zadanie uruchamiane na konkretnej maszynie wirtualnej, składające się z jednego lub wielu kroków. W ramach jednego workflow może być realizowane wiele zadań - równolegle lub szeregowo.

Akcja - W kontekście GHA, akcja to opakowany, zamknięty zestaw kroków, które możemy wykorzystywać ponownie w naszych Workflows. Akcja przypomina plugin do VS Code lub bibliotekę z npm, którą możesz opublikować na wewnętrzne lub zewnętrzne potrzeby.

Runner - Maszyna wirtualna, na której uruchamiamy poszczególne Joby. Może być dostarczana przez GitHuba (najczęściej), albo wystawiana przez ciebie (tzw. self-hosted runners). Wpływa na dostępne narzędzia i polecenia w ramach każdego scenariusza.

Środowisko uruchomieniowe

Domyślnie GitHub wspiera trzy rodzaje maszyn wirtualnych (systemów operacyjnych), na których możemy uruchamiać nasze scenariusze:

  • Środowisko Linuxowe - ubuntu-latest

  • Środowisko macOS - macos-latest

  • Dla Windowsa - windows-latest

Pracując na Linuxie lub macOS (tak jak my, tworząc to szkolenie) możesz zdecydować się na to samo ustawienie w scenariuszach, które zobaczysz już za chwile - ubuntu-latest. Większość prezentowanych tutaj komend jest w pełni kompatybilna z Ubuntu i te same operacje uruchomią się zarówno lokalnie, w trakcie developmentu, jak i zdalnie - już na GitHub Actions.

Jeśli jednak pracujesz na Windowsie, możesz zdecydować się na wykorzystanie maszyny windows-latest - wtedy zarówno komendy lokalne, jak i scenariusze GitHub Actions mogą działać nieco inaczej i wymagać innych narzędzi.

Rozwijając tę lekcję staraliśmy się zapewnić kompatybilność zarówno pod macOS, Ubuntu jak i Windowsa, ale jeśli zauważysz problemy, koniecznie nas o tym poinformuj - rozwiążemy je wspólnymi siłami!

Tworzymy pierwszy scenariusz

Znając terminologię, utwórz teraz pierwszy scenariusz automatyzacji (workflow).

Dodaj do repozytorium nowy plik hello.yml, który umieścisz w folderze .github/workflows.

Pamiętaj o sekcji runs-on - dopasuj ją do twojego środowiska pracy zgodnie z poradnikiem powyżej.

# <project_root>/.github/workflows/hello.yml

name: Hello world!

on:
  workflow_dispatch:

jobs:
  app-setup:
    runs-on: ubuntu-latest 
    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Configure node
        uses: actions/setup-node@v4
        with:
          node-version-file: '.nvmrc'

      - name: Install dependencies
        run: npm ci

      - name: Greeting
        run: echo "Brawo! Znakomita robota!" >> $GITHUB_STEP_SUMMARY

Ten scenariusz zawiera wszystkie opisane powyżej terminy. Mamy workflow (cały plik), mamy jeden Job (app-setup), mamy runnera (ubuntu-latest), a także zestaw kroków, z których dwa wykonujemy przez gotowe akcje (checkout i setup-node), a ostatni wykonujemy my sami (npm ci).

Dane wejściowe i konteksty

Wprowadźmy teraz pewną modyfikację do workflow deploy.yml, która pozwoli nam pobierać dane od użytkownika uruchamiającego to zadanie. Dzięki temu będziemy mieć jeszcze więcej kontroli nad jego przebiegiem.

Inputs

Github Actions pozwala nam definiować i wprowadzać dane wejściowe na kilka sposobów - możemy to realizować poprzez sekrety z repozytorium, poprzez zmienne środowiskowe a także przez inputy, czyli dane pobierane od użytkownika.

Do poznanego wcześniej triggera workflow_dispatch dodajmy teraz pierwszy input typu boolean.

on:
  workflow_dispatch:
    inputs:
      deploy-netlify:
        type: boolean
        description: Enable Netlify deployment
        default: true

Input definiujemy poprzez nazwę (deploy-netlify) oraz trzy dodatkowe pola - typ, opis i wartość domyślna.

Kiedy wrzucisz do repozytorium tę zmianę, to w momencie uruchamiania zadania Github zaktualizuje dla nas interfejs danej akcji:

Aby na poziomie scenariusza pobrać wartość danego inputa, korzystamy z tzw. context objectu, albo po prostu kontekstu. Kontekst jest obiektem zawierającym informacje o wybranym aspekcie danej automatyzacji, maszyny wirtualnej czy serii zadań, z którego możemy pobierać dane.

Do kontekstu uzyskujemy dostęp poprzez następującą składnię:

${{ context.property }}

W naszym przypadku będzie to:

${{ inputs.deploy-netlify }}

Może ci to przypominać wykorzystywane wcześniej sekrety - to bardzo dobra intuicja, zarówno secrets jak i inputs to dwa z wielu kontekstów, z których możesz korzystać pisząc scenariusze automatyzacji.

Innymi kontekstami mogą być chociażby:

  • ${{ github.<prop> }} - szeroki kontekst Githuba - adres API, nazwa użytkownika czy nazwa scenariusza

  • ${{ job.<prop> }} - status joba, informacje o maszynie wirtualnej i rezultat serii kroków

  • ${{ strategy.<prop> }} - kontekst zadań uruchamianych grupowo - poznamy go w kolejnych lekcjach

Wracając do naszego przykładu, dodajemy teraz nowe pole - if - do akcji Netlify, które uwarunkuje ten krok od włączenia checkboxa w momencie uruchamiania akcji:

- name: Deploy to Netlify
  uses: nwtgck/actions-netlify@v3.0
  if: ${{ inputs.deploy-netlify }}
  with:
    ...

W tym momencie krok “Deploy to Netlify” wykona się tylko wtedy, kiedy przy uruchomieniu automatyzacji będziesz miał zaznaczony checkbox “Enable Netlify deployments”. Aby to sprawdzić, przetestuj scenariusz uruchamiany w dwóch trybach - przy włączonym i wyłączonym inpucie.

Warto dodać, że boolean to nie jedyny typ danych, które możemy pobierać na starcie. W zależności od tego, jakich danych potrzebujemy, możemy korzystać z inputów innego typu. GHA wspiera cztery możliwości:

  • string

  • number

  • boolean

  • choice (single-select)

Od twojej kreatywności i realizowanego zadania zależy, jakich danych wejściowych potrzebujesz od użytkownika, a jakie będziesz definiował wewnątrz scenariusza. Kolejne przykłady pobierania i przekazywania danych przetestujemy już w kolejnej lekcji.

Podsumowanie

Gratulacje! Twoja aplikacja jest automatycznie wdrażana na popularną chmurę frontendową, wykorzystując przy tym ręcznie skonfigurowany workflow Github Actions. Co prawda sam skrypt nie wygląda optymalnie, ale już w kolejnej lekcji zajmiemy się kompozycją akcji, kontrolą przepływu danych i nowymi triggerami.

Czas na kilka ćwiczeń praktycznych, które rozbudują przygotowany przez nas scenariusz.

👨‍💻 Ćwiczenia praktyczne

  • Na potrzeby wszystkich lekcji z modułu trzeciego prosimy o utworzenie prostego projektu w dowolnej technologii, którego repozytorium znajduje się na GitHubie. Zrealizujemy na nim kolejne ćwiczenia.

  • Na tym etapie twój projekt powinien zawierać co najmniej jeden scenariusz - deploy.yml - który będzie wykonywany na żądanie. Jego finalnym krokiem powinno być wdrożenie aplikacji na Netlify.

  • Rozbuduj pierwszy workflow o uruchomienie testów jednostkowych

    • W szablonie frontend-bootstrap dodany jest Vitest - napisz prosty test jednostkowy, który uruchomisz przy pomocy Github Actions. Alternatywnie, dodaj pierwsze testy do własnego projektu.

    • Wykorzystaj istniejącą komendę z npm scripts (package.json) do uruchomienia testów w trakcie scenariusza automatyzacji na Github Actions.

  • Rozbuduj scenariusz o dodatkowy deployment na platformę Vercel

    • Przykładowy scenariusz znajdziesz pod tym linkiem - będziemy go realizować instalując Vercel CLI w momencie uruchamiania scenariusza.

    • Potrzebujesz nowych sekretów z twojego konta na Vercelu:

      • ID projektu: Project Settings → General → Project ID

      • ID organizacji: Settings → General → Team ID

      • Tokena, który wygenerujesz w ustawieniach konta.

    • Dodaj kolejny input, który będzie odpowiedzialny za deployment na Vercela.

    • W razie problemów, gotowy scenariusz znajdziesz w folderze “_solutions” do dzisiejszej lekcji.
