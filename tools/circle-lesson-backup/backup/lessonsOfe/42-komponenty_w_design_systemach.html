Core components

Core components to kluczowa warstwa design systemu, zwłaszcza z punktu widzenia frontend developera. To właśnie tutaj żyją wszystkie przyciski, kontrolki formularzy i inne proste komponenty, których nie chcemy w kółko kodzić na nowo. Solidne core components są zaprojektowane tak, aby były łatwe w kompozycji, elastyczne i gotowe do dostosowania do konkretnych potrzeb różnych produktów lub aplikacji.

Repozytorium core components

Repozytorium kodu jest krytycznym aspektem warstwy core components. Z punktu widzenia użytkownika to właśnie tutaj, a nie w Figmie, jest źródło prawdy na temat design systemu (przestrzegam: dla wielu designerów nie jest to łatwe do zaakceptowania). W tym repozytorium core components są tworzone, rozwijane i wersjonowane - często to główne miejsce kolaboracji międzyzespołowej dla frontend engineerów.

Na początkowym etapie prac nad design systemem, to właśnie tutaj czeka na nas jedna z kluczowych decyzji dotyczących wyboru technologii i architektury. Czy zbudować core components przy użyciu Web Components czy konkretnego frameworka, takiego jak React, Angular lub Vue.

Co wybrać: web components czy framework frontendowy?

Web Components to zestaw interfejsów API platformy webowej, które umożliwiają developerom tworzenie niestandardowych elementów wielokrotnego użytku. Są one zbudowane przy użyciu natywnych technologii internetowych, takich jak HTML, CSS i JavaScript, dzięki czemu są niezależne od frameworka i wysoce interoperacyjne.

W ostatnich latach znaczną popularność zdobył Lit od Google, który jest frameworkiem Web Components (popularność lit vs svelte - zdziwisz się!). Lit ułatwia nam tworzenie web components, zapewniając topowy developer experience, składnię i najlepsze rozwiązania znane z wielkiej trójki.

Z drugiej strony, frameworki takie jak React, Angular i Vue, które proponują własne standardy tworzenia komponentów, mogą się pochwalić wielkimi ekosystemami bibliotek oraz większą ilością developerów, którzy mieli już styczność z dana technologią.

Podobnie jak Brad Frost uważam, że budowanie core-components przy użyciu web components, a konkretniej Lita, jest preferowanym podejściem dla większości organizacji. Web Components oferują kilka zalet w porównaniu do komponentów specyficznych dla frameworka:

  1. Interoperacyjność: Web Components mogą być używane z dowolnym frameworkiem lub biblioteką JavaScript, dzięki czemu są bardzo wszechstronne i przyszłościowe. Pozwala to organizacjom na korzystanie z tych samych komponentów interfejsu użytkownika w różnych projektach, niezależnie od stacku technologicznego. Nawet jeżeli organizacja działa obecnie tylko w jednym frameworku, może być to jedynie kwestią czasu choćby za sprawą przejęcia mniejszej firmy, która rozwijała swoje produkty w innym stacku (been there, seen that)

  2. Wydajność: Web components są lekkie i mają minimalny narzut w porównaniu do komponentów specyficznych dla frameworka, co skutkuje krótszym czasem ładowania i lepszą wydajnością.

  3. Długowieczność: Web Components są oparte na standardach webowych i nie są powiązane z żadnym konkretnym frameworkiem. Oznacza to, że jest mniej prawdopodobne, że staną się przestarzałe lub będą wymagały znacznej refaktoryzacji w miarę ewolucji frameworków lub pojawiania się nowych.

  4. Dostępność: Web components są budowane przy użyciu natywnych elementów HTML, które z natury obsługują funkcje dostępności. Ułatwia to tworzenie dostępnych komponentów interfejsu użytkownika, które są zgodne z wytycznymi dotyczącymi dostępności, aczkolwiek Shadow DOM czasem rodzi nieoczywiste problemy na tym polu.

Tworzenie core components przy użyciu frameworków takich jak React, Angular lub Vue może być kuszące, zwłaszcza jeśli organizacja w dużym stopniu polega na jednym z tych frameworków. Jednak takie podejście często prowadzi do istotnych wyzwań:

  1. Lock-in: Komponenty specyficzne dla frameworka są ściśle powiązane z frameworkiem, w którym zostały zbudowane. Może to prowadzić do sytuacji lock-in, w której organizacja staje się silnie uzależniona od konkretnego frameworka, co utrudnia zmianę lub przyjęcie nowych technologii w przyszłości. Kto widział ile pracy wymagała migracja z Angular.js do Angulara ten wie, że lepiej nie oddawać serca naszych interfejsów na łaskę wojny frameworków.

  2. Koszty utrzymania: Frameworki szybko ewoluują, a nadążanie za najnowszymi wersjami, aktualizacjami i przełomowymi zmianami może być znacznym obciążeniem. Może to prowadzić do zwiększonych kosztów developmentu i wolniejszego wdrażania nowych funkcji.

  3. Ograniczona możliwość ponownego wykorzystania: Komponenty specyficzne dla frameworka nie są łatwe do ponownego wykorzystania w różnych projektach lub frameworkach. Może to prowadzić do powielania wysiłków i niespójności w implementacji design systemu.


Podczas gdy niektóre duże organizacje, takie jak Microsoft, dysponują zasobami pozwalającymi na utrzymanie zarówno pełnoprawnych core components w React, jak i Web Components, dla większości organizacji jest to niewykonalne.

Utrzymywanie wielu wersji komponentów UI wymaga znacznych inwestycji pod względem czasu developmentu, testowania i dokumentacji. Większość organizacji musi wybierać między Web Components a komponentami specyficznymi dla frameworka, biorąc pod uwagę swoje zasoby, wymagania techniczne i długoterminowe cele.

W SmartRecruiters postawiliśmy na Lita, z dwóch powodów:

  1. Aplikacje produktowe są rozwijane w dwóch technologiach: Angular oraz React, stąd interoperacyjność Web Components staje się dużym plusem.

  2. Chcieliśmy zabezpieczyć się na wypadek powtórki z Angular.js’a. “Nowy” Angular również może kiedyś zostać zastąpiony, czy to przez kolejny framework od Google, czy też miażdżącą konkurencję - rozsądnie uodpornić core components na taką sytuację.

Oczywiście każda organizacja jest inna. O to istotne powody, które mogą przechylić bilans na rzecz frameworka frontendowego:

  1. Istniejący code base i doświadczenie: Jeśli organizacja dokonała już znacznej inwestycji w konkretny framework frontendowy i posiada dużą bazę kodu zbudowaną przy użyciu tego frameworka, bardziej praktyczne może być tworzenie komponentów interfejsu użytkownika przy użyciu tej samej technologii. Pozwala to organizacji wykorzystać istniejącą wiedzę, narzędzia i biblioteki, zmniejszając krzywą uczenia się i umożliwiając szybszy development. Warto jednak zwrócić uwagę, że nauka Lita nie zajmuje dużo czasu - mamy do dyspozycji świetny interaktywny tutorial oraz dokumentację a składnia przypomina mix tego co najlepsze w Angularze i React’cie.

  2. Bogaty ekosystem i wsparcie społeczności: Frameworki frontendowe, zwłaszcza React, mają duże i aktywne społeczności, które współtworzą ich ekosystemy. Oznacza to dostęp do szerokiej gamy bibliotek firm trzecich, narzędzi i bibliotek komponentów UI, które można wykorzystać jako solidny fundament dla design systemu, co przyspieszy jego development, zwłaszcza na początkowych etapach.

  3. Renderowanie po stronie serwera i optymalizacja wydajności: Frameworki frontendowe oferują możliwość renderowania po stronie serwera bez dodatkowej konfiguracji. Może to być korzystne dla design systemów, które priorytetowo traktują optymalizację pod kątem wyszukiwarek (SEO) i wydajność początkowego ładowania strony. W przypadku Web Components, SSR od lat rodzi wyzwania a Lit obecnie oferuje eksperymentalne wsparcie dla tej techniki.

  4. Integracja z istniejącymi narzędziami: Jeśli organizacja zainwestowała już w tooling, który jest ściśle zintegrowane z konkretnym frameworkiem frontendowym, użycie tego frameworka dla komponentów UI design systemu może zapewnić płynniejszą integrację.


Jak widzisz jest to złożona decyzja, w której trzeba wziąć pod uwagę kontekst organizacji, jej dotychczasową historię, obecne i przyszłe cele oraz dostępne zasoby do rozwijania core components.

Core components Storybook

Storybook to narzędzie open-source do tworzenia, testowania i dokumentowania komponentów UI w izolowanym środowisku. Zapewnia wygodne środowisko programistyczne, które pozwala nam budować i prezentować komponenty niezależnie od głównej aplikacji. Storybook obsługuje różne technologie frontendowe, w tym Web Components, React, Angular, Vue i wiele innych.

Kluczowe funkcje i zalety:

  1. Isolated development: Storybook zapewnia środowisko do budowania i testowania komponentów UI w izolacji, bez konieczności poruszania się po aplikacji. Ta izolacja ułatwia skupienie się na poszczególnych komponentach i zapewnia, że są one odpowiednio hermetyzowane i przygotowane do wielokrotnego użytku.

  1. Interaktywna dokumentacja: Storybook automatycznie generuje interaktywną dokumentację dla komponentów UI w oparciu o historyjki tworzone przez developerów. Dokumentacja ta prezentuje różne stany i warianty każdego komponentu, ułatwiając designerom, developerom i stakeholderom zrozumienie, jak działają komponenty i jak efektywnie z nich korzystać. Zobacz sam na przykładzie historyjki komponentu Search w Storybooku Fluent Web Components.

  1. Ekosystem addonów: Storybook oferuje bogaty ekosystem dodatków, które rozszerzają jego funkcjonalność. Dodatki te zapewniają takie funkcje, jak testowanie responsywnego układu, sprawdzanie dostępności, wizualne testy regresji i wiele innych. Dodatki pomagają zapewnić, że komponenty UI są dokładnie testowane i spełniają niezbędne standardy jakości. Zainstalowane addony można znaleźć w panelu addonów, wyświetlanym przez Storybook pod canvasem z komponentem. Każda karta pokazuje wygenerowane przez dodatek metadane, logi lub analizę statyczną dla aktywnej historyjki komponentu.

Jak wskazuje dokumentacja, podstawowe addony domyślnie aktywowane dla każdego projektu, to:

  1. Controls, które udostępniają interfejs do dynamicznej interakcji z propsami komponentu. Ciężko wyobrazić sobie lepszy sposób na zapoznanie się z core componentem, niż poprzez obserwację jak zmieniające się wartości propsów wpływają na jego wygląd i zachowanie.

  1. Actions, które pomagają zweryfikować, czy interakcje generują prawidłowe eventy za pośrednictwem wywołań callbacków. Przykładowo możemy wykorzystać fn do spy'owania handlera onClick komponentu Button. 

// Replace your-framework with the name of your framework
import type { Meta } from '@storybook/your-framework';
import { fn } from '@storybook/test';

import { Button } from './Button';

const meta: Meta<typeof Button> = {
  component: Button,
  // 👇
  args: { onClick: fn() },
};

export default meta;

Po kliknięciu przycisku w historyjce, jeżeli handler będzie prawidłowy podpięty, to jego nazwa zostanie wyświetlona w panelu Actions:

  1. Interactions zapewnia interfejs użytkownika do debugowania testów interakcji za pomocą funkcji play. Plugin pozwala na symulowanie interakcji z komponentem, to szczególnie przydatne dla warstwy receptur, o której więcej w dalszej części lekcji.

Oczywiście to dopiero początek, mamy ogrom addonów dostępnych w marketplace, w tym:

  • Accessibility - uruchamia testy axe-core na historyjce komponentu. To świetny sposób na podstawowe testy automatyczne dostępności, więcej na ten temat w lekcji #Czym jest dostępność (a11y)?. Komponenty design systemu powinny spełniać najwyższe standardy dostępności wymagane wewnątrz firmy, stąd warto korzystać z tego rozszerzenia oraz zadbać o bogaty pakiet testów a11y w Playwright.

  • msw - integracja pomiędzy API Storybooka a biblioteką msw.js. Pozwala na mockowanie zapytań do API wykonywanych przez historyjkę komponentu. Core components z zasady są “głupie” i odpowiadają jedynie za prezentację. Plugin przydaje się w warstwie Smart components, o której w dalszej części lekcji. msw poznaliśmy już przy okazji testów UI w lekcji #Wprowadzenie do testów e2e i Playwright.

  • Designs - pozwala podpiąć design komponentu z design library w Figmie do historyjki komponentu.

  • Storysource - wyświetla panel z kodem źródłowym historyjki, co pozwala developerem lepiej zrozumieć jak API komponentu jest wykorzystywane w środowisku historyjki.

  • Themes - pozwala przełączać się pomiędzy motywami komponentów, np. ciemny/jasny.

  • Chromatic - zapewnia integrację Storybooka z narzędziem Chromatic, które służy do testów wizualnych komponentów i zapewnia przyjazny interfejs dla wszystkich osób pracujących nad design systemem. Chromatic poznamy bliżej w dalszych lekcjach.

  1. Współpraca i feedback: Storybook ułatwia współpracę między członkami zespołu, zapewniając wspólne środowisko do omawiania i zbierania opinii na temat komponentów. Designerzy, developerzy i stakeholderzy mogą wchodzić w interakcje z komponentami w Storybook, zostawiać komentarze i sugerować ulepszenia, wspierając lepszą komunikację w zespole.

  2. Niezależność technologiczna: Storybook obsługuje szeroką gamę technologii frontendowych, dzięki czemu jest wszechstronnym narzędziem do rozwoju design systemu. Bezproblemowo współpracuje z Web Components, React, Angular, Vue i innymi popularnymi frameworkami i bibliotekami.

Storybook jest stosunkowo prosty w konfiguracji, o czym przekonasz się w kolejnych lekcjach, gdy będziemy stawiali fundamenty prostego design systemu. Dzięki temu można go szybko zaimplementować w większości firm, nawet jeżeli mamy do dyspozycji jedynie kilka komponentów wielokrotnego użytku. Jednocześnie Storybook świetnie się skaluje i jest w stanie obsłużyć rozbudowane biblioteki komponentów takie jak Fluent UI React Components v9.

Storybook jest szczególnie ważnym narzędziem w kontekście rozwoju design systemu z kilku powodów:

  1. Centrum kolaboracji: Storybook często służy jako centralny hub dla komponentów design systemu. Developerzy mogą organizować komponenty w kategorie, tworzyć hierarchiczną strukturę i dostarczać szczegółową dokumentację dla każdego komponentu. Storybook może być używany do prezentacji palety kolorów, typografii, ikon i innych design tokenów, zapewniając spójność wszystkich komponentów.

  2. Testowanie i jakość: Storybook ułatwia testowanie komponentów UI. Developerzy mogą pisać testy jednostkowe, testy UI i visual regression testy dla każdego komponentu za pomocą addonów. Dzięki temu Storybook tworzy środowisko, w którym komponenty są dokładnie testowane, a wszelkie regresje są wychwytywane na wczesnym etapie developmentu.

  3. Dystrybucja interaktywnej dokumentacji: Storybook można zbudować i wdrożyć jako statyczną aplikację webową za pomocą Netlify/Vercel/AWS, co ułatwia udostępnianie design systemu w całej organizacji. Designerzy, developerzy i stakeholderzy mogą korzystać ze Storybooka, aby eksplorować dostępne komponenty, przeglądać ich dokumentację i wchodzić z nimi w interakcje bez konieczności logowania się na produkcję.

Wykorzystując Storybook w design systemie, organizacje mogą stworzyć solidną, dobrze udokumentowaną i łatwą w utrzymaniu bibliotekę komponentów UI. Wsparcie Storybook dla różnych technologii, bogaty ekosystem dodatków i funkcje ułatwiające współpracę sprawiają, że jest to niezbędne narzędzie do efektywnego budowania i zarządzania design systemami.

Paczka biblioteki core components

Pakiet (aka paczka) core components jest tworzony poprzez budowanie kodu źródłowego modułu core-components w monorepozytorium do artefaktu, który jest publikowany w repozytorium (takim jak npm) i może być wykorzystywany przez inne projekty, które konsumują design system.

Strona referencyjna

Strona referencyjna służy jako centrum wiedzy o design systemie, łącząc wszystkie różne komponenty i zasoby pod jednym dachem.

Powinna gromadzić zasoby, takie jak design tokeny, ikony, komponenty, dokumentację i wytyczne dotyczące marki. Centralizuje wszystkie te składniki, aby zapewnić interdyscyplinarne wskazówki dla designerów i developerów korzystających z design systemu w różnych produktach.

Strona referencyjna działa jednocześnie jako witryna sklepowa, strona marketingowa, źródło dokumentacji i kanał wsparcia dla zespołów korzystających z design systemu.

Biorąc pod uwagę tyle różnorodnych kontekstów, strona referencyjna realizuje różnorodne cele:

  • Zapewnienie pojedynczego źródła prawdy dla wytycznych, zasad i zasobów design systemu.

  • Promowanie design systemu wewnątrz organizacji (witryna zamknięta) oraz wśród społeczności open source (witryna otwarta).

  • Dokumentowanie komponentów, wzorców i wytycznych dotyczących użytkowania design systemu.

  • Umożliwienie łatwego przeglądania i odkrywania zasobów design systemu.

  • Ułatwienie komunikacji i wsparcia wokół design systemu.

Istnieje kilka opcji tworzenia witryny referencyjnej:

  1. Zbuduj od podstaw - zespół może zbudować niestandardową stronę referencyjną od podstaw, pobierając zawartość i komponenty z różnych źródeł design systemu (biblioteki Figma, repozytoria kodu itp.). Pozwala to na pełne dostosowanie strony referencyjnej, ale wymaga większego nakładu pracy ze strony developerów. Do budowy strony referencyjnej polecamy Astro, na które postawił również Microsoft w przypadku pięknej strony referencyjnej Fluent 2 (case study o Astro w Fluent 2).

  2. Użyj narzędzia innej firmy - rozwiązania takie jak Zeroheight zapewniają gotowe narzędzia do generowania referencyjnej strony internetowej bezpośrednio z istniejących źródeł designu (Figma) i kodu (Storybook). Zmniejsza to nakład pracy, ale ogranicza możliwość dostosowania strony referencyjnej do naszej marki oraz rozbudowywania zaawansowanych funkcji. Jako przykład zastosowania Zeroheight polecam Base design system od Ubera.

Jeśli twój design system nie jest open source lub jest na wczesnym etapie rozwoju, korzystanie z samego Storybooka powinno w zupełności wystarczyć. W stronę referencyjną warto zainwestować, kiedy design system ma już solidne fundamenty.

Opcjonalna implementacja pod platformę

W dojrzałym ekosystemie design system, warstwa implementacji specyficzna dla platformy jest opcjonalnym, ale często niezbędnym komponentem. Warstwa ta przekłada core design system na konkretne implementacje techniczne, ułatwiając wykorzystanie komponentów design systemu przez developerów pracujących z konkretnymi stackami technologicznymi.

Potrzeba istnienia tej warstwy wynika z faktu, że podczas gdy core design system może być zbudowany przy użyciu technologii takiej jak Web Components, które są interoperacyjne z dowolnym frameworkiem webowym, frameworkowe wrappery nadal mogą okazać się pomocne. Może to być spowodowane kodem legacy, preferencjami developerów lub potrzebą wykorzystania funkcji specyficznych dla frameworka w danym komponencie (najczęściej dotyczy to receptur oraz smart components). Niektóre frameworki - zwłaszcza wcześniejsze wersje Reacta - wymagają dodatakowej integracji, aby Web Components działały w 100% poprawnie. Libka @lit/react jest przykładem wrappera, który pozwala aby Web Components działały płynnie ze sposobem, w jaki React obsługuje przekazywanie danych i obsługę eventów. Przykładem świetnej integracji Web Components z Reactem jest First-class React support w Shoelace.

Repozytorium kodu frameworka

Są dwa podejścia do przechowywania kodu komponentów powiązanych z frameworkami:

  1. Pierwszym z nich, popularniejszym, jest trzymanie tych komponentów w dedykowanej paczce monorepo, obok podstawowych komponentów w web components.

  2. Drugim z nich jest wydzielenie tych wrapperów do osobnego repo, które jest utrzymywane przez zespół/zespoły wykorzystujące dany framework w swoich projektów.

Niezależnie od tego, gdzie znajduje się warstwa frameworka, ważne jest, aby warstwy podstawowe i specyficzne dla frameworka były ze sobą zsynchronizowane. Może w tym pomóc tooling do zarządzania zależnościami, ale ostatecznie wymaga to koordynacji między warstwami, za które mogą odpowiadać różne zespoły.

Storybook dla frameworka

Jeśli istnieje design system w wersji dla React, konieczne jest utworzenie dedykowanego Storybooka, aby zapewnić React-devom odpowiednią składnię, do której mogą się odwoływać, i dzięki temu kopiować i wklejać gotowe komponenty do projektów.

Właśnie z tego powodu Microsoft utrzymuje osobny storybook dla Fluent Web Components i Fluent React Components.

Paczka frameworkowa

Każdy specyficzny dla frameworka wariant biblioteki design systemu jest budowany i publikowany w rejestrze typu npm, co pozwala developerom pracującym w określonym frameworku na pobranie odpowiedniego wariantu design systemu jako zależności do projektu.

Implementacje poza-webowe

Aplikacje mobilne na iOS i Androida są jednymi z bardziej powszechnych poza-webowych produktów cyfrowych, ale w organizacji może istnieć ogromna liczba innych interfejsów oprogramowania typu embedded. Niezależnie od konkretnej technologii zastosowanej w tych interfejsach, obowiązują te same wytyczne: utwórz dedykowane repozytorium dla wspólnego kodu specyficznego dla platformy i wdróż ten kod do aplikacji za pomocą jakiejś formy rejestru pakietów.

Opcjonalna warstwa receptur

Zarządzanie UI w dużych organizacjach bywa wyzwaniem. Zespół opiekujący się design systemem często jest przeciążony śledzeniem i implementacją każdego żądania dotyczącego UI, co prowadzi do opóźnień i wypalenia zawodowego.

Warto jednak pamiętać, że design system nie musi obejmować każdego elementu UI. Kluczowym jest zapewnienie podstawowego zestawu komponentów (core components) i motywowanie zespołów do tworzenia własnych „receptur” z tych elementów. Taki podejście odciąża zespół DS, umożliwiając zespołom produktowym projektowanie specyficznych komponentów UI przy zachowaniu ogólnych standardów design systemu.

Przykładem receptury może być komponent Pricing z Tailwind UI, który składa się z przycisków, typografii, listy oraz dividera.

Większość design systemów, w tym Fluent 2, nie udostępnia swojej biblioteki receptur - często zawierają one smart components, rozszerzone o logikę biznesową, która powinna być znana jedynie developerom wewnątrz organizacji. Jak widać poniżej, sekcja receptur w design systemie Carbon jest dostępna tylko dla pracowników IBM:

Kluczowe elementy warstwy receptur:

  • Receptury: są to zestawy skomponowanych core components, z których można tworzyć złożone interfejsy użytkownika. Umożliwiają zespołom tworzenie kompozycji spełniających indywidualne potrzeby rozwijanych przez nie produktów.

  • Biblioteki i Repozytoria Receptur: Zawierają komponenty i kompozycje dostosowane do potrzeb poszczególnych jednostek biznesowych. Zwykle utrzymywane przez jednostkę biznesową, a nie zespół DS.

  • Paczki z Recepturami: Aby ułatwić dostęp i wykorzystanie, receptury są publikowane jako gotowe do użycia pakiety, konsumowane przez zespoły z wybranej jednostki biznesowej.

  • Storybook i witryna referencyjna receptur: Służą jako przewodniki po specyficznych dla produktu komponentach i ich zastosowaniu, zapewniając wytyczne i przykłady użycia.

Zdarza się, że niektóre receptury z czasem trafiają do warstwy core components. Ma to sens, gdy receptura staje się szeroko stosowana w całej organizacji.

Przekształcenie podejścia do design systemu poprzez implementację warstwy receptur daje zespołom produktowym większą elastyczność i autonomię w tworzeniu UI, przy jednoczesnym zachowaniu spójności i jakości projektowania.


Opcjonalna warstwa Smart Components

Warstwa smart Components wprowadza logikę do "głupich" core components. Core components, skoncentrowane na prezentacji, nie posiadają wbudowanej logiki biznesowej ani połączeń z usługami backendowymi. Natomiast smart components wzbogacają je o logikę potrzebną do interakcji z backendem, oferując gotowe do użycia rozwiązania dla developerów.

Zakres użycia takich komponentów jest szeroki - od walidacji formularzy, przez procesowanie płatności, po dynamiczne zapytania w komponencie wyszukiwarki. Smart components pozwalają zespołom skupić się na szybszej implementacji złożonych, ale powtarzalnych interfejsów użytkownika, eliminując potrzebę ciągłego wynajdywania koła na nowo.

Zarządzanie tymi komponentami często spoczywa na zespole technicznym, który zajmuje się produktem/usługami związanymi z danym komponentem. Repozytoria i pakiety smart components mogą mieć różną strukturę, zależnie od architektury produktu, ale zwykle są organizowane jako odrębne projekty w dedykowanych repozytoriach, co ułatwia ich użytkowanie i aktualizację. W przypadku wielu komponentów, można stworzyć mono-repo dla powiązanych smart components, aby uprościć ich konfigurację i utrzymanie.

Warstwa Produktu

Warstwa produktu to miejsce, w którym design system ożywa, zasilając rzeczywiste interfejsy użytkownika stron internetowych i aplikacji organizacji. Jest to warstwa, z którą użytkownicy końcowi wchodzą w bezpośrednią interakcję, co czyni ją najbardziej widoczną i wpływową częścią design systemu.

W warstwie produktu, designerzy i developerzy wykorzystują bloki konstrukcyjne dostarczane przez core design system, a także wszelkie opcjonalne warstwy, takie jak implementacje specyficzne dla technologii, receptury i smart components. Product designerzy tworzą designy interfejsu użytkownika w narzędziach takich jak Figma, subskrybując biblioteki core design systemu, biblioteki core components i wszelkie odpowiednie biblioteki receptur. Pozwala im to efektywnie projektować widoki oraz doświadczenia użytkownika, zachowując jednocześnie spójność z ogólnym językiem projektowania organizacji.

Po stronie developera, code base produktu wykorzystuje odpowiednie pakiety z design systemu, receptur i smart components jako zależności.

Wykorzystując bloki konstrukcyjne design systemu, zespoły produktowe mogą skupić się na tworzeniu unikalnych funkcji i doświadczeń użytkowników, które wyróżniają ich produkty, zamiast wymyślać na nowo wspólne elementy interfejsu użytkownika i interakcje.

Case study: Wykorzystanie Fluent 2 przez Microsoft w aplikacjach Teams i Outlook

Fluent Design System jest doskonałym przykładem efektywnego wykorzystania design systemu w warstwie produktu. Dwa z flagowych produktów Microsoft, Teams i Outlook, przyjęły Fluent 2, aby stworzyć spójne, nowoczesne i dostępne interfejsy użytkownika.

Jak zaprezentowane w case study Microsoft Teams, komponenty i wzorce Fluent 2 są używane w całej aplikacji, od paska bocznego nawigacji i paska poleceń po interfejs czatu i elementy sterujące połączeniami wideo. Zastosowanie Fluent 2 zapewnia, że aplikacja Teams zachowuje spójny wygląd z innymi produktami firmy Microsoft, zapewniając jednocześnie środowisko użytkownika dostosowane do unikalnych potrzeb zdalnej współpracy i komunikacji.

Podobnie z case study Microsoft Outlook dowiadujemy się, że zespół wykorzystał Fluent 2 w nowej aplikacji pod Windowsa, dzięki czemu odświeżono ten produkt, zachowująć spójność z resztą ekosystemu firmy.

Trzy strategie wdrażania design systemów w firmie

Okej, mamy to! Omówiliśmy już wszystkie warstwy i elementy anatomii design systemu, teraz zastanówmy się kiedy design system ma sens i jakie strategie wdrożeniowe możemy obrać.

Stworzenie pełnoprawnego design systemu prawdopodobnie nie wygeneruje pożądanego zwrotu z inwestycji w krótkim terminie. W końcu główną korzyścią jest przyspieszenie i podniesienie jakości projektów, które będą realizowane w przyszłości. Jeśli wątpisz w przydatność design systemu, warto rozważyć ramy czasowe, których używasz do oceny zwrotu z inwestycji. Design systemu są najsensowniejsze, gdy firma przewiduje lata przyszłych, powtarzalnych prac nad produktem bądź rodziną produktów.

Bardzo duży wpływ na zwrot inwestycji ma strategia wdrażania design systemu, którą wybierzemy:

  • Przyjęcie (Adopt) istniejącego design systemu, np. Fluent, i wykorzystanie go w projektach bez większych modyfikacji.

  • Adaptacja (Adapt) istniejącego design systemu poprzez ustalenie własnych design tokenów, tworzenie wrapperów i dodawanie własnych komponentów.

  • Tworzenie (Create) własnego design systemu od podstaw.

Każda z tych metod ma swoje plusy i minusy, ale ogólnie rzecz biorąc, im bardziej stawiamy na konfigurację i niezależność, tym więcej czasu i pieniędzy zajmie wdrożenie DS. W związku z tym korzystanie z istniejącego design systemu jest najtańszym podejściem i wymaga najmniej czasu na wdrożenie. W krótkim terminie, i tak będzie to wymagało więcej czasu niż w przypadku kontynuowania designu i developmentu w “zwykły” sposób.

Inwestycja we własny design system będzie opłacalna, jeśli organizacja ma szczególne potrzeby związane z brandingiem i interakcjami, których nie mogą zaspokoić design systemy typu open source. Wraz ze wzrostem liczby customozacji i modyfikacji, oszczędności wynikające z korzystania z istniejącego design systemu zmniejszą się.

Upewnij się, że wiesz, czego potrzebuje Twoja organizacja, zanim podejmiesz wysiłki związane z design systemem i ocenisz kompromisy związane z tymi trzema strategiami.

Zespół zarządzający design systemem

Design system jest tak skuteczny, jak zespół, który nim zarządza. Niezależnie od tego, czy jest tworzony od podstaw, czy dostosowywany, design system wymagają ciągłego utrzymania i nadzoru. Wielkość zespołu utrzymującego DS może się różnić, biorąc pod uwagę, że same systemy projektowe mogą przybierać różne rozmiary i poziomy dostosowania, ale zespół co najmniej powinien obejmować 1 designera i 1 programistę, z których każdy ma pomóc w napisaniu wytycznych, tworzeniu przykładów wizualnych i dostarczaniu specyfikacji implementacji dla każdego elementu.

Wreszcie, warto pozyskać sponsora (z szeregów kierownictwa) w celu koordynowania wysiłków związanych z design systemem. Chociaż brak sponsora nie będzie przeszkodą, sponsorzy mogą zabezpieczyć budżet i zasoby, jednocześnie przekazując strategiczne znaczenie design systemu reszcie organizacji.

Oczywiście skuteczny design system to wynik efektywnej kolaboracji większości designerów i frontendowców w firmie. Zespół zarządzający nadzoruje pracę, podczas gdy indywidualni kontrybutorzy implementują brakujące rozwiązania zgodnie z modelem InnerSource. Więcej na ten temat w lekcji trzeciej tego modułu.

Od czego zacząć wdrażanie design systemu?

Zanim przejdziemy do designu i developmentu, należy wykonać kilka kroków przygotowawczych, które zapewnią nam niezbędną wiedzę i fundamenty pod sprawną implementację przemyślanego design systemu. Są one genialnie opisane w thedesignsystem.guide:

  1. Przeprowadź audyt designu

  2. Ustal fundamenty (wytyczne designerskie, komponenty)

  3. Zdefiniuj design tokeny

  4. Ustal model współpracy

Dlaczego warto stosować design systemy?

Teraz gdy mamy już pełne zrozumienie czym są design systemy oraz jak wdrażać je do organizacji, dokładniej przyjrzyjmy się korzyściom, które oferują:

  1. Praca designerów i frontendowców może być powielana szybko i na dużą skalę.

Podstawową zaletą design systemów jest ich zdolność do szybkiego powielania designów poprzez wykorzystanie gotowych komponentów i elementów interfejsu użytkownika. Zespoły mogą stale korzystać z tych samych elementów, zmniejszając potrzebę odkrywania koła na nowo, a tym samym ryzykując niezamierzoną niespójność.

  1. Tworzy ujednolicony język w zespołach międzyfunkcyjnych i pomiędzy nimi

Jeśli nie ma wspólnego języka, grupie ludzi trudno jest współpracować. Zwłaszcza, gdy członkowie zespołów zmieniają się lub gdy zespoły są rozproszone geograficznie. Każdy pracownik ma wtedy inny sposób myślenia i zestaw celów, które chce osiągnąć.

Ujednolicony język zmniejsza marnowanie czasu na design oraz development w związku z nieporozumieniami. Na przykład, funkcjonalność lub wygląd dropdown menu nie będzie przedmiotem dyskusji, ponieważ termin ten jest zarezerwowany dla konkretnie zdefiniowanego elementu w design systemie.

  1. Odciążają designerów i pozwalają skupić się na większych, bardziej złożonych problemach.

Ponieważ prostsze elementy interfejsu użytkownika są już stworzone i nadają się do ponownego wykorzystania, designerzy mogą poświęcać mniej czasu na implementacje bazowych widoków, a bardziej na bardziej złożonych problemach takich jak ustalanie priorytetów informacji, optymalizacja przepływu pracy i zarządzanie podróżami użytkowników. Chociaż korzyści te mogą wydawać się niewielkie, gdy tworzy się tylko niewielką liczbę widoków, stają się one znaczące, gdy trzeba koordynować wysiłki w dziesiątkach zespołów i przy dziesiątkach widoków.

Z punktu widzenia developerów odzyskujemy cenny czas, który możemy przeznaczyć na techniki inżynierii jakości, które przedstawialiśmy w drugim module.

  1. Tworzy spójność wizualną między produktami, kanałami i (potencjalnie silosowymi) działami.

Zwłaszcza gdy zespoły pracują w silosach, gdzie każdy produkt lub kanał działa niezależnie od innych, brak design systemu obejmującego całą organizację może prowadzić do niespójnego wyglądu wizualnego i doświadczeń, które wydają się fragmentaryczne lub niezwiązane z marką. Design systemu zapewniają jedno źródło komponentów, wzorców i stylów oraz ujednolicają rozłączne doświadczenia, tak aby były spójne wizualnie i sprawiały wrażenie części tego samego ekosystemu. Dodatkową zaletą jest to, że wszelkie większe wizualne rebrandy lub przeprojektowania mogą być zarządzane na dużą skalę za pośrednictwem design systemu (a dokładniej design tokenów).

  1. Może służyć jako narzędzie edukacyjne i punkt odniesienia dla młodszych designerów i developerów.

Wyraźnie napisane wytyczne dotyczące użytkowania i przewodniki po stylach pomagają we wdrożeniu nowych współpracowników, zwłaszcza jeżeli mają małe doświadczenie komercyjne.
