Code review z asystentem AI

Do zrealizowania pół-automatycznego (bo wciąż wymagającego opieki programisty) Code Review, będą nam potrzebne co najmniej trzy elementy:

1) Nowy scenariusz GitHub Actions

2) Input do modelu, czyli zmiana w kodzie, której dotyczy Pull Request

3) Integracja z platformą OpenAI (dla modelu GPT-4o) / Anthropic (dla modelu Claude 3)

Zacznijmy od tego, co już znamy, a mianowicie od utworzenia nowego scenariusza GitHub Actions.

Jak zareagować na nowy Pull Request?

W trzecim module dowiedziałeś się, że dzięki triggerom GitHub Actions można realizować scenariusze automatyzacji bazujące na zdarzeniach w obrębie repozytorium.

Do przeprowadzenia Code Review będziemy potrzebować scenariusza, w trakcie którego model językowy przeanalizuje zmiany zgłaszane do repozytorium. Możemy to zrobić z wykorzystaniem poznanego wcześniej triggera pull_request, rozbudowanego o nowe kryterium - dodanie labelki “AI_CODE_REVIEW”.

W wybranym repozytorium, w folderze .github/workflows/code_review.yml zaczynamy pracę nad nowym scenariuszem:

name: PR Code Review

on:
  pull_request:
    types: [opened, synchronize, reopened, labeled]
    branches: [master]

Scenariusz będzie reagował na cztery aktywności w obrębie PRa - otwarcie, synchronizacja z branchem docelowym, ponowne otwarcie oraz dodanie nowej labelki (w razie, gdyby labelka AI_CODE_REVIEW nie była dostępna na starcie).

Dodajmy teraz pierwszy job, który będzie uzależniony od naszej labelki:

jobs:
  code-review:
    runs-on: ubuntu-latest
    if: contains(github.event.pull_request.labels.*.name, 'AI_CODE_REVIEW')
    steps:
      - name: Greeting
        run: echo "Hello world!"

Co się zmieniło?

Skąd pobrać informacje o zmianie wprowadzanej na poziomie Pull Requesta? Z punktu widzenia systemu kontroli wersji, Pull Request to nic innego jak różnica pomiędzy dwoma branchami w ramach tego samego repozytorium.

Polecenie git diff pokaże nam różnicę pomiędzy branchami, a konkretniej dwoma commitami na różnych branchach. Dla nas, danymi wejściowymi będzie ostatni commit na masterze (referencja) oraz ostatni commit na poziomie Pull Requesta.

git diff [commit_sha_1] [commit_sha_2]

Skąd te dane uzyskać? Lokalnie możemy je podejrzeć wykonując polecenie `git log` - zaznaczone na żółto:

Na poziomie automatyzacji te same wartości daje nam kontekst ${{ github }} i właściwości pull_requesta.

Zmodyfikujmy nasz scenariusz żeby sprawdzić, czy możemy zmiany wprowadzane przez PR wyświetlać w logach danej automatyzacji. Potrzebujemy do tego dwóch kroków - pobrania samego repozytorium, oraz właśnie polecenia git diff:

jobs:
  code-review:
    runs-on: ubuntu-latest
    if: contains(github.event.pull_request.labels.*.name, 'AI_CODE_REVIEW')
    steps:

      - name: Checkout repository
        uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: Print git diff
        run: git diff ${{ github.event.pull_request.base.sha }} ${{ github.event.pull_request.head.sha }}

Jak efekty?

Jest dobrze! Mamy automatyzację, która reaguje na nowe Pull Requesty, a także dostęp do zmian, które powinny być poddane ocenie. Czas na krok trzeci, czyli integrację z modelem językowym.

Dokumentacja projektu ze wsparciem AI

Kolejny ważny aspekt zespołowych procesów dużej skali, to rozwijanie i utrzymywanie bazy wiedzy o projekcie.

Każda firma tworzy takie bazy wiedzy w nieco inny sposób, ale główne założenie jest takie samo - chcemy utrwalać i centralizować ważne informacje, wnioski i decyzje, które pojawiły się w trakcie trwania danej inicjatywy. Nie chcemy, aby ich pozyskanie zależało od tego, czy Franek, nasz architekt, jest dzisiaj w pracy, albo czy Basia, lead developerka, pojechała akurat na urlop.

W zależności od tego, kto taką bazę tworzy i do kogo jest skierowana, może ona zawierać nieco inne informacje. Niezależnie od formy, pomaga ona m.in. w:

  • łatwiejszym wdrażaniu nowych programistów do projektu

  • łatwiejszym wdrażaniu nowych pracowników do firmy

  • szybkim pozyskiwaniu informacji na temat aktualnego stanu projektu

  • pozyskiwaniu informacji asynchronicznie, bez rozpraszania konkretnych osób

Niestety, teoretyczne założenia nie zawsze przekładają się na praktykę. Bazy wiedzy bywają aktualizowane z różną częstotliwością, nie zawsze zawierają aktualne informacje, a ich format i kształt może być niespójny.

W kolejnej części tej lekcji sprawdzimy, czy model językowy może nam w tej kwestii pomóc.

Projektowa dokumentacja

Po zbudowaniu pół-automatycznego procesu Code Review sprawdźmy, jak modele językowe OpenAI mogą budować dla nas dokumentację projektu, a także odpowiadać na pytania z nią związane.

Wykorzystamy do tego tzw. Assistants API oraz Vector Stores dostępne na platformie OpenAI.

  • Assistants API - To funkcjonalność, dzięki której zbudujemy nasze osobiste “instancje” modeli językowych, które pamiętają poprzednio prowadzone rozmowy w formie wątków, a także uzyskują dostęp do przekazanych przez nas plików. Asystenci są odpowiednikami Custom GPTs tworzonych z poziomu ChataGPT.

  • Vector Stores - Wektorowe bazy danych to jeden ze sposobów na przechowywanie treści w takim formacie, który może być odpowiednio interpretowany przez model językowy. W tego typu bazie przechowamy naszą dokumentację i będziemy mogli o niej porozmawiać z GPT-4.

W przypadku OpenAI, asystenci i bazy wektorowe są ze sobą mocno powiązane. Oba elementy łączą się ze sobą, dzięki czemu z łatwością przeprowadzimy z takim asystentem konwersację na temat wcześniej wrzuconych na platformę plików i danych.

W naszym przypadku inputem do konwersacji będzie dokumentacja projektu, którą najpierw przygotuje dla nas… nic innego jak model językowy.

Czy to się opłaca?

Całościowy koszt integracji będzie składał się z trzech części - kosztu generowania samej dokumentacji, kosztu przechowywania danych wektorowych a także kosztu komunikacji z modelami w momencie wysyłania zapytań o projekt

W zależności od potrzeb i tego, jak precyzyjna musi być twoja dokumentacja, możesz korzystać z modeli językowych różnej “generacji” i “rozmiaru”. Model od OpenAI o największych możliwościach, czyli GPT-4o, na dzisiaj wyceniany jest na 5$ za każdy milion tokenów na wejściu, oraz 15$ za milion tokenów na wyjściu.

Przyjmuje się, że w języku angielskim token to ok. ¾ słowa. Więcej o przeliczaniu tokenów na słowa możesz przeczytać w tym miejscu.
