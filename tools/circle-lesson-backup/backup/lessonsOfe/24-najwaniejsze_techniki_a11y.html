Obsługa klawiatury i focus management

Oto lista klawiszy i skrótów wspieranych przez przeglądarki:

Kluczowa część obsługi klawiatury po stronie frontend developera koncentruje się na nawigacji i zarządzaniu focusem. Focus odnosi się do tego, który element na ekranie aktualnie otrzymuje dane wejściowe z klawiatury.

Największym wrogiem dostępności w przypadku zarządzania focusem jest taki css:

*:focus {
  outline: none;
}

W ten sposób wszystkie elementy utracą domyślny kontur nadawany przez przeglądarki elementom z focus state. Jeżeli domyślne stylowanie nie pokrywa się z stylistyką Twojego projektu, to nadpisz domyślne style outline, pamiętaj tylko o wybraniu koloru, która zapewni odpowiedni kontrast (temat opisany w kolejnej sekcji).

Wszystkie elementy interaktywne muszą być dostępne z klawiatury (wyjątek: elementy w stanie disabled). Interaktywne elementy HTML takie jak button, linki czy kontrolki formularzy są automatycznie focusowalne.

Jeżeli masz element, który powinien być focusowalny z klawiatury, a normalnie nie otrzymuje focus state (np. customowy komponent oparty o divy), powinieneś dodać do niego dodać atrybut tabindex="0".

Ważne: Jeżeli element, któremu nadasz tabindex="0" ma na poziomie JS dodany click handler, to musisz pamiętać również o obsłudze eventów keydown/keyup, aby wykryć kliknięcie spacji lub entera (zwykle można podpiąć łatwo rozszerzyć ten sam handler o obsługę klawiatury). W innym wypadku użytkownicy keyboard-only nie będą mogli wejść z tym elementem w interakcje. Oczywiście nie dotyczy to elementów <button> oraz <a>, które automatycznie współdzielą click handlery z klawiaturą.

W przypadku komponentów, które przez większość czasu pozostają ukryte i są wywoływane przez kod JS, np. modale oraz błędy walidacji, należy wykorzystać tabindex=”-1”. Dzięki temu w momencie ich wywołania, możesz im nadać im focus state za pomocą HTMLElement.focus().

Ważne numero uno: Unikaj używania tabindeksów z liczbami dodatnimi, aby zapobiec potencjalnym problemom z kolejnością focusu.

Wyjątek od numero uno: cookie bannery sponsorowane przez RODO. Powinny one otrzymywać focus jako pierwsze, aby umożliwić użytkownikom szybkie pozbycie się tego arcydzieła Unii Europejskiej (kącik brukselski w kursie musi być, pozdrawiam słuchaczy Opanuj.AI).

Ważne numero duo: Nie nadużywaj tabindeksów, nie ma potrzeby dodawania ich do każdego z nagłówków lub paragrafów - czytniki ekranowe mogą do nich nawigować bez wykorzystania tabindeksów, a może to być irytujące dla pozostałych użytkowników klawiatury.

Focus state podczas nawigacji

Focus musi być również zarządzany, gdy użytkownik przechodzi ze strony na stronę. Jest to szczególnie istotne w SPA, gdzie nawigacja nie powoduje odświeżania przeglądarki.

Podczas przechodzenia między stronami, jako zespół musimy zadecydować, gdzie przenieść focus state po załadowaniu strony. Najpopularniejsze rozwiązania to:

  • Umieszczenie focusu na głównym kontenerze z ogłoszeniem aria-live.

  • Umieszczenie focus na linku, który pozwala wrócić do głównej zawartości.

  • Przeniesienie focusu do nagłówka h1 nowej strony.

Focus trap w modalach

Zarządzanie focus trapem w modalach/oknach dialogowych to kluczowy aspekt tworzenia dostępnych interfejsów użytkownika, szczególnie dla osób korzystających z AT oraz dla użytkowników nawigujących za pomocą klawiatury. Focus trap zapewnia, że podczas otwarcia modalu, focus jest "uwięziony" wewnątrz modalu, co oznacza, że użytkownik nie może przejść z focusem poza modal, używając Tab lub Shift + Tab, dopóki modal nie zostanie zamknięty. Jest to ważne dla zachowania kontekstu interakcji i zapewnienia, że użytkownicy klawiatury mogą łatwo nawigować po modalu.

Implementacja trap focus za pomocą tabindex:

  1. Zapamiętaj element, który miał fokus przed otwarciem modalu: To pozwoli na łatwy powrót focusu do tego elementu po zamknięciu modalu.

let focusedElementBeforeModal;

document.getElementById('open-modal-btn').addEventListener('click', function() {
    focusedElementBeforeModal = document.activeElement;
    // Wyświetl modal
    document.getElementById('modal').style.display = 'block';
    // Ustaw focus na modalu (lub jego pierwszym elemencie)
    document.getElementById('modal').focus();
});

<div id="modal" tabindex="-1">
    <button id="close-modal-btn">Zamknij modal</button>
    <button>Kontynuuj naukę</button>
</div>

  1. Utwórz focus trap wewnątrz modalu: Kiedy użytkownik nawiguje po modalu, zablokuj przenoszenie focusu poza modal.

document.getElementById('modal').addEventListener('keydown', function(e) {
    var isTabPressed = e.key === 'Tab' || e.keyCode === 9;

    if (!isTabPressed) {
        return;
    }

    const focusableModalElements = this.querySelectorAll('button, [href], input, select, textarea, [tabindex]:not([tabindex="-1"])');
    const firstElement = focusableModalElements[0];
    const lastElement = focusableModalElements[focusableModalElements.length - 1];

    if (e.shiftKey) /* shift + tab */ {
        if (document.activeElement === firstElement) {
            lastElement.focus();
            e.preventDefault();
        }
    } else /* tab */ {
        if (document.activeElement === lastElement) {
            firstElement.focus();
            e.preventDefault();
        }
    }
});

  1. Przywróć focus po zamknięciu modalu: Gdy modal jest zamykany, przywróć fokus do elementu, który był aktywny przed jego otwarciem.

document.getElementById('close-modal-btn').addEventListener('click', function() {
    document.getElementById('modal').style.display = 'none';
    // Przywróć focus
    if (focusedElementBeforeModal) {
        focusedElementBeforeModal.focus();
    }
});


Skip link

Aby ułatwić nawigację klawiaturę, warto dodać do strony “skip link”. Prowadzi on bezpośrednio do treści <main>, i pozwala użytkownikom pominąć treści, które są powtarzana na każdej stronie (np. główna nawigacja).

Skip link należy umieścić na samym początku body. Budując aplikację SPA sprowadza się to do utworzenia komponentu <SkipLink /> i umieszczeniu go jako pierwszego dziecka w głównym komponecie aplikacji (zwyczajowo <App />)

<a href="#main">Skip to main content</a>

Link powinien być domyślnie ukryty i pojawiać się jedynie gdy otrzyma focus z klawiatury. 

.skip-link {
  white-space: nowrap;
  margin: 1em auto;
  top: 0;
  position: fixed;
  left: 50%;
  margin-left: -72px;
  opacity: 0;
}

.skip-link:focus {
  opacity: 1;
  background-color: white;
  padding: 0.5em;
  border: 1px solid black;
}

Jest to narzędzie przeznaczone dla użytkowników keyboard-only, czytniki ekranu mają wbudowane możliwości nawigowania po treści.

Kontrast kolorów

Kontrast kolorów odgrywa kluczową rolę w dostępności treści cyfrowych, ułatwiając czytelność tekstu i rozróżnialność elementów graficznych na stronie. Zapewnienie odpowiedniego kontrastu między kolorem tekstu a tłem jest niezbędne dla osób z zaburzeniami widzenia, takimi jak niewyraźne widzenie czy ślepotą barw. Odpowiedni kontrast kolorów jest kluczowy dla zwiększenia czytelności tekstu oraz ułatwienia nawigacji i zrozumienia interfejsu użytkownika.

Wytyczne WCAG 2.1 określają minimalne stosunki kontrastu:

  1. Zwykły tekst czyli poniżej 18px lub poniżej 14px dla tekstu pogrubionego wymaga stosunku kontrastu 4.5:1 (AA) lub 7:1 (AAA).

  2. Duży tekst czyli 18px i większy lub 14px i większy dla tekstu pogrubionego wymaga stosunku kontrastu 3:1 (AA) lub 4.5:1 (AAA).

Istnieje wiele narzędzi online i wtyczek do przeglądarek, które pozwalają na sprawdzanie i dostosowywanie kontrastu kolorów, np. Who Can Use (narzędzie online) lub Wave Evaluation Tool (wtyczka).

O kontrast kolorów należy zadbać na wczesnych etapach projektu, najlepiej przed implementacją. Warto korzystać z palet kolorów zaprojektowanych z myślą o dostępności, które już na wczesnym etapie uwzględniają wymagania dotyczące kontrastu. Zmiana kolorów tła, zwłaszcza jeżeli nie wykorzystujemy CSS variables, będzie kosztowna i żmudna.

Jeżeli masz problem z kontrastem kolorów pomiędzy białym napisem a kolorowym tłem, warto zastosować tę samą barwę dla napisu i tła, czyniąc tekst ciemniejszym a tło jaśniejszym:

Typografia

Nie ma jednoznacznych danych jaki typ czcionek jest bardziej dostępny: szeryfowe czy bezszeryfowe. Wiadomo za to, że zgodnie z wytycznymi dotyczącymi dostępności treści internetowych (WCAG), rozmiar czcionki 16px jest najlepszy dla tekstu głównego.

Co ważne, musimy pamiętać o osobach, które mają ustawiony większy rozmiar tekstu w ustawieniach przeglądarki bądź korzystają z funkcji zoom. Z tego powodu powinnismy do tekstu wykorzystywać jednostki rem zamiast px, ponieważ reagują one na konfiguracje tych użytkowników (więcej o jednostkach w CSS).

1rem to 16px dla domyślnej wielkości tekstu.

Są dwa znane mi sposoby ułatwienia sobie pracy z remami, które są mniej naturalną jednostką dla programistów niż piksele.

  1. Wykorzystywanie funkcji calc() aby przeliczać remy operując na wielkościach pikselowych. Minusem tego rozwiązania jest rozwlekłość i konieczność pamiętania tej formuły za każdym razem, gdy korzystamy z font-size.

p {
  /* Daje 1,125rem. Odpowiednik 18px */
  font-size: calc(18rem / 16);
}
h3 {
  /* Daje 1.3125rem. Odpowiednik 21px */
  font-size: calc(21rem / 16);
}
h2 {
  /* Daje 1.5rem. Odpowiednik 24px */
  font-size: calc(24rem / 16);
}
h1 {
  /* Daje 2rem. Odpowiednik 32px */
  font-size: calc(32rem / 16);
}

  1. Zdefiniowanie CSS variables, które enkapsulują poszczególne wartości rem i operowanie na tych zmiennych.

html {
  --font-size-xs: 0.75rem;
  --font-size-sm: 0.875rem;
  --font-size-base: 1rem;
  --font-size-lg: 1.125rem;
  --font-size-xl: 1.3125rem;
  --font-size-2xl: 1.5rem;
  --font-size-3xl: 2.652rem;
  --font-size-4xl: 4rem;
}

p {
  font-size: var(--font-size-md);
}

Ja osobiście polecam metodę numer dwa lub korzystanie z TailwindCSS, który ogarnie temat za Ciebie.

Dostępne formularze

Aby zapewnić pełną dostępność formularzy musimy wziąć pod uwagę kilka istotnych zagadnień.

Powiązanie labelki z kontrolką

Jednym z najważniejszych elementów dostępnych formularzy jest ustanowienie jasnego i dokładnego połączenia między polem formularza (aka kontrolką) a jego labelką. Jest to istotne z punktu widzenia następujących niepełnosprawności:

  • Niepełnosprawność ruchowa - ułatwia tej grupie użytkowników nadanie stanu focus na określonym polu formularza, dając im szerszą powierzchnię do kliknięcia (mogą kliknąć również w labelkę). Jest to również istotne dla wszystkich użytkowników urządzeń mobilnych.

  • Upośledzenia wzroku - ułatwia tej grupie wypełnienie formularza, ponieważ czytniki ekranu ogłoszą labelkę za każdym razem, gdy kontrolka otrzyma focus.

Poprawne labelki informują użytkownika o celu pola, o tym czy pole jest wymagane, a także mogą dostarczać informacji o wymaganiach pola, takich jak format danych wejściowych. Etykiety muszą być widoczne przez cały czas i dokładnie opisywać pole formularza dla użytkowników.

Poprawne sposoby łączenia labelki z kontrolką

  1. Atrybut for=”control-id” - wskazujemy ID kontrolki, która jest opisywana przez labelkę (wiązanie jawne)

<label for="username">Username</label>
<input name="username" id="username"></input>

PS. Zamykam tutaj <input> bo inaczej podświetlanie składnie się sypie 🙈

  1. Zagnieżdżenie kontrolki wewnątrz labelki (wiązanie niejawne)

<label>
  Username
  <input name="username"></input>
</label>

Poza labelką możemy poszerzyć opis kontrolki za pomocą pól tekstowych. Opisy pól nie są wymagane w celu zapewnienia dostępności, jeśli etykiety lub instrukcje formularza są wystarczająco opisowe.

Istnieją jednak sytuacje, w których dodanie dodatkowych informacji jest przydatne w celu uniknięcia błędów w formularzu, takich jak przekazywanie informacji o minimalnej długości dla hasła lub informowanie użytkownika, jakiego formatu kalendarza należy użyć (np. MM-DD-RRRR).

<label for="password">
 Password
</label>
<span id="password-reqs">Minimum 8 characters, including: 1 uppercase, 1 special</span>
<input type="password" name="password" id="password" aria-describedby="password-reqs"></input>

Nie polegaj na placeholderach

Wbudowane placeholdery mają istotny problem: mają bardzo niski kontrast a jeżeli go podwyższymy za pomocą selektora :placeholder to kontrolka wygląda jakby była już wypełniona.

Stąd z punktu widzenia dostępności placeholdery można uznać za nieistniejące. Należy zapewnić niezbędny kontekst za pomocą labelek i dodatkowych instrukcji z wykorzystaniem aria-describedby.

Kiedy możemy wizualnie ukrywać labelki?

Labelka zawsze jest nam potrzebna a ukrywanie jej powinno być wyjątkiem. Dobrym tego przykładem jest komponent wyszukiwarki, zagnieżdżony zwykle w głównej nawigacji, gdzie nie ma przestrzeni na labelkę. Wtedy możemy ją ukryć za pomocą klasy visually-hidden, dzięki czemu etykieta będzie widoczna dla AT.

<label for="main-search" class="visually-hidden">Search the site</label>
<input name="main-q" id="main-search" type="search"></input>
<button>Search</button>


Zezwalaj na autocomplete

Czasami spotyka się formularze, w których programiści wyłączają funkcję autouzupełniania formularza - najpewniej dlatego, że nie działa ona z niskiej jakości mechanizmami walidacji. Z punktu widzenia użytkownika to nieprzyjemne doświadczenie, ponieważ muszą ponownie wypełniać formularz, jeśli popełnią błąd. Co ważne, autouzupełnianie pomaga osobom z niepełnosprawnościami szybciej wypełniać formularze, ponieważ nie muszą one za każdym razem ręcznie wprowadzać swoich danych.

Autouzupełnianie jest domyślnie włączone - co warto dodatkowo zrobić, to pomóc przeglądarkom powiązać pola z zapamiętanymi wartościami:

<label for="job-title">Title: </label>
<input type="text" id="job-title" name="job-title" autocomplete="organization-title"></input>

Atrybut autocomplete pozwala przypisać wiele różnych wartości, warto doczytać na MDN.

Grupuj checkboxy i radio buttony za pomocą <fieldset> i <legend>

Najczęściej spotykamy się z grupowaniem kontrolek radio i checkbox za pomocą div’ów i tworzeniem labelek grupy za pomocą <p>/<span>:

<div class="form-group">
    <p class="form-group-label">Gender</p>
    <label class="radio-inline">
        <input id="gender-male" name="gender" value="M" type="radio"> Male
    </label>
    <label class="radio-inline">
        <input id="gender-female" name="gender" value="F" type="radio"> Female
    </label>
</div>

Mamy jednak do dyspozycji semantyczne elementy fieldset i legend, które są przeznaczone do tego celu i zapewniają kompatybilność z AT:

<fieldset class="form-group">
    <legend class="form-group-label">Gender</legend>
    <label class="radio-inline">
        <input id="gender-male" name="gender" value="M" type="radio"> Male
    </label>
    <label class="radio-inline">
        <input id="gender-female" name="gender" value="F" type="radio"> Female
    </label>
</fieldset>

Obsługa błędów

Błędy zdarzają się podczas wypełniania formularzy przez użytkowników, więc musimy upewnić się, że wszelkie pojawiające się komunikaty o błędach są dostępne dla użytkowników z różnymi niepełnosprawnościami.

Gdy użytkownik napotka błąd formularza, pierwszym krokiem jest poinformowanie go o błędzie. Pole, w którym wystąpił błąd, musi być wyraźnie zidentyfikowane, a sam błąd musi być opisany w sposób, który pomoże użytkownikowi skorygować ten błąd.

Programiści, często ustawiają komunikaty o błędach w ten sposób:

<label for="username">Username</label>
<input type="text" id="username" name="username"></input>
<span>Error: Name is required and should have more than 5 characters</span>

Ten typ obsługi błędów jest niedostępny dla użytkowników niedowidzących, którzy polegają na czytnikach ekranu, ponieważ nie powiedzieliśmy AT, że błąd jest powiązany z kontrolką username. Oznacza to, że jeśli wystąpią jakiekolwiek błędy, użytkownik nie będzie wiedział, na czym polega problem.

Aby zapewnić dostępność komunikatów o błędzie, należy skorzystać z następujących atrybutów aria:

  • aria-describedby: w tym przypadku pomaga czytnikom ekranu zrozumieć, że komunikat o błędzie jest powiązany z danymi wejściowymi w kontrolce.

  • aria-live="polite": mówi technologii wspomagającej, aby ogłosiła aktualizację na stronie, przy czym aktualizacja polega na pojawieniu się komunikatu o błędzie dla pola formularza.

<label for="username">Username</label>
<input type="text" id="username" name="username" aria-describedby="username-validation"></input>
<span id="username-validation" aria-live="polite">
  Error: Name is required and should have more than 5 characters
</span>

Należy pamiętać, że span z komunikatem o błędzie powinno być obecne w DOM, ale domyślnie puste. Zapewni to, że technologie wspomagające ogłoszą komunikat o błędzie w momencie dodania treści błędu za pomocą textContent.

Komunikat błędu powinnien być jasno oznaczony, nie tylko za pomocą koloru czerwonego. Jeżeli zamiast słowa “Error” Twój projekt korzysta z ikon, pamiętaj o poprawnym wskazaniu znaczenia ikony.

Istnieją dwie dostępne metody wyświetlania komunikatów o błędach, takie jak:

  • Inline w pobliżu miejsca wystąpienia błędu

  • Zbiór błędów zgrupowanych w jednym większym komunikacie u góry strony. Dlaczego na górze a nie na dole? Ponieważ ułatwia to ponowną nawigację przez formularz w celu poprawy błędów, zwłaszcza jeżeli połączymy komunikaty błędów z linkami do kontrolek.

Z punktu widzenia dostępności najlepiej połączyć te dwie metody, chociaż komunikaty błędów inline wraz z przeniesieniem focusu na pierwszy z nich po submicie formularza jest wystarczający.

Oznaczaj pola wymagane nie tylko w sposób wizualny

Bardzo często spotyka się formularze, gdzie informacja o tym, że pole jest wymagane ogranicza się do asteriksa (*) obok labelki i zapiętej customowej walidacji w samym JS. 

<label for="username">Username (*)</label>
<input type="text" id="username" name="username"></input>

Z punktu widzenia dostępności nie jest to wystarczające.

W pierwszej kolejności należy skorzystać z atrybutu required na kontrolce w połączeniu z atrybutami client-side form validation.

Jeżeli tworzymy customowe kontrolki, należy skorzystać z aria-required=”true” i pamiętać o ręcznym zablokowaniu wysłania formularza, dopóki to pole nie zostanie poprawnie wypełnione, aby zachować spójność z tym jak na formularze wpływa atrybut required.

Warto również rozważyć rezygnację z asteriksa (*) na rzecz słownego oznaczenia (required) lub (wymagane), zapewni to lepszy UX dla użytkowników AT.

<label for="username">Username (required)</label>
<input type="text" id="username" name="username" required minlength="5"></input>

Wykorzystuj dostępne typy inputów

Często kusi nas tworzenie customowych inputów operujących na type=”text”, ale pamiętajmy że mamy do dyspozycji ponad 20 typów, które zdejmują z nas ciężar samodzielnego dbania o dostępność za pomocą atrybutów ARIA oraz zapewnienie topowego wsparcia dla urządzeń mobilnych.

Warto dobrze przeanalizować rozwiązania dostępne w HTML, zanim postanowimy wynajdywać koło na nowo.

Dostępne customowe komponenty

Nie da się ukryć, że to temat rzeka, o którym można zrealizować osobny kurs. HTML daje nam dużo możliwości, ale jest wiele komponentów, które musimy budować samodzielnie w oparciu o jego fundamenty.

Świetnym źródłem wiedzy jak robić to w sposób dostępny i zgodny ze standardami jest biblioteka wzorców komponentów W3C.

Każdy z komponentów jest tam kompleksowo opisany:

  • jaką rolę spełnia w interfejsie użytkownika

  • jak zapewnić pełne wsparcie dla klawiatury

  • z jakich atrybutów ARIA należy skorzystać

  • jak wygląda dostępny przykład takiego komponentu

Zresztą zobacz sam na przykładzie komponentu Accordion.

Testowanie dostępności

Aby zweryfikować obecny stan dostępności naszego projektu, pracować nad jego poprawą oraz na bieżąco weryfikować nowe komponenty, należy korzystać z metod testowania automatycznego oraz manualnego.

Testy automatyczne

Istnieje szereg rozszerzeń do przeglądarki oraz narzędzi, które możemy zintegrować z naszym workflow, aby wyłapać podstawowe błędy dostępności:

  1. Rozszerzenia do przeglądarki:

    1. Szybkie testy: Lighthouse Accessibility Audit

    2. Rozszerzone testy: WAVE

  2. Rozszerzenia do testów jednostkowych i e2e: axe-core, które zobaczysz w akcji w ramach lekcji “AI Edition” do tego modułu.

Testy manualne

Poniżej znajdziesz kluczowe informacje o tym jak przeprowadzać manualne testy dostępności za pomocą czytników ekranowych i klawiatury.

Czytniki ekranowe

Najważniejsze skróty klawiszowe przy obsłudze czytników ekranowych:

Najważniejsze kroki podczas testów z czytnikami:

  • Sprawdź, czy dokument HTML ma atrybut języka, aby czytniki ekranu odczytywały go z prawidłowym akcentem i wymową. Na przykład: <html lang="en">.

  • Upewnić się, że możesz nawigować po stronie zgodnie z jej strukturą. Użyj funkcji Element Rotor do przeglądania stron. Rotor wyświetla listę typowych elementów, takich jak nagłówki, linki i obrazy, i pozwala nawigować bezpośrednio do wybranego elementu.

  • Upewnij się, że wszystkie linki i obrazy są odpowiednio opisane.

  • Jeśli strona zawiera formularz, upewnij się, że czytnik ekranu odczytuje labelki i komunikaty błędów.

Obsługa klawiatury

Najważniejsze kroki podczas testów z czytnikami:

  • Użyj klawiatury, aby poruszać się po stronie za pomocą klawisza Tab.

  • Upewnij się, że możesz dotrzeć do wszystkich interaktywnych elementów i uruchomić je za pomocą spacji, klawisza Enter lub klawiszy strzałek.

  • Sprawdź, czy fokus jest zawsze widoczny i pojawia się w logicznej kolejności.

  • Upewnij się, że żadna zawartość nie skupia się poza ekranem lub nie jest ukryta.

  • Sprawdź, czy strona zawiera link pomijający nawigację (jeśli nawigacja znajduje się przed <main>). Pozwoli to użytkownikom pominąć nawigację, aby dotrzeć do głównej treści strony.

Kiedy accessibility jest priorytetem?

W idealnym świecie zawsze mielibyśmy czas na pokrycie 100% naszego kodu testami, zadbanie o topową wydajność oraz zgodność naszych interfejsów z kryteriami AAA WCAG 2.2.

Niestety nie żyjemy w idealnym świecie, mamy ograniczone zasoby i ogrom zadań do zrealizowania. Stąd jeżeli budujemy projekt do wewnętrznego użytku w firmie możemy sobie pozwolić na więcej kompromisów niż w przypadku aplikacji internetowych dla klienta z branży publicznej, gdzie mamy już określone wymogi prawne Unii Europejskiej.

Są one określone w dyrektywie (UE) 2016/2102 Parlamentu Europejskiego i Rady z dnia 26 października 2016 r. w sprawie dostępności stron internetowych i aplikacji mobilnych organów sektora publicznego. Ta dyrektywa ma na celu zapewnienie, aby osoby z niepełnosprawnościami miały lepszy dostęp do aplikacji internetowych i mobilnych oferowanych przez organy sektora publicznego.

Główne wymogi dyrektywy dotyczą:

  1. Zgodności ze standardami dostępności: Strony internetowe i aplikacje mobilne muszą być zgodne ze wspólnymi standardami dostępności WCAG (Web Content Accessibility Guidelines) 2.1 na poziomie AA.

  2. Oświadczenie o dostępności: Organizacje muszą udostępniać i regularnie aktualizować oświadczenie o dostępności, w którym opisują poziom zgodności swoich stron internetowych i aplikacji mobilnych ze standardami dostępności oraz znane problemy.

  3. Mechanizm informacji zwrotnej: Organizacje muszą zapewnić mechanizm, za pomocą którego użytkownicy mogą zgłaszać problemy z dostępnością i prosić o dostępne alternatywy dla treści niedostępnych.

  4. Monitoring i sprawozdawczość: Państwa członkowskie UE muszą wprowadzić system monitorowania i sprawozdawczości dotyczący zgodności stron internetowych i aplikacji mobilnych z wymogami dostępności.

Dyrektywa ta dotyczy stron internetowych i aplikacji mobilnych organów sektora publicznego, w tym administracji państwowej, samorządów terytorialnych, uniwersytetów, szkół, bibliotek publicznych, szpitali i innych instytucji publicznych. W 2019 roku zakres dyrektywy został rozszerzony, aby obejmować także inne organizacje świadczące usługi publiczne.

Choć dyrektywa ta skupia się na sektorze publicznym, wprowadza ona również standardy, które mogą być przykładem dobrej praktyki dla sektora prywatnego, szczególnie w obszarach, w których usługi są świadczone na szeroką skalę lub są istotne z punktu widzenia dostępu do informacji i usług przez osoby z niepełnosprawnościami.

Najczęstsze mity związane z dostępnością

Podobnie jak w przypadku innych dobrych praktyk, takich jak testowanie, o dostępności narosło wiele mitów i błędnych przekonań. Po zapoznaniu się z tą lekcją, będziesz gotowy na rozbrajanie tych mitów w swojej codziennej pracy:

  1. "Dostępność ogranicza design": Dostępność może być włączona do eleganckiego i innowacyjnego projektowania; nie ogranicza kreatywności, lecz wymaga przemyślenia najlepszych praktyk na wczesnym etapie designu komponentów.

  2. "Tylko niewielka liczba użytkowników potrzebuje dostępności": W rzeczywistości szeroki zakres użytkowników korzysta z funkcji dostępności, w tym osoby starsze i użytkownicy z tymczasowymi ograniczeniami.

  1. "Technologie wspomagające są tylko dla osób niepełnosprawnych": Technologie wspomagające mogą być używane przez każdego, kto z różnych przyczyn potrzebuje alternatywnych sposobów interakcji z treścią cyfrową, np. programiści korzystający z klawiatury jako power userzy.

  2. "Dostępność jest zbyt kosztowna lub trudna do wdrożenia": Wczesne włączanie dostępności do procesu designu może zmniejszyć koszty i ułatwić implementację. Koszta wdrożenia dostępności obniżają również design systemy.

  3. "Semantyczny HTML to tylko dla SEO": Chociaż semantyczny HTML poprawia SEO, jego głównym celem jest ułatwienie dostępu do treści.

  4. "Dodanie ARIA rozwiązuje wszystkie problemy z dostępnością": ARIA (Accessible Rich Internet Applications) może poprawić dostępność, ale nie zastąpi dobrej praktyki semantycznego HTML i powinna być używana tylko wtedy, gdy semantyczne znaczniki HTML są niewystarczające.

  5. "ARIA jest tylko dla ekspertów": Chociaż ARIA jako całość jest złożona, wiele jej aspektów jest stosunkowo proste do zrozumienia i zaimplementowania, a zasoby online oferują szeroki zakres poradników i najlepszych praktyk.

  6. "Dobry design nie może mieć wysokiego kontrastu": Można projektować estetycznie interfejsy z zachowaniem wysokiego kontrastu; odpowiedni dobór palety kolorów pozwala na połączenie estetyki z dostępnością. Jest wiele gotowych palet dostępnych w sieci.

  7. "Tylko osoby z niepełnosprawnościami wzrokowymi potrzebują wysokiego kontrastu": wysoki kontrast poprawia czytelność dla wszystkich użytkowników, w tym dla osób w jasnych środowiskach lub na urządzeniach z ograniczoną jakością wyświetlacza.

  8. "Kontrast jest ważny tylko dla tekstu": wysoki kontrast jest równie ważny dla elementów interaktywnych, takich jak przyciski i linki, aby ułatwić użytkownikom nawigację po stronie.

  9. "Testowanie dostępności jest tylko dla specjalistów": w rzeczywistości, podstawowe testy dostępności mogą być przeprowadzone przez każdego dewelopera, a wiele narzędzi ułatwia ten proces.

  10. "Automatyczne narzędzia do testowania dostępności są wystarczające": chociaż narzędzia te są użyteczne, wykryją do 50% problemów z dostępnością. Testowanie manualne i testowanie z użytkownikami są niezbędne.

    1. Wykonaj testy interakcji z komponentem za pomocą klawiatury

    2. Dodaj komponent do prostego formularza z kontrolkami: imię, nazwisko, email i zgodą na przetwarzanie danych osobowych (switch). Wszystkie pola powinny być wymagane:

      1. Imię dłuższe niż 1 znak, bez znaków specjalnych

      2. Nazwisko dłuższe niż 1 znak, bez znaków specjalnych

      3. Email zgodny z formatem

      4. Switch musi być zaznaczony

    3. Dodaj walidację błędów do formularza.

    4. Przetestuj dostępność formularza za pomocą czytnika ekranowego (NVDA dla Windows, VoiceOver dla MacOS).
