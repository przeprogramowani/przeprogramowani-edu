TypeScript oferuje bogaty zestaw narzędzi do zarządzania typami, takich jak adnotacje zmiennych, asercje typu czy operator satisfies. Pozwalają one na precyzyjne kontrolowanie typów w złożonych aplikacjach. W tej lekcji przyjrzymy się każdemu z tych mechanizmów, ilustrując ich zastosowanie zupełnie nowymi przykładami, aby w pełni zrozumieć ich działanie.

Adnotacje zmiennych i wartości

Adnotacje w TypeScript pomagają określić, jakie typy danych mogą być przypisane do zmiennych. Dzięki temu kompilator może skutecznie wykrywać błędy jeszcze przed uruchomieniem aplikacji.

Typowanie zmiennych w praktyce

Kiedy przypisujemy adnotację do zmiennej, określamy jej oczekiwany typ. W praktyce oznacza to, że TypeScript weryfikuje poprawność przypisywanej wartości, ale także odrzuca szczegóły jej typu literalnego.

type RGB = { r: number; g: number; b: number };
type Color = string | RGB;

const palette: Record<string, Color> = {
  primary: { r: 255, g: 0, b: 0 },
  secondary: "blue",
};

// Poniższa operacja zgłasza błąd, ponieważ TypeScript "zapomniał" szczegóły o RGB.
console.log(palette.primary.r); // Error: 'r' nie istnieje w typie 'Color'.

W tym przypadku, mimo że palette.primary to obiekt typu RGB, TypeScript widzi go tylko jako Color, co wymaga jawnego zawężenia typu.

console.log((palette.primary as RGB).r);

Typowanie wartości

Jeżeli zamiast korzystać z adnotacji typu, wykorzystamy mechanizm inferencji, TypeScript zachowa pełną wiedzę o strukturze obiektu:

const palette = {
  primary: { r: 255, g: 0, b: 0 },
  secondary: "blue",
}; // TypeScript w pełni zna strukturę 'RGB'.

console.log(palette.primary.r); // Działa poprawnie.

Jednakże w przypadku dynamicznych kluczy lub bardziej złożonych typów, ta technika prowadzi do spadku czytelności i bezpieczeństwa kodu, stąd zalecamy korzystanie z operatora satisfies.

Operator satisfies

Operator satisfies pozwala na sprawdzenie, czy dana wartość spełnia określone kryteria typu, jednocześnie zachowując precyzyjne informacje o tym obiekcie. Dzięki temu TypeScript może dokładniej inferować typy, co prowadzi do lepszej autouzupełniania i wczesnego wykrywania błędów:

type RGB = {r: number; g: number; b: number};
type Color = string | RGB;

const palette = {
  primary: {r: 255, g: 0, b: 0},
  secondary: "blue",
} satisfies Record<string, Color>;

console.log(palette.primary.r); // Brak błędu

Zapewnienie zgodności podczas rozwoju aplikacji

Kiedy rozwijasz aplikację, możesz chcieć, aby nowe właściwości w obiektach nie naruszały istniejących typów. Operator satisfies pomaga w utrzymaniu spójności typów:

type Settings = {
  theme: string;
  notifications: boolean;
};

const userSettings = {
  theme: "dark",
  notifications: true,
  language: "en", // Błąd: Property 'language' does not exist on type Settings
} satisfies Settings;

Kiedy używać satisfies?

Operator satisfies jest najbardziej efektywny w następujących przypadkach:

  1. Precyzyjna kontrola typów: Gdy chcesz upewnić się, że obiekt dokładnie spełnia wymagania typu bez rozszerzania go.

  2. Weryfikacja złożonych typów: Przy definiowaniu skomplikowanych struktur danych, gdzie ważne jest, aby obiekt spełniał określone kryteria.

  3. Unikanie problemów z autouzupełnianiem: Zapewnia lepsze wsparcie dla edytorów kodu, co pomaga w unikaniu błędów typów poprzez lepsze autouzupełnianie.

Asercje typu – praktyczne zastosowania

Asercje typu umożliwiają nadpisanie domyślnej inferencji TypeScript. Czasami jako programiści mamy więcej informacji o typie, niż kompilator może wnioskować.

as – wymuszanie typu

Asercje typu są często stosowane w sytuacjach, gdy interfejsy nie są jednoznaczne. Rozważmy przykład pracy z DOM:

const button = document.querySelector("#submit") as HTMLButtonElement;

// Gdyby element nie istniał, poniższa operacja spowoduje błąd runtime:
button.click();

Aby poprawić bezpieczeństwo kodu, asercję można połączyć z warunkiem:

const button = document.querySelector("#submit");

if (button instanceof HTMLButtonElement) {
  button.click(); // Teraz mamy pewność co do typu.
}

Non-null assertion (!)

Non-null assertion skraca kod w sytuacjach, gdy wartość może być niejawnie null lub undefined. Przykład:

const value = document.getElementById("app")!.textContent; // Zakładamy, że element istnieje.

Jednakże lepszym podejściem byłoby wcześniejsze sprawdzenie wartości:

const app = document.getElementById("app");

if (app) {
  console.log(app.textContent);
}
