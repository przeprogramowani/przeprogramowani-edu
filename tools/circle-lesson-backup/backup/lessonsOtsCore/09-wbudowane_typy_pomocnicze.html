Wprowadzenie

W ramach swojej biblioteki standardowej, TypeScript oferuje zestaw typów wbudowanych, które oszczędzają czas programisty i pozwalają dynamicznie sterować strukturą typów obiektów. Typy wbudowane pozwalają na bardziej deklaratywne opisywanie zachowania typu, z czego warto korzystać aby utrzymać kod w spójnym kształcie.

W tej lekcji przedstawimy ci kilka z takich typów, a w kolejnych lekcjach nauczymy cię, jak budować je od zera z wykorzystaniem operatorów i zaawansowanej algebry typów TypeScript.

Kolejne spotkanie z typami generycznymi

W tej lekcji opieramy się o poznane wcześniej typy generyczne. Dla przypomnienia - o ich działaniu możesz myśleć jak o “funkcjach w świecie typów”:

// W świecie wartości - wykonaj operację z pomocą argumentu typu T
function foo(param: T) { ... }

// W świecie typów - zbuduj nową strukturę w oparciu o typ T
type Foo<T> = ...

Poniżesz zobaczysz nowe scenariusze, w których generyki wykazują swój potencjał i pozwalają budować prawdziwie reużywalne i skalowalne struktury.

Readonly<T> i ReadonlyArray<T>

Dzięki słowu kluczowemu readonly, Typescript umożliwia tworzenie obiektów, których pola po inicjalizacji służą tylko do odczytu:

type PersonalDetails = {
  readonly name: string;
};

const person: PersonalDetails = {
  name: "John Doe",
};

person.name = "Jane Doe";
  // ^ - Cannot assign to 'name' because it is a read-only property.ts(2540

Czasami zdarza się, że tego typu konstrukcję chcemy utworzyć w locie, bo nie mamy dostępu do oryginalnej definicji typu (lub nie chcemy jej modyfikować).

Dzięki wbudowanemu typowi Readonly<T>, który stosujemy na istniejącym typie T możesz sprawić, że wszystkie jego właściwości będą tylko do odczytu (ang. readonly). Dzięki temu obiekt nie może być modyfikowany po jego utworzeniu, co zwiększa bezpieczeństwo i przewidywalność kodu:

type PersonalDetails = {
  name: string;
};

const person: Readonly<PersonalDetails> = {
  name: "John Doe",
};

person.name = "Jane Doe";
// ^ - Cannot assign to 'name' because it is a read-only property.ts(2540

Readonly jest przydatny, gdy chcemy upewnić się, że obiekty nie zostaną zmodyfikowane po ich utworzeniu. To szczególnie użyteczne w przypadku danych, które powinny być traktowane jako immutable, np. ustawienia konfigurowane tylko raz, przy starcie aplikacji.

TypeScript daje nam również odpowiednik tej funkcjonalności dla tablic:

let readonlyNumbers: ReadonlyArray<number> = [1, 2, 3, 4, 5];
// readonlyNumbers.push(6); // Błąd: metoda push nie jest dostępna

Partial<T>

Czasami zdarza się, że dwa typy muszą mieć podobną strukturę, ale jeden z nich procesujemy tylko we fragmentach. Przykładowo, kiedy aktualizujemy istniejące dane w systemie i udostępniamy użytkownikowi formularz, nie mamy gwarancji, jakie elementy zostaną wypełnione.

Bez typu wbudowanego taką relację można osiągnąć poprzez znak zapytania dodawany do nazwy pola, co zmienia go w pole opcjonalne:

type PersonalDetails = {
  name: string;
  age: number;
  email: string;
  phone: string;
};

const person: PersonalDetails = {
  name: "John Doe",
  age: 20,
  email: "john.doe@example.com",
  phone: "1234567890",
};

type PersonalDetailsUpdatePayload = {
  name?: string;
  age?: number;
  email?: string;
  phone?: string;
};

const payload: PersonalDetailsUpdatePayload = {
  name: "John Doe",
};

const updatedPerson: PersonalDetails = {...person, ...payload};

Zamiast tworzenia kopii obiektu, którą dodatkowo będziemy musieli synchronizować ręcznie, możemy wykorzystać typ Partial<T>:

const person: PersonalDetails = {
  name: "John Doe",
  age: 20,
  email: "john.doe@example.com",
  phone: "1234567890",
};

type PersonalDetailsUpdatePayload = Partial<PersonalDetails>;

Dzięki temu uzyskujemy typ o podobnym kształcie jak typ bazowy, którego wszystkie pola stają się opcjonalne.

Required<T>

Typ Required<T> działa odwrotnie do Partial<T>, wymuszając, aby wszystkie właściwości typu T były wymagane. Jest przydatny, gdy chcemy mieć pewność, że wszystkie pola obiektu będą zdefiniowane.

type User = {
  id: number;
  name?: string;
  email?: string;
  address?: string;
};

// Funkcja do aktualizacji częściowych danych użytkownika
function updateUser(data: Partial<User>) {
  // ...logika aktualizacji formularza, gdzie tylko część pól jest wymagana
}

// Funkcja do zapisywania pełnych danych użytkownika w bazie
function createUser(user: Required<User>) {
  // Możemy założyć, że wszystkie właściwości (name, email, address) są tutaj obecne
  console.log(`Saving user: ${user.name}, ${user.email}, ${user.address}`);
}

Required pomaga uniknąć sytuacji, w których obiekt nie ma zdefiniowanych wszystkich potrzebnych właściwości. Może być używany do doprecyzowania typów i zmniejszenia ryzyka błędów wynikających z brakujących wartości. W przypadku zapisywania danych w bazie, wymusza dodatkowy krok związany z zapewnieniem wartości domyślnych dla brakujących pól.

Record<K, V>

Typ Record<K, V> pozwala na tworzenie obiektów, gdzie klucze są typu K, a wartości typu V. Jest często używany do definiowania struktur, które pełnią rolę map lub obiektów przechowujących wartości przypisane do konkretnych kluczy.

Taki typ można wykorzystać chociażby do przedstawienia uprawnień użytkownika względem poszczególnych sekcji na stronie:

type AppSections = "home" | "settings" | "profile" | "search";
type SectionVisibility = boolean;

type NavigationPermissions = Record<AppSections, SectionVisibility>;

const permissions: NavigationPermissions = {
  home: true,
  settings: false,
  profile: true,
  search: true,
};

Co ważne, typ Record połączony z literałami (jak powyżej) dba o to, żeby kluczami obiektu były wszystkie dostępne wartości, a nie tylko kilka z nich:

const permissions: NavigationPermissions = {
  home: true,
  settings: false,
};
// ^ - Type '{ home: true; settings: false; }' is missing the following properties from type 'NavigationPermissions': profile, search

Co ważne, pierwszym z parametrów rekordu musi być typ prosty - string, number lub symbol (lub literały):

type User = {
  name: string;
  email: string;
  age: number;
};

type Permission = boolean;

type NavigationPermissions = Record<User, Permission>;
                                 // ^ - Type 'User' does not satisfy the 
                                 //     constraint 'string | number | symbol'.ts(2344)

👉 Dla ciekawskich: TypeScript zawiera wbudowany typ PropertyKey, który jest aliasem na dopuszczalne typy kluczy obiektów i rekordów. Dzięki temu nie musimy ich budować ręcznie:

type Permission = boolean;
// type PropertyKey = string | number | symbol

type NavigationPermissions = Record<PropertyKey, Permission>;

Chociaż Record ma swoje ograniczenia, to wciąż jest to świetne rozwiązanie do typowania obiektów, które przechowują dane zależne od wartości klucza, np. uprawnienia użytkowników lub ustawienia aplikacji, a do tego wszystkie klucze powinny być wzięte pod uwagę.

Pick<T, K>

Typ Pick<T, K> tworzy nowy typ zawierający tylko wybrane właściwości z istniejącego typu T. To szczególnie przydatne, gdy pracujemy z dużymi obiektami, a interesuje nas tylko część ich właściwości.

type User = {
  id: number;
  name: string;
  email: string;
  gender: "male" | "female";
  age: number;
  isActive: boolean;
  role: "admin" | "user";
  externalId: string;
};

type MetadataProps = "id" | "name";
type UserMetadata = Pick<User, MetadataProps>;

const preview: UserMetadata = {
  id: 1,
  name: "Alice",
};

Pick ułatwia tworzenie nowych typów, które są podzbiorem istniejących typów.

Przydaje się, gdy chcemy ograniczyć ilość danych przekazywanych do komponentów lub funkcji.

Omit<T, K>

Typ Omit<T, K> jest odwrotnością Pick – tworzy nowy typ na podstawie typu T, ale z pominięciem wybranych właściwości. To użyteczne, gdy chcemy pracować z typem bez określonych właściwości.

type User = {
  id: number;
  name: string;
  email: string;
  gender: "male" | "female";
  age: number;
  isActive: boolean;
  role: "admin" | "user";
  externalId: string;
};

type HiddenDetails = "id" | "age" | "gender" | "email";
type SafeUser = Omit<User, HiddenDetails>;

const user: SafeUser = {
  name: "John",
  isActive: true,
  role: "admin",
  externalId: "123",
};

Omit pomaga tworzyć wersje typów bez pewnych, wrażliwych właściwości, takich jak hasło.

Jest używany, gdy potrzebujemy typów do API lub UI, które nie muszą zawierać wszystkich danych danego obiektu.

Obiekty zagnieżdżone

Zarówno Pick<T, K> jak i Omit<T, K> działają jedynie na polach bezpośrednio znajdujących się na najwyższym poziomie w strukturze obiektu. Nie mają one możliwości modyfikacji pól zagnieżdżonych, czyli właściwości potomnych.

W poniższym przykładzie usunięcie zagnieżdżonego pola name nie jest możliwe z wykorzystaniem tych typów - zamiast tego powinniśmy zastosować dedykowany typ o mniejszym zakresie pól i przemapowanie odpowiednich wartości:

type UserWithRole = {
  id: number;
  role: {
    id: number;
    name: string;
  };
};

type AnonymousUser = Omit<UserWithRole, "name">;

const user: AnonymousUser = {
  id: 1,
  // ❌ Property 'name' is missing in type '{ id: number; }' but required in type '{ id: number; name: string; }'
  role: {
    id: 1
  },
};

Exclude<T, K>

Exclude pozwala na usuwanie określonych składowych z unii typów:

type UnionOfTypes = string | number | symbol | undefined | null;

type ExcludeNulls = Exclude<UnionOfTypes, undefined | null>; // string | number | symbol

NonNullable<T>

Przydatny typ pomocniczy, który z unii typów pozbywa się undefined oraz null. Można powiedzieć, że jest to szczególna wersja typu Exclude, która domyślnie usuwa warianty “null | undefined”.

type UnionOfTypes = string | number | symbol | undefined | null;

type NoNullableTypes = NonNullable<UnionOfTypes>; // string | number | symbol

Podsumowanie

W tej lekcji poznałeś przydatne typy wbudowane w język TypeScript, które, w zależności od potrzeb, mogą wpłynąć na tworzenie bardziej elastycznego lub spójnego kodu:

  • Readonly<T> i ReadonlyArray<T>

  • Partial<T>

  • Required<T>

  • Record<K, V>

  • Pick<T, V>

  • Omit<T, V>

  • Exclude<T,K>

  • NonNullable<T>

Każdy z tych typów jest typem generycznym, czyli typem powstałym w oparciu o istniejący typ bazowy. Generyki to jedna z najbardziej istotnych konstrukcji możliwych do wykorzystania w języku TypeScript i właśnie poznałeś kolejny argument przemawiający za tą tezą.

Czas na kolejne zaawansowane scenariusze typowania danych - zapraszamy do następnej lekcji!
