Rozszerzanie typów globalnych i przestrzeni nazw w TypeScript to techniki pozwalające definiować typy, które są dostępne w całej aplikacji. Są szczególnie przydatne w sytuacjach, gdy aplikacja korzysta z globalnych zmiennych, takich jak process.env, window, lub wymaga obsługi globalnych akcji w systemach zarządzania stanem (np. Redux).

Poniżej omówimy te techniki na trzech praktycznych przykładach.

Rozszerzanie process.env

W standardowej konfiguracji TypeScript zmienne środowiskowe dostępne przez process.env są typowane jako string | undefined, co prowadzi do błędów, gdy próbujemy je wykorzystać jako string bez dodatkowej walidacji:

const myFunc = (envVar: string) => console.log(envVar.toUpperCase());
myFunc(process.env.NODE_ENV); // Błąd: Argument of type 'string | undefined' is not assignable to parameter of type 'string'.

Załóżmy, że w projekcie mamy zmienne środowiskowe DATABASE_URL oraz PORT.

Rozszerzenie typu ProcessEnv w przestrzeni nazw NodeJS:

// @types/globals.d.ts
namespace NodeJS {
  interface ProcessEnv {
    DATABASE_URL: string;
    API_KEY: string;
    NODE_ENV: 'development' | 'production' | 'test';
  }
}

Wykorzystanie rozszerzonych typów:

// app.ts
console.log(process.env.DATABASE_URL); // Typ: string
console.log(process.env.NODE_ENV); // Typ: 'development' | 'production' | 'test'

Musisz jednak pamiętać, że w ten sposób nie dokonujemy walidacji zawartości naszych zmiennych środowiskowych. Bierzemy na wiarę, że faktycznie będą one poprawnie ustawione podczas działania aplikacji. Takie założenie może sprawdzić się w 95% przypadków, ale nieoczekiwane 5% może mieć fatalny wpływ na aplikację, włącznie z poważnymi awariami.

Dodanie walidacji zmiennych środowiskowych w aplikacjach TypeScript za pomocą biblioteki zod jest jedną z najbardziej efektywnych metod zapewnienia bezpieczeństwa i poprawności typów. Dzięki temu podejściu możemy zweryfikować, czy wszystkie wymagane zmienne środowiskowe są dostępne, a ich wartości mają oczekiwany typ i spełniają wymagane kryteria.

Wykorzystanie zod pozwala również na automatyczne generowanie typów, co upraszcza korzystanie z process.env w projekcie.

Definicja schematu walidacyjnego

Zaczynamy od utworzenia pliku, w którym zdefiniujemy schemat dla zmiennych środowiskowych. W naszym przykładzie użyjemy pliku env.ts. Za pomocą zod tworzymy obiekt, w którym każda zmienna środowiskowa jest dokładnie opisana:

// env.ts
import { z } from 'zod';

const envSchema = z.object({
  PORT: z.coerce.number().min(1000).max(65535),
  DATABASE_URL: z.string().url(),
  NODE_ENV: z.enum(['development', 'production', 'test']).default('development'),
  API_KEY: z.string().min(1),
});

Tutaj każda zmienna ma przypisany specyficzny typ i reguły:

  • PORT jest liczbą, którą wymuszamy nawet jeśli w środowisku jest przechowywana jako string, a dodatkowo sprawdzamy, czy znajduje się w zakresie od 1000 do 65535.

  • DATABASE_URL musi być prawidłowym URL-em.

  • NODE_ENV to zmienna o stałym zestawie wartości (development, production, test) z domyślną wartością development.

  • API_KEY to ciąg znaków o minimalnej długości 1, co gwarantuje, że klucz API nie będzie pusty.

Walidacja process.env

Po zdefiniowaniu schematu możemy przeprowadzić walidację zmiennych środowiskowych. Używamy metody parse, która sprawdza zawartość obiektu process.env zgodnie z regułami zdefiniowanymi w schemacie:

// env.ts
const env = envSchema.parse(process.env);
export default env;

Jeśli którakolwiek ze zmiennych nie spełnia wymagań schematu, aplikacja zgłosi błąd w czasie uruchomienia. Jest to szczególnie przydatne podczas rozwoju, ponieważ natychmiast sygnalizuje problemy z konfiguracją.

Użycie zmiennych środowiskowych w kodzie

Po walidacji zmienne środowiskowe są dostępne w postaci obiektu env, który można łatwo importować i używać w aplikacji:

import env from './env';

console.log(`Server running on port ${env.PORT}`);
console.log(`Connected to database at ${env.DATABASE_URL}`);

Dzięki walidacji z zod mamy pewność, że zmienne te są poprawnie typowane i spełniają wymagania biznesowe.

Rozszerzenie typu process.env

Aby w całym projekcie można było korzystać z autouzupełniania i automatycznego typowania, można rozszerzyć globalny interfejs process.env. W tym celu rozszerzamy ProcessEnv za pomocą typu generycznego z.infer:

// @types/globals.d.ts
import { z } from 'zod';

declare global {
  namespace NodeJS {
    interface ProcessEnv extends z.infer<typeof envSchema> {}
  }
}

Dzięki temu process.env automatycznie dziedziczy typy zdefiniowane w naszym schemacie, co eliminuje konieczność importowania env w każdym pliku.

Więcej o biblitece zod dowiesz się z lekcji #Ekosystem języka TypeScript.

Walidacja w czasie działania i CI/CD

Największą zaletą użycia zod jest możliwość walidacji w czasie działania aplikacji. Jeśli schemat zostanie naruszony, aplikacja nie uruchomi się, a my otrzymujemy szczegółowe informacje o brakujących lub niepoprawnych zmiennych środowiskowych. Na przykład, jeśli w pliku .env brakuje PORT nie jest liczbą z wskazanego zakresu, zod zgłosi czytelny błąd:

codeError: Invalid value at "PORT". Expected a number between 1000 and 65535.

W przypadku błędów walidacyjnych build zakończy się niepowodzeniem, co zapobiega wdrożeniu aplikacji z błędną konfiguracją.

Rozszerzenie obiektu window

W projektach frontendowych bardzo często pojawia się potrzeba dodania własnych właściwości do globalnego obiektu window. Mogą to być na przykład konfiguracje aplikacji, funkcje użytkowe lub właściwości wprowadzone przez biblioteki zewnętrzne. Jednak domyślna definicja window w TypeScript, która jest zawarta w pliku lib.dom.d.ts, nie uwzględnia tych rozszerzeń. Próba użycia własnych właściwości bez ich uprzedniego zadeklarowania prowadzi do błędów typu:

Property 'config' does not exist on type 'Window & typeof globalThis'.

Rozwiązaniem jest rozszerzenie interfejsu Window, co pozwala na wzbogacenie jego definicji o dodatkowe właściwości. TypeScript wspiera tę operację za pomocą mechanizmu tzw. declaration merging.

Definicja rozszerzenia w pliku globalnym

Najbardziej uniwersalnym sposobem rozszerzenia window jest użycie pliku .d.ts, który automatycznie wprowadza zmiany w globalnym zakresie. 

// @types.globals.d.ts
declare global {
  interface Window {
    config: {
      apiUrl: string;
      featureFlag: FeatureFlag;
    };
  }
}

export {};

W powyższym przykładzie rozszerzamy interfejs Window o właściwość config. Jest to obiekt, który przechowuje adres wewnętrznego API (apiUrl) oraz obiekt z flagami (featureFlag). Koniecznie musimy pamiętać o export {} (niezależnie czy korzystamy z moduleDetection: force).

Dzięki rozszerzonej definicji TypeScript automatycznie podpowiada nazwy właściwości oraz sprawdza ich typy. Jeśli przypisane wartości nie spełniają wymagań typów (np. apiUrl nie jest stringiem), kompilator zgłosi błąd.

Podczas korzystania z window.config w innych częściach aplikacji, np. w komponentach, możemy bezpiecznie odwoływać się do tej właściwości:

// app.ts
console.log(window.config.apiUrl); // 'https://api.example.com'
console.log(window.config.featureFlag); // true

Rozszerzenie lokalne dla pojedynczego modułu

Jeśli zmiana dotyczy tylko jednego modułu lub pliku i nie chcemy wpływać na globalny zakres, możemy użyć declare const zamiast declare global. Ten sposób jest mniej inwazyjny, ale ogranicza widoczność zmian tylko do określonego miejsca w kodzie.

// app.ts
declare const window: {
  config: {
    apiUrl: string;
    featureFlag: boolean;
  };
} & Window;

Taka deklaracja oznacza, że obiekt window w bieżącym pliku ma dodatkowe właściwości config, ale w innych plikach window pozostaje niezmieniony. To rozwiązanie jest przydatne w sytuacjach, gdy rozszerzenie window ma charakter tymczasowy lub dotyczy tylko konkretnego modułu.

Różnice między metodami implementacji

Rozszerzenie obiektu window może być realizowane na różne sposoby, w zależności od potrzeb projektu:

  1. Plik .d.ts: Jest to najlepszy wybór dla globalnych rozszerzeń. Wszystkie zmiany są widoczne w całym projekcie, co upraszcza ich użycie i utrzymanie.

  2. declare global w pliku .ts lub .tsx: Działa podobnie jak .d.ts, ale zmiany są umieszczane w zwykłym module. Może to prowadzić do bałaganu, szczególnie w dużych projektach, gdzie rozszerzenia są używane w wielu miejscach.

  3. declare const: Rozwiązanie lokalne, ograniczające widoczność zmian do jednego modułu. To podejście jest odpowiednie, gdy nie chcemy wprowadzać zmian globalnych.

Rozszerzenie obiektu window jest nie tylko praktyczne, ale również zgodne z zasadami dobrego zarządzania typami w TypeScript. Odpowiedni wybór metody implementacji (globalne lub lokalne) pozwala na dostosowanie tego rozwiązania do specyfiki projektu.

Podsumowanie

W trakcie lekcji zapoznaliśmy się z technikami rozszerzania typów globalnych i przestrzeni nazw w TypeScript, które umożliwiają definiowanie i zarządzanie typami dostępnymi w całej aplikacji. Te techniki pozwalają na lepsze dopasowanie typów do specyficznych potrzeb projektu, zwiększając bezpieczeństwo, czytelność i spójność kodu.
