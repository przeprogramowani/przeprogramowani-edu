Wprowadzenie

Efektywna organizacja kodu to kluczowy element skalowalnych projektów TypeScript. Obejmuje ona nie tylko organizację typów, ale również świadome podejście do modułów i przestrzeni nazw. W tej lekcji przyjrzymy się kluczowym aspektom:

  1. Przestrzeniom nazw – rozwiązanie historyczne dla organizacji kodu.

  2. Modułom TypeScript – nowoczesne podejście do izolacji i importowania kodu.

Przestrzenie nazw (namespaces)

Przestrzenie nazw (ang. namespaces) zostały wprowadzone do TypeScript jako odpowiedź na problemy z organizacją kodu w języku JavaScript sprzed ery ES6 (2015). W tamtym czasie brakowało wbudowanego mechanizmu modułów, co prowadziło do częstych konfliktów nazw w dużych projektach.

Przestrzenie nazw pozwalają grupować powiązane ze sobą funkcje, klasy, zmienne czy typy w jeden logiczny obszar, unikając problemów z globalnymi zmiennymi.

namespace Utils {
  export function calculateTotal(a: number, b: number): number {
    return a + b;
  }

  export const CONSTANT = 42;

  export namespace Formatting {
    export function formatUSD(value: number): string {
      return `$${value.toFixed(2)}`;
    }
  }
}

console.log(Utils.calculateTotal(5, 10)); // 15
console.log(Utils.Formatting.formatUSD(50)); // "$50.00"

Z wykorzystaniem namespaces możemy budować zagnieżdżone struktury, co ułatwiało zarządzanie większymi projektami.

Jak działają przestrzenie nazw?

Przestrzenie nazw były transpilowane na zwykły kod JavaScript poprzez zamykanie deklaracji w natywnych funkcjach IIFE (ang. Immediately Invoked Function Expression). Dzięki temu zapewniały izolację nazw, zachowując zgodność z ówczesnym środowiskiem JavaScript.

namespace MathUtils {
  export const PI = 3.14;

  export function calculateCircumference(radius: number): number {
    return 2 * PI * radius;
  }
}

Transpilacja do JavaScript: 

var MathUtils;
(function (MathUtils) {
  MathUtils.PI = 3.14;
  MathUtils.calculateCircumference = function (radius) {
    return 2 * MathUtils.PI * radius;
  };
})(MathUtils || (MathUtils = {}));

Dlaczego przestrzenie nazw zostały zastąpione przez moduły?

Przestrzenie nazw są dostępne tylko w TypeScript i wymagają transpilacji do JavaScript, co może prowadzić do problemów z debugowaniem. Ich użycie utrudnia współpracę z narzędziami opartymi na modułach ES6 (np. tree shaking w bundlerach takich jak Webpack).

Rekomendacja: stosuj moduły zamiast przestrzeni nazw

ES6 wprowadziło natywne moduły (import/export), które oferują lepszą izolację i bardziej elegancką składnię (import/export). ES Modules (ESM) działają w lokalnym zakresie, co dodatkowo minimalizuje ryzyko kolizji nazw.

Zamiast grupować funkcje w przestrzeni nazw, wystarczy używać oddzielnych modułów:

// utils/calculations.ts
export function calculateTotal(a: number, b: number): number {
  return a + b;
}

// utils/formatting.ts
export function formatCurrency(value: number): string {
  return `$${value.toFixed(2)}`;
}

// main.ts
import { calculateTotal } from './utils/calculateTotal';
import { formatCurrency } from './utils/formatting';

console.log(calculateTotal(5, 10)); // 15
console.log(formatCurrency(50)); // "$50.00"

Dodatkowo, ES modules pozwalają na dynamiczne ładowanie kodu (lazy loading), co może poprawić wydajność aplikacji.

function exportToPDF() {
    import("jspdf").then(({ jsPDF }) => {
      const doc = new jsPDF();
      doc.save();
    }).catch(error => console.log(error));
}

Jest to szczególnie istotne, gdy do obsługi funkcjonalności potrzebujemy ciężkiego modułu, jak jspdf, a użytkownicy sporadycznie korzystają z tej opcji. Dzięki wykorzystaniu lazy loading, moduł jspdf zostanie załadowany w momencie wywołania funkcji “Eksportuj do PDF” a nie przy każdym załadowaniu strony.

Kiedy jeszcze można używać przestrzeni nazw?

Chociaż przestrzenie nazw są uważane za przestarzałe w nowych projektach, mogą być przydatne w specyficznych sytuacjach:

  1. Starsze projekty: Projekty oparte na TypeScript przed sprzed ES6. Jednak od premiery ES6 minęło już prawie 10 lat, więc to naprawdę sporadyczne przypadki.

  2. Typowanie globalnych bibliotek: Przestrzenie nazw nadal sprawdzają się w definiowaniu globalnych API, np. w rozszerzeniach globalnych obiektów w d.ts.

Namespaces wciąż znajdują zastosowanie w bibliotekach takich jak @types/react, szczególnie w przypadku definiowania i rozszerzania interfejsów globalnych, takich jak JSX. Ten konkretny przykład ilustruje, jak przestrzeń nazw JSX jest wykorzystywana do integracji JSX z Reactem w środowisku TypeScript.

Dlaczego namespace JSX jest używane w React?

Przestrzeń nazw JSX jest istotna, ponieważ TypeScript musi wiedzieć, jak obsługiwać składnię JSX (np. <div></div>). W ramach tej przestrzeni definiowane są typy, które TypeScript wykorzystuje do walidacji i rozpoznawania składni JSX w czasie kompilacji.

export namespace JSX {
    type ElementType = React.JSX.ElementType;
    interface Element extends React.JSX.Element {}
    interface ElementClass extends React.JSX.ElementClass {}
    interface ElementAttributesProperty extends React.JSX.ElementAttributesProperty {}
    interface ElementChildrenAttribute extends React.JSX.ElementChildrenAttribute {}
    type LibraryManagedAttributes<C, P> = React.JSX.LibraryManagedAttributes<C, P>;
    interface IntrinsicAttributes extends React.JSX.IntrinsicAttributes {}
    interface IntrinsicClassAttributes<T> extends React.JSX.IntrinsicClassAttributes<T> {}
    interface IntrinsicElements extends React.JSX.IntrinsicElements {}
}

Jakie są korzyści stosowania namespaces w @types/react?

  1. Globalność:
    JSX działa globalnie, niezależnie od tego, czy jest używany w React, czy w innych frameworkach. Przestrzenie nazw pozwalają na definiowanie globalnych interfejsów bez kolizji z innymi elementami kodu.

  2. Rozszerzalność:
    Dzięki namespaces biblioteki i użytkownicy mogą rozszerzać istniejące typy lub dodawać nowe, bez modyfikacji kodu źródłowego @types/react.

  3. Wsteczna kompatybilność:
    Niektóre starsze mechanizmy JSX wciąż korzystają z przestrzeni nazw, co umożliwia współdziałanie z narzędziami używanymi przed popularyzacją ES6 modułów.

W nowych projektach użycie przestrzeni nazw jest ograniczone do takich właśnie przypadków, jak JSX, które wymagają globalnych interfejsów lub integracji z zewnętrznymi narzędziami.

Moduły i skrypty w TypeScript

Czym są moduły w TypeScript?

Moduły w TypeScript (i szerzej w JavaScript) to odizolowane fragmenty kodu, które pozwalają nam organizować aplikacje w sposób modularny, kontrolując, które elementy kodu są dostępne w innych częściach projektu. Moduł definiuje własną przestrzeń nazw, co eliminuje ryzyko konfliktów nazw między różnymi częściami kodu.

Każdy plik w TypeScript, który zawiera przynajmniej jedną instrukcję import lub export, jest traktowany jako moduł. Moduły umożliwiają lepsze zarządzanie kodem, ułatwiają jego testowanie i ponowne wykorzystanie, a także pomagają w utrzymaniu czytelności dużych aplikacji.

// math.ts
export const add = (a: number, b: number): number => a + b;

// app.ts
import { add } from './math';
console.log(add(2, 3)); // Output: 5

W powyższym przykładzie math.ts definiuje moduł zawierający funkcję add, która jest eksportowana. Plik app.ts importuje tę funkcję i może ją użyć w swoim zakresie.

Czym są skrypty w TypeScript?

Skrypty to pliki TypeScript, które nie zawierają żadnych instrukcji import ani export. TypeScript traktuje je jako część globalnego kontekstu aplikacji, podobnie jak tradycyjne skrypty w JavaScript ładowane za pomocą <script> bez atrybutu type=”module”. W praktyce oznacza to, że wszystkie zmienne i funkcje zdefiniowane w takim skrypcie są automatycznie dostępne w innych częściach projektu, o ile również są skryptami.

// globalScript.ts
const globalVar = "Hello, World!";

// app.ts
console.log(globalVar); // Output: Hello, World!

Tutaj zmienna globalVar jest dostępna w app.ts bez potrzeby importowania. Taka organizacja kodu może być wygodna w małych projektach, ale w większych prowadzi do problemów związanych z kolizją nazw.

Dlaczego warto preferować moduły?

Stosowanie modułów pozwala unikać typowych problemów, takich jak:

  • Kolizje nazw: Wszystkie zmienne w module są lokalne, o ile nie są eksportowane.

  • Niejasność kodu: Eksporty i importy jasno wskazują zależności między plikami.

  • Problemy z narzędziami: Moduły są zgodne z nowoczesnymi standardami ECMAScript, co zapewnia lepszą kompatybilność z narzędziami developerskimi (Vite, Webpack itd.)

Praktyczne wyjątki

Skrypty mogą być odpowiednie w określonych sytuacjach, np.:

  1. W prostych projektach lub prototypach: Gdy szybkość tworzenia kodu jest ważniejsza niż jego izolacja.

  2. W starszych aplikacjach: Aby uniknąć kosztownej migracji istniejącego kodu.

  3. Do definiowania globalnych zmiennych lub konfiguracji: W projektach, gdzie część kodu celowo działa w globalnym zakresie (np. pliki konfiguracyjne).

W nowoczesnych aplikacjach rekomendujemy jednak konsekwentne stosowanie modułów jako domyślnej praktyki. Skrypty powinny być używane tylko w szczególnych przypadkach, kiedy ich zalety przeważają nad ryzykiem związanym z globalnym zakresem.

Mając to na uwadze, domyślnie warto wyeliminować mechanizm podziału na moduły i skrypty na poziomie tsconfig.json. Wystarczy skorzystać z “moduleDetection: force”.

{
  "compilerOptions": {
    "moduleDetection": "force" // Domyślnie "auto", które wprowadza opisany powyżej podział
  }
}
