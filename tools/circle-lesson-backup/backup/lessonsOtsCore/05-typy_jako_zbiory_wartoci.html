Wprowadzenie

W poprzedniej lekcji omówiliśmy tzw. typy proste (primitive types), które wywodzą się wprost z typów dostępnych w języku JavaScript (string, number, boolean). Wprowadziliśmy również pojęcie inferencji, czyli mechanizmu rozpoznającego typ bez jawnego oznaczenia go przez programistę, a także wstępnie omówiliśmy dwa nowe koncepty, których JS nie posiada - aliasy typów oraz interfejsy.

Przejście przez “typowy happy path” nie wymaga zbyt wiele wysiłku, ale schody zaczynają się w momencie wprowadzania niestandardowych scenariuszy, takich jak “zwykły obiekt”, błędnie deklarowany przez puste nawiasy klamrowe. Tego typu przypadków na puste, błędne lub “dziwne” typy i wartości TypeScript zawiera jeszcze kilka, ale do ich zrozumienia potrzebujemy lepszej teorii.

W tej lekcji przedstawimy ci uniwersalną “filozofię TypeScriptu”, która w praktyce odnosi się do matematycznych operacji na zbiorach danych. Bez obaw - nie będzie to wykład wprost z uczelni, a raczej praktyczne narzędzie do zrozumienia typów i ich transformacji.

Problematyczny obiekt

Celem tej lekcji będzie przedstawienie kluczowych informacji, które pomogą ci zrozumieć poniższy fragment kodu:

// Zamiar - someObj to będzie "jakiś obiekt", więc wykorzystam puste nawiasy:
let someObj: {} = {};

// Zgodnie z oczekiwaniami, mogę do niego przypisać inne obiekty:
someObj = {name: "John"};

// Ale uwaga - to również poprawne przypisania:
someObj = true;
someObj = "hello";
someObj = [];
someObj = 0.12;

console.log(someObj); // 0.12 (!)

Dlaczego typ zdefiniowany jako pusty obiekt może pasować do liczb, tablic i stringów?

Typy jako zbiory

Na typy w języku TypeScript powinieneś patrzeć jak na zbiory akceptowalnych wartości.

Trzy typy proste (string, number i boolean) to nic innego jak trzy zbiory:

  • string to zbiór, który zawiera nieskończenie wiele łańcuchów znaków

  • number to zbiór, który zawiera nieskończenie wiele liczb

  • boolean to zbiór, który zawiera dwie wartości - true lub false

👉 Zapamiętaj: Weryfikując typy, TypeScript “ocenia” dopasowanie określonej wartości do zbioru.

Operacje na zbiorach

W teorii zbiorów można wyróżnić kilka operacji, których odwzorowanie znajdziemy w konkretnych operatorach TypeScript.

Unie typów (sumy zbiorów)

TypeScript daje nam możliwość łączenia zbiorów po to, aby utworzony typ obejmował więcej akceptowalnych wartości. Przykładowo, możemy utworzyć sumę zbiorów string i number. W nowym typie zmieści się dowolny łańcuch znaków lub liczba, jaką akurat chcemy się posługiwać:

W kodzie taka operacja realizowana jest przez pipe “|” i tworzy tzw. unię typów (union type):

type StringOrNumber = string | number;

const num: StringOrNumber = "25";
const str: StringOrNumber = "John";

Unia typów akceptuje wartości z obu zbiorów, które łączymy. Dotyczy to zarówno typów prostych, jak i typów złożonych, tworzonych przez aliasy:

type Cat = {
  kind: string;
};

type Dog = {
  breed: string;
};

type Animal = Cat | Dog;

const cat: Animal = {kind: "cat"}; // obiekt typu Cat
const dog: Animal = {breed: "labrador"}; // obiekt typu Dog

Taka operacja ma jednak swoje konsekwencje - o ile nowy typ akceptuje więcej wartości niż jego składowe, to w przypadku konkretnych operacji musimy doprecyzować, którym zbiorem wartości chcemy się posługiwać:

// ❌ Błąd - value nie zostało sprowadzone do właściwego typu
function processValue(value: StringOrNumber) {
  return value.toUpperCase();
         // ^ - Property 'toUpperCase' does not exist on type 'number'.ts(2339)
}

// ✅ Sukces - przed wywołaniem funkcji, wprowadzamy dodatkową weryfikację typu
function processValue(value: StringOrNumber) {
  if (typeof value === "string") {
    return value.toUpperCase();
  }
  return value;
}

Tego typu konstrukcje to tzw. type guards - dodatkowe wyrażenia warunkowe, które zawężają typ zmiennej na podstawie sprawdzenia wartości. Dzięki temu TypeScript może bezpiecznie operować na wartościach połączonych typów bez ryzyka przepuszczania błędów w runtime.

Co ważne, strażnicy typów muszą weryfikować takie cechy zmiennych, które będą istnieć w momencie działania programu. Z tego powodu nie możemy odwoływać się do aliasów typów, ponieważ są to konstrukcje ze świata TypeScriptu - w docelowym kodzie JavaScript nie istnieją.

Aby uniknąć błędów z tym związanych, powinniśmy sprawdzić zawartość obiektu, z którym pracujemy:

type Cat = {
  kind: string;
};

type Dog = {
  breed: string;
};

type Animal = Cat | Dog;

const cat: Animal = {kind: "cat"}; // obiekt typu Cat
const dog: Animal = {breed: "labrador"}; // obiekt typu Dog

const sayHello = (animal: Animal) => {
  
  // ❌ Błąd - typ Cat należy do świata "TypeScriptu" - nie może być sprawdzany w runtime
  // if (typeof animal === Cat) {
                    // ^ - 'Cat' only refers to a type, but is being used as a value here.ts(2693)

  // ✅ Sukces - zawartość obiektu będzie możliwa do sprawdzenia w runtime
  if ("kind" in animal) {
    console.log(`Hello, I am a ${animal.kind}`);
  } else if ("breed" in animal) {
    console.log(`Hello, I am a ${animal.breed}`);
  }
};

Literały, czyli zbiory jednoelementowe

Częstym przypadkiem wykorzystywania unii są typy zawierające wyłącznie jeden element - to tzw. literal types, czyli typy zawierające wyłącznie jedną wartość (string, number lub boolean).

Poniższy typ - ShowerState - nie jest stringiem, a literałem - zbiorem z jednym elementem (hot):

type ShowerState = "hot";

To popularna praktyka służąca do budowania typów o kilku stałych wartościach, często wykorzystywana np. w maszynach stanu albo uproszczonych typach, które nie powinny akceptować nieskończonej liczby wartości:

type ShowerState = "hot" | "cold"; // Nowy typ akceptuje wyłącznie dwie wartości

const hotWater: ShowerState = "hot";
const coldWater: ShowerState = "cold";
const mildWater: ShowerState = "mild";
                          // ^ - Type '"mild"' is not assignable to type 'ShowerState'.ts(2322)

Przecięcia typów (iloczyny zbiorów)

TypeScript wspiera jeszcze jeden operator, którego zachowanie możemy wprost wydedukować z teorii zbiorów. Mowa o ampersand “&” - którego efektem są przecięcia typów (intersection types). Zostając w świecie zbiorów, wyjaśnienie jego działania jest stosunkowo proste - nowy typ powstaje poprzez wybranie części wspólnej obu zbiorów, czyli wartości pasujących do obu typów jednocześnie.

Wprowadzone do tej pory operatory można przetestować tworząc dwa nowe typy:

  • Unia utworzy typ string, ponieważ oba łączone zbiory zawierają się w tym typie

  • Przecięcie utworzy typ dwóch literałów, ponieważ to jedyne wartości pasujące do obu zbiorów

type A = string;
type B = "john" | "doe";

type C = A | B; // string
type D = A & B; // "john" | "doe"

Czy przecięcie może pozbawić nas zbioru akceptowalnych wartości? Tak - przykładowo, kiedy zastosujemy go na zbiorach rozłącznych, które nie mają części wspólnej, otrzymamy typ never:

type F = string;
type G = number;

type H = F & G; // never

Nie ma w języku TypeScript wartości, która byłaby jednocześnie stringiem i numberem. Stąd… wynik oznaczający brak akceptowalnych wartości. Szczegóły typu never omówimy w kolejnej lekcji.

Przecięcia typów złożonych

Przecięcia typów prostych są stosunkowo łatwe do przewidzenia. W przypadku obiektów sprawa zaczyna się komplikować. Przecięcie zastosowane na dwóch typach złożonych, de facto łączy właściwości obu typów, zamiast wyszukiwania części wspólnej:

type A = { name: string };
type B = { age: number };

type C = A & B; // {name: string, age: number}

// ✅ Sukces - poprawne przypisanie
const c1: C = { name: "John", age: 20 };

// ❌ Błąd - obiekt nie zawiera pola age
const c2: C = { name: "John" };
// ❌ Błąd - obiekt nie zawiera pola name
const c3: C = { age: 20 };

Dlaczego tak jest?

Musisz wiedzieć, że TypeScript wspiera tzw. strukturalny system typów. Oznacza to, że kompilator przymyka oko na pewne nieścisłości w definiowaniu typów tak długo, jak dany obiekt spełnia minimalny zestaw wymagań określonego typu.

Na poniższym przykładzie widać działanie tego mechanizmu:

type A = {name: string};

function foo(a: A) {
  console.log(a.name);
}

const a1 = {name: "John"};
const a2 = {name: "Mark", country: "UK", job: "clerk"};
const a3 = {region: "EU"};

foo(a1); // ✅ Pasuje
foo(a2); // ✅ Pasuje
foo(a3); // ❌ Property 'name' is missing in type '{ region: string; }'

// UWAGA:
foo({name: "Mark", country: "UK"}); // ❌ Object literal may only specify known properties, and 'country' does not exist in type 'A'


// ---

type B = {age: number};

function bar(b: B) {
  console.log(b.age);
}

const b1 = {age: 20};
const b2 = {age: 21, role: "admin", skill: "drawing"};
const b3 = {access: "full"};

bar(b1); // ✅ Pasuje
bar(b2); // ✅ Pasuje
bar(b3); // ❌ Property 'age' is missing in type '{ access: string; }'

// UWAGA:
bar({age: 25, role: "admin"}); // ❌ Object literal may only specify known properties, and 'role' does not exist in type 'B'

Chociaż typy A i B powinny być obiektami z jednym polem, to istnieją sytuacje, kiedy możemy posługiwać się “szerszą wersją” tych typów - tak długo, jak znane pola są w nich zawarte (i nie są to literały obiektów).

Wróćmy do wcześniejszego pytania - dlaczego iloczyn typów łączy właściwości obu obiektów

Rozumowanie jest takie:

Jaki typ muszę utworzyć, aby na końcu pasował do obu typów bazowych?

Najprostsze rozwiązanie to połączenie obu obiektów - wtedy, przy ocenianiu struktury obiektu - finalny typ będzie “luźno dopasowany” do obu zbiorów określonych na początku:

To rozumowanie potwierdza się w kodzie:

const person = {name: "John", age: 20};

type A = {name: string};
type B = {age: number};

// Funkcja foo przyjmuje obiekty pasujące do zbioru A
function foo(a: A) {
  console.log(a.name);
}

foo(person); // => John ✅ Obiekt person tutaj pasuje - ma pole name!

// Funkcja bar przyjmuje obiekty pasujące do zbioru B
function bar(b: B) {
  console.log(b.age);
}

bar(person); // => 20 ✅ Obiekt person nadal pasuje - ma pole age!

Co ważne, takie zachowanie będzie wspierane wyłącznie w językach o strukturalnym systemie typów.

👉 Dla ciekawskich: Przeciwieństwem tego mechanizmu jest nominalny system typów (np. w języku Java), gdzie obiekt oceniany jest nie tylko pod kątem struktury, ale jego pochodzenia (nominału). Poniżej widać dwie klasy o identycznym kształcie, które w Javie nie mogą być stosowane naprzemiennie:

class Person {
    String name;
    int age;

    public Person(String name, int age) {
        this.name = name;
        this.age = age;
    }
}

class Employee {
    String name;
    int age;

    public Employee(String name, int age) {
        this.name = name;
        this.age = age;
    }
}

public class Main {
    public static void main(String[] args) {
        Person person = new Person("John", 30);
        Employee employee = new Employee("Jane", 25);

        // Próba przypisania obiektu typu Employee do zmiennej typu Person
        person = employee; // ❌ Błąd kompilacji: incompatible types
    }
}

W TypeScript natomiast strukturalny system typów sprawia, że przypisanie obiektów zależy bardziej od zgodności kształtu niż od ich pochodzenia. W praktyce oznacza to, że jeżeli obiekt posiada wymagane właściwości, będzie traktowany jako zgodny z typem – nawet jeśli jest to zupełnie inny obiekt niż ten, który był pierwotnie przewidziany.

Rozszerzanie typów

Praktyczne zastosowanie operatora przecięcia to rozszerzanie istniejących typów:

type Id = {
  id: number;
};

type Name = {
  firstName: string;
  lastName: string;
};

type Greeter = {
  greet: (name: string) => string;
};

type User = Id & Name & Greeter;

const u1: User = {
  id: 1,
  firstName: "John",
  lastName: "Doe",
  greet: (name: string) => `Hello ${name}`,
};

Zwróć jednak uwagę, że w przeciwieństwie do rozszerzania interfejsów, których składowe mogą znajdować się w wielu plikach i wciąż dotyczyć tej samej struktury, aliasy typów zawsze tworzą nowy typ:

type Car = {
  brand: string
};

Car = Car & { age: number; } // ❌ Błąd

type AgedCar = Car & { age: number; } // ✅ OK

Jest to warty zapamiętania mechanizm, z którego będziemy korzystać w lekcjach o zaawansowanym typowaniu

Rozwiązanie zagadki

Czy teraz jesteś w stanie odpowiedzieć, dlaczego puste nawiasy klamrowe nie określają pustego obiektu, albo “zwykłego obiektu”, ale wiele różnych wartości?

// Zamiar - someObj to będzie "jakiś obiekt", więc wykorzystam puste nawiasy:
let someObj: {} = {};

// Zgodnie z oczekiwaniami, mogę do niego przypisać inne obiekty:
someObj = {name: "John"};

// Ale uwaga - to również poprawne przypisania:
someObj = true;
someObj = "hello";
someObj = [];
someObj = 0.12;

console.log(someObj); // 0.12 (!)

Najłatwiej zrozumieć to na zbiorach - idąc “od ogółu do szczegółu”, puste nawiasy klamrowe określają wszystkie niepuste wartości, które mają pewne cechy obiektu. Wprowadzając kolejne pola, zawężamy obiekty, które pasują do naszego typu.

Nic jednak nie stoi na przeszkodzie, żeby na string, number i boolean również patrzeć jak na obiekty, w końcu mają konkretne właściwości, które możemy wywoływać i wywodzą się z “obiektowych” typów pisanych wielką literą. Poniżej przedstawiamy to w bardzo dosłowny sposób ;)

const a: String = "John"; // String
const b: Number = 20; // Number
const c: Boolean = false; // Boolean

a.toUpperCase();
b.toFixed();
c.valueOf();

type KindaObjectIfYouLookClosely = {};

function foo(maybeObject: KindaObjectIfYouLookClosely) {
  console.log(maybeObject);
}

foo(a); // ✅ Pasuje
foo(b); // ✅ Pasuje
foo(c); // ✅ Pasuje

Typy jako zbiory

Zaprezentowany w tej lekcji sposób myślenia o typach może się wydawać wymagający, ale często jest najlepszą odpowiedzią na najtrudniejsze zagadnienia typowania w TypeScript.

Jeśli w przyszłości zdarzy ci się zastanawiać, dlaczego określona wartość pasuje do dziwnego typu, najprawdopodobniej odpowiedź znajdziesz w teorii zbiorów. Możesz do tej lekcji wracać i przypominać sobie najważniejsze aspekty tego zagadnienia.

Żeby jednak omówić go do końca, musimy jeszcze porozmawiać o zbiorach… bez wartości. Ale o tym kolejna lekcja - zapraszamy!
