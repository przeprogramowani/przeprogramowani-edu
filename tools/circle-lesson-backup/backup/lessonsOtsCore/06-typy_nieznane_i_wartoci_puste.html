Wprowadzenie

Jeśli do nauki TypeScriptu podchodziłeś już wcześniej, być może spotkałeś się ze słynnym typem “any”, którego stosowanie powszechnie uważany jest za antywzorzec. W tej lekcji wyjaśnimy ci dlaczego tak jest (i czy zawsze?), ale omówimy też inne typy, które mogą się pojawić w twoim kodzie na skutek pracy z TypeScriptem i poszczególnymi operatorami.

Typy nieznane - any i unknown

Oba typy wskazują na brak wiedzy o tym, czym jest przetwarzana wartość. Mają one istotną różnicę z punktu widzenia działania aplikacji i bezpieczeństwa kodu.

Typ any jest używany, gdy:

  • Chcemy całkowicie wyłączyć sprawdzanie typów dla danej zmiennej. Jest to typ, który „wyłącza” TypeScript i pozwala na wykonywanie operacji bez żadnych ograniczeń.

  • Nie zależy nam na bezpieczeństwie typu w danej sytuacji, np. gdy szybko prototypujemy coś w kodzie i chcemy skupić się na logice, a nie na dokładnym typowaniu.

  • Pracujemy z kodem napisanym w czystym JavaScript, gdzie typy nie zostały określone. W takich przypadkach any pozwala uniknąć błędów kompilacji, szczególnie jeśli chcemy szybko zintegrować jakiś moduł JS.

  • Migracja kodu do TypeScript jest w toku i jeszcze nie wiadomo, jaki dokładnie typ powinien zostać przypisany do pewnych zmiennych. Wtedy any może być stosowany tymczasowo jako miejsce na bardziej precyzyjne typy, które zostaną dodane później.

Przyzwyczajenie do any potrafi uzależnić i to główny powód, dla którego jest to typ uważany za antywzorzec. Traktuj go jako tymczasową taśmę klejącą, a w dłuższej perspektywie wymieniaj na precyzyjne typy. Kompilator może ci w tym pomóc oferując dodatkową opcję, która wyłącza możliwość niejawnego posługiwania się tym typem w projekcie: noImplicitAny (w sekcji compilerOptions).

Typ unknown jest używany gdy:

  • Nie znamy typu wartości podczas kompilacji, ale chcemy zachować pewien poziom kontroli nad typem przed użyciem tej wartości. unknown wymusza na nas sprawdzenie typu, zanim wykonamy na tej zmiennej konkretne operacje.

  • Przyjmujemy dane z zewnętrznego źródła, np. z API lub z formularza, gdzie typ danych jest niepewny. unknown jest dobrym wyborem, bo pozwala zachować bezpieczeństwo typu – przed użyciem musimy wykonać sprawdzenie i rzutowanie zmiennej na określony typ danych.

  • Chcemy zabezpieczyć kod przed nieoczekiwanymi błędami typów. Użycie unknown zmusza nas do jawnego sprawdzenia typu przy pomocy operatora typeof lub innych mechanizmów. Dzięki temu unikamy sytuacji, w której próbujemy wykonać operację na typie niezgodnym (np. próbujemy wywołać metodę, której dana wartość nie ma).

Brak wartości - never i void

W tej sekcji omówimy dwa specjalne typy w TypeScript: never i void, które często są mylone ze sobą ze względu na ich podobne zastosowania. Dowiesz się, kiedy i jak ich używać oraz dlaczego są one istotne w kontekście pisania bezpiecznego i czytelnego kodu.

Void

Typ void jest stosowany głównie do określania funkcji, które nie zwracają wartości. W kontekście JavaScript, funkcje tego typu zwracają undefined - na przykład, kiedy nie używamy słowa kluczowego return, funkcja automatycznie zwróci undefined. Typ void wskazuje, że zwracana wartość funkcji nie ma znaczenia i nie jest używana.

function logMessage(message: string): void {
    console.log(message);
}

W praktyce, przy tego typu funkcjach określenie typu zwracanego pomijamy, a TypeScript sam zajmie się jego rozpoznaniem.

Never

Patrząc na typy jak na zbiory wartości, never będzie oznaczał pusty zbiór wartości - coś, co powstało na skutek naszej pracy z kodem lub typami, a co przestaje mieć wartości możliwe do przypisania. Jak to możliwe?

Przykładowo, kiedy poprzez unię spróbujemy uzyskać część wspólną stringów i numberów, otrzymamy typ never - nie ma wartości, która jest jednocześnie łańcuchem znaków i liczbą:

type F = string;
type G = number;

type H = F & G; // never

Typ never może również służyć do oznaczania funkcji, które nigdy się nie zakończą (pętle) lub nie zwrócą wartości możliwej do obsłużenia (wyjątki):

function alwaysThrows(): never {
  throw new Error("Always throws");
}

const a: never = alwaysThrows();

function alwaysLoops(): never {
  while (true) {}
}

const b: never = alwaysLoops();

Kiedy taki typ może się przydać? Ciekawym przykładem są konstrukcje switch połączone z literałami typów - musimy wykorzystać wszystkie “kejsy”, a to, co pozostanie, będzie właśnie typu never. Jeśli jakiegoś przypadku nie obsłużymy (np. blue) lub dodamy nowy, system typów wskaże błąd (sprawdź sam):

function unknownColor(x: never): never {
    throw new Error("Nieznany kolor!");
}

// Wprowadź nowy kolor i zobacz zachowanie TypeScriptu
type Color = 'red' | 'green' | 'blue';

function getColorName(c: Color): string {
    switch(c) {
        case 'red':
            return 'is red';
        case 'green':
            return 'is green';
        case 'blue':
            return 'is blue';
        default:
            return unknownColor(c);
    }
}

Void czy never?

Zdarza się, że programiści mylą te dwa typy, zakładając, że funkcja, która "nic nie zwraca", powinna mieć typ never. W rzeczywistości never jest znacznie bardziej restrykcyjny i oznacza brak możliwości zwrócenia czegokolwiek, nawet undefined.

// ❌ Nieprawidłowe podejście
function logWarning(message: string): never {
    console.warn(message);
}

// ✅ Prawidłowe podejście
function logWarning(message: string): void {
    console.warn(message);
}

Null i undefined

I call it my billion-dollar mistake. It was the invention of the null reference in 1965 - Tony Hoare

Na koniec krótkie omówienie dwóch typów, które wprost nie reprezentują zbiorów, ale są raczej odwzorowaniem wartości z języka JavaScript:

  • null - symbolizuje wartość “pustą”

  • undefined - symbolizuje zadeklarowaną zmienną bez wartości początkowej

let noValue: null = null;
let undefinedValue: undefined;

console.log(noValue); // null
console.log(undefinedValue); // undefined

Szczególnie w przypadku wartości null, często stosuje się ją w parze z innymi typami, aby zaznaczyć opcjonalne występowanie określonej wartości:

const img: HTMLImageElement | null = document.querySelector("img");

// Type guard - zmiana typu z "HTMLImageElement | null" na "HTMLImageElement"
if (img) { 
  img.src = '...';
}

Jeśli mamy pewność, że dany element istnieje, możemy wykorzystać wykrzyknik aby to zaznaczyć:

const img: HTMLImageElement | null = document.querySelector("img");

img!.src = "https://...";

Ważne ustawienie kompilatora

W ustawieniach kompilatora TypeScript istnieje ustawienie strictNullChecks, które wpływa na to, jak TypeScript traktuje puste wartości (null i undefined).

Gdy to ustawienie jest włączone (domyślnie), null i undefined są traktowane jako osobne typy, co oznacza, że przypisanie ich do zmiennej o określonym typie, który nie uwzględnia tych wartości, spowoduje błąd kompilacji. Dzięki temu musisz jawnie określić, kiedy dana zmienna może przyjąć wartość pustą, co zwiększa bezpieczeństwo i czytelność kodu, a także zmniejsza prawdopodobieństwo błędów w runtime.

Wpływa to również na pracę z biblioteką standardową - niektóre metody w bibliotekach TypeScript mają sygnatury, które wymagają uwzględnienia pustych wartości. Jest to stosowane wyłącznie jeżeli strictNullChecks jest ustawione na true:

// strictNullChecks: true
const img: HTMLImageElement | null = document.querySelector("img");

// strictNullChecks: false
const img: HTMLImageElement = document.querySelector("img");

Przydatny operator

Zarówno JavaScript jak i TypeScript wspierają ten sam operator do zabezpieczania się przed wartościami null i undefined - nullish coalescing operator “??”. Jeśli zmienna nie ma przypisanej wartości lub jest ona pusta (null), pod uwagę zostanie wzięta prawa część tego operatora:

const cfg = null;
const generate = () => 5;

const result = cfg ?? generate();
console.log(result); // 5

// Cukier składniowy względem:
// const result = (cfg !== null && cfg !== undefined) ? cfg : generate();
