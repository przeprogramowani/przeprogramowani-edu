Wzorzec repozytorium

Jedno z najczęściej spotykanych zastosowań generyków wiąże się z wzorcem Repozytorium.

Wzorzec Repozytorium (ang. Repository Pattern) to wzorzec projektowy, który oddziela warstwę dostępu do danych (Data Access) od logiki biznesowej (App). Zamiast bezpośrednio korzystać z bazy danych w logice aplikacji (i za każdym razem kontrolować zarządzanie połączeniami, budowanie zapytań czy sprawdzać stan bazy), używamy specjalistycznych modułów (repozytoriów), które obsługują operacje na danych – np. pobieranie, zapisywanie, usuwanie czy aktualizowanie danych. Dzięki temu aplikacja staje się bardziej modularna, łatwiejsza do testowania oraz utrzymania.

Zauważmy, że komunikacja z bazą będzie wyglądać tak samo niezależnie od tego, którą kolumnę akurat obsługujemy. Wykonywane zapytania będą się zwykle różnić listą parametrów oraz nazwą tabeli, przy czym ogólny kształt zapytań będzie bardzo podobny:

-- Wyszukiwanie po id
SELECT id, name, age FROM users WHERE id = 123
SELECT id, price, variant FROM products WHERE id = 432

-- Usuwanie po id
DELETE FROM users WHERE id = 123
DELETE FROM products WHERE id = 432

Ta prawidłowość wpisuje się w nasze rekomendacje dotyczące generyków - jeśli poszczególne operacje różnią się wyłącznie typem danych, prawdopodobnie możemy przeprowadzić dodatkową refaktoryzację w kierunku typów generycznych.

Punkt startowy

Naszą refaktoryzację przeprowadzimy na kodzie JavaScript składającym się z trzech elementów:

  • db.js - moduł udostępniający połączenie z bazą

  • data-access.js - moduł wystawiający operacje na danych

  • index.js - docelowa logika aplikacji

Całość prezentuje się następująco (korzystamy z Node.JS, bazy PostgreSQL i biblioteki pg):

// db.js
import pg from "pg";
const {Pool} = pg;

const connectionString = "postgres://postgres:postgres@localhost:5432/postgres";

// 👉 Obiekt pool działa jak "brama do bazy danych" - pozwala na jednorazowe
// nawiązanie połączenia, wykonanie serii niezbędnych zapytań i zamknięcie dostępu.
const pool = new Pool({connectionString});

export default pool;
// data-access.js
import pool from "./db.js";

const db = {
  getUserById: async (id) => {
    const query = "SELECT * FROM users WHERE id = $1";
    const values = [id];
    const res = await pool.query(query, values);
    return res.rows[0];
  },
  getAllUsers: async () => {
    const query = "SELECT * FROM users";
    const res = await pool.query(query);
    return res.rows;
  },
  insertUser: async (user) => {
    const {name, email} = user;
    const query = "INSERT INTO users (name, email) VALUES ($1, $2) RETURNING *";
    const values = [name, email];
    const res = await pool.query(query, values);
    return res.rows[0];
  },
  getProductById: async (id) => {
    const query = "SELECT * FROM products WHERE id = $1";
    const values = [id];
    const res = await pool.query(query, values);
    return res.rows[0];
  },
  getAllProducts: async () => {
    const query = "SELECT * FROM products";
    const res = await pool.query(query);
    return res.rows;
  },
  insertProduct: async (product) => {
    const {name, price} = product;
    const query =
      "INSERT INTO products (name, price) VALUES ($1, $2) RETURNING *";
    const values = [name, price];
    const res = await pool.query(query, values);
    return res.rows[0];
  },
};

// 👉 Funkcja pomocnicza, która pozwoli nam zrealizować wiele operacji na jednej puli połączeń
export const useDb = async (callback) => {
  try {
    await callback(db);
  } catch (error) {
    console.error("Database operation failed:", error);
    throw error;
  } finally {
    // 👉 Po wykonaniu wszystkich operacji zamykamy połączenie z bazą
    await pool.end();
  }
};
// index.js
import {useDb} from "./data-access.js";

(async () => {
  // 👉 Obiekt "db" reprezentuje nasz moduł data-access i dostępne tam funkcje
  await useDb(async (db) => {
    try {
      const insertedUser = await db.insertUser({
        name: "Jane Doe",
        email: "jane.doe@example.com",
      });
      console.log("Inserted User:", insertedUser);

      const insertedProduct = await db.insertProduct({
        name: "Smartphone",
        price: 699.99,
      });
      console.log("Inserted Product:", insertedProduct);
    } catch (error) {
      console.error("Error during database operations:", error);
    }
  });
})();

Na potrzeby tej lekcji moduły db.js oraz index.js pozostawimy bez zmian - refaktoryzacja będzie przede wszystkim dotyczyć samej warstwy dostępu do danych, gdzie obecnie mieszają się metody z domeny użytkowników i produktów, a do tego same metody wyglądają niemal identycznie dla różnych typów obiektów.

Refaktoryzacja z użyciem TypeScript

W pierwszym kroku zajmiemy się rozdzieleniem złożonego modułu data-access.js na dwa repozytoria:

  • UserRepository - zarządzanie użytkownikami

  • ProductsRepository - zarządzanie produktami

W warstwie dostępu do danych (data-access.ts) wprowadzamy dedykowane repozytorium które będzie w sobie zawierać wszystkie dostępne operacje na kolumnie users:

// data-access.ts
import {Pool} from "pg";
import pool from "./db";

type User = {
  id?: number;
  name: string;
  email: string;
};

export class UserRepository {
  private pool: Pool;

  constructor(pool: Pool) {
    this.pool = pool;
  }

  async getUserById(id: number) {
    const query = "SELECT * FROM users WHERE id = $1";
    const values = [id];
    const res = await this.pool.query(query, values);
    return res.rows[0];
  }

  async getAllUsers() {
    const query = "SELECT * FROM users";
    const res = await this.pool.query(query);
    return res.rows;
  }

  async insertUser(user: User) {
    const {name, email} = user;
    const query = "INSERT INTO users (name, email) VALUES ($1, $2) RETURNING *";
    const values = [name, email];
    const res = await this.pool.query(query, values);
    return res.rows[0];
  }
}

    return res.rows;
  }

  async insertProduct(product: Product) {
    const {name, price} = product;
    const query =
      "INSERT INTO products (name, price) VALUES ($1, $2) RETURNING *";

    const res = await this.pool.query(query, values);
    return res.rows[0];
  }

Na koniec pozostaje nam rozszerzyć funkcję useDb o jawne typowanie i możliwość posługiwania się połączeniem do bazy (pool):

export async function useDb(callback: (pool: Pool) => Promise<void>) {
  try {
    await callback(pool);
  } catch (error) {
    console.error("Database operation failed:", error);
    throw error;
  } finally {
    await pool.end();
  }
}

W docelowej aplikacji możemy teraz zaimportować wybrane repozytorium i przy pomocy funkcji useDb zrealizować na istniejącym połączeniu z bazą wymagane operacje:

// index.ts
import {UserRepository, useDb} from "./data-access";

(async () => {
  await useDb(async (db) => {
    try {
      // 👉 Nowość - praca z dedykowanym repozytorium (przekazujemy do niego połączenie)
      const userRepo = new UserRepository(db);

      const insertedUser = await userRepo.insertUser({
        name: "Jane Doe",
        email: "jane.doe@example.com",
      });
      console.log("Inserted User:", insertedUser);

      // Pozostałe operacje na danych
    } catch (error) {
      console.error("Error during database operations:", error);
    }
  });
})();

Pierwszy krok refaktoryzacji poskutkował rozdzieleniem “god objectu” na dwa dedykowane repozytoria.

Dodatkowo, wprowadziliśmy jawne typowanie poszczególnych elementów naszej aplikacji, dzięki czemu zyskujemy opiekę kompilatora TypeScript nad wykonywanymi operacjami.

Zobaczmy teraz co zyskamy (i jakie będą tego konsekwencje) wprowadzając jeszcze jedną optymalizację.

Refaktoryzacja z generycznym repozytorium

Wprowadźmy teraz generyczne repozytorium, które będzie zawierać zestaw wspieranych przez nas operacji na poszczególnych kolumnach. O tym, jakie to kolumny i jakie typy będą przekazywane, będą decydować konsumenci repozytorium:

// repository.ts
import {Pool} from "pg";
import {Identifiable} from "./models";

export class Repository<T extends Identifiable> {
  private pool: Pool;
  private tableName: string;

  constructor(pool: Pool, tableName: string) {
    this.pool = pool;
    this.tableName = tableName;
  }

  async getById(id: number): Promise<T> {
    const query = `SELECT * FROM ${this.tableName} WHERE id = $1`;
    const values = [id];
    const res = await this.pool.query<T>(query, values);
    return res.rows[0] as T;
  }

  async getAll(): Promise<T[]> {
    const query = `SELECT * FROM ${this.tableName}`;
    const res = await this.pool.query<T>(query);
    return res.rows;
  }

  async insert(item: T): Promise<T> {
    const columns = Object.keys(item).filter((key) => key !== "id");
    const values = columns.map((_, i) => `$${i + 1}`);
    const placeholders = columns.map((col) => (item as any)[col]);

    // Np. INSERT INTO users (id, name, email) VALUES $1, $2, $3
    const query = `
      INSERT INTO ${this.tableName} (${columns.join(", ")})
      VALUES (${values.join(", ")})
      RETURNING *
    `;

    const res = await this.pool.query<T>(query, placeholders);
    return res.rows[0];
  }
}

Repozytorium generyczne będzie reprezentować zestaw operacji na określonej kolumnie. Aby kod był naprawdę uniwersalny, nazwy kolumn musimy przekazywać w konstruktorze, obok zainicjalizowanego połączenia z bazą.

Jeśli wkleiłeś ten kod do swojego edytora to widzisz, że czegoś tutaj brakuje. TypeScript wskazuje, że parametr generyczny nie jest sprecyzowany w wystarczający sposób, co psuje typowanie na przestrzeni całego repozytorium. Potrzebujemy ograniczenia na parametr naszego typu T:

// repository.ts
import {Identifiable} from "./models";

export class Repository<T extends Identifiable> {
// ...

Poniższy interfejs Identifiable wymusi pracę z obiektami, a dodatkowo pozwoli nam rozszerzyć istniejące typy o spójny identyfikator (to jedna z dostępnych opcji):

// models.ts
export interface Identifiable {
  id?: number;
}

export interface User extends Identifiable {
  name: string;
  email: string;
}

export interface Product extends Identifiable {
  name: string;
  price: number;
}

Alternatywnie, moglibyśmy wykorzystać typ QueryResultRow z biblioteki pg, który reprezentuje uniwersalny wynik operacji na bazie - na końcu musielibyśmy go rzutować na nasz model danych.

// repository.ts
import {Pool, QueryResultRow} from "pg";

export class Repository<T extends QueryResultRow> {

// node_modules/@types/pg/index.d.ts
export interface QueryResultRow {
    [column: string]: any;
}

Mając wprowadzone generyczne repozytorium możemy teraz zbudować dedykowany moduł, którym będzie się posługiwał użytkownik końcowy. Będzie on zawierał konkretne implementacje dla wspieranych typów danych oraz kolumn, które mają być z nimi połączone:

// data-access.ts
import pool from "./db";
import {Product, User} from "./models";
import {Repository} from "./repository";

class DbContext {
  users: Repository<User>;
  products: Repository<Product>;

  constructor() {
    this.users = new Repository<User>(pool, "users");
    this.products = new Repository<Product>(pool, "products");
  }
}

const ctx = new DbContext();

export async function useDb(callback: (db: DbContext) => Promise<void>) {
  try {
    await callback(ctx);
  } catch (error) {
    console.error("Database operation failed:", error);
    throw error;
  } finally {
    await pool.end();
  }
}

Nowa klasa DbContext jest opcjonalna, jednak pozwala uniknąć ręcznego zarządzania repozytoriami w docelowym kodzie aplikacji. Zamiast tego otrzymujemy wygodny sposób dostępu do danych, gdzie na obiekcie db mamy od razu dostęp do poszczególnych kolumn i wspieranych operacji:

// index.ts
import {useDb} from "./data-access";

(async () => {
  await useDb(async (db) => {
    const users = await db.users.getAll();
    console.log(users);

    const products = await db.products.getAll();
    console.log(products);

    // ...
  });
})();

Generyczne repozytorium - mocne i słabe strony

Wzorzec repozytorium, a także jego wariant generyczny, to przydatne narzędzia do rozwiązywania określonego typu problemów (utrudnione i rozproszone zarządzanie danymi, realizowane przez powtarzający się kod i trudne do zlokalizowania bugi).

W klasycznych aplikacjach typu CRUD, gdzie dla użytkownika końcowego wystawiamy tabele i formularze do modyfikacji danych, generyczne repozytorium sprawdzi się znakomicie. To jednolity interfejs do wszystkich elementów bazy danych, który pozwala łatwo i szybko wprowadzać nowe modele, a także unikać powielania logiki związanej z zarządzaniem danymi.

W tzw. “systemach głębokich”, gdzie procesowanie danych odbywa się w tle, bez wiedzy użytkownika, a także powoduje wiele efektów ubocznych, repozytorium może być ograniczeniem. Jeśli dodatkowo realizowane operacje wymagają kontekstu wielu tabel, warto rozważyć alternatywy (np. Commands & Queries) - wszystko jak zwykle zależy od problemu, który rozwiązujesz. Tym niemniej, repozytorium jako koncept zdecydowanie warto poznać, a rozszerzanie jego możliwości przez generyki to rozwijające ćwiczenie dla każdego pasjonata TypeScriptu.

Dobre praktyki w posługiwaniu się generykami

Typy generyczne to naprawdę potężny mechanizm, dzięki któremu wprowadzisz do swojego kodu więcej reużywalnych, modularnych struktur i algorytmów. Warto jednak korzystać z nich świadomie, aby uniknąć dodatkowych kosztów utrzymania projektu.

Stosuj czytelne nazewnictwo

Wcześniej wspomnieliśmy, że zgodnie z konwencją typy generyczne określamy pojedynczymi literami - T, K, V, itd.

Nic nie stoi jednak na przeszkodzie, aby w przypadku bardziej złożonych typów ich nazwa oddawała intencję programisty. Wiele projektów korzysta z takiej możliwości, co potwierdza przegląd rozwiązań dostępnych na GitHubie. Poniżej przykład z biblioteki react-redux:

export interface UseStore<StoreType extends Store> { ... }

// Tożsame z:

export interface UseStore<T extends Store> { ... }

A to przykład z biblioteki Drizzle ORM:

export interface ColumnBaseConfig<
	TDataType extends ColumnDataType,
	TColumnType extends string
> { ... }

// Tożsame z:

export interface ColumnBaseConfig<
	T extends ColumnDataType,
	K extends string
> { ... }

Podobnie wygląda to w bibliotece react-query, gdzie dodatkowo widzimy parametry domyślne dla typów generycznych:

export function useQuery<
  TQueryFnData = unknown,
  TError = DefaultError,
  TData = TQueryFnData,
  TQueryKey extends QueryKey = QueryKey,
> { ... }

// Tożsame z:

export function useQuery<
  T = unknown,
  K = DefaultError,
  V = TQueryFnData,
  Z extends QueryKey = QueryKey,
> { ... }

Na którą wersję nazewnictwa powinieneś się decydować? Tutaj znowu warto pomyśleć o perspektywie odbiorcy danego fragmentu kodu - jeśli tworzysz krótkie funkcje, typy lub klasy, to jednoliterowe typy generyczne są akceptowalne. Jeśli złożoność rośnie, a typy mają oddawać konkretne scenariusze lub przypadki użycia, wprowadzaj pełne nazewnictwo.

Unikaj nadmiarowego stosowania generyków

Zdobywanie nowej wiedzy w programowaniu często wiąże się z tym, że staramy się ją zastosować w pierwszym możliwym momencie. Przykładowo, możemy wprowadzić do naszego kodu “uniwersalną” funkcję do procesowania wszystkich dostępnych typów. W praktyce okaże się, że na poziomie implementacji pracujemy wyłącznie z jednym typem.

Może to być typ string, tak jak w funkcji poniżej, a może to być każdy inny przypadek, gdzie wprowadzony typ generyczny natychmiast opakowujemy wyrażeniami warunkowymi, które ograniczają realne pole manewru:

function processValue<T>(value: T): string {
  if (typeof value === "string") {
    return value.toUpperCase();
  } else {
    throw new Error("This function only supports strings");
  }
}

Tutaj prawdopodobnie wystarczy nam zwykła funkcja z jednym typem:

function stringToUpper(value: string): string {
  return value.toUpperCase();
}

Czasami stosowanie generyków jest nadmiarowe, a złożoność funkcji niepotrzebnie wzrasta.

Aby unikać takich nadmiernie złożonych konstrukcji, autorzy TypeScriptu zalecają stosowanie generyków tam, gdzie relacje i wiązania pomiędzy typami stają się łatwo zauważalne (np. dotyczą relacji parametrów funkcji i typu na wyjściu):

Type parameters are for relating the types of multiple values. If a type parameter is only used once in the function signature, it’s not relating anything.

Jeśli generyk ma zastąpić wyłącznie jeden typ w jednym miejscu (np. na poziomie funkcji), zastanów się nad alternatywnymi rozwiązaniami.

Zwracaj uwagę na relacje miedzy typami

Relacje pomiędzy typami to jeden z najbardziej powszechnych scenariuszy stosowania generyków, który dodatkowo wnosi realne korzyści i okazję do tworzenia prawdziwie reużywalnych narzędzi.

Jakie scenariusze mogą bazować na takich relacjach? Wyobraźmy sobie narzędzie do podglądu danych tabelarycznych (np. Google Sheets lub Airtable), gdzie na start wczytujesz wszystkie kolumny, ale na skutek filtrowania, użytkownik powinien zobaczyć wyłącznie podzbiór kolumn (zestaw pól obiektu).

Na początku zrealizujmy to bez generyków - w naszym kodzie mamy funkcję pickColumns która obsługuje relację typu User do typu zwracanego z kolumnami name i email (mapuje jeden typ na drugi):

type User = {
  id: number;
  name: string;
  email: string;
  age: number;
};

// user (User) -> result (name, email)
function pickColumns(users: User[]): { name: string; email: string }[] {
  return users.map(user => ({
    name: user.name,
    email: user.email
  }));
}

// Przykład użycia:
const users: User[] = [
  { id: 1, name: "Alice", email: "alice@example.com", age: 25 },
  { id: 2, name: "Bob", email: "bob@example.com", age: 30 }
];

console.log(pickColumns(users));
// Output: [{ name: "Alice", email: "alice@example.com" }, { name: "Bob", email: "bob@example.com" }]

Kiedy zauważymy taką relację, to możemy ją uogólnić z wykorzystaniem generyków:

  • Typ User zamienimy na parametr TBase

  • Typ TColumns posłuży do określenia podzbioru wybieranych pól (wcześniej: literał obiektu)

type TDetails = { name: string; email: string };

function mapItems<TBase extends TColumns>(items: TBase[]): TColumns[] {
  return items.map(item => ({
    name: item.name,
    email: item.email
  }));
}

Zauważ, od teraz nasza funkcja mapItems może być stosowana do typów innych niż User - tak długo, jak typ bazowy ma pola name oraz email, możemy go wykorzystywać na wejściu.

Co ciekawe, w refaktoryzacji możemy iść o krok dalej i sprawić, aby dowolne dwa typy współpracowały ze sobą w podobny sposób. Wykorzystamy do tego dwie poznane wcześnie konstrukcje:

  • extends keyof - do nadania właściwego ograniczenia na typ K

  • Pick<T, K> - do określenia typu obiektu zwracanego

function mapItems<T, K extends keyof T>(items: T[], keys: K[]): Pick<T, K>[] {
  return items.map((item) => {
    const mappedItem = {} as Pick<T, K>;
    keys.forEach((key) => {
      mappedItem[key] = item[key];
    });
    return mappedItem;
  });
}

Nasza generyczna funkcja spełnia rekomendację związaną z relacjami - wiąże ona typy T i K w taki sposób, że z jednej strony możemy je podmieniać dynamicznie, a z drugiej wciąż muszą od siebie zależeć (jeden musi być podzbiorem drugiego). W efekcie uzyskujemy:

type User = {
  id: number;
  name: string;
  email: string;
  age: number;
};

type Product = {
  id: number;
  name: string;
  price: number;
};

// Przykład użycia:
const users: User[] = [
  {id: 1, name: "Alice", email: "alice@example.com", age: 25},
  {id: 2, name: "Bob", email: "bob@example.com", age: 30},
];

const products: Product[] = [
  {id: 1, name: "Product 1", price: 10},
  {id: 2, name: "Product 2", price: 20},
];

console.log(mapItems(users, ["name"])); // [{name: "Alice"}, {name: "Bob"}]
console.log(mapItems(products, ["price"])); // [{price: 10}, {price: 20}]

Podsumowanie

W tej lekcji omówiliśmy praktyczne zastosowanie generyków w TypeScript, koncentrując się na wzorcu Repozytorium oraz jego generycznej implementacji - wiesz już kiedy warto go stosować, a kiedy unikać. Podzieliliśmy się także dobrymi praktykami, które sprawiają, że wykorzystywanie generyków nie jest nadmiarowe i prowadzi do rzeczywistego podniesienia jakości całego projektu.

W następnej lekcji zagłębimy się w zaawansowaną logikę typów, gdzie odkryjesz, jak używać złożonych elementów TypeScript, takich jak typy mapowane, warunkowe oraz rozwiązywane dynamicznie poprzez mechanizm inferencji. Zapraszamy!
