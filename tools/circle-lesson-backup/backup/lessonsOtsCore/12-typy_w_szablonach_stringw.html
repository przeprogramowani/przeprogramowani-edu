Wprowadzenie

W tej lekcji wcielimy się w programistę, który tworzy nową bibliotekę do zarządzania danymi. Jednym z wymagań jest utworzenie nowego typu, który pozwoli na zarządzanie tabelami w bazie z wykorzystaniem łatwych do zrozumienia funkcji. Przykładowo:

interface User { ... }
interface Product { ... }
interface Order { ... }

// Nasz wymagany typ - Table
type Table<Model, TableName> = ?

// Oczekiwane działanie
const usersTable: Table<User, "user">;
usersTable.createUser(...);

const productsTable: Table<Product, "product">;
productsTable.deleteProduct(...);

const ordersTable: Table<Order, "order">;
ordersTable.getOrder(...);

Czy takie zadanie można zrealizować wyłącznie jednym typem Table? Wydaje się, że obsługa wszystkich modeli danych powinna skutkować:

  • 1) jednym ogromnym interfejsem z obsługą wszystkich możliwych funkcji LUB

  • 2) tworzeniem odrębnych typów na każdy model

Innej opcji nie ma. A jednak - TypeScript ma na to sposób, a polega on na połączeniu typów mapowanych z typami opartymi o szablony stringów. Zobaczmy jak to działa.

Pierwsze kroki

Podobnie jak w przypadku standardowych stringów, w języku TypeScript typy można łączyć poprzez szablony. Nie dotyczy to jednak dowolnych typów, a literałów stringów:

type Model = "user";
type Identifier = "id";

type ModelID = `${Model}_${Identifier}`;

const userID: ModelID = "user_id";
const productID: ModelID = "product_id"; // ❌ Błąd - jedyna akceptowalna wartość to user_id

Jak widzisz, do typu ModelID przypiszemy wyłącznie stringi formatowane w jeden konkretny sposób.

Co ciekawe, ten mechanizm działa też dla wielu wartości tworzonych poprzez unie - wtedy dostępne wartości są wynikiem “mnożenia” obu składowych:

type Model = "user" | "product" | "order";
type IdField = "id";

type ModelID = `${Model}_${IdField}`; // 'user_id' | 'product_id' | 'order_id'

Utworzony w ten sposób typ (ModelID) może przyjąć trzy opisane powyżej wartości, które będą dynamicznie rozszerzane - wprowadźmy nowy model (feature) aby to sprawdzić:

type Model = "user" | "product" | "order" | "feature";
type IDField = "id";

type ModelID = `${Model}_${IDField}`; // 'user_id' | 'product_id' | 'order_id' | 'feature_id'

Po wprowadzeniu nowej wartości w typie Model, ModelID został automatycznie zaktualizowany.

Wspomniane mnożenie dobrze widać na kolejnym przykładzie - typ Model zawiera trzy wartości, Field zawiera dwie wartości, więc FieldModel aż sześć - to połączenie każdego z akceptowanych typów:

type Model = "user" | "product" | "order";
type Field = "id" | 'createdAt';

// user_id | product_id | order_id | user_createdAt | product_createdAt | order_createdAt
type FieldModel = `${Model}_${Field}`;

Gdzie taki mechanizm może się przydać?

Wyobraźmy sobie, że tworzymy system do rekrutacji, który procesuje specjalne kody, z których można wydedukować informację o dziale firmy i doświadczeniu kandydata. Na końcu nadawany jest numer otwartej pozycji (wykorzystamy typ prosty number jako symbol wieloznaczny, tzw. wildcard).

type Department = "sales" | "marketing" | "engineering";
type Seniority = "junior" | "mid" | "senior";

type HiringCode = `${Department}_${Seniority}_${number}`;

const juniorSalesRole: HiringCode = "sales_junior_021"; // ✅ OK
const midEngineeringRole: HiringCode = "engineering_mid_554"; // ✅ OK
const seniorMarketingRole: HiringCode = "marketing_senior_228"; // ✅ OK

Wykorzystując ten specjalistyczny typ możemy się zabezpieczyć przed sytuacjami, kiedy inny moduł w tym samym systemie zacznie procesować identyfikatory złego formatu (odwrócony dział i seniority):

function resolveHiringCode(code: HiringCode) {
  const [department, seniority, id] = code.split("_");
  // ...
  // Dodatkowe procesowanie wykrytych wartości
}

resolveHiringCode("junior_sales_15"); 
// ❌ Błąd - Argument of type '"junior_sales_15"' is not assignable

👉 Co ważne - mówimy tutaj o zgodności typów, a nie wartości przekazywanych w trakcie działania aplikacji. Tam, niestety bez naszej wiedzy, wciąż może się zdarzyć użycie niewłaściwego formatu. Przed tym powinny nas przestrzegać dodatkowe funkcje walidacyjne.

Operacje na literałach stringów

Korzystając z typów opartych o literały stringów, do dyspozycji mamy cztery wbudowane funkcje, które wykonują dla nas modyfikację poszczególnych znaków:

type Model = "User";

type UpperModel = Uppercase<Model>; // "USER" (Wszystko z wielkiej litery)
type LowerModel = Lowercase<Model>; // "user" (Wszystko z małej litery)
type CapitalizeModel = Capitalize<Model>; // "User" (Pierwszy znak z wielkiej litery)
type UncapitalizeModel = Uncapitalize<Model>; // "user" (Pierwszy znak z małej litery)

Będziemy z nich korzystać budując docelowe API naszej biblioteki do zarządzania danymi.

Literały i typy mapowane

Opisywane w tej lekcji typy pokazują cały potencjał w momencie kiedy połączymy je z typami mapowanymi - to szczególnie interesująca możliwość dla twórców wszelkiego rodzaju bibliotek i narzędzi, którzy mogą dynamicznie kształtować API swoich rozwiązań (a właściwie - dają przepis, z którego korzystają później użytkownicy).

Przykładowo - wprowadźmy typ, który wprowadza zestaw getterów dla bazowego typu Type:

// Wszystkie klucze z Type zamieniamy na funkcje
type Getters<Type> = {
  [Key in keyof Type as `get${Capitalize<string & Key>}`]: () => Type[Key];
};

Powyższy typ będzie opisywał obiekty, w których znajdują się wyłącznie funkcje o nazwach dopasowanych do pól typu bazowego.

Rozłóżmy to na czynniki pierwsze:

  • Key in keyof Type - Dla wszystkich pól w obiekcie o typie Type

  • as - przeprowadź mapowanie

  • `get${Capitalize<…>} - wprowadź pola o nowej nazwie - get<Nazwa>

  • string & Key - zagwarantuję Ci, że klucze będą stringami

  • [ … ]: () => Type[Key]; - każde pole to od teraz funkcja zwracająca ten sam typ co pole bazowe

Dzięki takiej konstrukcji uzyskamy dynamicznie formatowane nazwy, które będą tworzone zgodnie z tym samym wzorcem:

  • id => getId()

  • name => getName()

  • age => getAge();

Dodatkowo mamy gwarancję, że zwracany typ będzie taki sam jak typ pola bazowego. A w praktyce:

interface Person {
  id: number;
  name: string;
}

const personGetters: Getters<Person> = {
  getName: () => "John",
  getId: () => 1,
};

interface Product {
  quantity: number;
}

const productGetters: Getters<Product> = {
  getQuantity: () => 10,
};

Robi się ciekawie!

Zarządzanie tabelami

W tym momencie mamy do dyspozycji trzy narzędzia:

  • typy mapowane

  • typy oparte o szablony stringów

  • wbudowane typy pomocnicze

Możemy je wykorzystać do zbudowania docelowej struktury:

Kod z filmu znajdziesz poniżej:

type Entities = "user" | "product" | "order";

interface User {
  id: number;
  name: string;
  email: string;
}

interface Product {
  id: number;
  name: string;
  price: number;
}

interface Order {
  id: number;
  productId: number;
  quantity: number;
}

type CreateMethod<TModel, TName extends Entities> = {
  [Prop in `create${Capitalize<TName>}`]: (data: Partial<TModel>) => void;
};

type ReadMethod<TModel, TName extends Entities> = {
  [Prop in `get${Capitalize<TName>}`]: (id: number) => TModel | null;
};

type UpdateMethod<TModel, TName extends Entities> = {
  [Prop in `update${Capitalize<TName>}`]: (
    id: number,
    data: Partial<TModel>
  ) => void;
};

type DeleteMethod<_, TName extends Entities> = {
  [Prop in `delete${Capitalize<TName>}`]: (id: number) => void;
};

type Table<TModel, TName extends Entities> = 
  CreateMethod<TModel, TName> &
  ReadMethod<TModel, TName> &
  UpdateMethod<TModel, TName> &
  DeleteMethod<TModel, TName>;

declare const usersTable: Table<User, "user">;
declare const productsTable: Table<Product, "product">;
declare const ordersTable: Table<Order, "order">;

productsTable.deleteProduct(1);
ordersTable.getOrder(1);
usersTable.updateUser(1, { name: "John" });

Indeksy o ściśle określonym formacie

Jeszcze inny, wykorzystywany w praktyce scenariusz dla typów opartych o szablony, dotyczy typowania komponentów lub węzłów drzewa DOM, których atrybuty powinny być dopasowane do ściśle określonych standardów.

Przyjęło się, że rozszerzanie elementów DOM o niestandardowe atrybuty powinno być realizowane prefixem data- (tzw. data attributes):

<div data-access="false" data-priority="high"></div>

Takich atrybutów na każdym elemencie może być więcej. Dzięki przedstawionym w tej lekcji szablonom możemy sprawić, żeby typowana reprezentacja każdego węzła trzymała się właśnie takiego standardu.

Jak to zrobić? Używając indeksów:

type HTMLNode = {
    id: number;
    name: string;
    [key: `data-${string}`]: string;
    [key: `x-marker-${string}`]: number;
}

Tym sposobem wskazujemy, że obiekty typu HTMLNode wspierają indeksy zaczynające się od dwóch prefixów - standardowego data-, oraz wprowadzonego przeze mnie x-marker-. Możemy ich użyć w momencie tworzenia obiektu, a także już po tym fakcie:

const MyComponent: HTMLNode = {
    id: 12,
    name: 'div',
    ['data-testid']: 'topbar',
    ['data-priority']: 'high',
    ['x-marker-loadtime']: 21124
}

MyComponent['data-access'] = 'false';

Na poziomie indeksów mamy pełne wsparcie dla weryfikacji typów i wartości, jakie do nich przypisujemy.

const loadtime: number = MyComponent['x-marker-loadtime'];
console.log(loadtime); //21124

Jeśli wypiszemy teraz na konsolę wszystkie pola takiego obiektu, to otrzymamy:

for (let prop in MyComponent) { 
    console.log(prop); // id, name, data-testid, data-priority, x-marker-loadtime, data-access
}

Dodatkowo, TypeScript zabezpieczy nas przed indeksowaniem nieznanymi nazwami:

MyComponent['does-not-exist'] = 15;
// ❌ Expression of type '"does-not-exist"' can't be used to index type 'HTMLNode'

Szablony to potężny mechanizm, który może nam pomóc tworzyć bezpieczne typy, przy jednoczesnym zachowaniu elastyczności dla ich konsumenta.

Ćwiczenia

Pierwsza to implementacja typu Code, który będzie przyjmował “trójki” cyfr 0 i 1 połączone w formacie “{group}-{group}-{group}”, np. “110-001-101”. Po ukończeniu implementacji, drugim krokiem jest naprawa funkcji codeToDecimal, która konwertuje każdą z trójek na liczbę w formacie dziesiętnym, a następnie zwraca trzyznakowy kod dostępu. Zapoznaj się z testami i przygotuj niezbędne zmiany.

  2. 422-literals-table - Wprowadzony w trakcie lekcji typ Table przyjmuje obecnie dwa parametry - model danych oraz nazwę tabeli w postaci stringa. Zaproponuj własny format typów, które umożliwiają wykorzystywanie jednego parametru - modelu danych. Zastanów się jak definiować mapowanie między modelem a nazwą tabeli w taki sposób, żeby istniejąca funkcjonalność pozostała bez zmian.
