TypeScript łączy dynamiczność JavaScriptu ze statycznym typowaniem, co pozwala pisać bardziej przewidywalny kod. W tej lekcji skupimy się na zaawansowanych funkcjach języka, takich jak "ewoluujące any", dynamiczne typowanie struktur danych, zarządzanie nadmiarowymi właściwościami oraz różnice między otwartymi i zamkniętymi typami obiektów. Dzięki temu nauczysz się unikać typowych błędów i lepiej projektować typy w swoich aplikacjach.

Mechanizm "ewoluującego any"

TypeScript pozwala zmiennym zadeklarowanym z użyciem let przyjąć typ na podstawie przypisywanej wartości. Jeśli brak jest początkowej adnotacji typu, typowanie zmiennej ewoluuje w miarę przypisywania różnych wartości. Mechanizm ten bazuje na dynamicznej inferencji typu.

let myVar; // Typ: any
myVar = 42; 
console.log(myVar.toExponential()); // Wywołuje metodę specyficzną dla liczby

myVar = "tekst";
console.log(myVar.toUpperCase()); // Wywołuje metodę specyficzną dla stringa

Jest to zgodne z dynamicznym charakterem JavaScriptu, ale wymaga ostrożności, aby uniknąć nieprzewidywalnych błędów.

W powyższym przykładzie TypeScript automatycznie zmienia typ myVar na podstawie przypisanych wartości.

Mechanizm "ewoluującego any" ma także zastosowanie w tablicach. Jeśli tablica zostanie zadeklarowana bez przypisania jej typu, TypeScript automatycznie rozszerzy jej typ na podstawie zawartych w niej elementów. Początkowo taka tablica jest traktowana jako pusta i niezdefiniowana pod względem typu. W momencie dodawania elementów, TypeScript tworzy unie typów, które obejmują wszystkie przypisane wartości.

const evolvingArray = [];
evolvingArray.push(123);  // Typ tablicy: number[]
evolvingArray.push("tekst"); // Typ tablicy: (number | string)[]

Choć to zachowanie zapewnia dużą elastyczność, wprowadzenie bardziej precyzyjnego typowania w początkowej deklaracji może znacznie poprawić przewidywalność kodu.

Aby zachować bezpieczeństwo typów, warto zdefiniować typ tablicy od razu:

const numberArray: number[] = [];
numberArray.push(123); // Działa poprawnie
numberArray.push("tekst"); // Błąd kompilacji: Typ 'string' nie jest przypisywalny do 'number'

Deklarując typ tablicy na number[], upewniasz się, że w tablicy znajdą się tylko liczby, co znacząco zwiększa czytelność i stabilność kodu.

Ewoluujące tablice

Podobnie jak w przypadku pojedynczych zmiennych, TypeScript potrafi dynamicznie określać typ tablic na podstawie jej zawartości. Jeśli na początku tablica nie ma określonego typu, jej typ ewoluuje w miarę dodawania nowych elementów.

Rozważmy tablicę, do której dodawane są różne typy elementów:

const evolvingArray = []; // Typ: any[]
evolvingArray.push(123);  // Typ: number[]
evolvingArray.push("tekst"); // Typ: (number | string)[]

Po dodaniu liczby do pustej tablicy, jej typ zmienia się na number[]. Gdy dodamy do niej tekst, TypeScript automatycznie rozszerza typ tablicy na number | string.

Aby uniknąć nieprzewidywalnych błędów, warto zdefiniować typ tablicy od razu:

const numberArray: number[] = [];
numberArray.push(123); // Działa poprawnie
numberArray.push("tekst"); // Błąd kompilacji: Typ 'string' nie jest przypisywalny do 'number'

Deklarując typ tablicy na number[], upewniasz się, że w tablicy znajdą się tylko liczby, co znacząco zwiększa czytelność i stabilność kodu.

Otwarte i zamknięte typy obiektów

W TypeScript obiekty są domyślnie traktowane jako otwarte w kontekście ogólnej filozofii języka. Jednak w praktyce, kiedy definiujesz obiekt za pomocą literału, jego typ jest zamykany do dokładnie tych właściwości, które zostały zdefiniowane. Przykładowo:

const user = { name: "Alice" }; // Typ: { name: string }
user.age = 25; // Błąd: Property 'age' does not exist on type '{ name: string; }'.

TypeScript przypisuje obiektowi dokładny typ { name: string }, co oznacza, że próba dodania nowej właściwości, takiej jak age, skutkuje błędem. To zachowanie różni się od JavaScriptu, gdzie można swobodnie dodawać nowe właściwości do obiektów.

Indeksowane typu obiektów

Jednym z rozwiązań jest zadeklarowanie obiektu jako mapy klucz-wartość:

type User = { [key: string]: unknown }; 

const user: User = { name: "Alice" };
user.age = 25; // Działa poprawnie, ponieważ typ User pozwala na dowolne klucze
console.log(user); // { name: "Alice", age: 25 }

W tym przypadku mówimy TypeScriptowi, że obiekt może zawierać dowolne klucze ([key: string]) i dowolne wartości (any). Jednak stosowanie any osłabia mechanizmy typowania, dlatego lepszym rozwiązaniem może być użycie bardziej precyzyjnych typów.

Użycie Record

Możesz również użyć typu Record, aby określić typ kluczy i wartości:

const user: Record<string, string | number> = { name: "Alice" };
user.age = 25; // Działa poprawnie
console.log(user); // { name: "Alice", age: 25 }

W tym przykładzie obiekt user akceptuje klucze typu string oraz wartości będące string lub number.

Otwarte typy a iterowanie po kluczach

Otwarte typy mogą prowadzić do problemów, kiedy iterujesz po kluczach obiektu, ponieważ TypeScript nie zakłada, że wszystkie klucze są zdefiniowane w typie.

const album = { title: "Revolver", releaseYear: 1966 };
for (const key in album) {
  console.log(album[key]); // No index signature with a parameter of type 'string' was found on type '{ title: string; releaseYear: number; }'
}

Aby uniknąć błędów, możesz jawnie wskazać, że klucze odpowiadają właściwościom obiektu, na przykład za pomocą operatora keyof:

for (const key in album) {
  console.log(album[key as keyof typeof album]); // Bez błędu
}

Jak zamknąć typy w TypeScript

Aby uniemożliwić dodawanie nowych właściwości do obiektu, możesz zastosować operator as const lub wprowadzić bardziej restrykcyjny typ, który wymaga zgodności z dokładnym zestawem właściwości.

Użycie as const

Jeśli zależy ci na tym, aby obiekt był niemodyfikowalny, możesz oznaczyć go jako stały (const assertion):

const user = { name: "Alice" } as const;
user.name = "Bob"; // Error: Cannot assign to 'name' because it is a read-only property.

Powyższe podejście nie tylko blokuje dodawanie nowych właściwości, ale również zmienia istniejące właściwości na tylko do odczytu (readonly).

Nadmiarowe właściwości w obiektach

TypeScript czasami ignoruje dodatkowe właściwości w obiektach, co może prowadzić do zaskakujących sytuacji. Rozważmy prosty interfejs Album, który definiuje dwa pola: title i releaseYear.

interface Album {
  title: string;
  releaseYear: number;
}

Jeśli przypiszesz obiekt zawierający dodatkową właściwość do zmiennej, a następnie przekażesz tę zmienną do funkcji, TypeScript nie zgłosi błędu:

function processAlbum(album: Album) {
  // Do something with album
}

const album = {
  title: "Rubber Soul",
  releaseYear: 1965,
  label: "Parlophone", // Nadmiarowa właściwość
};

processAlbum(album); // Wszystko w porządku

Ale jeśli przekażesz ten sam obiekt bezpośrednio do funkcji, TypeScript zacznie protestować:

processAlbum({
  title: "Revolver",
  releaseYear: 1966,
  label: "EMI" // Błąd: "label" nie istnieje w typie Album
});

Dlaczego tak się dzieje? Kiedy obiekt jest przypisywany do zmiennej, TypeScript zakłada, że może być używany w innych miejscach, gdzie nadmiarowe właściwości są potrzebne. Ale w przypadku literałów obiektowych, które są używane tylko w jednym miejscu, TypeScript może sprawdzić, czy wszystkie właściwości pasują do zdefiniowanego interfejsu.

Jeśli chcesz wymusić sprawdzanie nadmiarowych właściwości, możesz użyć operatora satisfies:

const album = {
  title: "Abbey Road",
  releaseYear: 1969,
  label: "Apple",
} satisfies Album; // Błąd: "label" nie istnieje w Album

Alternatywnie, możesz użyć jawnej adnotacji typu:

const album: Album = { 
  title: "Help!", 
  releaseYear: 1965, 
  label: "EMI" // Błąd: "label" nie istnieje w typie Album
};

Podsumowanie

Podczas tej lekcji poznaliśmy zaawansowane aspekty TypeScript, takie jak "ewoluujące any", dynamiczne typowanie struktur danych oraz mechanizmy zarządzania nadmiarowymi właściwościami w obiektach. Dzięki tym technikom możesz pisać kod, który lepiej odzwierciedla dynamiczność JavaScriptu, ale jednocześnie jest bardziej przewidywalny. Warto stosować jawne typowanie wszędzie tam, gdzie to możliwe, aby uniknąć nieprzewidywalnych błędów.
