# Research: Allow launching preferred dialogues by available keys in QaOverlay.svelte

**Date**: 2026-03-30T12:00:00+02:00
**Researcher**: Claude
**Git Commit**: 959167aa
**Branch**: master
**Repository**: przeprogramowani-sites

## Research Question

How to add dialogue launching capability to the QaOverlay, allowing testers to trigger any available dialogue by key.

## Summary

The dialogue system is **event-driven** — any component with access to the Phaser game event bus can trigger a dialogue by emitting `GameEvents.DIALOGUE_SHOW` with a `{ dialogueId: string }` payload. The QaOverlay already has access to `game.events` (the bus) and uses similar patterns for map jumping and state manipulation. All dialogue keys are accessible at runtime via `getAllDialogues()` from levelLoader. Adding a dialogue launcher requires:

1. Import `getAllDialogues` from levelLoader and `GameEvents` (already imported)
2. Add a `<select>` + button UI section (matching existing QaOverlay patterns)
3. Emit `GameEvents.DIALOGUE_SHOW` on button click

No new public methods on GameScene are needed.

## Detailed Findings

### 1. Dialogue Triggering API

The **only thing needed** to start a dialogue is emitting an event on the game bus:

```typescript
game.events.emit(GameEvents.DIALOGUE_SHOW, { dialogueId: 'some-key' });
```

**Event definition** (`src/explorers/events/GameEvents.ts:15,83-85`):
```typescript
DIALOGUE_SHOW: 'dialogue:show',

export interface DialogueShowPayload {
  dialogueId: string;
}
```

**Listener** (`src/explorers/scenes/DialogueScene.ts:33-36`):
- DialogueScene listens for `DIALOGUE_SHOW` and calls `showDialogue(dialogueId)`
- DialogueManager looks up the sequence in the global `allDialogues` map
- If found, renders it; if not, warns and emits `DIALOGUE_DISMISSED`

### 2. Existing Patterns in QaOverlay

The QaOverlay already:
- Has `game: Phaser.Game` prop with access to `game.events` (the bus)
- Emits events: `GameEvents.TRANSITION_START` for map jumping (line 119)
- Uses `getMapDisplayNames()` from levelLoader for map listing (line 5, 112)
- Has `blurQaTarget()` and `preventQaButtonFocus()` helpers for all interactive controls
- Uses select + button pattern for flag addition and map jumping

### 3. Getting Available Dialogue Keys

**From levelLoader** (`src/explorers/levels/levelLoader.ts:154-156`):
```typescript
export function getAllDialogues(): Map<string, DialogueSequence> {
  return allDialogues;
}
```

This returns ALL loaded dialogues across all levels + rank-up dialogues. Keys are globally unique (loader throws on duplicates).

**Total dialogue count**: ~47 unique sequences across 4 levels plus rank-up dialogues.

### 5. Important Caveats

**Side effects**: Many dialogues have `onComplete` effects (set flags, grant XP, activate/complete quests, open URLs). Triggering these from QA will apply those effects. This is actually **desirable for testing** but worth noting in the UI.

**Player state during dialogue**: When GameScene triggers a dialogue via `startDialogue()`, it:
- Sets `inDialogue = true`
- Puts player in `'dialogue'` state
- Disables input controller
- Freezes NPCs

When QaOverlay emits `DIALOGUE_SHOW` directly, **only DialogueScene responds** — the GameScene won't freeze the player or disable input because it doesn't know a dialogue started through its own `startDialogue()` method. The dialogue will still display and work, but the player may be able to move during it. This is acceptable for QA testing.

**Dialogue completion**: When the dialogue finishes, DialogueScene emits `DIALOGUE_DISMISSED`. GameScene listens for this to restore player control — but since GameScene didn't set `inDialogue = true`, the restore logic won't fire either. This is fine for QA-triggered dialogues.

### 6. Suggested Implementation

Add a new section to QaOverlay between the "Jump to map" section and the zone debug button:

```svelte
<!-- In <script> -->
import { getAllDialogues } from './levels/levelLoader';

let selectedDialogue = '';

function getAvailableDialogues(): Array<{ key: string }> {
  const dialogues = getAllDialogues();
  return [...dialogues.keys()].sort().map(key => ({ key }));
}

function launchDialogue(dialogueId: string) {
  if (!dialogueId) return;
  game.events.emit(GameEvents.DIALOGUE_SHOW, { dialogueId });
}

function onLaunchDialogueClick(event: MouseEvent) {
  launchDialogue(selectedDialogue);
  blurQaTarget(event);
}

<!-- In template -->
<div>
  <label class="text-gray-500" for="qa-dialogue-launch">Launch dialogue:</label>
  <div class="flex gap-1 mt-0.5">
    <select
      id="qa-dialogue-launch"
      class="flex-1 px-1.5 py-1 bg-gray-800/80 border border-gray-600/50 rounded
             text-[11px] text-gray-300 cursor-pointer outline-none focus:border-teal-600"
      bind:value={selectedDialogue}
      on:mousedown|stopPropagation
      on:change={blurQaTarget}
    >
      {#each getAvailableDialogues() as d}
        <option value={d.key}>{d.key}</option>
      {/each}
    </select>
    <button
      type="button"
      on:mousedown={preventQaButtonFocus}
      on:click={onLaunchDialogueClick}
      class="px-2 py-1 bg-purple-900/60 hover:bg-purple-800/80 border border-purple-700/50
             rounded text-purple-300 text-xs font-bold cursor-pointer transition-colors"
    >Play</button>
  </div>
</div>
```

## Architecture Insights

1. **Event bus is the universal API** — All dialogue triggers (GameScene, SmartTerminal, QuestManager, ExamManager, ArcadeScene, PhaserGame.svelte) use the same `game.events.emit(GameEvents.DIALOGUE_SHOW, { dialogueId })` pattern. QaOverlay should follow the same convention.

2. **Global dialogue namespace** — All dialogues are loaded into a single flat map at boot time. No scoping by level is needed to trigger them — any ID works from anywhere.

3. **QaOverlay already has the bus** — The `game` prop provides `game.events`, which is the same bus all scenes and managers use.

4. **Existing UI patterns** — The "Jump to map" select + action button pattern is directly reusable. The flag add section is another good template.

## Open Questions

1. **Should dialogues be grouped by level in the dropdown?** — With ~50 keys, a flat sorted list works but `<optgroup>` by level could improve usability.
2. **Should the QA overlay show dialogue metadata?** — e.g., line count, mode types, whether it has onComplete effects — useful for QA but adds complexity.
3. **Should there be a search/filter input?** — With 50+ dialogues, a text filter could be helpful.
