Ocena przez LLM (Large Language Models) to potężna technika w prompt engineeringu, która wykorzystuje zdolność modeli językowych do analizy, rozumienia i oceny różnorodnych treści, w tym kodu, pomysłów i decyzji. Ta sekcja skupi się na dwóch kluczowych technikach: self-criticism (samokrytyka) oraz "Panel of Experts" (panel ekspertów).

Przykładowe zastosowania

LLM-y, dzięki swojej zdolności do rozumienia kontekstu i wyciągania wniosków, mogą służyć jako wartościowe narzędzie do oceny kodu, projektów interfejsu użytkownika czy decyzji architektonicznych w web developmencie. Ocena przez LLM może:

  • Dostarczyć szybkiej, wstępnej analizy jakości kodu

  • Zidentyfikować potencjalne problemy z użytecznością interfejsu

  • Ocenić zgodność rozwiązań z najlepszymi praktykami branżowymi

  • Symulować różne perspektywy przy ocenie decyzji projektowych

Poniżej znajdziesz dwie popularne techniki oceny przez LLM: samokrytyka oraz panel ekspertów.

Self-criticism (Samokrytyka)

Self-criticism to technika, w której LLM jest proszony o ocenę i krytykę własnych wcześniejszych odpowiedzi lub wygenerowanych treści.

Jak działa self-criticism?

  1. Generowanie początkowej odpowiedzi: LLM tworzy pierwszą wersję kodu, projektu lub rozwiązania.

  2. Prompt do samokrytyki: Model otrzymuje instrukcję do oceny swojej własnej odpowiedzi.

  3. Analiza i feedback: LLM analizuje swoją pracę, identyfikując mocne strony i obszary do poprawy.

  4. Iteracyjne udoskonalanie: Na podstawie samokrytyki, model może zaproponować ulepszenia lub alternatywne rozwiązania.

Przykładowa konwersacja z Self-criticism

Początkowy prompt:

Wygeneruj funkcję JavaScript do walidacji adresu email.

Model generuje funkcje, na co my odpowiadamy:

Dokonaj krytycznej analizy swojego kodu, wskazując jego mocne strony i potencjalne obszary do poprawy.

Model odpowiada:

Naturalnie. Przeprowadzę krytyczną analizę ulepszonej wersji funkcji walidacji adresu email.

Mocne strony:
1. Kompleksowość: Funkcja teraz obejmuje więcej aspektów walidacji, co czyni ją bardziej niezawodną.
2. Obsługa błędów: Dodano sprawdzanie typu wejściowego, co zapobiega nieoczekiwanym błędom przy nieprawidłowych danych wejściowych.
3. Walidacja długości: Wprowadzono sprawdzanie minimalnej i maksymalnej długości adresu email, co eliminuje oczywiste przypadki nieprawidłowych adresów.
4. Ulepszone wyrażenie regularne: Nowe wyrażenie regularne jest bardziej precyzyjne i obsługuje szerszy zakres poprawnych formatów adresów email.
5. Przykłady użycia: Dodano więcej przykładów testowych, co pomaga w zrozumieniu działania funkcji i ułatwia testowanie.

Obszary do poprawy:
1. Złożoność wyrażenia regularnego: Chociaż nowe wyrażenie regularne jest bardziej dokładne, jest też trudniejsze do zrozumienia i utrzymania.
2. Brak modularności: Funkcja wykonuje kilka zadań (sprawdzanie typu, długości, dopasowanie do wzorca). Można by rozważyć podzielenie jej na mniejsze, bardziej specyficzne funkcje.
3. Ograniczona informacja zwrotna: Funkcja zwraca tylko wartość boolean, co nie daje szczegółowych informacji o tym, dlaczego walidacja się nie powiodła.
4. Brak walidacji specyficznych dla domeny: Funkcja nie sprawdza, czy domena najwyższego poziomu jest prawidłowa lub czy istnieje.
5. Statyczne limity długości: Obecne limity długości są sztywno zakodowane i mogą nie być odpowiednie dla wszystkich przypadków użycia.

Propozycje dalszych ulepszeń:

1. Modularyzacja:
function isString(input) {
  return typeof input === 'string';
}

function isValidLength(email, minLength = 3, maxLength = 320) {
  return email.length >= minLength && email.length <= maxLength;
}

function matchesEmailPattern(email) {
  const emailRegex = /^[a-zA-Z0-9.!#$%&'*+/=?^_`{|}~-]+@[a-zA-Z0-9](?:[a-zA-Z0-9-]{0,61}[a-zA-Z0-9])?(?:\.[a-zA-Z0-9](?:[a-zA-Z0-9-]{0,61}[a-zA-Z0-9])?)*$/;
  return emailRegex.test(email);
}

function validateEmail(email) {
  if (!isString(email)) {
    throw new Error('Email must be a string');
  }
  return isValidLength(email) && matchesEmailPattern(email);
}

2. Bardziej szczegółowa informacja zwrotna:
function validateEmail(email) {
  if (!isString(email)) {
    return { isValid: false, reason: 'Email must be a string' };
  }
  if (!isValidLength(email)) {
    return { isValid: false, reason: 'Email length is invalid' };
  }
  if (!matchesEmailPattern(email)) {
    return { isValid: false, reason: 'Email format is invalid' };
  }
  return { isValid: true };
}

Podsumowując, ulepszona wersja funkcji jest znacznie lepsza od oryginalnej, ale nadal istnieje przestrzeń do dalszych udoskonaleń, szczególnie w zakresie modularności, elastyczności i szczegółowości informacji zwrotnej. Wybór konkretnych ulepszeń zależałby od specyficznych wymagań projektu, w którym funkcja miałaby być używana.

🥳 Korzyści

  • Zwiększona dokładność: Poprzez krytyczną analizę własnych odpowiedzi, LLM może dostarczyć bardziej przemyślane i dopracowane rozwiązania.

  • Identyfikacja słabych punktów: Samokrytyka pomaga w wykrywaniu potencjalnych problemów, które mogły zostać przeoczone w początkowej odpowiedzi.

  • Uczenie się na błędach: Proces ten symuluje iteracyjny charakter developmentu, gdzie kod jest stale analizowany i udoskonalany.

🫣 Wyzwania

  • Potencjalna stronniczość: LLM może być stronniczy wobec własnych odpowiedzi, co może prowadzić do niedostatecznej krytyki lub nadmiernej pewności siebie.

  • Ograniczenia wiedzy: Model może nie być świadomy najnowszych praktyk lub specyficznych dla danego projektu wymagań, co może prowadzić do niepełnej lub nieodpowiedniej krytyki.

  • Brak kontekstu zewnętrznego: Self-criticism opiera się wyłącznie na wiedzy modelu, bez uwzględnienia szerszego kontekstu projektu czy specyficznych wymagań klienta.

  • Ryzyko nadmiernej iteracji: Ciągłe stosowanie self-criticism może prowadzić do zbyt wielu iteracji i potencjalnego odejścia od pierwotnych celów zadania.

🧑🏻‍💻 Ćwiczenie

Jesteś senior web developerem pracującym nad nowym projektem e-commerce. Twoim zadaniem jest stworzenie funkcji JavaScript do walidacji formularza zamówienia.

Wykorzystaj technikę self-criticism, aby wygenerować, przeanalizować i udoskonalić funkcję walidacji formularza zamówienia.

Kroki

  1. Sformułuj początkowy prompt: Napisz prompt, który poprosi LLM o stworzenie funkcji JavaScript do walidacji formularza zamówienia. Twój prompt powinien zawierać:

    • Krótki opis formularza (np. pola: imię, nazwisko, adres email, adres dostawy, numer telefonu)

    • Prośbę o uwzględnienie podstawowych zasad walidacji

    • Sugestię użycia komentarzy wyjaśniających logikę

  2. Analiza wygenerowanego kodu: Po otrzymaniu kodu od LLM, przeanalizuj go pod kątem:

    • Kompletności (czy wszystkie pola są walidowane?)

    • Czytelności i struktury kodu

    • Potencjalnych luk w bezpieczeństwie

  3. Prompt do self-criticism: Napisz prompt, który poprosi LLM o krytyczną analizę wygenerowanego kodu. Prompt powinien zachęcać do:

    • Wskazania mocnych stron kodu

    • Identyfikacji obszarów wymagających poprawy

    • Sugestii konkretnych ulepszeń

  4. Ocena self-criticism: Przeanalizuj odpowiedź LLM zawierającą samokrytykę. Zwróć uwagę na:

    • Trafność zidentyfikowanych problemów

    • Jakość i wykonalność proponowanych ulepszeń

    • Aspekty, które mogły zostać pominięte

  5. Iteracja (opcjonalnie): Na podstawie self-criticism, możesz poprosić LLM o wygenerowanie ulepszonej wersji funkcji.

  6. Refleksja: Zastanów się nad procesem i odpowiedz na następujące pytania:

    • Jak technika self-criticism wpłynęła na jakość końcowego rozwiązania?

    • Jakie korzyści i wyzwania napotkałeś podczas stosowania tej techniki?

    • Jak możesz wykorzystać self-criticism w swojej codziennej pracy jako web developer?

Wskazówki

  • Zwróć szczególną uwagę na bezpieczeństwo i obsługę różnych przypadków brzegowych

  • Pamiętaj o zasadach czystego kodu i najlepszych praktykach JavaScript

  • Rozważ, jak funkcja mogłaby być rozszerzona lub zmodyfikowana w przyszłości

Kryteria sukcesu

  • Wygenerowany kod jest funkcjonalny i spełnia podstawowe wymagania

  • Self-criticism LLM identyfikuje rzeczywiste problemy i proponuje sensowne ulepszenia

  • Jesteś w stanie krytycznie ocenić zarówno wygenerowany kod, jak i samokrytykę LLM
