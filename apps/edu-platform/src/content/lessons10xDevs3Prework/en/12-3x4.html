In the previous prework lesson, you learned how to manage the thread's lifecycle and how the context gets exhausted with each turn. Now it's time for a decision that sets this budget from the start - in which language should you write to the agent?

You open the editor, copy the error from the console, and you are supposed to write: "Make this an asynchronous job and push it." But wait - since the latest models understand Polish perfectly, why struggle with... broken technical English?

The answer in 2026 is purely engineering: the context window budget, API cost, and noise in mapping system concepts.

It's not that Polish is "worse." The point is that as a 10xDevs participant, you will work with agents for a long time, in many rounds, and often on a real codebase, so the language becomes part of the ergonomics of the entire workflow.

During 10xDevs we will return to this topic when designing repository rules, working in branches, PR reviews, and building our own contracts with the agent. For now, let's establish a practical starting rule.

Tokenization does not like Polish

Every text you enter into the agent - in the Claude Code terminal or in the Cursor window - passes through a tokenizer before reaching the neural network layers.

In practice, you will encounter various tokenizers today, often based on approaches from the BPE or unigram family. Details differ between models, but the effect for the programmer is usually similar: English technical vocabulary is often packed more efficiently than its Polish equivalents.

This asymmetry results in a measurable phenomenon - fertility ratio, that is the number of tokens needed for the model to "absorb" one recorded word.

flowchart LR
    A["Word: zaimplementowaliśmy"] --> B["Text fragment: za"]
    A --> C["Text fragment: implement"]
    A --> D["Text fragment: owali"]
    A --> E["Text fragment: śmy"]
    B --> B1["Token ID: number"]
    C --> C1["Token ID: number"]
    D --> D1["Token ID: number"]
    E --> E1["Token ID: number"]

For English in IT topics, many concepts fit into single tokens. For Polish, due to inflection, endings, and less typical character clusters, this cost often increases.

You don't need to remember a specific ratio. Practical intuition is enough: the same instruction in Polish will often take more tokens than its English equivalent.

Since modern models have windows ranging from 200 thousand to 2 million tokens, why should this difference matter to you? The secret lies in context rot, which we talked about in the previous lesson.

The more tokens land in the session history – especially when the agent runs long analytical loops, pastes logs, and performs tests with errors – the faster the model's attention fades.

By forcing Polish for every command prompt, you voluntarily approach the moment when you will have to do /compact or /clear. It seems harmless, but in the middle of refactoring, it can hurt.

To see this overhead, do a simple test in the OpenAI tokenizer. For tiktoken o200k_base this example looks like this:

Operation CommandInput Text (Prompt)Number of Tokens
English (default)Implement a retry mechanism with exponential backoff.9 tokens
Polish (forced)Implement a retry mechanism with exponential delay.20 tokens

In this specific comparison, we get over 100% markup. In another model, the numbers may be different, but the direction often remains the same.

You can also check your own pairs:

  • Find all failing tests and propose the smallest fix. / Find all tests that do not pass and propose the smallest fix.
  • Refactor this service without changing the public API. / Refactor this service without changing the public API.
  • Explain why this migration is unsafe. / Explain why this migration is unsafe.

It's not about academically counting tokens with every prompt. It's about the sense that later helps during long sessions with the agent.

The effects can be seen in three places at once:

  1. longer conversation history
  2. faster window wear
  3. higher bill in the pay-per-token model.

Let's set aside economic issues for a moment and ask another question - what about the quality of answers depending on the language?

Reasoning is cross-linguistic

There was a long-standing myth that in order for a model to logically think through a difficult architectural problem, you need to write it a proper English essay with precisely chosen verbs.

In practice, the latest models handle many languages much better than old intuitions from the days of simple chatbots would suggest. Good news for everyone who doesn't think in English in the shower.

In the OneRuler benchmark, researchers tested how models handle information extraction from long context in multiple languages. This is not a test of "whether the model writes better code in Polish," but it clearly shows something else: the input language does not automatically destroy the quality of reasoning.

In selected configurations, Polish performed very well there, also in tasks with long context. This is a surprising result, but note - it is worth reading it together with the methodology, and not as a slogan "Polish always wins" as seen on social media.

It can therefore be stated that the model will not lose its programming abilities just because it received a detailed casting error log in Polish.

Moreover, attempts to force English structures by a tired engineer can generate more ambiguity than a well-written problem description in the native language. It boils down to a simple conclusion: with a Polish prompt, we usually don't worry about the model's logic itself, but rather about the cost, context length, and conceptual noise.

Alright then - when should you use English, and when is Polish simply a better tool?

English as the default working mode

The technology stack does not like translations. The vast majority of domain names, library methods, exceptions, and discussions on StackOverflow are in English.

Translating in your head "exponential backoff retry" to "exponential delay of retries" adds an additional layer of mapping. The indexing system that tools in Cursor use when querying the codebase works most clearly when tokens from the prompt match the names from the project.

Treat English in operational commands the same way as branch names, commits in Git, or comments in classes. It is a language that stabilizes the process and minimizes misunderstandings with the agent.

The file CLAUDE.md, AGENTS.md or Cursor Rules - your master contract with the agent - should usually be a precise document in English. This ensures stable instructions.

That doesn't mean that Polish has no place in working with the agent. As a participant of 10xDevs, you will quickly notice that Polish works great as a "debug mode" - a tool for breaking your own cognitive barriers.

When working in Plan Mode on a troublesome piece of refactoring, don’t waste time polishing English grammar. Think out loud in Polish just like you are explaining the problem to a teammate.

This is a conscious trade-off: you pay for these turns with a slightly higher token cost in exchange for clarity of your own thinking. During the main program, we will practice this division when planning, implementing, and reviewing the agent's work.

When the plan is accepted and it's time for bulk file edits and the tool loop - return to the sterile language of engineering. You save tokens on every confirmation, every diff, and every subsequent step.

Simple language policy

It is easiest to think of the AI working language as a project configuration. You don't choose one language "forever," but assign a language to the type of task.

SituationRecommended languageWhy
Project rules (AGENTS.md, CLAUDE.md, Cursor Rules)EnglishStable contract for the agent and good mapping to tools, libraries, and patterns from documentation.
Variable names, functions, commits, branchesEnglishConsistency with the technology stack and less noise when searching the repository.
Operational commands to the agentPrefer EnglishShorter notation, better fit to code and documentation.
UI, messages, emails, content for usersProduct languageIf the application speaks to the user in Polish, the agent should also generate copy in Polish.
Business rules, stakeholder requirements, domain contextLanguage of the clearest descriptionSometimes a Polish description of accounting, education, or support is more precise than forced English.
Planning, debugging, describing a strange errorPolish is OKIf you get to the point faster in Polish, use Polish and only then switch execution to English.

This table is not a dogma. It is the default profile that you can override when a specific project or team has different rules.

There is one more detail. Instead of writing to a modern model "Think step-by-step," more often ask for something you can actually evaluate: a short plan, assumptions, risks, a list of files to change, or a summary of reasoning.

You do not directly control the hidden reasoning process anyway. However, you control the artifacts that the agent shows before editing the code – and it is precisely on them that it is worth basing the cooperation.

What is worth knowing

  • Decision rule: by default, control the agent in English, but generate results in the product language. Keep code, commits, project instructions, and operational commands in English; write UI copy, messages, and user materials in Polish if the product requires it.
  • Security check: when working in Polish, check if the agent has not translated technical terms in a way that conflicts with the code. Pay special attention to library names, exceptions, endpoints, flags, and domain-specific terms.
  • Action for today: take the three most common commands you use with the agent and prepare their English versions. Then compare the tokens in the tokenizer and save the best form in your project rules.
