In the previous lesson, we showed you how a good prompt can guide the agent on the right track and what prompt today would already be considered suboptimal.

But what if the same agent, after forty minutes of work, starts behaving as if someone has tampered with its memory, even though it continues to receive high-quality prompts?

The situation always looks similar - not long ago the agent read the files, located the error, and proposed a diff with a reasonable explanation. And now it repeats itself, "discovering" files that it found itself three rounds earlier, losing path names, and generating increasingly long responses... which do not lead to any decision.

Your prompt has not changed. The model has not changed. What has changed is what is inside the context window.

Previous prework lessons have established two foundations: context is a resource, and a good prompt needs a goal and constraints. What is missing is the third - conscious management of context over time. When to clear it, when to compress it, when to isolate it, and when to simply start over.

What is context engineering

In June 2025, Andrej Karpathy wrote a sentence on X that quickly became the definition of a new discipline. Context engineering is "the delicate art and science of filling the context window with exactly the right information for the next step."

A few days earlier, Harrison Chase from LangChain added the operational version - "building dynamic systems that deliver the right information and tools in the right format so that the LLM can reliably complete the task."

The difference between prompt engineering and context engineering is not cosmetic. Prompt engineering is text editing - you write better instructions and get better answers.

Context engineering is the design of a system that decides what actually enters the model's context window, when, and in what form. In an agent operating over dozens of turns, the prompt alone is just a starting point.

The outcome is determined by what the agent sees in turn 15, 30, or 50 – and that depends on the context architecture, not a single instruction. You will learn all these issues thoroughly throughout the entire 10xDevs, but as a light introduction, it’s worth seeing the four basic moves now.

LangChain in the article "Context Engineering for Agents" proposed four strategies that create a useful map of the terrain:

  • Write — record information outside the context so the agent can return to it. Notes in a file, plans in markdown, commits of partial results. This is memory that survives a session restart.
  • Select — choose what is relevant for the next step. Instead of uploading 20 files “just in case,” allow the agent to fetch context dynamically - grep, semantic search, on-demand reading, or references with the @ sign (in @-mentions).
  • Compress — reduce volume while preserving meaning. Compression of stories, summaries, trimming older messages.
  • Isolate — separate responsibilities. The sub-agent explores the repository in its own context window, and only the summary returns to the main conversation.

These four strategies are not a list of techniques to memorize, but a decision map. Each round of work with the agent is – consciously or not – a decision of Write / Select / Compress / Isolate.

As a 10xDevs participant, you will use this map daily. A good agent operator makes decisions deliberately and does not let the context grow until the model starts to choke.

How the thread with the agent runs

Let's now see what the life cycle of a single conversation looks like and when to make which decisions.

Every thread with the agent starts with a fresh context window - system prompt, AGENTS.md or Cursor Rules and your first message. This is the moment of highest quality response, because the model has little data and no one has cluttered its mind with unnecessary information.

With each subsequent turn, the window receives the model's responses, tool call results, file readings, error messages. Thariq Shihipar from Anthropic states this directly: "the model's performance degrades as the context length increases, because the attention mechanism stretches over more and more tokens."

A key operational skill is recognizing degradation signals. The agent repeats itself and "rediscovers" things it found itself a few turns earlier. It loses file names and paths. It generates increasingly longer, evasive answers instead of concrete decisions. It slows down.

Anthropic adds one more signal - the agent denies its previous conclusions. When you see two or more such signals, it’s time to make a decision, not another prompt.

You have four options, and they are not synonyms (using Claude Code as an example):

SituationToolWhy
The session has grown, but the direction is good/compact <what to keep>Compress history to ~4K tokens while preserving decisions and direction
Change of topic or end of task/clearComplete history reset; system prompt and CLAUDE.md remain
The last step was wrong/rewind or Esc EscUndo 1–N turns without resetting the entire session
The session is beyond savingNew threadCommit knowledge to a file, start from scratch with a better prompt

When to use /clear and when /compact? It depends on the situation.

Thread clearing is an irreversible operation, but it recovers the most space in the window. Use it when a longer session with the agent was used solely for prototyping, and not for the target task. Also when you have achieved a specific task and want to move on to a new thread. Compaction, on the other hand, is a "soft reset" with specifying what should be retained (in a simplified form) in the context window. The thread history is still available, but in a simplified form.

Anthropic in the documentation recommends a specific rule - if you have corrected the agent more than twice on the same matter, it is often better to do /clear and start over with a better prompt than to continue in a cluttered context or execute /compact.

Lee Robinson from Cursor adds a complementary rule: “start a new thread when you move on to a different task, when the agent repeats the same mistakes, or when you finish a logical unit of work”.

What to save before a reset

The biggest mistake when using /clear or deciding to start a new thread? Deleting the history you’ve just built up every time.

Before a reset, save the following in a file, commit or short summary:

  • the current task objective,
  • approved technical decisions,
  • files that have been modified or should be modified,
  • commands executed so far and their results,
  • the status of tests, builds or manual verification,
  • open risks and things the agent should not touch,
  • the next prompt from which you want to resume work.

This is the Write strategy in its simplest form. Not very flashy, but very effective.

When consolidating, you can use a template like this:

Keep: final decisions, file paths, executed commands, test status, accepted constraints and unresolved risks.
Remove: failed attempts, duplicate logs, intermediate hypotheses, repeated explanations and details that do not affect the next step.
Next step: <one specific action from which the agent should start after consolidation>.

During 10xDevs, we’ll come back to this when planning work with agents, during testing and in change reviews. For now, just remember this simple rule: if something is to survive a reset, document it outside of the conversation.

Isolation and External Memory

Compact and clear operate within a single thread. But what if the thread itself is not the right place for a given operation?

Subagents are an architectural pattern for context isolation, not a convenient feature. Each subagent in Claude Code operates within its own context window – file read results, search results, and analysis stay with it. Only the summary returns to the main conversation.

Anthropic in the article about context engineering mentions this as one of the three techniques for long sessions, alongside compaction and structured notes. Simply put: instead of cluttering the main context with data from codebase exploration, you delegate it to a subagent, and your window remains clean for decisions.

Built-in subagents with a specific role can be found in the /agents command (Claude Code and Codex), or by customizing the Agent's custom instructions (Cursor).

The subagent can work in two ways. In foreground mode, it blocks the main agent until the result is returned - the natural choice when you need an answer before the next step. Background mode is parallel work - long repository exploration, a series of tests, or log analysis run in the background, and you return to the results when they are ready.

Both Claude Code and Cursor allow you to create your own specialized subagents as markdown files with YAML frontmatter - you define the name, role description, model selection, and instructions there. The key to effective delegation is the description field - the agent reads it to decide when to hand over the task to the subagent. You place the files in .cursor/agents/ or .claude/agents/, and then invoke the subagent with the command /name or simply describe the task and let the agent choose the appropriate specialist.

Repeated architectural patterns emerge from subagents - a verifier that independently checks whether the declared change actually works; an orchestrator that connects the planner, implementer, and tester in a chain; or a debugger specialized in root cause analysis. It is worth keeping one limitation in mind - each subagent consumes tokens independently, so five parallel subagents cost roughly five times as much as a single agent. This is not a reason to avoid them, but a reason to delegate purposefully, not "just in case".

The second form of isolation is external memory, which survives session restarts. Examples include CLAUDE.md, which is injected into every conversation, or Cursor Rules, which load automatically based on glob patterns. You can create such files dynamically yourself, for the needs of the current task being performed.

Plans saved in markdown files, commits of partial results, notes in scratchpads - these are all forms of the Write strategy. Anthropic even recommends committing progress to git halfway through a long session, doing a /clear and resuming with a clean context and knowledge saved in the repository.
