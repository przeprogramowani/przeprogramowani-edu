Hi, in this video I will show you the basics of working with Cursor - an environment that enables programming with AI.

I'm deliberately talking today about the environment rather than strictly the code editor, because Cursor has already undergone several significant evolutions in the way it works with artificial intelligence - initially we had suggestions and local modifications, then Cursor Composer which worked more cross-functionally, and from version 3.0 we have Cursor as the control center for multiple agents.

Cursor - Operational Basics

If you are using another tool

If you want to use an editor other than Cursor, make sure to take care of a few essential elements corresponding to what we presented in the above clip.

To start, take a look at the privacy policy documentation. Look for the answer to a simple question: can your code, prompts, and work history be used for model training, product analytics, or cloud services? If so, and you are working with commercial code, the default consent for broad data sharing should raise a warning flag for you.

The second thing is the project context. Check whether the tool indexes the repository, how it refreshes this index, and how you can exclude files from the AI view. Look for equivalents of .gitignore, AI-specific ignore files, indexing settings, and ways to manually specify files during the conversation.

The third thing is modes of operation. A good tool should allow at least local editing of a code snippet, discussion about the specified file, the agent working on multiple files, and change control before approval. Without this, we quickly return to copy-pasting with chat, which is... well, back to 2023.

If you're looking for an alternative close to this mode of work, check out Windsurf or GitHub Copilot in Agent mode. They don't have to work exactly like Cursor, but they should give you a similar set of basic capabilities: project context, editing multiple files, running commands, and a clear view of changes.

Support for conventions becoming standards will also be important, including AGENTS.md and .agents/skills.

  • AGENTS.md - standardization of Agent onboarding in the project
  • .agents/skills - new opportunities and knowledge for Agents

At the end, check the models and costs. In the documentation, look for information about default models, the option to choose a more powerful model, subscription limits, API usage pricing, and whether the tool automatically switches models to a cheaper or more expensive option. During 10xDevs we will revisit this topic, because model selection is not a technical detail, but a decision about quality, time, and budget of work.
