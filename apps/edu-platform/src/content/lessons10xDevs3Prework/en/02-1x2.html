Exactly - but how to control it? What really is "AI" today in the context of programming, and where should we assist the agent, where the model decides the success, and where are we guided by the harness built by the solution vendors?

To effectively work with tools like Cursor, Codex, or Claude Code in 2026, we must stop treating the word "AI" as a marketing synonym for a smarter chatbot.

You need to distinguish three layers of the modern ecosystem: model as the engine generating subsequent tokens, agent as the system executing the task through tools, and harness as the environment that provides the agent with access, constraints, and working memory.

We will explore all these topics in more detail in 10xDevs. For now, we aim for a light, practical introduction that will help you better evaluate tools and your own prompts.

An Agent is More Than Just ChatGPT

The most common mistake when transitioning to advanced agent environments is the constant attempt to talk to them as if they were a classic ChatGPT interface.

The chatbot operates in a one-time, static loop: it delivers the next text message based on a single prompt or a series of prompts (input -> LLM -> output).

You receive the response and the chatbot's lifecycle ends. The entire burden of executing commands, evaluating results, and physically integrating the code with the project falls back on you.

Of course, you can continue the same thread, but the model, instead of working for you, rather works... with you - describing the next steps for manual implementation.

An agent is something much more proactive and autonomous - it is a system controlled by an LLM that has a decision-making mechanism and direct influence on your work environment through tools (tool use).

In the world of software engineering, an agent rarely responds solely with text to copy. Most often, it expresses the intention to use a specific tool, waits for the result, interprets it, and makes another attempt to get closer to the goal.

Underneath, the LLM still predicts the next token. The difference is that this token may no longer only represent a part of the response, but also a declaration: "I want to run a test," "I want to read a file," "I want to change this component."

See what a simplified tool usage declaration looks like - a typical JSON structure returned by an agent application:

{
  "role": "assistant",
  "content": [
    {
      "type": "text",
      "text": "Tests in auth.spec.ts are failing. I'm running validation to analyze the stack trace."
    },
    {
      "type": "tool_use",
      "id": "toolu_01A09q90qw90lq917835lq9",
      "name": "execute_bash",
      "input": { "command": "npm run test:unit src/auth.spec.ts" }
    }
  ]
}

Receiving such an instruction from an LLM, the agent application can execute the indicated command in your terminal. When the command completes successfully or with an error, the harness injects the result back into the model's context.

The model sees a new situation, updates the action plan, and proceeds with the next iteration. The operation itself still takes place outside the LLM - on your computer, in your repository, with your dependencies.

This is an important distinction because the same model can produce very different results depending on where it is run.

GPT or Claude in a simple editor plugin can excellently explain an error, but not necessarily search the repository well, maintain context, run tests, and show you a readable diff. The same model in a strong coding harness has access to better tools, better information flow, and more predictable control.

What really affects the result

When the agent performs well or poorly, it's easy to attribute everything to the "model." That's convenient but too simplistic.

The result of the work consists of four layers:

  • Model - reasons, generates text, chooses the next step, and uses tools available in a given environment.
  • Harness - provides tools, manages context, requests permissions, limits risky actions, and displays the work results in a clear format.
  • Local environment - contains the repository, dependencies, tests, scripts, environment variables, and tools such as git, node, ffmpeg or gh.
  • User policy - determines whether the agent can only read files, or also edit them, run commands, install packages, or work without approving each step.

Sounds not very magical? Very good.

At 10xDevs we will often return to this perspective, because most problems with AI in a programmer's work do not stem from a single "stupid model". More often the problem is a poorly chosen level of autonomy, lack of tests, weak context, or too broad permissions.

Harness as a control layer

The ability to autonomously modify files would not work without a control framework protecting the agent from loops, accidental changes, and misuse of tools.

The missing concept is the harness - the runtime, integration, and configuration layer for the AI assistant. It largely determines the quality of work of tools such as Claude Code, Codex, or Cursor.

A good harness gives the agent not only "access to files". It also provides a way to operate safely.

A typical harness includes several mechanisms:

  • Tools and their schemas - describe what the agent can do: read files, search the repository, edit code, run commands, fetch data from APIs, or use the browser.
  • Permissions and sandboxing - specify which actions require your consent, where the agent can write, whether it can install dependencies, and when it should stop before a risky operation.
  • Context and working memory - determine what gets into the model, what is compressed, what disappears from the thread, and how the tool handles a long work session.

There are also less visible but very important elements: task scheduling, tool error handling, telemetry, diff for review, buttons for approving changes, and a UI that allows you to quickly understand what the agent just did.

Without this layer, the agent would be just a model that really wants to help, but lacks agency, and you – in relation to it – control.

An agent can also be a chatbot

The agent does not have to immediately edit files, run tests, and rewrite half of the repository. Sometimes the best use of the agent is a simple conversation.

Exploring the problem, asking questions, organizing concepts, searching for missing assumptions, and deepening requirements are still very enriching. Not because "the chatbot is enough," but because conversation is one of the cheapest ways to improve the quality of further work.

You can ask the agent to explain the error, compare several solutions, find risks in the idea, or ask you a series of questions before implementation. It is still work with AI, but at the thinking stage, not the execution of changes.

In practice, a mature workflow often begins exactly with such a textual exploration. Only then do you move on to planning, file editing, testing, and diff review.

You will get to know this during work on the project PRD, where the conversation with AI helps to refine the problem, scope, and decisions before anything goes into the code.

From conversation to goal achievement

When you realize the potential of agent systems, the old mental model of working with AI changes.

The agent embedded in the CLI has access to a similar set of tools that you use in the operating system. You move forward by delegating tasks and controlling the outcome, not by rewriting successive instructions from the chatbot's responses.

A certain anti-pattern becomes tedious micro-management of the goal-reaching algorithm. Instead of giving the agent every command to transcribe, you describe the desired end state, constraints, and the verification method.

This perspective activates new patterns of collaboration:

  • Bulk file operations: "Review all .svg files in the /assets folder, optimize them using SVGO, rename them to kebab-case, and generate an index.ts with exports."
  • Working with local tools: "Compress video.mp4 below 5 MB using ffmpeg, but keep the text on the screen readable. Check the file size after each attempt".
  • Migrations and tests: "Update dependencies with the aws- prefix, run the linter and tests, and if there are compatibility issues, suggest a safe migration."

Your message becomes more declarative: you describe the goal, boundaries, and success criteria. You leave the dirty imperative work to the system operating in a controlled environment.

This is something you need to get used to gradually.

How to evaluate the harness

Before you entrust the agent with a bigger task, ask yourself a few simple questions about the environment in which it operates:

  • Can they read and search files well, instead of guessing the project structure?
  • Can they safely edit code, show the diff, and stop before a risky operation?
  • Can they run commands with permission control and respond sensibly to tool errors?
  • Does it manage long context through notes or other working memory mechanisms?
  • Does it give you a clear overview of what he did, what he missed, and what he couldn't do?

If the answer to most questions is "I don't know," the agent can still help but should work under close supervision.

A new era of programming with AI

The combination of the three pillars of working with AI - the model, the agent, and the harness - means that you stop being a typical ChatGPT user and start acting like a next-generation programmer.

Your role is no longer just to ask questions. Increasingly, you design the work environment, choose the level of autonomy, set safety boundaries, and approve higher-level decisions.

You can delegate the physical modification of files, tedious directory traversing, and repetitive commands to the machine. You focus on intention, risk, quality, and checking whether the result actually delivers value.

Not everything is rosy, there are also some downsides or consequences.

Autonomy can increase the cost of labor - if the agent gets stuck in the tools, it will run tests senselessly, lose the decision after context compaction, or start "fixing" the problem by hiding the exception - you will pay with tokens.

That is why at 10xDevs we will practice not only writing prompts but also designing controls: plans, checkpoints, review diffs, tests, branches, and moments when it is worth telling the agent "stop".

For a good start

A practical set of rules for an effective programmer transitioning from chatbots to agents:

  • Decision: train cooperation with the agent and try to delegate increasingly complex tasks. Avoid moments when AI controls you – that is a pattern from the past.
  • Control: before allowing the agent to change the project, ask for a plan, scope of edits, and success criteria. It's cheaper to fix the plan than to clean up after bad autonomy.
  • Action: prepare the work environment: rules, working dependencies, tests, linters, git status without surprises, and a short document with project guidelines for the agent.
