The beginnings of working with AI agents always look the same - you paste a large dump of logs with a production error and ask for a fix by writing a simple prompt "take care of this".

After a few seconds, confident, you receive a perfectly formatted diff which, after a lightning-fast deployment... crashes the rest of the application. Quite a lot of work for a single command!

Frustration in such moments most often stems from the mistaken assumption of an all-powerful AI that will guess our intentions, choose the one right way to act, and additionally, in case of problems, describe its work step by step. Unfortunately - although the potential of AI grows month by month, it is still not a magic wand.

To stop fighting with AI, we need to look at it coolly, as a system based on probability calculus with hard, measurable limits. The input resource has its strict cost, and making errors is not a workplace accident, but the core of the entire process.

Token Prediction in Practice

There are two main sins in the daily integration of AI into the software development lifecycle – ignoring the potential of agents and overestimating them. As a member of 10xDevs, you are certainly not in the first group, but you may be prone to the second extreme – which is equally dangerous.

Instead of AI-voo-doo, we want to look at these solutions through the eyes of an engineer. What they are, what architecture they are based on, and where our intervention is still necessary. It won't hurt to start with a few words about the basics.

When leading models (such as GPT-5.4 or Claude Opus 4.7) generate code in your IDE, they perform a continuous, massive next-token prediction operation. The neural network's weight base is a complex machine that decides which fragment of text and syntax statistically best fits as the direct continuation of the prompt you provided.

The model in the pre-training layer does not have an embedded mental image of your software. It does not know what well-executed database transactionality is and does not experience cognitive dissonance when generating code that violates SOLID principles. Its sole mathematical task is to minimize the prediction error with respect to all the projects and code snippets it has been trained on.

How it looks in the code

During training, the model viewed vast collections of text and code: repositories, documentation, test examples, bug descriptions, snippets of technical conversations. It does not remember them as one large searchable library, but updates weights so that after a given sequence of tokens it can better predict the next token.

The simplest image of predicting the next token can be understood with a simple code snippet:

function calculateTotal(items: Item) {
  return items.reduce((sum, item) => sum + item. //which field to take from the item object?

If we have the classic autocomplete model enabled on such a fragment, after the fragment item. field names typical for thousands of similar examples may appear.

And here the practical problem begins. If in your project the correct field is item.grossAmount, but in context there is not a clear enough signal, the model may choose item.price, because such a continuation looks statistically familiar.

function calculateTotal(items: Item) {
  // Property 'price' does not exist on type 'Item'.
  return items.reduce((sum, item) => sum + item.price;

From a practical point of view, this means that the model is calibrated to produce text that "looks" the safest, but not necessarily functional. A fabricated piece of logic may perfectly mimic naming conventions, a new endpoint may at first glance look just like the others, and a new data model may be "just right," yet in the end something will still go wrong.

Such mistakes usually do not look spectacular. The code passes through the formatter, imports match, function names sound familiar... and that's exactly why it's easy to let it pass further.

In practice, you will see three recurring classes of errors:

  1. Syntax-correct code but semantically incorrect - e.g. handles the happy path but loses the transaction, retry, or edge case in the data.
  2. Test that checks implementation instead of behavior - looks professional, but only reinforces a faulty model assumption.
  3. Ignoring the context center - the model remembers the beginning of the conversation and the last command, but skips the decision buried dozens of messages earlier.

The mere similarity of the code to what already exists in the project, and even the correctness of imports, does not eliminate the chance of mistakes.

Can this be prevented? Yes, partly through gained experience, and partly through strict security gates and rigorous context control, which are techniques you will learn in 10xDevs.

From guessing to reasoning

Since classical models are simply fast statisticians, how can we entrust them with architectural tasks that require hypothesis verification and predicting blind alleys?

The industry's answer is reasoning models. Instead of relying solely on the knowledge base from the pretraining stage, this architecture allows allocating additional budget for inference after the original question has been asked.

Instead of immediately spitting out the result, such a model uses hidden reasoning tokens as an auxiliary tool. The model operates with thousands of such tokens, allowing it to consider several variants of further action: would this database migration lock the table for other processes? Can I do it in stages? Do I need to change the deployment order? In this way, the system gains a better chance of solving structural problems, not just generating additional lines of code.

Such power, however, comes at a price. Every step in hidden reasoning is tokens that you normally pay for.

Sending a "thinker" for a simple formatting change is a great way to burn through the budget. On the other hand, trying to force a cheap base model into deep refactoring quickly increases the risk of context degradation. Hence, in solutions like Cursor or Claude Code, after the model name, so-called effort annotations appear such as:

  • Gemini 3.1 Pro (low)
  • GPT-5.4 (high)
  • GPT-5.3-Codex (xhigh)

Terms such as low / high / xhigh denote the allowable level of reasoning (budget for hidden reasoning tokens), which correlates with the quality of the result, but also with the cost of the entire operation. If you are planning the architecture, aim for higher levels - if, on the other hand, you are dealing with implementation on an existing plan, you can set this control a bit lower.

Model reasoning does not exempt you from verifying the results. The model may analyze the problem longer and still propose a change that does not pass tests, ignores real environmental constraints, or is based on an outdated assumption.

Additionally, in many editors and environments, you will see at most a summary of the train of thought, instead of a raw "chain of thoughts." This is an additional argument for maintaining full control over what AI does for us in the project.

Hard wall of context degradation

And now another aspect of working with models, which directly results from their architecture.

Programmers often fall into a certain trap: since the LLM's output matches the input, let's throw the entire repository, documentation, and logs into the model's window – it will somehow manage.

The truth about processing huge input is different: loading irrelevant content baggage reduces the model's analytical capabilities. This applies equally to cheap base models as well as the most expensive reasoning models. Research on the performance of leading solutions and practical work with agents point to a barrier that has been provisionally called MECW (Maximum Effective Context Window).

After exceeding it, the model's ability to extract important information often decreases. The fact that the model reports 2 million tokens available in the window does not at all mean that it will correctly take into account a hint buried on the hundredth page of your logs.

Degradation results, among other things, from how models distribute attention over long input and how tools append context to the conversation. The model may cling more to the beginning of the input data and the last lines of the chat, while losing the logic embedded in the middle of the prompt.

How to deal with this? Control the occupancy of the context window at every important stage of the conversation (especially before starting a crucial stage of work). Our "rule of thumb" is values like those below:

Important clarification: the problem is not the existence of a large repository itself. Good tools can index, search, and selectively fetch code fragments. The problem begins when you dump unfiltered context into the current window and expect the model to distinguish signal from noise on its own.

Token Budgeting

Understanding both of the above phenomena - the power of reasoners and the effect of context degradation - leads us to the most important engineering principle when working with AI. The model's context window is an extremely valuable space that must be cared for especially carefully.

Token resources in the window will be used, among others, by:

  1. System instructions: - System prompt prepared by the solution creator.
  2. Constant design instructions: AGENTS.md / CLAUDE.md / Cursor Rules - files with design instructions that are used by the model in every session.
  3. Commands in the current session: - Current tokens as well as thought tokens generated for the user's session, e.g. when executing tools/commands.
  4. Tool definitions / MCP - Tools that are available to the model in a given session.

Depending on the problem, we need to learn the appropriate allocation of these resources to avoid overspending the budget. There should be a lot of knowledge and context, but... not too much. It should be high-quality content, but without a bulky structure and unnecessary stylistic tricks. Experience and work at 10xDevs will help you master this aspect.

For the purposes of daily work, you can adopt a simple division:

Task typeReasonable choice
Mechanical change, formatting, simple transformationcost-effective model with low inference budget
Unclear architecture, migration, difficult production bugreasoning model with higher effort

In our opinion, it's quite intuitive. As the importance of the problem increases, we can invest more tokens in the model's work itself. We will get better results, but we will pay a bit more for them. A reasonable compromise.

What is worth knowing

  • Decision rule: choose the model according to the depth of the problem. Leave routine fixes and styles for cheaper models, and direct difficult architectural problems to the models reasoning with the greatest effort.
  • Safety check: do not trust the certainty of the answer without external verification. Check the diff, run tests, build the project, or compare the claim with the documentation.
  • Action for today: before asking the agent to complete a task again, consider what it actually sees at the moment - perhaps the project instructions, rules, or the prompt itself contain redundant content?
