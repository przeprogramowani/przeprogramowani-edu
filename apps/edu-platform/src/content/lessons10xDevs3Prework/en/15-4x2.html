You already have a few ideas in mind. One looks ambitious, the second practical, the third sounds like something that would look great at Demo Day. And right at this moment, it's easy to confuse a course project with a product, a startup, or a case study for a six-month portfolio.

In 10xDevs the course project has a different task. It is supposed to give you a real environment to work with agents, artifacts, tests, CI/CD, and technical decisions. It doesn't have to be big. It doesn't have to be web-based. It doesn't have to be built in our stack. It must be concrete enough to be delivered together with the course.

A good course project is not the one that sounds the nicest in the description. A good project is one that quickly delivers the first working user flow, has clear business logic, and naturally meets certification requirements.

A good project is smaller than you think

The most important word in this lesson is MVP (Minimum Viable Product), which is the smallest version of the product that still provides the user with concrete value. Not a "prototype version to be discarded." Not a "senseless version." Not "half an application." MVP is the smallest coherent flow that can be launched, tested, and demonstrated.

For a sample flashcard application, an MVP could mean: I paste the text, the application generates flashcard suggestions, I accept the chosen ones and save them in the database. No support is needed for organization, payments, PDF import, mobile application, or a spaced repetition algorithm based on machine learning.

For a training planning app, an MVP can mean: the user provides a goal and constraints, the app generates a weekly plan, the user marks completed sessions, and the system updates the recommendation. This is already business logic. It already has data. It can already be tested from the user's perspective.

The rule is simple: if the first working flow requires more than a week of overtime, reduce the MVP. A week is not a formal limit. It's a smoke detector. If just getting to "something working" takes you several weeks or months, during the course you'll run out of time for documentation, tests, CI/CD, fixes after feedback (and normal life).

MVP has one more advantage: it helps the agent. The agent works better when they have a clear goal, a short verification loop, and a specific context. A project with a small, clear core is easier to describe in a PRD (Product Requirements Document), easier to break down into tasks, and easier to verify with an end-to-end test.

What the final project must have

Certification does not evaluate whether you made the biggest application in the cohort. We assess whether the project demonstrates practical use of the workflow from the course and meets the minimum technical requirements.

The project should have an access control mechanism appropriate to the type of application. In a web application, this will usually be login. In a desktop application, it could be a local user profile, access key, or another mechanism that realistically fits the domain. The point is for the application to distinguish the user, state, or permissions, and not be an anonymous demonstration page.

The second element is data management. The project should allow creating, reading, updating, and deleting items in a way that makes sense for the domain. It's not about an artificial CRUD (Create, Read, Update, Delete) list added just to tick a requirement off. It's about data that the user needs: flashcards, tasks, notes, workouts, recipes, transactions, submissions, measurement records.

The third element is business logic. It can be AI, but it doesn't have to be. What's important is that the application makes some domain decision: classifies, recommends, validates, calculates, creates a plan, checks conditions, generates a proposal, or guides the user through a process. If you can't describe this logic in one sentence, the project is still too vague.

The fourth element is the project artifacts from modules 1-3: PRD, specifications, plans, context for AI. In 10xDevs, it’s not just about the final code. It’s also about whether you can prepare a context for the agent that the agent can work effectively with.

The fifth element is at least one test verifying the operation from the user's perspective. It doesn't have to be an extensive test suite. There must be a test that checks the key flow and gives the signal: the user can perform the most important action.

The sixth element is CI/CD (Continuous Integration / Continuous Delivery or Deployment), which is a pipeline that builds the application and runs tests. Public deployment, App Store, or an installable package are very welcome, but treat them as a layer above the minimum, not as the first problem to solve.

Various paths are accepted: your own application with business logic or extending an existing project with a new module. The stack is also open: web, desktop, mobile, embedded, JavaScript, .NET, Java, Python, PHP, or something else. The key question is not "is this our stack?", but "will the project allow you to practice the AI workflow and meet requirements without wasting time on ramp-up?".

A bad design has a high zero-to-one threshold

The most dangerous project doesn't always look bad. Often it sounds great: your own company automation system, an app with five integrations, a cross-platform mobile product, a tool on a niche framework you've long wanted to learn.

The problem starts when you bury yourself in technical tasks for a long time. You configure authorization with an external provider. You struggle with a library that the agent barely knows. You design integration with an API before a basic user flow exists. You work for a week, but still can't perform one meaningful action.

It is precisely the high zero-to-one threshold: you have to put in a lot of work before something functional is created. In a course conducted after hours, this quickly kills momentum.

The second common mistake is an MVP that is too large. The description starts innocently: “personal finance management application.” After five minutes, it already has bank import, receipt scanning, charts, budgets, alerts, family roles, an AI coach, and calendar integration. Each function individually makes sense. Together, they create a project that competes with a product application, not with the course scope.

The third mistake is an empty CRUD. The application allows adding, editing, and deleting records, but there is no domain decision. A list of books, movies, contacts, or tasks can be a good foundation, but the list alone is usually not enough. Add a rule: recommendation of the next book, review plan, task prioritization, data validation, scoring, approval workflow, ticket classification.

A good project yields results quickly. A bad project is a company, not a side project.

If you have two ideas, don't choose the one that sounds flashier. Choose the one that has a shorter path to the first flow and a more obvious business logic. The impressiveness comes later, when the working core can be refined.

What is worth knowing

  • Decision rule: the first working flow should be achievable within a week of after-hours work. If it isn’t, reduce the MVP before you start writing the PRD.
  • It's worth remembering: describe the business logic in one sentence. If the only sentence is "the user can add and delete records," the project needs a stronger domain.
