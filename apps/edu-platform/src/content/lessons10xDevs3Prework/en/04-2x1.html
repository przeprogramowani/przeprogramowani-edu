The following lessons will prepare you to work with the AI agent in three incarnations and allow you to choose the preferred working environment throughout 10xDevs.

Cursor will represent a classic code editor with a built-in AI assistant — folders, files, diffs - a familiar affair. Claude Code, on the other hand, will be a representative of Agents in the terminal — here AI work is primarily about the goal, tools, and verification of the final result. You will also get to know completely new environments where the priority is delegating work to Agents, not the code itself. Cursor 3.0, Claude, and Codex Desktop are representatives of this generation of tooling - you open the editor and see a prompt field with a list of active agents.

Three tools, three philosophies. Instead of a firm recommendation, an analysis of pros and cons and the most important features to start with.

To begin with, a brief overview of where each approach has its strengths and where it naturally limits the potential of AI.

Finally, we will suggest an optimal setup, but you, throughout the entire 10xDevs, will be able to configure it in a very individual way.

Classic IDE with built-in Agent

This is the most familiar path, where a safe choice will be Cursor or alternatives such as Windsurf or VS Code with Copilot.

You open the editor, see the directory tree, the terminal at the bottom, and the window for working with AI in a separate panel.

The advantage of this form of working with AI is obvious: you don’t change your programming habits. Keyboard shortcuts, extensions, window layout — everything in its place. The AI agent is an addition to the environment you have known for years.

What might be insufficient today? The classic editor was created as a tool for manual file editing. The directory tree, tabs, project navigation — this is an interface for a human who writes code themselves. Working with Agents (and today we often work not with one, but many) can be more difficult here. Depending on the solution, using AI can be either a pleasure (e.g., Cursor) or something that causes a headache (early versions of JetBrains editors with the JetBrains AI plugin).

If the preferred environment does not integrate Agents, but only Chatbots and forces work on a copy-paste basis, then unfortunately it does not meet the needs of the modern AI workflow. It may also happen that the same model in the chosen editor will give worse results than when we operate it from the terminal level. This is already a matter of the so-called harness, about which we wrote more in one of the previous lessons.

If you want to enter the world of programming with AI precisely in this format, we recommend Cursor, whose basic operations you will learn in the next lesson.

Terminal as an agent environment

The agent in the terminal is the other pole of cooperation with AI. You won't find a file tree here, you won't find tabs — there will be a standard text field and an Agent waiting for your commands.

Workflow? You build context, define a goal or intention, and the Agent gets to work. It works in a loop, uses tools, can correct its actions and (hopefully) comes back with ready results. Examples - Claude Code, Codex, or OpenCode.

The strength of this approach lies in its versatility. If you need multiple Agent sessions, you can simply run multiple terminal sessions. The terminal is the interface through which almost everything in the development ecosystem passes: git, npm, docker, ssh, ffmpeg. An Agent embedded in the terminal does not require special integrations — it has access to the same set of tools as you do (Agents in your IDE should be able to do this today as well - if not, look for a better environment).

The terminal also offers something the editor does not natively provide: headless mode. claude -p allows you to run the agent without an interface — in a CI/CD pipeline, in cron, in an automation script. You provide the goal, limit permissions with flags, and parse the result into JSON.

git diff main | claude -p \
  "Review these changes for security issues and regressions" \
  --allowedTools "Read,Grep,Bash(npm run test:*)" \
  --max-turns 6 \
  --output-format json > review.json

This is a domain where the code editor simply does not exist.

Of course, this format has its limitations. For many developers, especially those who work daily in GUI editors, the CLI is an uncomfortable environment. More difficult to handle diffs, one highly standardized layout, more work without immediate feedback, with commands and shell. A slight difficulty at the start may be a completely new set of commands that do not resemble file editing or classic refactoring. However, in the lesson on the basics of Claude Code, as a representative of AI in the terminal, you will learn the most important commands that will make your start easier.

Agent-Native IDE and Cloud

Terminal or classic IDE - or maybe something completely different?

The "Agent-Native" tools immediately after launching give you one main prompt field with a list of Agent sessions. Files? Folders? They exist somewhere, but they are not the most important.

The priority here is not code editing, but controlling agents and achieving goals by managing a virtual team. This direction is taken by desktop versions of Claude or Codex, where the prompt field replaces the classic terminal view.

The mentioned environments were designed from scratch to work with multiple agents simultaneously. Easy launching of several independent sessions, support for GIT Worktrees, tools for previewing the results of their work. And... no more being tied to your computer.

By using Cursor or Claude in the cloud, you no longer need to clone the repository, install dependencies, or sit at your computer at two in the morning, saving a production incident. In this format, you specify the repo, the agent configures its remote environment and works in the background. You come back for the result when it's ready.

The biggest downside? The code layer goes out of sight. For many developers, this is a real loss — they want to see what changes, line by line.

And it's hard to be surprised. In Agent-Native IDE, you work more like a manager delegating tasks to a team than like a programmer with hands on the keyboard. It's a different perspective that requires trust in the process, which you do not control at the level of each diff.

We will show you this using the example of Cursor 3.0 and Claude Desktop.

Each of these environments is optimized for a different way of working and a different level of trust in the Agent.

Do you need control over every change, and the comfort of a familiar editor is key for you? A classic IDE with agent functions is a safe start. You lose some of the agent's versatility, but you gain the certainty that you know what is happening in the project.

You value agency and want the agent to use the same set of tools as you? The terminal is a natural choice. The entry barrier is higher, but headless mode, CI/CD integration, and the full power of the CLI open up scenarios unavailable in the editor.

Are you thinking about delegating many tasks simultaneously and working in the background? Agent-Native IDE provides an interface designed precisely for this and lets you experience a "new dimension of programming." It requires a shift in perspective from "I write code" to "I manage agents," but the more autonomous the agents become, the more this perspective makes sense.

The most valuable skill is not loyalty to a single environment. It is the ability to consciously switch between them depending on the task. A small fix in a familiar file? IDE. Multi-file migration in a large repo? Terminal. Five independent tasks to complete? Agent-Native with worktrees and the cloud.

Importantly, more and more often one subscription is enough for you to use every format presented here, for example:

  • Claude Code - AI Agent in the terminal
  • Claude Desktop - Agent-Native IDE
  • Claude.ai - Cloud-based Agent Service

Recommended setup

Today, you will feel the greatest potential of AI using terminal Agents, such as Claude Code or Codex. This does not mean that you have to lose sight of the classic file and folder tree.

You can simply run the mentioned agents in the terminal session of a "big IDE" like WebStorm, VS Code, or alternatives. Then you use familiar habits and keyboard shortcuts for code editing, and at the same time you have access to a universal, ready-to-adapt Agent.

What about Agent-Native IDE? It is an option for the brave and for those who want to experiment with a new programming format. Most solutions of this type have appeared on the market just this year, so we can expect a period of continuous iteration and optimization for full stability. On our side, we recommend Claude Desktop and Codex Desktop.
