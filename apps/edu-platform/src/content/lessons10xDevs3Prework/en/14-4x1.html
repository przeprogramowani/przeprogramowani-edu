What makes the stack "agent-friendly"

Choosing a stack for working with an AI agent is not a matter of fashion. In the lesson about LLMs, we discussed that the model predicts tokens based on patterns from training data.

Therefore, a stack good for an agent should provide many clear signals. Four are the most important.

Typed. Explicit data contracts are not decoration for the agent, but a map. When UserProfile has defined fields, the agent does not have to guess whether the user has name, fullName, displayName, or something else from outer space.

TypeScript, Java, C#, Python with type hints, or other typed environments provide the agent with a similar kind of support. They differ in syntax, but the principle is the same: the less guessing, the fewer errors.

Convention-based. A convention is predictability. If a project has established places for routing, components, backend logic, configuration, and tests, the agent understands more quickly how to navigate it.

It doesn't have to be our stack. The important thing is that in your ecosystem there is a mature way of creating a project, not improvisation.

Popular in training data. Models perform better with technologies that have many public examples, tutorials, repositories, and discussions. It's not about blindly choosing the most popular solution, but whether the agent has seen enough similar patterns during training.

An exotic stack can be great for a team of experts. For an agent, however, it can be like working with a map without street names.

Well documented. Official documentation, up-to-date examples, and clear guides help the agent stick to real APIs instead of making up methods that sound plausible.

At 10xDevs we will repeatedly return to the same pattern: the agent works best when it receives clear context, not when it has to guess your intentions. The stack is part of that context.

Where a wrong choice can really cause trouble

The worst choice is not "other than course".

The warning sign is simple: you often correct the agent not because they solved the problem incorrectly, but because they are unable to effectively use your technologies.

Therefore, before choosing a stack, don't ask: "can I write with AI in this?". Ask: "will the agent get types, conventions, a starter, and current documentation?"

Of course, the gaps in the above are not a verdict, we can supplement the context ourselves through instruction files (AGENTS.md, CLAUDE.md, or rules), but this requires additional work and may not yield as good a result as working with a stack that provides these elements in a natural way.

Astro manages routing, page building, server-side rendering, and API endpoints. It is in Astro that you define the application's custom API, e.g., handling a form, a webhook, or an endpoint acting as an intermediary between the frontend and backend services.

Its distinctive feature is the islands architecture: the page is by default HTML, and interactive parts, such as a form, modal, or dynamic list, are embedded as isolated "islands".

In January 2026, The Astro Technology Company team joined Cloudflare. This strengthens the integration with Cloudflare, but Astro remains open source, under the MIT license and platform-agnostic.

React is responsible for interactive islands. You don't build the entire application in React, you only use it where static Astro HTML is not enough.

TypeScript covers the entire project: components, API endpoints, and backend logic. For the agent, this is the most important navigation signal. It sees interface Product { id: string; price: number } and doesn't have to guess the shape of the data.

Tailwind CSS keeps styling close to the component. Instead of scattering context between the component file and a separate CSS, the agent sees the structure and style in one place:

<div class="flex items-center gap-4 p-6 bg-white rounded-lg shadow">
  <h2 class="text-lg font-semibold text-gray-900">Card title</h2>
</div>

This is not the only good way to style, but in the course starter 10x-astro-starter it works great.

Supabase provides a PostgreSQL database, auth, storage, and SDK in TypeScript. In our stack, it does not replace the API layer in Astro, but rather delivers ready-made backend services used by Astro endpoints and application code.

Cloudflare is responsible for deployment and edge runtime. Astro has official integration with Cloudflare, so the path from a local project to a working online version is short and well documented.

This is our default path. It will be easiest to get support, compare your code with examples, and use our starter.

But still: default does not mean the only one.

If you prefer your own stack

If you work professionally in another ecosystem, you don't have to abandon it just because the course shows examples in Astro, React, and TypeScript.

Most of the 10xDevs lessons are about the way of working: how to work with context, how to plan with an agent, how to do code review with AI, and how to build workflow. These skills transfer across technologies.

The condition is simple: your stack must be readable enough for the agent and familiar enough for you.

Check it like this:

  • Does it have types or at least explicit data schemas?
  • Does the project start from an official starter or a mature template?
  • Are the folder structure, routing, tests, and configuration based on conventions?
  • Is the documentation up-to-date and easy to link to the agent?
  • Can you recognize by yourself when an agent is doing something inconsistent with the practice of your ecosystem?

If the answers are "yes," you can carry out the project in your environment. If several answers are "I don't know," the course stack will be a safer path to start with.

The course stack provides full support: ready-to-use starters, examples, and proven patterns. Your own stack offers greater transfer to your work, but the heavier burden of technical decisions remains on your side.

How much you need to know at the start

Short answer, regardless of the chosen stack: orientation matters, not proficiency.

The course is designed so that the agent helps you work in an environment you do not know perfectly. The stack is a tool, not the learning goal.

What to have before Module 1:

  • Familiarity with some type system: TypeScript, Java, C#, Python with type hints, or another system of explicit contracts. You don't need to know the syntax of every framework. You need to understand why types help the agent.
  • General concept of HTTP and API: endpoint, request, response, JSON. Whether you are building on Supabase or on your own backend, this layer does not change - unless you will not be implementing a web application (which is also acceptable).
  • The ability to run the starter: git clone, dependency installation, the dev command or its equivalent in your ecosystem. If this is natural for you, there is nothing to worry about.

What you do not need to know:

  • The syntax of no specific framework — you will learn it in M1, programming with the agent.
  • Configuration of static code analysis tools — default configurations in starters work immediately.
  • Cloud deployments — this is the topic of M1, the lesson about deployment.
  • Advanced patterns of a given ecosystem — course projects are designed so that this is not required at the start.

During 10xDevs, we will focus much more on how you manage the agent, how you check their work, and how you build a secure workflow. The stack is meant to help you with this, not to become a separate source of stress.

What is worth knowing

  • Decision rule: if you don't have a strong preference, go with the course stack. If you have a production stack, stick with it when it meets the four agent-friendly criteria.
  • Worth remembering: do not start from an empty folder. Choose a starter with types, linting, formatting, and a working dev command.
  • Action: compare the course stack and your own stack in the table from this lesson. You don't have to finalize your choice yet, but it's worth knowing where your decision comes from.

You will finalize the decision about the project and stack in module 1. For now, one thing is enough: you do not need to know our stack before starting to benefit from the course.
