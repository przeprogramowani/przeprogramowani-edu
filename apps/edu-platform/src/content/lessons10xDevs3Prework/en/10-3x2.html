In the previous lesson of our prework, we showed you how LLMs predict the next tokens and why flaws such as hallucinating logic are inherent to their nature.

Now it's time to apply this knowledge in practice.

The beginnings of working with agents always look similar - you type "fix this error", and then watch as the system wanders through the code for several minutes and finally... breaks your build. And when it actually fixes something, you have the feeling "damn, this could be done better". Classic.

The same short instruction that provides a safe text suggestion in ChatGPT often leads to strange modifications in many files in an agent environment. The difference does not lie in a "better model." A chatbot prompt is a one-time low-risk query, while an agent prompt is a long-term high-stakes contract.

During 10xDevs we will break down this contract into very specific workflows: planning, implementation, code review, testing, working on branches, and team rules for using AI. Here we build the foundation.

Two modes of working with AI

The word "prompt" today supports two completely different modes of dialogue with AI. When working with a browser chatbot, you enter an instruction, and after receiving the text, you decide what to do with it. The turn usually takes seconds, there are no side effects, and minor inaccuracies at most slightly worsen the style of the response.

In the case of an agent, the same text triggers an autonomous decision loop. The system plans what to do, uses tools, observes the results, and adjusts the effects of its work itself - often through dozens of consecutive iterations.

Therefore, you must remember three key differences between a typical query to ChatGPT and, for example, Claude Code:

  1. Duration. A prompt to the chatbot lives shortly - one turn. In the agent, ambiguity that seems harmless at the beginning, by the tenth iteration of using tools, can cause the code to start being modified in a completely unexpected way.
  2. Side effects. The agent edits files, runs tests, and modifies the repository. Your communication style must clearly specify not only what to do but also what to strictly avoid: which files not to touch, whether the public API should remain unchanged, whether migration can alter data, what performance or availability budget applies. A system without boundaries will improvise.
  3. Error cascade. The chatbot does not consider the consequences of its responses. The agent - quite the opposite. It takes every failed task very seriously, trying to improve and implement the necessary changes. Hence - without caution - the often observed "no-exit loop" effect - a situation in which the agent loses a sense of purpose and focuses in a loop on fixing what is completely insignificant from the perspective of the original command.

Does this mean you need to write perfect, multi-page essays every time you use AI? Quite the opposite. In a well-configured agent environment, your prompt is short and precise — because most of the instructions live elsewhere, in the hierarchy of layers that you configure once and that work in every session.

Anatomy of an Agent Prompt

In the chat, your prompt is usually the only thing the model sees. In the agent environment, it's the opposite — the message you enter is the last layer in a stack of hierarchically built instructions.

Imagine this stack from the bottom:

  • System prompt of the harness — built-in behavior: security, context compression, tool management. You do not change this — this is the manufacturer's job (we talked about the harness in lesson 1.2).
  • Design rules (CLAUDE.md, AGENTS.md) — permanent rules specific to your project. You write them once, and they apply in every session. This is where conventions, constraints, and architectural patterns go. There will be a dedicated lesson about this.
  • Memory — context built from previous sessions, if the given tool and organization settings allow it. Sometimes the agent remembers your preferences, decisions made, and feedback. Sometimes memory is turned off or limited, so important decisions should still be recorded in the project.
  • Skills — ready-to-use prompt templates triggered on demand (/commit, /code-review, /plan). They standardize repetitive tasks so you don't have to write the same instructions every time.
  • Your prompt — a specific goal to achieve here and now.

Pay attention to who and how often configures each layer:

LayerWho configuresHow oftenExample
System promptTool manufacturerNever (on your side)Security, context compression
Design rulesYou or the teamFrom time to time — evolutionary changesCLAUDE.md: "Vitest tests, Svelte, not React"
MemoryTool or configurationDepends on the environment"User applies conventional commits"
SkillsYou or the teamReusable, as needed/commit, /code-review
PromptYou, here and nowEvery task"Fix bug in calculateDiscount()"

That's why a good agent prompt can be surprisingly short. Constraints, context, and conventions live in the lower layers — you don't have to repeat them every time. Your message focuses on what really changes: a specific goal and success criteria.

You prepare a good agent prompt even before you start writing it — first, you configure the layers below, and then you determine how you will check if the work has been done correctly.

In a conversation with the chatbot, you explain to the model how it should respond to you. In the agent, you specify how the system will recognize that the task has been fully completed (e.g., successful build, tests, and no linter errors). The agent already has the rest of the context on board.

In the main 10xDevs program, we will also practice which rules are worth keeping in AGENTS.md or CLAUDE.md, and which are better to move to separate skills and procedures. Here, one intuition is enough: a prompt in a well-configured environment does not need to describe the entire work system from A to Z.

Reasoning Models in Practice

Leading reasoning models are able to break down complex problems into smaller parts.

In their case, traditional "hand-holding" techniques, such as imposing step-by-step thinking or providing many examples of solved tasks, can actually worsen the results. These models analyze the problem in their own learned way, and our attempts to impose a specific line of thought often introduce noise.

The optimal prompt for such models is a clear goal combined with ironclad constraints, without any additional clever tricks.

Let's look at three concrete examples, each of which directly results from how these models work underneath.

1. Do not impose steps if you don't have to.

A reasoning model breaks down the problem internally. When you impose a sequence of steps on it, you force it to follow your path instead of finding the optimal one.

❌ "Analyze this function step by step. First check the types of the arguments, then trace the data flow, next identify the error and suggest a correction."

✅ "Fix the bug causing double discount calculation in calculateDiscount(). Tests in discount.test.ts must pass."

In the first case, you force a specific line of reasoning which - depending on the problem - may be worse than what the model would choose on its own. In the second, you provide a goal and an objective success criterion, and the model decides by itself how to achieve it.

There are exceptions, however. In data migrations, security, payments, accessibility, or strict team conventions, these steps may be necessary. Then you are not guiding the model by hand for fun, but rather overseeing a process that must not be skipped.

2. Do not give examples that close the problem.

Reasoning models have been trained to independently arrive at problem solutions. When you provide them with a ready-made pattern, you narrow the search space to your way of thinking — and it is not always the best.

❌ "Here is how I optimized a similar function: [15 lines of code]. Optimize rendering in userList.ts in a similar way."

✅ “Optimize the rendering of the user list in userList.ts — with 5000 records, the rendering time should not exceed 200ms.”

In the first prompt, the model may obediently copy your pattern, even if there is a much better solution. In the second, it receives a specific, measurable goal and the freedom to choose the strategy.

That does not mean the examples are bad. They are great when you are showing the style of the response, the report format, the changelog structure, or the rules of your own DSL. The problem starts when the example pretends to be the best technical solution.

3. Do not impose the technique — provide the constraints.

This is a natural consequence of the two previous principles. When you tell the model "use the Observer pattern" or "write this using reduce," you cut it off from potentially better alternatives.

❌ "Use the Observer pattern to synchronize state between components. Create an EventBus class with a Subscriber interface."

✅ “The Dashboard and Sidebar components must respond to filter changes in real time. Currently, the state is inconsistent after rapid switching. Fix the synchronization — without changing the public API of the components."

The first prompt is a ready-made technical specification that leaves no room for reflection for the model. The second defines the problem, symptom, and boundary — the rest the model will determine itself, and it is very likely that this will be a good enough approach.

Important: Of course, there are situations when imposing a specific pattern makes sense. If after 1–2 attempts the agent still does not meet your expectations, and you have strong preferences regarding the implementation style, then it is worth being more specific. You can also ask the agent to propose, for example, three different approaches to completing the task — considering complexity and best practices — and only then choose one and give the green light for implementation.

So it's not about banning instructions. It's about not confusing the goal with micromanagement.

Common Prompting Anti-Patterns

Since we know the proper structure, let's look at common pitfalls. Working with an agent simply requires much more discipline than a casual chat conversation.

1. No limits.

Most often, simple boundaries are missing: "edit only src/billing/*", "do not change the public API", "do not add dependencies", "show the plan first", "do not run migrations without my consent".

2. Reloading the entire context at the start.

3. Lack of objective criteria.CLAUDE.md, as well as reusable commands or hooks.

Even better if the criterion checks behavior, not just implementation. A test added by the agent can of course cover exactly the same error it just generated. It’s green, but that’s not what it was about.

4. Hierarchy compression into one layer.CLAUDE.md absolutely everything — rules, procedures, preferences, context — causes the model to lose the most important instructions in the noise. Repetitive procedures are better moved to skills, preferences left to memory, and only the fundamental project rules should be kept in the configuration file.

5. Trusting the correct syntax.

This is the moment when, as a 10xDevs participant, you should think like an engineer, not like an AI show spectator. Correct syntax is just the beginning of the conversation.

Are you able to avoid all these mistakes on the first run? Probably not, but systematically building good prompts and the entire agent environment – which we will practice exactly on live code in 10xDevs – will very quickly become a habit for you.

Prompt contract

A good agent prompt does not have to be long. However, it should answer a few specific questions:

  • Goal: what should be true after the work is completed?
  • Scope: which files, modules, or paths are in play?
  • Boundaries: what can the agent not change without consent?
  • Context: where to get the truth from: issue, logs, test, documentation, code snippet?
  • Verification: with which command or script will you check the result?
  • Report: what should it return at the end: diff summary, risks, tests, decisions?
  • Permissions: is it allowed to edit files, or only to explore and plan?

If you already have AGENTS.md / CLAUDE.md pre-configured, you can skip some of these elements (e.g., how to run tests after implementation).

Ready-made templates

The templates below are not magic formulas. Treat them as starting contracts that you adapt to your repository.

Bugfix

We are fixing a critical incident. Here is the support ticket:
<TICKET>
[short symptom description].
</TICKET>

The probable scope of changes concerns @{module/directory/file}
First, find the cause and propose a minimal plan, then implement the fix.
Do not change the public API without my permission.

Verification: run [test/build command] and describe the result.
Finally, summarize the changed files and risks.

Refactor

Refactor the [area], preserving current behavior.

Goal: [e.g., reduce duplication / separate concerns].
Scope: [files/modules].
Do not change the contract of public functions or data format.
If you find a need to change the public API, stop and inform me.

Verification: [command] and a brief description of how you confirmed no change in behavior.

Code review

Review changes for errors, regressions, and missing tests.

Here are the team rules: {rules}

Focus on behavior, security, edge cases, and adherence to project conventions.
Return only specific comments with file and line.
At the end, add a short list of missing tests or tests that cannot be evaluated from the diff.

Exploration without editing

Investigate how [area/feature] works in this project.

Find the most important parts of the code, dependencies, and data flow between components.
Finally, propose 2-3 possible approaches to solve [problem], along with their risks.
Stop before implementing and do not edit any files.

Implementation according to plan

Implement the approved plan from [link/section].

If the plan is outdated, stop and explain the conflict.
Verification: [command].

The most important pattern is simple: first explore by building up context, and then edit (this should be almost trivial after gathering the information beforehand).

The most important rules

  • Define success criteria, not the path. Before you start writing a prompt, consider: "How will the system know it has finished?". Lack of clear verification criteria is the fastest way to endless, unproductive iterations.
  • Break down instructions into layers. Design rules in CLAUDE.md, repeatable procedures in skills, preferences in memory, goal in the prompt. Each layer has its own scope — do not crush them into one place.
  • Do not impose the process on new models. Reasoning models require a specific goal and clear boundaries. Attempts to provide dozens of examples or to lead them step by step disrupt their effective, built-in analytical processes. Bend these rules in special cases when the model really has no idea how to approach a given problem.

What can you do right away?

  • Decision rule: if the task is unclear or involves multiple files, first ask for exploration and a plan, and only then allow editing.
  • Security control: every prompt with editing rights should have scope, limits, and a verification command or appropriate instructions in the project rules.
  • Action for today: based on the knowledge gathered, consider what your prompts have been lacking so far. Which elements should be given explicitly, and which should be included in the project's environment configuration?
