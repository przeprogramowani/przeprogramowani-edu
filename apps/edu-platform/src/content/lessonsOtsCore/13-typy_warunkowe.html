Wprowadzenie

W tej lekcji wracamy do wątku, który wstępnie pojawił się już przy okazji typów mapowanych. Mowa o typach warunkowych, czyli kolejnej użytecznej konstrukcji, która na poziomie systemu typów pozwala wprowadzić rozgałęzione scenariusze rodem z klasycznych algorytmów ze świata wartości.

Typy warunkowe

Jak sama nazwa wskazuje, typy warunkowe to element języka, który pozwala na dynamiczne tworzenie nowych typów w efekcie wykonania wyrażenia warunkowego:

// Jeśli typ T rozszerza string, zwróć typ true - w przeciwnym razie typ false
type IsString<T> = T extends string ? true : false;

type A = IsString<string>; // true
type B = IsString<number>; // false

Po co nam taki mechanizm? To kolejny element języka (po typach generycznych i typach mapowanych), który pozwala rozwijać elastyczne, dopasowujące się do potrzeb użytkownika typy danych.

Zaprezentujemy to na kilku kolejnych przykładach.

Selekcja fragmentów unii

Załóżmy, że w naszym kodzie znajduje się reprezentacja wszystkich identyfikatorów obiektów, które wprowadzano na kolejnych etapach rozwoju systemu:

type Identifier =
  | `user-${string}`
  | `user-v2-${string}`
  | `product-${string}`
  | `product-v2-${string}`
  | `order-${string}`
  | `order-v2-${string}`;

Wykorzystywanie takiego typu jest niepraktyczne - chcielibyśmy sprawić, żeby poszczególne funkcje czy obiekty przyjmowały wyłącznie identyfikator dopasowane do jednego rodzaju obiektu (np. User, Product lub Order). Jak zbudować taki typ?

Rozumowanie jest takie:

Zbuduj nowy typ, który pobiera wszystkie wartości unii rozpoczynające się do wzorca

Pierwsze podejście może wyglądać tak:

type Id<Pattern extends string> = `${Pattern}${string}` extends Identifier
    ? Identifier
    : never;

Niestety wszystkie typy utworzone w ten sposób będą typu never - TypeScript weźmie pod uwagę całą unię Identifier, więc żaden pojedynczy wzorzec nie spełni warunku “pattern extends Identifier”. My musimy się dostać do pojedynczych wartości, co możemy osiągnąć następująco:

type Id<Pattern extends string, Type = Identifier> = 
  Type extends `${Pattern}${string}` ? Type : never;

const u1: Id<"user"> = "user-1"; // ✅ OK
const u2: Id<"user-"> = "user-19"; // ✅ OK
const u3: Id<"user-"> = "user-v2-84"; // ✅ OK
const u4: Id<"user-v2"> = "user-19"; // ❌ Wartość musi się rozpoczynać od user-v2
const u5: Id<"user-v2"> = "product-15"; // ❌ Wartość musi się rozpoczynać od user-v2

Jak widać, przypisywane wartości są teraz porównywane do tych fragmentów unii, które pasują do wzorca.

Co więcej, nasz nowy typ może być wykorzystywany w wielu innych miejscach - może służyć do oznacza identyfikatorów innego typu, lub typować parametry funkcji:

type User = {
  id: Id<"user">;
  name: string;
  createdAt: Date;
};

function processUserV2(id: Id<"user-v2">) {
  console.log(id); // Argument id zawsze rozpocznie się od wartości "user-v2..."
}

Selekcja pól określonego typu

W połączeniu z typami mapowanymi, typy warunkowe umożliwiają łatwe klonowanie istniejących typów z zachowaniem pól określonego rodzaju. Zacznijmy od istniejącego typu Person, który posiada właściwości o trzech różnych typach (number, string i boolean):

type Person = {
  id: number;
  age: number;
  name: string;
  isActive: boolean;
}

Napiszmy nowy typ mapowany, który umożliwi doprecyzowanie typu Person do pól numerycznych:

// Dla przypomnienia
// type PropertyKey = string | number | symbol;

type PickPropsOfType<Type, PropType extends PropertyKey> = {
  [Prop in keyof Type as Type[Prop] extends PropType ? Prop : never]: Type[Prop];
};

Przeanalizujmy go krok po kroku:

  • Prop in keyof Type - przeiteruj po wszystkich polach danego typu

  • as - wykonaj mapowanie

  • Type[Prop] extends PropType ? - czy określone pole jest typu PropType?

    • Jeśli tak - zachowaj je (zwróć Prop)

    • Jeśli nie - usuń je (zwróć never)

Wykorzystanie tego typu wygląda następująco:

type NumericalProps = PickPropsOfType<Person, number>;

// Tożsame z:

type NumericalProps = {
    id: number;
    age: number;
}

Odwrotnością takiego typu byłoby pozbywanie się pól określonego rodzaju. Ten przypadek czeka na ciebie w sekcji ćwiczeń - powodzenia!

Selekcja pól do modyfikacji

W lekcji o typach wbudowanych poznaliśmy typy takie jak Required czy Partial, które na wszystkich polach danego obiektu stosują określone modyfikatory. A co, jeśli chcielibyśmy wybrać tylko pewną część obiektu, która ma się poddać takiej operacji? Bez typów warunkowych byłoby to niemożliwe - na szczęście Typescript daje nam taką możliwość.

Zacznijmy od początku - wprowadźmy typ, który wszystkie pola obiektu zamienia na readonly:

type FreezeProps<Type> = {
  readonly [Prop in keyof Type]: Type[Prop]
};

Co dalej? Potrzebujemy znaleźć sposób na to, aby wszystkie pola danego obiektu rozbić na dwa odrębne zestawy - jeden, gdzie readonly jest zastosowane i drugi, gdzie nie jest.

Możemy to osiągnąć dzięki poznanym wcześniej przecięciom:

type Person = {
  id: number;
  name: string;
};

const p1: Person = {
  id: 1,
  name: "John",
};

// Przypisanie działa, bo obiekty p1 i p2 mają ten sam zestaw właściwości (id, name)
const p2: {id: number} & {name: string} = p1;

Zbudujmy nasz typ FreezeProps w podobny sposób:

type FreezeProps<Type> = 
  { readonly [Prop in keyof Type]: Type[Prop]; } &
  {          [Prop in keyof Type]: Type[Prop]; };

Stosując go na typie Product uzyskamy dość nietypową konstrukcję, gdzie wszystkie pola mają dwa różne modyfikatory (możesz przetestować sklejanie różnych zestawów pól i obserwować efekty):

type Product = {
  id: number;
  name: string;
  price: number;
};

type FrozenProduct = FreezeProps<Product>;

type FrozenProduct = {
    id: number;
    name: string;
    price: number;
} & {
    readonly id: number;
    readonly name: string;
    readonly price: number;
}

Docelowo chcielibyśmy parametryzować zestaw pól, który ma zyskiwać określony modyfikator. W tym celu musimy rozszerzyć nasz typ FreezeProps o zestaw kluczy:

type FreezeProps<Type, Keys extends keyof Type> = 
  { readonly [Prop in keyof Type]: Type[Prop]; } &
  {          [Prop in keyof Type]: Type[Prop]; };

// Oznacz "id" i "name" jako readonly
type FrozenProduct = FreezeProps<Product, "id" | "name">;

Jak sprawić, aby dwa składowe obiekty zawierały odpowiednie pola? Tutaj do akcji wkraczają typy warunkowe, które sprawdzą, czy dane pole należy (lub nie) do naszego parametru:

type FreezeProps<Type, Keys extends keyof Type> = 
   { readonly [Prop in Keys as Prop extends Keys ? Prop : never]: Type[Prop]; } &
   {          [Prop in Exclude<keyof Type, Keys>]: Type[Prop]; };

I gotowe! Przeanalizujmy ten typ - składa się on z dwóch części:

  • Pola przekazywane jako parametr zyskują atrybut readonly

    • Jeśli nie wchodzą w skład parametru, zostają “wyrzucone” przez never

  • Pola, których nie ma w parametrze, uzupełniają te wyrzucone w drugiej części typu

    • Uzyskujemy to poprzez wbudowany typ Exclude, który z wszystkich typów usuwa podane Keys

Co najlepsze, to jeden z wielu sposobów na zbudowanie typu FreezeProps - w sekcji ćwiczeń zmierzysz się z jego alternatywą bez użycia typów warunkowych.

Podsumowanie

Podobnie jak w przypadku typów opartych o szablony stringów, potencjał typów warunkowych odkryjesz łącząc je z innymi konstrukcjami języka TypeScript. W połączeniu z dynamicznym rozwiązywaniem typów i słowem kluczowym infer, które poznasz w kolejnej lekcji, będą one znakomitym sposobem na typowanie narzędzi, które przekazujesz innym programistom.
