Wprowadzenie

Przedstawione w naszym kursie elementy języka TypeScript niemal jednoznacznie wskazują, że w projektach typu “greenfield” powinien być to domyślny wybór większości programistów. Zdajemy sobie jednak sprawę, że wielu z was pracuje w tzw. “kodzie zastanym”, wciąż rozwijanym w oparciu o JavaScript, gdzie migracja w trybie “wszystko na raz” jest niemożliwa do zrealizowania.

Właśnie dlatego jedną z lekcji naszego szkolenia poświęcimy dedykowanym poradom na temat migracji i integracji ekosystemów TS i JS, aby przejście w kierunku “typowanego JavaScriptu” odbyło się tak płynnie i szybko, jak to możliwe.

Integracja z JavaScriptem

Typowe kroki migracji zasymulujemy w oparciu o testowy projekt służący do konwersji HTMLa w postaci stringa na tzw. drzewo składniowe (w tej lekcji konkretna funkcjonalność nie jest istotna).

Struktura projektu wygląda następująco - src/parser to nasz “kod legacy”:

html-ast-parser/
└── src/
    ├── parser/
    │   ├── HTMLParser.js
    │   ├── NodeParser.js
    │   ├── AttributeParser.js
    │   └── TokenStream.js
    └── tests/
        ├── HTMLParser.test.js
        ├── NodeParser.test.js
        ├── AttributeParser.test.js
        └── TokenStream.test.js

Na tym przykładzie będziemy obserwować zachowanie kompilatora TypeScript w momencie, kiedy obok naszego modułu parser zaczynamy budować nową aplikację w oparciu o TS.

Kompilacja z JavaScriptem

Na początku musimy oczywiście zainstalować sam typescript (npm install typescript -DE) i dodać do projektu prostą konfigurację kompilatora i skrypt pomocniczy do kompilacji:

tsconfig.json

{
  "include": ["src/**/*"],
  "exclude": ["node_modules", "src/tests"],
  "compilerOptions": {
    "outDir": "./dist"
  }
}

package.json

{
  "build": "rm -rf dist && tsc"
}

Nasza nowa aplikacja od początku rozwijana będzie w języku TypeScript - na razie składa się z dwóch modułów, gdzie korzystamy z naszego parsera napisanego w języku JavaScript.

src/index.ts

import HTMLParser from "./parser/HTMLParser.js";
import {HTML_INPUT} from "./input";

const parser = new HTMLParser();
const ast = parser.parse(HTML_INPUT);

console.log(JSON.stringify(ast, null, 2));

src/input.ts

export const HTML_INPUT =
  '<div class="container"><p>Hello <span>World</span>!</p></div>';

Pierwsza kompilacja

Kiedy w oparciu o istniejącą konfigurację uruchomimy polecenie npm run build, w folderze dist zauważymy tylko dwa pliki:

html-ast-parser/
└── dist/
    ├── index.js
    └── input.js

Nowa aplikacja zostanie skompilowana poprawnie, ale bariera “TS/JS” będzie nie do przejścia - jak widać poniżej, w pliku wyjściowym znajduje się odwołanie do importowanego parsera, ale sam kompilator ten plik ignoruje - nie ma go w folderze dist.

dist/index.js

var HTMLParser_js_1 = require("./parser/HTMLParser.js");

(Zwróć również uwagę na sam format importów, którym na razie jest CommonJS)

Żeby kompilator TS brał pod uwagę wszystkie dostępne pliki, musimy dodać ustawienie allowJS: true.

tsconfig.json

{
  ...
  "compilerOptions": {
    "outDir": "./dist",
    "allowJs": true
  }
}

Nowe ustawienie powoduje, że odwołania do JavaScriptu z plików .ts są brane pod uwagę i po kompilacji umieszczane w folderze docelowym:

dist
├── parser
│   ├── AttributeParser.js
│   ├── HTMLParser.js
│   ├── NodeParser.js
│   └── TokenStream.js
├── index.js
└── input.js

Jeśli chcemy dodatkowo zaktualizować sam format docelowego JavaScriptu, w sekcji “compilerOptions” możemy dodać ustawienie:

{
  "module": "ES2022" // wartość przykładowa
}

Bezpieczeństwo typów

Warunki naszego projektu sprawiają, że naszego parsera nie możemy modyfikować w sposób inwazyjny (np. przepisując go całościowo na TypeScript). Jak w takim razie sprawić, żeby migracja przebiegała stopniowo, a nowa aplikacja pisana już w języku TypeScript była świadoma oczekiwanych typów?

Komentarze JSDoc

Bezpieczną metodą zapewniającą częściową zgodność typów jest oznaczenie fragmentów kodu JavaScript komentarzami w formacie JSDoc. Jeśli oznaczymy typy wejściowe i wyjściowe, kompilator TS zacznie brać je pod uwagę, uniemożliwiając niepoprawną kompilację.

src/parser/HTMLParser.js

class HTMLParser {
  /**
   * Creates AST from html string
   * @param {string} html - Input string
   * @returns {Object} Final AST
   */
  parse(html) {
    const tokenStream = new TokenStream(html);
    const nodeParser = new NodeParser(tokenStream);
    return nodeParser.parseNode();
  }
}

src/index.ts

const parser = new HTMLParser();
const ast1 = parser.parse(HTML_INPUT); // ✅ OK
const ast2 = parser.parse(32); 
// ❌ Błąd - Argument of type 'number' is not assignable to parameter of type 'string'.

W przypadku typów złożonych możemy wykorzystać kolejny element standardu dokumentacji JSDoc, a mianowicie tagi @typedef. Dzięki nim, na poziomie JavaScriptu możemy opisywać bardziej znaczące definicje typów (np. zwracanych) niż ogólny Object:

src/parser/HTMLParser.js

/**
 * AST Returned by HTMLParser
 * @typedef {Object} AST
 * @property {string} id - Some identifier
 * @property {number} quantity - Some quantity
 */

class HTMLParser {
  /**
   * Creates AST from html string
   * @param {string} html - Input string
   * @returns {AST} Final AST
   */
  parse(html) {
    const tokenStream = new TokenStream(html);
    const nodeParser = new NodeParser(tokenStream);
    return nodeParser.parseNode();
  }
}

export default HTMLParser;

Jak możesz zauważyć, praca z @typedef wygląda podobnie do typów w TypeScript - najpierw musimy opisać szczegóły nowego typu, a następnie możemy go zastosować w wybranym miejscu (np. jako typ zwracany) na funkcji parse.

W efekcie, po stronie TS otrzymamy znacznie lepsze podpowiadanie składni i możliwość weryfikacji typów złożonych:

Co najważniejsze, zmiana polegająca na dodawaniu komentarzy JSDoc nie będzie inwazyjna z punktu widzenia innych konsumentów modułu parser, które nadal będą mogłby współpracować z kodem JavaScript bez potrzeby migracji na TypeScript.

Zabezpieczanie plików JS w obrębie kodu JS

Wprowadzone komentarze pozwoliły nam zwiększyć bezpieczeństwo typów na granicy JavaScriptu i TypeScriptu. Niestety, na poziomie samej biblioteki JavaScript wciąż mogą znajdować się błędy, które przez brak integracji z TSem pozostają niewykryte.

Przykładowo, niech TokenStream określa poprzez JSDoc, że w jego konstruktorze zawsze powinien być przekazywany typ string:

src/parser/TokenStream.js

class TokenStream {
  /**
   * Creates a token stream from a string
   * @param {string} input - Input string
   */
  constructor(input) {
    this.input = input;
    this.pos = 0;
  }

Przez pomyłkę, w innym pliku JavaScript ktoś próbuje go wywoływać z typem number i zastanawia się, czemu cały kod nie działa:

src/parser/HTMLParser.js

const tokenStream = new TokenStream(10);

Co gorsza, tak zbudowany kod możemy teraz zbudować, a TypeScript - dopóki go o to nie poprosimy - nie wskaże tego błędu. Można to jednak zmienić.

Tworzona przez nas integracja z TypeScriptem sprawia, że kompilator może się zaopiekować nie tylko kodem TypeScript, ale również importowanymi zależnościami JavaScript.

Służy do tego flaga - checkJS:

"compilerOptions": {
    "allowJs": true,
    "checkJs": true,
    // ...
}

To ustawienie sprawia, że kompilator TypeScript zacznie weryfikować typy nie tylko na granicy JS/TS, ale również w obrębie całego drzewa zależności JavaScript. Kiedy spróbujemy skompilować taki kod, otrzymamy błąd kompilacji:

src/parser/HTMLParser.js:11:41 - error TS2345: Argument of type 'number' is not assignable to parameter of type 'string'.

11     const tokenStream = new TokenStream(10);

To rozwiązanie to prawdziwy game-changer. Dzięki niemu kompilator TypeScript może wprowadzić do projektu nowy poziom bezpieczeństwa bez wymuszania radykalnych zmian w strukturze projektu. Nasza biblioteka parser wciąż w 100% tworzona jest w czystym JSie, ale od teraz mamy nad nią nowego opiekuna w postaci kompilatora tsc.

Kiedy błędów jest zbyt dużo

Może się zdarzyć, że na skutek włączenia “opieki” tsc nad kodem JavaScript, odkrytych błędów będzie na tyle dużo, że w projekcie nastąpi paraliż. Aby tym zarządzić, mamy kilka opcji.

👉 Wybór plików do weryfikacji - @ts-check

Zamiast włączania flagi “checkJs” w obrębie całego projektu, możemy precyzyjnie wybierać te pliki, które powinny być weryfikowane. Służy do tego komentarz @ts-check dodawany na początku pliku JavaScript. Po jego dodaniu, TypeScript będzie oceniał dany moduł pod kątem zgodności typów.

src/parser/HTMLParser.js:

// @ts-check
class HTMLParser {
...

👉 Ignorowanie weryfikacji plików - @ts-nocheck

Jeśli natomiast flaga checkJs powinna pozostać włączona, a wybrany plik jest szczególnie problematyczny, możemy go wyłączyć z procesu weryfikacji przez komentarz @ts-nocheck.

src/parser/HTMLParser.js:

// @ts-nocheck
class HTMLParser {
...

Czas życia takiego komentarza powinien być stosunkowo krótki - możemy nim np. odblokować development w obrębie projektu, równolegle pracując nad jego usunięciem na osobnym branchu.

👉 Lokalne wyłączanie kontroli typów - @ts-ignore

Komentarz @ts-ignore pozwala wyłączyć sprawdzanie typów na poziomie jednej linijki. To bardzo radykalne podejście, które spełnia swoje założenia, ale nie polegamy się do niego przyzwyczajać:

// @ts-ignore
const tokenStream = new TokenStream(10);

👉 Tymczasowe oczekiwanie na błąd - @ts-expect-error

Polecaną alternatywą do @ts-ignore jest komentarz @ts-expect-error, którego działanie można zdecydowanie łatwiej odwrócić (a przynajmniej - nie zapomnieć o jego stosowaniu). Jak to działa?

Działanie tego komentarza można sprowadzić do następującego rozumowania:

  1. Wiem, że psuję wymagania względem typów

  2. Pomimo tego gwarantuję, że ten kod zadziała

  3. W momencie, kiedy “po drugiej stronie” typy będą naprawione, daj mi znać

To właśnie @ts-expect-error - pozwala nam unikać błędów kompilacji tylko do czasu, kiedy te błędy faktycznie występują.

// @ts-expect-error
const tokenStream = new TokenStream(10);

👉 Sterowanie ustawieniami kompilatora

Jeśli lokalne oznaczenia i komentarze są zbyt uciążliwe, a określony rodzaj problemu powtarza się w zbyt wielu miejscach, możesz rozważyć poluzowanie reguł kompilacji. Przede wszystkim dotyczy to ustawienia “strict”, które omawialiśmy w pierwszej lekcji tego kursu.

W momencie tworzenia tej lekcji, ustawienie “strict” odpowiada za 7 różnych aspektów systemu typów:

  • strictNullChecks

  • strictBindCallApply

  • strictFunctionTypes

  • strictPropertyInitialization

  • noImplicitAny

  • noImplicitThis

  • useUnknownInCatchVariable

W nowych projektach zdecydowanie łatwiej trzymać się wszystkich wspomnianych tutaj reguł. Jeśli natomiast weryfikację typów włączamy na setkach albo tysiącach plików w kodzie legacy, rygorystyczne zasady mogą zablokować development.

TypeScript zezwala nam na to, żeby zamiast jednej flagi “strict”: “true”, w naszym tsconfig pozostawić podzbiór reguł z powyższej listy, a część jawnie wyłączyć. Czasami może to być niezbędne aby proces migracji do TypeScriptu posuwał się do przodu (np. dedykowany programista), bez zablokowania reszty projektu błędami, na które wcześniej zespół przymykał oko bo były bardziej stylistyczne, a nie funkcjonalne.

W tym samym momencie powinniśmy sobie postawić za cel (np. czasowy) przywrócenie pełnego trybu zgodności, np. na koniec kwartału. Tutaj liczy się odpowiedzialność i dojrzałość zespołu - jeśli twoje środowisko spełnia te warunki, warto przetestować strategię luzowania reguł.

Brakujące deklaracje

Opisywane do tej pory narzędzia do integracji JS i TS zakładały, że do pewnego stopnia możemy wpływać na zawartość modułów JavaScript. W praktyce, w przypadku zewnętrznych bibliotek lub zależności, jest to po prostu niemożliwe. Tutaj łatwo o problemy:

  • 1) Włączamy allowJs aby korzystać z istniejących modułów JavaScript

  • 2) Reszta projektu TypeScript działa w oparciu o tryb strict

  • 3) Funkcje i zawartość modułów JS jest nieznana i określana jako any, co generuje konflikt z ustawieniem noImplicitAny (tryb strict)

Wyobraźmy sobie, że w naszym projekcie korzystamy z biblioteki legacy-utils.js, która eksportuje funkcje obliczeniowe:

// legacy-utils.js
exports.calculateTax = function (amount, rate) {
  return amount * rate;
};

exports.isPrime = function (number) {
  if (number < 2) return false;
  for (let i = 2; i < Math.sqrt(number); i++) {
    if (number % i === 0) return false;
  }
  return true;
};

Importując tę bibliotekę w projekcie TypeScript, możemy napotkać problem:

import { calculateTax, isPrime } from 'legacy-utils';

const tax = calculateTax(100, 0.2); // Could not find a declaration file for module 'legacy-utils'.js. Module implicitly has an 'any' type

TypeScript nie wie, jakiego rodzaju dane calculateTax i isPrime przyjmują, ani co zwracają. Bez deklaracji typów kompilator nie pozwoli na użycie tych funkcji, ponieważ niezgodność z zasadą noImplicitAny uniemożliwia bezpieczną integrację.

Rozwiązanie: użycie słowa kluczowego declare

Słowo kluczowe declare pozwala na zdefiniowanie typów dla kodu, który jest zewnętrzny i nieznany TypeScriptowi. Nie wymaga ono modyfikacji oryginalnego kodu, co czyni je idealnym rozwiązaniem w przypadku zewnętrznych bibliotek lub modułów JS. Możemy utworzyć dedykowany plik deklaracyjny legacy-utils.d.ts, który opisze funkcje eksportowane przez bibliotekę:

// legacy-utils.d.ts
declare module 'legacy-utils' {
  export function calculateTax(amount: number, rate: number): number;
  export function isPrime(number: number): boolean;
}

Warto zwrócić uwagę, że plik deklaracyjny nie dodaje żadnej funkcjonalności do modułu – jego jedyną rolą jest poinformowanie TypeScriptu, jakie typy są obsługiwane przez funkcje.

Jeżeli spróbujemy dodać implementację do pliku d.ts, kompilator zgłosi błąd.

Mechanika declare

Słowo kluczowe declare informuje TypeScript, że deklarowany element istnieje w kodzie w czasie działania aplikacji (runtime), ale szczegóły jego implementacji są pomijane. Deklaracja jest „obietnicą” wobec TypeScriptu, że taki element rzeczywiście istnieje, i opisuje jego typy. Ważne jest zrozumienie, że TypeScript nie weryfikuje, czy deklaracje są zgodne z rzeczywistą implementacją. Jeśli na przykład w deklaracji podamy błędne typy:

// legacy-utils.js
exports.calculateTax = function (amount, rate) {
  return amount * rate;
};

// legacy-utils.d.ts
declare module './legacy-utils' {
  export function calculateTax(amount: string, rate: string): string; // Błędne typy
}

TypeScript nie zgłosi błędu przy kompilacji, ale nasz kod może zakończyć się błędem w czasie działania aplikacji, ponieważ calculateTax rzeczywiście oczekuje liczb, a nie ciągów znaków. Stąd warto wzbogacać pliki JS o JSDoc, które zapewnia nam informacje o typach.

Automatyczne generowanie plików deklaracyjnych

Zamiast manualnie tworzyć pliki d.ts warto skorzystać z narzędzi automatyzujących to zadanie. Dla pojedynczych modułów JS wewnątrz naszego projektu wystarczy skorzystać z tsc:

npx -p typescript tsc src/*.js --declaration --allowJs --emitDeclarationOnly --outDir @types

Dla zewnętrznych bibliotek warto skorzystać z narzędzia dts-gen.

Niestety musimy liczyć się z tym, że automatyczne generowanie plików deklaracyjnych z modułów pozbawionych JSDoc prowadzi do dużej ilości “any”, które będziemy musieli ręcznie zastąpić.

Rozszerzanie istniejących modułów

Słowo kluczowe declare jest również używane, gdy chcemy rozszerzyć lub zmodyfikować typy istniejących modułów. Załóżmy, że korzystamy z biblioteki express, która udostępnia interfejs Request, ale potrzebujemy dodać do niego dodatkowe pole user (obiekt dodawany przez jeden z napisanych przez nas middleware’ów do autoryzacji). Możemy to zrobić, tworząc plik express-augment.d.ts:

// @types/express-augment.d.ts
import 'express';

declare module 'express' {
  export interface Request {
    user?: { id: string; role: string };
  }
}

Teraz pole user jest dostępne w całym projekcie jako część interfejsu Request, bez potrzeby modyfikowania oryginalnych plików biblioteki.

Tworząc pliki deklaracyjne, warto pamiętać o kilku zasadach:

  • Deklaracje powinny być minimalne, tj. zawierać tylko te funkcje i obiekty, które są używane w projekcie. Nie ma potrzeby deklarowania całej biblioteki, jeśli korzystamy tylko z jednej funkcji.

  • Typy powinny być tak dokładne, jak to możliwe, aby uniknąć problemów z bezpieczeństwem typów. W razie wątpliwości warto używać ogólnych typów, takich jak unknown czy Record<string, any>, zamiast any.

  • Dobrą praktyką jest trzymanie plików deklaracyjnych w dedykowanym katalogu, np. @types, co ułatwia ich organizację i utrzymanie.

Automatyzacja

Jeśli w nadchodzącym projekcie twoim zadaniem będzie migrowanie kodu JavaScript na TypeScript, spróbuj rozpocząć od narzędzia, które wstępny szkic migracji przygotuje za ciebie. Dobre praktyki migracji

Jak każda zmiana w obrębie projektu, migracja kodu JavaScript na TypeScript wiąże się z wyzwaniami wychodzącymi poza aspekty techniczne. Na koniec tej lekcji dzielimy się kilkoma poradami, które zwiększą twoje szanse na sukces:

  • Inicjatywy związane z migracją całego projektu w jednym podejściu to duże ryzyko. Zamiast tego określ taką taktykę migracji, która umożliwi zmianę plików JS→TS na żądanie, bez wymuszania zmian w modułach, które nie są na to gotowe.

  • Konfiguracja kompilatora TypeScript powinna być na tyle elastyczna, żeby brać pod uwagę nie tylko dobre praktyki nowego kodu, ale również akceptować pewne wady kodu istniejącego. Pamiętaj, że ostatecznym celem jest ukończenie migracji, a nie zwiększenie frustracji na działanie TypeScriptu.

  • W zależności od dojrzałości zespołu i stanu samego projektu, tryb strict może być ustawiony bardziej lub mniej precyzyjnie. Niektórzy radzą wymuszanie pełnej zgodności od pierwszego migrowanego pliku, inni decydują się na więcej swobody aż do końca procesu. Od ciebie będą zależeć priorytety, a wynikają one ze środowiska, w którym pracujesz.

  • Refaktoryzacja może działać w dwóch kierunkach - do istniejącej aplikacji JavaScript możemy chcieć dołączać moduły TypeScript, albo odwrotnie, budować aplikacje TypeScript w oparciu o istniejące moduły JavaScript. Każdy z przypadków będzie wiązał się z nieco inną konfiguracją kompilacji, więc warto przeanalizować konsekwencje obu strategii.

  • Dla osób, które wcześniej nie pracowały z TypeScriptem, wprowadzanie rygorystycznych reguł kompilacji może kojarzyć się z ograniczeniami i odbieraniem swobody programowania. Zadbaj o to, aby zespół odpowiednio przygotować - pokazać narzędzia, dobre praktyki i przede wszystkim korzyści, które TypeScript może wnieść do projektu. Zdecydowanie ważniejsze od aspektów technicznych będzie takie przygotowanie zespołu, żeby wszyscy “brali udział w migracji”, a nie narzekali na zaskakujące pomysły jednego inżyniera.

Podsumowanie

Integracja JavaScript z TypeScript, podobnie jak migracja w kierunku TSa, wymaga początkowego nakładu pracy — dostosowania projektu, skonfigurowania narzędzi i zrozumienia różnic między tymi językami. Naszym zdaniem cały wysiłek jest zdecydowanie opłacalny. TypeScript wnosi do projektu większą przejrzystość kodu, bezpieczeństwo typowania i skalowalność budowanych rozwiązań, co jest szczególnie widoczne w większych projektach i zespołach.

Przeprowadzając zespół przez proces migracji pamiętaj, że cała inicjatywa dotyczy aspektów technicznych ale i czysto ludzkich. Zarządzanie zmianą powinno dotyczyć zbudowania odpowiednich kompetencji, podzielenia się wiedzą, korzyściami i sposobami na problemy, które mogą się pojawić już po starcie. Pomimo tego jest to inicjatywa warta swoich kosztów, a my przez cały ten kurs staraliśmy się to udowodnić.
