Wprowadzenie

Poznane do tej pory elementy systemu typów, takie jak generyki, typy mapowane, typy warunkowe a także te oparte o szablony stringów, pozwolą ci zrealizować wiele scenariuszy typowania danych na różnym poziomie złożoności.

W niektórych sytuacjach opisanie skomplikowanych zależności może być jednak na tyle trudne, że bez dodatkowych mechanizmów pozwalających na "rozpakowywanie" lub analizowanie typów wejściowych trudno byłoby zapewnić oczekiwaną spójność i zgodność typów.

Tutaj właśnie pojawia się infer, czyli nowe słowo kluczowe i zarazem mechanizm dedukcji typów.

Czym jest infer?

infer pozwala zdefiniować zmienną typu wewnątrz warunku typu (extends w Conditional Type), co umożliwia dynamiczne wyciąganie części złożonego typu. Używa się go głównie w celu "rozpakowania" typów generycznych.

Pierwsze kroki

Słowo kluczowe infer (en. wnioskuj / wywnioskuj) wykorzystujemy w połączeniu z klauzulą extends w typie warunkowym.

Infer można traktować jak “pułapkę na typ”, którego jeszcze nie znamy. To przystępna metafora, która pomoże ci zrozumieć działanie tego mechanizmu. Przykładowo, możemy utworzyć typ generyczny z parametrem T, gdzie w momencie wywołania, nasza pułapka o nazwie infer przechwyci przekazywany typ. My z kolei możemy tego typu użyć w dalszej części całego wyrażenia (to coś na wzór zmiennej lokalnej).

W najprostszej wersji, korzystanie z infer nie różni się od bezpośredniego wykorzystywania parametru generycznego:

type JustType<T> = T;

type NumberType = JustType<number>; // number
type StringType = JustType<string>; // string
type BooleanType = JustType<boolean>; // boolean

type InferType<T> = T extends infer R ? R : never;

type NumberType2 = InferType<number>; // number
type StringType2 = InferType<string>; // string
type BooleanType2 = InferType<boolean>; // boolean

Skoro na pierwszy rzut oka nie ma tutaj oczywistych korzyści, to w jakim celu wprowadzono ten mechanizm do Typescriptu? Poniżej przedstawiamy najpopularniejsze zastosowania.

Mockowanie funkcji

Pisząc testy jednostkowo często wykorzystujemy tzw. mechanizm mockowania - funkcje wywołujące realne endpointy lub korzystające z systemu plików zastępujemy przez proste “zaślepki”, które w momencie wywołania natychmiast podają nam oczekiwaną wartość.

Aby mockowanie miało sens, definicje obu funkcji (prawdziwej i “udawanej”) muszą być identycznie - zarówno na poziomie parametrów jak i wartości zwracanej. Jak w takim razie rozpoznać typy, które zastosowano w funkcji bazowej?

Potrzebujemy do tego dwóch typów pomocniczych:

  • Typu przechwytującego parametry funkcji

  • Typu przechwytującego typ zwracany

Zanim zaczniemy, wprowadźmy najbardziej ogólny typ dowolnej funkcji:

type AnyFunction = (...args: any[]) => any;

const f1: AnyFunction = () => false;
const f2: AnyFunction = (name: string) => `Hello, ${name}`;
const f3: AnyFunction = (id: number) => { name: 'john' };

Typ AnyFunction akceptuje funkcje o dowolnym zestawie parametrów i typu zwracanego - użyjemy tej składni aby odkryć jakie funkcje biorą udział w testach.

Zacznijmy od rozpoznawania parametrów. Potrzebujemy do tego nowego typu generycznego FnParamType, który w miejsce parametrów zastawi pułapkę na typ (infer P):

type FnParamType<T> = T extends (...args: infer P) => any ? P : never;

type MyFunc1 = (a: string, b: string) => boolean;
type Params1 = FnParamType<MyFunc1>; // [a: string, b: string]

type MyFunc2 = (user: {name: string}) => string;
type Params2 = FnParamType<MyFunc2>; // [user: { name: string }]

Zweryfikujmy działanie naszego nowego typu budując funkcję passParams, która akceptuje:

  • funkcję bazową

  • parametry funkcji bazowej

Zgodność typów musi być oczywiście zachowana.

function passParams<T extends (...args: any[]) => any>(
 fn: T,
 params: FnParamType<T>
): FnParamType<T> {
  return params;
}

// Asynchroniczna funkcja korzystająca z zewnętrznego serwisu (uproszczona implementacja)
function checkPermissions(userId: number, featureId: string): Promise<boolean> { 
  return Promise.resolve(false);
}

passParams(checkPermissions, [10, 'modals']); // OK
passParams(checkPermissions, [12, 15]); // ❌ Błąd - drugi z parametrów musi być typu string

Mamy to - czas na drugi krok, czyli rozpoznawanie typu zwracanego. Analogicznie jak wcześniej, zaczniemy od ogólnej definicji funkcji, ale tutaj infer zastosujemy na końcu. Dzięki temu odkryjemy typ wartości zwracanej:

type FnReturnType<T> = T extends (...args: any[]) => infer R ? R : never;

Przetestujmy nowy typ:

function getMockedValue<T extends (...args: any[]) => any>(
  fn: T,
  returnValue: FnReturnType<T>
): FnReturnType<T> {
  return returnValue;
}

getMockedValue(checkPermissions, Promise.resolve(false)); // ✅ OK
getMockedValue(checkPermissions, false); // ❌ Błąd - funkcja checkPermissions zwraca Promise

Mamy więc dwa typy, które pozwalają nam rozpoznawać zarówno parametry jak i wartości zwracane.

Połączmy wszystko w jedną całość aby zbudować docelową funkcję:

function createMock<T>(
  fn: T,
  expected: FnReturnType<T>
): (...params: FnParamType<T>) => FnReturnType<T> {
  return (...params: FnParamType<T>) => expected;
}

const mockFn1 = createMock(checkPermissions, Promise.resolve(false)); // ✅ OK
const mockFn2 = createMock(checkPermissions, Promise.resolve(true)); // ✅ OK
const mockFn3 = createMock(checkPermissions, 'what'); 
// ❌ Argument of type 'string' is not assignable to parameter of type 'Promise<boolean>'

mockFn1(10, 'none').then(result => { console.log(result); }) // false
mockFn2(10, 'none').then(result => { console.log(result); }) // true

Jak działa createMock? Przede wszystkim przyjmuje ona dwa parametry:

  • funkcję do zastąpienia (fn)

  • statyczną wartość zwracaną o typie zwracanym z funkcji fn

Logika jest naprawdę prosta - niezależnie co przekażesz do środka, zawsze zwracaj tę samą wartość czyli expected.

W trakcie wywoływania funkcji createMock TypeScript dba o to, żeby drugi parametr miał ten sam typ co wartość zwracana z funkcji. Co ważne, otrzymana w efekcie funkcja mockująca przyjmuje dokładnie te same parametry co oryginalna, więc bez obaw będziemy mogli jej używać jako zastępstwo oryginału.

Podsumowując - infer w tym przykładzie posłużył jak pułapka na typ dwóch elementów funkcji. Pomaga to budować bardziej złożone rozwiązania, gdzie relacje między typami muszą być ściśle określone.

👉 Warto wiedzieć: TypeScript posiada wbudowane typy pozwalające uzyskać to samo zachowanie co nasze dedykowane typy ze słowem kluczowym infer. Przetestujesz je w sekcji ćwiczeń.

Wyszukiwanie wzorców

Wróćmy do przykładu, w którym budujemy odporny na użytkownika system typów dla modułu służącego do kontaktu z bazą danych. Użytkownik może wykorzystywać dynamicznie tworzone typy, które będą przewidywać potencjalne błędy w zastosowaniu.

Każdy z typów przypisanych do tabeli powinien mieć cztery metody:

  • insert<TableName>, delete<TableName>, update<TableName>, find<TableName>

Dla spójnego interfejsu chcemy zapewnić, aby TableName zawsze pojawiał się w liczbie pojedynczej i zaczynał się od wielkiej litery:

  • createTable<User, ‘user’> => insertUser, deleteUser, …

  • createTable<User, ‘users’> => insertUser, deleteUser, …

  • createTable<User, ‘User’> => insertUser, deleteUser, …

TypeScript musi nas wyposażyć w narzędzie, które z wielu przekazywanych przez użytkownika nazw tabel zawsze wyłuska nazwę w liczbie pojedynczej. Jak to zrobić?

Tutaj z pomocą przychodzi nam infer i mechanizm tzw. pattern matchingu, czyli wyszukiwania wzorców:

type NormalizedTableName<TableName extends string> = 
  TableName extends `${infer Base}s` ? Base : TableName;

Dzięki wprowadzeniu infer, do nowego typu NormalizedTableName możemy wprowadzić logikę:

  • Jeśli nazwa typu kończy się na “s”, pobierz poprzedzający fragment

  • W przeciwnym razie zwróć całą nazwę

Infer pozwala nam zaglądać w poszczególne fragmenty danego typu i pobierać z nich to, co akurat nas interesuje. Do pełnego zwycięstwa potrzebujemy jeszcze upewnienia się, że pierwsza litera typu będzie “wielka”:

type TableToFunc<TableName extends string> = Capitalize<NormalizedTableName<TableName>>;

To oczywiście dość naiwna implementacja wspierająca przypadki z języka angielskiego, ale dobrze pokazuje ona działanie naszej pułapki na typy.

Zastosowanie nowego typu

Po wprowadzeniu naszego typu do formatowania nazw, możemy go teraz wykorzystać do zbudowania docelowego typu Table:

type MethodNames<T extends string> =
  | `insert${TableToFunc<T>}`
  | `delete${TableToFunc<T>}`
  | `update${TableToFunc<T>}`
  | `find${TableToFunc<T>}`;

type Table<Model, TableName extends string> = {
  [Prop in MethodNames<TableName>]: () => void;
}

Typ mapowany Table będzie zawierał cztery nazwy metod dopasowanych do naszej tabeli. Na potrzeby tego przykładu pomijamy parametry i wartości zwracane, bo te fragment omawialiśmy w poprzednich lekcjach.

Dekompozycja istniejących typów

Ta najbardziej uniwersalna korzyść wynikająca ze stosowania narzędzia infer to możliwość rozpakowywanie typów, do których nie mamy dostępu. Można to wykorzystać choćby do odpakowania typów, które wykorzystywane są w standardowych typach Promise, Map czy Set:

type UnwrappedPromise<T> = T extends Promise<infer U> ? U : T;
type UnwrapMapEntries<T> = T extends Map<infer K, infer V> ? [K, V] : never;
type UnwrappedSet<T> = T extends Set<infer U> ? U : never;
type UnwrapArrayPromise<T> = T extends Promise<Array<infer U>> ? U : never;

Nie jest to coś, od czego zwykle będzie zależał biznesowy sukces, ale wciąż warto być świadomym możliwości tego mechanizmu i rozważyć wprowadzenie go do firmowych typów, które mogą wchodzić w skład reużywalnej biblioteki.

Podsumowanie

Chociaż praca z mechanizmem dedukcji typów jest naprawdę interesująca i pokazuje potencjał całego silnika typów w TypeScript, to warto korzystać z niego rozważnie. Przesadne poleganie na infer w momencie, kiedy istnieją prostsze rozwiązania, może prowadzić do kodu o nadmiernej złożoności.

Naszą rekomendacją jest dobre zaznajomienie się z typami wbudowanymi w sam język, a następnie ich rozszerzanie o nowe generyki i typy mapowane. Jeśli to nie wystarcza, a tobie zdarzy się pracować np. z bibliotekami dla innych programistów, gdzie typy należy rozwiązywać dynamicznie, być może słowo kluczowe infer stanie się niezbędne.

To, co dodatkowo zyskujesz poznając infer, to możliwość łatwiejszego poruszania się po projektach i bibliotekach Open Source - przyglądanie się temu, jak inni pracują z tym rozwiązaniem, może być kolejnym sposobem na pełniejsze zrozumienie ekosystemu języka TypeScript. Przykłady takich projektów przedstawimy w ostatnich lekcjach modułu “Core Pro”.

Ćwiczenia

  • 441-infer-reward - Twoim zadaniem jest implementacja typu RewardRadar<T>, który na podstawie skanów terenu rozpozna lokalizację ukrytej nagrody. Skany dostarczane są w postaci innych typów, które należy odpowiednio zinterpretować. Dokumentacja radaru została dostarczona w postaci testu jednostkowego oraz kilku przykładowych sesji skanowania.

  • 442-infer-flags - Interfejs rozwijanej aplikacji webowej opiera się o tzw. feature flagi, czyli przełączniki sterujące dostępem do wybranych funkcjonalności. Niestety, w obecnej wersji backendu ten sam serwis zwraca kolekcję flag dla dwóch wersji interfejsu użytkownika, co staje się coraz trudniejsze w utrzymaniu i dokumentowaniu. Zespół rozwijający nową wersję frontendu postanawia przesłonić starsze typy poprzez typ mapowany, który wyszuka flagi dla aplikacji w wersji “v2” i wystawi je dla swojej aplikacji, bez problematycznego oznaczenia wersji. Dokończ implementację typu FlagsV2<T> aby zrealizować ten cel.
