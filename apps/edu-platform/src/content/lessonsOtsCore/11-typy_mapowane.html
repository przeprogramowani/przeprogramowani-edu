Wprowadzenie

W poprzedniej lekcji poznałeś zestaw wbudowanych typów pomocniczych, które z wykorzystaniem mechanizmu generyków modyfikują tworzone przez nas typy bazowe.

W tej lekcji poznasz mechanizm, dzięki któremu twórcy TypeScriptu byli w stanie utworzyć opisywane wcześniej narzędzia. Rozpoczynamy pracę z typami mapowanymi (mapped types).

Typy mapowane - pierwsze kroki

Obejrzyj kolejny film aby poznać typy mapowane:

Przedstawiony na filmie typ Clone prezentuje się następująco:

type Person = {
  firstName: string;
  lastName: string;
  age: number;
};

type Clone<T> = {
  [Prop in keyof T]: T[Prop];
};

type PersonClone = Clone<Person>;

Mechanizm typów mapowanych będziemy wykorzystać przez kilka kolejnych lekcji - pierwszym przykładem ich zastosowania będzie odtworzenie typów wbudowanych, które poznałeś we wcześniejszej wersji kursu.

Będzie to ćwiczenie które pozwoli nam uniknąć poczucia pracy z “magicznymi” typami wbudowanymi bo szybko okaże się, że ich odpowiedniki można z łatwością zbudować samemu.

Odtwarzanie typów wbudowanych

Readonly - pola do odczytu

Spróbujmy teraz odtworzyć wbudowany typ Readonly<T> - jego zachowanie opisujemy poniżej:

type Person = {
  firstName: string;
};

// Obiekt staje się zamknięty na modyfikacje
const person: Readonly<Person> = {
  firstName: "John",
};

person.firstName = "Mark";
         // ❌ ^ - Cannot assign to 'firstName' because it is a read-only property.

Odtworzenie tego mechanizmu z wykorzystaniem typu mapowanego wygląda następująco:

type ReadonlyClone<Type> = {
  readonly [Prop in keyof Type]: Type[Prop];
};

const person2: ReadonlyClone<Person> = {
  firstName: "John",
};

person.firstName = "Mark";
         // ❌ ^ - Cannot assign to 'firstName' because it is a read-only property.

Nasz typ ReadonlyClone wykorzystuje słowo kluczowe readonly po to żeby “przepisać” wszystkie pola obiektu bazowego właśnie w trybie tylko do odczytu. Dzięki temu uzyskujemy blokadę dalszych modyfikacji pól obiektu z tym typem.

Żeby operację odwrócić, możemy również wprowadzić “usuwacza” trybu readonly - nowy typ wygląda niemal identycznie - różni się dodatkiem znaku minus, usuwającego modyfikator pola.

type ReadonlyRemoval<T> = {
  -readonly [P in keyof T]: T[P];
};

W ramach eksperymentu możemy połączyć go z poprzednio wprowadzonym typem ReadonlyClone aby “wyzerować” jego działanie:

const justPerson: ReadonlyRemoval<ReadonlyClone<Person>> = {
  firstName: "John",
};

justPerson.firstName = "Mark"; 
console.log(justPerson); // { firstName: "Mark" }

Partial / Required - pola opcjonalne i wymagane

Kolejny typ wbudowany, który spróbujemy odtworzyć, to Required - ustawia on wszystkie pola jako wymagane:

type Account = {
  nickname?: string;
};

const account: Required<Account> = {};
   // ❌ ^ - Property 'nickname' is missing in type '{}' but required in type 'Required<Account>'

Jego przeciwieństwem jest typ Partial, który pozwala na budowanie “częściowo gotowych” obiektów:

type ProductDetails = {
  name: string;
};

const product: Partial<ProductDetails> = {};

product.name = "MacBook";

Jak odtworzyć mechanizm dostępności wszystkich pól bez użycia typu wbudowanego? Wykorzystamy ten sam operator (?) co w przypadku standardowego typu - tyle, że w kontekście typu mapowanego:

// Wszystkie pola stają się opcjonalne
type OptionalClone<Type> = {
  [Prop in keyof Type]?: Type[Prop];
};

// Wszystkie pola stają się wymagane
type RequiredClone<Type> = {
  [Prop in keyof Type]-?: Type[Prop];
};

Sprawdźmy najpierw pierwszy z nowych typów - istniejące pola staną się opcjonalne:

type User = {
  age: number;
};

const u1: OptionalClone<User> = {age: 25};
const u2: OptionalClone<User> = {};

A teraz drugi z nowych typów - istniejące pola staną się wymagane:

type Cart = {
  discount?: number;
};

const cart: RequiredClone<Cart> = {};
 // ❌ ^ - Property 'discount' is missing in type '{}' but required in type 'RequiredClone<Cart>'

Pick i Omit - wybór i pomijanie określonych pól

Dwa kolejne typy wbudowane, które możemy odtworzyć, dotyczą selekcji pól obiektu. Pierwszy z nich to Pick, który pozwala określić te pola, które powinny pozostać częścią nowego typu:

type Book = {
  title: string;
  author: string;
  pages: number;
};

const book: Pick<Book, "title" | "author"> = {
  title: "1984",
  author: "George Orwell",
  // pages: 422
  // ❌ ^ - (...) 'pages' does not exist in type 'Pick<Book, "title" | "author">'
};

Omit to jego przeciwieństwo - w jego przypadku wskazujemy pola do pominięcia:

type BankAccount = {
  id: number;
  bankName: string;
  balance: number;
};

const accountPreview: Omit<BankAccount, "balance"> = {
  id: 133,
  bankName: "Bank of TypeScript",
  // balance: 1000,
  // ❌ ^ - (...) 'balance' does not exist in type 'Omit<BankAccount, "balance">'
};

Jak odtworzyć pierwszy z nich? Musimy wykorzystać generyk z dwoma parametrami, gdzie drugim parametrem będą klucze pierwszego.

Wykorzystajmy uwagę z początku lekcji o tym, jak określać dozwolone pola obiektu mapowanego:

[Prop in /* POLA DOZWOLONE */]

Stąd:

type PickClone<Type, AllowedKeys extends keyof Type> = {
  [Prop in AllowedKeys]: Type[Prop];
};

// lub w dwóch krokach:

type TypeKeys<Type> = keyof Type;

type PickClone<Type, AllowedKeys extends TypeKeys<Type>> = {
  [Prop in AllowedKeys]: Type[Prop];
};

Dzięki temu uzyskamy możliwość wskazywania podzbioru pól wchodzących w skład typu bazowego:

const book2: PickClone<Book, "title" | "author"> = {
  title: "1984",
  author: "George Orwell",
};

W przypadku typu Omit sytuacja jest nieco bardziej wymagająca - zacznijmy od ogólnego kształtu nowego typu. Znowu potrzebujemy dwóch parametrów - typu bazowego oraz listy kluczy niedozwolonych:

type OmitClone<Type, BannedKeys extends keyof Type> = {
  [Prop in /* ? */]: Type[Prop];
};

Jak sprawić, żeby nowy obiekt nie zawierał tych kluczy? Zróbmy krok wstecz i zobaczmy, jak mogłoby to wyglądać w standardowej funkcji:

const keys = ['a', 'b', 'c'];
const banned = ['b'];

const filteredKeys = keys.filter(k => banned.includes(k) ? false : true); // ['a', 'c']

W przypadku typów mapowanych, naszym filtrem staje się połączenie słowa kluczowego “as” oraz ewentualnego zwracania typu never:

type OmitClone<Type, BannedKeys extends keyof Type> = {
  [Prop in keyof Type as Prop extends BannedKeys ? never : Prop]: Type[Prop];
};

Można to interpretować w ten sposób:

  1. Prop in keyof Type - Iteruj przez wszystkie pola (Prop) z kluczy typu Type (keyof Type),

  2. as - zmapuj pole na coś nowego:

    1. Jeśli pole należy do zestawu BannedKeys, zwróć typ never (pomiń to pole),

    2. W przeciwnym razie zachowaj go.

Co ciekawe, podobnie jak w przypadku zwykłej funkcji, nasz “filtr” można wyodrębnić:

type FilterType<A, B> = A extends B ? never : A;

type OmitClone<Type, BannedKeys extends keyof Type> = {
  [Prop in keyof Type as FilterType<Prop, BannedKeys>]: Type[Prop];
};

const acc: OmitClone<BankAccount, "balance"> = {
  id: 23,
  bankName: "Bank of TS",
  // balance: 24,
  // ❌ ^ 'balance' does not exist in type 'OmitClone<BankAccount, "balance">'
};

Typy warunkowe w akcji

Powyższy przykład wymusił na nas zastosowanie tzw. typu warunkowego (conditional type) - to kolejny ważny element algebry typów, który będziemy poznawać w kolejnych lekcjach. Jego podstawowe założenia nie powinny być dla ciebie problemem.

Pamiętaj jednak, że wszystkie operacje odbywają się na typach, a nie wartościach:

interface User {
  firstName: string;
}

type IsString<T> = T extends string ? true : false;

type Result = IsString<User["firstName"]>; // true

const t1: Result = true; // OK
const t2: Result = false; // ❌ Type 'false' is not assignable to type 'true'

Podsumowanie

Poznane w tej lekcji typy mapowane to niezwykle ważny element zaawansowanego systemu typów języka TypeScript. Być może od strony aplikacji nie będziesz go wykorzystywał aż tak często, ale budując reużywalne biblioteki i narzędzia, będzie on niezwykle ważny.

W kolejnych lekcjach połączymy go z nowymi rodzajami typów i zaprezentujemy przykłady operacji, w których to dynamiczne mapowanie pól i typów staje się niezbędne.

Co ważne, typy mapowane to również istotny wyróżnik w debacie “typy vs interfejsy” - na dzisiaj tę funkcjonalność można osiągnąć wyłącznie z poziomu typów.

Ćwiczenia

  1. 411-mapped-access - Rozwijasz projekt, w którym typy mapowane wykorzystywane są do określania dostępu do poszczególnych stron w aplikacji. Zapewniają one, że żadna nowa strona nie zostanie pominięta, a dostęp do niej przypadkowo zablokowany. Twoim zadaniem jest zbudowanie mapowanego typu PagesAccess oraz implementacja funkcji checkAccess, które potwierdzą dostęp do każdej dostępnej strony. Działanie mechanizmu potwierdzą testy jednostkowe.

  2. 412-mapped-extender - Zaimplementuj typ mapowany type FieldExtender<T, K>, który dla podanego typu T tworzy nowy typ rozszerzony - lista pól jest taka sama, ale zamiast typów prostych każde pole staje się typem złożonym z dodatkowym polem “value”. Dostępne testy i kod przedstawiają docelowy obiekt, którego kompilacja powinna zostać przez ciebie odblokowana.
