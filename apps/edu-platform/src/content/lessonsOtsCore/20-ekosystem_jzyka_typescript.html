Wprowadzenie

Dojrzałość języka programowania nie wynika wyłącznie z jego składni, wsparcia dla szczególnego rodzaju typów lub łatwej instalacji na wielu platformach, ale w głównej mierze z tego, czy powstała wokół niego silna społeczność. Organiczny wzrost popularności powoduje, że dany język może stać się fundamentem narzędzi i projektów realizujących o wiele bardziej kreatywne zadania, niż wynikałoby z założeń jego twórców.

W tej lekcji poznasz kilka takich projektów z ekosystemu TypeScript, które pokazują szerokie zastosowanie tego języka. Przedstawimy ci nie tylko popularne biblioteki takie jak Zod czy Drizzle, ale również narzędzia ułatwiające migrowanie projektów, integrację z API czy wykonywanie modyfikacji kodu w dużych projektach.

Każdy element przedstawimy w oparciu o rzeczywisty scenariusz, który możesz napotkać w trakcie rozwijania aplikacji produkcyjnych. Zaczynajmy!

Biblioteki typów

Wyobraź sobie, że twoim zadaniem jest napisanie funkcji zwracającej wartość danego pola z obiektu JSON, na podstawie przekazanej ścieżki. Sama funkcjonalność nie wydaje się szczególnie trudna, ale korzystając z możliwości języka TypeScript chcesz zabezpieczyć użytkownika przed literówkami i pomyłkami na poziomie argumentu funkcji. Potrzebujesz typu, który zawiera w sobie wszystkie możliwe ścieżki dostępu do pól w obiekcie - dzięki temu nie będzie można pobrać pola, które nie istnieje.

Niezbędny typ możesz napisać samemu, ale mogą ci w tym pomóc popularne biblioteki typów takie jak type-fest, który zawiera wszystko, czego potrzebujesz (np. typ Paths<T>):

import {type Paths} from "type-fest";

type Person = {
  details: {
    firstName: string;
    lastName: string;
  };
  address: {
    city: string;
    street: string;
  };
};

function getProp<T>(path: Paths<T>) { ... }

getProp<Person>("details.firstName"); // ✅ OK - pole pod taką ścieżką istnieje
getProp<Person>("details.city"); // ❌ FAIL - nie ma pola pod taką ścieżką

Generyczny typ Paths<T> zwraca unię wszystkich ścieżek, jakie dotyczą pól w obiekcie. Unia aktualizowana jest na bieżąco, na podstawie przekazywanego parametru. Taki typ sprawdzi się idealnie w każdym scenariuszu, gdzie budowana funkcja, narzędzie lub biblioteka powinna zapewnić kontrolowany dostęp do struktury obiektu.

Tego typu scenariuszy jest naprawdę mnóstwo, dlatego zamiast pisać każdy typ na nowo, zapoznaj się z najpopularniejszymi bibliotekami typów z ekosystemu TypeScript. Najpopularniejsze zestawy prezentujemy poniżej - są one ułożone wg czasu aktualizacji, od najnowszych do nieco starszych:

  • type-fest: https://github.com/sindresorhus/type-fest

  • utility-types: https://github.com/piotrwitek/utility-types

  • ts-toolbelt: https://github.com/millsp/ts-toolbelt

Generatory i procesory kodu

Statyczna analiza kodu TypeScript to coś, co dało początek całej serii generatorów konwertujących istniejące typy na dokumentację, klienty API czy kontrakty danych oparte o JSON Schema. Ten ostatni element znakomicie sprawdza się w budowaniu integracji z modelami językowymi, gdzie od niedawna pojawiło się wsparcie dla tzw. structured outputs. Czym jest ta funkcjonalność?

Kiedy chcemy zbudować stabilną integrację z modelem językowym to naszym celem jest to, żeby odpowiedzi były przewidywalne i zawsze posiadały ten sam format. W związku z losową naturą LLMów nie zawsze można to zagwarantować, ale przykładowo firma OpenAI wspiera podawanie oczekiwanego formatu odpowiedzi właśnie przez JSON Schema.

Przetestujemy to w oparciu o dwie wiadomości, na które odpowiedzieć ma model GPT-4o:

  • Zadanie: Your task is to extract personal details based on provided text.

  • Dane wejściowe: Jan Nowak, ul. Główna 12, 00-001 Warszawa

Wprawny programista zauważy, że w promptach nie ma żadnego oczekiwanego formatu danych, ale wykonujemy takie polecenie aby sprawdzić, czy na nasz typ będzie można zmapować chociaż fragment odpowiedzi:

// src/person.ts
export type Person = {
  details: {
    firstName: string;
    lastName: string;
  };
  address: {
    city: string;
    street: string;
  };
};

Oto nasz kod:

import OpenAI from 'openai';
import 'dotenv/config';

const openai = new OpenAI();

async function main() {
  const completion = await openai.chat.completions.create({
    messages: [
      {
        role: 'system',
        content: 'Your task is to extract personal details based on provided text.',
      },
      {
        role: 'user',
        content: 'Jan Nowak, ul. Główna 12, 00-001 Warszawa',
      },
    ],
    model: 'gpt-4o',
  });

  console.log(completion.choices[0]);
}

main();

Niestety, odpowiedź z modelu w tym momencie nie jest możliwa do automatycznej interpretacji:

'Name: Jan Nowak  \nAddress: ul. Główna 12, 00-001 Warszawa'

Wykorzystajmy teraz Structured Outputs podając definicję naszego obiektu zgodną z JSON Schema. Nie chcemy tego jednak robić ręcznie - wykorzystamy bibliotekę typescript-json-schema, która na podstawie istniejącego typu TypeScript wygeneruje nam schemat JSONa.

Najpierw instalacja:

npm install typescript-json-schema -DE

Następnie dodajemy skrypt do package.json:

{
  "script": {
     "schemas": "typescript-json-schema src/person.ts Person --out src/person.json"
  }
}

Po wykonaniu tej komendy uzyskamy definicję naszego obiektu zgodną z JSON Schema:

{
    "$schema": "http://json-schema.org/draft-07/schema#",
    "properties": {
        "address": {
            "properties": {
                "city": {
                    "type": "string"
                },
                "street": {
                    "type": "string"
                }
            },
            "type": "object"
        },
        "details": {
            "properties": {
                "firstName": {
                    "type": "string"
                },
                "lastName": {
                    "type": "string"
                }
            },
            "type": "object"
        }
    },
    "type": "object"
}

Mając definicję obiektu, możemy ją wczytać z pliku i przekazać do pola response_format:

import OpenAI from 'openai';
import 'dotenv/config';
const openai = new OpenAI();

// Wczytujemy definicję obiektu Person (lub przez JSON import)
import fs from 'fs';
const schema = fs.readFileSync('src/person.json', 'utf8');

async function main() {
  const completion = await openai.chat.completions.create({
    messages: [...], // Pozostawiamy bez zmian
    model: 'gpt-4o',
    // Ustawiamy oczekiwany format odpowiedzi
    response_format: {
      type: 'json_schema',
      json_schema: { name: 'Person', schema: JSON.parse(schema) },
    },
  });

  console.log(completion.choices[0]);
}

main();

Wystarczy tylko uruchomić ten kod raz jeszcze, a w odpowiedzi z modelu otrzymamy poprawny JSON, który z łatwością można zintegrować z resztą aplikacji - win!

{
  "details": { "firstName": "Jan", "lastName": "Nowak" },
  "address": { "street": "ul. Główna 12", "city": "Warszawa" }
}

To tylko jeden z przykładów tego, jak statyczne typowanie i generatory z ekosystemu TypeScript mogą usprawniać rozwój nowoczesnych aplikacji (w tym - zintegrowanych z AI).

Generowanie klientów OpenAPI

Kolejny scenariusz wykorzystania toolingu TypeScriptu dotyczy budowania bezpiecznych integracji z API. Jeśli korzystamy z zasobów, które opisywane są formatem OpenAPI, to w łatwy sposób możemy się nimi integrować poprzez narzędzie openapi-typescript.

Co robi to narzędzie? Po wskazaniu adresu zasobu (np. REST API), które wystawia poprawną specyfikację, jest w stanie ją zinterpretować a następnie przygotować lokalną bibliotekę, dzięki której będziemy realizować bezpieczna wymianę danych. Taka lokalna biblioteka zawiera sam moduł do pobierania danych (np. poprzez Fetch lub axiosa), ale również typowane endpointy (zarówno na poziomie parametrów i modeli zwracanych).

Jest to istotne ułatwienie i zwiększenie poziomu bezpieczeństwa w sytuacji, kiedy zależy nam na dokładnym odwzorowaniu endpointów danego API na poziomie np. kodu frontendowego. Więcej informacji na temat budowania bezpiecznych integracji w oparciu o OpenAPI znajdziesz w szkoleniu Opanuj Frontend, a także w drugim module tego kursu, czyli O:TS React Pro.

Modyfikacje kodu w dużej skali

W dużych projektach od czasu do czasu zdarza się przeprowadzać duże manipulacje kodu - czy to ze względu na aktualizację frameworka, zmianę modeli danych lub lepszy sposób realizowania wybranej operacji. Ręczne modyfikacje mogą być czasochłonne, trudne do analizowania i podatne na błędy, dlatego najlepsze firmy programistyczne realizuje takie zadania w oparciu o tzw. “codemods”.

Codemods to narzędzia, które przy pomocy kodu manipulują składowymi danego projektu, czy to na poziomie zmiennych, wywołań określonych funkcji lub komponentów. Możesz o nich myśleć jak o bardziej wyszukanych wyrażeniach regularnych, które nie pracują na poziomie stringa, a pełnoprawnych struktur danych.

W środowisku TypeScript warto zainteresować się narzędziem ts-morph, które może stać się takim operatorem migracji w dużej skali. Dzięki niemu w formie algorytmu opiszesz operację, którą chcesz zrealizować na systemie plików, możesz ją krok po kroku debugować a także odwracać. Może to być świetny sposób na przeprowadzenie operacji, która do tej pory wydawała się zbyt czasochłonna.

Środowiska i bundlery

Kilka lat temu społeczność frontendowa zmagała się ze słynnym “frontend fatigue” - było to globalne znużenie tym, jak skomplikowany i wielowarstwowy stał się tooling frontendowy. Nie ma co ukrywać, że TypeScript sprawy długo nie ułatwiał. Brakowało typów, konfiguracja była trudna do zrozumienia a główne środowisko uruchomieniowe, czyli Node, nie potrafiło rozmawiać w tym języku.

Dzisiaj sytuacja wygląda znacznie lepiej, a to za sprawą środowisk i bundlerów TypeScript-first czyli takich, których już na etapie projektowania postanowiono wspierać tę technologię.

Deno (https://deno.com/)

Runtime Deno (czyt. Dino) został stworzony przez Ryana Dahla - tego samego programistę, który stworzył Node.js w 2009 roku. Dahl opracował Deno w odpowiedzi na problemy, które po czasie zaczęły występować w jego najpopularniejszym projekcie.

Deno zostało zaprojektowane jako nowoczesna alternatywa dla Node.js, eliminując jego ograniczenia i problemy projektowe. Deno obsługuje TypeScript natywnie, co oznacza, że nie trzeba kompilować kodu przed jego uruchomieniem.

Zgodnie z założeniami autora, Deno jest domyślnie bezpieczne: dostęp do systemu plików, sieci, środowiska itp. jest wyłączony, chyba że użytkownik explicite zezwoli na to za pomocą flag (np. --allow-read). Mamy również wsparcie dla modułów ES (ECMAScript Modules) zamiast CommonJS, co jest zgodne z najnowszym standardem JavaScript. Obok samego runtime dostajemy też formatter, linter i test runner.

Dodatkowo, Deno jest dostarczane jako pojedynczy binarny plik, co ułatwia jego instalację i dystrybucję. Poza językami JavaScript i TypeScript, Deno wspiera także WebAssembly (WASM) oraz może być rozszerzane za pomocą pluginów pisanych w językach takich jak Rust.

Bun (https://bun.sh/)

Bun to kolejne środowisko uruchomieniowe dla JavaScriptu, TypeScriptu i WebAssembly, które zostało zaprojektowane jako nowoczesna alternatywa dla Node.js i Deno.

Bun wyróżnia się na tle Node.js i Deno dzięki swojej wydajności, prostocie i wbudowanym narzędziom uzupełniającym standardowe funkcje uruchamiania kodu. Jest idealnym rozwiązaniem dla programistów, którzy szukają narzędzia kompatybilnego z ekosystemem Node.js, ale szybszego i bardziej wszechstronnego. Mimo że Bun jest stosunkowo młodym projektem, już teraz zdobywa uznanie w społeczności programistów dzięki innowacyjnemu podejściu i skupieniu na szybkości.

Bundlery ze wsparciem dla TypeScriptu

Obecnie, coraz więcej bundlerów - narzędzi zajmujących się “pakowaniem” składowych aplikacji webowej - traktuje TypeScript na równi JavaScript. Pokazuje to, jak mocną pozycję wyrobił sobie w środowisku opisywany przez nas język.

Narzędzia z tej kategorii, dla których TypeScript to obywatel pierwszej kategorii, to m.in.:

  • rspack - https://rspack.dev/

  • Vite - https://vite.dev/

  • esbuild - https://esbuild.github.io/

Node kontra TypeScript - znacznie lepiej, niż kiedyś!

Backendowe środowisko uruchomieniowe JavaScriptu - Node - przez długi czas nie radziło sobie z procesowaniem kodu TypeScript. W nowych wersjach Node’a (22+) sytuacja wygląda inaczej - w momencie uruchamiania kodu, do dyspozycji mamy dwie eksperymentalne flagi:

  • --experimental-strip-types

  • --experimental-transform-types

Po włączeniu pierwszej z nich, Node stara się uruchomić “prosty” kod TypeScript poprzez usunięcie deklaracji typów. Nie wykonuje przy tym żadnych dodatkowych transformacji, więc po pierwsze nie każdy kod z tą flagą zadziała, a po drugie nie uruchomimy niczego, co nie istnieje po stronie JavaScriptu (np. enumy lub przestrzenie nazw).

Druga z flag, wprowadzona w Node 22.7.0, wprowadza obsługę transformacji TS do JS - dzięki temu uruchomimy zdecydowanie bardziej złożony kod, którego procesowanie wygląda podobnie do budowania frontendu. Obie z flag możemy dodać do skryptu uruchamiającego plik w Node:

node --experimental-strip-types --experimental-transform-types script.ts

Autorzy Node’a wspominają, że obie flagi to wciąż funkcje eksperymentalne, które w przyszłych wersjach tego środowiskach zostaną usunięte na rzecz natywnego wsparcia TypeScriptu.

Jeśli nie chcesz polegać na eksperymentalnych flagach, lepszym rozwiązaniem mogą być takie narzędzia jak ts-node lub tsx, które umożliwiają uruchamianie kodu TypeScript w środowisku Node bez dodatkowych kroków transpilacji.

Popularne biblioteki w ekosystemie ts

Na koniec prezentujemy ci naszą piątkę rekomendowanych narzędzi, które w pełni korzystają z potencjału TypeScriptu i które warto rozważyć w kolejny projekcie webowym.

1. Zod (https://zod.dev/) - Walidacja danych

Zod to biblioteka do walidacji danych, która świetnie integruje się z TypeScriptem, pozwalając definiować mocno typowane schematy danych. Dzięki temu można ocenić, czy deklarowana struktura danych na pewno wygląda tak, jak powinna. Zod sprawdza się zarówno w walidacji formularzy frontendowych, jak i w middleware na backendzie, np. w aplikacjach korzystających z tRPC czy Next.js. Na Zoda postawili również twórcy frameworka Astro (Content API) oraz biblioteki OpenAI (Structured Outputs).

2. DrizzleORM (https://orm.drizzle.team/) - Dostęp do bazy

DrizzleORM to lekka, typowana biblioteka ORM, pozwalająca na budowanie zapytań SQL z pełnym wsparciem typów i podpowiadania składni. Dzięki integracji z narzędziami, takimi jak Vite czy TurboRepo, DrizzleORM doskonale wpisuje się w nowoczesne standardy full-stackowe, oferując jednocześnie intuicyjny interfejs dla operacji na bazie danych. Idealna do aplikacji, gdzie potrzebna jest kontrola nad SQL przy zachowaniu bezpieczeństwa typów.

3. tRPC (https://trpc.io/) - Integracja warstw aplikacji

tRPC to framework pozwalający na budowanie w pełni typowanego API, które nie wymaga ręcznego pisania kontraktów między frontendem a backendem. W połączeniu z TypeScriptem umożliwia wystawianie endpointów w formie klasycznych funkcji, co eliminuje problemy z niespójnością danych. tRPC jest świetnym wyborem dla aplikacji rozwijanych w ramach monorepo, w których kluczowa jest spójność typów i szybkość integracji wielu różnych warstw.

4. TanStack (https://tanstack.com/) - Przydatne narzędzia

TanStack to zbiór popularnych narzędzi OpenSource. Gwiazdą zestawu jest TanStack Query, czyli narzędzie do zarządzania stanem zapytań i synchronizacją danych w aplikacjach frontendowych. Obsługuje automatyczne buforowanie, ponawianie zapytań oraz odświeżanie danych w tle. Dzięki doskonałej obsłudze typów w TypeScript, integruje się z aplikacjami, które wymagają precyzyjnego zarządzania danymi z API, np. z tRPC czy GraphQL, zwiększając produktywność programistów.

5. Effect (https://effect.website/) - Efekty uboczne

Effect to warta uwagi biblioteka do zarządzania tzw. efektami ubocznymi. Opiera się na paradygmacie programowania funkcyjnego (FP) i stara się eliminować problematyczne aspekty zarządzania asynchronicznością, błędami czy współbieżnością czyli tym, co odwraca naszą uwagę od dostarczania wartości dla użytkownika końcowego.

Podsumowanie

W tej lekcji zobaczyłeś, jak szeroki i dojrzały jest obecnie ekosystem TypeScriptu.

Chociaż nadal można spotkać biblioteki bez deklaracji typów lub takie, gdzie wersje typów nie nadążają z bieżącym developmentem, w przypadku projektów mainstreamowych jest to praktycznie niemożliwe.

Dzięki starterom, bibliotekom, zestawom typów a także generatorom i procesorom kodu wiele projektów produkcyjnych już od startu decyduje się na język TypeScript, bo jego skalowalność i bezpieczeństwo to coś, czego nie może zaoferować nawet najbardziej nowoczesny JavaScript. Podpisujemy się pod tą decyzją i trzymamy kciuki za to, aby twoje przyszłe projekty były tworzone właśnie w oparciu o tę technologię.
