Wprowadzenie

W tej lekcji dowiesz się więcej o samym procesie transpilacji, czyli konwersji kodu TypeScript na JavaScript, czasami nazywanym kompilacją.

Transpilacja to proces, w którym kod źródłowy napisany w jednym języku programowania jest przekształcany na kod źródłowy w innym języku, na podobnym poziomie abstrakcji. Jako, że narzędzia ekosystemu TypeScript, takie jak tsc, mają w sobie coś z kompilacji (np. analiza kodu, sprawdzanie typów), terminów transpilacja i kompilacja będziemy używać zamiennie.

Zakładamy, że na tym etapie masz zainstalowanego TypeScriptu na jeden z opisywanych wcześniej sposobów. Jeśli korzystasz z instalacji globalnej, kompilator będzie dostępny bezpośrednio w konsoli:

tsc

W przypadku rekomendowanej przez nas instalacji lokalnej powinieneś utworzyć nowy projekt (npm init), zainstalować TypeScript jako devDependency i dodać pomocniczy skrypt w package.json:

"scripts": {
  "tsc": "tsc"
}

Zobacz jak rozpocząć pracę z kompilatorem:

Ważne aspekty konfiguracji

Poniżej przedstawiamy najważniejsze elementy konfiguracji i to, jak poszczególne ustawienia przekładają się na rezultat w postaci kodu JavaScript. Pola, które wcześniej omówiliśmy na filmie, możesz pominąć na rzecz tych mniej popularnych, które również znajdują się w poniższej sekcji.

Pliki wejściowe

Dwa najważniejsze pola, które decydują o kompilowaniu właściwych plików, to include oraz exclude - umieszczamy je na głównym poziomie konfiguracji:

{
  "include": ["*.ts", "src/**/*.ts"],
  "exclude": ["node_modules"]
}

W pierwszym z nich możemy podać listę ścieżek z tzw. wildcardami, do miejsc, gdzie znajdują się pliki TypeScript, a w drugim - jak łatwo się domyślić - możemy określić foldery wyłączone z całego procesu.

Pliki wyjściowe

Po wybraniu odpowiednich plików czas na decyzję o tym, gdzie umieścić efekt kompilacji projektu.

Docelowy JavaScript może być tworzony w kilku miejscach:

  • obok źródłowych plików .ts (ustawienie domyślne)

  • we wskazanym folderze, gdzie każdy plik .ts ma odpowiednika w postaci pliku .js:

{
  "compilerOptions": {
    ...
    "outDir": "./dist",
    ...
  }
}

  • w jednym pliku, gdzie znajdzie się połączona zawartość plików źródłowych (po konwersji na JS):

{
  "compilerOptions": {
    ...
    "outFile": "./dist/bundle.js",
    ...
  }
}

(Zwróć uwagę, że w przeciwieństwie do poprzedniej sekcji, oba ustawienia są zagnieżdżone w obiekcie compilerOptions)

Która opcja jest preferowana? Na potrzeby nauki języka i własnych eksperymentów, prawdopodobnie .js umieszczany obok .ts, jako odrębny plik, pozwoli ci łatwiej analizować proces kompilacji (a więc outDir lub zachowanie domyślne, bez tych pól).

W kontekście budowania całych aplikacji, ta kwestia wygląda nieco inaczej. W rozwiązaniach produkcyjnych, kiedy obok TypeScriptu korzystamy z bundlerów takich jak Vite, Parcel czy Webpack, ścieżki docelowej nie konfigurujemy na poziomie tsconfig.json. To nie sam kompilator tsc, a właśnie wspomniane bundlery umieszczają docelowe pliki w określonym miejscu i to ich konfiguracja jest w tym względzie priorytetowa.

Ustawienia kompilacji

Powyższe sekcje służą do zarządzania fizycznymi plikami i folderami biorącymi udział w kompilacji, ale nie wpływają na same transformacje kodu. Ten proces skonfigurujesz z wykorzystaniem sekcji “compilerOptions”.

Przejdźmy teraz przez najważniejsze pola wspomnianej sekcji żeby zrozumieć różne warianty ustawień procesu kompilacji i ich wpływ na wyprodukowany kod JavaScript. Na początku skupimy się na tych polach, które są niezależne od stylu czy nawyków programisty - w kolejnych lekcjach rozbudujemy konfigurację o nowe ustawienia, które możesz stosować zgodnie z twoimi preferencjami.

👉 target

Pole target decyduje o kompatybilności docelowego JavaScriptu z określoną wersją standardu ECMAScript (a tym samym - o zgodności z przeglądarkami użytkowników).

Zobaczmy, jak wygląda efekt kompilacji poniższej klasy:

class Person {
  constructor(public name: string, public age: number) {}
}

Bez ustawionego pola “target”, zastosowanie ma wartość domyślna, czyli ES3 - w obecnych czasach kod może być niepotrzebnie “uwsteczniony” względem dzisiejszych możliwości JavaScriptu, co najlepiej widać po konwersji klasy na funkcję oraz zastosowaniu IIFE:

var Person = /** @class */ (function () {
    function Person(name, age) {
        this.name = name;
        this.age = age;
        this.id = Math.random();
    }
    return Person;
}());

Z wartością “ES2015/ES6” uzyskujemy m.in. natywne wsparcie dla klas - na dzisiaj jest to rekomendowane ustawienie, wspierane obecnie przez wszystkie nowoczesne przeglądarki. Kod jest bardziej czytelny i zdecydowanie prostszy w interpretacji, jednak w szczególnych warunkach (starsze środowiska) może nie być obsługiwany poprawnie:

class Person {
    constructor(name, age) {
        this.name = name;
        this.age = age;
    }
}

Z wartością “ESNext” możemy obserwować brzegowe możliwości nowoczesnego JavaScriptu, w tym chociażby dedykowane pola w klasach wprowadzone w 2022r. Niestety, w tym wariancie kod wynikowy jest najmniej stabilny:

class Person {
    name;
    age;
    constructor(name, age) {
        this.name = name;
        this.age = age;
    }
}

👉 ESNext to ustawienie specjalne, które na przestrzeni kolejnych wersji TypeScriptu może dawać inne rezultaty kompilacji (na skutek ciągłej ewolucji ECMAScript). Jeśli nie korzystasz z dodatkowych narzędzi zapewniających kompatybilność kodu takich jak Vite, Rollup czy Babel, ustawienie “ESNext” nie jest rekomendowane i w wybranych przeglądarkach może generować błędy.

Co ciekawe, w niektórych sytuacjach “target” nie może być ustawiony zbyt nisko. Przykładowo, kiedy do naszej klasy wprowadzimy pola prywatne (w rozumieniu JavaScriptu), to kompilator nie pozwoli nam wygenerować kodu w standardzie bez tego elementu:

class Person {
  #id;
  constructor(public name: string, public age: number) {
    this.#id = Math.random();
  }

  identify() {
    console.log(this.#id);
  }
}

Dla ustawienia “ES5” otrzymamy błąd:

TS18028: Private identifiers are only available when targeting ECMAScript 2015 and higher

Skąd ten błąd? Kompilator, interpretując pole “mocno prywatne”, wg standardu ECMAScript (z hashem), musi wygenerować klasę o podobnej konfiguracji jak plik źródłowy. Aby w JavaScripcie otrzymać prawdziwie prywatne pole, możemy:

  • stosować wyższy standard języka, gdzie ta opcja po prostu istnieje

  • lub wykorzystywać struktury pomocnicze, przez które otrzymamy podobny efekt.

Kompilator TypeScript wykorzystuje do tego celu obiekt WeakMap, wprowadzony w ES2015, a na niższy standard po prostu nie zezwala (poniżej efekt kompilacji):

var __classPrivateFieldSet = (this && this.__classPrivateFieldSet) || function (receiver, state, value, kind, f) {
    if (kind === "m") throw new TypeError("Private method is not writable");
    if (kind === "a" && !f) throw new TypeError("Private accessor was defined without a setter");
    if (typeof state === "function" ? receiver !== state || !f : !state.has(receiver)) throw new TypeError("Cannot write private member to an object whose class did not declare it");
    return (kind === "a" ? f.call(receiver, value) : f ? f.value = value : state.set(receiver, value)), value;
};
var __classPrivateFieldGet = (this && this.__classPrivateFieldGet) || function (receiver, state, kind, f) {
    if (kind === "a" && !f) throw new TypeError("Private accessor was defined without a getter");
    if (typeof state === "function" ? receiver !== state || !f : !state.has(receiver)) throw new TypeError("Cannot read private member from an object whose class did not declare it");
    return kind === "m" ? f : kind === "a" ? f.call(receiver) : f ? f.value : state.get(receiver);
};
var _Person_id;
class Person {
    constructor(name, age) {
        this.name = name;
        this.age = age;
        _Person_id.set(this, void 0);
        __classPrivateFieldSet(this, _Person_id, Math.random(), "f");
    }
    identify() {
        console.log(__classPrivateFieldGet(this, _Person_id, "f"));
    }
}
_Person_id = new WeakMap();

👉 lib

Pole “lib” wpływa na zestaw dostępnych deklaracji typów, z których możemy korzystać rozwijając kod TypeScript w naszym edytorze. Będzie ono wpływać na takie aspekty pracy z TypeScriptem jak podpowiadanie składni, informowanie o błędach lub sugerowanie bardziej precyzyjnego posługiwania się typami.

Domyślnie, wartość tego pola jest kopią ustawienia z pola “target” i nie musimy jej ustawiać ręcznie - w końcu chcemy widzieć podpowiedzi takich metod, funkcji czy obiektów, jakie finalnie wspiera docelowy JavaScript:

{
  "compilerOptions": {
    "target": "ES2015",
    "lib": ["ES2015"]
  }
}

Kiedy w takiej konfiguracji spróbujemy skorzystać z funkcji toSorted, wprowadzonej w ECMAScript2023, w edytorze (jeszcze przed wykonaniem kompilacji) zobaczymy błąd - ta metoda nie jest dostępna:

const names = ['John', 'Jane', 'Doe'];

names.toSorted
          // ^ Property 'toSorted' does not exist on type 'string[]'.ts(2339)

Może się jednak zdarzyć, że takie nowoczesne funkcje chcemy dostarczyć w inny sposób - być może mamy polyfille, o których TypeScript nie wie, albo zestaw dodatkowych bibliotek, które w runtime rozszerzają obiekty wbudowane. Wtedy - na własną odpowiedzialność - możemy oba te pola zmodyfikować niezależnie:

{
  "compilerOptions": {
    "target": "ES2015",
    "lib": ["ES2023"]
  }
}

Efektem kompilacji będzie nadal kod w standardzie ES2015, jednak TypeScript zacznie podpowiadać definicję nowej metody korzystając z typów standardu ES2023:

const names = ['John', 'Jane', 'Doe'];

names.toSorted()
       // ^ Returns a copy of an array with its elements sorted.

Co ważne - kod wynikowy nie będzie w żaden sposób zmodyfikowany i nie pojawią się w nim dodatkowe polyfille lub funkcje pomocnicze. TypeScript założy, że w tym momencie to my przejmujemy odpowiedzialność za dostarczenie tej funkcji do przeglądarki (lub innego środowiska):

// index.js
const names = ['John', 'Jane', 'Doe'];
names.toSorted();

W projektach, które używają dodatkowych bundlerów (np. webpack, Babel), możesz być bardziej odważny w ustawieniach pól “target” oraz "lib" - wspomniane narzędzia często zajmują się kompatybilnością z różnymi środowiskami i zapewnią ci dodatkowe polyfille.

Spostrzegawczy zauważą, że pole lib akceptuje tablicę wartości - obok standardu ECMAScript możemy tam dodać (lub świadomie pominąć - np. w projektach Node.js) wbudowane deklaracje typów, które opisują poszczególne obszary środowiska przeglądarki.

Przykładowo, aby zacząć pracę z obiektem document lub window, sekcja lib powinna zawierać “DOM”, zawierający typowania dla obiektów z Document Object Modelu:

{
  "compilerOptions": {
    "target": "ES2015",
    "lib": ["ES2023", "DOM"],
  }
}

Jeśli z kolei pracujemy z kodem backendowym, to ten przykład można zinterpretować odwrotnie - o ile obiekty przeglądarkowe nie są nam potrzebne, to pozbycie się wpisu “DOM” może oszczędzić kilka cykli pracy kompilatora poprzez ignorowanie środowiska Document Object Modelu (wtedy pozostawiamy tylko wersję standardu ECMAScript).

Dostępne rozszerzenia znajdziesz na repozytorium GitHub.

👉 module

Pole "module" określa system modułów używany w wygenerowanym kodzie JavaScript. To ustawienie jest kluczowe dla określenia, w jaki sposób TypeScript przekształci importy i eksporty w twoim kodzie.

Zobaczmy, jak różne wartości tego pola wpływają na kompilację prostego modułu utils.ts:

// utils.ts
export const formatName = (name: string) => {
  return name.trim().toLowerCase();
};

// index.ts
import { formatName } from './utils';

Z ustawieniem "CommonJS" (wartość domyślna dla Node.js) otrzymamy:

// utils.ts
"use strict";
Object.defineProperty(exports, "__esModule", { value: true });
exports.formatName = void 0;
const formatName = (name) => {
    return name.trim().toLowerCase();
};
exports.formatName = formatName;

// index.js
"use strict";
Object.defineProperty(exports, "__esModule", { value: true });
const utils_1 = require("./utils");
console.log((0, utils_1.formatName)("John Doe"));

Z kolei używając "ESNext" lub "ES2015", kod zostanie przekształcony w natywne moduły ES:

// utils.js
export const formatName = (name) => {
    return name.trim().toLowerCase();
};

// index.js
import { formatName } from "./utils";
console.log(formatName("John Doe"));

Wybór odpowiedniej wartości zależy od środowiska, w którym kod będzie uruchamiany:

  • "CommonJS" - standardowy wybór dla aplikacji Node.js

  • "ES2015"/"ESNext" - dla nowoczesnych przeglądarek i narzędzi bundlujących

  • "UMD" - dla bibliotek, które mają działać zarówno w przeglądarce jak i Node.js

  • "AMD" - dla starszych aplikacji korzystających z RequireJS (na dziś rzadko spotykane)

W kontekście posługiwania się modułami warto jeszcze wspomnieć o trzech ustawieniach, które starają się zapewniać maksymalną kompatybilność pomiędzy różnymi formatami importów, eksportów i wykorzystywanych zależności:

  • allowSyntheticDefaultImports - Umożliwia importowanie modułów w stylu ES6, nawet jeśli moduły te nie mają eksportów domyślnych. Może być przydatne w sytuacjach, kiedy zależności projektu nie są w pełni kompatybilne z ES6, ale chcemy korzystać z bardziej przejrzystej składni import x from 'module'.

// allowSyntheticDefaultImports: false
import * as React from "react";

// allowSyntheticDefaultImports: true
import React from "react";

  • esModuleInterop - To ustawienie rozszerza działanie pola allowSyntheticDefaultImports dodając tę samą funkcjonalność, a także kod, który zwiększa kompatybilność między modułami CJS i ES6.

  • verbatimModuleSyntax - To pole wprowadzone w TypeScript 5.0, które wymusza na nas bardziej precyzyjne określanie zawartości modułów, którymi posługujemy się w kodzie. Zanim go włączymy, możemy przetestować kompilację dwóch plików, gdzie jeden z nich zawiera wyłącznie typ, bez dodatkowej logiki (Person), a drugi stosuje importowany typ na nowym obiekcie:

// utils.ts
export type Person = {
  name: string;
  age: number;
};

// index.ts
import {Person} from "./utils";

const person: Person = {
  name: "John Doe",
  age: 20,
};

console.log(person);

W trakcie kompilacji, TypeScript zastosuje tutaj tzw. mechanizm module elision - mechanizm ten optymalizuje zbędne importy/eksporty usuwając je z wynikowego JavaScriptu. Poniżej widać, że wynikowy index.js nie zawiera już referencji do modułu utils (zawierał wyłącznie typ Person, więc został oceniony jako zbędny):

// index.js
const person = {
    name: "John Doe",
    age: 20,
};
console.log(person);
export {};

Pole “verbatimModuleSyntax” zostało wprowadzone aby ten mechanizm optymalizacji importów stał się jawny. Po jego włączeniu TypeScript informuje nas, że Person musimy zaimportować jako typ…

import type {Person} from "./utils";

…a typy - z definicji - będą usunięte z docelowego JavaScriptu. Wspomniane ustawienie sprawia, że moduły eksportujące typy będą optymalizowane w bardziej przejrzysty sposób.

👉 strict

Włączenie flagi strict skutkuje przejściem na bardziej rygorystyczne reguły sprawdzania typów i bardziej świadome posługiwanie się dostępnymi funkcjami, obiektami czy strukturami w projekcie. Jest to rodzina ustawień, które w łatwy sposób możemy aktywować jednym polem:

{
  "compilerOptions": {
    "strict": true
  }
}

Efektem włączenia trybu ścisłego jest rygorystyczna analiza kodu, która wychwytuje potencjalne błędy już na etapie kompilacji. Przykładowo, bez trybu strict, zmienna shortName będzie uznana za string, ale już w przypadku typu ścisłego, będzie to “string | undefined”:

const names = ['John', 'Jane', 'Doe'];
const shortName = names.find((name) => name.length === 2);

W trybie ścisłym kompilator wymusi na nas obsługę przypadku, w którym funkcja find nie znajduje żadnego elementu - inaczej moglibyśmy ten błąd wykryć dopiero na etapie działania naszej aplikacji.

Włączenie opcji strict jest jednoznaczne z włączeniem zestawu niezależnych reguł, takich jak chociażby:

  • noImplicitAny - zakaz posługiwania się niejawnym typem “any” (więcej o nim w kolejnych lekcjach)

  • strictNullChecks - obowiązek obsługi wartości null i undefined (np. przy wyszukiwaniu elementu)

  • strictBindCallApply - weryfikacja parametrów metod, które wywołujemy poprzez bind, call i apply

  • useUnknownInCatchVariables - obowiązek jawnej obsługi typów przy obsłudze błędów

Jeśli uznasz, że w trybie strict rozwijanie kodu jest problematyczne (np. przy złożonych integracjach czy migracji z JavaScript na TypeScript), to każdą z reguł możesz obsłużyć w niezależny sposób:

{
  "compilerOptions": {
    "strict": true,
    "noImplicitAny": false
  }
}

Podobnie jak w przypadu “target”: ESNext, zakres tego, co zawiera w sobie pole strict, w przyszłych wersjach języka może ewoluować. Ważne jest więc świadome aktualizowanie TypeScriptu i weryfikacja zmian wprowadzanych w kolejnych wersjach języka.

👉 noEmit

Może się zdarzyć, że działanie kompilatora TypeScript będziesz chciał ograniczyć tylko do wsparcia programisty w edytorze kodu, a część odpowiedzialności wydelegujesz do innych narzędzi:

{
  "compilerOptions": {
    "noEmit": true
  }
}

Pole noEmit można wykorzystać wtedy, kiedy za kompilowanie produkcyjnego TypeScriptu odpowiadają dodatkowe, zainstalowane niezależnie narzędzia, takie jak Vite, swc lub rspack. Korzystając z szablonów projektowych opartych właśnie o te narzędzia (przykład z Vite), noEmit będzie najczęściej włączone, a za produkcję JavaScriptu będzie odpowiedzialny dodatkowy bundler.

W projektach prywatnych, przy prostej konfiguracji i bez dodatkowego toolingu, tego pola nie włączamy.

👉 noEmitOnError

Pole noEmitOnError, przyda się w produkcyjnych scenariuszach CI/CD i wyłączy domyślne ustawienie kompilatora TypeScript, który - niezależnie od błędów - stara się nie blokować twojej pracy i zawsze produkować pewną formę kodu JavaScript.

Może to prowadzić do sytuacji, w których na poziomie TypeScriptu znajdują się ewidentne błędy w kodzie, mimo tego powstaje wynikowy JavaScript, a kolejne etapy procesu budowania aplikacji zaczynają przeprowadzać wdrożenie. Aby tego uniknąć, wyłączaj produkcję JavaScriptu z błędnego TS:

{
  "compilerOptions": {
    "noEmitOnError": true
  }
}

Korzystaj z tego pola w świadomy sposób - w procesie debugowania lub integracji zewnętrznych bibliotek może się zdarzyć, że pozornie błędny TypeScript chcesz jednak przetestować. Tymczasowe wyłączenie tego pola może ci w tym pomóc.

👉 skipLibCheck

Pole skipLibCheck wpływa na to, czy TypeScript powinien pomijać sprawdzanie typów w tzw. plikach deklaracji (.d.ts) - są to specjalne pliki, które zawierają tylko definicje typów i interfejsów dla modułów, bez ich implementacji. Są tworzone np. na potrzeby integracji bibliotek JavaScript z systemem typów TypeScript, pozwalając na zapewnienie typowania w projektach TypeScript przy korzystaniu z zewnętrznych zależności. Dzięki temu deweloperzy mogą korzystać z typowania dla bibliotek, które same w sobie mogą nie być napisane w TypeScript, co poprawia bezpieczeństwo typów i wygodę pracy z kodem.

Gdy wspomniane pole jest ustawione na true, kompilator TypeScript nie będzie wykonywał pełnego sprawdzania typów, a skupi się jedynie na kodzie, który bezpośrednio kontrolujemy.

Włączenie tego pola ma sens w dwóch sytuacjach:

  • Optymalizacja procesu kompilacji - dla dużych projektów, korzystających z rozbudowanej listy zależności, wyłączenie type-checkingu na poziomie bibliotek może w znaczący sposób skrócić czas kompilacji kodu

  • Sprzeczne deklaracje typów - może się zdarzyć, że dwie lub więcej bibliotek opisują tę samą zależność w inny sposób - jeśli jesteśmy pewni tego, że konflikt dotyczy wyłącznie deklaracji, a nie samego kodu, możemy sobie pozwolić na wyłączenie powiadomień o błędach:

// node_modules/lib1/index.d.ts
declare module "shared-module" {
  export function configureService(p1: string, p2: string, p3: number): void;
}

// node_modules/lib2/index.d.ts
declare module "shared-module" {
  export function configureService(config: Configuration): void;
}

// Nasz kod
import { configureService } from "shared-module";

// Pomimo rozbieżności typów, sama biblioteka rozpoznaje parametry i nadal działa poprawnie
doSomething({ ... });

Mimo korzyści płynących z użycia "skipLibCheck", należy zachować ostrożność przy jego stosowaniu.

Włączenie tej opcji może maskować rzeczywiste problemy z typami w bibliotekach, prowadząc do trudnych do wykrycia błędów w czasie wykonania. Rekomenduje się, aby używać "skipLibCheck" selektywnie i w połączeniu z innymi rygorystycznymi opcjami sprawdzania typów, takimi jak "strict" czy "noImplicitAny".

Warto również rozważyć okresowe wyłączanie tej opcji podczas rozwoju projektu, aby wychwycić potencjalne problemy z typami. Pamiętaj, że najlepszym długoterminowym rozwiązaniem jest aktualizacja lub naprawa niespójnych definicji typów, zamiast ich pomijania.

👉 allowJs / checkJs

To dwa ważne pola, które umożliwiają stopniową integrację TypeScriptu z istniejącymi projektami JavaScript. Pierwsze z nich (allowJs) umożliwia importowanie kodu JS do istniejących modułów TS, a drugie (checkJs) pozwala kompilatorowi tsc na wykonywanie oceny jakości na poziomie plików JavaScript. Oba ustawienia mogą się okazać kluczowe w projektach legacy, gdzie nie ma mowy o przepisaniu całego projektu, ale stopniowe wprowadzanie TSa może podnosić ich jakość.

Do obu ustawień wrócimy w lekcji poświęconej migracji projektów do TypeScriptu.

Łatwy sposób na domyślną konfigurację

Aby utworzyć wstępnie wypełniony plik konfiguracyjny, a tym samym uzyskać standardowy zestaw reguł do kompilacji TypeScriptu, w projekcie możesz wywołać komendę:

tsc --init

W efekcie tego polecenia utworzony zostanie plik konfiguracyjny tsconfig.json, którego zawartość będzie wpływać na proces kompilacji, wybór plików źródłowych oraz docelową lokalizację JavaScriptu.

Ustawienia z tego pliku są nieco bliższe oczekiwaniom współczesnego programisty:

  • target: es2016

  • module: commonjs

  • strict: true

  • esModuleInterop: true

  • skipLibCheck: true

  • forceConsistentCasingInFileNames: true

👉 Ważne: Kompilator TypeScript, uruchomiony z dodatkowymi parametrami (np. nazwą pliku wejściowego), traktuje te parametry z najwyższym priorytetem, a samą konfigurację ignoruje. 

tsc index.ts

Warto o tym pamiętać - czasami pójście na skróty w postaci przekazania nazwy pliku skutkuje bólem głowy i pozornymi błędami kompilacji, która działa inaczej niż oczekujemy.

Jeśli chcemy skorzystać z ustawień pliku tsconfig.json, musimy wykonać polecenie:

tsc

# lub

tsc --project {path_to_tsconfig}

Przeprowadzenie kompilacji w ten sposób sprawi, że kompilator sam wyszuka plik tsconfig.json i wykona swoją pracę na jego podstawie.

Dodatkowe rozszerzenia konfiguracji

Oprócz podstawowych ustawień kompilacji, TypeScript oferuje zaawansowane możliwości konfiguracji, które pozwalają na lepsze dostosowanie środowiska pracy do potrzeb projektu. W tej sekcji omówimy trzy ważne rozszerzenia: łączenie konfiguracji, aliasy ścieżek oraz konfiguracje zewnętrzne.

Dziedziczenie ustawień

TypeScript umożliwia łączenie wielu plików konfiguracyjnych, co jest szczególnie przydatne w przypadku dużych projektów z różnymi wariantami ustawień. Typowymi scenariuszami dla rozszerzania konfiguracji są:

  • oddzielne projekty frontendowe i backendowe rozwijane w ramach jednego repozytorium

  • oddzielne konfiguracje dla kodu aplikacji i kodu testowego

  • oddzielne konfiguracje dla kodu rozwijanego od zera oraz kodu migrowanego z JavaScript

Aby dziedziczyć ustawienia z innego pliku konfiguracyjnego, wykorzystaj pole extends w pliku tsconfig.json:

// tsconfig.base.json
{
  "compilerOptions": {
    "target": "ES2020",
    "module": "CommonJS",
    "strict": true
  }
}

// tsconfig.test.json
{
  "extends": "./tsconfig.base.json",
  "compilerOptions": {
    "noImplicitAny": false
  },
  "include": ["test/**/*.ts"]
}

W tym przypadku, konfiguracja testowa dziedziczy wszystkie ustawienia z pliku bazowego, ale wyłącza noImplicitAny dla większej elastyczności w pisaniu testów.

Konfiguracje zewnętrzne

Poza dziedziczeniem konfiguracji w obrębie projektu, pole extends można również wykorzystać do pobierania konfiguracji z zewnętrznych bibliotek i frameworków. Jednym z przykładów jest chociażby Astro, które oferuje nam własny tsconfig.json i stosuje go w projektach tworzonych z Astro CLI:

{
  "extends": "astro/tsconfigs/strict",
  "compilerOptions": { ... }
}

Jeśli zajrzymy nieco głębiej, w folderze Astro znajdziemy aż trzy warianty konfiguracji:

node_modules
└── astro
    └── tsconfigs
        └── base.json
        └── strict.json
        └── strictest.json

Każda z tych konfiguracji to inny poziom elastyczności i swobody korzystania z TypeScriptu.

Przykładowo, poza konfiguracją bazową, mamy jeszcze niezależny wariant z trybem strict:

{
  "$schema": "https://json.schemastore.org/tsconfig",
  "extends": "./base.json",
  "compilerOptions": {
    "strict": true
  }
}

…a także konfigurację rozbudowaną o dodatkowe reguły, jeszcze bardziej standaryzujące całe środowisko pracy:

{
  "$schema": "https://json.schemastore.org/tsconfig",
  "extends": "./strict.json",
  "compilerOptions": {
    "noFallthroughCasesInSwitch": true,
    "noImplicitOverride": true,
    "noImplicitReturns": true,
    "noUnusedLocals": true,
    "noUnusedParameters": true,
    "noUncheckedIndexedAccess": true,
    "exactOptionalPropertyTypes": true,
    "allowUnreachableCode": false,
    "allowUnusedLabels": false
  }
}

Jeśli zdarzyło ci się pracować w środowisku, gdzie zespoły zaczynają tracić kontrolę nad standardem rozwijania projektów TypeScript, jednolita konfiguracja może być rozwiązaniem tego problemu.

Możesz ją dostarczać chociażby poprzez współdzieloną bibliotekę, którą doda do swoich aplikacji każdy zespół i na żądanie będzie rozszerzał bazowe reguły ustalone na poziomie organizacji.

Bezpieczne startery

Jeśli chcesz korzystać z bazowych konfiguracji rozwijanych przez społeczność TypeScript, to zapoznaj się z bibliotekami z serii @tsconfig/bases:

https://github.com/tsconfig/bases

Konfigurację dopasowaną do twojego środowiska instalujesz tak jak bibliotekę, np.:

npm install @tsconfig/node20 -DE

Po instalacji możesz z niej korzystać dzięki właściwości “extends” (w razie potrzeby nadpisując poszczególne pola):

{
  "extends": "@tsconfig/node20/tsconfig.json",
  // Pozostałe pola
}

Aliasy ścieżek

Kolejne ustawienie, które poprawia doświadczenia programisty w rozbudowanych projektach, to aliasy ścieżek. Aliasy ścieżek pozwalają na tworzenie skrótów do często używanych ścieżek w projekcie, co ułatwia importowanie modułów i poprawia czytelność kodu.

Aby skorzystać z tej opcji, dodaj dwa nowe pola w sekcji compilerOptions:

  • baseUrl - bazowa lokalizacja procesu rozwiązywania ścieżek

  • paths - aliasy na poszczególne obszary projektu:

{
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "@utils/*": ["src/utils/*"],
      "@components/*": ["src/components/*"]
    }
  }
}

Praktyczny przykład

Zamiast używać długich, relatywnych ścieżek:

import { formatDate } from '../../../utils/date-formatter';
import { Button } from '../../../components/Button';

Możemy teraz pisać:

import { formatDate } from '@utils/date-formatter';
import { Button } from '@components/Button';

W naszych projektach często używamy tego ustawienia do aliasowania typowych zestawów plików, takich jak:

  • obrazki

  • pliki publiczne (PDFy i inne)

  • komponenty

Udany początek

Dzięki tej lekcji zyskałeś umiejętność konfigurowania i personalizacji pliku tsconfig.json pod kątem różnych wymagań projektowych.

Potrafisz już zarządzać ustawieniami, które poprawiają kompatybilność, wydajność i bezpieczeństwo kodu TypeScript. Wprowadzenie tzw. trybu ścisłego (strict mode) i lepsze zrozumienie opcji kompilatora pozwoli na pisanie bardziej niezawodnego, czytelnego i łatwiejszego do utrzymania kodu, a także zapobiegnie błędom, które mógłbyś nieświadomie wprowadzić do projektu.

Kolejne aspekty procesu kompilacji poznasz w dalszej części szkolenia - tym razem już od strony praktycznej, poprzez implementację kodu TypeScript. Zaczniemy od zrozumienia systemu typów, czyli tego, co realnie decyduje o potencjale omawianego języka.

Ćwiczenia

  1. 121-compiler-strict - Zostałeś poproszony o naprawę modułu do zarządzania użytkownikami. W projekcie wdrożono ścisłą weryfikację typów i kod przestał się kompilować. Twoim zadaniem jest umożliwienie kompilacji bez wprowadzania funkcjonalnych zmian do systemu.

  2. 122-compiler-misconfigured - pracujesz nad nowoczesnym projektem, który pozwala na wykorzystywanie najnowszych funkcji JavaScript i TypeScript. Niestety zastany plik konfiguracyjny 001-compiler-misconfigured/tsconfig.json nie jest dostosowany do potrzeb projektu i powoduje niepotrzebne błędy kompilacji. Dostosuj plik konfiguracyjny, aby pozbyć się błędów kompilacji.
