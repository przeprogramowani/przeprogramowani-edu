Wprowadzenie

W poprzedniej lekcji zaprezentowaliśmy ci jeden z potencjalnych scenariuszy na wykorzystanie generyków - była to funkcja służąca do łączenia dwóch obiektów:

function mergeObjects<T, U>(obj1: T, obj2: U): T & U {
    return { ...obj1, ...obj2 };
}

Spostrzegawczy zauważą, że bezpieczne wykorzystanie tej funkcji zależy wyłącznie od dobrej woli programisty, bo na teraz można z niej skorzystać na wiele sposobów:

mergeObjects({ name: 'john' }, { surname: 'doe' }); // { name: 'john', surname: 'doe' }
mergeObjects('john', 'doe'); // { '0': 'd', '1': 'o', '2': 'e', '3': 'n' }
mergeObjects(12, false); // {}

Nie musimy chyba dodawać, że w założeniach chcieliśmy wspierać wyłącznie pierwszy przypadek?

Kompilator TypeScript nie blokuje naszych zamiarów, bo parametry generyczne są zdefiniowane na tyle szeroko, że do funkcji można przekazać dowolną wartość. Poszukajmy więc sposobu na to, aby funkcja była możliwa do wywołania wyłącznie z obiektami (a przy okazji poznajmy inne sposoby na bardziej precyzyjne definiowanie generyków).

Ograniczenia parametrów

Ograniczenie typu generycznego (type constraint) wprowadzimy przez słowo kluczowe extends dodawane do symbolu, który określa parametr funkcji generycznej:

// Typ argumentu "str" powinien pasować do zbioru stringów
function toUpper<T extends string>(str: T): string {
  return str.toUpperCase();
}

// ✅ Poprawne wywołanie
toUpper("hello");

type States = "hot" | "cold" | "warm";
const state: States = "hot";

// ✅ Poprawne wywołanie - literał to nadal pochodna stringa
toUpper(state);

// ❌ Błąd - typ number nie ma związku ze stringami
toUpper(123);
// ^ - Argument of type 'number' is not assignable to parameter of type 'string'.ts(2345)

W przypadku typów złożonych, na generykach możemy wymuszać rozszerzanie określonego interfejsu:

interface HasNumericValue {
  value: number;
}

function logValue<T extends HasNumericValue>(param: T) {
  console.log(param.value);
}

// ✅ Poprawne wywołanie - argument implementuje interfejs HasNumericValue
logValue({ value: 5 });

// ❌ Błąd - argument nie implementuje interfejsu HasNumericValue
logValue({ value: "hello" });

👉 Zwróć uwagę: Staraj się pisać swój kod w taki sposób, aby generyki nie stały się nadmiarowe. Dobra praktyka z dokumentacji TypeScriptu mówi o tym, że generyki maja sens wtedy, kiedy na poziomie deklaracji pojawiają się co najmniej dwa razy (np. jako argument i wartość zwracana). W powyższym przykładzie nie ma to zastosowania i okazuje się, że istnieje prostszy sposób na wymuszenie tego samego kształtu funkcji:

interface HasNumericValue {
  value: number;
}

function logParam(param: HasNumericValue) {
  console.log(param.value);
}

logParam({ value: 5});

Relacje między parametrami

Załóżmy, że chcemy zbudować funkcję, która z obiektu o określonym kształcie pobierze wartość pola o podanej nazwie. Wprowadźmy nowy interfejs Person i funkcję, do której można przekazać nazwę klucza:

interface Person {
  name: string;
  age: number;
  gender: "male" | "female";
}

function getProperty(obj: Person, key: string) {
  return obj[key];
      // ❌ - Element implicitly has an 'any' type because expression of type 'string' can't be used to index type 'Person'.
}

Jak widać, TypeScript tego nie akceptuje. Nazwa klucza o typie string jest zdefiniowana zbyt luźno, a konsument takiej funkcji mógłby wywołać ją z czymś, co nie istnieje w obiekcie typu Person.

Rozwiążmy ten problem wprowadzając nowy typ (PersonKeys) przechowujący pola obiektu Person - możemy je wypisać ręcznie, ale zdecydowanie lepsze będzie słowo kluczowe keyof, które zbuduje dla nas dynamicznie aktualizowaną unię pól, które istnieją w interfejsie Person:

interface Person {
  name: string;
  age: number;
  gender: "male" | "female";
}

type PersonKeys = keyof Person; // "name" | "age" | "gender"

function getProperty(obj: Person, key: PersonKeys) {
  // ✅ Sukces - gwarantujemy pobieranie znanych pól
  return obj[key];
}

Dzięki temu już na etapie wywoływania funkcji dowiemy się, czy pobieramy właściwe dane:

const p: Person = {name: "John", age: 30, gender: "male"};

getProperty(p, "age"); // ✅ => 30
getProperty(p, "id"); // ❌ Błąd - pole "id" nie istnieje w interfejsie Person

Powyższy przykład można uogólnić wykorzystując typy generyczne, co pozwoli nam wykorzystać ten sam mechanizm w wielu innych obiektach i interfejsach.

Zamiast interfejsu Person i typu PersonKeys wprowadźmy parametry T i K. Ich relacja będzie taka sama jak wcześniej - typ K będzie zawierał nazwy pól z obiektu T, co wyrazimy poprzez extends keyof T:

function getProperty<T, K extends keyof T>(obj: T, key: K): T[K] {
  return obj[key];
}

const x = { a: 1, b: 2, c: 3, d: 4 };

getProperty(x, 'a'); // ✅ => 1
getProperty(x, 'm'); // ❌ Błąd - pole "m" nie istnieje w zmiennej x

Generyki z konstruktorem

W przypadku bardziej złożonej logiki biznesowej może się okazać, że potrzebujemy generycznej funkcji tworzącej instancje określonej klasy.

Przykładowo, utworzymy w kodzie centralną fabrykę obiektów, która poza utworzeniem instancji klasy zaloguje ten fakt na konsoli. Tutaj pojawia się pytanie - jak określić typ parametru c, aby przy jego pomocy tworzyć obiekty typu T i zablokować możliwość przekazywania typów prostych?

function createInstance<T>(c: ???): T {
  console.log('Creating instance of', c.name);
  return new c();
}

W języku TypeScript mamy do wyboru dwie opcje:

function createInstance<T>(c: new () => T): T {
  console.log('Creating instance of', c.name);
  return new c();
}

// lub

function createInstance<T>(c: { new (): T }): T {
  console.log('Creating instance of', c.name);
  return new c();
}

W obu przypadkach T reprezentuje typ instancji, która zostanie utworzona przez konstruktor przekazany jako argument c. Typ T jest określany automatycznie na podstawie typu zwracanego przez konstruktor.

Obie wersje funkcji createInstance są równoważne pod względem funkcjonalności. Różnica polega na sposobie zapisu typu konstruktora:

  1. new () => T to skrócona notacja dla funkcji konstruktora.

  2. { new(): T } to pełna notacja obiektu z sygnaturą konstruktora.

Wybór między tymi dwoma zapisami jest kwestią preferencji stylistycznych. Obie wersje pozwalają na to samo: przekazanie klasy jako argumentu i utworzenie jej instancji.

class Person {
  constructor() {}
}

interface Human {}

const p1 = createInstance(Person); // ✅ const p1: Person
const p2 = createInstance(Human); // ❌ - Nie można utworzyć instancji interfejsu

Funkcja łącząca obiekty

Zobacz teraz, jak wykorzystać nową wiedzę do ulepszenia funkcji łączącej dwa obiekty:

function mergeObjects<T, U>(obj1: T, obj2: U): T & U {
    return { ...obj1, ...obj2 };
}

Jakiego rodzaju typu potrzebujemy? Musi on spełniać następujące wymagania:

  • Musi akceptować tylko obiekty (w domyśle - zabraniać typów prostych)

  • Nie może akceptować wartości null i undefined

  • Nie może akceptować tablic ani funkcji

  • Powinien działać z interfejsami i aliasami typów

Sprawa nie jest tak oczywista, jak mogłoby się wydawać, ale na kolejnym filmie zobaczysz, jak te reguły wprowadzić:

Zaktualizowana funkcja wraz z nowym typem znajduje się poniżej:

function mergeObjects<T, U>(
  obj1: MergeableObject<T>,
  obj2: MergeableObject<U>
): T & U {
  return {...obj1, ...obj2};
}

type MergeableObject<T> = T extends object
  ? T extends any[] | Function
    ? never
    : T
  : never;

Podsumowanie

Mechanizm ograniczenia typów (type constraints) pozwala na pełniejsze wykorzystanie generyków, dając możliwość definiowania bardziej precyzyjnych, bezpiecznych i reużywalnych funkcji oraz struktur danych. Dzięki ograniczeniom typów, generyki mogą operować tylko na wybranych typach, co pozwala na eliminację błędów i zwiększa bezpieczeństwo kodu.

Warto jednak zachować umiar, ponieważ nadmiarowe stosowanie generyków może skutkować kodem trudnym do odczytania i utrzymania. Generyki mają sens głównie wtedy, gdy istnieje realna potrzeba stosowania dynamicznie określanych typów lub budowania powiązań pomiędzy nimi – w przeciwnym razie warto trzymać się konstrukcji, które będą łatwiejsze w utrzymaniu.
