In the previous lesson, we talked about which internal tools and automations are worth implementing within a team and at the organizational level.

Now we will focus on the purely technical side of such projects.

Over the past four modules, you have been controlling an agent in the IDE and terminal. You typed commands in Cursor, delegated tasks to Claude Code, and watched Codex patch tests. But each time it was someone else's agent. Someone else's loop, someone else's tools, someone else's default settings, and manual control dependent on operator availability.

And now you need something that works exactly your way and in a more autonomous manner, tailored to the team's life cycle rather than a single session of one of you. An agent that conducts reviews in your team's style, that you can run wherever you want, and that doesn't require the rest of the team to open a specific editor.

The next two lessons tackle a problem that is one of the most common symptoms of "agent acceleration." So much code is created that people can't keep up with reviewing it. Review becomes a bottleneck – it is where all the speed gained from the agent is lost.

So what to do? Hire another virtual employee – but not one who generates code, rather one who guards its quality, directly on the path to production.

In this lesson, we will learn the first steps in the entire process of building such an assistant — how to assemble it from available SDKs, how to run it locally, and how to estimate the operating cost. In the next one, we will integrate it into a real review system with a human-in-the-loop — the agent will prepare a preliminary assessment, but the final word will belong to the human.

In this lesson, we are giving up store-bought "ready-made" solutions. Our team-tailored agent will be something you assemble yourself from available building blocks.

These building blocks are precisely the SDK (Software Development Kit). And here comes the first decision that will affect everything you do next: whether you choose an SDK that gives you a preformed agent in a package (e.g., Codex, Claude, Cursor), or an SDK that provides its components, where you add tools and control mechanisms yourself (e.g., OpenRouter SDK, Vercel AI SDK).

Ready-made agent versus agent to assemble

When studying the market, you might get the impression that all "agent SDKs" are more or less the same, just with a different manufacturer's label. This assumption can cost you a lot of time because SDKs are divided into two categories that solve different problems.

SDK with a ready-made agent gives you a whole agent controlled programmatically (scripted). The loop, file tools, bash, sandboxing, repository reading — all tied into a working harness. You provide a goal ("review this diff") and... that’s basically all it takes for the agent’s work to start. This is how Claude Agent SDK, Codex SDK, and Cursor SDK work.

The price for convenience: you are tied to the models, runtime, and pricing of a given provider.

SDK for assembling an agent gives you the agent components and mounting points: call the model, run the tool, attach the result, repeat, stop. The Harness, which you get ready-made on the other side, here you assemble yourself — you add your own tools, and in return you get a model-independent interface. This is how the OpenRouter Agent SDK and Vercel AI SDK work.

The question "which SDK is the best?" is poorly framed. A better question is: "do I want a ready-made coding agent or a framework to build one myself?". Almost every difference between these tools stems from this one division.

And it's best to feel it firsthand by getting acquainted with both integration approaches we can choose here.

The same agent on two SDKs

To feel the difference between two types of SDKs, we will now discuss two approaches to creating a minimal code review agent, currently running locally and in an independent location. We will learn the similarities and differences, and at the end of this stage, we will implement one of the options in our project.

The agent will receive a git diff as input and return a structured evaluation in JSON format: five criteria scored on a scale of 1-10 (implementation correctness, idiomatic usage, complexity, test coverage, security), a binding verdict (pass/fail), and a brief summary.

Everything locally, with a single command git diff | npx tsx review.ts.

The scenario is fixed — the same diff as input, the same JSON as output. The variable is the SDK category, and as a side effect of the chosen option, the number of decisions you have to make along the way. We will use TypeScript - currently the programming language that gives us the most possibilities in this particular category of tools.

All snippets from this lesson — in a complete, runnable version with imports, schemas, and error handling, which we omit here for brevity — can be reviewed and run by downloading the agent-sdk-examples repository. Just clone it, run npm install, and execute the examples one by one with a single command (npm run ...).

Let's start with what is identical in both versions: the system prompt, which calibrates the agent's further operation, and the output result schema. We separate them once into a separate module (in the agent-sdk-examples repository, this is common/review-schema.ts) so that both implementations use a single definition.

We write the output schema based on the zod library — this is our single source of truth, from which we also derive the Review type. At this point, we are not strictly focusing on the quality of the prompt for the review, but rather we want to put everything together into one whole and continue iterating on it.

import { z } from "zod";

const SYSTEM_PROMPT = `You are a precise, constructive code reviewer evaluating a pull request.
Assess the given diff against five criteria on a scale of 1-10 (1 = serious gaps, 10 = exemplary):
implementation correctness, idiomaticity, complexity, test coverage relative to risk, security.
Then issue a binding verdict (pass/fail) for the whole change and include a short summary (2-3 sentences)
in Markdown, on which the PR author will be able to act.`;

// We keep the scores as plain z.number(): Anthropic's structured output rejects
// minimum/maximum on the integer type, so we enforce the 1-10 range with the field description and the prompt,
// not the schema itself.
const REVIEW_SCHEMA = z.object({
  implementationCorrectness: z.number().describe("Implementation correctness: whether the code does what it declares (scale 1-10)"),
  idiomaticity: z.number().describe("Idiomaticity: conformance with the conventions of the language and the project (scale 1-10)"),
  complexity: z.number().describe("Complexity: simplicity of the solution relative to the problem (scale 1-10)"),
  testRiskCoverage: z.number().describe("Test coverage proportional to the risk of the changed paths (scale 1-10)"),
  securitySafety: z.number().describe("Security: no vulnerabilities and no secret leaks (scale 1-10)"),
  verdict: z.enum(["pass", "fail"]).describe("Binding verdict for the whole change"),
  summary: z.string().describe("Summary in Markdown, ready as a comment on the PR"),
});

// Configuring the target field ensures compatibility between zod and the Claude Agent SDK
const REVIEW_JSON_SCHEMA = z.toJSONSchema(REVIEW_SCHEMA, { target: "draft-07" });

type Review = z.infer<typeof REVIEW_SCHEMA>;

The AI SDK will take the output definition directly from zod objects, and the Claude Agent SDK will receive a JSON Schema derived from the same object via z.toJSONSchema().

Note that we described each field with .describe(). This is not just aesthetics — the field description is a crucial lever for controlling the model when generating the expected response format. It works with high effectiveness, so this is exactly where you refine ambiguous names, units, or format. In our schema, the description even conveys the rating range (scale 1-10) — because, as in the comment above, we do not enforce it through the schema itself.

You can push this lever further. Instead of a one-sentence description, you can insert an entire rubric that explains to the model what an extreme assessment looks like:

const REVIEW_SCHEMA = z.object({
  implementationCorrectness: z.number().describe(
    "Implementation correctness: whether the code does what it declares (scale 1-10). " +
    "1: the logic is wrong or silently breaks existing behavior. " +
    "10: correct on the main path, in edge cases, and in error handling."
  ),
  // ...
});

The impact of this field on the agent's operation can be tested iteratively, based on the instructions from the next lesson.

Version with a ready-made agent (Claude Agent SDK)

In this scenario, you install a single agent package (plus zod for the shared schema) and... that’s basically it (at least for the beginning). Using the documentation, you configure the agent core (prompt or the so-called preset, the number of tool handling turns, their definitions, or the output format), provide the input parameter, and then the actual evaluation process begins.

Before you run the code below, you need a minimal setup: a Node project as an ES Module and one runtime package plus developer tools to run TypeScript.

npm init -y && npm pkg set type=module       # project as ESM
npm install @anthropic-ai/claude-agent-sdk zod # SDK with a ready-made agent + zod for the data type contract
npm install -D tsx @types/node                # running TS + Node types

We leave key authentication for later — if you have an active Claude Code session, the script will use your credentials without an explicit key (we'll come back to this in the costs section). If you're developing code in a repo where you have working changes, you can pass the current diff directly to the reviewer:

git diff | npx tsx review.ts

Alternatively, you can also load the diff from a file or any other source. The next part is to use the available functions from the downloaded SDK:

import { query } from "@anthropic-ai/claude-agent-sdk";
import { z } from "zod"; // only to convert the shared schema

// Reading arguments from stdin
async function readDiff(): Promise<string> {
  const chunks: Buffer[] = [];
  for await (const chunk of process.stdin) chunks.push(chunk as Buffer);
  return Buffer.concat(chunks).toString("utf8");
}

// Review process based on the git diff
async function review(diff: string): Promise<Review> {

  // Agent configuration
  const result = query({
    prompt: `Review this diff:\n\n${diff}`,
    options: {
      systemPrompt: SYSTEM_PROMPT,
      model: "claude-sonnet-4-6",
      tools: [],
      maxTurns: 2,
      outputFormat: { type: "json_schema", schema: REVIEW_JSON_SCHEMA },
    },
  });

  // Processing the response and handling errors if any
  for await (const message of result) {
    if (message.type !== "result") continue;
    if (message.subtype === "success") {
      const parsed = REVIEW_SCHEMA.safeParse(message.structured_output);
      if (!parsed.success) throw new Error(`Invalid structured output: ${parsed.error.message}`);
      return parsed.data;
    }
    throw new Error(`Review failed (${message.subtype}): ${message.errors.join("; ")}`);
  }
  throw new Error("The agent returned no result");
}

// Entry point of the whole process
const diff = await readDiff();
console.log(JSON.stringify(await review(diff), null, 2));

Note what you got for free, and what you need to remember:

  • The entire agent workflow is in query(). You don't compose it manually — you provide the goal and options declaratively, the harness handles the rest.
  • Output format. It is enough to provide outputFormat, and the SDK will validate the result against the schema by itself. However, structured_output is typed as unknown, so in the end, we pass it through REVIEW_SCHEMA.safeParse() to obtain a guarantee of the input type.
  • At least two turns. We set maxTurns: 2 because the run is divided into two steps: in the first turn, the model reads the diff and formulates the evaluation, and in the second, it emits it as a structured JSON according to the schema.
  • Dozens of tools in reserve. But here you disable them with tools: [] because we want to limit accidental invocations.
  • Role of the reviewer. Defined by the systemPrompt field - we will return to this later.
  • You catch the error yourself. When subtype is not success, you read errors and finish your way. If you don't do this and the loop continues, the SDK will throw a raw exception instead of a readable message.
  • Start without a key. If you have an active Claude Code session on your machine, you don't need to explicitly provide a key at the start in the Claude Agent SDK - the script will use your credentials. However, this must change, for example, when integrating the agent in CI/CD - we will return to this later.

Version to deploy (Vercel AI SDK 6)

The same scenario, the same JSON output. But you install three separate packages – the base (ai), the model (@openrouter/ai-sdk-provider), and the schema (zod), working as three interchangeable pieces giving you more flexibility.

npm init -y && npm pkg set type=module
npm install ai zod @openrouter/ai-sdk-provider
npm install -D tsx @types/node

And we take advantage of this flexibility right away: instead of an LLM from Anthropic, we point to a model outside of that ecosystem (GLM via OpenRouter).

import { ToolLoopAgent, Output, stepCountIs } from "ai";
import { openrouter } from "@openrouter/ai-sdk-provider";

// Reading arguments from stdin
async function readDiff(): Promise<string> {
  const chunks: Buffer[] = [];
  for await (const chunk of process.stdin) chunks.push(chunk as Buffer);
  return Buffer.concat(chunks).toString("utf8");
}

// Review process based on the git diff
async function review(diff: string): Promise<Review> {

  // Using the so-called agent loop from the AI SDK
  const reviewer = new ToolLoopAgent({
    model: openrouter("z-ai/glm-5.1"),
    instructions: SYSTEM_PROMPT,
    tools: {},
    output: Output.object({ schema: REVIEW_SCHEMA }),
    stopWhen: stepCountIs(2),
  });

  const { output } = await reviewer.generate({
    prompt: `Review this diff:\n\n${diff}`,
  });
  return output;
}

const diff = await readDiff();
console.log(JSON.stringify(await review(diff), null, 2));

Conceptually, working with ai-sdk version 6 resembles declarative solutions like the Claude Agent SDK. This is all thanks to the new ToolLoopAgent. This API was introduced to the library only in AI SDK 6 (stable since December 2025) — in AI SDK 5 the same class was experimental and called Experimental_Agent (with a system field instead of instructions). The consequences of this change will be discussed later in the lesson.

And what do we achieve with this configuration?

  • You import the model explicitly. Here we immediately take advantage of the mentioned flexibility by pointing to a model from the broad OpenRouter directory. Switching providers is a one-line change: instead of @openrouter/ai-sdk-provider, you insert @ai-sdk/anthropic or @ai-sdk/openai.
  • The workflow starts from the ToolLoopAgent class. You provide the tools yourself — at this moment we do not need custom definitions.
  • The same schema, without conversion. Output.object takes our REVIEW_SCHEMA (zod) directly, without additional conversion.
  • You set the stop condition manually. stopWhen: stepCountIs(2) is your equivalent of maxTurns. The loop would stop anyway by default settings after stepCountIs(20) — we consciously shorten it to two steps because the review does not need more.
  • The API key must be explicitly loaded. Without OPENROUTER_API_KEY this code won't run: @openrouter/ai-sdk-provider needs your credentials.

With both implementations in mind, the question arises - which option should you choose?

None of these SDKs is "better". An SDK with a ready-made agent is a way to quickly deploy a coding agent working on your file system. You reach for an SDK like Vercel’s when the choice of model, switching provider, or full freedom to replace tools matters to you more than built-in config and harness.

Here lies the most common mistake: someone chooses an SDK based on brand familiarity ("I use Claude, so I'll take the Claude SDK"), treats the choice as a cosmetic detail, and hits a wall when the task actually required a different category.

Therefore, it is worth starting with the question of how much flexibility you need and how much you would like to iterate with different models, and only then move on to "which solution to choose?". If costs do not matter, you trust the provider, and you want to use the same engine as Codex CLI or Claude Code, then ready-made packages with agents (Codex SDK, Claude Agent SDK) will be a better option for you. However, if you want to have control over the entire process or remain in the process of continuous optimization of each component - then it is better to assemble the agent yourself.

What the agent loop really does

In both versions of the script, we conventionally referred to the agent's work as a "workflow" - in the first case it was hidden by query and in the second by ToolLoopAgent. However, it needs to be specified that the agent's workflow is the so-called tool loop. What does this mean? When working with agents in an IDE or Terminal, you saw this live - it's all the intermediate reasoning operations and external tool invocations that happen between your query and the agent's final response.

Here, in the script version, it is not a side process, but actually the core of what we build. It affects the costs, capabilities, and resilience of the agent.

Anthropic describes it in five steps:

  1. Input. The model receives your prompt along with the system prompt, tool definitions, and the conversation history so far.
  2. Decision. The model assesses the state and chooses what to do next: respond with text, request the invocation of one or several tools, or both at the same time.
  3. Execution of tools. The requested tools are launched and their results are collected.
  4. Return. The results return to the model as input for the next round. Steps 2-3 repeat in a loop.
  5. Result. When the model responds with text only, without requesting tools, the loop ends and returns the result — along with the cost and token usage.

One complete cycle (model decision, tool execution, result return) is one turn (hence settings like maxTurns). A simple question is resolved in one or two turns; a complex task can involve a dozen turns (reading files, editing code, running tests) until the model decides it is done. In our review, we consciously cut this down to 2-3 steps, because a diff review doesn't need more.

The entire difference between a ready-made agent and a version to assemble boils down to one question: who writes steps 3 and 4 of the whole loop?

Loop hidden behind ready-made functions and classes. The more blocks you get in the package, the more self-service the agent's loop will be. In Claude Agent SDK, this entire mechanism is hidden by the query() call. The ready agent runs the requested tools itself, attaches their results to the conversation itself, and loops back to the model itself — and it does this without relinquishing control until the agent finishes. In AI SDK 6, this is taken over by ToolLoopAgent, which is also already based on a more declarative setup.

Loops written manually, for AI SDK 5. Before ToolLoopAgent appeared in Vercel AI SDK, loops on AI SDK were written by hand — and this is exactly the code that determines the quality of the agent, or rather its harness.

In simple terms, it looked like this:

const messages: ModelMessage[] = [{ role: "user", content: prompt }];
let step = 0;
while (true) {
  const result = await generateText({ model, messages, tools }); // Generate the model's response
  messages.push(...result.response.messages);  // Append the message to the history
  if (result.finishReason !== "tool-calls") break; // No need to use tools
  for (const call of result.toolCalls) {
    const output = await runTool(call); // Find and invoke the tool
    messages.push({ // Process the result of the next tool
      role: "tool",
      content: [{ type: "tool-result", toolCallId: call.toolCallId, toolName: call.toolName, output }],
    });
  }
  if (++step >= MAX_STEPS) break;                         // you watch the stop condition yourself
}

Count how many things you have to control yourself here: the messages array and appending to it after each turn, the while loop itself, reading the finishReason, manually calling each tool, formatting the result as a tool message and sending it back to the model, finally your own step counter as a stop condition. A mistake in any of these places results either in an agent that does not respond to the tool’s result or a loop that does not want to stop.

What ToolLoopAgent does now. AI SDK 6 wraps exactly this skeleton into a single class. The entire while loop, appending messages, reading the finishReason, running tools, and returning their results happen underneath generate(). You only declare the components: model, tools, output, and stopWhen — and the loop that you previously composed manually is ready inside. That’s why our version to assemble is as short as the one with the ready-made agent.

The ready-made agent (Claude Agent SDK, Codex SDK) and ToolLoopAgent hide the same loop; they differ in how much you get outside the loop in the package (tools, sandbox, compaction in the ready-made agent) versus how much remains on your side (model selection, custom tools, persistence in the version to be assembled). The loop is the common denominator — the rest is exactly that category difference we started with.

Does the agent remember the previous session flow

The memory of our reviewer is fleeting – you run it, it returns JSON, and finishes. The next npx tsx review.ts starts from scratch — it knows nothing about the previous run. When reviewing diffs, this is even convenient because each run is clean. But as soon as the agent enters a real work cycle, the question returns: the review comes after fixes, someone asks for clarification, the change evolves. Does the agent remember what it has already analyzed, or does it read everything from scratch every time?

This is one of the biggest differences between the two categories and runs exactly along our main axis of "ready-made versus to be assembled". It's about where the state lives: whether it is held for you by the harness or by your own code.

The ready agent maintains the session by itself. The Claude Agent SDK saves the session state on disk and allows it to be resumed. From the first message (init), you catch the session_id, and then you pass it back on the next call — and the entire context (read files, previous analysis, conversation history) returns to the table, without loading the diff again:

// first pass: remember the session id
let sessionId: string | undefined;
for await (const message of query({ prompt: `Review this diff:\n\n${diff}`, options })) {
  if (message.type === "system" && message.subtype === "init") sessionId = message.session_id;
  // ...receive the result as before
}

// next pass (e.g. after fixes): resume the same context
query({
  prompt: "The author applied fixes. Check whether they address your earlier remarks.",
  options: { ...options, resume: sessionId },   // the agent remembers what it has already seen
});

You can even fork a session to explore an alternative path without breaking the original. The state lives in the harness — you only operate on its identifier. The other ready-made agents work the same way: Codex keeps threads in ~/.codex/sessions and you resume them by id (resumeThread).

The version to be submitted has nothing to resume — because there is no session. Vercel AI SDK does not recognize the concept of a "session" saved on disk. Each generate() call is a separate shot; the ToolLoopAgent class stores configuration (model, tools, stop condition), but not history. If the next run is supposed to remember anything, you carry the history with you — you keep an array of messages and pass it back:

// no session: you manage the history yourself and provide it with each call
const messages: ModelMessage[] = [{ role: "user", content: `Review this diff:\n\n${diff}` }];
const first = await reviewer.generate({ messages });
messages.push(...first.response.messages);                    // append what the model replied

messages.push({ role: "user", content: "Check whether the fixes address your remarks." });
const second = await reviewer.generate({ messages });         // continuation = you carried the context over

Note: You set output on the agent (in the ToolLoopAgent constructor), not on a single generate() call. Therefore, the same reviewer with structured output would also enforce the schema on the follow-up question — if the second round should return plain text, use a separate agent without output for it, passing the same messages array.

A ready-made agent does part of the work for you (persistence, resuming, forking sessions), but in return you are tied to its runtime; the version to assemble does nothing on its own, but you decide what survives to the next run, where you save it (file, database, queue), and when you load it. In a one-time review this difference doesn't hurt. It becomes crucial with asynchronous and long-living agents, which are the focus of the lesson Async & Remote Agents (M5L5): the whole point there is that the agent returns to the task after hours and needs to know where it left off.

What your agent inherits from the repo: rules, skills, security

Since you have layered the rules for the agent across four modules — CLAUDE.md, AGENTS.md, skills in .claude/skills, hooks, permission rules — does the agent you are currently assembling from the SDK even read all of this?

The answer is exactly the same in nature as the entire lesson: it depends on the category. These rules are not magic built into the model — they are files on disk that someone has to load and inject into the agent. A ready-made agent can pick them up on its own, because it’s the same harness that reads them when you control it manually from the terminal. In the version to be assembled there is no “automatic” at all. If you want your agent to know the repo rules, you load them with your own code.

The ready agent inherits the repo configuration — because it is the same harness. Claude Agent SDK is literally Claude Code packaged into a library, so it knows the same places you check when working in the terminal:

WhatWhere the ready agent loads from (Claude Agent SDK)
Project memory (instructions)CLAUDE.md or .claude/CLAUDE.md
Skills (auto or /name).claude/skills/*/SKILL.md
Commands (legacy).claude/commands/*.md
Hooks (e.g. PreToolUse, PostToolUse)configuration + callbacks in options
Permissions / edit modeallowedTools, permissionMode
MCP servers, pluginsmcpServers, plugins

By default, the SDK loads settings from .claude/ in the working directory and from ~/.claude/ (you can narrow this down with the settingSources option — user / project / local). But there's one nuance here that's easy to trip over, and our review.ts is a perfect example of this: to actually have the agent load CLAUDE.md, you need to enable the system preset claude_code. In our review, we provided a custom systemPrompt (reviewer's role) and tools: [] — deliberately cutting off access to project memory, skills, and built-in tools. This was intentional: the diff review is meant to be narrow and predictable, not controlled by a random CLAUDE.md. If you wanted the opposite — an agent that respects repo conventions just like your Claude Code — you use the preset and settings loading:

// A ready agent inherits rules, skills, and repository memory
const result = query({
  prompt: `Review this diff:\n\n${diff}`,
  options: {
    systemPrompt: { type: "preset", preset: "claude_code" }, // pulls in CLAUDE.md + conventions
    settingSources: ["project"],                              // .claude/ from the repo: skills, settings
    skills: "all",                                            // expose the discovered skills to the model
    allowedTools: ["Read", "Glob", "Grep"],                   // let it look into neighboring files
  },
});

Two things determine whether the skill will actually work here. skills: "all" is the only place in the SDK where you expose discovered skills to the model (you can also provide a list of names, e.g. ["greeting"]) — the preset and settingSources provide the files, but this option lets them into the list seen by the model. And settingSources: ["project"] loads .claude/ from the working directory — if your .claude/skills is located somewhere other than where you run the script from, explicitly specify that directory with the cwd option.

The other ready-made agents work the same way, each in its own ecosystem. Codex SDK runs the codex binary as a subprocess, so it inherits exactly what the CLI has: the AGENTS.md files from the repo, the configuration from ~/.codex/config.toml, and the sandbox presets — and through the config option in the SDK you add overrides (they go to the CLI as --config flags). Cursor SDK loads skills from .cursor/skills/ and hooks from .cursor/hooks.json — the same project config that powers the Cursor editor. In all three, the rule is the same: it's the same harness, so the rules you wrote for manual work travel with the agent for free.

The assembly agent does not inherit anything — because there is no harness to read it. Vercel AI SDK and OpenRouter Agent SDK have no knowledge of CLAUDE.md, AGENTS.md, .claude/skills, or .cursor/hooks.json. They only receive what you explicitly provide them in instructions / system. Your repo rules for such an agent simply do not exist until you load them yourself:

// Version to assemble: rules won't load themselves — you read and inject them yourself
import { readFileSync } from "node:fs";

const projectRules = readFileSync("CLAUDE.md", "utf8"); // repo conventions, loaded manually
const reviewer = new ToolLoopAgent({
  model: openrouter("z-ai/glm-5.1"),
  instructions: `${SYSTEM_PROMPT}\n\nProject conventions:\n${projectRules}`,
  // ...
});

The build version gives you full control over what goes into the prompt, in exchange for nothing happening automatically. Skills as auto-loaded abilities, lifecycle hooks, project memory — in this world these are patterns you implement yourself (you attach the rule to instructions, create a "hook" with your own StopCondition or logic around the loop), rather than files that the framework finds for you.

Security follows the same division. Rules for the agent are not only "how to write code" — they also include "what it is not allowed to do." And here, the ready-made agent again gives you a ready mechanism: a list of allowed tools (allowedTools), permission mode (permissionMode), the PreToolUse hook that can block risky calls before they execute, and in Codex, sandbox presets set per turn (read_only for planning, workspace_write for changes). These are the same safeguards that protect you when the agent operates in the terminal — except now you configure them from code. In the build-your-own version, there is no sandbox or permission system: the agent can only access tools you wrote yourself, and the only gatekeeper is your own approval gate (needsApproval in Vercel, HITL modes in OpenRouter).

Practical takeaway for choosing an SDK: if your team has already invested in rules, skills, and config for a specific tool (CLAUDE.md, AGENTS.md, .cursor/), a ready-made agent from the same ecosystem will carry that work along without rewriting. If you prioritize model swapping and vendor independence, you accept that you maintain rule consistency yourself — and this work, i.e., turning repo-local rules into a shared, team-wide registry of skills and conventions that the entire team uses, is the topic of the lesson Shared AI Registry: skills, commands, and rules for the team (M5L4).

Who can see your diff now

The SDK itself does not directly affect the privacy policies of individual solutions. Whether your code can be used for training, how long it is stored, and where it physically lands is determined on the backend, specifically: it depends on the authentication path you choose.

The Claude Agent SDK has two different data modes depending on how you log in:

  • Key from the console (commercial). Commercial terms explicitly state that Anthropic does not train models on your content, and the producer directs agent creators to use this key. It is a safe default choice for corporate code.
  • Subscription login (consumer). If you don't set a key, the agent silently uses your Claude Code login — and claude -p and the entire Agent SDK on a Pro/Max subscription fall under consumer terms. Since August 28, 2025, training is enabled by default there (you disable it in Claude's privacy settings), and retention lasts up to 5 years until you disable it — compared to 30 days and no training on a commercial key (Anthropic consumer terms).

The rule repeats for every provider: the cheapest and most convenient path is often the most "leaky." Consumer login Claude, Codeks login through free ChatGPT, free models on OpenRouter. Each of them sacrifices some data protection in exchange for lower cost or fewer formalities. So you choose the path based on the sensitivity of the data, not the other way around.

In the version to be assembled, there is yet another plot twist: the ai library (Vercel) by default does not interfere with your data at all. The request goes directly to the provider you imported, and only their terms apply. The SDK provider and the entity processing your code can be two completely different companies. Only enabling the optional proxy (Vercel AI Gateway) inserts Vercel into this path.

And if you are bound by data residency requirements in the EU, it's good to know in advance where the gaps are: Vercel's AI Gateway only operates in the USA, and Cursor does not offer self-service residency in the EU. In both cases, the solution is often to specify your own provider in the EU region. This is again the same leverage: in the deployable version, you keep control on your side.

Cost control

Since the agent is supposed to run frequently, two operational questions arise: how much does it cost and how to control that cost. The same authentication path that just determined the data also determines the bill. Beyond that, the answer again depends on the SDK category — and again, the ready-made agent does part of the work for you.

You retrieve tokens directly from the response object. In the Claude Agent SDK, the same result message, from which you extract structured_output, also carries the usage field (input and output tokens, broken down by cache):

if (message.subtype === "success") {
  message.usage.input_tokens; // tokens (snake_case, broken down by cache)
  message.total_cost_usd;     // ready cost in USD — no rate table
  message.modelUsage;         // breakdown per model (camelCase: inputTokens, costUSD…)
}

In the version for assembly (AI SDK), the generate() function returns totalUsage alongside the result — in our code we only destructure { output }, but the tokens are waiting right next to it:

const { output, totalUsage } = await reviewer.generate({ prompt });
// totalUsage.inputTokens, totalUsage.outputTokens, totalUsage.totalTokens

Real-time consumption. During a longer loop, sometimes you want to see the consumption incrementally — if only to catch a run that started looping before the algorithm reaches stepCountIs. The build-it-yourself version gives you a hook for this: onStepFinish fires after each turn and hands over the step number, token consumption, and reason for termination, so you build per-step telemetry yourself:

await reviewer.generate({
  prompt,
  onStepFinish: ({ stepNumber, usage, finishReason }) => {
    console.log(`step ${stepNumber}: ${usage.inputTokens} in / ${usage.outputTokens} out (${finishReason})`);
  },
});

The cost in dollars is another story. The Claude Agent SDK calculates it for you — result has a ready field total_cost_usd, as well as modelUsage (breakdown per model), num_turns, and duration_ms. You get a complete set of operational metrics without any rate table.

The version for composition gives you only tokens, and you calculate the cost yourself. And here again, it is the provider, not the SDK, that contributes part of the solution: the OpenRouter provider we used returns the actual, billed cost in providerMetadata.openrouter.usage.cost — provided that you enable usage reporting (usage: { include: true }). With a provider that does not report cost, you are left with multiplying the tokens from totalUsage by the model’s rates.

The hard cost limit breaks out of this division. With tokens and cost in dollars, the rule was "a ready-made agent does part of the work for you." With the hard ceiling, this rule no longer applies: whether you get a ready-made primitive depends on the specific SDK, not on whether it is a ready agent or a version to assemble. A ready agent provides it through the maxBudgetUsd option — once exceeded, it stops execution and returns error_max_budget_usd, the same error subtype that you catch in the block above.

On the side to assemble, the ready-made primitive maxCost belongs to the OpenRouter Agent SDK. And watch out for the name: in our review code we use OpenRouter as a provider under the Vercel AI SDK — which is completely different from the OpenRouter Agent SDK, a separate SDK for assembling an agent. The same brand, two different things:

// ready agent (Claude Agent SDK):  options: { maxBudgetUsd: 0.5, ... }
stopWhen: maxCost(0.5) // OpenRouter Agent SDK: hard ceiling of 50 cents per run

The exception is Vercel AI SDK 6: the named maxCost is not present here, and simply connecting OpenRouter as a provider does not change that. You can achieve the same effect, but you need to write your own StopCondition that counts the cost step by step.

One thing worth knowing in advance: starting June 15, 2026, developer use of Claude on subscription plans (claude -p and Claude Agent SDK) will consume budget from a new, separate monthly Agent SDK credit, distinct from interactive work limits — meaning the agent in a loop will no longer eat into your budget for manual terminal sessions and vice versa. The mechanics and amounts can change, so before deployment, check the Agent SDK credit description and the current pricing.

The token rates are also variable. For mid-2026, the same pattern you got used to in the prework continues: a more powerful model (Opus) costs significantly more per token than the daily (Sonnet) or the cheap one (Haiku). That's why in the example with the ready agent we left Sonnet as the default model. This is the same logic of matching the model to the role that you encountered in the prework [3.5].

We are still sticking to the local run here: the agent runs on your machine, and you read tokens and cost directly from its response.

What can be built from this

The diff reviewer is just the first example, because reviews tend to become the bottleneck the quickest. But since you already know how to assemble an agent from the SDK, the same pattern — take an event from the development cycle, hand the tedious part to the agent, collect a structured result — applies to most places where people waste time on repetitive work. A few integrations into the SDLC that truly relieve the team's burden:

  • Code review as the first pass. The agent evaluates the diff before a human does: it catches obvious errors, test gaps, and deviations from conventions, leaving the reviewer to make decisions that truly require human judgment. This is exactly our agent, just integrated into the pull request — how to do this with a human in the loop is shown in the lesson Code Review in the AI Era (M5L3).
  • Self-heal of a red build or ticket. The agent receives a failed CI log or a report, reproduces the problem remotely, suggests a fix, and opens a draft PR for approval. Instead of "someone has to tackle this tomorrow morning," you get a ready starting point.
  • Triage of the backlog and bug reports. The agent labels new submissions, adds reproduction steps, indicates probable duplicates, and links related code — a tedious sorting process that usually blocks at the entry point.
  • Migrations and codemods. Mechanical rewriting of the same pattern in hundreds of files (changing a library API, unifying imports, refactoring to a new convention) — a boring task for a human, perfect for delegating to an agent.
  • Completing tests, docstrings, and changelogs. The agent periodically adds missing test cases, generates function descriptions, or composes release notes from the diff itself — small backlog items that never naturally reach the top of the priority list.

One caveat before you imagine an agent constantly monitoring the repo: a 24/7 running agent is a budget burner. Every run consumes tokens, and a loop running nonstop "just in case" burns the budget on work nobody ordered. Therefore, two common-sense rules. First, run the agent on an event that matters (PR opening, red build, new issue). Second, for each task, choose the cheapest model that does it well enough — triage or labeling can easily be handled by a cheap model, and the powerful (and expensive) one is reserved for when real code reasoning is needed. Additionally, run experiments and optimize the environment — you'll learn the details in the next lesson, using an in-depth Code Review example.

Agent SDK Cheatsheet

We showed two SDKs, but there are more in the game. The decision is easiest to make in two steps: first the category (ready-made agent or an agent to be assembled?), then the ecosystem (what do you need for the future?). The cheat sheet below is a help for that second step, not a ranking.

Claude Agent SDK

  • Category: ready agent
  • Strength: top coding agent, 14+ tools, native structured output, ready total_cost_usd and built-in cap maxBudgetUsd
  • Be careful with: Anthropic models; TypeScript V2 version with streaming is in preview stage

Codex SDK

  • Category: ready agent
  • Strength: wraps the codex CLI (JSONL via stdio), sandbox presets per turn (plan in read-only, changes in workspace-write)
  • What to watch out for: requires the current CLI; update SDK and CLI together

Cursor SDK

  • Category: ready-made agent
  • Strength: runtime cloud/self-host/local, repository indexing, continuing work in IDE
  • Watch out for: public beta, payloads and types may change; no first-class structured output

OpenRouter Agent SDK

  • Category: to be assembled
  • Strength: 400+ models in one callModel, fallback routing, hard maxCost
  • What to watch out for: this is the OpenRouter SDK (not Anthropic); you add the tools yourself

Vercel AI SDK 6

  • Category: to be submitted
  • Strength: standard in the TS/Next.js world, the best streaming to UI, clean structured output, stable since December 2025
  • What to watch out for: no named maxCost (custom StopCondition); v7 already in canary

Decision shortcuts that work in practice:

  • Do you want a coding agent editing the real file system? You take a ready-made agent and choose from the ecosystem: Anthropic is Claude, OpenAI/Codex is Codex, and a fleet of agents in the cloud with auto-PR is Cursor.
  • What matters is the choice of model plus a hard cost ceiling? OpenRouter Agent SDK, as the only SDK for assembly with built-in maxCost, fallback routing, and 400+ models in a single call (callModel) — the ready agent also has a hard cap (maxBudgetUsd); without a ready-made solution, only Vercel AI SDK remains.
  • You build in TypeScript/Next.js and want a standard framework to assemble an agent? Vercel AI SDK 6.

If your choice also depends on what you are coding in, here is the whole group across language/runtime (status as of mid-2026 — still pin versions in your code):

  • Claude Agent SDK — Python, TypeScript
  • Codex SDK — TypeScript/JavaScript (Node 18+), Python (3.10+)
  • Cursor SDK — TypeScript only
  • OpenRouter Agent SDK — TypeScript, Python (Go in client SDKs)
  • Vercel AI SDK 6 — TypeScript/JavaScript, UI integrations are universal (Node, Next.js, React, Svelte, Vue, Expo)

The same model, different effects?

In the production version, we replaced the provider with a single line and moved on. It's easy to conclude from this that the model is a portable commodity: transfer any weights to any harness, and you will get the same result. This is a simplification worth knowing before it starts to surprise you in practice.

Because the coding model does not emerge in a vacuum. Large labs train it with their own harness in a loop: Anthropic fine-tunes Claude under Claude Code, OpenAI trains Codex through reinforcement learning on real tasks in environments that simulate its production runtime. So the model does not learn coding itself in isolation — it also learns the conventions of a specific harness: the names of tools, the shape of schemas, the protocol of "first make a plan" or "at the end, try to run tests." These habits soak into the weights, not just the system prompt.

The consequence is that the same weights can behave differently depending on what wraps them — and the deciding factor here is largely the harness configuration, not the model itself. The clearest proof can be seen when leaving the model alone and only changing the settings: on the Terminal-Bench 2.0 benchmark, the same Claude Opus scored 77% on the default configuration and 92.1% with the fine-tuned CLAUDE.md (the community "Mythos" config) — more than 15 percentage points just from the instruction file alone, without changing the model or touching the prompt.

For comparison, the same Opus in Cursor achieved as much as 93% — and here it is easy to mistakenly conclude that "Cursor has a better model." It does not: it is the same model. The creators of Cursor also put a lot of effort into properly packaging the model with additional instructions and tools before it is included in the final product.

The same pattern is clearly visible on the public Terminal-Bench 2.0 board, where the "Agent" column is simply the harness, and the "Model" — the weights. The same weights can end up several to a dozen places apart depending on who wrapped them.

  • Claude Opus 4.6 in the Terminus 2 harness (Terminal-Bench team) scored 62.9%, and in the "official" Claude Code — 58.0%.
  • GPT-5.3-Codex: 75.1% in Simple Codex versus 64.7% in Terminus 2 — over 10 points difference on the same weights.
  • Gemini 3 Pro: 69.4% (agent Ante) versus 56.9% (Terminus 2).
  • GPT-5.2: 62.9% (Codex CLI) versus 54.0% (Terminus 2).

If someone were to read this table "by model," they would get a completely different ranking than reading it "by agent." The same name in the "Model" column does not guarantee either the same position or the same result — because part of the work is done by the harness around the weights. As you can see, depending on the model, sometimes the creators and sometimes the community manage to get the most out of a given setup.

There is another layer here that calls for reading such tables with caution: some entries are custom agents tuned directly for benchmarks — someone's proprietary harness whose sole task was to squeeze the maximum out of a specific set of tasks. These are often unofficial runs (reported by the community, not by the model labs), so the result says more about the quality and tuning of the harness than about the model itself. In other words: a high position can obscure the picture — it’s easy to mistake it for a “better model” when in reality you are looking at a better-fitted (sometimes even over-optimized for the benchmark) wrapper.

There is also good news in this: since the result is determined by tuning, it is a lever you hold in your hand — in the build-your-own version, you fine-tune the default config yourself (your equivalent of adding your own CLAUDE.md). Hence the practical habit: select the model and harness as a pair, and after swapping the model in the build-your-own version, measure the result again instead of assuming it will be the same. How to measure this quality systematically — in the next lesson, during evaluations.

Agent as an independent, deployable unit

Throughout the lesson, the agent ran locally: npx tsx review.ts, print the result, end. You already know how to build and run it. But what if the agent needs to respond to HTTP requests, maintain state between calls, have its own database handy, and run continuously without managing infrastructure?

Cloudflare Agents SDK (npm install agents) addresses this need. It is a library that allows you to create agents based on so-called Durable Objects, with their own SQLite database, wake-up scheduler, and a URL accessible globally on Cloudflare.

Durable Objects as a foundation

Cloudflare Workers are stateless scripts. Each request starts from scratch — without memory of the previous invocation.

Durable Objects work differently. Each instance is:

  • globally unique, addressable object with a persistent identity,
  • single execution thread (no race conditions),
  • built-in SQLite database collocated with the instance,
  • the ability to sleep and wake without losing state.

The Cloudflare Agents SDK builds on this foundation the class Agent<Env, State>. Each agent class is a Durable Object class. Each named instance (e.g. user-abc123 or review-bot-default) is an independent entity with its own database and state.

Class Agent — what you get out of the box

import { Agent, routeAgentRequest } from "agents";

export class ReviewAgent extends Agent<Env, { reviewCount: number }> {
  initialState = { reviewCount: 0 };

  async onRequest(request: Request) {
    const diff = await request.text();
    const result = await runReview(diff, this.env.AI);
    this.setState({ reviewCount: this.state.reviewCount + 1 });
    return Response.json(result);
  }
}

export default {
  fetch(request: Request, env: Env) {
    return routeAgentRequest(request, env) ?? new Response("Not found", { status: 404 });
  },
};

Three things change compared to the local script:

  • this.state — state persisted in SQLite; setState() saves, sends to connected WebSockets, and triggers onStateChanged().
  • this.env — Cloudflare bindings: Workers AI without a key, KV, R2, queues, other agents.
  • routeAgentRequest — routes HTTP requests to the appropriate agent instance by the URL: https://your-worker.dev/agents/review-agent/instance-name.

The state that undergoes hibernation

An instance may go to sleep after about 70–140 seconds of inactivity and wake up automatically on the next request. But beware - not everything survives this:

In SQLite, everything that has gone through it survives. The key here is the agent's field this.state and the scheduled tasks (this.schedule).

However, the in-memory variables of the class (e.g., this.cache) and active timers (setTimeout, setInterval) are lost.

The rule is simple: what needs to survive goes into this.state or SQL. Class variables are cache, not the source of truth.

WebSockets with hibernation

The agent handles WebSocket connections throughout their entire lifecycle, freeing up the thread between messages:

async onConnect(connection: Connection) {
  connection.send(JSON.stringify({ type: "welcome" }));
}

async onMessage(connection: Connection, message: string) {
  const data = JSON.parse(message);
  this.broadcast(JSON.stringify({ from: connection.id, ...data }));
}
  • this.broadcast(msg, exclude?) — sends to all connected clients (optionally excluding selected IDs)
  • getConnectionTags(connection) — assigns tags to connections; allows broadcasting to subgroups (max. 9 tags per connection)
  • The per-connection state survives hibernation

For a code review agent, this is not a primary need — but for an agent streaming the PR progress status to the client in real-time, it is a ready infrastructure.

Per-instance schedule

Instead of global triggers at the Worker level, each agent instance has its own schedule:

// run in 10 minutes
this.schedule(600, "runReview", { prId: "123" });

// every hour, a cron expression
this.schedule("0 * * * *", "syncStats", {});

// every 30 seconds
this.scheduleEvery(30, "heartbeat", {});

Scheduled tasks survive restarts — they are stored in SQLite. Identical calls (callback + expression + payload) are idempotent by default, so each time you reconnect, you do not create duplicates.

Deployment Configuration (wrangler.toml)

The agent requires two declarations in addition to the typical Worker:

name = "review-agent"
main = "src/server.ts"
compatibility_date = "2026-06-12"
compatibility_flags = ["nodejs_compat"]

[ai]
binding = "AI"  # Workers AI without an API key

[[durable_objects.bindings]]
name = "ReviewAgent"
class_name = "ReviewAgent"

[[migrations]]
tag = "v1"
new_sqlite_classes = ["ReviewAgent"]  # starts the collocated SQLite database
  • new_sqlite_classes — a one-time migration activating SQLite for the class; without it, this.state and this.sql do not work.
  • nodejs_compat — required for greater SDK compatibility with the Node runtime.
  • [ai] binding — optional; provides this.env.AI to call Workers AI without managing keys.

After wrangler deploy the agent is available at an automatically generated URL: https://review-agent.nazwa.workers.dev/agents/review-agent/{instance-name}. The instance name is any string: user-id, pr-123, default.

Where it makes sense

The local script npx tsx review.ts is easy to configure but limited in resources and lifecycle. The Cloudflare Agent comes into play when:

  • the agent is to be invoked by any external actor (webhook, developer, system on demand),
  • must store and remember the previous state (how many PRs it has reviewed, what the results were),
  • should react asynchronously (accept the request, respond "in progress", send the result when ready),
  • you need a per-instance schedule without a separate cron service
  • you do not want to build dedicated infrastructure for long-living agents

Cloudflare Agents SDK, Durable Objects, and Workers AI is an interesting solution in situations where, beyond simple communication and a few steps of a tool loop, a full agent lifecycle and persistence of its knowledge matter to us. And a global environment, not a remote one.
