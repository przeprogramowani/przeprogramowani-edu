Chatbots VS Agents

In the previous lesson, your project idea underwent a Socratic session and ended up on the disk as prd.md: a contract describing what you are building, for whom, and how you will recognize that it works.

Natural reflex: open a new session, paste the PRD, and write "ok, choose a stack for this project." The model will respond. Probably even sensibly.

But this answer mainly lives in this one session. You can add "save it to a file," sure. But then a week later you again have to remember which file, in what format, and according to what rules.

In the prework [3.2] Prompting patterns and antipatterns, "skills" were one of the items in the instruction hierarchy: system prompt → design rules → memory → skills → prompt. Now this item is being promoted to a fully-fledged work mechanism.

At the end of this lesson, you will run /10x-tech-stack-selector on your own PRD and receive a tech-stack.md file, which the AI-Powered Bootstrap lesson (M1L3) will read and turn into a ready project skeleton. Do you have an existing project instead of a new one? /10x-stack-assess awaits you, which will evaluate your stack for working with the agent. Let's get started.

What is an Agent Skill

An Agent Skill is a folder on the disk with one required file: SKILL.md.

Inside you will find:

  • SKILL.md — instructions that the agent reads after activation. The recommended limit is ~500 lines. Exceeding it? Add additional files instead of expanding the main document.
  • references/ — additional documentation, diagrams, registers, examples. The agent loads them only when needed.
  • scripts/ — executable code (validator, generator, migration script). The agent runs it via bash; the code itself does not go into the context window, only the execution result does.
  • assets/ — templates, icons, fonts used in output files.

Folder with markdown and optional assets. Nothing more.

Structure is just the beginning. More important is how the agent uses these files. In short, the agent discovers skills according to the principle of progressive disclosure in three stages:

  1. Metadata (~100 tokens) — name and description. The agent sees them from the start of the session and based on this decides whether the skill is needed.
  2. Instructions (up to ~5000 tokens) — contents of SKILL.md. They load only after activation. If the skill does not start, these tokens do not exist in the context window.
  3. Resources (unlimited) — files from references/, scripts/, assets/. The agent accesses them during work when it needs specific information.

The context window is the budget (you know this from the prework [3.1] and [3.3]). Progressive disclosure is the way in which skills respect this budget.

Twenty installed skills cost you ~2000 metadata tokens, not ~100,000 tokens of full instructions. The agent knows about all of them, but only reads those it activates.

Skill vs one-time prompt

When you enter a prompt in the chat, you create a one-time instruction. It works here and now, in the context of this specific session. You can ask the agent to save a file, you can force JSON, you can describe the input and output.

The difference is therefore not that the prompt cannot produce the artifact. The difference is that the prompt itself does not provide a permanent, versioned mechanism that you can return to in a subsequent session.

The skill provides five guarantees that a one-time prompt usually does not provide on its own:

  1. It exists on disk. The file(s) in the repository, not text in chat history. You can open it, audit it, modify it, and version it.
  2. Loads progressively. The agent always sees the description (~100 tokens), but reads the full instructions only after activation. The prompt enters the context immediately and in full.
  3. Has an input/output contract recorded in the procedure. It reads a file from disk (e.g., prd.md), saves the result as a file (e.g., tech-stack.md), and does this according to a repeatable pattern. The prompt can also request a file, but every time you must recreate the rules.
  4. It runs by name. You type /10x-tech-stack-selector and you know what you'll get. You don't recreate the full command from memory or copy it from a notebook every time.
  5. It is repeatable between sessions. You close the session, open a new one a week later, and return to the same procedure. The model's output may still vary (lack of determinism), but the work contract remains the same.

The decision rule is simple: repeatable process - reach for a skill. One-time exploration, explanation, or editing - write a prompt and move on.

A heuristic for everyday use: do you catch yourself writing the third similar prompt? It's a sign that it's worth creating a skill.

Test yourself on five scenarios:

SituationSkill or prompt?
"Fix the typo in the README"Prompt — one-time edit.
"Explain to me what this function does"Prompt — one-time explanation.
"Choose a stack for this PRD"Skill — repeatable step with named input (prd.md) and output (tech-stack.md).
"Generate an implementation plan for login"Skill — repeatable step with a named contract (/10x-plan).
"Summarize this long chat session for me"Prompt — one-time analysis of the current context.

There are also other layers of agent configuration, e.g., project rules (AGENTS.md/CLAUDE.md). Here we stick to the contrast between skill vs one-time prompt, because the prompt is the layer you actively use from the first day of working with the agent.

Where to get skills

You already know what a skill is and when to reach for it. The remaining question is: where to get it from?

You don’t have to write everything from scratch. The ecosystem has ready registries:

  • anthropics/skills — official Anthropics skills, including skill-creator (a meta-skill for creating new skills).
  • skills.sh — a public registry with thousands of skills and tens of thousands of installations. It supports many agents: Claude Code, Cursor, Codex, Gemini CLI, and others.
  • Skills of tool creators — e.g. supabase/agent-skills (Postgres best practices, 8 categories of rules) or vercel-labs/agent-skills (skill react-best-practices, 40+ rules for React/Next.js).

Audit before installation. Skills may contain executable code in scripts/. Anthropic states clearly: treat skill installation like software installation.

Before you add a skill:

  • Open SKILL.md — read the goal, scope, and allowed-tools.
  • Check the scripts/ — what does it do? What does it affect?
  • Check the author and source — a known producer, verified repo, or a random fork?

Do not install anything you haven't read. It sounds like a truism, but many people treat skills like ordinary text files, and combined with an agent, a skill can perform unwanted operations.

Two types of skills

There are two types of skills worth distinguishing:

  • Process-based (e.g., /10x-tech-stack-selector) — you run it explicitly, it reads a file, produces a file. A link in the chain.
  • Advisory (e.g., react-best-practices) — loads automatically when you work in a given domain because its description names activation contexts. It does not produce an artifact but influences every decision the agent makes in the session.

Both have the same specification. The difference is in how the skill describes its work and how you run it.

Activation: explicit vs automatic

Advisory skills promise automatic activation: the agent reads the description, recognizes the context, and loads the instructions itself. Sounds great, but it doesn't always work as reliably as the description suggests.

The problem is simple: the model does not know what it does not know. If it thinks it knows the answer, it may not feel the need to load a skill, even when that skill contains exactly the instructions that would be needed.

This does not discredit skills. It only reminds that automatic activation assumes a difficult thing: the model must recognize its own knowledge gap. We leave the details of one of Vercel's evals in the Deep Dive.

What does this mean in practice?

  • Process skills (/10x-tech-stack-selector, /10x-plan) — you explicitly invoke by name. You make the decision, so the agent does not have to guess whether the skill is needed.
  • Advisory skills (react-best-practices, supabase-postgres-best-practices) — treat automatic activation as assistance, not a guarantee. If the outcome matters, name the skill explicitly or revisit it in the design rules we will discuss in lesson four.

Automatic activation will improve along with models and harnesses. For now, plan the workflow so that important steps are explicit, and treat automatic activation as a bonus.

PRD → tech-stack.md in practice

The theory is behind us. It's time to see how the skill works on a real project.

Let's go back to the PRD from the lesson From Idea to PRD (M1L1). We have the prd.md of the 10xCards project, an AI flashcard app.

First: ad-hoc prompt.

You open a new session, add a reference to @prd.md, and write: "ok, pick a stack for this project." The agent will suggest something, probably even accurately.

But what do you actually get?

  • Response in the session, usually without a file on disk.
  • Most likely you will not get any questions about your preferences, only a ready-made proposal.
  • No information whether the starter related to the tech stack is tested and supported.

The result could be reasonable, but /10x-bootstrapper in the next lesson AI-Powered Bootstrap (M1L3) will not be able to generate a project skeleton based on these vague suggestions.

Now: the same PRD via skill.

/10x-tech-stack-selector @context/foundation/prd.md

The skill reads the premises from the PRD (project type, timeline, functional requirements) and gives you a recommendation based on the built-in registry of starters. If the PRD is unambiguous, the standard path closes quickly.

The result of the process will be context/foundation/tech-stack.md with the following structure:

starter_id: 10x-astro-starter

The document will also include a "Why this stack" section linking the requirements from the PRD with the capabilities of the chosen technology stack.

This is the contract. /10x-bootstrapper in the AI-Powered Bootstrap lesson (M1L3) will read this file, see starter_id, bootstrapper_confidence: first-class and will know what to do next.

The agent will not have to guess or improvise.

Why will the skill do it predictably?

Let's take a look at SKILL.md in /10x-tech-stack-selector:

  • description — the agent knows when to reach for a skill ("what stack should I use", "what to choose for the project").
  • allowed-tools — list of tools that the agent can use within this skill. Explicit permissions contract.
  • references/starter-registry.yaml — a registry of verified starters with quality gates. It loads on demand and does not take up the window upfront.
  • references/handoff-schema.md — the output file schema that /10x-bootstrapper can read. The contract is explicit.
  • references/agent-friendly-criteria.md — criteria for assessing whether a stack is agent-friendly. An ad-hoc prompt will not verify this unless you provide such rules to it.

The skill turns the PRD into a repeatable process step and leaves an artifact on the disk that the next link in the chain can read.

Assessment of the Existing Stack

If you are working on a brownfield project (existing codebase), you do not choose the stack from scratch. You assess the one you are working with.

/10x-stack-assess does exactly that: it enters the project directory, detects the stack components (language, framework, build tool, test runner, package manager), and evaluates each through four quality gates, the same ones that /10x-tech-stack-selector uses to select a greenfield stack:

  1. Typed — does the language/framework have a type system that the agent can read?
  2. Convention-based — are there directory and naming conventions that the agent understands?
  3. Popular in training data — is the tool popular enough for the agent to have knowledge about it?
  4. Well-documented — is the documentation available and up to date?

The difference is in the application. /10x-tech-stack-selector uses gates as a selection filter: it rejects starters that do not meet them. /10x-stack-assess uses them as a diagnostic tool. It does not reject your stack, but points out where the agent will face friction and provides a compensation plan.

Output: context/foundation/stack-assessment.md with an assessment of each component individually, identified gaps, and a compensation plan.

Three possible verdicts:

  • ready — the stack meets the gates, the agent is working properly.
  • ready-with-compensation — the stack has gaps, but the compensation plan patches them. Most existing projects end up here.
  • significant-friction — the gaps are serious; consider migrating key components (e.g., adding TypeScript).

Example: your project uses Express.js without TypeScript.

verdict: ready-with-compensation

Express passes the "popular" and "documented" gates, but not "typed" (no type system) or "convention-based" (no enforced directory structure). Without compensation, the agent will generate code that works, but may break your project's conventions.

The compensation plan consists of proposed entries for the agent's configuration files (CLAUDE.md, AGENTS.md). How to create, edit, and maintain these files is the topic of the Agent Onboarding lesson (M1L4).

Both paths (greenfield and brownfield) will converge in this lesson with an equivalent context. The entry point differs, not the way of working.

Skills as the building blocks of 10xWorkflow

/10x-tech-stack-selector is the first visible consumer of PRD. But this is just the first link; the rest of the course consists of more links of the same format.

The entire module 1 prepares the environment for working with the agent: PRD as a contract (M1L1), skills and stack selection (this lesson), project bootstrap (M1L3), agent configuration (M1L4), and from localhost to production (M1L5). Once the environment is ready, module 2 introduces the implementation workflow, where these skills become everyday tools.

The chain you will see in action from module 2:

/10x-research — code exploration with parallel sub-agents. You break down the research question into areas, the skill launches 2–4 agents searching the repository simultaneously, and then merges their results into one document with file:line references. Input: question → Output: research.md.

/10x-frame — questioning the assumptions of the problem before planning a solution. Separates observation from the presumed cause, builds a map of the problem dimensions, tests hypotheses with parallel agents, and asks Socratic questions: not about the solution, but about where in the map the real problem lies. Input: problem description or ticket → Output: frame.md with a reframed (or confirmed) problem statement.

/10x-plan — implementation plan with a scaled question. The more artifacts it receives as input (research + frame), the fewer questions it asks. It assesses the task complexity, adjusts the depth of questions (from 4 to 15), and each option includes a recommendation with a compromise analysis. Input: problem + research + frame → Output: plan.md with phases + plan-brief.md (short summary).

/10x-implement — execution of the plan phase by phase with a full verification ritual. It tracks touched files, runs tests, stops at a manual verification gate, commits indicated files (never git add -A), and after closing the phase enters the commit SHA into the Progress section. Input: plan.md → Output: implemented code with commit history.

/10x-impl-review — implementation review versus plan. Two parallel agents: one searches for drift between the plan and the code (MATCH / DRIFT / MISSING / EXTRA), the other scans security, performance, and adherence to design patterns. Six evaluation dimensions and an interactive decision loop: fix, accept the risk, or save as a rule for the future. Input: plan.md + code → Output: impl-review.md with a list of findings and decisions made.

Each of these skills has the same format: SKILL.md + references/ + optionally scripts/. Each reads the file produced by the previous step and saves the result as input for the next one.

You do not learn a new abstraction in each module. You learn the next link based on the same foundation of the Agent Skills specification.

You will master this way of working once, in module 1. In module 2, when the environment is ready (project set up, agent configured, deployment set), you will run new skills on real tasks in your project. You will see that each of them works exactly like /10x-tech-stack-selector: reads a file, produces a file, the next link moves forward.

This is our definition of "AI-native software engineering": not just single prompts, but a flexible set of skills that grow with you.

Creating custom skills

Consuming ready-made skills is the entry point. But the real change in working with an agent begins when you create your own, tailored to your stack, your domain, your repetitive tasks.

It's simpler than it seems. There are two paths. They differ in cost, formality, and predictability of outcomes.

Path 1: conversation with the model

The simplest way: you open a session with an architect-type model and describe what the skill should do.

You don't need any special tool for this. Just an agent (Claude Code, Cursor, Codex) and your contract description:

  • What does the skill receive as input? (e.g., the prd.md file)
  • What should it produce? (e.g., a tech-stack.md file with specified frontmatter)
  • What steps should it go through?
  • What should he not do?

The agent writes SKILL.md. You read it, run it, see what works and what doesn’t, go back to the conversation, and make corrections. Three to four iterations and you have a working skill.

This path is fast and cheap. It does not require any infrastructure, only your ability to assess whether the result is good. For most skills that you write for yourself, this is enough.

Path 2: skill-creator with evals

Anthropic publishes the meta-skill skill-creator. You describe the input and output contract, and skill-creator generates a SKILL.md skeleton and optionally a set of evaluations that automatically check whether the new version of the skill produces correct results.

Evale automate verification: instead of manually testing each change, you run a set of test cases and receive a report. This provides more predictable results, especially when the skill has many users or is part of a longer chain.

But there is a tradeoff.

Evals cost time and money: each eval is a model call. And designing a good eval requires understanding what exactly you are verifying. Default evals can be too favorable to the new skill version: they test what the new version does well, rather than what it should do well. To truly protect quality, you need to design the eval for the boundaries and edge cases of your contract, not for the happy path.

This is not a flaw of skill-creator. It is a fundamental difficulty of automatic verification. A good eval is more difficult than a good skill.

Which path to choose?

  • Skill for yourself, for your own workflow — path 1. Conversation, iteration, manual tests. Entry threshold: zero.
  • Skill shared with the team or part of the pipeline — path 2, when you care about repeatability and verification without manually testing each change.
  • First 1, then 2 — a natural progression. You start with the conversation, the skill works. When you start changing it more often or pass it on to others, you add evals.

Are you caught writing the third prompt doing the same thing? Open a session with the agent and tell it what goes in and what comes out. You don't need anything more to start with.

Issues with automatic activation

In the lesson, we left a practical conclusion: call important skills explicitly, and treat automatic activation as a help, not a guarantee.

Where does this caution come from?

Vercel verified the effectiveness of automatic skill activation during evaluations of tasks related to Next.js 16. The test concerned new APIs, such as connection(), 'use cache', and forbidden(), areas in which the model could not succeed with training knowledge alone. It had to turn to the proper documentation.

The results were as follows:

ConfigurationEffectiveness
Baseline (no documentation)53%
Skill (default behavior)53%
Skill with explicit activation instructions79%
AGENTS.md with indexed documentation100%

The most important conclusion: the skill with default behavior gave the same result as no documentation. In 56% of cases, the agent did not use it at all, even though it had access to it.

That doesn't mean the skills are faulty. Vercel itself states that they fit well with vertical, specific workflows, especially those that the user invokes consciously. This is exactly what we teach at 10xDevs: process skills that we trigger manually: /10x-tech-stack-selector, /10x-plan, /10x-implement.
