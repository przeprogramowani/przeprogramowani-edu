In previous lessons, we wrapped up the team workshop on working with AI: review standards, custom agents, shared skills, rules, and the artifact distribution mechanism. The agent then ceases to be a private assistant at your desk. Increasingly often, it operates in an environment that has repeatable rules for the entire team.

In the lesson Innovate: more features, less waiting with multiple agents (M2L5) we scaled the work while you are still sitting at your computer managing several parallel tracks. This lesson takes the next step: what if you want to start a task, walk away from your laptop, check the status from your phone, or come back to the result only after the scheduled run?

Sounds like a loss of control? Only if you understand control as constantly watching the Agent’s every move.

When working with remote and asynchronous agents, control does not disappear. It changes location. Instead of watching the Agent’s actions in real time, you choose the moment of control: you configure the environment before starting, monitor selectively, and conduct a review after completion.

A simple idea, but it has a few pitfalls worth knowing about in advance.

When do you control the agent's work?

The biggest mistake with remote agents is lumping everything into one bag: "agent in the cloud," "remote coding," "background agent," "loop." These names sound similar but describe different operational situations.

Instead of starting with the tool, start with the question: when should a human control the work? The answer breaks down into three modes, which differ in the timing of control, place of execution, and main risk:

Mode of operationWhere the agent runsWhat you controlWhen it fitsMain risk
Mode 1: remote control of a local agentYour machine or your own serverAgent session, only from another locationQuick fix, diagnosis, continuing work from a phoneYou are responsible for the environment and machine security
Mode 2: cloud sandboxManaged sandbox in the cloudSetup, network access, secrets, task scopeLimited task that can complete after closing the laptopPoorly prepared environment or overly broad permissions
Mode 3: loops and routinesCloud or loop run on a scheduleTrigger, success criteria, stop conditionsRegular review, report, backlog control, self-checkGreen run without real success, cost, looping

This table is more important than product names. If you know when you want to regain control, it’s much easier to choose the right tool. Regardless of the selected mode, you still end up in the same place: reviewing the work done.

Mode 1: remote control of the local agent

The simplest version of remote work does not transfer computations to the cloud. The agent still runs on your machine or on your server. Only the control interface changes.

The classic option is SSH or mosh to your own development machine and an agent running in tmux. The session survives disconnection, so you can return to it from another device. This solution has one major advantage: you don't add a new provider to the trust chain.

It also has an obvious cost: everything is on your side. You maintain the server, access, updates, secrets, network, and the convenience of working from a phone on your own.

You also have a real choice of workplace here: a daily machine or a separate VPS. The VPS is usually safer because the Agent then does not have access to your private files, keys, and entire home directory. The cost is setting up the environment, but with the help of the Agent itself, it usually takes hours, not days.

If you work in Claude Code, you have a simpler, first-party version of the same idea today: Remote Control. You run claude remote-control (or /remote-control in an ongoing session) and take control of it from the browser at claude.ai/code or from the Claude mobile app. The mechanics are the same as with SSH: The agent runs locally all the time, and the phone and browser are just a window to that session.

The connection is made exclusively via outgoing traffic over HTTPS, so you don't open any ports, and the session automatically resumes after the laptop wakes from sleep or a brief network loss. Everything goes through the Anthropic API, so you don't add any external intermediary to the control path.

The feature is in research preview, works on Pro, Max, Team, and Enterprise plans (not on the API key), and requires Claude Code version 2.1.51 or newer. On Team and Enterprise, the administrator must enable it first.

And where does Happy fit into all this? It's a good option when Remote Control doesn't suit you: you work with an API key, prefer an open-source solution, or control Codex instead of Claude Code. Happy wraps a local Claude Code or Codex and allows you to take over the session from your phone through an encrypted relay. Just like with Remote Control, the Agent still runs locally, and the Happy cloud is only an intermediary passing encrypted messages between your computer and phone.

Both solve a practical problem that SSH often does not handle elegantly: the phone and laptop are on different networks, behind NAT, without a stable address and without the desire for port forwarding. Remote Control or Happy provides convenience, while the actual execution still remains with you.

With Happy, remember one thing: it is still the third party in the control flow. Before it enters the team workflow, check the encryption model, the repository, the device pairing method, and whether such a relay fits your security policy. It is still a low entry barrier to controlling the local agent from the phone, just with an additional trust decision that you don't have to make again with Remote Control from Anthropic.

You choose this mode when you want mobility but do not want to hand over execution to a managed sandbox. It's still work "on your machine," just not necessarily "at your desk."

Mode 2: cloud sandbox

The second mode is qualitatively different. The agent no longer operates on your machine. It gets a fresh environment in the cloud, clones the repository, runs the setup, and continues working even when the laptop is closed.

This is a good model for tasks like: "take this small change, prepare a PR, run tests, and I'll check the result later." You don't need to maintain your own development machine. You don't need to keep a local session alive. You pay for this with environment configuration and trust in the provider's sandbox.

The most important skill here is not about clicking "start". It is about preparing the environment so that the Agent has everything it needs, and does not have what it should not have.

In practice, the sandbox configuration can be reduced to six questions.

What do you prepare before starting?

The setup script runs before the Agent starts working. Here you install system dependencies, missing CLIs, runtime environments, package managers, or other elements that are not present in the base image.

Do not treat setup as a place for experiments. It is part of the execution contract. If the task requires gh, test database migration, or a specific package manager, include it in the setup script instead of hoping the Agent will "somehow handle it."

When does the sandbox have internet access?

The network is one of the most important switches. Some environments allow a trusted allowlist, others disable the internet by default during the Agent's runtime phase and leave it only for setup. This difference is not cosmetic.

If dependencies are to be downloaded during the setup, and the Agent later operates without internet, then the prompt "install the missing package during implementation" will end in frustration or a series of failed attempts. This is not an Agent error. It is a poorly set work boundary.

How do you transfer MCP and tool configuration?

In the previous lesson, we talked about shared AI artifacts as code. The sandbox shows why: the remote environment only sees the tool configuration that it can read at startup.

If the MCP configuration lives only in your local user profile, the cloud will not see it. The remote environment needs the configuration saved in the repository or provided through the official mechanism of the given tool.

In Claude Code on the web, one of the practical patterns is the .mcp.json stored in the repository, which is exactly the type of artifact that we learned to treat as part of the shared workshop in the Shared AI Registry lesson (M5L4).

This is a good test of the team's maturity: does the Agent in the cloud get the same tool context as the local Agent, or does each have its own hidden configuration?

What goes into the cache?

Sandboxes usually keep the setup result or the container state in cache to speed up subsequent sessions. The other side of the coin: changing the setup script, variables, secrets, or network can invalidate this cache.

For you, the principle is important, not the exact number of hours. If the setup is repeatable, short, and explicit, cache helps. If the setup is a collection of manual fixes, cache only hides the problem.

How do you restrict secrets?

The secret in the sandbox should have minimal scope and a reasonable lifespan. Do not give production secrets to a remote (or local) agent.

That sounds obvious, of course, until a "quick fix" comes along and someone inserts a token with broad access into the environment because otherwise the tests won't pass. In asynchronous work, such shortcuts hurt more because the Agent can operate for a long time without your constant supervision.

Which tools do you assume are available?

Do not mix tool lists between providers. Claude Code Web, Codex Cloud, and other sandboxes may have similar operating systems, but they differ in the base image, network model, MCP configuration, and cache.

Instead of assuming a "standard cloud agent," explicitly specify it:

  • what the task requires,
  • what is in the setup script,
  • what should be available through MCP,
  • which hosts the Agent can connect to,
  • which secrets it is not allowed to get.

This flow is less flashy than "The agent works for me in the cloud," but much closer to the truth. Asynchronous work begins before launching the Agent.

Before we move on to the third mode, let's pause at the question that comes up with each of them: why do all these restrictions even allow you to calmly step away from the computer?

Why constraints allow you to leave the desk

Sandbox can be annoying. The package didn’t download, the host is blocked, the local MCP configuration didn’t transfer to the cloud, a secret is missing, the setup takes too long. The first reaction? “This tool is getting in my way.”

And it is precisely these same limitations that are the reason you can safely step away from your desk in the first place.

An agent without filesystem and network isolation has a much larger attack surface. If something goes wrong, it can read files outside the project, attempt to send data outside the environment, or use accesses that the task does not require at all. In interactive work, you can more often stop such activity with permission or manual review. In work without constant supervision, you need boundaries set beforehand.

Therefore, isolation is not just a brake. It is a condition for autonomy.

The simplest rule is this: the less you watch the runtime, the more the environment must enforce the scope of work.

If the Agent has only the task repository, an allowlist of required hosts, short secrets, and a clear setup, you can allow it to operate longer without continuous approval. If it runs on your local machine with full access to the home directory and network, closing the laptop is no longer delegation but a blindly taken operational risk.

Let's look at a small example. Suppose you delegate the update of the test plan for the payment module after subscription limit changes. This is not a task for the production secret, access to Stripe, or a product decision. A good scope: read the diff and module documentation, add missing test scenarios, do not change the application code.

Before submitting such a task, consciously make a few decisions: goal, boundaries, setup, network, MCP, secrets, stop condition, and review criteria. This is not a separate file or tool format. Some of these decisions are set in the sandbox configuration (network, secrets, setup), and the rest you enter directly in the command for the Agent. For our example, such a command might look like this:

/goal Done when the payment module test plan covers subscription-limit scenarios.
Scope: edit only docs/test-plans/payments.md; read code from src/modules/payments/**, do not change it.
Do not change production code without separate approval, and do not merge.
Setup: `npm ci` in the preparation phase; dependencies are fetched during setup, not while working.
Network: disabled during the work phase; packages are downloaded earlier in setup.
MCP: only configuration visible to the sandbox (e.g. .mcp.json from the repo), no servers from your local profile.
Secrets: none; no production or write-enabled tokens.
Stop after updating the test plan and a short report of the gaps (limit ~20 turns).
Review (a green run is NOT success): the diff touches only the test plan, the report lists assumptions and uncertainties, and the Agent did not go beyond scope or ask for broader secrets.

Each line is a decision that you have to make anyway. The point is just to make it before starting, not in panic after the first failed run.

Now the phone has a very specific role. It is not used for reading the entire diff while waiting in line for coffee. It is used for quick checks:

  • whether the session has completed setup and did not get stuck on dependency installation,
  • whether a network lock, MCP, or secrets have not stopped the task,
  • whether the Agent is not trying to extend the scope to production code,
  • whether it is worth clarifying the command, stopping the process, or leaving it for review.

The full evaluation still takes place later, at the computer: diff, logs, tests, report, and criteria that you established before starting.

Forget about the "programming from the beach" image. The phone acts here as a control panel: it helps decide whether to continue, refine, stop, or leave the task for review.

In a good async workflow, you return to the Agent at selected moments instead of monitoring each of its steps.

Mode 3: loops and routines

The third mode begins when control is to return cyclically, not "after this one session".

In the lesson Innovate: more features, less waiting with multiple agents (M2L5), you learned about goal delegation and patterns like /goal. We will not be teaching them again here.

We are interested in another variant: a routine, a planned course, or a tool loop like /loop, that is, a mode of operation in which the Agent receives a repetitive task, triggered according to a schedule or event.

Examples:

  • every hour check open PRs and prepare a list of risks,
  • after each release, check the changelog,
  • once a day, review the overdue issues with the ai-candidate label,
  • periodically run self-check for the selected module and save the report.

These are not tasks that require sitting at the screen. They do, however, require a stronger contract.

This flow should have:

  • goal: what exactly should happen,
  • stop condition: when the Agent should stop,
  • scope limit: which files, repositories, or labels the work concerns,
  • cost limit: how many iterations, time, or budget you accept,
  • success criteria: what it means that the run actually delivered value,
  • failure criteria: when the green technical status is still a task failure.

It is safest to think of a routine as a small contract that ends with a review, rather than automatic trust:

The last point is the most important. A green run can only mean that the session ended without infrastructure errors. It does not mean that the recommendation makes sense, the PR is ready, the tests cover the risk, or the cost was justified.

In loops, risk rarely looks like a spectacular failure. More often it looks boring: An agent tries for two hours to download a package from a host blocked by network policy, creates a similar report a second time, fixes the symptom instead of the cause, or leaves a branch that no one can safely merge.

Therefore, the loop does not exempt anything from review. It only automates the moment when the work begins.

What exactly are you delegating for execution

Three modes tell you when you regain control. The question remains, what exactly do you hand over to the Agent to do without supervision. Most often, it is not a loose prompt, but a plan you already have.

In the course workflow, such a plan is created via /10x-plan, and so far you have run it interactively via /10x-implement: the Agent does the phase, you watch, approve, and move on. This is desk work. Asynchronous delegation needs an equivalent that does not wait for your click.

For this lesson, we provide exactly such an equivalent. Download it via npx @przeprogramowani/10x-cli@latest get m5l5:

/10x-goal-implement executes the same plan but without a human in the loop. You run it under /goal (interactively or headless via claude -p), and every decision you would normally make yourself is replaced by an explicit policy: The Agent implements the phase, runs it through quality gates (success criteria from the plan, deliberate-break check — that is, verifying that the new test actually protects something: the Agent temporarily breaks the production code and ensures the test fails red, then restores the code — along with the full test suite), commits on green in the Conventional Commits format, and proceeds to the next phase. When it encounters ambiguity or structural divergence from the plan, it stops with a STOP block instead of guessing.

This is the "configure before, review after" model in an executive version. Before starting, you set boundaries with a plan: file scope, success criteria, stop condition. During the process, you don’t need to watch because the gates ensure that a green run means something real. After the run, you get a report and a list of manual steps (#### Manual) for review.

That is why this skill wraps up the lesson topic: in a cloud sandbox or in a headless robot it is a definite "delegate and focus on something else." The plan acts like a runbook, /goal is the stop condition, and /10x-goal-implement is the policy that meets this condition without your constant presence.

The control did not disappear

Finally, let's gather the three modes into one image.

In local remote control, you still run the Agent, just from a different device.

In the cloud sandbox, you control the setup, network, MCP, secrets, and task scope before starting, and then receive the result in the review.

In loops and routines, you control the trigger, stop condition, budget, and success criteria.

This is a stronger model than "trust the agent". The agent can operate without your constant presence because you previously prepared a safe path for it.

At the module level, it forms a complete picture of teamwork with AI. First, we set the team standards, then our own agents and review pipelines, followed by shared AI artifacts. Now you are adding the final element: the ability to run this work remotely, asynchronously, and with a clear moment of accountability.

In practice, this boils down to a simple sequence: choose a control mode, configure the environment, delegate a limited task, return to review.

Details of cloud sandboxes

In the main part of the lesson, we deliberately did not create a complete table of specific product features. In preview version tools, such details can change faster than the course material.

The durable model is simpler:

  • setup script prepares the environment before starting the Agent,
  • network policy specifies what the Agent is allowed to communicate with,
  • MCP configuration determines which tools and servers are visible in the remote environment,
  • cache speeds up repetitive setup but can hide non-reproducibility,
  • secrets should be minimal and limited to the task,
  • tools available in the base image are convenient, but they cannot be an invisible assumption that you have not documented anywhere.

Claude Code Web and Codex Cloud are good examples of the same model. One can work by default with a trusted network allowlist, the other can split internet access during the setup and runtime phase of the Agent. In both cases, the mechanism teaches the same thing: installations and access to external hosts are planned before starting.

If you are preparing production material for the team, check the current supplier documents before providing the team with specific limits, tiers, lists of preinstalled tools, or cache lengths.

SSH, tmux, Remote Control and Happy

SSH with tmux is still a sensible point of reference. Especially when the team already has their own development machines, VPN, access standards, and doesn’t want to add another provider. tmux provides session persistence, and an SSH client on a phone allows you to return to the terminal from another location.

However, this solution does not provide automatic isolation. If the agent is running on your server with wide access to directories and the network, security depends on your configuration, not merely on using tmux.

Remote Control from Anthropic works exactly in this gap for Claude Code. The execution happens locally, but you take control from the Claude application or from claude.ai/code, without exposing SSH to the world and without configuring port forwarding. Unlike Happy, you don’t add an external intermediary because the traffic goes through the Anthropic API.

Happy takes a similar approach for teams for whom Remote Control is not suitable. It leaves execution local but facilitates control from a phone via an encrypted relay. This is more convenient than manually exposing SSH to the world, but unlike first-party Remote Control, it requires a trust decision regarding the tool and its intermediary model.

In practice, compare these options by asking: is the problem the control interface, or the place where the work is done? If it's only the interface, a local Agent plus remote control may be enough. If you want to isolate the local machine and work in a sandboxed environment, you need a sandbox or a VPS.

Cloudflare Agents and Workflows

Cloudflare Agents and Workflows are not a fourth variant of a "cloud coding sandbox" for the same table. Rather, it is a platform for teams that want to build their own persistent asynchronous environment for agents.

The difference is significant. Claude Code Web or Codex Cloud run the task in a managed environment. Cloudflare Agents provide the building blocks: a persistent agent identity, state, sessions, tools, schedules, and integration with Workflows for long processes, retries, waiting for events, or human approvals.

It's an interesting path when you're building your own agent system for a team: a bot performing regular audits, a support agent, an approval process, or a long background task with checkpoints.

The most important boundary: here you learn to delegate work to existing remote and asynchronous runtimes. Building your own runtime environment is a separate project.

Ralph, /goal, /loop and routines

Ralph is a historically important pattern: the agent operates in a loop, saves the state to disk, receives fresh context, and iterates until success criteria are met. This pattern helped name the way of thinking about work as "until you finish."

In the lesson Innovate: more features, less waiting with many agents (M2L5) you saw goal delegation and parallel work. Here we are interested in what happens when the loop becomes remote, scheduled, or cyclic.

Routines are a practical form of this approach, and treat /loop here as the name of a tool-dependent loop mode, not as a promise of a single stable syntax across all environments. Such a workflow can run a prompt or command periodically, respond to a trigger, or operate in a cloud session without your laptop. This is convenient, but only when it has a stop condition and review. Otherwise, it may result in uncontrolled resource consumption and suboptimal performance.
