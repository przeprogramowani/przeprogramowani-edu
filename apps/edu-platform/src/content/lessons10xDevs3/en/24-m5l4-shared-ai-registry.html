You have completed five weeks of the course. During this time, you have built a full toolkit for working with AI: skills that close repetitive tasks, prompts refined for your stack, rules that keep the Agent in check, and your own methods of guiding it through larger changes. This is no longer just individual interactions with AI. This is your workflow.

Since this all works for you, how do you transfer it to the rest of the team?

In the fourth module, we established that the real problem when working with AI is not duplication itself, but maintenance and synchronization. This is most strongly felt in a multi-repo system, where each project lives in isolation. After a month, no one knows which version of the skill is the current one.

The answer to these challenges is a single source of truth about how we work with AI, plus a distribution mechanism tailored to the needs of your environment. Such a mechanism can be built in three ways, and we will examine each of them in this lesson.

First, however, let's pause on the assumption that ties all three together: AI artifacts are code.

AI artifacts are code

Skills, rules, and commands are sometimes treated like notes: something kept in a notebook, pasted on Slack, or uploaded to the company wiki.

Meanwhile, they actually control how the Agent writes code and conducts reviews in your project. These are not margin notes, but instructions executed with every change.

And since the artifact is executed, its outdated or misaligned version is not an "old note." It is a silent bug that seeps directly into the system. That is why AI artifacts need the same things as any other code: versioning, change control, and controlled distribution.

Let's go back to the world before npm, pip, or maven. Back then, libraries were copied manually, file by file, between projects. It worked as long as there was only one project. Today, we are exactly at this point with AI artifacts: we copy them between repos and pretend it works.

Package registries, such as npm, PyPI, or Maven Central, solved this years ago: someone publishes a versioned package, you declare a dependency, and the package manager takes on all the tedious work. You know this from your daily work as a package consumer. This time, you will be on the publisher's side.

Two properties matter to us, which are easy to overlook in this familiar picture. The first is access control: a package can be public or private, only for your organization. The second is a single source of truth with a complete version history and control over who pulls what, which is exactly what was missing in the copy-paste approach.

However, the most important thing is that the registry doesn’t care what’s inside the package. The index.js script or the SKILL.md instruction for the Agent? For the registry, these are simply versioned files. Since AI artifacts are files, and we want to share, version, and update them, we can use proven package infrastructure for this purpose.

Let's clear up one doubt right away: you don't have to live in the JavaScript world to use this. We show the mechanism on npm because our toolkit is in that stack, but every ecosystem has its own registry, whether PyPI, Maven, or Cargo. And the artifact itself, even SKILL.md, is a plain text file in an open standard, independent of language or tool.

What AI Artifact Distribution Needs

Before we look at any solution, let's establish what such a mechanism must provide:

  • Single source of truth — one place from which the artifact originates. Without this, in six months there will be three versions of the skill circulating and no one will know which one is current.
  • Versioning — every change has its semantic label. Without this, you don't know whether the version in the repo you're working on is up to date, what has changed along the way, and how demanding the update will be.
  • Authentication — control over who can read and who can publish. Without this, either everything is public, or no one has access.
  • Installation, update, and uninstallation — the artifact can be pulled in, bumped, and cleanly removed. Without controlled uninstallation, after a year you will have remnants of five skill versions in your repo.
  • Multi-tool delivery — the same artifact goes to Claude Code, Cursor, or Codex. Without this, each tool in the team gets its own diverging copy.

These five points are our measure.

Repository as the Source of Truth

Models differ in infrastructure, but each of them stands on the same foundation.

The foundation is the division into two worlds. On one side, you have the repository of the source of truth: one repo where your AI artifacts live and from which they are published. You make changes here, in one place. On the other side, you have the consumer repositories: all the team's projects that pull and use these artifacts locally.

Between these worlds operates an installer: code that on the consumer side pulls in artifacts, places them where your tool reads them, and can later cleanly update or remove them. In the simplest model, the installer is a script run by the package manager after downloading the package from the registry. In subsequent models, this layer takes on increasingly more responsibilities.

What does the source of truth repository look like from the inside? In practice, it's simply one package, and the artifacts are arranged within it according to type:

ai-toolkit/
├── package.json          # package metadata + publishing configuration
├── skills/
│   ├── 10x-plan/         # SKILL.md: planning a change
│   └── 10x-research/     # SKILL.md: research in the codebase
├── rules/
│   └── CLAUDE.md         # rules appended to the consumer project
├── prompts/              # ready-made prompts for specific tasks
└── config-templates/     # tool settings templates

Each folder represents a single artifact category. And here you can go really broad: these are exactly the skills, prompts, and rules you've been collecting over the past five weeks. Adding another one means adding a file and bumping the version, and the rest of the team gets it on the next install.

The entire remainder of the lesson boils down to one question: how to best combine these two worlds in your environment. Three models offer three answers.

Three Distribution Models

These three methods differ primarily in the recipient and infrastructure.

The first is the registry you already have. If the team stores code on GitHub, it's easy to use GitHub Packages. The entire "setting up infrastructure" comes down to a single field in the package configuration. This is the default recommendation for most teams and where we will start.

The second is a registry as ready cloud infrastructure, for example AWS CodeArtifact set up with Terraform. This is a deliberate choice for an organization that needs advanced service management or uses its own registry like Nexus. In the case of AWS, the infrastructure is ready, but there is much more to configure here than in the first model.

The third is API and CLI. If you don't have a shared registry or recipients use multiple technology stacks, the registry ceases to be the simplest answer. In that case, a solution like 10x-cli makes sense, which we use in the course: the API manages access, and the CLI saves artifacts in your project bypassing the registry.

There are two things to clear up before the first model: how do we know this is the right direction, and what about ready-made marketplaces. The idea of distributing skills from a single source of truth using a package or CLI is not an exotic concept invented for the course. More and more developers are arriving at it independently. In June, Matt Pocock described exactly this recipe on X, which we break down below: an npm package with skills plus a postinstall script that links them to the appropriate tools. He summarized it briefly as a simple, versioned, and intuitive distribution of skills.

What about marketplaces?

One question might be on your mind. You’ve seen plugin marketplaces in Claude Code or Cursor and you think: why a registry when there’s a ready-made catalog? It’s a simple option requiring minimal work, so it’s worth taking seriously.

The Marketplace in Claude Code is a catalog in the git repository that the team links through the project settings so that new members receive an installation suggestion. For a team working on a single tool, this is a reasonable approach.

The catch is dependency on a single tool (vendor lock-in). The marketplace is tied to a specific ecosystem, and the turnover of AI tools today is simply too high to base the entire team's distribution on it. When you change the default tool or when a second tool appears in the team, the distribution channel starts to drift apart. The SKILL.md itself is portable, but its delivery through the marketplace is not.

If you work with a single tool, a marketplace can be a reasonable shortcut. If you want a portable standard for your team, let's return to the three models and go through each of them, starting with the simplest.

Model 1: GitHub Packages

If your team stores code on GitHub, you already have a package registry. It's called GitHub Packages and does not require configuring infrastructure.

The source of truth repository, consumer repositories, and the installer between them you already know from the previous section.

The entire "infrastructure setup" is a single field in package.json, in the source of truth repository:

{
  "name": "@your-team/ai-toolkit",
  "publishConfig": {
    "registry": "https://npm.pkg.github.com"
  }
}

This field contains the entire configuration on the publisher's side: publishConfig only specifies where to publish. The publishing authentication is added separately by CI, because the token is never committed to the repo.

On the consumer side, there is a second, equally short file. This is the .npmrc, which maps your scope to GitHub Packages so that only the @your-team/* scope uses the private registry, while everything else continues to use the public npm. This file is committed to the consumer's repo and contains only this one line, without any secret:

@your-team:registry=https://npm.pkg.github.com

The line with the token (//npm.pkg.github.com/:_authToken=${GH_PKG_TOKEN}) is not added to this file in the repo. It is added only during installation: from the environment variable in CI. So only the mapping remains in the repo, and the secret never enters the git history.

This is enough to move from configuration to a regular npm package pipeline.

In the source of truth repository, you package artifacts into a single npm package. CI on merge to the main branch publishes it to the registry. Each consumer repo installs it like a regular dependency. Changed a rule or refined a skill? You bump the version, merge, and the rest of the team gets the update with a single install. These are still the same two contexts: the change originates from the source of truth repo, and all consumer repos pull it in.

This is the path I actually recommend to most teams. This is exactly how we distribute our internal toolkit today: the source-of-truth repository from the previous section is one npm package with publishConfig, which you just saw, and the rest of the team pulls it like any other dependency.

The same set of artifacts can reach different recipients through various channels. Our team pulls the toolkit directly from the registry, exactly as in this model. You, as a course participant, receive the same artifacts differently, through the CLI. The registry is not inferior; it just stops fitting with a different type of recipient. We will explain this in the third model.

Where the only real difficulty hides

Since it's so simple, where's the catch? In one place: in authentication. And it's in a rather surprising asymmetry between writing and reading.

Saving is free and temporary. When CI publishes a package, it uses the GITHUB_TOKEN token, which GitHub Actions injects automatically for the duration of a single run. You just need to add permissions: packages: write in the workflow file. No secrets to manage.

Reading is more problematic. To pull a private package, you need a token that must be manually placed everywhere the package is installed: on the developer's machine, in the CI of each consumer repo, and sometimes on an external build platform (e.g., Cloudflare). This is not a one-time token, but a long-lived secret.

If you use the organization feature in GitHub, a consumer repo within the same organization can read the package with the ephemeral GITHUB_TOKEN, exactly as when writing. A persistent PAT only becomes unavoidable beyond the organization's boundary: in a repo owned by another owner or on an external CI that does not recognize this token.

And the costs? Public packages are completely free. Private ones fall within the limit included in your GitHub plan and only start costing after exceeding it. At that point, storage and transfer matter, but downloading in GitHub Actions via GITHUB_TOKEN is free. For most teams, this means zero cost. We link specific limits at the end of the lesson, as they are subject to updates.

What does this token handling look like in practice? Below is a lot of code, but you only need to remember one thing: the token never goes into the repo, and the rest is done by the installer for you. The whole trick starts with a single line that the installer adds to the consumer repo's package.json as a preinstall script:

[ -n "$GH_PKG_TOKEN" ] && echo '//npm.pkg.github.com/:_authToken=${GH_PKG_TOKEN}' >> .npmrc || true

This line appends an entry with the token to .npmrc only when the GH_PKG_TOKEN variable is set. In CI, the secret is injected, so the token is placed accordingly. Locally, the variable is usually not set: the developer logs in once via npm login, the token is stored in their ~/.npmrc, the condition is false, and || true ensures that the script does not block the installation.

It is more difficult on external build platforms, such as Cloudflare Pages or Workers, which do not have access to GitHub secrets. There, the installer adds a step syncing the token to the project via the platform's API, directly to the production and preview environments, because otherwise the preview build will fail due to the lack of a token.

In the production toolkit, this logic is not a README instruction but installer code. You don't need to read it line by line. Three properties matter: idempotency (run it twice, get the same result), no secrets in the repo, and gentle handling of the existing user's package.json. In short: it first appends the registry mapping to .npmrc, then attaches preinstall only if it's not already there:

const REGISTRY_LINE = "@your-team:registry=https://npm.pkg.github.com";
const PREINSTALL_SCRIPT =
  "[ -n \"$GH_PKG_TOKEN\" ] && echo '//npm.pkg.github.com/:_authToken=${GH_PKG_TOKEN}' >> .npmrc || true";

function ensureGitHubPackagesAuth(projectRoot: string) {
  ensureLine(path.join(projectRoot, ".npmrc"), REGISTRY_LINE);

  const pkg = readPackageJson(projectRoot);
  const existing = (pkg.scripts?.preinstall ?? "").trim();
  pkg.scripts ??= {};
  pkg.scripts.preinstall = existing.includes("GH_PKG_TOKEN")
    ? existing
    : existing
      ? `${existing}; ${PREINSTALL_SCRIPT}`
      : PREINSTALL_SCRIPT;
  writePackageJson(projectRoot, pkg);
}

This is a more important example than just a shell line: it shows that the distribution of AI artifacts is quickly becoming ordinary product engineering.

The same installer also patches workflows where it sees actions/setup-node and npm ci, adding registry-url, scope, and NODE_AUTH_TOKEN. Therefore, the consumer does not have to remember the entire checklist: the package itself prepares the repo for fetching its subsequent versions.

Effort does not disappear anywhere, it just changes form. In GitHub Packages, you do not pay for infrastructure, but you pay attention when injecting tokens into environments. In the model with your own registry, you pay earlier, with infrastructure configuration.

Versioning without manual bumping

Since publishing goes from CI on every merge, manually bumping the version quickly becomes a bottleneck. It can be automated, but with one safeguard.

We recommend versioning with conventional commits: the commit type (fix:, feat:, feat!:) indicates whether it is a patch, minor, or major. This works great as long as the types in commits are correct. In practice, they can be wrong, whether we commit manually or with AI assistance. Someone might label a typo fix in the documentation as feat: and suddenly a release with a new version comes out, even though no real change entered the package. Or conversely: a real skill change is tagged as chore: and no release happens at all.

Therefore, it is worth adding a second check in CI/CD: verifying with git diff whether the files included in the package have actually changed between the last tag and now. A commit can lie about its intention, git diff shows what has really changed. This check kills false releases from mistakenly tagged commits.

You don't have to write all this logic yourself. semantic-release calculates the version from the commit messages and publishes it right away. release-please also works from commits, but instead of publishing, it opens a PR with the release for approval. You can also write your own logic in CI/CD to have full control over the process.

⚠️ Model 1 — what to watch out for

A handful of pitfalls from our production deployment that will save you hours of debugging:

  • The token type for reading is a moving target (status as of mid-2026). Support for fine-grained tokens for the npm registry on GitHub Packages can be inconsistent, and classic PATs, which many teams still rely on, are being gradually deprecated. Before basing reading in CI on a specific token type, check the current status in GitHub documentation — this is the fastest-changing detail of this section.
  • External build platforms do not see GitHub secrets. CF Pages/Workers will only build once you synchronize the token to the project in their own secret management system. This is the synchronization step described above (on other platforms you perform a similar step yourself).
  • GitHub Packages rejects duplicate version (error class 409). Automatic versioning from the previous section makes you never think about it; manual version bumping will sooner or later generate a conflict.

Model 2: Registry as Ready Infrastructure

Model 2 is a ready-made, managed registry in the cloud, for us AWS CodeArtifact. Before I show how it works and when to choose it, a brief story about how I ended up on this path, because this is the one I built during the webinar promoting this edition of 10xDevs.

Why do I have this reflex with packages and CLI? From experience. As a platform engineer at SmartRecruiters, I was responsible for sharing working methods across the entire organization: libraries, frameworks, design systems. All of this was delivered to teams precisely through packages and CLI tools. When I looked at the skills and rules, I saw the same distribution problem, just with a different payload.

Before I chose GitHub Packages for the toolkit, I built a platform version: a private registry on AWS CodeArtifact, fully deployed with Terraform, with CI authentication via OIDC, meaning the exchange of short-lived identity tokens instead of a long-lived AWS key.

In exchange for maintaining the registry server, which is not present here, you get more control over the permissions and behavior of the registry itself than in the first model.

The most complex and impractical solution for most teams is setting up your own registry from scratch, such as Nexus or Verdaccio on your own cluster. Then you are responsible for the process, storage, and availability. CodeArtifact lies somewhere in the middle because the cloud takes care of all that machinery.

Similar to GitHub Packages, for most teams the cost will be zero thanks to the free tier of AWS CA. In contrast, running your own Verdaccio on ECS results in a real bill of several dozen dollars per month and significantly more code to maintain. Details of operation and billing can be found in the CodeArtifact documentation.

So why do we ultimately recommend GitHub Packages instead of CodeArtifact? We want you to start from the lowest entry threshold: a solution that will work for a team already using GitHub, without creating an AWS account or writing Terraform code. Both are valid, but best practice says: choose what requires the least effort while meeting the needs and fitting the environment.

I recorded a full walkthrough of this pipeline, step by step, in the webinar: CodeArtifact deployed with Terraform, OIDC, and CI/CD from start to finish. Treat this recording as a story of how I arrived at today’s recommendation, not as a path that everyone must follow.

So when does model 2 make sense? When your organization already uses AWS, or when you need management at the registry level itself, which GitHub does not provide you. It is a real, conscious choice for a specific context, not a default recommendation for everyone.

What this registry looks like inside

Let's start with terminology, because CodeArtifact uses the word "repository" differently than you're used to. For you, a repository is a place where you store code, like a repo on GitHub. In CodeArtifact, a repository is a package store inside a registry. The hierarchy looks like this: one domain (for us devs10x) ties together repositories with packages. In our configuration, there are two: one private for your own packages and another that is a proxy to the public npm. Thanks to this, the developer gets one endpoint for everything, both for company packages and those from the public registry.

Also watch out for two names that look similar but mean different things. The package scope (@10xdevs) is a prefix in the npm package name. The domain (devs10x) is the name of the resource in CodeArtifact. When logging in, you provide the domain, and the scope only determines which packages go through the private registry and which continue from the public npm. These are two different things that are easy to conflate in a single login command.

The rest of the configuration is declared in the pipeline. The registry resources alone amount to about 70 lines of Terraform, and the full configuration with permission policies and remote state grows to 250–400 lines. CI authentication goes through OIDC, so no long-lived AWS key lands in the repo, and access tokens expire after a few hours. The IAM role and OIDC provider are intentionally separated, one-time bootstraps, just like creating the bucket for Terraform state. This is not "manual work alongside the pipeline," but a clear boundary between what you manage using Terraform code and what you set up once at the start of the project.

The prompts and specifications I used during the webinar can be found in the materials for this lesson.

⚠️ Model 2 — what to watch out for

Complications worth knowing in advance:

  • The domain name must begin with a lowercase letter. 10xdevs will be rejected, hence devs10x in our configuration. Keep this in mind if the company or team name starts with a digit.
  • The AWS provider in version 5 weighs about 700 MB, so the first terraform plan tends to fail due to a timeout. Set TF_PLUGIN_TIMEOUT=300 before the first run.
  • The CI/CD pipeline must have permissions: id-token: write. Without this, OIDC fails silently and the publication cannot authenticate.

Each of these pitfalls can be seen in the webinar recording above, in the context of a real configuration.

Patterns That Will Survive Any Model

Some design decisions do not depend on where the registry is located. Regardless of whether you choose GitHub Packages, CodeArtifact, or something else, three patterns transfer between models.

Sentinel markers for rules. You put the skill into a separate folder so it doesn't interfere with anything. But project rules often reside in a file that a person edits in every project, for example in CLAUDE.md or AGENTS.md. How to inject team conventions into it without overwriting what the team has prepared? This is where the installer comes in, the same one that, with the first model, pulled in artifacts on the consumer side. Instead of overwriting the entire file, it appends its fragment between a pair of markers:

<!-- BEGIN @your-team/ai-toolkit -->
... team rules ...
<!-- END @your-team/ai-toolkit -->

The secret lies in idempotency. With each update, the installer finds this pair of markers, replaces only the block between them, and leaves the rest of the file untouched. Your own notes in the same CLAUDE.md will survive every install. You will see this in your own final project: open the file with the rules, find the 10x-cli markers, and add something outside of them. When pulling the package for the next lesson, the content between the markers will change, and your addition will remain intact.

In the simplest variant, such an update boils down to the operation: find the block, cut out the old middle, insert the new one. This is enough to make the installer predictable:

const BEGIN = "<!-- BEGIN @your-team/ai-toolkit -->";
const END = "<!-- END @your-team/ai-toolkit -->";

function applyRules(existing: string, teamRules: string) {
  const block = `${BEGIN}\n${teamRules.trim()}\n${END}`;
  const start = existing.indexOf(BEGIN);
  const end = existing.indexOf(END);

  if (start !== -1 && end !== -1) {
    return existing.slice(0, start) + block + existing.slice(end + END.length);
  }

  return existing.trimEnd() + "\n\n" + block + "\n";
}

If one tag exists and the other disappeared after manual editing, the production installer does not pretend that nothing happened. This is a case of a corrupted block that must be handled separately to avoid duplicating rules during the next installation.

Installation manifest. The installer writes to a small manifest file exactly what it installed: which package, which version, and for which tool, as well as a complete list of files broken down by individual skills. In your final project, this is .claude/.10x-toolkit-manifest.json:

{
  "package": "@przeprogramowani/10x-cli",
  "version": "1.2.0",
  "lessonId": "m1l5",
  "tool": "claude-code",
  "files": {
    "skills": {
      "10x-init": { "files": ["SKILL.md"] },
      "10x-shape": { "files": ["references/prd-schema.md", "SKILL.md"] }
    },
    "configs": ["settings.json.template", "mcp.json.template"]
  }
}

Thanks to this list, uninstallation is certain: the installer removes exactly the files it once added, instead of guessing based on the directory's contents. Importantly, the manifest is standalone. Cleaning does not depend on whether node_modules still exists, nor on package manager hooks, which do not always trigger when removing dependencies.

SKILL.md as a portable standard. Let's remember that this standard does not belong to any tool. It should work the same way in Claude Code, Cursor, and Codex. Therefore, the same artifact can be delivered to multiple tools at once: since the format is neutral, the tools only differ by the target directory. The installer (or, as you will see in the third model, tool profiles in the CLI) places the same file where the given tool looks for it.

The practical effect is that your work from the last five weeks is not tied to today’s choice of tool. The delivery channel changes, not the content. This is the same boundary you saw with marketplaces: the SKILL.md itself is portable, and only the distribution mechanism can lock you into a single ecosystem.

Model 3: Full product with API and CLI

The third model is a fully-fledged product with its own API and CLI. You reach for it when you want to become independent from the technology stack of your users. This is how the solution you use in this course works: 10x-cli, through which you have been downloading lesson materials since the beginning of the course.

Why aren't the course materials simply available as a package on npm?

  • Recipients sit on different stacks. A registry that has to serve multiple ecosystems at once is more complicated than a single CLI talking to a single API. The diversity of stacks is easier to manage with tool profiles in the CLI than by multiplying registries.
  • You want to distribute artifacts precisely. In the package model, the material is either published or not. With 10x-cli I decide what and for whom is available, as well as the order and portions in which it reaches the recipients. In the registry, such dosing is much more difficult.

These two things shift the distribution from package configuration towards the CLI.

Architecturally, this product consists of three interchangeable layers.

Source of truth is the place where the built artifact bundles reside. In the course, this is private content built from 10x-toolkit, but in a company, it can be any repository with artifacts.

API checks who is asking, whether they have the right to get a given resource now, and then delivers the appropriate bundle via REST. It is not just a protective layer. It is a distribution layer.

CLI saves the received content in the directory appropriate for the user's tool. Here begins a very practical part: a single API returns a neutral bundle, and the CLI decides whether the skill goes to .claude/skills, .cursor/skills, .agents/skills, or somewhere else.

In 10x-cli this decision is a simple tool profiles configuration:

export const PROFILES = {
  "claude-code": {
    skillPath: (name) => `.claude/skills/${name}/SKILL.md`,
    promptPath: (name) => `.claude/prompts/${name}.md`,
    rulesFile: "CLAUDE.md",
    manifestDir: ".claude",
  },
  cursor: {
    skillPath: (name) => `.cursor/skills/${name}/SKILL.md`,
    promptPath: (name) => `.cursor/prompts/${name}.md`,
    rulesFile: ".cursor/rules/10x-course.mdc",
    manifestDir: ".cursor",
  },
  codex: {
    skillPath: (name) => `.agents/skills/${name}/SKILL.md`,
    promptPath: (name) => `.agents/prompts/${name}.md`,
    rulesFile: "AGENTS.md",
    manifestDir: ".agents",
  },
};

And that is the whole point of the CLI as an "applicator": you don't have to learn the API conventions of each tool. The API manages access and content, while the CLI manages the local file semantics.

To make sure this doesn't remain an abstraction, let's take an inside look at the product you use in this course. One caveat: in this example, you are the consumer at the end of the chain, but this is the 10xDevs context, not the corporate one we discuss throughout the lesson. In your team, colleagues from other projects will be at the end of that chain.

API: access that the registry can’t handle

Login starts with a magic link, and beneath that the CLI receives a short-lived access token: a JWT valid for about an hour, refreshed in the background. A key property of the API: permissions are checked at each token refresh, not just once at login.

Translated to the team, this means you get access in one place, and the token expires no later than within an hour — without manually clearing tokens scattered across repositories.

How does the API know if you have the right to the content? The source of truth about permissions is your business system: employee database, membership system, subscription list. In 10xDevs, such a system is Circle.so, and the local store is KV in Cloudflare, because we don't want to query Circle with every content retrieval (Circle has monthly API request limits).

A practical pattern is as follows: extract permissions from the business system to a local store and check them locally, instead of querying that system with every request. Local lookup is fast and doesn't fail when the source system experiences a temporary outage. If in your company the permissions system does not have such limitations, which is common, a local store is unnecessary.

The API also provides something that a regular registry cannot: timed content dosing. A given module only unlocks when it is supposed to be available, and control over who gets what and when is held by the API. In a registry, a package is either published for everyone or does not exist at all; releasing in portions is much more complicated.

CLI as a security boundary

It’s easy to think of a CLI as something that simply writes files to disk. If a CLI is supposed to accept content from an API and modify the user’s workspace, it needs several safeguards. We will demonstrate them with the real 10x-cli code: this is our course delivery client, but your team’s equivalent looks the same, just under a different name.

Package signing (Ed25519). The API signs each bundle with an Ed25519 key, and the CLI verifies the signature before writing anything to disk. The public key is embedded in the CLI itself, so merely compromising the API is not enough to feed you forged content. Verification is a strict requirement (REQUIRE_SIGNATURES = true): a package without a valid signature is rejected, and nothing is written to disk. This protects against man-in-the-middle attacks on the supply chain of your AI artifacts.

In the shortened code, it doesn't look magical. The CLI hashes the raw response, compares it with the header, and only then checks the signature over the canonical string:

function verifyBundleSignature(rawBody, signature, keyId, headerHash) {
  const computedHash = sha256(rawBody);
  if (computedHash !== headerHash) {
    throw new Error("Bundle content hash mismatch");
  }

  const canonical = `v1:${keyId}:${computedHash}`;
  const valid = verifyEd25519(canonical, signature, publicKeyFor(keyId));
  if (!valid) {
    throw new Error("Bundle signature verification failed");
  }
}

First the integrity of the content, then the authenticity of the sender. Without this, the CLI would be just a convenient curl | write.

API host allowlist. The CLI communicates exclusively with its own API endpoint: the production host over HTTPS or localhost in development mode. If the API address could be arbitrarily changed by an environment variable, an attacker could redirect the login to their own server and capture the token. Therefore, a list of allowed hosts is necessary.

Guard for sentinel-injection. Remember the sentinel markers from the previous section? The tool appends them, surrounding the team rules block. If the rule content itself contains such markers, it is a warning signal: the next CLI usage could take the inserted marker as genuine and delete your content outside the block. Therefore, the CLI refuses to save a rule that itself carries sentinel markers.

Anatomy of the installer

The installer from models 1 and 2 is the package’s scripting layer: code responsible for installation, updating, and uninstallation of artifacts. It’s more than just “copy files”, even when everything fits within a single npm package. A few decisions worth examining:

  • Two installation modes. The symlink mode (npm install) links artifacts to the project’s .claude/ so they travel with the repo. The copy mode (npx ... install) physically copies files and works in any project, even without a package.json (Python, Go, Rust). Copy mode is mandatory here because the npx cache is ephemeral, so symlinks would break after the command finishes.
  • Manifest instead of guessing. The installer records the mode and the exact list of installed files to .claude/.10x-toolkit-manifest.json. Uninstallation reads the manifest and removes exactly what was added, without relying on node_modules still existing. Cleanup must be self-sufficient because uninstall hooks of the package manager cannot be relied upon in every dependency removal scenario.
  • Presets and dependencies. Skills declare requires in the frontmatter; the installer fetches missing dependencies and warns if something is missing. A preset that skips a skill with requires would silently break it, so filtering by preset still has to go through dependency resolution.
  • Migration of old versions. The installer cleans up artifacts from previous toolkit versions, but with a corrupted manifest, it prefers to leave the files alone rather than risk data loss.

Contract between private repo and public CLI

In model 3, the private repo with the content and the public CLI are two separate repositories with different visibility. They meet on an explicit contract: the API publishes a generated OpenAPI 3.1 specification, from which the CLI generates its types at build time. Thanks to this, a change on the API side that would break the client is revealed already during the CLI compilation, not at the user's side. This is a pattern for any two repos that can evolve independently but must not diverge.
