In the previous lesson, we deliberately stopped before refactoring.

You have the Deep Focus report: the message recording flow recreated step by step (②) and the fragility map with specific types of risk (③). You know where a change can silently corrupt the data, where there is no safety net, and where the risk is only apparent.

Exactly, so what now? You could enthusiastically try the prompt: "AI: refactor this module." However, often our reaction is paralysis. Refactor... but what exactly and how?

In this lesson, you add the element ④ Refactor opportunities to the report: a ranking of refactoring options grounded in the code and history, each of which you can defend before the team. Then you go through the moment when options turn into a decision, and the decision into a plan. One that protects you with tests before you touch anything.

Hotspots from report ③ (Technical debt) are problems, not tasks. Refactoring only begins once you give the problem a final form, anchor it in history, and plan a reversible path.

Let's start by figuring out what is not worth doing.

Antipattern: refactor without shape

"Refactor this module" is a prompt that can yield an impressive result. The agent will gladly rewrite half the layer, the diff will span forty files, names will be nicer, functions shorter. It looks like progress.

The problem is that this diff is missing three things.

There is no goal: no one said what shape it should be in the end, so the agent chose a shape that "usually fits." There is no history: neither you nor the agent knows which quirks of this code are accidental and which are deliberate decisions that the whole system is based on. And there is no way back: you can't verify forty files at once bit by bit or roll back without pain.

The other extreme is just as costly. You know that the module "should be better," so you reach for the pattern catalog and modernize everything that doesn't run away. Layers grow, abstractions multiply, and the value... well, where is the value?

The conclusion from both extremes is the same: the agent is not immediately tasked with refactoring. The agent is assigned to present options, and the refactor begins with the decision you make. For this decision, you need three perspectives: the target shape, the history, and the reversible path. We will take them one by one.

Target module shapes

Spectrum of archetypes

The question "what to modernize to?" has a surprisingly old, practical answer: a catalog of archetypes of business logic organization that the industry has used for more than two decades (you can find more information in Deep Dive).

  • Transaction Script — logic organized into procedures; each procedure handles one request from start to finish.
  • Table Module — one instance handles the logic for all table rows or the view.
  • Domain Model — an object-oriented domain model where data and behavior live together; the application's operations are then organized by the Service Layer, a thin layer that defines available operations and coordinates the response to each of them.

This is a spectrum, not a ranking. None of these shapes is "better" – each has a break-even point depending on the amount and variability of logic.

Why do you need this spectrum? Because it turns "this code is ugly" into a thesis that can be defended or refuted.

Take a look at our material from Mattermost. SaveMultiple in the store layer is 180 lines: a validation loop, counter accumulation, a transaction with a batch INSERT, and fixes after commit, all inline – while next to it, four responsibilities are already separated into named helpers. The file app/post.go has 3957 lines and 60 exported methods from several different domains.

A thesis you can propose: an overgrown Transaction Script whose logic needs a domain model behind the service layer. This is a sentence of a different kind than "let's rewrite this". It points in a direction, allows for disagreement, and does not yet determine whether it is even worth it.

Because the spectrum also works the other way: a small domain module with a bit of logic has every right to remain a Transaction Script. We'll return to this shortly.

Story as an intention test

Before you suggest changing the shape, you must answer the question that distinguishes refactoring from ruining someone else’s work: is what looks like trash actually a conscious decision?

It helps here to distinguish between two types of complexity.

Essential arises from the nature of the problem - you can't remove it with any refactor, you can only arrange it better. Accidental arises from how the problem was originally coded - and that's what refactoring actually removes. The catch is that often, from the code alone, it’s not clear which is which.

The answer lies in the decision history. First, you check the ADRs (architecture decision records): short notes on "what we decided, in what context, and with what consequences." If the team maintains them, you read them before you "fix" anything.

Mattermost does not maintain ADRs. And this is the most common case, so it's good that it appeared in the demo: what remains is git archaeology. git log -L on the function, blame, justifications in commit and PR descriptions.

Let's take a candidate who in report ③ (Technical debt) looked like textbook accidental complexity: positional column arrays when saving a post. Eighteen columns whose order must match in two functions at once, positional writing, reading by names, no safeguards. You just want to ask "who wrote this?!"... right?

Git archaeology tells a different story. The commit 27d536b212 from March 2020 intentionally introduced this form: it replaced the reflective writing of individual rows with a positional, batch INSERT to stop the mass import of messages from being a bottleneck. And in the file's history, there isn't a single commit fixing a column order mistake. Six years, zero slip-ups.

The verdict is reversed: this is not accidental complexity, but a deliberate limitation. A crucial decision on which import performance depends.

Guard, not rebuild

Alright, but the risk hasn't disappeared. Nothing enforces the consistency of these arrays, and swapping two adjacent columns of the same type would quietly pass through the entire test suite.

And here comes the rule worth remembering for all legacy work: guard, not rebuild. The shape does not change with conscious limitation. You add a cheap, deterministic shield that turns "easy to forget" into "impossible to break." In our case: a unit test checking the consistency of both arrays. A pure addition, reversible with a single revert, and the performance decision from 2020 remains untouched.

This is the adjustment of the refactoring scale to the problem (right-sizing). This filter completes the first perspective:

  • a small module with simple logic remains a Transaction Script,
  • conscious limitation gets a guard,
  • you reserve refactoring for accidental complexity, whose cost actually increases.

If you skip this filter - you end up creating art for art's sake: patterns everywhere, value nowhere.

Reversible path to the goal

Let's assume a candidate goes through both analysis paths: the history is checked, the type of complexity identified, and the verdict is "random, worth pursuing." How to reach the target shape without system failure?

Two strategies cover most cases:

Strangler Fig: new code grows on the edges of the old. Small fragments of behavior are transferred to the new structure, the old system is gradually phased out until it can be removed. The most important reason to choose this path instead of rewriting from scratch is lower risk: at any moment you have a functioning whole.

Branch by Abstraction: a major change without a long-lasting, locked branch. You insert an abstraction over the point of interaction with the old code and continue step by step, while the main branch remains releasable throughout the duration of the change:

The issue of the order of steps remains. With a big change, everything depends on everything – where to start? The practical mechanics look like this: try the target change, see what stops working, write it down, and then undo the change and solve the problems from the farthest leaf to the root of the change. The "big change" becomes a graph of small steps, each of which is verifiable and individually reversible. In the literature, this method has a name and its own graph – you will find them in Deep Dive.

And the last piece of the puzzle, a safety net for external code. A characterization test captures the current behavior – not necessarily correct! – so that a structural change doesn’t silently alter it. It’s worth adding it in a place where you can observe behavior without editing the code you are protecting. The rule is: add the test before you touch. Here returns a warning about the oracle problem from the lesson on the test plan (M3L1) – a characterization test preserves the existing state, so you consciously decide that this fragment of behavior is worth preserving.

We have a complete set of perspectives on potential refactoring: shape, history, and path. It's time to hand these three lenses over to the agent. On our terms.

Element ④ (Refactor opportunities): exploration and ranking of options

All of the above could be done manually... but who has time for that in someone else's large-scale repo? This is exactly the kind of work that AI can do for us: read every recorded issue, analyze its current state and history, assess the feasibility of migration – and give you a ranking of options.

We return to the familiar cycle: /10x-new opens a change, /10x-research explores. The novelty lies in the contract you feed the research with, and in one principle that ties the whole lesson together: exploration ends with a report, not a decision.

Does history really change decisions? That's exactly what you just saw. Without the perspective of intentionality, C2 would be at the top, C4 at the bottom – and we would spend many hours rebuilding something that has worked for six years, instead of closing a hole that already lets records slip past indexing today.

Read before you decide

The exploration is over, the report is in the changes folder... and this is the moment when it's easy to ruin the entire work with one click.

The ranking looks like a final decision. It is not. It is a proposal—and it is no coincidence that the contract forbade the agent from ending with the question "do you approve?". Asking a question immediately after a fresh result forces an answer under pressure, without review. You know this mechanism from the prework [1.3]: generate, then understand. A quickly closed task is not the same as a decision you can defend.

So read research.md carefully. Pay attention to three things:

  • candidate list - whether all issues from the previous Deep Focus report are included and their classification candidate/non-candidate holds up,
  • intentionality verdicts - does each have evidence (a specific commit, ADR, PR), and not just a hunch,
  • ranking - do you agree with the proposed order and arguments.

There is also a fourth move - you know it from the previous lesson (M4L3). The ranking is based on structural assertions: the number of methods in the interface, "overrides X but not Y", the number of call-sites. This is exactly the category where numbers that look certain often turn out to be approximations - and your decision will soon depend on these numbers. Before moving on to planning, confirm them with the same ast-grep investigation you used to verify report ② (Feature overview) and ③ (Technical debt):

Decision Gate in /10x-plan

The final decision on what we will focus on is made in /10x-plan. You already know the planner mechanics from module 2. This time we will use it to make a decision about refactoring.

/10x-plan refactor-opportunities

The planner reads research.md from the change folder automatically, so we don't have to answer diagnostic questions - the entire question budget goes toward design decisions. In our run, there were seven questions, including one that the planner did not have in the scenario - we will return to that shortly. The first one was exactly as it should be: which option are we implementing? The contract clause applied: the ranking is a proposal, and the decision is made at the planning stage. Thanks to this, accepting the ranking became an explicit question, not a silent assumption.

Example choice: C4 plus test-guard from C2 (the cheap, quick gain "regardless of ranking"). Subsequent questions revolved around trade-offs: how to enforce completeness (validation in the layer generator, under the existing blocking CI gate, instead of new infrastructure), how to record exceptions (an explicit list with a mandatory reason per method), what the pilot scope should be, and what the rollout should look like.

And then something happened that is worth remembering as a verification pattern. The planner, in its research phase, checked an open question from the ranking: is the SaveMultiple loophole a live bug? And it came back with solid evidence: the only callers are three batch import locations, nothing automatically reindexes after import, and the internal delegation Save→SaveMultiple is below the search layer, so overriding will not duplicate anything. Imported posts are unsearchable until the admin manually triggers reindexing. And this new evidence was not quietly absorbed into the plan. It was here that the announced off-script question was posed — the seventh, forced by new evidence: do we want to address this?

Along the way, for the third time in this process, the test design for linked arrays was refined. This thread is worth highlighting upfront, as it runs through all stages of the work:

  1. The "round-trip" test (save → read → compare) sounds natural but is worthless: reading is done by names and is forgiving, saving is positional and is not forgiving. Such a test passes with swapped columns.
  2. Sketch from the research (comparison of types per index) is also fragile: four out of eighteen columns do not pass naive type comparison (three serialize to JSON, one is a pointer), and swapping adjacent columns of the same type passes without a trace.
  3. Final project from planning: sentinel-values - a characteristic value in each field and an identity assertion per column name.

Each stage caught what the previous one let through. The closer to implementation, the sharper the eye - provided we can rely on a solid analysis from the previous stages.

It is worth focusing on two properties of the plan, because they carry the entire philosophy of the lesson:

  • Add the test before you touch. Phase 2 begins with a test characterizing the existing behavior of Save→indexPost (this path currently has no coverage) and only then adds the override for SaveMultiple. Test first, edit later. Not "side by side".
  • The mechanism lands on green, enforcement is enabled separately. The plan also doesn’t include Strangler or Branch by Abstraction—and this too is matching the action to the problem. The chosen candidate is a guard and patching the hole, not migrating the structure, so both strategies are waiting for candidates like C3. They did their job anyway: they were the perspective through which the research assessed the feasibility of each option.

  • each phase has verification criteria - automatic and manual,
  • mechanisms turn green, and enforcement is explicitly enabled in a separate step.

After your own review, verify the plan with /10x-plan-review in a new session, just like in the "Plan handover" section - a fresh context will catch what both you and the planner missed.

From a Single Step to a Campaign

The plan from this lesson completes several selected refactoring candidates. But we still have a broader backlog: the contract drift gate (C1), covering error paths, extending verified guards to additional sub-stores. If you see weeks of such work ahead of you, rather than just one step, it's worth using /10x-test-plan, which you know from the lesson on test planning (M3L1).

In our usage on Mattermost, fueled by the decision (plan + both researches), it did two noteworthy things.

First, it did not duplicate work. It recognized that the existing plan already delivers the goal of the first rollout phase, and adopted it as phase 1, instead of opening a parallel change folder. Two skills merged into one process because both read the same artifacts from context/.

Secondly, they managed the rest of the backlog. The runner-ups from the ranking were arranged into subsequent rollout phases: the contract drift gate Post as phase 2, error path coverage as phase 3, extension of verified guards to additional sub-stores as phase 4 – each with its own change folder and a known cycle of research → plan → implementation.

When is it worth using /10x-test-plan during refactoring? When the backlog has several worthwhile candidates, the complexity of the work is assessed as high, and the completion time is measured in weeks. When not: a single narrow slice, as in this lesson - then the plan already provides enough safety margin.

Where do the techniques used in the lesson come from?

  • Business Logic Archetypes — Martin Fowler, Patterns of Enterprise Application Architecture (2002). Transaction Script, Table Module, Domain Model, and Service Layer come directly from the pattern catalog, along with the trade-off table on which our right-sizing is based.
  • Essential and accidental complexity — Frederick Brooks, "No Silver Bullet" (1986). The original terms are essence and accidents; "essential/accidental complexity" is the industry-adopted, popularized phrase.
  • Strangler Fig — Martin Fowler, 2004 (originally as "Strangler Application", renamed in 2019). The strongest verified argument: lower risk compared to a cut-over rewrite.
  • Branch by Abstraction — a term coined by Paul Hammant (2007), popularized by Fowler. Hammant clarifies that he did not invent the practice, and the name itself is credited to Stacy Curl.
  • The Mikado Method — Ola Ellnestam and Daniel Brolund, The Mikado Method (Manning, 2014). The prerequisite graph (Mikado Graph) is exactly the "prerequisite sketch" required by element ④ (Refactor opportunities); the "try, undo, solve from the leaves" loop is the mechanic we used when arranging the order of steps.
  • Seams and characterization tests — Michael Feathers, Working Effectively with Legacy Code (2004). The technique of adding new behavior in fresh, tested units instead of editing uncovered code also comes from there.
  • ADRs — Michael Nygard, "Documenting Architecture Decisions" (2011). Format: Title, Context, Decision, Status, Consequences. Reading ADRs for "non-goals" is our practice of interpreting the context and consequences sections.

A broader perspective on refactoring

When selecting candidates for refactoring, it is worth considering two more perspectives:

  1. Technical Debt Quadrant (Fowler; the debt metaphor itself originates from Ward Cunningham) classifies debt along the axes conscious/unconscious × deliberate/reckless, and the capital and interest metaphor suggests the order of repayment: first debt with the highest interest, i.e. the one that most slows down current changes. Our ranking did this implicitly (cost of debt vs cost of change); the quadrant provides dedicated concepts for this.

  2. The classic Joel Spolsky essay "Things You Should Never Do" (2000) on why rewriting a system from scratch is a strategic mistake. The text was written long before AI, and that is precisely why it is worth revisiting today: the cheaper it becomes to generate new code, the cheaper a rewrite seems – but the costs Spolsky writes about (losing knowledge embedded in patches) have not decreased at all.
