AI-powered bootstrap

In the lesson From Chatbot to Agent (M1L2), the tech-stack.md landed on the disk. Together with prd.md, you now have two artifacts: a business contract and a technical contract.

Only that the project does not exist yet.

This is the first moment in 10xDevs where the agent stops just talking. In a moment, it will invoke the CLI, create directories, and fetch dependencies.

And here comes the question that everyone who gives an agent access to the disk for the first time knows: how far should I let it go?

Too little trust and you drown in clicking "Yes" at every step. Too much, and after a week you discover that the agent had access to things it shouldn't have.

There are alternative methods, and this is precisely what we address in this lesson.

What you already have on disk

The /10x-shape → /10x-prd → /10x-tech-stack-selector → /10x-bootstrapper chain operates on context files in the /context folder. Each link reads files and writes new artifacts to the disk.

In this lesson, this flow is completed: tech-stack.md will be used by /10x-bootstrapper, and as output, it will create a project skeleton and a verification.md report.

Do you have an existing project? You will use the /10x-health-check skill, which will check the state of the repository. We will discuss this path later in the lesson.

Running the bootstrapper

You type in the terminal:

/10x-bootstrapper @tech-stack.md

The skill starts with one verification: does context/foundation/tech-stack.md exist on the disk?

If it does not exist, the skill refuses to execute and redirects you to /10x-tech-stack-selector. It does not try to extract the stack from the conversation history, nor does it ask "so which stack?" in an ad-hoc dialog. It does not improvise.

It sounds trivial until you see how many sessions get into trouble because "yesterday we agreed on Astro," but today the agent has a clean context and honestly doesn't know what you're talking about.

Hand-off as a prerequisite is the first of three gates you will learn about in this lesson. The pre-execution gate: before the agent executes anything, it checks if it has the necessary foundations to do so.

Here: whether there is an input file. In other skills, this can be, for example: whether the tests pass, whether there are no uncommitted changes. The rule is the same: without permission, the agent does not move.

Why the agent does not write the project on its own

When tech-stack.md is on the disk, the bootstrapper reads starter_id from it, searches for this starter in the internal registry, and executes the command from the cmd_template field. For a JS/TS project, this might be, for example:

npm create astro@latest -- --template minimal

Why does the agent not try to write this project from scratch at all but uses the starter CLI?

Because it is exactly the same job that the Astro team codified in their official creator. npm create astro@latest will generate a correct package.json, correct astro.config.mjs, correct tsconfig.json, and a directory structure tailored to Astro 6.x. It knows the conventions, knows the default integrations, and knows what is current and what is not.

If the agent tried to replicate it from memory, it would deliver a half-working boilerplate. package.json with "astro": "^4.0.0", astro.config.mjs with syntax from two versions ago – because such combinations most often appear in the training data. You will spend two days patching what npm create astro would do in thirty seconds.

Hence a universal rule:

If there is a CLI that can accomplish the task, the agent should delegate to it rather than acting based on training data.

The Bootstrapper is one instance of this rule. But the same logic applies everywhere:

  • Do you want to add a Dockerfile? First docker init (there is such a command from Docker), and the agent will add what the CLI does not cover.
  • Do you want a new component generator? First ng generate component in Angular or bun create vite with a template, not "write me a React component from scratch".
  • Do you want a new Prisma migration? npx prisma migrate dev --name add_users, no generating SQL from scratch.

The model behavior you know from prework (confidence despite incomplete knowledge) is most costly where there is a tool nearby with an authoritative answer. So it is always worth considering: is there a CLI to which I can delegate this?

Three execution gates

Bootstrapper is a specific skill, but the pattern it uses is universal. Every execution of the agent - bootstrap, health-check, refactor, migration generation - goes through three gates:

  1. Pre-execution. Before the agent does anything on the disk or in the network, it checks whether it has the mandate to do so. In the bootstrapper, this is the verification of the hand-off (tech-stack.md on disk) and the recency check: whether the starter has not been abandoned.
  2. In-execution. During execution, the interaction takes place between the agent and the harness, which is the tool the agent operates in (Claude Code, Cursor, Codex). The harness asks for your permission or executes the command without asking, depending on the policy.
  3. Post-execution. The skill audits the result and saves the report. In the bootstrapper, this is npm audit plus verification.md with the status of each phase.

Each gate has a different owner. Pre-execution controls the skill contract. In-execution is the harness and your permission policy. Post-execution involves external tools: npm audit, tests, CI.

Three questions for each skill: what does it check before it starts? What does the harness want from me during the process? What will I get as output for verification? In this lesson, we will focus most on the in-execution gate, because your harness configuration has an impact on it.

Permissions during work

After the hand-off, Skill will start working with your repo. And this is where the harness comes into play. Let's see how it looks in practice:

In Claude Code, during work you will see requests for permissions on the screen:

Claude wants to run a command:
  npm create astro@latest

[1] Yes
[2] Yes, don't ask again for npm commands in this directory
[3] No, and tell Claude what to do differently

Each such request is a moment when the harness asks: do you accept this? By default, it asks about everything that touches the disk or network. Without asking, it only lets through purely reading commands (ls, cat, grep, git status).

Three decisions you have to make with every prompt:

  • Yes — single consent, one-time. The next session will ask you the same.
  • Yes, don't ask again — the harness adds a rule to .claude/settings.json in the project and stops asking about this category.
  • No — you reject; the agent receives feedback and must solve the problem differently.

Are you clicking "Yes" every time? By the evening, you will have spent two hours confirming routine prompts. Are you clicking everything "Yes, don't ask again" without thinking? After a week, the agent may execute any bash command in the project directory.

How to decide? A filter with every prompt: what can this pattern break?

"Nothing" (e.g., npm install in the project directory) - feel free to add the rule "Yes, don't ask again".

"Everything that git push can send to the remote" - you leave individual consent.

"Potentially the entire disk" (e.g., Bash(*)) - you do not agree with such a rule.

Rules in Claude Code are evaluated in the order of deny → ask → allow. The first match wins. *Bash(git push ) on the deny list has priority over *Bash(git add ) on the allow list, regardless of the entry order.

The built-in harness tools (Read, Edit, Write) and bash commands are different worlds. The rule Read(./.env) in deny will block reading .env from the agent level, but it will not stop cat .env in bash, because that is a different execution path.

The policy filters tool calls by name and arguments, not by effect.

It is important to note that the permission policy for the agent is not absolute.

In other tools, the pattern is similar, the syntax is different:

  • In Codex: --sandbox workspace-write --ask-for-approval on-request. Configuration in ~/.codex/config.toml.
  • In Cursor: allowlist in ~/.cursor/permissions.json or in the settings UI.

Links to the documentation can be found in the Additional Materials. The syntax differs, the way it works remains the same.

Recommended starter configuration

Instead of learning the policy dryly, start with a specific file. After the first bootstrap, open .claude/settings.json in your project and paste this configuration:

{
  "permissions": {
    "allow": [
      "Bash(npm *)",
      "Bash(npx *)",
      "Bash(node *)",
      "Bash(git add *)",
      "Bash(git commit *)",
      "Bash(git diff *)",
      "Bash(git log *)",
      "Bash(git status *)",
      "Bash(git branch *)",
      "Bash(git checkout *)",
      "Bash(git stash *)",
      "Read",
      "Edit",
      "Write"
    ],
    "ask": [
      "Bash(curl *)",
      "Bash(wget *)",
      "Bash(git push *)",
      "Bash(git push)"
    ],
    "deny": [
      "Bash(rm -rf *)",
    ]
  }
}

Why these particular rules?

Allow: local operations within the project:

  • Bash(npm *), Bash(npx *), *Bash(node ) — dependency installation, running scripts, tools from the npm registry. The only output outside the project is the network layer (npm registry), but you need that anyway to work.
  • Bash(git add *), Bash(git commit *), *Bash(git diff ) and other local git operations — read and write in .git/, but do not publish anything to remote. We separate them instead of writing a broad Bash(git *) so that the agent does not get implicit permission for git push, git reset --hard, or git clean -fd.
  • Read, Edit, Write — built-in Claude Code tools for file operations. They operate within the working directory.

Ask: operations the agent will ask about before execution:

  • Bash(curl *), *Bash(wget ) — the agent should not independently download and execute scripts from the internet without your knowledge. A single curl to the documentation? You will see the prompt and decide.
  • *Bash(git push ) and Bash(git push) — push to publication decision. Two patterns, because plain git push (without arguments) also publishes to remote. The agent will ask you each time, instead of publishing blindly.

Deny: operations unconditionally blocked:

  • *Bash(rm -rf ) — can destroy the entire home directory if the agent substitutes the argument incorrectly. You don't even want to see the prompt; you want it to never go through.

How does it change over time?

This list is a starting point, not a goal. After a week of work, you will notice patterns: if you approve Bash(docker compose *) every session, add it to allow. If it turns out that *Bash(npx ) ran a command you didn’t want, move it to deny or break it down into narrower patterns.

Initial configuration → mature configuration. Allow grows with patterns that have passed your filter. Deny grows with patterns that surprised you. Rules that do not match any of these lists remain in ask. And that is their proper place.

Permission policy and YOLO mode

The configuration from the previous section takes just a few minutes of work. It gives you an intermediate state: the routine proceeds without asking, destruction is blocked, and the rest requires your approval.

On the other end of the scale sits YOLO mode, in Claude Code formally bypassPermissions (flag --dangerously-skip-permissions). Everything goes through without asking.

The natural reflex after two hours of clicking "Yes": I turn on YOLO and keep going. Anthropic knows this reflex and explicitly states in the documentation (paraphrase):

Use this mode only in isolated environments: containers, virtual machines, dev containers without network access. Where Claude Code cannot damage your host system.

Dev container in practice

What does this actually mean? A dev container is a container in which you run Claude Code instead of on your own system. The agent only sees what you have mounted: your repo, selected dependencies, nothing more. It does not have access to ~/.ssh, cloud keys, or production databases.

In practice, it comes down to three conditions: the agent runs as a non-root user, has limited network traffic, and only sees the trusted repository with the necessary files. The dev container protects the host system but does not exempt you from project control because the files in the mounted repository still end up on your disk.

If you ever conclude that YOLO mode meets your boundary conditions, start with the reference devcontainer.json from the Claude Code documentation. You can find the link in the Additional Materials.

In such conditions, --dangerously-skip-permissions is a conscious choice. Without it, you are delegating your own system permissions to a model whose reasoning is not fully accessible to you.

Permission Scale

So you have two ends of the scale. On one end, permission paralysis: a default policy, every prompt confirmed manually, two hours a day clicking "Yes". On the other end, full delegation: YOLO mode without boundary conditions, the agent does whatever it wants.

Between them lies the intermediate state. And this is the course recommendation for this stage.

An intermediate state is an operating mode in which:

  • you have several deliberately chosen allow rules (e.g., Bash(npm *), Read, Write), so Claude Code does not ask about the routine,
  • you have several deny rules for dangerous patterns (e.g., Bash(rm -rf *), Bash(git push *)), which it blocks without asking,
  • everything except for these two lists still requires your consent.

In practice: the bootstrapper runs npm create astro@latest without asking (because *Bash(npm ) is on the allowlist), but when it wants to run an unclassified command, it stops and waits. You are not drowning in prompts, but you don’t lose control over what is not obvious.

The key difference compared to YOLO: in the intermediate state, Claude Code still asks you about operations you did not anticipate. It is a safety net for behaviors outside your rules. In YOLO mode, this net is absent.

Claude Code also offers an auto mode (research preview), which automatically reduces the number of prompts without disabling safety controls, e.g. explicitly blocking Bash(*) and wildcarded interpreters on input. It is worth monitoring its development as a potential alternative to manually shaping the allowlist.

It is worth remembering, however, that:

The permission policy is a probabilistic control, not an absolute one. It raises the cost of a mistake (the agent must stop, you have the opportunity to notice), but it does not eliminate it.

There are documented cases where Claude Code, having the npx command blocked on a denylist, accessed the same binary file through the path /proc/self/root/usr/bin/npx, bypassing pattern matching.

In another scenario, the model suggested: the bubblewrap sandbox is failing... let me try disabling the sandbox, then asked the user to run the command without the sandbox.

These are isolated, documented cases, not everyday occurrences. However, they are enough to correct the mental model: the permissions policy is not a lock on the door. It is a threshold on the floor – it will slow down and alert, but it won't stop every time.

Hence the course recommendation for this lesson: policy as a starting point, YOLO as a conscious decision with boundary conditions, defense in depth as a principle. Stick to these three until you develop the instinct for when it is safe to loosen any of them.

verification.md as the bootstrapper report

The skill has finished working: the scaffold has landed on the disk, .git/ was skipped (bootstrapper v1 does not run git init; it is your decision when to initialize the history).

A context/changes/bootstrap-verification/verification.md appeared in the directory.

What is there?

This is the third gate, post-execution. The agent gave you a report of its work in a file format, not conversational. You will return to this file in a week and still know what happened then.

In this case, we have three verification phases. Each has a status: passed, warned, failed, skipped. Each briefly explains why.

  • Phase 1 — pre-scaffold verification. Is the starter (e.g., astro@latest) not abandoned? Fresh? passed. Last publish two years ago? warned, the skill does not stop, but flags a signal in the report.
  • Phase 2 — scaffold and merge. Running npm create astro@latest. It ended with exit code 0 (passed) or non-zero (failed, HARD-STOP). About "HARD-STOP" in a moment.
  • Phase 3 — post-scaffold verification. npm audit (for JS/TS projects). Five severity levels: info, low, moderate, high, critical. A non-zero exit code does not stop the bootstrapper; this information provides a signal but does not block.

warned means that you know where to start the investigation. failed means that the scaffold did not complete and intervention is required.

Health-check: audit of an existing project

Do you have an existing project? Do not run the bootstrapper that was designed for greenfield projects.

In the case of brownfield, /10x-health-check is the proper entry point. It takes an existing project and runs it through the same analogous analytical gates:

  1. Pre-check — dependency audit (npm audit, pip-audit, cargo audit, depending on the ecosystem), presence of a lockfile, known vulnerabilities. Before the agent starts working with the project, it checks the state of the foundation.
  2. In-check — read-only analysis. Is there a test runner and do the tests run? Is there a CI/CD pipeline? Are configurations missing (tsconfig.json with strict, .prettierrc, .editorconfig)?
  3. Post-check — agent-readiness verdict: healthy, needs-attention, or critical-issues. Plus a prioritized list of fixes in two categories: fixes to be done now (Category A: missing test runner, critical vulnerabilities, missing lockfile) and fixes discussed in subsequent lessons (Category B: missing CI/CD is the topic of Sprint Zero lesson (M1L5), missing AGENTS.md is the topic of Agent Onboarding lesson (M1L4)).

If in the previous lesson (M1L2) you ran /10x-stack-assess, health-check reads stack-assessment.md and links its findings to the detected vulnerabilities.

Example: stack-assess identified that your stack does not meet the "typed" gate, and the health-check confirms that the CI does not have a type-check step.

Two reports together provide a consistent picture: where the stack has gaps and where the project infrastructure does not compensate for those gaps.

Result: context/foundation/health-check.md, a report file that serves the same purpose as verification.md in greenfield: it provides the subsequent links in the chain with a file-based context about the project status before agent onboarding (M1L4).

Both paths converge in the Agent Onboarding lesson (M1L4) with an equivalent context.

Four safe modes

We discussed security patterns during the lesson. But what about anti-patterns? Four modes that students most often fall into after the first week with the agent.

Approve-everything Stockholm. You manually approve every permission request because you don't know what is safe. After two days, the workflow becomes a test of endurance, not productivity, and you abandon the agent as "unmaintainable."

Antidote: a few deliberately added allow rules after the first bootstrap, plus a filter "what could break outside my repo" with every decision.

YOLO mode without conditions. Bash(*) on the allowlist after the first frustrating day. The agent gets access to production integrations through MCP. A few days later, a command appears that shouldn’t be there, and the database or the secrets file disappears.

Antidote: two deny rules for dangerous patterns, no prod-data on the agent side, repo in git. Conscious YOLO mode is possible but requires boundary conditions that this course has not yet led you to.

AI creates a project from scratch. You skip the skill and prompt: just create an Astro project with auth for me. The agent generates a semi-working boilerplate, and you lose two days patching what npm create astro would have done correctly in thirty seconds.

Antidote: with each prompt, ask if there is a CLI that the agent can delegate this to.

Brownfield-as-greenfield. You have an existing project, but instead of auditing it, you run a bootstrapper. The scaffold conflicts with the existing code, .scaffold siblings multiply throughout the directory, and time is spent resolving collisions.

Antidote: if the project exists, /10x-health-check is the proper entry point. The bootstrapper is for an empty directory.

All four modes are natural behaviors of the first week: anxiety, frustration, illusion of model competence, the "let's start from scratch" reflex.

YOLO mode performed by 10xDevs

This question arises in almost every cohort: do you, the instructors, confirm every action of the agent?

No. Since September 2025, we have been mainly working with the active --dangerously-skip-permissions for main projects. Nothing spectacular happened, except for one case where the agent, during debugging... deleted the local database. Subsequent attempts like this were later stopped by a rule in CLAUDE.md, which explicitly prohibits destructive operations on the database without interactive confirmation.

But: we work in a safe environment. Without MCP servers connected to production, with git versioning and branch discipline. Our YOLO mode meets the boundary conditions that Anthropic describes in the devcontainer documentation.

Hence the course recommendation: stay in the intermediate state (allow/deny plus Claude Code asking about the rest) until you have the instinct for when one of the boundary conditions stops applying.

Your code in the cloud: privacy and model training

The permission policy protects your disk. But when the agent reads a file and sends it to the API, your code leaves the machine. What happens to it on the other side?

The answer depends on two things: who is the model provider and which tier you are working on.

Three tiers, three different rules

Most providers divide their products into three layers, each with a different privacy policy:

  1. Free application (ChatGPT Free, Gemini, claude.ai Free) — Your data by default trains the model. You can disable this, but you must do so actively in the service settings.
  2. Paid individual subscription (ChatGPT Plus/Pro, Claude Pro/Max, Gemini Advanced) — depends on the provider. With Anthropic and OpenAI, you can disable training. With Google, the paid consumer subscription does not change the training policy (you need to manually disable "Keep Activity," just like on the free tier).
  3. API / business plans (Anthropic API, OpenAI API, Google Vertex AI, Claude Team/Enterprise, ChatGPT Team/Enterprise) — Your data by default does not train the model. This is a fundamental difference compared to consumer tiers.

Key takeaway: APIs and business plans are private by default. Consumer applications are not.

American suppliers: what to pay attention to

Anthropic (Claude, Claude Code): On consumer plans (Free, Pro, Max), Anthropic displays a pop-up asking if you allow training. If you agreed during registration and want to change it, go to claude.ai/settings → Data Privacy Controls section and disable it. On commercial plans (Team, Enterprise, API), your data does not train the model, unless the admin explicitly enrolls the organization in the Development Partner Program.

OpenAI (ChatGPT, Codex): On the free and Plus/Pro toggle "Improve the model for everyone" is enabled by default. You disable it in Settings → Data Controls. Note: disabling this toggle does not apply to Codex. Codex has a separate setting at chatgpt.com/codex/settings/general. On the API, data has not been trained since March 2023. For Team/Enterprise, training is disabled by default.

Google (Gemini): The only one from the big three where the paid consumer subscription does not change the training policy. Gemini Advanced / Google One AI Premium treats data the same way as the free tier. If "Keep Activity" is enabled (default), Google uses your conversations for training and human reviewers can read them. You disable this at myactivity.google.com.

Chinese Suppliers: Jurisdiction Matters

DeepSeek and Alibaba (Qwen) offer models that can compete on price with American providers. But the difference does not lie in the quality of the model. It lies in the data jurisdiction.

DeepSeek stores all data in China (PRC). The privacy policy explicitly states: "We directly collect, process and store your Personal Data in People's Republic of China." There is no option to choose the region. The Stanford FMTI Transparency Report (December 2025) gave DeepSeek 0 out of 5 points for data usage transparency.

Alibaba Cloud (Qwen/DashScope) is better in one respect: it offers deployment modes with data storage outside of China (Singapore, USA, EU). However, the parent company remains subject to Chinese law.

Why is this important? Three Chinese laws create a framework in which no Chinese company can refuse to provide data to Chinese authorities:

  • National Intelligence Act (2017), art. 7: Every organization and citizen must support, assist, and cooperate with state intelligence.
  • Data Security Law (2021): Prohibits sharing data stored in China with foreign law enforcement agencies without the consent of the Chinese government.
  • PIPL (2021): Chinese equivalent of GDPR, but with a one-time state access right without a refusal mechanism.

There is no adequacy decision regarding data protection between China and the EU/EEA. DeepSeek does not have a GDPR representative in the EU. None of the Chinese providers offer a Data Processing Agreement comparable to those provided by Anthropic, OpenAI, or Google.

Practical consequence: Code sent to the DeepSeek API ends up on servers in China and potentially trains the model, without a clear opt-out path. If you are working with company code, check your employer's policy before sending anything through the Chinese API. The price does not negate legal consequences.

Checklist before the first session with the agent

  • Check which tier you are working on (free / subscription / API / business).
  • If consumer tier — disable model training in provider settings.
  • If you are working with company code — confirm with compliance whether the provider meets the requirements (DPA, data storage region, certificates).

What if delegation fails

The "delegate to authoritative CLI" pattern works as long as the CLI itself works. But what if it doesn't?

The bootstrapper treats this as a HARD-STOP. If npm create astro@latest returns a non-zero exit code, the skill stops. It does not try to fix the situation from the agent side, does not improvise missing files, does not continue to npm audit.

What remains on the disk?

  • The .bootstrap-scaffold/ directory with what the CLI managed to generate before the error.
  • Partial verification.md with a failed status and a brief error description.
  • Your original tech-stack.md untouched. The contract from the previous lesson (M1L2) returns as input for retry.

You open .bootstrap-scaffold/ and look for the cause. The most common are: lack of network access, lockfile from an older npm version, directory name conflict, missing environment variable. You fix it at the system level, not at the agent level, and restart /10x-bootstrapper.

Pattern to remember: when the authoritative CLI says no, the agent also says no. Attempts to bypass this ("let the model write this file manually") lead straight to semi-functional boilerplate that cannot be maintained in subsequent modules.

How the bootstrapper treats existing files

Let's assume that you already have a package.json in the directory, left from a previous exploration or a manually written project version. npm create astro@latest wants to save its own package.json. What happens?

The bootstrapper does not overwrite. Your package.json remains, the version from the scaffold goes to package.json.scaffold as a sibling. You compare both and make a decision: keep yours, accept the scaffold, or merge manually.

One directory is treated specially: context/. Everything placed there (prd.md, tech-stack.md, any other notes) is preserved verbatim, without exceptions. These are your traces of design decisions from previous lessons (M1L1, M1L2). The bootstrapper treats them as data, not candidates for replacement.

.gitignore is specially handled: the bootstrapper merges your entries with the entries from the scaffold. You don’t lose local ignores, but get a fresh set from the starter.

This entire mechanism is in-execution in practice: you don't want the agent to have free rein over your disk. You want the tool to have built-in rules that you don't have to enforce with a prompt.
