Uruchom agenta

Uruchom podstawowego agenta lokalnie w trybie interaktywnym przez terminal:

npm run start:dev-agent -- --runMode=interactive

Konfiguracja agenta

Każdy agent (czyli model, instancja agenta, która wykonuje zadania) może mieć własną konfigurację w pliku agentConfig.json. Przykładowa konfiguracja wygląda następująco:

{
  "agentName": "10xDevs Basic Agent",
  "runMode": "interactive",
  "toolsEnabled": ["terminal"],
  "context": "You are an assistant helping to write code and explain concepts.",
  "maxTokens": 2048
}

Główne pola konfiguracji:

  • agentName - nazwa agenta, która będzie widoczna w logach i interfejsie.
  • runMode - tryb pracy agenta, np. interactive lub batch.
  • toolsEnabled - lista narzędzi, z którymi agent może się integrować, np. terminal, fileEditor.
  • context - ciąg tekstowy określający rolę agenta i kontekst pracy.
  • maxTokens - limit tokenów modelu podczas generowania odpowiedzi.

Dodawanie narzędzi do agenta

Agent może korzystać z zewnętrznych narzędzi, by lepiej wykonywać zadania. Narzędzia można dołączyć przez dodanie ich do listy toolsEnabled w konfiguracji agenta, np. ["terminal", "fileEditor", "ghClient"].

Oto przykłady narzędzi:

  • Terminal – pozwala agentowi wykonywać polecenia systemowe, takie jak git lub ffmpeg.
  • File Editor – umożliwia agentowi czytanie i zapisywanie plików w projekcie.
  • GitHub Client (ghClient) – pozwala agentowi przeglądać repozytoria, tworzyć issue i pull requesty.

How HTML and CSS provide layout possibilities?

Flexbox - Flexible Box Layout

Flexbox is a CSS layout mode designed to provide a more efficient way to lay out, align, and distribute space among items in a container, even when their size is unknown or dynamic.

  • Display flex defines a flex container and enables flex context for all its direct children.
  • Flex direction determines the direction of the flex items (row or column).
  • Justify content aligns flex items along the main axis.
  • Align items aligns flex items along the cross axis.

What the agent sees and does not see at the start

Every new programming session with the agent starts with a pre-filled working memory (context window) of the AI model. This is the space where system instructions, project rules, inter-session memory, available commands are located, and your current prompt may also be placed there. This is the first important observation - even in an empty repo, the Agent does not start from scratch. However, it still lacks essential contextual information.

In the prework [3.2] the hierarchy of instructions loaded into the context window appeared for the first time. Now we return to it from a practical perspective. First, the general structure:

This command /context in Claude Code visualizes it well - note that when running this command for the first time, we already have 22 thousand tokens used!

Every component that goes into the model's context window has a different owner:

  • System prompt defines the tool manufacturer. These are the security rules, tool handling, environment isolation, context compression, and general work rules. You do not edit this layer.
  • Auto-memory project memory that the agent builds during each session. It is convenient but worth reviewing regularly.
  • CLAUDE.md / AGENTS.md constant rules developed by you and your team. This is where project conventions, non-obvious constraints, and rules that the agent will not infer from the code itself are placed.
  • Skills and commands standardize repetitive tasks, for example /init, /commit, /review, or your own work procedures.
  • Task prompt / Messages A series of messages exchanged with the agent in the context of a given task.

This whole arrangement of instructions should lead to a practical conclusion: your prompt is just the tip of the iceberg that influences the direction of the agent's work.

All other layers affect what happens after writing the prompt and pressing "ENTER" - how the agent interprets it, what it does first, what in another step, which files it loads, and which it ignores.

Of course, you can always interrupt the process, reject the diff, undo steps, or add a correction, but that would be a waste of time and tokens. In practice, without instructions, you give the agent a wide scope for wandering.

If the agent misinterprets the intention or does not know the local rules, the error does not have to end with one wrong answer. It can trigger a cascade: wrong file, wrong pattern, wrong command, a fix to its own erroneous fix.

Therefore, design instructions are not only meant to avoid repeating the project conventions, response format, test commands, and forbidden patterns in every prompt. They shift part of the control before the session starts. Before the agent begins to operate, it receives a framework that limits the scope of guessing.

It is possible to work without this layer, but then the task prompt simultaneously becomes the command, the project memory, and the emergency brake. This quickly turns into a ritual, and with agents, it also becomes a source of costly deviations.

From /init to the first draft

How to implement a specific set of onboarding rules and instructions for an agent? It's good to start with built-in commands - this is a starter that will give us safe defaults.

In Claude Code, this will be the /init command – after executing it, the agent analyzes the repository and generates a starter CLAUDE.md: inside you can find build commands, test and formatter run commands (if any), the project structure, and initial conventions. If the file already exists, /init should suggest improvements instead of overwriting everything from scratch.

Codex will create AGENTS.md as a result of the same command - the assumptions are identical.

In the package for this lesson, you also get an experimental skill /10x-agents-md. Treat it as an alternative to the built-in /init command or as a fallback if your IDE or agent doesn’t have its own project file generator. The skill is intended to help prepare the first draft of AGENTS.md, but it does not exempt you from manual review: you still need to check whether the file describes your local conventions rather than summarizing documentation the agent can read anyway.

GitHub Copilot does not have a local equivalent of /init in the CLI. Its main design convention is .github/copilot-instructions.md. Copilot also supports AGENTS.md, so within a team it is worth clearly deciding which file is the source of truth.

Let us now compare different approaches to such rule initialization:

Experimental /init in Claude Code

In newer builds of Claude Code, there is also an experimental variant of /init, toggled by the CLAUDE_CODE_NEW_INIT flag. This is not about a second description of the same generator CLAUDE.md, but an extension of the command towards a short project onboarding.

When the /init flag is enabled, it may ask whether to prepare a team CLAUDE.md, a private CLAUDE.local.md, or both files. It may also suggest skills and hooks.

Skills are meant to fit repetitive, but not necessarily automatic workflows: deeper project verification, session report, deploy to sandbox, release checklist, or procedure for fixing a specific type of problem. Hooks are rather intended for mechanical checks that the agent should not skip: formatter after file editing, quick lint or test for the changed file, command executed at the end of the turn. The prompt distinguishes these levels quite sensibly: a hook is meant to be a deterministic signal from a tool, a skill is a procedure called when you actually need it.

An important difference is procedural: the agent first examines the repository, then only inquires about gaps that cannot be inferred from the code, and only at the end proposes specific artifacts.

The run looks like this:

CLAUDE_CODE_NEW_INIT=true claude

And then in the session:

/init

This is still an experimental feature dependent on the Claude Code build, so treat it as a convenient shortcut for configuration work, not as a stable team standard. Evaluate the final effect with the same filter as a regular sketch: does the given instruction, skill, or hook eliminate a real, repeatable agent error.

How to clean a sketch

We have an onboarding file - now let's take care of its content.

First instinct? More project details, more about the architecture, an extensive description of the tech stack, the most important framework conventions, rewriting the commands from the README, and the classic – "write readable code".

The file grows to several hundred lines and looks professional.

Unfortunately, such content can paradoxically worsen the quality of the agent's work. On one hand, extensive rules in the context window leave less space for your commands. The busier the window, the worse the interpretation quality of subsequent messages. On the other hand – redundancy. If you rewrite things that the agent can find in the README or, even worse, has already seen during training (e.g., description of the Observer design pattern or a fragment of the FastAPI documentation), you fill its context window with noise.

The creators of Claude Code suggest a practical limit: CLAUDE.md should usually stick to about 200 lines per file — not because line 201 suddenly breaks the model, but because a larger file increasingly means you are paying in context for things the agent doesn't need in every session.

Cursor in the documentation of the rules puts it more operationally: a good rule is focused, executable, and limited in scope. It does not describe the entire project. It tells the agent what to do differently in a specific workplace.

This provides a simple sketch cleanup filter:

  • if the rule applies to the entire repo and is truly meant to always be enforced, leave it in the main AGENTS.md or CLAUDE.md,
  • if it applies only to part of the code, move it closer to that code: to a nested AGENTS.md, a local CLAUDE.md, or a tool equivalent like .cursor/rules with a glob,
  • if the rule repeats the README, style guide, framework documentation, or linter configuration, remove it and leave a reference to the canonical file,
  • if the rule sounds like an intention ("write readable code", "care about quality"), change it to a verifiable behavior (e.g. avoid the type "any") or remove it,
  • if the rule does not result from a recurring agent error, do not add it "just in case".

Instead of copying a sample component, endpoint, or migration into the instruction file, point the agent to a template file: @src/features/users/user.service.ts, @docs/api-errors.md, @migration-template.sql. The rule stays short, and the example doesn’t become outdated in two places at once.

The impact of various types of rules on working with Agents is described by two studies, which we place at the end of the lesson, in the Deep Dive / Additional Materials section.

For now, you need just one conclusion from this: it’s not about "short file good, long file bad" — it’s about a practical intuition derived from how the model we’re working with is built. Cursor provides a practical mechanism: start simple, break down large rules into smaller ones, reference files instead of copying their content, and update instructions only when you notice a recurring agent error.

How to evaluate the quality of rules and instructions for the Agent?

Inclusion test: before writing a rule, ask yourself:

Could the agent know this without this file? Could publicly available training data (books, articles, blogs, repositories in your stack) have prepared it for this?

If yes, usually do not add. If not, add.

AGENTS.md or CLAUDE.md may include:

  • non-obvious project conventions, for example the error response format,
  • unusual file and module naming,
  • import rules that cannot be safely inferred,
  • pitfalls characteristic for the project,
  • "embarrassing" workarounds that must be maintained because they result from code history or a dependency bug.

They do not belong there:

  • general documentation of a mainstream framework,
  • commands detailed in the README, if the agent is going to read them anyway (a reference via @README.md is sufficient),
  • popular rules like "use TypeScript strict mode" which you already have configured in tsconfig.json,
  • general statements such as: "write clean code", "take care of quality", "apply good practices".

Remember - AGENTS.md (and rule files in general) is not documentation for a person who is seeing TypeScript for the first time. It is onboarding for an agent who knows TypeScript but is not familiar with your local conventions.

There is one more important effect: the tools listed in AGENTS.md are used much more frequently by agents. In one study, this increase was 1.6-2.5x. This file effectively directs the agent's behavior.

After creating the first draft - via /init, /10x-agents-md, or manually - use the second tool from this lesson's package for a quick rules review:

/10x-rule-review <rule_file>

For example:

/10x-rule-review AGENTS.md

Skill evaluates the specified file in five dimensions: length, presence of pasted code or configuration fragments, language precision, redundancy relative to the knowledge the agent already has, and the order of rules in the file.

The result is an evaluation table with verdicts OK, WARN, or FAIL and specific fixes. By default, the skill does not edit the file - it provides a diagnosis, and the decision to make changes remains yours.

Treat /10x-rule-review as a static quality review of the rule before or after the experiment. If you have added a new rule in AGENTS.md, you can immediately check if it is short, specific, non-redundant, and well placed in the file.

Then start a fresh session with a representative task anyway. Only then will you see if the rule actually changes the agent's behavior.

Five patterns to test

The best way to calibrate rules is simple: check where the agent actually breaks your conventions.

Select one pattern that matches your project:

  1. Error response format - the project uses { error: { code, message, context } } instead of { error: string }.
  2. File naming - the project uses feature.handler.ts, not featureHandler.ts.
  3. Import convention - the project requires absolute imports with @/, not ../../.
  4. Module structure - each module has an index.ts, types.ts, and __tests__/.
  5. Date handling - the project always uses UTC and the formatDate() helper, not loose new Date().toISOString().

Then do a simple test:

  1. Ask the agent to implement without a rule in the instruction file (preferably 3-5 times, each time starting from the same repository state)
  2. Check if and where the convention was broken. Note the available metadata: agent's working time, executed commands, and the visible cost/tokens.
  3. Add a minimal rule in 1-3 sentences.
  4. Request the same change in a fresh session and compare the result: convention, time, number of explored files, and number of iterations.

Such a simple test should show you whether the rule reduces the agent's wandering and helps it find the local convention faster.

If the agent hits the convention without a rule, you don't need an entry. If it systematically selects the wrong pattern, you've found a good custom instruction.

Instruction hierarchy - AGENTS.md and CLAUDE.md

But what if the instructions already exist and, worse, there are many of them? Which ones are taken into account by the agent?

Claude Code loads CLAUDE.md from several locations, and the order matters. At the start of the session, the agent looks for the file in the user's directory (e.g., on macOS it will be ~/.claude/CLAUDE.md), then in the main repository folder, and then in the subdirectories where you are currently working — the deeper in the project structure, the more specific the rules can be and can overwrite or complement those from higher levels. If you edit a file in src/api/, the agent may load both the main CLAUDE.md at the repo level and a separate CLAUDE.md inside src/api/, which contains rules specific only to that part of the code. General rules live higher, area-specific rules closer to the code. In practice, we will explore this in the lesson on scaling context, in the later part of the 10xDevs training.

Codex and GitHub Copilot work similarly for AGENTS.md. Codex searches for the AGENTS.md file in directories from the current one up the tree — the nearest found file has priority. GitHub Copilot applies the same rule: the closest AGENTS.md in the directory tree wins. You can have a global AGENTS.md in the root of the project with rules for the entire team and separate files in subdirectories for specific modules. Claude Code natively reads CLAUDE.md, but supports import via @AGENTS.md — just add this marker to CLAUDE.md so the agent loads both conventions. In teams using multiple tools simultaneously, this is a natural starting point for unifying rules in one file.

Loading multiple files raises the question of how the loading order affects the weight of individual instructions. Research on language models describes the phenomenon of U-shaped attention: models clearly pay more attention to the content at the beginning and at the end of the context than to those in the middle. Instructions buried deep in a long file—or loaded as the fourth or fifth document in a row—statistically have a lower chance of being consistently followed than those that the agent sees right after the system prompt. This explains why the recommendation of "around 200 lines per file" is not an effective guideline: a long monolithic CLAUDE.md causes the rules in its lower half to land in the area of the model's weakest attention.

Practical conclusion: place the most important design rules at the top of the file. If the tool supports scoped rules and loads them selectively — for example, src/api/CLAUDE.md only when working in that part of the code — each shorter file goes entirely into the context window and close to the beginning of its section. The agent does not have to "swim down" to rule No. 47 at the bottom of a 400-line document. This is one of the specific benefits of granular rules instead of one big file.

Lessons from Agent Incidents

A good onboarding document for an agent contains practical working rules, but good rules rarely come from theory. They most often come from incidents.

In previous lessons, context/foundation/ served as a place for the project's permanent contracts: prd.md, tech-stack.md, and other decisions that subsequent skills can read without guessing. In m1-l4 we add one more file to this directory: context/foundation/lessons.md.

This is a log of repeatable lessons appended only at the end of the file. It contains things that should influence the future setup of the problem, research, planning, implementation, or review.

It's not about a log of every agent error. It's about rules that will be returned.

The skill from the m1-l4 package is used to add such lessons:

/10x-lesson

You can also provide the rule seed directly in the command:

/10x-lesson feature flags should always have a kill date

The skill will ask four questions and append one lesson to context/foundation/lessons.md:

  • Context — where this rule applies: part of the system, work phase, file pattern,
  • Problem — what exactly breaks without this rule,
  • Rule — the rule itself in the imperative mood, preferably in 1-2 sentences,
  • Applies to — which skills should weigh it the most: frame, research, plan, plan-review, implement, impl-review or all.

The entry has a fixed format:

## Do not add lodash without an explicit reason

If the file does not yet exist, /10x-lesson will create it with the header # Lessons Learned. If it exists, it will append a new section at the end.

This is an important convention: this file grows by appending, not by rearranging history. You do not sort it, deduplicate it incidentally, or modify earlier entries when you want to add a single observation.

The difference between an incident and a lesson is practical. If the agent did something strange once, note it locally or fix the prompt. If you see a class of errors that would change decisions in previous tasks and will likely come back in future ones — run /10x-lesson.

You can also place a reference to the registry in the main project instructions so that agents outside 10xWorkflow know where to look for these rules:

## Lessons learned

After a few weeks, such a file becomes a concrete map of your agent environment: where the model guesses correctly, where it guesses incorrectly, and which rules are justified by real repeatable problems.

Other 10xWorkflow skills can read this file as a reference point, so one well-written lesson starts working in the next stages of the project.

Team settings

The previously discussed tips assume that we stick to a single common instruction standard, but this is not a rule. Currently, the AI community is leaning towards AGENTS.md, although the creators of Claude Code have a different opinion. In such a case, Claude can import AGENTS.md with a single line entry in CLAUDE.md:

This file provides guidance to AI Agents when working with code in this repository:

If Claude Code is the main tool but you want Codex or Copilot to use the same rules, the simplest setup looks like this:

  • AGENTS.md as a common source of truth,
  • CLAUDE.md as a thin layer of Claude Code with an import of @AGENTS.md,
  • .github/copilot-instructions.md only when Copilot needs its own clarifications.

Additionally, if tools require different file names and you do not want to maintain duplicates, you can use symlinks:

ln -s AGENTS.md CLAUDE.md

Most importantly: one rule should not live in three places. Duplicate instructions diverge just as quickly as duplicate code.

  • Generate and filter the first draft of rules. Run /init (or /10x-agents-md) in your project, then /10x-rule-review AGENTS.md. System prompts of popular tools

In the previously presented context stack, system instructions and tools were at the top. Do they have to remain a black box for us? Not necessarily – the Open Source community throws us a lifeline.

These are collections of system prompts for popular coding agents available on GitHub. They show a recurring pattern of instructions: first the agent’s identity and security restrictions, then system operations, followed by the project context, task execution rules, and tool documentation.

Let's look at specific examples. One of the best publicly documented examples is the Claude Code system prompt (v2.1.120, ~610 lines).

Observation 1 — security hierarchy above all. The first instruction in the entire prompt concerns security, not code:

Assist with authorized security testing, defensive security, CTF challenges,

There is no list of frameworks here. There is no preferred language. The first rule states what the agent is not allowed to do. This is an intentional signal from the creator: the agent's identity must be based on safe actions.

Observation 2 — "blast radius" as a working principle. A dozen lines later, a principle appears that requires the agent to estimate the reversibility of each action before performing it:

Carefully consider the reversibility and blast radius of actions.

The prompt literally categorizes types of risky actions: destructive operations (rm -rf, dropping tables in a database), hard to reverse (force push, amending a published commit), visible to others (comments on GitHub, messages on Slack). The agent therefore must not only know how to write code — it must be able to weigh the consequences.

Observation 3 — memory degradation built into the prompt. In the section concerning the auto-memory mechanism, a rarely encountered epistemic clause appears:

A memory that names a specific function, file, or flag is a claim that it existed

This means that the tool's manufacturer directly built a warning into the system prompt: memory can lie. The agent is instructed to verify every memory before using it, rather than treating it as a fact.

Observation 4 — lazily loaded tools. The Claude Code system prompt does not contain full schemas of all available tools. Some of them — including CronCreate, Monitor, TaskCreate, NotebookEdit — are listed only by name in the Deferred Tools section. Their full schemas must be fetched on demand by a separate ToolSearch tool. This is an intentional trade-off: a smaller base prompt, less context usage, schemas fetched only when truly needed.

For comparison, the Cursor system prompt (~18 KB) shows a different architecture philosophy.

Observation from the Cursor — terminal state as a file. Cursor does not provide an agent with a dedicated API to inspect the terminal. Instead, information about running processes and recent commands is passively injected as text files in the terminals/ directory:

The terminals folder contains text files representing the current state

This is a fundamental difference compared to Claude Code, where every action is an explicit tool invocation. In Cursor, a significant part of the IDE context is sent to the model as text — which facilitates integration but also means the agent has less control over what and when it reads.

Comparing both prompts, a consistent logic is visible: the place for AGENTS.md is always after the producer layer, and before the task prompt. The producer defines what the agent can do, AGENTS.md defines how you work in this specific project. The two layers have different owners and different goals, and here this is clearly visible.

Traces from Sessions

Let's now return to the auto-memory layer from the diagram at the beginning of this lesson.

In many agent tools, an automatic memory of the project and our actions will be built between sessions. This is also something that will be included in the context window of future agent tasks.

For example, in Claude Code you can check and edit it via /memory, and disable it with the autoMemoryEnabled setting or the CLAUDE_CODE_DISABLE_AUTO_MEMORY variable.

Memory is local to the machine and project (using macOS as an example):

~/.claude/projects/<nazwa-katalogu>/memory/MEMORY.md

<directory-name> is created from the repository path, where slashes are replaced with hyphens. Example:

/Users/jan/projekty/moja-aplikacja

At the beginning of the session, Claude Code loads the memory index: the first 200 lines or 25 KB of MEMORY.md. More detailed topic files may be loaded only when needed.

After an important session, check what the agent remembered:

cat ~/.claude/projects/<path>/memory/MEMORY.md

If you see an incorrect decision, remove it. If an important decision did not make it to memory, but in your opinion it should be there, add it manually – this can be another layer of verification alongside AGENTS/CLAUDE.md.

Reset is simple: delete or rename the memory/ folder. During the next session, the agent will start rebuilding the memory.

Another option is to change the configuration in the settings under ~/.codex/config.toml:

[features]

When Codex is enabled, it can carry useful context from previous threads to subsequent ones: stable preferences, repeatable workflows, technology stack, project conventions, and known pitfalls. In a single thread, you control memory usage with the /memories command.

Codex memory files are stored under CODEX_HOME, by default in ~/.codex/memories/. You can view them during debugging or before sharing the Codex directory, but do not consider manual editing of these files as the primary way to control the agent.

The rule is the same as with Claude Code: local memory helps, but it is not the source of truth. The rules that are to apply to the team should be kept in AGENTS.md or in the documentation in the repository.

📚 Additional materials

  • On the Impact of AGENTS.md Files on the Efficiency of AI Coding Agents — Lulla et al. A study on OpenAI Codex: about 28% shorter median execution time and about 16% fewer tokens generated by the agent in small PRs up to 100 lines of code.
  • Evaluating AGENTS.md: Are Repository-Level Context Files Helpful? — Gloaguen et al., ETH Zurich + DeepMind. Study on Claude Code, Codex, and Qwen Code: redundant instruction files increase costs, while minimal rules make more sense than a copy of the documentation.
  • Cursor Rules documentation — practical rules for creating rules: focused, executable, limited in scope, with references to files instead of duplicating documentation.
  • GitHub Copilot Custom Instructions and AGENTS.md Support — GitHub Changelog. Support for AGENTS.md and .github/copilot-instructions.md.
  • OpenAI Codex documentation - AGENTS.md and Memories — principles of using AGENTS.md as the project context and /memories as memory control in the current thread.
  • Lost in the Middle: How Language Models Use Long Contexts — Liu et al. A study showing u-shaped attention: models achieve the best performance for information at the beginning and end of the context, and the worst for content located in its middle — the basis for recommendations regarding the order and length of instruction files.
