Map as an Input Contract

The project map from the previous lesson is not a note "to read." It is an input contract for the agent.

If you followed the previous lesson correctly, you have a document with a fixed structure in context/map/repo-map.md — because it was created from a fixed synthesis prompt. Regardless of which repo you mapped, your map should have the same sections:

  • TL;DR — what a repo is and where the work is focused,
  • Terrain — areas of high responsibility vs peripheries, deep and shallow modules, activity over time,
  • Real couplings — what really changes together: couplings, layers, cycles,
  • Risk zones — several high-risk areas with a one-line "why",
  • Who to ask — candidates for context per zone,
  • First day — list of 5–8 input files,
  • Constraints — time window, method, and what the map DOES NOT say.

These are exactly the pieces of information the agent needs to avoid reading randomly. In this lesson, the three most important of them will be: Risk Zones (from here you will choose the goal), First Day (from here you will take the entry points), and Constraints (from here the initial unknowns will arise).

Note the order. First the map, then the goal selection, then the scope, then the semantic analysis, and finally the structural confirmation. In large legacy systems, this is the difference between working with an agent and watching the agent work its own way.

From the map to a single flow

The map is not a goal in itself — it is a tool that allows moving from "we got to know the repo" to specific tasks on the backlog. And there are always several such tasks in a real project: someone reports a free endpoint, someone wants to add a field to the model, someone plans a bigger refactor and asks what to be afraid of.

Let's take one of them — one that is "hanging" on our simulated issue tracker. The task concerns a critical area of the application: message saving (PostStore.Save). This is the business core of Mattermost — every sent message goes through here — and the idea came up to optimize this flow, maybe prepare it for debt repayment. The problem is that no one on the team can say off the top of their head exactly how it works and what will break if we mess with it.

The map already suggests that this is a sensitive area: the storage layer with the toughest coupling in the entire repo. But "sensitive area" is not enough to touch anything. Before in the next lesson about refactoring (M4L4) we start thinking about changes, in this lesson we need to truly understand this one flow — and I will take it as the Deep Focus goal until the very end.

And here lies the essence of what the map provides. Without it, analyzing a large repo always starts the same way: the agent reads randomly, opens files that happen to match the name, and burns through context in areas that don't concern us. With a map, we reverse this dynamic — we tell the agent where to burn tokens, where to intensify research, and where not to even look.

The map transforms the broad "analyze storage" into focused work:

  • we know where to intensify the research — in the risk zone, not in the periphery,
  • we know which file to start reading from — from the First day section,
  • we know what to expect from coupling — from Real connections,
  • we know which questions have been left open — from Constraints.

Instead of asking the agent to "understand this module," we give it a narrowed goal supported by evidence from the map — and only then we let it go deeper.

In your project, such an area could be account registration, payment and webhook, flashcard generation, synchronization with integration, account deletion, file import, or sending a notification.

One thing is important: the further you go, the more specifics. Narrow, but deep.

What the map won't tell you — and why we reach for an agent

Let’s pause for a moment and summarize what we really have on hand after the previous lesson.

We have a map. And quite a large one: we know where the work is concentrated, which areas hold significant weight in the overall project context, where the risk zones are, what the real connections are, and which files to start reading from. This turns "analyze the repo" into guided work — an invaluable advantage at the start.

But the map has three built-in constraints. Not because we made it wrong — but because that is its purpose.

  • Shows width, not depth. The map answers "where to look?", not "how does it work?". It knows that a given flow is sensitive. It does not know where the data actually flows, what validates it, and what breaks along the way.
  • It describes the structure, not the behavior. It is a static picture: dependencies, layers, entry points. What happens during execution (order of steps, side effects, error paths) lies outside of it.
  • It is a snapshot in time. It was created from the history window and describes activity, not the current state of contracts. Some of its details have become outdated, and its graph by design does not cover the entire repo, so there remain white spots marked as unknowns.

In other words: we built the map broadly and cheaply, mainly using the CLI, scanning everything. That was the right method for a Wide Scan. But the questions change for Deep Focus — from "which areas?" to "how does this one flow really work and where are the pain points?". Such questions cannot be answered with a cheap scan. You need to dive fresh into the code, reconstruct the flow step by step, review the change history, and firmly separate what is proven from what is guessed.

This is exactly the moment when the map alone stops being sufficient, and we bring in an agent to work. Not to start from scratch — but to take the map as a prior and deepen exactly what it lacks: the real trace, runtime behavior, and the current state of the code, with the rigor of evidence / inference / unknown, so that an outdated prior does not silently overwrite the fact.

The map says "dig here, not there." Now let's take the shovel.

The skill you know is based on parallel sub-agents: it takes one research question, decomposes it into several areas, and launches 2–4 agents simultaneously. Each one examines a different dimension of the problem, then synthesizes the results into a single artifact. In a new project, this decomposition tends to be loose. In legacy, we want to control it.

  • sub-agent blast radius → what changes together with this flow when you move it.

It is no coincidence that these three in particular. If you are thinking about working in legacy, you need to know how it works, if there is a safety net, and how far the change reaches. The rest is details.

Technical debt

Here the report did something the map couldn't: it turned a general "this was a sensitive area" into a specific type of risk.

Deepening the report with the ast-grep tool

Before analyzing and refactoring legacy code, the precision in the report matters in direct proportion to the risk of change. And this entire process relies on models that become increasingly reliable month by month (especially with CLI support), but can still miss that one piece that determines the success of the entire change.

If you have ever updated code after reviewing the model, you know it firsthand. That file skipped, that method ignored, that parameter not taken into account.

Why? Because numbers and structural statements ("only here", "always through X", "exactly four calls") are the weakest analyzed by the agent, and the report reader often takes them on faith. This is the same class of error that causes the model to make mistakes counting letters in a word.

Hence the counterintuitive rule: the more confident a number looks in a report, the more it is worth checking. We will use a tool for this that truly understands the code structure, not just variable and class names.

Once again CLI. In the previous lesson, we used git (what changes together) and dependency analysis (who imports what). Both answered questions about relations and aggregates, meaning history and graph. Now we need something from a different category: a tool that at the syntax level will tell exactly where a specific pattern occurs in the code, with no guessing, no sampling, fully reproducible. It's the same philosophy as before — a cheap, deterministic CLI as a counterpoint to an expensive, probabilistic agent — but aimed at a different axis.

The agent is great for meaning and behavior (how this flow works, what it validates, where it hurts). The CLI is great for numbers and positions (exactly how many, which lines, and is it really only here). Confirming numbers is not a lack of trust in the agent, but a division of labor.

What is an AST and ast-grep. When an agent looks at code, it sees text, i.e., a string of characters. That’s why a search like grep "Save(" will also catch comments, strings, and SaveMultiple( — for text matching, these are all the same string. A compiler sees something different: a syntax tree (AST, or abstract syntax tree), where "this is a method call," "this is its receiver," "this is the list of arguments." Each element is a separate, named node with its own metadata. Essentially, it’s the same model that an agent builds in its mind when reading code, except the AST is explicit, queryable, and deterministic.

ast-grep is a tool that allows you to search a code syntax tree in a repeatable way.

You use it by writing patterns (you or the agent) in the syntax of the examined language, with placeholders to be filled in where a specific fragment will only be revealed during the exploration phase.

Examples of using this tool (no worries - the agent will solve this for us):

ast-grep -p '$X.Post().Save($$$A)' -l go

This pattern will match only actual Save calls — it will skip comments, strings, and, crucially, distinguish Save from SaveMultiple. This is the difference between “four call-sites roughly” and “two Save and three SaveMultiple, file by file.”

Installation and cross-platform. ast-grep is a single binary file written in Rust, with precompiled releases for macOS, Linux, and Windows — it works everywhere you work.

Select any channel:

npm i -g @ast-grep/cli      # easiest if you already have Node
brew install ast-grep       # macOS / Linux (Homebrew)
cargo install ast-grep --locked   # if you have Rust
pip install ast-grep-cli    # via Python
# Windows: scoop install ast-grep  /  winget install ast-grep

The command is called ast-grep (there is also a shorter alias sg, but on Linux it conflicts with the system command sg for switching groups, so use the full name). Check the installation: ast-grep --version.

ast-grep vs classic grep — and why you want both

It’s easy to think that ast-grep replaces grep. It doesn’t replace it — it complements it, because their weaknesses are complementary. grep is blunt but honest: it searches text, so it will find an occurrence everywhere, including in comments and twin-names. ast-grep is precise but picky: if your pattern doesn’t match the actual shape in the code, you get zero — and it’s easy to read that as “there are no such calls,” when it actually means “my pattern was wrong.”

Hence the rule worth remembering for this entire analysis phase: use ast-grep for precision, but always confirm any zero matches with grep. A zero from the structural matcher is suspicious until a cheap textual grep confirms that it really means no occurrences, and not a typo in the pattern. Two cheap, deterministic tools watching over both the agent and each other — that's the whole trick of this lesson.

ast-grep can not only read but also rewrite code

Although in this lesson we will not yet deal with making changes, we will briefly highlight the other side of the described tool. Namely - ast-grep is not only a scanner but also an operator for precise code modifications.

The same pattern that finds a shape in the syntax tree can also rewrite it.

It is enough to add -r (rewrite) and provide the target form — with the same meta-variables as in the pattern:

ast-grep -p 'var $A = $B' -r 'let $A = $B' -l js src/

Metavariables captured on the left ($A, $B) are reused on the right. This is not a search-and-replace on text — it is rewriting at the AST level, so it preserves the structure and works only where the shape truly matches. That’s why ast-grep distinguishes Save from SaveMultiple even during rewriting, which sed will never do.

By default, -r only shows the diff — it does not save anything. To review and approve changes one by one, add --interactive; to apply all at once, use -U/--update-all:

ast-grep -p 'var $A = $B' -r 'let $A = $B' -l js --interactive src/   # review change by change
ast-grep -p 'var $A = $B' -r 'let $A = $B' -l js -U src/              # apply all

And if a given rewrite is to be repeatable — part of the process, not a one-time command — you save it as a YAML rule with the fix field and run it through ast-grep scan:

id: no-var
language: JavaScript
severity: warning
message: "Użyj `let` lub `const` zamiast `var`. Znaleziono: $NAME"
rule:
  pattern: var $NAME = $VAL
fix: let $NAME = $VAL

ast-grep scan --interactive

This is the second half of the tool: the same deterministic patterns we used for proof become a tool for change — structural, repeatable, capturing exactly that shape and no other.

Important — in this lesson we are not using this part yet. We are still at the stage of gathering evidence: analysis, not modification. I'm showing you rewrite now so it's clear that such a possibility exists. You will reach for it when one of your future changes turns out to be a mechanical, shape-preserving transformation.

For now, ast-grep remains in the role you just learned: a verifier of structural assertions. The same pattern you use today to confirm the report will carry the actual change itself in the next lesson on refactoring.

This section contains additional in-depth knowledge on selected topics related to the lesson. In this Deep Dive you will find:

A Hotspot is complexity × change frequency

In the ③ Technical debt section, you look for risky spots, not just ugly ones. The difference is one observation: risk lies at the intersection of two axes — complexity and frequency of changes.

Each of them alone is misleading. A dense file, but frozen for two years, is just old — no one touches it, so it won't do any harm. A trivial file, but touched in every other commit, is usually a config or a barometer, not debt. Only the product — code that is complicated and constantly changing — indicates a place where every subsequent change is costly and risky, because it has to be made often and each time in difficult terrain.

And here is the non-obvious part: which exact complexity proxy you choose is less important than the actual intersection itself. Lines of code, indentation depth, cognitive complexity — these are different approximations of the same "how hard it is to understand," and practitioners of code history analysis deliberately use these inexpensive measures to count them consistently across the entire repo. Don’t anchor your report on a single complexity metric as if it were a verdict; treat it as one axis that you multiply by another.

This closes the loop from the previous lesson. You built the project map, among other things, from the git history — so you already have the frequency of changes in the Terrain. Deep Focus adds the missing axis: on the selected flow, you ask the agent about the complexity of specific files, and the hotspot is those among them that fall into both buckets at once.

Connascence — a brittleness glossary instead of "tightly coupled"

“Tightly coupled” is a verdict, not a description. It says that something is tight, but it does not say what kind — and you start refactoring with the knowledge of exactly which type of coupling you need to untangle. Connascence is a practical dictionary that gives names to couplings instead of putting them all in one bag.

Several labels that are most often useful in a report:

  • Position/order coupling — two places must agree on the order of arguments or steps. In our flow, this is writing to the database, which relies on order, and no compiler enforces it.
  • Meaning / convention coupling — two places must agree on the interpretation of the same value. In our case: two halves of the model, backend and front, with nothing holding them together toolwise.
  • Name / type coupling — a shared name or type used in many places, for example a public contract of the store layer.

Key distinction: static connascence is visible in a single place in the code (the compiler or ast-grep will catch it), dynamic connascence only reveals itself at runtime — and it is the latter that is the silent debt, because nothing watches over it. Additionally, the strength of coupling increases with distance: agreeing on the order of arguments within one function is cheap, but agreeing on the meaning of a field between backend and frontend, whose git history almost never touches in the same commit — that is exactly the kind of brittleness this report aims to name explicitly.

Treat this as a set of working labels, not an academic taxonomy to be recited. The value is practical: instead of "this module is tightly coupled," you write "the backend and frontend share connascence of meaning over a long distance, without a tooling barrier" — and it’s immediately clear where the pain points are and what to look for.

Logical coupling: pair vs hub

The sub-agent blast radius combines the static graph with co-changes from the git history — files that change together despite no obvious import. This is a powerful signal, but it is easy to interpret it too literally. It helps to break it down into two different questions.

First: which pair is tightly coupled? If A and B almost always appear in the same commit, there is a hidden contract between them — often invisible in the code, because no import expresses it. This is a candidate for the connascence of meaning described above.

The second: which file is the hub? A file coupled with ten others poses a different risk than a tight pair — it is a node with a wide blast radius, where one change radiates across many areas. Summing couplings per file (instead of looking only at pairs) reveals exactly such hubs. Confusing one with the other is a classic mistake: a ranking of "what changes most with many things" answers a different question than "which pair is most tightly coupled".

And two caveats without which co-changes lie. First, they are evidence within a time window — they show what changed together, never what should have been synchronized but in history is not (this remains as unknown). Second, a high co-change can be innocent: a massive refactor, reformatting, or a single "tidy up everything" commit can link files that are not actually related. Therefore, the report ranks risk from two sources simultaneously — the static graph and the co-change — and leaves contradictions between them as questions for verification, not as facts.
