Test plan, unit tests, hooks, E2E tests. Over the four lessons of this module, you have built a proactive quality pipeline that catches known risks before they reach the repository.

And then a ticket comes from the user.

"I rate the flashcard as »Good«, but it still comes back for review — I keep seeing the same cards."

No stack trace. No error code. No screenshot. One sentence and the diagnosis to be made.

The pipeline from previous lessons? This time it might have missed it. This particular application path didn’t have any test that would trigger it. Why — we will break it down at the end of the lesson.

That doesn't mean the pipeline is bad. Every such mechanism has its blind spots. Proactive tests catch risks that you can anticipate. Production reveals the ones you didn't foresee.

This lesson teaches responding. Instead of an agent that writes tests "upfront" based on risks, you will guide an agent that gathers evidence from multiple sources, formulates a hypothesis, and confirms it with a test. Diagnostics that fill the gap after proactive tests.

The bug we will be diagnosing was intentionally introduced into 10xCards for the purposes of the lesson. At this stage, the application had no bugs, and we did not want to change the project context to something else.

We will go through this entire investigation step by step below, just as it would happen with a real ticket: from parsing the report, through data from Sentry, runtime logs and local reproduction, to a test reproducing the bug and a verified fix. Ready commands and configuration for each tool are collected in the 🔎 Deep Dive section, and links to the official documentation can be found in the 📚 Additional Materials. Return to them when you are reproducing this process on your own.

One note before we begin. Sentry, wrangler tail, and Playwright are a specific set of tools for the 10xCards stack (Astro on Cloudflare). In your project, they may look different: Datadog instead of Sentry, vercel logs or flyctl logs instead of wrangler, Cypress instead of Playwright. What doesn't change is the investigation structure: where to get evidence from, in what order, and how to combine it into a diagnosis. At each step, we name the pattern so you can translate it to your own tools.

From ticket to investigation plan

Parsing a ticket with an agent

User ticket is the hardest entry point in debugging. When a unit test fails, you have an assertion and a stack trace. When Sentry catches an exception, you have a location in the code. When a ticket comes in, you usually have one vague sentence.

Instead of guessing the cause yourself, give the agent the ticket and ask for it to be structured. This is the same principle you know from the lesson Unit and Integration Tests with the Agent (M3L2): the more organized the context you provide to the agent, the better result you get.

In testing, you gave the agent a specific, structured input related to risk. In debugging, you give it a raw ticket and ask to break it down into specific diagnostic questions.

The agent parses "I evaluate the flashcard, but it still comes back for review" as:

  • Reproduction steps: open the review → mark the card as “Good" → refresh → the same card is still pending
  • Scope: the evaluation of the card returns success, but the schedule does not change
  • Frequency: every time
  • Possible problem area: review route → writing to review_states (the review works, but the new review date is not saved)

Now you have something to work with. The agent does not know the answers yet, but it knows the questions.

Triage — where to look first

You now have specifics: reproduction steps, scope of the problem, frequency, possible area. It's time to decide where to look.

"The evaluation returns success, but the schedule does not change" is a signal that the API does not report an error, yet something silently does not work. The first step should be monitoring. Is there an error appearing somewhere in the system that the API does not propagate outward?

Not every symptom leads to monitoring. Different signals suggest different sources:

In our case: the API returns 200, but the evaluation does not change the schedule. We start with Sentry.

Collecting evidence

Sentry MCP — production data

Sentry is an error monitoring service: the SDK integrated into the application captures exceptions in production and sends them along with the stack trace, breadcrumbs (a trail of events preceding the error), and request context. If you don't have Sentry on your project yet, no worries — the installation process and free plan are described in the Deep Dive of this lesson. Here, we assume that monitoring is already collecting data from production.

With Sentry configured, the agent queries it through the MCP server: it searches issues, retrieves stack traces and breadcrumbs without leaving the terminal. Why MCP, and not sentry-cli? The Sentry CLI handles release tasks: uploading source maps, creating releases, managing debug files, and deployments. It does not have a dedicated command for searching issues. This is done by the MCP server: it exposes issues, stack traces, and breadcrumbs as structured data, which the agent queries and immediately combines in its loop. The diagnosis remains on the agent side, in line with what this lesson teaches.

But since the endpoint returns 200, what exactly is Sentry supposed to see here? It depends on one detail of the configuration. On a failed save, the handler performs console.warn(...) (we will see this shortly in the code), and the project has the captureConsoleIntegration integration enabled, which forwards such logs to Sentry as events. Thanks to this, a silent warning from production appears on the issues list, even though the user received a 200 status. We show the configuration of this integration in the Deep Dive.

The agent finds the issue review/rate: update_failed. The stack trace points to rate.ts, at the handling of the write result to review_states: UPDATE of the schedule returns an error, but the handler does not propagate it. Breadcrumbs show the context of the card rating, which directly precedes the error.

This is the first layer of evidence. We now know that:

  • An error exists — failed UPDATE to review_states
  • Location — rate.ts, handling the save result
  • Context — the error occurs when evaluating the card

Without monitoring, this information would be buried somewhere in the server logs (if your setup stores them) or most likely nowhere at all.

But we still don't know how often this happens or why the write fails. For that, we need additional sources.

Runtime logs as a supplementary signal

Then the Sentry agent checks the runtime logs. For Cloudflare, it is wrangler pages deployment tail:

npx wrangler pages deployment tail --project-name 10xcards-30 \
  --format json --search "update_failed"

Logs confirm: console.warn with the message review/rate: update_failed appears after each card rating. This is not an occasional problem. It is a systematic error affecting every user.

Sentry gave the location, the logs provide the scale. That's enough to move on to reproduction.

The specific tool depends on the platform. Wrangler is an example for Cloudflare, Fly.io has flyctl logs, Vercel has vercel logs, AWS has CloudWatch.

The pattern is the same: query the platform log stream and filter by the message related to the symptom.

Local reproduction

Production evidence points to rate.ts and a failed write to review_states. Before the agent starts reading the code, let's confirm that the problem reproduces locally.

The agent launches the application, opens the replay, rates the card as "Good," refreshes. The same card is still waiting in the queue. Symptom confirmed.

Here, Playwright CLI (or Playwright MCP) comes to the rescue, the same tool you got to know in the E2E Tests lesson (M3L4). This time, it is not used for testing but for gathering information: it gives the agent access to the browser, including console messages (browser_console_messages) and network traffic (browser_network_requests). These are two channels that show what happens on the client side before we look into the server code.

Browser console (browser_console_messages): no JavaScript errors. This excludes a frontend-side problem.

Network requests (browser_network_requests): POST to the rating endpoint returns 200 with the card, which appears to be updated. But subsequent GET to the replay endpoint returns 200 with the same card in the queue. The server reports success, but the new deadline is never saved.

Now we know: it's a server-side problem, in the data. No crash, no network error, no rendering issue. The schedule change simply does not reach review_states.

Three sources, three layers of information. Now the agent has enough data to read the code with a purpose instead of blindly.

At this point, one might ask: since the bug is anyway in rate.ts, wouldn't it be faster to just open that file right away and read the code? For this specific bug, maybe. But reading code “in the dark” fails exactly where debugging hurts the most: with errors visible only in production, not reproducible locally, in unknown or large project codebases, or appearing randomly. Then you don't even know which file to open. That's why we practice the method of gathering evidence from multiple sources, not a trick for a single case. Besides, even here the evidence earned its place: Sentry provided the location, the logs showed the scale (every rating, not a single incident), and reproduction isolated the frontend from the backend, which you wouldn’t have learned from rate.ts alone.

Visual signal: reproducing UI issues

For this bug, the console and network were enough — the symptom was server-side, in the data. But some bugs live exclusively in the rendered interface: overlapping components, wrong z-index, clipped text, broken animation. Here browser_console_messages is clean, browser_network_requests returns 200, yet the user still sees something broken. The DOM and logs are silent because the problem exists in pixels, not in the structure.

Then local reproduction has one more channel: a screenshot given to the vision model (VLM) with an open diagnostic question — "what's wrong here?". Playwright MCP in vision mode (--caps=vision) takes a screenshot and operates on coordinates instead of element refs, so it sees what the accessibility tree does not express. This is not the same as visual verification in the E2E Tests lesson (M3L4): there, vision confirmed a narrow, known assertion (whether this element looks as it should); here you ask an open question because you don’t yet know what broke, so the model is there to help you name the symptom.

Which model? You need a model with image support (VLM). Three categories:

  • Frontier (e.g. Claude Opus, GPT with vision support) — highest quality, highest cost.
  • Budget (e.g., Gemini Flash) — good vision, low cost, fast responses. A reasonable default choice for repeatable diagnostics.
  • Open-weight (e.g. Qwen-VL) — zero API costs on your own hardware, at the expense of speed.

The cost is real, though small: one screenshot via the API is a fraction of a cent and takes a few seconds, a local model is free but slower. Specific names age quickly — stick to categories and refresh your choices every few months.

Vision has limitations that must be kept in mind during diagnosis:

  • Accuracy. Vision models shift coordinates toward the center of the screen; elements near the edges are often poorly located. The question must be precise.
  • Hallucinations. The model may report a problem that does not exist (false alarm — frustrating), or overlook one that does (false negative — more dangerous). Treat its response as a hypothesis to be confirmed, not hard evidence.

And the limit regarding testing: for pixel-level regression in tests you use deterministic tools (toMatchSnapshot, Argos, Lost Pixel) — this is the work from the E2E Tests lesson (M3L4), cheap and repeatable. Playwright MCP with a vision helps in something else: in an open diagnostic question that a deterministic tool will not ask.

Diagnosis and fix

Synthesis — the evidence converges

The agent reads rate.ts. The handler calculates the new card state in memory (updated), saves it to review_states, and then handles the result of the save:

if (updateError) {
  console.warn(`review/rate: update_failed ...`);
  return jsonResponse(200, { ok: true }); // ← błąd połknięty, zwracamy „sukces"
}

We compare this with data from Sentry: an error on the UPDATE to review_states. The write fails (the handler writes a value to the column that the database rejects on every attempt), but the error does not propagate. Instead of returning a 500, the handler logs a console.warn and responds with 200 { ok: true }. The frontend only looks at the response status, so it moves on to the next card as if nothing happened — and the new deadline never reached the database.

Hence the 200 OK and the card that appears to be rated. And since the review selects cards by due <= now, an unchanged due date means that the same card returns with every subsequent opening of the review.

Note: no single source was sufficient for a complete diagnosis.

  • Sentry provided the location (rate.ts, failed UPDATE) and the error name (update_failed)
  • Logs provided the scale (each rating)
  • Reproduction with Playwright provided context (the response returns 200, but the card returns)
  • Code gave the root cause (swallowed write error and returned as success)

The agent combined these signals into a coherent diagnosis. This is the skill you are learning here — not reading code or configuring tools, but conducting an investigation where each source adds something that others did not have.

First, a test that reproduces the bug

Before the agent starts fixing, it's worth writing a test.

In the lesson Unit and integration tests with the agent (M3L2), the test started from the risk and worked "forward": identify the risk → write the assertion → check if the code covers it. Here we work the other way around: you have a symptom → you write a test that confirms it → the test fails → you fix it → the test passes.

This technique has its name: test-driven bugfixing. It is an established practice in TDD — Kent Beck described it decades ago: when you find a bug, before fixing it, write a test that reproduces it, then bring it to green. What’s new is that the agent can guide this process for you, from symptom to finished assertion.

The agent writes an integration test that checks the saved state rather than the API response:

test('rating a due card advances its persisted schedule', async () => {
  // arrange: seed a review_states row that is already due
  const { flashcardId } = await seedDueCard(accountId);

  // act: rate it "Good" via the endpoint
  await rateCard(flashcardId, 'Good');

  // assert: the PERSISTED row moved into the future
  const row = await getReviewState(flashcardId);
  expect(new Date(row.due).getTime()).toBeGreaterThan(Date.now());
  expect(row.reps).toBe(1);
});

How to get the agent to such a test? First, describe the symptom and request a test that fails for the correct reason, i.e., checks the persistent state in the database rather than the response status:

We have a bug: the flashcard is marked with a score of 200, but the schedule isn’t saved and the card goes back to ‘for review’.
Write an integration test for the grading endpoint (`rate`) that:
- seeds a row in `review_states` with a `due` date in the past (the card goes straight to ‘for review’),
- triggers a “Good” rating via the endpoint,
- reads the SAVED row from the database (not the API response) and verifies that `due` has moved
  into the future and `reps` has increased.
The test should fail at this stage — this is a reproduction of the bug, not a fix.

This is a debug-as-test loop: a red test before you start fixing. On the skills side, reach for /10x-tdd — it leads the cycle "first a failing test, then minimal code" for unit and integration tests, which is exactly our case. If the reproduction had to go through the browser (a bug visible only in the rendered UI), the E2E layer is handled by /10x-e2e from the E2E Tests lesson (M3L4).

This test remains in the repository as protection against regression. Moreover, it is the first test of the correctness of SRS schedule transitions — exactly the layer that our test pipeline did not have yet.

Fix and multi-layer verification

The fix has two parts. First, we stop swallowing the error: instead of returning 200 with the memory card, we propagate the failure:

// before:
if (updateError) {
  console.warn(`review/rate: update_failed ...`);
  return jsonResponse(200, { ok: true });
}

// after:
if (updateError) {
  console.warn(`review/rate: update_failed ...`);
  return jsonResponse(500, { error: "rate_failed" });
}

Secondly, we fix the actual save operation so that the UPDATE goes through. Only the combination of both changes results in a card that truly disappears from the queue: simply exposing the error (instead of swallowing it) turns the silent 200 into a visible 500, and the correct save causes the schedule to actually shift. This is, after all, the essence of this type of bugs. First, you stop hiding what’s happening in the system, and only when the error is visible can you properly handle and fix it.

But just "works locally" is not enough. Every diagnostic source that showed the problem should now confirm that it no longer exists:

Verification is a mirror image of investigation. You diagnosed through four channels, you verify through four. If you skip any layer, you cannot be sure that the fix is comprehensive.

Swallowed errors — a class of problems visible only in monitoring

Let's take a step back from this specific bug and notice the pattern.

The API returns 200 OK. The review card appears to be saved. But the side effect (saving the new schedule to review_states) ends with a silent error.

Try/catch catches the exception, logs a warning, and continues — returning to the client the value calculated in memory. The user sees success because they received the evaluated card. The permanent record was lost silently.

Swallowed error is an error that the system swallows instead of propagating. OWASP in the 2025 edition added this class to the Top 10 as A10: Mishandling of Exceptional Conditions. It is one of the most common patterns leading to bugs that users report for weeks before anyone tracks them down.

Why didn't our pipeline, built from the first to the fourth lesson of this module, catch this?

  • Unit and integration tests (M3L2) cover a different path: session saving and its contract. The rating route and SRS logic do not have a single test — none run rate.ts. Not because they were forgotten, but because the testing phase for this risk simply hasn't been created yet (the bug itself, as you remember from the beginning of the lesson, was deliberately introduced here).
  • Hooks (M3L3) check the source code. TypeScript is correct — the value we write to the database has the correct type, but only its content at runtime is rejected by the database.
  • E2E Tests (M3L4) run through the flow of generating and saving flashcards, not the rating path. They check the visible happy-path result, not the persistent saving of a new schedule in rate.
  • Test plan (M3L1) anticipated this layer: the correctness of the SRS schedule transitions is listed as a separate risk. However, a named risk without a constructed test phase is not yet a gate. This is exactly the gap closed by the test just written that reproduces the bug.

Automated gates that actually trigger on every change had no chance to catch a bug in the code that none of them touch — and that the API actively hides. The consistent signal comes only from monitoring. The diagnostics you've practiced here are not an "additional tool," but a safety net under your proactive tests.

One workflow, four entry points

The process you just went through works regardless of where you get the initial signal from:

What changes? The entry point and the amount of data at the start. A unit test gives you an assertion and a stack trace immediately — you decide faster where to look. A user ticket usually gives you one sentence, and you have to find the error in the system yourself.

What does not change? Synthesis from multiple sources, hypothesis confirmation by test, verification at every layer.

If your stack uses other tools, the process is the same: change the data source, keep the investigation structure.

Sentry — configuration and free plan

The free Sentry plan (Developer) provides 5000 errors per month, 30 days of retention, and full API access. For a course project, this is more than enough. You can check the current limits and prices on the pricing page.

For an Astro project on Cloudflare, you need two packages: @sentry/astro and @sentry/cloudflare. The SDK automatically detects the Cloudflare adapter starting from version 10.40.0. In wrangler.toml, you need to enable nodejs_compat.

Note — Astro 6 on Cloudflare: Astro 6 requires the @astrojs/cloudflare adapter version 13+, which integrates with Cloudflare via a custom entry point. Since @sentry/astro 10.44.0 (issue #19762), this path is supported — and this is exactly the configuration for 10xCards (Astro 6.3.1, deploy on Workers). Instead of the adapter's default entry point, you specify your own file in wrangler.toml, which wraps the Astro handler in Sentry:

# wrangler.toml
main = "./sentry.server.config.ts"  # zamiast "@astrojs/cloudflare/entrypoints/server"
// sentry.server.config.ts
import * as Sentry from "@sentry/cloudflare";
import handler from "@astrojs/cloudflare/entrypoints/server";

export default Sentry.withSentry(
  (env: Env) => ({
    dsn: env.SENTRY_DSN,
    // przekaż console.warn / console.error do Sentry jako zdarzenia
    integrations: [Sentry.captureConsoleIntegration({ levels: ["warn", "error"] })],
  }),
  handler,
);

Purely frontend Astro (without the Cloudflare adapter) is based on issue #19753 — check the current status there before deploying on Astro 6.

It is precisely captureConsoleIntegration that causes swallowed console.warn from our bug to appear in Sentry at all. However, there is a cost-related catch here: captured warnings and errors end up in the same issue stream and consume the same error quota as unhandled exceptions (5000 events per month on the free plan). At an early stage of the project, with low traffic, catching all warnings and errors is a reasonable default choice because you want maximum visibility. As traffic grows, this doesn't scale: narrow down levels (e.g., to ['error']) or report selected cases manually via Sentry.captureException to avoid burning through the quota and flooding real incidents with noise.

If you do not provide a DSN (or provide an empty string), the SDK initializes in no-op mode. Events are captured but not sent. This allows you to have the same code in environments with and without Sentry.

You don't have to configure this manually. You can use the /10x-new → /10x-plan → /10x-implement path from module 2 to have the agent guide you through the entire setup.

Sentry MCP — details of the diagnostic process

The Sentry MCP server (@sentry/mcp-server) allows the agent to query Sentry without leaving the terminal.

Configuration in Claude Code:

npx @sentry/mcp-server@latest --access-token=<YOUR_TOKEN>

Two main tools in the diagnostic process:

search_issues searches for issues matching the query. Without an LLM key (OpenAI/Anthropic), the tool operates in pass-through mode: you pass the query directly in Sentry syntax, e.g. message:update_failed is:unresolved. No additional API key is needed — a free Sentry account and an access token are sufficient.

get_sentry_resource is a universal tool for fetching resources with two invocation modes:

  • URL mode — you provide the url (e.g. a link to an issue), resource type detected automatically
  • Explicit mode — you provide resourceType (e.g., "issue"), resourceId (short ID like PROJECT-123), and organizationSlug

Sample run for our bug:

search_issues(query="update_failed", organizationSlug="...", projectSlugOrId="...")
→ finds an issue with a short ID

get_sentry_resource(resourceType="issue", resourceId="PROJECT-123", organizationSlug="...")
→ full stack trace, tags, breadcrumbs in Markdown format

Sentry MCP is pre-1.0. Tool names may change. Check the repository before first use.

Runtime logs outside Cloudflare

Earlier in the lesson, we used wrangler pages deployment tail for real-time logs. There are a few things worth knowing about this tool:

  • The results show outcome (ok/error/canceled), logs[], and exceptions[], but not the HTTP status or response body
  • Max. 10 concurrent tail clients (dashboard + CLI together)
  • Streaming stops above 100 req/s sustained for 5 minutes

Cloudflare also offers persistent logs through the [observability] section in wrangler.toml (200K events/day on the free plan, 3 days retention). This feature is documented for Workers. If your project is hosted on Cloudflare Pages, unfortunately, you do not have access to it.

On other platforms, the pattern is the same, only the tool changes:

  • Fly.io: flyctl logs --app my-app
  • Vercel: vercel logs my-project --follow
  • AWS Lambda: CloudWatch Logs + aws logs tail
  • Railway: railway logs --follow

Playwright CLI/MCP — diagnostic tools

In the main part, we used Playwright for diagnostics through MCP tool names. Here are their equivalents in the CLI and a third tool that is useful when you want to see the exact server response:

Tool (MCP)CLIWhat it provides
browser_console_messagesconsole [min-level]console messages filtered by level (error, warning, info, debug)
browser_network_requestsrequestslist of requests with method, URL, and status
browser_network_requestrequest <index>full details of a single request: headers, body, status, duration

All three are core tools — they do not require the --caps flag. It's easy to get confused here: --caps=network concerns mocking routes (not reading), and --caps=devtools concerns tracing and recording (not console diagnostics). No flag is needed for just reading the console and network traffic.

In our case, browser_network_request is useful to closely inspect the response of POST /api/review/<id>/rate — to see that the status is 200, and the body is a card that appears to be updated, even though the database did not accept it.

Besides these tools, Playwright CLI provides browser_navigate, browser_snapshot, and other navigation commands useful for replaying reproduction steps.
