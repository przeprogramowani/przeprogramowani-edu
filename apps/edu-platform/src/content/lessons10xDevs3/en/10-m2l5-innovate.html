In previous lessons, we went through full change cycles for simpler and more complex slices.

And immediately the next question arises: what about the rest of the roadmap?

If there are six slices left in the roadmap, the simplest reflex is: "let's launch six agents." Tempting. For the first five minutes, it looks like the future of programming, and then you have to handle several sessions with the agent, several branches, several PRs, and one question: who is supposed to understand it all?

Still you.

More agents without management and control does not mean greater productivity. It means more accidental code for which you will have to take responsibility.

In this lesson, you will learn how to conduct work in parallel in an organized manner. We will ensure a separate scope, a separate git worktree, and a separate agent.

In the prework, you learned about worktrees and context isolation as concepts. Here we will see how to apply these parallel work mechanisms in practice.

Start with independent slices

If two slices touch the same areas of the project, parallel work will not speed up progress at all. It will produce a conflict that someone will then have to manually untangle.

Therefore, do not choose "any two tasks from the roadmap." Choose two that can reach PRs without getting in each other's way.

While working on the previous slices, you probably noticed minor interface issues, missing actions, annoying limitations. Instead of adding them to the existing slice, we added a new entry to the 10xCards roadmap:

SliceTopicWhy it is suitable for parallelization
S-05Account deletion (GDPR)Account deletion and data retention logic, separate project area
S-06UX/UI fixesBulk actions, session reset, separate project area

S-06 is exactly such a slice that appeared during work: minor interface improvements discovered during the implementation of earlier changes. It does not affect business logic, so it can run in parallel with S-05.

How to add new tasks to the roadmap? Through a conversation with the agent, a simple prompt is enough:

During work on S-01 through S-04 I noticed missing bulk actions on candidate review, no way to reset a review session, and poor loading states. Add a new slice S-06 "UX improvements" to the roadmap — status planned, prerequisites F-01, parallel with S-05. Keep it concise and consistent with existing slice format.

Not sure if two slices are independent? Ask the agent before implementation:

Check the roadmap and plans for S-05 and S-06. Assess whether they can be implemented in parallel. Pay attention to shared files, contracts, layers and other elements that may cause conflicts.

This verification will allow you to make a decision: whether I can work on these tasks in parallel, or if I need to approach them sequentially (one after the other).

Worktree is a separate lane

When you have two independent slices, you need two separate workspaces. That's what git worktrees are for.

A worktree is a separate working directory connected to the same repository. Each has its own branch, its own HEAD, and its own index. Agent A edits files in one directory, agent B in another, and the main repo does not mix their states.

git worktree add ../10xcards-account-deletion -b feature/account-deletion
git worktree add ../10xcards-ux-improvements -b feature/ux-improvements

After these commands, you have three folders:

  • the main project folder where you read the roadmap and coordinate work,
  • worktree for S-05,
  • worktree for S-06.

This is the moment when the prework strategy Isolate gets concrete. You don't just isolate the conversation. You isolate the directory, branch, agent session, and task context.

There is an important boundary here. The worktree isolates the git state and working files, but it does not isolate the entire environment.

Two worktrees can still compete for the same local development server port, the same local database, or the same payment sandbox. We need to address this separately.

Two modes: conversation or delegation

You have worktrees and independent slices. Now the question is: how do you run the agent?

For complex changes where you make decisions along the way, work as in previous lessons: interactive session, /10x-implement, manual pauses, ongoing control. This is the best, most universal working mode.

But if the plan is specific, the scope is closed, and the completion criteria are measurable, you can delegate the implementation without interaction. Especially for simpler tasks.

In Claude Code and Codex this is done using /goal:

/goal Use 10x-implement skill to implement all phases of context/changes/ux-improvements/plan.md. Each phase is committed separately. All phases marked done in plan progress. Stop after 20 turns if not complete.

One command and one condition. The agent starts working immediately. After each turn, a separate small model (evaluator) checks whether the condition is met. If not, it continues. If yes, it closes the session.

The control shifts later: to the PR, review, and your decision. This is a deliberate reduction of the level of interaction, not a reduction of responsibility.

In the second worktree, you can also use /goal or work with the workflow you learned in previous lessons - the decision depends on your preferences, the tool you use, and your budget.

Multiple sessions in one view

Two worktrees with two agents is a good start. When parallel work becomes a daily routine, you naturally look for a tool that will show you all sessions in one place instead of juggling several terminal/IDE windows.

Tools such as Cursor Agents, Antigravity (Google), Superset, Conductor, and Claude Code Agent View solve the same problem: orchestration of multiple agents from a single interface.

The pattern is the same everywhere:

  1. Isolate context per task.
  2. Delegate the goal to the agent.
  3. Review the result in one place.

You don't have to choose a single tool forever. Links to each of them can be found in the additional materials. More important than the choice is understanding the pattern, because that will not change, even when specific products rotate every quarter.

Marcin used Superset for a while but quickly returned to working with Claude Code in the Ghostty terminal. In several sessions, he works by creating new terminal tabs. The choice of tools is a matter of personal preference and what is most comfortable for you – we encourage you to experiment.

🧑🏻‍💻 Practical tasks

Your first parallel session

If you want to deploy several changes to your project in parallel, don't start with five agents. Seriously, don't do that. Start with two slices.

Your checklist:

  1. Select two slices that do not directly depend on each other.
  2. Check shared files, contracts, migrations, and external services.
  3. Create a separate worktree and branch for each slice.
  4. Run a separate agent session in each worktree.
  5. Use the mode that fits the task and your preferences - use /goal or the interactive /10x-implement.
  6. Perform a Solo Code Review (M2L3) for each change.
  7. Open a separate PR for each change and merge according to the project's workflow process.

After this exercise, you will have a less romantic but much more useful picture of parallelism. Not "AI does everything at once," rather: "I run several well-isolated cycles and do not exceed my own throughput in terms of decision-making and review."

And that is precisely when the question arises, which opens the third module.

I deliver faster. But how do I know it really works?

In this module, we learned an efficient process for planning, implementing, and reviewing changes.

We still lack consciously designed quality gates. We have no tests. We have no automatic gates that will say "this change broke something" before the user notices it.

We do not solve this here. This is the starting point of module three.
