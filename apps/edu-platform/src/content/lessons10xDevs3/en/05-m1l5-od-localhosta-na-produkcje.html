From Localhost to Production

This lecture covers typical deployment methods for a classic web application.


What is deployment?

Deployment is the act of making your program available to users and placing it on a computer - a server. You can deploy to your local machine, a machine on your home network, a virtual server in a data center, on cloud infrastructure, on an IoT device, or even on a phone or tablet.

The most common for web applications is placing the application on a computer somewhere on the internet, with a public IP address, which anyone can use from their browser.

You can deploy your web app to known infrastructure providers - Amazon AWS, Microsoft Azure, Google Cloud, etc. You can hire a professional web hosting company or run your own server in a virtual private server (VPS) or a container.

Deployment involves:

  • Making the app's code and resources available on the internet, usually on a hosting provider.
  • Configuring security, ensuring the app is accessible on the internet via a domain name.
  • Configuring servers that run the app and respond to user requests.
  • Setting up databases.
  • Setting up web servers that serve static files and reverse proxy.
  • Setting up SSL certificates.
  • Monitoring and scaling the app.

Deployment means getting your software from your development environment to the users.


How to deploy a web app?

Basics:

  • Build the project - prepare all necessary files for deployment.
  • Upload or copy files to production server.
  • Inform the web server to serve your app files.
  • Configure the domain name and DNS to point to your server.
  • Configure SSL certificate for security (HTTPS).
  • Start the server or the app process.

For example, a simple static app can be deployed by just copying files to an nginx or apache server directory.

For dynamic web apps, deployment can be more complex and may involve packaging the app, running it as a service, configuring process managers, etc.


Deployment Methods Overview

  • Manual copy and run: Copy files manually or via scripts to the server and start the application manually or with simple process managers like pm2.
  • Using CI/CD pipelines: Set up a pipeline (with e.g. GitHub Actions, GitLab CI) that builds and deploys automatically after pushing code to the repository.
  • Containerization: Package your app as a Docker container and deploy it to container orchestration systems like Kubernetes.
  • Platform as a Service (PaaS): Use cloud services like Heroku, Vercel, or Netlify for simplified deployment.
  • Serverless: Deploy your functions or front-end as serverless functions.


Example: Manual deployment of static app on nginx

Assuming you have an app built and ready as static files in the build/ folder.

scp -r build/* user@production-server:/var/www/your-app
ssh user@production-server
sudo systemctl restart nginx


Example: Simple deployment of Node.js app with PM2

Copy code to the server, install dependencies on the production server, and run with PM2:

scp -r yourapp user@production-server:/home/user/
ssh user@production-server
cd /home/user/yourapp
npm install --production
pm2 start index.js --name yourapp
pm2 save


Summary

  • Deployment is making your app accessible for users.
  • It can range from simple file copying to complex pipelines and container infrastructure.
  • Pick proper deployment method suited for your app scale and complexity.

How to solve the problem and what does it look like in practice

  • Testing should be 3-4 times more valuable than development for the team
  • Automatize all tests, from unit through tests guiding algorithm development, to UI tests checking the flow between components
  • Create a hierarchy of tests where the most stable tests work at the bottom of the pyramid and less stable ones at the top
  • When tests falter, analyze whether the breakdown is caused by the application or by the test itself
  • Be aware of what type of test is failing, e.g., a unit test, high-level integration, system test, etc.
  • Take care of parameters like test speed, stability, cost of fixing, and instruct developers on these priorities to help them understand why testing is so important
  • Measure coverage close to 100% at the unit level to allow for risky changes without fear of breaking existing solutions
  • Don't rely only on code coverage; integrate other tools searches for logical errors or misuses since coverage shows only lines, not quality

There are many categories of tests:

  • Unit tests – testing individual code units (methods, functions)
  • Integration tests – testing the interaction between several components
  • Functional tests – checking business requirements and correctness of function behavior
  • Regression tests – ensuring new changes do not break previous functionality
  • Smoke tests – quick check before deeper testing
  • UI tests – end-to-end tests of application interface
  • Performance tests – assessing system speed and reliability under load

It's worth keeping in mind that more tests and automation give more confidence, but they also require more maintenance and cost to support.

Choosing a platform as a conscious decision

You could stop here and proceed with wrangler deploy for the 10xCards we are presenting, but the more important question is not "what do mentors recommend for 10xDevs?", but "how do you know this is the right choice for this project, this team, and this application?".

Three questions worth having answers to before you click deploy:

  • Does the platform fit your runtime and the way you build the application (SSR vs static, edge vs Node, long processes vs short requests)?
  • Does it have a sensible mode of operation with the agent – documented and up-to-date CLI, optional MCP server, logs accessible via API?

If you don't have an answer, you didn't choose a platform - one was chosen for you. Let's do it again - better and more universally. Not just for Astro and 10xCards, but for a project you simply want to deploy to production.

Anti-Confirmation Bias prompting - how to avoid "prompts tailored to you"

Every research conducted with an agent has one inherent flaw. Language models by default adapt to what they have already heard from you – this phenomenon has its own name, sycophancy, and it was once a major headache at OpenAI. In practice, this means that if you were subconsciously leaning towards a particular platform, the scoring will be designed so that this platform wins. Your research will confirm your beliefs instead of testing them.

The recipe is simple. Instead of asking the agent is platform X suitable for stack Y?, you ask why is platform X a bad choice for stack Y?.

Anti-bias is a series of questions designed so that the model has no way to escape. Before you approve the result, the agent runs it through three anti-bias tests. This broadens the range of available options and increases your awareness in the chosen area.

Devil's advocate - what is the worst thing that can be said about this choice?

The agent takes on the role of a skeptical architect whose sole task is to find flaws, hidden costs, and risks in a technology that looks like a winner. Where will this platform start to bite you? What limits, which you don’t see now, will hit you first? What seems great at first glance but will turn out to be a problem in the second quarter? The goal is not to prove that the choice is bad – the goal is to get a brutal list of weaknesses in writing.

Pre-mortem - imagine that it is 3 months later and you regret your choice.

A classic design research technique. Instead of asking "what could go wrong?" — to which the model will escape into generalities — you assume that it already went wrong, and together with the agent you recreate the failure scenario. What faulty assumptions led to this decision? Which trap did you not see from the day 1 perspective, but after 90 days was obvious? Pre-mortem shifts the perspective from prediction to narrative, and narratives are much richer in specifics than abstract risk lists.

Unknown unknowns - what didn't make it into the scoring because you didn't think of that question?

The toughest test, because it concerns things you didn't know you didn't know. Here, the agent's task is not to come up with anything sensational, but to list risk classes that the standard scoring captured poorly or not at all – compliance, backups, exit paths from the platform, the production status of a specific feature (GA or beta?), dependencies on decisions from other teams. These questions arise only when someone asks for them.

Each such test ends with one of three decisions:

  • accept risk - I know it, I agree with it, I move on.
  • I add to infrastructure.md as a known limitation - the decision stands, but the risk is consciously documented.
  • I need to change the choice - the test showed something that the scoring did not catch.

The third option happens rarely, but it is precisely for it that you do the entire cross-check. Without it, research turns into an exercise in rationalizing a decision you had already made anyway.

Techniques to avoid the confirmation bias are an excellent way to ensure any conversation with AI yields more objective results that consider various scenarios. The three tests above are built into the skill, but that's not all you should keep in mind. Two additional techniques don't require a skill – you can trigger them from a regular prompt whenever you feel the agent is starting to agree with you.

Comparison of alternatives - force the model to show the competition.

Instead of asking for an evaluation of a single solution (to which the model will respond with praise), ask for three alternative approaches in a comparative table. The table enforces a consistent structure and makes it harder for the model to choose "the same thing, but in different words." It works everywhere – from choosing a library, through module architecture, to organizational decisions like "whether to introduce code review through pair programming."

I am considering {{decision X}}.

Changing roles and perspectives - the same idea through the eyes of different people.

Every decision looks different from the perspective of the person who will live with it. Your decision about the platform will please the backend developer, but may annoy the person responsible for security who does not want a new vendor in the audit. Instructing the model to take on several specific roles one by one reveals burdens that you do not see from a single perspective.

I am considering {{decision X}}.

These two techniques complement the trio from the skill well. Devil's advocate, pre-mortem, and unknown unknowns test the same choice from different angles. Comparing alternatives and changing roles test the space around the choice – what else you could have chosen and who will live with that choice.

Deployment from Plan Mode

Let's now perform the first deployment of the application to the selected platform or production environment.

The input for this process should be infrastructure.md and tech-stack.md. The agent knows where the project is deployed and what it is built from. Time for setup and actual deployment.

What do we want to avoid? The classic - "the agent works, I watch." If something goes wrong halfway through and you have to save the situation, it would be good to know exactly what was supposed to happen, instead of reconstructing it from the Agent's work history.

Here, the so-called Plan Mode, built into many AI-Native tools, comes to the rescue.

This is a mode in which the agent operates read-only: it reads the repo, examines external sources, asks you about missing decisions, but does not modify files or perform project mutations. The final result is a plan – a textual description of what the agent intends to do, step by step. In a Claude Code session, you switch to this mode with the shortcut Shift+Tab (mode cycle: default → auto-accept → plan) – in the IDE this is usually a dedicated button below the message input field for the agent.

How to use the plan? It's simple:

  1. Initialize Plan Mode. The agent in planning mode reads the input (infrastructure.md, tech-stack.md), optionally queries about gaps, and returns the implementation plan procedure.
  2. Evaluation. You read the plan. If something is missing or something is wrong - you say it directly. You do not accept it immediately.
  3. Approval. You accept the plan. Only now does the agent exit plan mode and proceed to execute the task.
  4. Reference. The plan remains in the project. When something goes wrong (and it usually does in the first deployment), you have a specific document to refer to and answer the question "what was supposed to happen?".

On this occasion, it is also worth revisiting the skill /10x-lesson - if you notice that a particular skill or part of the agent's work does not match your preferences (e.g., the plan analyzes an irrelevant part of the repository), add this info to the rules for the future.

Net effect: the agent operates more predictably, and you have a point of reference that does not disappear along with the context.

The input for Plan Mode may vary depending on the task. Now it consists of two reference files and a brief goal: "Let's perform the first deployment based on @infrastructure.md, according to the stack from @tech-stack.md".

The plan generated by the agent should include:

  • automatic steps that will be executed by the agent
  • the list of accounts and services that need to be configured manually (Cloudflare account, possibly an external database),
  • a list of secrets to configure (environment variables, API keys),
  • specific deployment commands in the form of documentation (wrangler pages deploy or wrangler deploy depending on the choice of Pages vs Workers - these are not the same commands),

After the plan is approved, the agent executes the steps where it assigned itself as the "owner". Your role may involve filling in gaps, configuring secrets, or verifying the final state. Watch out for the agent trying to do something that is not in the plan.

The approved plan is saved in the repo as context/deployment/deploy-plan.md. This is not a "document on the shelf" – in the next lesson, when we plan the implementation milestones of the MVP itself, the agent will receive this file as context to know what is already set up, which secrets are configured, and which ready deployment paths the project uses.

CLI vs MCP - how the agent uses a live application

The application is deployed, but that’s not the end – the whole adventure with production maintenance really begins here. Due to subsequent extensions, changes, or decisions, your application will start to live its own life here. Sometimes everything will be under control, and sometimes it will be a state you’re not happy with. And this is where the AI agent also comes to help.

Using the example of communication with production, we will start here a topic that will recur through many subsequent lessons - integrating the agent with external services via CLI (terminal tools) and MCP (a dedicated external protocol).

This is one of the most significant differences between a chatbot living in the browser and an agent that can actually work for us. Of course, it will only do so when we inform it about specific tools and new capabilities.

In the prework [2.1] we said that the terminal (not only agent-based, but simply the system shell) is a universal environment for many tools - anything that can be executed in the terminal, the agent can also execute. This is the first path of agentic operability: you give the agent tools such as wrangler, gh, npm, psql, the agent calls them through Bash, collects output, and makes further decisions.

The second path is MCP (Model Context Protocol). Instead of executing arbitrary shell commands, the agent communicates with a dedicated server that exposes a specific set of tools - for example, "list of deployments", "logs from the last hour", "project environment variables". Each tool has explicitly defined input and output schemas, and the server enforces permissions.

Alright, so the obvious question arises – which option should you choose? Anthropic in their text about production agents states outright: API, CLI, and MCP are three complementary paths. A mature platform often provides all three. Netlify in the documentation recommends installing their CLI along with the MCP server, because MCP uses the CLI underneath where it makes sense.

When is the CLI sufficient, and when is it worth reaching for MCP?

  • CLI works well for operations that you want to be explicit and auditable. The command remains in the terminal, the output stays in the terminal, making it easy to paste into a report or failure-modes.md. Ideal for MVP - little setup, zero additional infrastructure.
  • MCP starts to make sense when the agent needs to operate in environments where the shell is impractical (cloud agent, remote session without access to your terminal), or when the operation requires auto-discovery - the agent has to find out "which command is used for X" on its own, rather than reading a 30-page CLI documentation.

There is one more argument in favor of CLI for MVP - safer defaults. Netlify CLI illustrates this well: netlify deploy by default performs a draft deploy (lands on a temporary URL for preview), and publishing to production requires an explicit --prod flag. The command explicitly requires someone to consciously say "yes, to production." In MCP, this same operation may be a single tool invocation, which does not have an analogous gate - unless the MCP server enforces it itself.

There is also a cost. Each connected MCP server contributes all the definitions of its tools to the context window. With several servers at once, this can consume thousands of tokens just... on their schema. Anthropic describes practical mitigations, such as progressive disclosure and lazy loading, which you have encountered, for example, in the context of Skills (we start with metadata, then based on decisions we load the API definition, and finally instruct the agent to use the active MCP).

The community of practitioners also has an established opinion on this topic. In discussions on HN about CLI vs MCP, one idea is most frequently repeated: CLI is great when you know what command to run. MCP is great when the agent decides what to run. For us, this is a practical tip - choose the interface for the task, not for the trend.

Demo: ready MCP server in agent session

The fastest way to see this in action is to connect a ready-made MCP server from your hosting. Since we are deploying on Cloudflare, we use Cloudflare MCP.

Before you start, it is worth knowing that Cloudflare offers two categories of MCP servers and each of them has a different purpose:

  • Code Mode server (mcp.cloudflare.com) - a single unified endpoint that gives the agent wide access to the entire Cloudflare API by executing code. You choose it when you want to cover many Cloudflare products at once and prefer a small set of universal tools, based on which the agent composes operations itself. The biggest benefit is very limited context window usage.
  • Domain-specific servers (*.mcp.cloudflare.com) - servers specialized for specific products, including docs.mcp.cloudflare.com/mcp (documentation), bindings.mcp.cloudflare.com/mcp (Workers Bindings: KV, R2, D1), observability.mcp.cloudflare.com/mcp (Workers logs and analytics), as well as radar.mcp.cloudflare.com/mcp (Radar) and browser.mcp.cloudflare.com/mcp (Browser Rendering). You choose these when you want ready, typed tools within a single product area - more predictable, more "guided", less improvisation on the agent's side.

In short: Code Mode = breadth, domain-specific = depth and safe defaults in the chosen domain. The connection pattern is the same for both – only the URL changes.

For our MVP, where we want to quickly check the status of the Pages deployment and glance at the logs, a unified endpoint is sufficient:

https://mcp.cloudflare.com/mcp

What is MCP in Code Mode?

Code Mode is a direct response to the problem we signaled earlier - each connected MCP server contributes the definitions of its own tools to the context window. Cloudflare in the article Code Mode: the better way to use MCP provides a concrete number - the classic exposure of all ~2500 Cloudflare API endpoints as separate MCP tools would consume around 1.17 million tokens in the model context. This is more than most context windows can accommodate - and that is before the agent performs any operation.

Code Mode cuts through this problem with a simple move - instead of exposing each endpoint as a tool, the server provides only two:

  • search() - the agent writes a fragment of JavaScript that searches the Cloudflare OpenAPI specification (by product, path, tags) to find the appropriate endpoint, without loading the entire specification into the context.
  • execute() - the agent writes code that performs an authenticated API call, handles pagination, combines operations, and returns only what is needed.

The generated code runs on the Cloudflare side in an isolated Dynamic Worker (V8 sandbox, without a file system, without environment variables vulnerable to prompt injection). According to Cloudflare, this means the unified Cloudflare MCP server takes up about 1000 tokens in the model context—regardless of whether the platform's API has 100 endpoints or several thousand.

For you as a user, this roughly means: a greater chance that the agent still remembers what the conversation was about during a long session, because the MCP tool definitions haven't pushed the rest of the context out of its mind. The price – operations on MCP require the model to write short code for each query, so good models handle this well, while weaker ones can get confused.

The same operation via wrangler CLI

Cloudflare also has a mature CLI - wrangler - and most of the things MCP does can be done through it without connecting a server. After installing the tool, we can execute the following prompts:

When was the last deploy of the opanuj-ai project? Use the wrangler cli.

The agent translates this into terminal commands, parses the results, and prepares a summary. The output is textual, not structural – the agent has to interpret it, but every step is recorded in the terminal history.

What exactly is different:

  • Setup. MCP requires one-time configuration in .mcp.json and appropriate credentials. The CLI requires logging in outside the agent session (wrangler login once) and is ready to use.
  • Format. MCP returns JSON with a schema. The CLI returns text (sometimes with the --json flag you can force the structure). For the agent, JSON is cheaper to parse. The CLI does not have to be specifically created for AI – MCP is an almost native standard.
  • Discovery. The question "what can you do with Cloudflare Pages?" - MCP will answer with a list of tools with descriptions. CLI usually starts with --help and reading the output, step by step.
  • Audit. The CLI leaves a history in the terminal and it may go to shell logs. MCP communicates with its server – the audit depends on what the server logs on the platform side.
  • Context cost. Each connected MCP server contributes the definitions of its tools to the window. The CLI doesn't contribute anything until you invoke it - but the descriptions in --help with each invocation end up in the context as text.

Note - if you use an internal CLI (or even 10x-cli), the tool definitions and handling method should be placed somewhere in the model context (skill, rules, inline prompt) anyway. Otherwise, the model will not manage the handling and will waste a lot of tokens on trial and error (hence we have a dedicated skill for tool handling).

In the context of MVP projects, CLI often wins by simplicity. MCP starts to pay off when the agent has to perform dozens of contextual queries ("show me the logs with a 500 error from the last 24 hours," "compare the bundle size between two deployments") - there, discovery and a structured response genuinely reduce the number of steps.

Best starting point? Start with the CLI, add MCP when you notice a recurring pattern of queries that the agent needs to probe through --help.

Configuration of another ready-made server is analogous - you change the endpoint and optionally the authorization method:

  • Vercel MCP - https://vercel.com/docs/ai-resources/vercel-mcp
  • Netlify MCP - https://docs.netlify.com/welcome/build-with-ai/netlify-mcp-server/
  • GitHub MCP - https://github.com/github/github-mcp-server

The pattern is the same, the code is platform-specific. If you chose hosting without a ready MCP, the CLI with the appropriate tool (wrangler, flyctl, vercel) takes care of it without compromise.

Some MCP servers (including Vercel MCP, AWS Deployment SOPs) are still in beta - check the feature status and regions before production use.

Wrangler and gh - a few commands worth knowing

Regardless of whether you choose MCP or stick with CLI, the agent will regularly use wrangler and gh after deployment – diagnostics, checking secrets, previewing PRs with preview deployment. Here is a handful of commands that are worth not having to look up in --help every time.

Wrangler - minimum to maintain Cloudflare Pages:

  • wrangler whoami - checks which Cloudflare account the session is logged into. The agent's first action before pages deploy, if you want to avoid deploying this project to the wrong account.
  • wrangler pages deployment list --project-name 10xcards - history of recent deployments with URLs and statuses. The basis for the question "what went to production, when, and from which commit".
  • wrangler pages deployment tail --project-name 10xcards - live log stream from the active deployment. It can be invaluable when the application runs locally but crashes only on the edge.
  • wrangler pages secret list --project-name 10xcards - list of secret names configured in the project. You won’t see the values, but you can immediately tell if, for example, JWT_SECRET is in place before you start debugging.

Full directory of Pages commands: https://developers.cloudflare.com/workers/wrangler/commands/pages/.

gh - contact with GitHub without leaving the terminal:

  • gh auth status - the equivalent of wrangler whoami on the GitHub side.
  • gh repo view --web - opens the repo in the browser from the project directory.
  • gh pr list - list of open PRs.
  • gh pr view <number> --web - opens a specific PR in the browser.
  • gh pr create --fill - creates a PR using the last commit message as the title and description. Add --draft if you don't want to involve reviewers yet.

Full manual: https://cli.github.com/manual/.

Agent access boundary to production

⚠ Using both the CLI and MCP, the agent can gain direct access to your infrastructure. This is the moment to pause.

The fundamental question is this: is this access absolutely necessary for the agent to do what it is supposed to do?

For the MVP on 10xCards the answer is usually "yes, for diagnostics and logs." It is not "yes, for dropping the database" nor "yes, for rotating DNS entries." The security pattern is simple: minimum permissions, human authorization for irreversible actions.

In practice, on Cloudflare it looks like this:

  • You create an API token in the Cloudflare panel or select a limited scope during authentication – only for the part of the platform that the agent actually uses (Pages or Workers for one specific project). No DNS, no Workers Secrets for unrelated projects, no billing.
  • You put the token in an environment variable, not in the .mcp.json committed to the repo. The agent receives it indirectly through the MCP server or env discovery of the chosen CLI, you do not paste it into the chat window.
  • You perform destructive actions yourself. If the agent suggests "delete the project from the panel" or "rotate the main secret," you log into the panel and do it manually. A manual click takes 30 seconds, cleaning up after an automatic mistake - several hours.

The pattern works identically on other clouds. On AWS and GCP, the equivalents are IAM roles with scoped access and console-only-user as well as the minimum scope of permissions you assign to a given user/token.

This is the setup for the MVP. As the project grows, the natural evolution leads towards a separate pre-prod environment – the agent gets full access to staging, and only read-only access to production. This is a topic we will discuss in later modules.
