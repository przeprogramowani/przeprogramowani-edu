In the previous lesson, we created the first version of an agent based on the SDK we learned earlier. This solution requires a bit more work than a terminal or editor "ready-made" option, but it also gives you the most flexibility.

Now it's time for the next step. Our agent from the generic LLM wrapper is to become a full-fledged CI/CD assistant. At the same time, we want to detach it from localhost and move it to where all changes heading to production intersect.

In this lesson, using GitHub Actions as an example, you will learn the key elements of an agent integrated with CI/CD processes. We will start with the basics of infrastructure, then prepare the agent itself, learn how to test it, and finally, we'll reach for a ready-made skill for code review (this is in the Deep Dive section).

Let's get started!

GitHub Actions: the minimum to get started

To give the agent more autonomy and agency, we need to disconnect it from one specific machine and put it on the "production highway." For this purpose, we will use GitHub Actions (GHA) – a service for automating everything related to software. It allows building, testing, and deploying applications continuously, and thanks to tight integration with the repository, it does not require configuring dedicated servers. If your repository is already on GitHub, you are just one file away from creating a simple scenario (workflow).

If you are using a different CI/CD provider, use the most important conceptual guidelines from this lesson and adapt the GHA syntax to the chosen infrastructure.

And how much knowledge about GHA itself do we need? Just enough for the agent to start working more independently. Let's start with a few concepts that will recur throughout the lesson:

  • Workflow (also known as: scenario) - an automated process consisting of one or more jobs, described by a YAML file in the .github/workflows/ folder at the repository level.
  • Trigger (event) - an event that starts the workflow, e.g. push, pull_request, or manual workflow_dispatch. One workflow can have more than one trigger.
  • Job - a task run on a specific virtual machine, consisting of one or multiple steps. Within a single workflow, jobs can run in parallel or sequentially.
  • Step - a single step within a job (e.g., installing dependencies or running an agent).
  • Action (action) - a wrapped, reusable set of steps. An action is similar to an editor plugin or an npm library - you publish it for internal or external use and utilize it in multiple scenarios.
  • Runner - a virtual machine on which we run jobs. It is most often provided by GitHub, but you can host your own (so-called self-hosted runner).

These concepts are not a flat list - they form a hierarchy of composition, from the entire scenario down to a single action:

By default, GitHub provides three types of machines: a Linux environment (ubuntu-latest), macOS (macos-latest), and Windows (windows-latest). For CI/CD scenarios in the Node.js stack, the natural choice is ubuntu-latest - it is fast, cheap, and compatible with most commands you run locally. As for costs: for public repositories on GitHub's standard runners, you use the service for free, and for private ones, you get a default pool of free minutes and disk space, after which you move into the pricing plan.

GitHub automatically recognizes workflows - just place a .yml file in the .github/workflows/ folder. Build your first minimal workflow that will run on every pull request to master (and also on demand, allowing you to test it manually without waiting for a PR):

# .github/workflows/review.yml
name: AI Code Review

on:
  pull_request:
    branches: [master]
  workflow_dispatch:

jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Configure node
        uses: actions/setup-node@v4
        with:
          node-version-file: '.nvmrc'

      - name: Install dependencies
        run: npm ci

      - name: Run agent
        run: npm run review
        env:
          LLM_PROVIDER_API_KEY: ${{ secrets.LLM_PROVIDER_API_KEY }}

This file contains all the concepts we just learned. We have a workflow (the entire file), one trigger (pull_request plus manual workflow_dispatch), one job (review), a runner (ubuntu-latest), and a set of steps – two of them use ready-made actions provided from the GHA directory (checkout and setup-node), and the rest we run ourselves (npm ci, npm run review). The step with the agent receives the API key not as text pasted into the file, but through secrets – this is a value defined in the repository settings that no one will see in logs or commit history.

This is the minimum after which the agent stops being a tool on your laptop and becomes a step in the team process. Every PR to master triggers a scenario, the scenario runs the agent, and the result goes where the team is already looking – to the pull request. In the next steps, you will turn this "assessment" into a hard Definition of Done and add a gate that will stop the merge if the criteria are not met.

Composite Action: agent as a reusable plugin

The workflow from the previous section has one drawback: it only lives in this one repository. If you want the same agent to monitor ten repositories of your team, copying the review.yml file to each of them results in ten diverging copies. This is exactly the problem that Composite Actions solve – and it is these that we use in the production version of the agent.

Composite Action is a way to extract a set of scenario steps into a separate, parameterized unit and inject it on demand in multiple places - a bit like a plugin for your pipeline. Most importantly, such an action can be developed in an independent repository (e.g., github.com/your-team/ai-reviewer), and the entire integration with the consumer repository is handled by GitHub Actions for you. This is one of two ways to create modular scenarios - the other is reusable workflows, which are used to compose whole, independent processes, while Composite Action packages a single, reusable step.

This way, you connect several independent, reusable actions into one overarching scenario:

You host a reusable action in two ways:

  1. In a separate repository - optimal when you want to use the action in multiple projects or publish it in the GitHub Actions Marketplace. The consumer refers to it via uses: your-team/ai-reviewer@<sha>.
  2. Locally, in the subfolder .github/actions/<name>/ of the same repo - simpler to start, but makes it harder to use the action in other projects. The consumer refers by relative path: uses: ./.github/actions/ai-reviewer.

For a start, we recommend option 2) - later it is easy to extract such an action to scenario 1), and we do not start by working on several repos.

The heart of reusable actions is the action.yml file at the main level of its repository (or subfolder), marked as composite. Here is what an action looks like that wraps the agent's execution:

# action.yml
name: AI Reviewer
description: Run the team's code review agent

# GitHub Marketplace - branding (optional)
branding:
  icon: terminal
  color: purple

# Input parameters
inputs:
  api-key:
    description: API key
    required: true

# Values returned to the consumer scenario
outputs:
  verdict:
    description: Pass/fail verdict from the agent
    value: ${{ steps.agent.outputs.verdict }}

# Action steps - injected into the consumer scenario
runs:
  using: composite
  steps:
    - name: Run agent
      id: agent
      run: node ${{ github.action_path }}/dist/review.js
      shell: bash
      env:
        LLM_PROVIDER_API_KEY: ${{ inputs.api-key }}

Pay attention to a few elements:

  • The using: composite clause means that the action will be "embedded" into the parent scenario. The action does not define a trigger or an operating system - that is decided by the consumer in their workflow.
  • Each step with run must explicitly specify the shell (e.g., bash). This is a requirement specific to composite actions - in a regular workflow the shell is default.
  • From the action level, you can run any script from its repository using the variable ${{ github.action_path }}, which points to the directory with the action. This way, you can integrate GHA with any AI SDK and technology - not just JS.
  • Inputs (inputs) parametrize the action - typically the mode of operation or an API key. 🚨 Remember the principle of least trust: action consumers may inject potentially dangerous values here, so treat them like any untrusted input.
  • Outputs (outputs) allow returning the result back to the scenario - here the agent's verdict, on which we will soon build the merge gate.

On the consumer side, the workflow from the first section is reduced to a single step – we moved all the logic into the action:

# .github/workflows/review.yml
name: AI Code Review

on:
  pull_request:
    branches: [master]

jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - id: review
        uses: your-team/ai-reviewer@<sha>
        with:
          api-key: ${{ secrets.LLM_PROVIDER_API_KEY }}

One security note, more important in 2026 than ever: pin actions to a specific commit (@<sha>), not to a moving tag like @v1. A tag can be shifted to different code without your knowledge, and the action runs inside your pipeline with access to secrets – pinning to a SHA freezes exactly the code you have verified. It’s the same principle of least privilege, but applied to third-party code you allow into your process.

At this point, the agent is packaged once, and you plug it into any repository with a single line.

The initial context is somewhat different, however. Sometimes, instead of extensive framing, shaping, or fetching data from the web, a brainstorming note is enough that preliminarily defines the requirements for what we want to achieve. We tentatively call this file requirements.md and place it in the folder with the new change.

For me, it looks as follows:

## Overall concept

- GHA workflow run for every new pull request to master
- composite action for the review itself so that main workflow is easy to reason about

## Input parameters

- pull request title
- pull request description (?? cost tradeoff)
- git diff

## Code Review Criteria

Each criterion is scored on a 1–10 scale, where 1 is the worst outcome and 10 is the best.

{{CR_CRITERIA}}

## Parked for later

- business alignment (require broader context)
- architectural fit (require broader context)

## Expected side-effects

- PR comment with summary
- labels: `ai-cr:failed` (red) OR `ai-cr:passed` (green)

## Expected behavior

- on-demand retry when label `ai-cr:review` is added

The {{CR_CRITERIA}} fragment is a list of subjective expectations regarding pull requests and diffs. Each team and developer will define it somewhat differently. Common expectations for every change usually concern readability, complexity, presence of tests, or documentation of changes. The AI itself will help you discover the most important practices, conventions, and patterns widely used in your stack. But the key contribution is yours – internal conventions and requirements that existed long before the agent's integration.

Having such a document, we can move on to the previously learned steps of codebase recognition and planning upcoming changes.

# Research
/10x-research ci-cd-code-review based on requirements from '@context/changes/ci-cd-code-review/requirements.md'

# Plan
/10x-plan ci-cd-code-review Plan the implementation of ci/cd workflow based on provided research and @context/changes/ci-cd-code-review/requirements.md

Note that this is the same pattern you use for the "regular" application code - we treat the CI/CD scenario like any other feature to be implemented, rather than like some magical configuration to copy from Stack Overflow.

We pay attention to what was included in the first version of the requirements. This is intentionally a narrow scope – the equivalent of an MVP, but for the pipeline rather than the application. Just as your application starts from a minimal version and then grows, the entire CI/CD process can begin with the simplest meaningful step and be developed to a large scale.

Here is what this first draft consisted of:

  • Concept - workflow triggered for every pull request to master, with the review itself separated into a composite action so that the main scenario is easy to grasp (exactly those two decisions we discussed above).
  • Input for the agent - the PR title, its description (with a note that it is a conscious cost compromise), and the git diff. As much as the agent realistically needs to assess the change.
  • Evaluation criteria - six dimensions scored on a scale from 1 to 10 - for us these are implementation correctness, idiomatic usage, complexity, test/ risk coverage, documentation, and security. Each criterion has a described state for "1" and for "10" so that the agent's evaluation is not discretionary.
  • For later - business and architectural alignment. Both require a broader context than just the diff itself, so they are not included in the MVP but are recorded as a direction for development.
  • Side effects - a comment with a summary in the PR and the labels ai-cr:passed (green) or ai-cr:failed (red). These turn the "rating" into a visible signal for the team.
  • Behavior - restarting the review on demand by adding the label ai-cr:review.

We do not try to immediately build the "perfect" reviewer who understands the entire architecture and product strategy. We start with a narrow but complete loop - a diff goes in, a verdict and label come out - and we add other dimensions (architectural context, business fit, richer input) iteratively, exactly as you develop any other part of the system.

And now for one key question - how do these three inputs from requirements.md, namely the PR title, description, and git diff, physically get from GitHub to the agent? We take the title and description directly from the pull_request trigger payload (github.event.pull_request.*), and we calculate the diff on the runner relative to the base branch and pass it to the action via with: - using the same inputs channel through which we previously provided the api-key:

# .github/workflows/review.yml - consumer fragment
- uses: actions/checkout@v4
  with:
    fetch-depth: 0   # full history - otherwise you cannot compute the diff against the base

- id: diff
  run: echo "value=$(git diff origin/${{ github.base_ref }}...HEAD)" >> "$GITHUB_OUTPUT"

- uses: your-team/ai-reviewer@<sha>
  with:
    api-key:  ${{ secrets.LLM_PROVIDER_API_KEY }}
    pr-title: ${{ github.event.pull_request.title }}
    pr-body:  ${{ github.event.pull_request.body }}
    diff:     ${{ steps.diff.outputs.value }}

On the action side, this means extending the inputs contract in action.yml with pr-title, pr-body, and diff — exactly the same move we used to add api-key there. One pitfall worth remembering: the default checkout is shallow, so without fetch-depth: 0, the git diff command has nothing to compare changes against and will return empty.

Claude in GHA - Claude Action

So far, we have been assembling everything manually: our own agent on the SDK, a composite action as a wrapper, the npm run review step, passing secrets. This provides maximum control – we are the owners of the prompt, response schema, and gateway ourselves. But for a large class of cases, building such a setup from scratch is overkill. Anthropic provides a ready-made solution directly from the GitHub Marketplace: Claude Code Action (anthropics/claude-code-action@v1).

In short: it’s the same Claude Code you know from the terminal, but running on a GHA runner. Instead of writing an agent, you get the entire harness—with access to the repository, tools for reading files, commenting on PRs, and leaving reviews—and you control it with a single prompt field. The action itself detects the mode of operation based on the context (so-called intelligent mode detection): it behaves differently when invoked by mention @claude in a comment versus when triggered automatically on a pull_request event.

Example 1: interactive mode (@claude in the comment). The simplest input. After setting up this scenario, anyone on the team can invoke Claude directly from the discussion under a PR or issue - "@claude explain what this module does" or "@claude suggest a fix for this bug" - and the reply will come back as a comment:

# .github/workflows/claude.yml
name: Claude - interactive mode

on:
  issue_comment:
    types: [created]
  pull_request_review_comment:
    types: [created]

jobs:
  claude:
    runs-on: ubuntu-latest
    steps:
      - uses: anthropics/claude-code-action@v1
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
          github_token: ${{ secrets.GITHUB_TOKEN }}

Example 2: automatic review of every PR. The same action, a different trigger, and an explicit prompt – and the on-demand assistant becomes a reviewer who speaks up on every pull request. This is a direct equivalent of what we built manually, just without writing an agent:

# .github/workflows/claude-review.yml
name: Claude - automatic review

on:
  pull_request:
    types: [opened, synchronize]

jobs:
  claude:
    runs-on: ubuntu-latest
    steps:
      - uses: anthropics/claude-code-action@v1
        with:
          prompt: |
            Review this PR for correctness, security, and readability.
            For each changed file, provide concrete suggestions.
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
          github_token: ${{ secrets.GITHUB_TOKEN }}

Example 3: review according to your criteria. prompt is any text, so there is nothing preventing you from pasting the same list of criteria we developed for our own agent – or directly specifying a skill from the repository. This is exactly how we integrate the course skill 10x-impl-review-ci into CI, as shown in the Deep Dive section. Thanks to this, Claude evaluates the PR not generically, but according to your Definition of Done, and the result is returned, among others, as a structured output (JSON) for further steps in the scenario (e.g., merge gate).

What do you really gain compared to manual assembly? Primarily time: you get the first review in minutes, without writing an agent, integrating with the GitHub API, and handling comments or labels. You get this out of the box, along with repository reading and inheriting skills from .claude/skills (we'll return to this in the Deep Dive). Moreover, the whole process runs on your runner, and you can connect the model not only via the Anthropic API but also through AWS Bedrock or Google Vertex AI.

So when is it worth assembling an agent manually, as in the previous sections? When you need full control over the loop: your own enforced verdict scheme (Output.object), a hard gate on a specific score field, custom tools, or a cheap model outside Anthropic for mass runs. These are exactly the two paths we develop in the Deep Dive: a ready-made agent (Claude Action) reusing the entire skill versus an SDK agent where you own the loop yourself. The rule is simple - start with the ready-made and only move to manual work where the ready-made actually limits you.

Just like with composite actions: ultimately pin the action to a specific commit (@<sha>), rather than to a moving @v1 - it's the same third-party code with access to your secrets, which we mentioned above.

Testing changes in Agents with promptfoo

The agent is already running on pull requests and finally returns the first verdicts. And here an interesting trap begins.

After the first successful run, a carousel of optimizations starts spinning in your mind. Maybe swap the model for a cheaper one to cut costs? Or maybe a more expensive one to capture more? Shorten the prompt with criteria or expand it? Give the agent more time to "think" or less? Each of these changes sounds reasonable, and after each one, you glance at one, maybe two diffs and mumble under your breath "probably better".

And that "probably" is exactly the problem. You change one parameter, check it on a random PR, and draw a conclusion from a sample size of one. This is not testing, it's reading tea leaves.

promptfoo comes to the rescue - a tool for testing and evaluating prompts and entire agents. In short: instead of judging a change by eye, you describe a set of cases and expectations once, and promptfoo responds whether your modification is better or worse. The question "model A or B" ceases to be a matter of intuition.

The whole idea of promptfoo comes down to three things gathered in one file promptfooconfig.yaml:

  • prompts - the prompt (or prompts) of the agent you want to test.
  • providers - models or entire configurations that you want to compare with each other. This is where you list side by side two (or five) candidates.
  • tests - a set of inputs along with assertions (assert), i.e. conditions that the result must satisfy.

promptfoo takes each test and runs it through each provider, building a matrix of results from this. For the code review agent, a single test is a pre-prepared diff, and the assertions describe what a good verdict must contain.

In the providers section, OpenRouter works great - one API key gives you access to models from different providers, so you settle the whole "maybe cheaper, maybe from another vendor" carousel without juggling multiple accounts. You specify models with the prefix openrouter: and the name in the format vendor/model:

# promptfooconfig.yaml
providers:
  # cheaper candidate
  - id: openrouter:anthropic/claude-haiku-4.5
  # pricier, stronger candidate
  - id: openrouter:anthropic/claude-sonnet-4.6
  # non-Anthropic model, same key
  - id: openrouter:openai/gpt-5-mini

prompts:
  - file://prompts/review.txt

tests:
  - vars:
      diff: file://fixtures/sql-injection.diff
    assert:
      # 1. the output is valid JSON (the gate stands on it)
      - type: is-json
      # 2. a second model evaluates the comment content against a criterion
      - type: llm-rubric
        value: The verdict rejects the change and points out the SQL injection vulnerability
      # 3. a hard condition on the score field from our Definition of Done
      - type: javascript
        value: JSON.parse(output).score <= 3

You provide the key once, as the environment variable OPENROUTER_API_KEY, and then you run the evaluation with a single command:

npx promptfoo eval

The most important assertion types for our case can already be seen in the config above:

  • is-json - the result is a valid JSON (and optionally matches your Definition of Done schema). This ensures that the agent returns the structure on which the gateway is based.
  • llm-rubric - the second model evaluates the content of the comment according to the described criterion, e.g., "the justification points to a specific line, not a generality".
  • javascript / python - custom code checking strict conditions, e.g. that score is within the 1-10 scale or that a known, introduced vulnerability received a rating below the threshold.

The result is a readable table: each row is a test, each column is one of the models, and in the cells you see pass/fail along with the cost and response time.

There is also a second, more subtle reason why it is worth having such a set of tests. You edit the prompt so that the agent better catches one type of error, and you do not notice that it has stopped detecting another one at the same time. Such a silent quality regression does not stand out in a single PR, but it is caught by the set of cases run every time.

The default threshold is absolute: just one failing case is enough for the entire evaluation (and the CI step along with it) to turn red. If your set tends to be "noisy," you can relax the threshold with the variable PROMPTFOO_PASS_RATE_THRESHOLD (e.g., 95), but initially keep the strictest standard – it's better for the agent to have to earn the green light.

Just like the agent itself and the CI/CD scenario, we conduct the integration of promptfoo with the project using our proven 10x-workflow - first research, then planning. The research phase doesn't immediately ask "how to hook up promptfoo," but checks whether our code is even suitable for evaluation: whether the prompt and the agent from @packages/code-reviewer can be reused and imported into tests.

# Research

If my tech stack is aligned with this tool, go in that direction. Otherwise, you can analyze other oss tools allowing me to eval my prompts and agents. Use Web Search or context7 to get the most up to date docs.

# Plan
/10x-plan code-review-evals Plan how to introduce promptfoo within '@packages/code-reviewer'. My goal is to create first configuration, allowing me to test the same code review prompt on three different models (z-ai/glm-5.1 and deepseek/deepseek-v4-flash). For test cases, there should be one, rather complex diff migrating React 16 component into React 19+ with three impactful flaws in it. LLM-as-a-judge should verify whether code review results correctly identify what is broken. You can also add static test verifying if code review actually fail.

Note two things. First, the research leaves AI a loophole: if promptfoo does not fit our stack, the agent is to look for another open-source tool – we leave the tooling choice for verification instead of assuming that the first intuition is correct. Second, the plan turns this research into a specific configuration that we discussed above: the same prompt for review run on three models side by side, one complex diff (migration of a component from React 16 to 19+ with three real errors) as a test case, and two layers of assertions – an LLM-as-a-judge checking whether the review caught the broken parts, and a strict static test ensuring that such a change actually does not pass.

More possibilities for the agent

Although we build the "agent" throughout the entire module, in the usage of the SDK so far the difference between it and a regular chatbot is minimal. We compose the input prompt, append git diff, call the model once, and wait for a single structured output. Even when using ToolLoopAgent from the Vercel AI SDK, we use it in the simplest possible variant - as a wrapper for a single LLM call with Output.object:

// Reviewer as a "scorer" - single input, single output, zero tools.
const agent = new ToolLoopAgent({
  model: provider(modelId),
  instructions: REVIEW_INSTRUCTIONS,
  output: Output.object({ schema: ReviewResult }),
});

const { output } = await agent.generate({ prompt: buildReviewPrompt(input) });

In this form, the entire ToolLoopAgent class looks like unnecessary ceremony - we could just as well call generateText and that would be it. And that's fine. A narrow, single-step loop "diff goes in, verdict comes out" is a conscious decision from our MVP, not an unfinished feature.

So what do we gain by sticking to this construction? The answer is hidden in the name - ToolLoopAgent. At the moment you decide that the agent should have more agency, you don’t rewrite anything from scratch. The same model, the same instructions, the same class - you add the tools field, and the agent transforms from a one-time "evaluator" into something that can read, decide, and act in a loop.

How the first tool works. A tool (tool) in the AI SDK is three things combined:

  • description - a natural language description based on which the model decides whether and when to use the tool.
  • inputSchema - a (Zod) schema describing the arguments. The SDK uses it bidirectionally: it provides the model with the shape of the input and validates what the model actually generates.
  • execute - an asynchronous function that does the actual work and returns the result back to the model.

Let’s add the agent's first, purely read-only tool - reading the implementation plan from the local ./context folder, so the agent can evaluate the diff relative to the plan that this diff supposedly implements:

import { tool } from 'ai';
import { z } from 'zod';

const readPlan = tool({
  // The model reads this description to decide whether to call the tool.
  description:
    'Read an implementation plan from context/changes/<change-id>/plan.md. ' +
    'Returns the plan contents, or { found: false } when none exists.',
  inputSchema: z.object({
    target: z.string().describe("A change-id (e.g. 'oauth-login') or a plan.md path."),
  }),
  execute: async ({ target }) => {
    const contents = await readPlanFromContext(target); // read from ./context
    return contents ?? { found: false };
  },
});

Registering a tool is literally one field - tools - added to the same class we used earlier:

const agent = new ToolLoopAgent({
  model: provider(modelId),
  instructions: REVIEW_RUNNER_INSTRUCTIONS,
  tools: { readPlan }, // <- the whole difference from the "scorer" version
});

We are building an agent from scratch here, so we write the file-reading tool ourselves. If instead you use a ready-made agent - Codex SDK or Claude Agent SDK - repository reading is already included. Rather than adding a tool, just extend the prompt (tell the agent that the plan is in ./context) and possibly adjust file permissions - the rest, i.e., finding the right file, the agent will handle on its own.

From this moment, calling the agent stops being a one-shot - it starts a tool loop:

At each step, the model either generates the final response (and the loop ends) or requests to call a tool. If it requests a tool, the SDK validates the arguments against the inputSchema, runs execute, attaches the result to the context, and returns the next turn to the model. The model sees the outcome and decides what to do next – either reach for another tool or close the review. This loop is what turns the LLM wrapper into an agent.

Agency under cost control. Each loop iteration is another model call – meaning additional tokens, cost, and time. Therefore, "more agency" goes hand in hand with a session length gate. The AI SDK by default stops the agent after 20 steps (stopWhen: stepCountIs(20)) – this is an intentional safeguard against runaway loops. You consciously change the threshold when the task requires it:

import { stepCountIs } from 'ai';

const agent = new ToolLoopAgent({
  model: provider(modelId),
  tools: { readPlan /* , ... */ },
  stopWhen: stepCountIs(8), // a short, predictable review session
});

🚨 There is also isLoopFinished(), which completely removes the step limit and allows the agent to operate until it stops calling tools on its own. It sounds convenient, but without an upper limit one runaway run can burn through the budget – and on CI/CD, where the scenario runs on every PR, the cost multiplies by the number of pull requests. Keep a hard step limit. Additionally, the onStepFinish callback provides insight into token usage step by step, so you can measure the session cost instead of guessing.

The ladder of agency: from reading to acting. The first tool was reading and local. But since the mechanism is always the same - description, diagram, execute - you add subsequent tools with the same move, climbing the ladder of agency:

  • Reading local context (read) - reading the ./context folder, implementation plans, team conventions, or the Definition of Done. The agent evaluates diffs not in a vacuum, but with respect to the documents you already have in the repository.
  • Writing to GitHub (write) - instead of returning a verdict as text, the agent itself comments on the PR via gh/GitHub API and assigns labels ai-cr:passed or ai-cr:failed. The side effects from our requirements stop being something we add around the agent - they become its tools.
  • Working across the entire GitHub - since the agent has access to the API, its domain is the entire repository, not just a single diff: related issues, status checks, previous PRs, file change history.
  • Going outside GitHub (read/write) - there is nothing preventing the tool from reaching another service: updating a ticket in Linear or Jira, sending a notification to Slack, or fetching context from an internal knowledge base.

Notice the shift. The agent ceases to be a "text input, verdict output" function and becomes an actor in your process – reading, making decisions, and taking actions where the team actually works. And this whole path, from the scorer to the actor, does not require an architectural change. It requires adding successive tools to the same loop, step by step, exactly as you develop any other part of the system. One such extension – a change plan as a full-fledged input to review – is developed in the Deep Dive section.

🧑🏻‍💻 Practical tasks

Before you start the tasks, one organizational note about the 10xChampion badge: you earn it with one practical project — either a pipeline for code review (tasks from M5L2 and M5L3) or an AI artifact registry (M5L4). Choose the path closest to your daily work. When the pipeline runs on the PR, take screenshots as proof for 10xChampion: a view of the pipeline with at least one job, job logs, and a comment from the LLM on the PR (the same categories announced in M5L1, Step 4).

The following tasks will guide you through the entire review contract on your own repository - from setting the criteria, through enforcing the verdict format, to comparing models. The first three build on each other, the fourth is an optional extension.

  1. Define the criteria for a good review. In a separate, fresh session, have a conversation with the agent about what a good code review looks like in your stack and what your acceptance criteria for pull requests are. The goal is to come away from it with five concrete criteria that you want to communicate to the agent operating on CI/CD. AI will help you discover practices and conventions typical for your ecosystem, but it is your internal requirements - existing long before the agent - that are the key contribution here.

  2. Embed the criteria into the agent with a forced response format. Provide these five criteria to the agent—either as part of the system prompt or together with a diff in the prompt. The most important thing: ensure a structured output and a schema that enforces a specific response format (for example, an evaluation per criterion plus an overall verdict). This schema transforms a vague "assessment" into something the pipeline can use to gatekeep and mechanically approve or block the change.

  3. Compare the models using evals. Create a short set in promptfoo on the same diffs and run 2-3 different models side by side through it. The idea is to decide the question "cheaper or more expensive model" with a hard matrix of results (pass/fail plus cost and time), rather than a gut feeling from a single PR. In the process, leave this set as a regression gate before further prompt changes.

  4. (Optional) Add agency to the agent. Try giving the agent 1-2 additional tools and extend the base prompt. Consider which additional actions realistically change the quality or usefulness of the review – whether at the project level (reading the plan, conventions, Definition of Done) or external services (comment in PR, label, ticket in Linear or Jira). Start with the read mode, and only then consider tools that modify the system state.

Implementation review on CI/CD - 10x-impl-review-ci

In the previous lesson, we left an open question: since you wrapped the repository with rules, skills, and conventions for the agent across four modules, will the agent you assemble under CI even use it? The answer depended on the SDK category. A ready-made agent (Claude Agent SDK) inherits these files by itself because it's the same harness you use in the terminal. An agent assembled from the AI SDK inherits nothing – what it needs to know, you load through your own code.

Now the same axis returns in the context of the pipeline, with a concrete example. In the course toolkit, you have a ready-made skill: 10x-impl-review-ci, which you download via the course CLI and invoke as /10x-impl-review-ci. This is a modification of the already known skill for CI/CD without an interactive mode and questions for you – it compares the implementation from the pull request with the plan that the PR supposedly executes, evaluates the change in seven dimensions, and issues a verdict of APPROVED, NEEDS ATTENTION, or REJECTED.

The 10x-impl-review-ci skill may be stored in a public repository. This is an allowed exception relative to the rest of the 10x-Workflow skills, making it easier to get a quick start with public agents running on CI/CD.

Most importantly, the same skill can be reused in both paths from the previous lesson. More agent autonomy, but also uninterrupted operation on CI/CD.

  • SKILL.md - execution mechanics. This contains all the "plumbing" for a specific runtime: how to find the plan in the repository, how to calculate the diff against the base branch, how to split the analysis into subagents, how to save the report, commit it with [skip ci], and expose a status check that will block the merge. This layer is tightly integrated with Claude Code – it uses subagents, tasks, and the MCP tool for inline comments.
  • skills/10x-impl-review-ci/references/impl-review-instructions.md - same evaluation criteria. Here is the knowledge about what exactly we compare and how we evaluate it: the plan as the source of truth, three independent analyses (drift relative to the plan, code safety and quality, test coverage), seven evaluated dimensions, and the findings grammar - severity, impact, and improvement proposals. This file assumes that you already have the plan and diff in hand, and deliberately does not say a word about where to get them or where to send the result.

SKILL.md does not rewrite the criteria itself - it refers to the instruction file at every step of the evaluation (progressive disclosure). This is exactly the same move you make in normal code when you separate domain logic (criteria) from the integration layer (mechanics). And this division determines how you hook the skill into each of the two paths.

Path 1: ready-made agent – you reuse the entire skill. If your agent in CI is Claude Code Action (that is the same Claude Code, but running on the GHA runner), you have a runtime at hand that discovers skills in .claude/skills on its own in the terminal. So you just need to provide it the skill and call it – exactly as you would do manually, but in a dedicated workflow step. In a production scenario, this boils down to two steps:

# .github/workflows/review.yml - fragment
- name: Expose the skill to the agent
  run: |
    mkdir -p "$HOME/.claude/skills"
    cp -r .github/skills/10x-impl-review-ci "$HOME/.claude/skills/"

- name: Run review in CI
  uses: anthropics/claude-code-action@v1
  with:
    anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
    prompt: |
      /10x-impl-review-ci
      You are running in CI on a pull request. Execute the skill: compare the PR
      with its plan, save a report, and post a comment on the PR.

Path 2: foldable agent - you reuse just the criteria. Vercel AI SDK has no concept of a skill or the .claude/skills folder - no one will load the SKILL.md for you, and even if they did, its mechanics assume Claude's runtime, which is not present here. But the criteria layer is agent-agnostic: it is pure knowledge about how to evaluate a diff, independent of who performs it. You extract it and inject it into the agent as additional context - through a tool that reads the same instruction file and delivers its content to the model:

import { tool } from 'ai';
import { z } from 'zod';

// The same criteria as in the skill - loaded from a file and passed to the agent.
const readReviewCriteria = tool({
  description:
    'Read the implementation-review criteria: the rubric for judging whether ' +
    'a diff faithfully implements its plan. Call this after a plan is found.',
  inputSchema: z.object({}),
  // loadImplReviewCriteria() reads references/impl-review-instructions.md
  execute: async () => ({ criteria: await loadImplReviewCriteria() }),
});

Thanks to this construction, the tool reads the rating column. The agent calls it when it finds a plan, receives criteria in response, and evaluates the diff against them. You write the mechanics (where the plan comes from, where the result goes) your way, but you don’t invent the criteria from scratch – you reuse the same prose that drives the ready skill.

That is exactly the purpose of this division. Evaluation criteria are a durable resource: you transfer them between the harness, your own agent on the AI SDK, or in extreme cases print them as a simple checklist for a human—without dragging along all of Claude's hydraulics. The mechanics, however, are interchangeable: you change the CI provider, runtime, or the way PR comments are made and only replace the SKILL.md, while the criteria remain untouched.

There is also the other side of the contrast that appeared with our agent. We enforced the response structure with a schema (Output.object with the ReviewResult schema) because we own the loop ourselves. The skill keeps the criteria as plain prose in Markdown, because in path 1 the harness itself loads the file and allows the model to structure the report in its own way. The same goal, a different strategy – and the division into two layers makes the prose with criteria portable regardless of what you run it in (including your agent from path 2, which adds its own schema on top of it).

Under the hood of the skill

To run the 10x-impl-review-ci skill in, say, the AI SDK, we need a way to provide the agent with additional context (the skill itself and the plan against which each successive review runs). To do this, we extend ToolLoopAgent with new tools:

  • readPlan - reads the plan from context/changes/<id>/plan.md:

    // description — the model uses it to decide when to reach for the tool
description:
  "Read an implementation plan from context/changes/<change-id>/plan.md. " +
  "Accepts either a change-id (e.g. 'oauth-login') or a path to a plan.md " +
  "under context/changes/. Returns the plan contents when found, or " +
  "{ found: false } when no such plan exists so you can proceed without it.",

  • readImplReviewCriteria - returns the evaluation rubric itself (the same criteria that live in the skill's references/impl-review-instructions.md):

    description:
  "Read the implementation-review criteria: the rubric for judging whether a diff faithfully " +
  "implements the plan it references (plan adherence and scope discipline). This is NOT a review " +
  "of the plan's own quality — it grades the implementation against the plan. Call this after " +
  "readPlan returns found: true, then review the diff against the plan using the returned criteria.",

  • postPrComment - publishes a comment on the PR (when we move this function into the agent, we should remove it from the .yml workflow itself)

How these three tools come together in a loop

We wrap the new tools in a prompt that, at the level of general requirements, splits into two parts:

  1. STEP 1 - always. Evaluate the diff against the six criteria and call postPrComment({ kind: "code" }). Unconditionally, on every run, for every verdict - even a clean pass with no findings gets a comment.
  2. STEP 2 - conditionally. The implementation review runs only when the PR explicitly points to a specific plan (a context/changes/<id>/plan.md path, a Plan: <id> marker, an unambiguous change-id).

This condition means that, at runtime, the agent decides on its own which path to take:

  1. Code Review only
  2. Code Review + Plan review

Keep in mind that opting for an agentic "review + plan review" raises the cost of the whole process and also introduces a new non-deterministic element into the entire integration. So, before you decide on this path, consider additional criteria, among others:

  • What kinds of PRs reach the agent at all?
  • What kinds of diff fragments are stripped out before the agent reads them (lockfiles, configs, etc.)?
  • Do we run Plan Review always, or only for PRs where Code Review passed green?
  • Should the whole process be handled by a single reasoning model, or are Code Review and Plan Review the work of two models (a more expensive one and a cheaper one)?

There are no ready-made answers here. The scale and volume of code review, together with continuous testing, are the best indicator of the direction you should take.

Experimental support for skills on the Vercel side

We just said that the Vercel AI SDK has no concept of skills - and at the level of the ai library itself, that is still true. But this boundary is starting to blur. Vercel has released experimental support for skills, but not in the core SDK; instead, it's in a separate package called bash-tool - the very same package that provides the agent with a sandbox containing a filesystem and a shell. It's worth knowing that this exists so you can consciously decide whether you even need it.

The mechanism boils down to a single function - experimental_createSkillTool. You point it to a directory with skills (folders, each having its own SKILL.md), and it returns three things: a skill tool that the agent uses to choose a specific skill, the files of that skill to be uploaded to the sandbox, and instructions for the prompt. You combine these three elements with createBashTool and provide them to the agent:

import {
  experimental_createSkillTool as createSkillTool,
  createBashTool,
} from 'bash-tool';
import { ToolLoopAgent } from 'ai';

// Folder with skills → tool + files for the sandbox + instructions
const { skill, files, instructions } = await createSkillTool({
  skillsDirectory: './skills',
});

const { tools } = await createBashTool({ files, extraInstructions: instructions });

const agent = new ToolLoopAgent({
  model,
  tools: { skill, ...tools },
});

Note the difference compared to path 2 from the main part of the lesson. There, we extracted only the criteria from the skill – plain prose from a single file – and injected them through our own tool. Here you load the entire skill as a folder: the model gets its files in the sandbox and accesses them via the shell, exactly like Claude's harness does, with progressive disclosure – it first sees the skill's metadata, and only accesses the content when it selects it. This is a closer representation of the original skill mechanics, but at the cost of adding a sandbox with bash to the agent.

Three caveats before you use this in production. First, the experimental_ prefix is not just decoration — the API is unstable and may change shape between versions. Second, this support lives in bash-tool, not in the SDK itself, so you’re bringing in the entire sandbox just to run the skill. Third, a skill tightly coupled to a specific SDK is harder to reuse later — the tool to load it exists, but the skill’s content itself may not transfer cleanly between different runtimes.

For our review in CI, path 2 - extracting just the criteria - is simpler and completely sufficient. But it’s good to know that the boundary "AI SDK doesn't know the skills" is now softer than it was just a few months ago.

Code review at the team level: what code alone cannot address

The agent goes through the first PRs, the gate works, the labels light up. It looks like a closed job – but that's just the starting point. When the same agent reviews the team's pull requests over the following months, issues emerge that you won't find in any review.ts or action.yml. This is no longer a code problem, but a problem with the process around it.

A good starting assumption: AI in review is throughput, not replacement. There are more and more agents writing code, so the bottleneck ceases to be writing changes and becomes reviewing them. The four threads below boil down to one thing: how to increase this throughput without removing humans from places where they are truly needed.

Triage: not every PR deserves the same treatment. A typo fix in documentation and a payment module refactor currently go through the exact same gate. This is wasteful on one hand and risky on the other. It is worth adding a cheap triage step upfront that, after several signals—the number and paths of changed files, diff size, labels—assigns a priority and complexity level to the PR. This level determines when and how intensively we run AI: trivial changes take a fast track, while those in sensitive areas undergo a deeper, more expensive process.

The result of the triage determines who has the final say. Since you already know the risk level of the PR, you can allocate attention based on it. Low-risk changes with a green verdict can be closed by the agent on your behalf. Those from critical paths always go to a human, and the agent only adds a preliminary opinion to them. This is not about "letting reviewers off the hook," but about directing human attention where it really matters, and taking it away from PRs that you would have just clicked through anyway.

Cost and time as a metric, not a surprise. Each run consumes tokens, money, and time. For a single PR, it's pennies, but with every pull request across the entire team, the bill starts to take on a life of its own. You already have the raw data at hand – the previously introduced onStepFinish callback reports token usage step by step. The missing element is pushing these numbers to an external tool, for example a Grafana dashboard or a simple table, and watching the cost per PR and per triage level over time. You manage what you measure – without this, you'll only notice the agent's cost on the invoice.

Learning loop: when a human really needs to look. The most important long-term signal is this: where did we call a human after the agent, even though it wasn’t necessary (AI was right, the reviewer just approved), and where the opposite happened – a bug landed in the PR that the agent let through as green? The first is wasted human attention, the second is a gap in the criteria. You collect both and feed them back into the system: you add the case to the set in promptfoo, close the criterion in the Definition of Done, adjust the triage thresholds. Over time, you stop guessing when a developer needs to look at the PR and start knowing it from the data.

Note that this all closes the loops we opened in the lesson. The Definition of Done and the eval set with promptfoo are not one-time configurations – these are places where knowledge from production returns. The agent itself in CI gives you a gate. Turning it into a lasting team practice involves triage, observability, and a feedback loop, none of which you will find in the agent's code itself.
