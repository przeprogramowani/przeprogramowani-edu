In the previous lesson, we laid the foundation for large projects: how to consciously manage what the agent sees at a given moment. This was a situation where you have control over the project. You can improve AGENTS.md, organize the documentation, create a lasting note-taking system, and gradually develop processes together with the repo.

And now you enter a project you don't know.

It may be legacy work. It may be an old module that you return to after a year. It may be a repo from another team, with documentation written back when the last deploy was still "temporary".

The first impulse is tempting: "Hey Agent, read the entire repo and explain the architecture to me."

Sounds reasonable. After all, we have large context windows, repo packaging tools, and models that can reason for a long time.

Except that in a really large project, such a prompt often fails right at the context window itself. A large window on the order of 200K tokens does not mean 200K tokens just for the code: within the same budget, system instructions, project rules, tool definitions, conversation history, and space for the model's response must fit.

The code also tokenizes unevenly. A simple converter "1 token ≈ 4 characters" only gives an intuition, because the number of tokens depends on the model, the text domain, and the language. In practice, 200K tokens corresponds to tens of thousands of lines of code in the best-case scenario, and after adding tests, configuration, documentation, generated files, and the overhead of the tools themselves, your budget disappears faster than the number alone suggests.

Without additional context management techniques, the task "explore this repo" often simply does not fit in the model window.

The second problem is more subtle. Even if a project fragment technically fits, "a lot of context" does not mean "good understanding." If you throw everything at the agent at once, you'll get a flat summary without priorities: some folders, a few relevant names, a couple of guesses, and a false sense that you understand the system.

In this lesson, you will learn a process that solves both problems. We tentatively call it Wide Scan -> Deep Focus. First, you perform a wide scan of the repository using cheap and deterministic CLI tools. This way, you can see which areas are key, active, coupled, and sensitive before asking the agent for costly code reading.

The result will be an artifact named Project Map.

Project map is an operational map of the legacy project territory. You collect cheap signals from the CLI, ask the agent for synthesis, and receive a short artifact: modules, entry points, dependency directions, cycles, sensitive areas, unknowns, and places that need to be safeguarded against major changes.

Why do we need this map? To avoid burning the context window on random file reading and not to change legacy blindly. The map tells the agent and the human: "here is the core of the system, here are the risks, here a change may pass through several layers, check the evidence here first."

Important: We present one of the example formats of such a map. In your projects, you can adapt these practices to your needs and expand the scope of the queries performed.

We will save it in the repo as context/map/repo-map.md. This is not meant to be an essay for a human or a chat dump. It should be an agent-friendly format: short sections, concrete lists, evidence traces from commands, explicit unknowns, and clear spots where one should be careful before making major changes.

Along the way, three working artifacts will be created in the same folder:

context/map/artifact-1-territory.md
context/map/artifact-2-structure.md
context/map/artifact-3-contributors.md

These are working notes from subsequent scans (the naming is completely subjective, you can organize it according to your own preferences). The final Project Map is created only after their synthesis in context/map/repo-map.md.

Why don't we start with the entire repo

The principle from the previous lesson is simple: the agent works better when given the right context at the right time. In a legacy project, the problem is more difficult because you don't yet have an organized memory of the project. You need to build it from scratch without wasting the context window on random file reading.

Two prework lessons lay the foundation for this decision: in [3.1] we said that the model does not have a magical project memory and operates only on what gets into the context window, and in [3.3] we added the Select strategy, that is, choosing the context needed for the next step.

This lesson is a practical application of both principles in legacy: you don’t throw everything in “just in case,” but first determine what is really worth reading. Similar to a programmer who, when settling down to fix a bug or make an improvement, more or less consciously selects which part of the project to “load into memory.”

When you ask the agent to read the entire repo, unfortunately you are mixing three different tasks:

  • searching the project structure (tool calle),
  • selecting areas important for architecture (reasoning + tool call),
  • deep understanding of specific behavior (reasoning).

That is too much at once. The agent may then correctly name several directories, but does not yet know which dependencies are critical, which entry points are actually used, where there are cycles, and where it is worth spending human attention.

Therefore, instead of a full dump, we first perform a broad scan and extract the context using traditional tools.

Note the difference between both approaches. CLI tools do not "understand the architecture" for you. They reduce the noise.

Instead of feeding the model hundreds of files, you provide it with command outputs: a list of entry points, a dependency graph, cycles, symbols, suspicious imports, and brief signals from the change history. The agent interprets smaller, better data.

This is the difference between "read everything" and "interpret the evidence".

Some of the tools that we call directly in this lesson can be executed "on the fly" by the agent during the reasoning process. However, our goal is for each tool call and every "peek" into a given area of the repo to be done deliberately, in order to obtain a specific and repeatable portion of information.

Wide Scan -> Deep Focus

Our two-step process for acquiring context from a large repo is a direct response to the limits of the context window and how unpredictable the behavior of LLMs can be without defined boundaries.

To understand the new project, we will therefore perform two stages of analysis:

Wide Scan is a broad, shallow analysis of the repository, for which a cheaper model is sufficiently reliable. You are looking for signals: module history, entry points, dependency directions, cycles, boundaries, active change flows, and areas that require caution. This is not yet a complete understanding of the system, but it narrows down further exploration.

Deep Focus is a deep analysis of one area, for which it is worth deploying a more powerful agent. You take one feature, one module, or one suspicious grouping of elements and track the actual data flow, behavior, risks, and areas for modernization.

In this lesson, we perform a Wide Scan and complete the Project Map. In the next lesson, we will choose one of the sensitive areas to enter Deep Focus.

The map itself is meant to answer the practical question: "where in this legacy do I need to be careful?".

This order of work protects you from two mistakes.

The first error is a random deep-read. The agent opens a file that looks important, then subsequent imports, then three neighboring modules, and after a dozen tool calls it turns out that it’s an old adapter that nobody uses anymore.

Technically it "read something," but you paid for it with the context window, tokens, and noise in the conversation. It's a pointless sieve: lots of traffic, few decisions.

The second mistake is a decision-less map. If the map does not indicate important areas of the project, potential risks, sensitive modules, and places for further inspection, it is just a picture.

A good map is operational. It should accelerate entry into key parts of legacy, not pretend that just drawing a graph means understanding the system.

A good map ends with the decision: "these areas are really important; here the change has the greatest chance to pass through several layers; here we first check the evidence, tests, PR history, and unknowns".

What the directory tree doesn't say

When you enter foreign legacy code, the file explorer gives a quick sense of control.

You see src/, pages/, services/, domain/, components/, shared/. You can expand the tree, count directories, find the largest files, and ask the agent: "describe the architecture based on the directory structure".

This is a good start, but a wrong model of reality.

Folders show a static snapshot: where something is located today. They do not show the project dynamics:

  • which places really change every week,
  • which folders are remnants of the old team division,
  • which modules always break together with a single change,
  • where the team most often fixes bugs,
  • which paths are "dead", and which are critical for the business,
  • who has hidden knowledge about a given area.

The classic search has the same problem. search: "payments" will find files with the word payments, but it won't tell you whether the payments are controlled by webhooks, jobs, the admin panel, feature flags, or manual data migrations. search: "auth" will show places where the name appears, but it won't distinguish the authentication core from the test helper, the old adapter, and the comment in the README.

The file explorer answers the question: "what is here?". The project map must answer the more difficult question: "what is important, connected, active, and risky here?".

At this stage, you don't need to evaluate the quality of each module yet. It's enough that you stop treating the directory structure as truth and start treating it as the first hypothesis to verify.

Effective Project Map

A useful map should answer several questions:

  • key modules and their role - which major parts make up the system and what they are responsible for,
  • change frequency - which areas are often touched in the git history and which appear to be stable,
  • sensitive areas - where a change can have a high cost: auth, payments, data, cache, integrations, migrations, runtime config,
  • coupling and blast radius – which modules are tightly connected through imports, cycles, shared helpers, public contracts, or frequent co-changes,
  • contributors - who most frequently worked on a given area and where hidden team knowledge may exist.

The most important thing is that the map should be useful for decision-making. It doesn't have to be complete. It should be good enough to say: "I understand the main layout of the project, I see areas requiring caution, and I know where a change might surprise me."

Natural language and predictable CLI

At this point, the agent starts to make a lot of practical sense. Not because it "magically understands the repo," but because it effectively connects two worlds: natural language and predictable CLI tools.

You can ask a question in plain language: "show me entry points," "check which modules import auth," "find places where the frontend directly accesses the database," "show files most frequently changed in the last year." The agent can translate such a question into a series of specific commands: rg, find, git log, import analysis, a dependency graph tool, or structural searching through AST.

As you already know, the CLI operates outside the model's context window until it returns results. The command can go through thousands of files, git history, or the entire import graph, and only a condensed result reaches the conversation: a list of paths, a change counter, a few cycles, a JSON snippet, a table of suspicious dependencies.

Then begins the part where the agent is most useful: you can talk with the results.

You don't have to know right away which command will be perfect. You can ask: "does this look like the core of the system?", "which of these results is noise?", "what should we check next?", "narrow this down to the payment module", "compare this with the change history". The agent guides you from a broad signal to the next, more precise question.

In Wide Scan, it's not about the agent coming up with the architecture on its own. It's about the loop:

Three components of the Project Map

To get a sense of a large legacy, you need three different questions and three different sources of signals that together build a single Project Map.

Territory — where the work actually takes place, which areas are active, which are frozen, which change together, and where frequent work may increase the risk of accidental regression.

Structure — entry points, system layers, import directions, connections, contracts, and places not visible from the directory tree itself.

Contributor context — who worked on difficult edge cases, protocols, migrations, permissions, or emergency fixes.

Each component answers a different question. Only together do they create the Project Map, on which it is worth making decisions. Important: you don't wait until the end to interpret. After each component, you record what already follows for work in legacy.

In the examples below, we work with the Mattermost repository, which is an open-source platform for team communication (backend in Go, frontend in React/TypeScript). It is a project with a long history, dozens of contributors, and several distinct layers: server/channels as domain logic, webapp/channels as the production UI, server/public as public models and API contract. And, most importantly, with a scale on the order of a million lines of code, which cannot fully fit into the context window of any model as of 2026. This is real legacy code that you can check yourself.

Familiarize yourself with this lesson using the example of Mattermost. Perform practical exercises on any project you have at hand — not necessarily the one from the lesson.

Territorial map — where the project lives

In this session, we do not start with a ready-made skill. At an early stage, short ad hoc prompts are sufficient. Only after several such exploratory sessions on various projects will you begin to notice repetitive procedures worth extracting into a separate, reusable skill for the agent.

Before you open any file, it is worth knowing which areas are active and which are frozen. Few things answer this question faster than Git history. But what is there to discover? After all, most of us use some client that can draw branches. We feel that this is not enough for a full understanding of inherited code – because activity is not only about changes and point operations, but a cross-sectional narrative about what happened in the project and thanks to whom.

Indeed - Git clients operate on the same models as Git itself - commits, branches, cherry-picks, fixes - you can see what's happening, but often to truly understand what is going on, that's not enough. That's why to navigate these complex structures, we invite an agent who has solved a few puzzles for us.

Structural Map — how it is built

The history of git tells "where to look". The next question is: "how is it built?".

Instead of opening additional files, we can use an agent to generate a useful dependency graph for a given code area.

Here, dependency-cruiser (depcruise) helps, which is a CLI that builds a static import graph of the entire codebase and emits it in many formats friendly for both humans and agents: DOT (rendered to SVG/PNG by Graphviz), Mermaid, JSON, Markdown, HTML, and CSV. For this lesson, we will use it in the JS/TS stack, and at the end of the section we will list alternatives for other languages.

It is worth knowing that from a single tool you can extract four different signals for the Project Map — and not only for the entire repo, but also for a single specified module:

  • Module territory map — by collapsing the graph to the folder level (--collapse), you will see modules as nodes, the direction of dependencies, and central hubs instead of hundreds of files.
  • Focus on one module — --focus or --include-only cuts out the selected area and its neighbors, making the graph clear and decisive.
  • Coupling metrics — --metrics counts the so-called afferent coupling (Ca, how many depend on the module), efferent coupling (Ce, how many the module depends on), and instability = Ce / (Ca + Ce). This turns "I feel this is highly coupled" into a concrete coordinate.
  • Cycles and tangled boundaries — the no-circular rule detects import cycles, which are a common source of hard-to-predict blast radius and a good candidate for Deep Focus targeting.

dependency-cruiser is installed like any Node.js tool, via npm, yarn, or pnpm, and run as a CLI where your JS toolchain operates. It supports JavaScript, TypeScript, LiveScript, and CoffeeScript, as well as ES6, CommonJS, and AMD modules, making it well-suited for frontends, Node.js backends, and JS/TS monorepos. If you only need quick cycle detection and a simple graph, reach for madge: it operates in the Node.js ecosystem and supports CSS preprocessors, but with TypeScript requires configuring tsconfig and resolvers.

How to tame an overly dense graph

The raw graph of the entire module almost always comes out as a hairball — hundreds of edges, unreadable, useless for decision-making. dependency-cruiser (and Graphviz itself) do provide specific levers that you can use to enforce the layout according to the question you ask. Ask the agent for one of these instead of settling for the first dump:

GoalLeverWhat it does
Top-down viewrankdir=TB (in DOT via reporterOptions.dot.theme.graph.rankdir or simply sed 's/rankdir="LR"/rankdir="TB"/')vertical hierarchy instead of horizontal
Collapse detail into structure--collapse "^src/[^/]+"nodes = folders, not individual files
Only upstream (who depends on X)--reaches "X"all modules that lead to X
Only downstream (what does X depend on)--focus "X" --focus-depth NX and its neighborhood to depth N
Ready architectural view--output-type archi (auto high-level)reporter designed for architecture diagram
Only hubs (degree filter)no native flag — compose from --metrics + --include-onlyshow only nodes with a high number of connections
No external packages (node_modules)--exclude 'node_modules' (not doNotFollow)cuts out react, redux, @floating-ui, etc. from the graph
Layout enginedot (hierarchical) instead of fdp/sfdp (force-directed)hierarchical DAG instead of a cloud of points

How to ignore node_modules and other noise. External packages almost never belong to the Project Map — you are interested in your code, not the graph of react or lodash. Beware of the trap: the default doNotFollow: 'node_modules' in the configuration does not remove the packages, it only stops delving into them — it still draws them as leaf nodes, which clutter the render. To make them disappear completely, use exclude (in configuration or with the --exclude flag). Use the same pattern for tests, stories, snapshots, and generated code: --exclude 'node_modules|\.test\.|\.stories\.|__snapshots__|\.d\.ts$'. Simply put, it's enough to say: "skip node_modules and test files on the graph".

You don’t have to remember all these flags. The whole fun of combining --collapse, --focus, --reaches, nesting thresholds, or rankdir is the agent’s job — it translates your human question into a specific depcruise command. You work at the level of requirements, not syntax: “the graph is too detailed, show only hubs,” “collapse to folders,” “flip to vertical,” “show only what depends on the work loop.” The agent generates, you look at the result and adjust it in the same natural direction — until the graph answers your question. This is exactly the "tool selection → goal → command → result → conversation" loop from the start of the lesson, just applied to the readability of the diagram itself.

Each of these tools has boundaries that are worth noting as unknowns in the map: jdeps does not see dependencies through reflection or Class.forName, goda does not automatically detect Go module boundaries, JS/TS tools lose imports due to misconfigured aliases, and the static graph will never show dependencies injected at runtime (dynamic require, build-time injection, feature flags, codegen). These gaps are part of the map, not its flaw.

Contributor map — who knows what and what to ask them about

This is the layer that is most often overlooked in legacy analysis. And that is a mistake.

The code can be read. Decisions, context, and edge cases often cannot. These things live in the minds of people who have touched the given area over the past year. The contributor map answers the question: "who to ask before making a change that someone already tried to make a year ago?". And here we return again to the power of git and the flexibility of agents.

Project map — final synthesis

After three components, you have a lot of signals, but you still don't have a map. A complete dependency graph is usually too dense to help in decision-making. If you paste an automatically generated image with hundreds of edges into the report, you get decoration, not a working tool.

The project map is meant to be a synthesis of evidence. You take results from the git history, dependency graph, and contributor context, and then you leave only what reveals the key and sensitive areas of the project.

Four Families of Repository Search

When working with a large repo, the question "how to search?" is more important than "which tool to use?".

You have four families. Each corresponds to a different type of question.

Lexical / pattern search (rg, grep) is text-based pattern searching. It is fast and cheap, but requires familiarity with the project's vocabulary. rg "handler|controller|route" will return a lot, but it won't distinguish between router registration, comments, or imports from tests.

Structural / symbol search searches by code structure or symbols — Probe, universal-ctags, SCIP, LSP, and similar code indexes. This is the proper step when you ask: "where do we define this symbol?", "which functions have this signature?", "where is the contract of this layer?". In this lesson, we treat this family as auxiliary guidance; the main structure signal comes from the dependency graph.

Semantic / RAG search uses embeddings over code chunks. It can help when you don't know the exact name, but it has its own costs: the index may be outdated, a chunk may cut off important context, and the result always requires verification in the code.

Agentic search is iterative narrowing by the agent: it makes a hypothesis, runs a tool, reads the result, changes direction. This is appropriate in Deep Focus. In Wide Scan it must be controlled, otherwise the agent deviates into a random area and uses the context budget on reading the old adapter.

Do not treat these families as a ranking. If you ask "where are the endpoints?", start with lexical or structural. If you ask "who imports the payment module?", take the dependency graph. If you ask "where is the definition of this type?", use symbol search. If you ask "which feature flows through these layers?" — that's a topic for the next lesson, Feature Analysis with AI (M4L3).

Boundaries of each component

None of the three components of the Project Map is an oracle.

Git history territory shows where the project has been touched — not why, nor whether it was right. An area with a large number of changes may be the product's core or a place where the same difficult bug was fixed for a year. Quarterly breakdown helps distinguish a stable core from a change campaign, but the conclusion is still a hypothesis. There is also a freshness trap here: history is evidence in a time window, not a snapshot of the state today. A file that changed a lot for a year may already have been deleted or rewritten — before basing your conclusion on it, check if it still exists in the repo. And the last limitation of co-changes: they show what changes together, but will never show a contract that should be synchronized but isn't in history (a classic: the backend model and its manually maintained counterpart on the front end). Such things remain as unknown to be checked in Deep Focus.

The structure from the dependency graph shows imports at a given moment. It does not show intent: shared/utils used by eight modules might be a well-thought-out cross-cutting concern or a catch-all bag for everything no one could name. A cycle between modules might be a real architectural problem or an artifact of build configuration. You decide.

Contributor context from commit history shows who touched the area — not who the formal owner is, not who is still in the company, not whether the decisions from those PRs are still valid. It is an entry point for conversation and reading PRs, not a list of authorities.

Two meanings of "semantic search"

It is easy to make a mistake when choosing a tool.

Some tools talk about "semantic code search," but it doesn't always mean embeddings and vector search.

Probe is a good example: it advertises itself as a "semantic code search engine," but its "semantic" means returning complete functions and classes instead of individual lines of text — based on AST parsing by tree-sitter, not embeddings. This is structural search, not RAG.

This distinction has practical significance:

FamilyMechanismRisk
Structural (Probe, ctags, LSP)Matching to symbol, function, or code structureRequires a good pattern or index; result still requires verification
Semantic / RAG (embeddings)Vector similarity of chunksStale index, poor chunking, probabilistic result

If you provide the agent with the result from the structural search tool and say "this is semantic search," the agent might start treating it as approximate similarity instead of precise structure matching. The map will receive the wrong type of confidence.

How to classify modules on the map

The module list alone is not enough. A good Project Map needs labels that help make decisions: where to read next, what not to touch by accident, which areas are just background, and which places have high sensitivity to change.

In the main part of the lesson, you use a dependency graph, change history, and contributor map to gather signals. Here, the next step is about distinguishing important signals from noise and recording them so that the agent can use the map in the next session.

It's not about an academic classification of architecture. It's about simple working labels that arise from evidence:

  • core / supporting / peripheral - whether the module is close to the main value of the product, supports it, or is an addition,
  • deep / shallow - whether the module contains real logic and many internal branches, or is a thin input layer or adapter,
  • stable / volatile / seasonal - whether the area changes regularly, hardly at all, or only during campaigns,
  • load-bearing / contained - whether many parts of the system depend on this module or its contracts,
  • high / medium / low sensitivity - how cautiously changes in this area should be handled.

The most important rule: a label without proof is a guess. If an agent writes that billing is core, they should show where they know it from: entry points, imports, change history, dependency paths, public types, workspace package, runtime configuration, or recurring co-changes.

In practice, the four questions are the most important.

First: is this the core? The core is not "the biggest folder" nor "the nicest name". The core is the area through which the main value of the product or a critical operational flow passes: authentication, payments, data, communication, orders, posts, permissions, synchronization, billing. Evidence may include an entry point, a public contract, an endpoint, a command, a job, a domain model, frequent use by other modules, or a connection to the main user flow.

Second: what is its coupling? Coupling refers to the degree of connection of a module with the rest of the system. In the Project Map, it is not about an abstract judgment of "good/bad coupling," but rather the question: what might move when I touch this spot?

You have several types of coupling that are worth distinguishing:

  • incoming coupling - how many places depend on this module; if many, changing the contract may have a wide blast radius,
  • outgoing coupling - how many places this module depends on; if many, its behavior may be fragile to changes in neighbors,
  • contract coupling - whether the module exposes types, API, events, data schema, or a shared package used by other layers,
  • runtime coupling - dependencies invisible in regular import: feature flags, DI, configuration, queues, webhooks, reflection, dynamic import, codegen,
  • co-change coupling - frequent co-changes in commits; in this lesson we treat it as a signal for the map, and leave the full analysis for the next lesson.

Third: what do the metrics from the tool say? Tools like dependency-cruiser do not indicate whether a module is "business-important." They measure the import graph. This is still very useful, but the metrics need to be read as signals:

SignalWhat it means practicallyHow to use in the Project Map
Ca / afferent couplinghow many modules depend on the examined modulehigh Ca suggests a contract or a system's supporting element
Ce / efferent couplinghow many modules the examined module depends onhigh Ce suggests a module that composes many external dependencies
instabilityratio of outgoing dependencies to the total dependencieslow value with high Ca may indicate a stable core; high value may indicate a thin adapter or orchestration
cyclesimport cycles between files or modulescandidate for caution area, as boundaries are tangled
orphan / leaf moduleselements without visible dependencies or dependentspossible periphery, dead code, or a separate entry point; requires verification

Do not save only the metric table in the map. Save the decision that the metric supports:

Module: server/public
role: supporting / contract layer

- high incoming coupling in the import graph

inference:
- changes to public types may propagate through several layers
caution:
- before making major changes, check for usage in the web app and server/channels
unknowns:
- the static graph does not show runtime dependencies or API compatibility on the client side

Fourth: what matters in static analysis of legacy? Static analysis shows the shape of the code without running the system. In legacy, the following matter most:

  • entry points: endpoints, pages, CLI commands, jobs, event handlers, webhooks,
  • direction of dependencies: who imports whom and whether the layers are not inverted,
  • contracts: public types, DTO, schema, shared packages, API clients,
  • graph centers: nodes with a large number of incoming or outgoing edges,
  • cycles: places where module boundaries are tangled,
  • adapters and thin inputs: files that appear important but only delegate further,
  • static analysis gaps: dynamic imports, DI, reflection, runtime configuration, generated code, feature flags.

The agent should receive not only the results but also the questions that enforce distinguishing proof from interpretation.

When performing static analysis of legacy code, ask the agent primarily about decisions, not about directory descriptions:

Which modules are probably core, but the evidence is weak?

Do not pick the first file just because it has a good name.
