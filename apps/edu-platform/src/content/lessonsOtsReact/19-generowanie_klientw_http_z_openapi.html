Wprowadzenie

W poprzedniej lekcji poznałeś popularne rozwiązania służące do komunikacji z backendem w środowisku Reacta i TypeScriptu. W takim podejściu, niezależnie od wykorzystywanej metody, narzut pracy i analizy zewnętrznego API pozostaje na programiście. Mowa tutaj zarówno o nawiązaniu podstawowej komunikacji, wymiany pierwszej porcji danych, ale również (szczególnie w aplikacjach opartych o TS) o typowaniu wszystkich obiektów i parametrów, które są wykorzystywane w takiej integracji.

Dodatkowym wyzwaniem, którego do tej pory nie omówiliśmy, może być niespójna dokumentacja - taka, która opisuje nieaktualne wersje endpointów lub błędne parametry.

Na szczęście, w świecie web developmentu został wypracowany standard, który stara się adresować większość wymienionych wcześniej problemów. Jest nim OpenAPI - to ujednolicony format opisu publicznych zasobów, z którego możemy skorzystać przy budowaniu integracji.

Standard OpenAPI

OpenAPI (dawniej Swagger https://swagger.io/specification/) to standardowy format opisu API w formie pliku (np. YAML lub JSON), który może być interpretowany w spójny i przewidywalny sposób. Dzięki OpenAPI można udostępnić zrozumiałą deklarację dostępnych endpointów, metod, modeli danych, a nawet informacji o zabezpieczeniach czy limitach połączeń.

Wykorzystywanie tego standardu wnosi kilka realnych korzyści:

  • Czytelność dla programistów - Publiczne API, wraz z wszystkimi jego cechami, może być opisane w jednym pliku, co ułatwia zrozumienie tego, jak przeprowadzić skuteczną integrację.

  • Interpretowalność dla narzędzi - Wykorzystywanie formatów YAML/JSON powoduje, że w oparciu o standard OpenAPI powstało mnóstwo narzędzi, które konwertują deklarację API np. na automatyczne klienty do pobierania danych (co przetestujemy w dalszej części lekcji).

  • Spójność i powtarzalność - Dokumentacja w formacie OpenAPI może być regularnie odświeżana i generowana na podstawie istniejącego kodu. Jej tworzenie może być uniezależnione od motywacji czy dobrej woli programistów, co niestety może skutkować nieaktualnymi informacjami dostępnymi publicznie.

Przykład dokumentacji

Poniżej możesz zapoznać się z przykładowym fragmentem dokumentacji opartym o OpenAPI - są w nim zawarte dwa endpointy (GET i POST) oraz kilka modeli opisujących oczekiwany format danych:

openapi: 3.0.3
info:
  title: Order Management API
  description: API to manage orders for an e-commerce platform.
  version: 1.0.0
servers:
  - url: https://api.example.com/v1
    description: Production Server
paths:
  /orders:
    get:
      summary: Get a list of orders
      description: Retrieve a list of all orders with optional filtering by status.
      parameters:
        - name: status
          in: query
          description: Filter orders by their status (e.g., pending, completed).
          required: false
          schema:
            type: string
            enum: [pending, completed, cancelled]
        - name: limit
          in: query
          description: Limit the number of orders returned.
          required: false
          schema:
            type: integer
            minimum: 1
            maximum: 100
        - name: offset
          in: query
          description: Offset the list of returned orders for pagination.
          required: false
          schema:
            type: integer
            minimum: 0
      responses:
        '200':
          description: A list of orders.
          content:
            application/json:
              schema:
                type: array
                items:
                  $ref: '#/components/schemas/Order'
        '400':
          description: Bad request due to invalid parameters.
        '500':
          description: Server error.
    post:
      summary: Create a new order
      description: Create a new order with the provided details.
      requestBody:
        required: true
        content:
          application/json:
            schema:
              $ref: '#/components/schemas/NewOrder'
      responses:
        '201':
          description: Order created successfully.
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/Order'
        '400':
          description: Invalid order data provided.
        '500':
          description: Server error.
components:
  schemas:
    Order:
      type: object
      properties:
        id:
          type: string
          description: Unique identifier for the order.
          example: "ord_12345"
        customerName:
          type: string
          description: Name of the customer.
          example: "John Doe"
        items:
          type: array
          description: List of items in the order.
          items:
            $ref: '#/components/schemas/OrderItem'
        status:
          type: string
          description: Current status of the order.
          enum: [pending, completed, cancelled]
          example: "pending"
        totalPrice:
          type: number
          format: float
          description: Total price of the order.
          example: 99.99
        createdAt:
          type: string
          format: date-time
          description: The date and time when the order was created.
          example: "2024-12-01T15:30:00Z"
    NewOrder:
      type: object
      required:
        - customerName
        - items
      properties:
        customerName:
          type: string
          description: Name of the customer placing the order.
          example: "John Doe"
        items:
          type: array
          description: List of items in the order.
          items:
            $ref: '#/components/schemas/OrderItem'
    OrderItem:
      type: object
      properties:
        productId:
          type: string
          description: Unique identifier of the product.
          example: "prod_54321"
        quantity:
          type: integer
          description: Number of units of the product.
          example: 2
        price:
          type: number
          format: float
          description: Price per unit of the product.
          example: 49.99

Taki opis można wystawić jako klasyczny, statyczny zasób w formie pliku, który mogą pobierać programiści budujący integrację w oparciu o opisywane API.

Można z niego wyczytać chociażby:

  • Wsparcie dla metod GET i POST, pod adresem /orders

  • Filtrowanie zamówień po parametrach “status”, “limit” i “offset”

  • Modele danych - Order, NewOrder oraz OrderItem, wraz z ich właściwościami

Ta definicja posłuży nam do zamiany wszystkich składowych w użyteczne typy i modele.

Klienty bez dodatkowych zależności

Jeśli nie chcesz korzystać z wrapperów na istniejące biblioteki do pobierania danych, takie jak openapi-react-query, jako alternatywę możesz przetestować generator Orval (https://orval.dev/).

Orval to narzędzie, które na podstawie istniejącej specyfikacji wygeneruje kod zgodny z preferowaną biblioteką, ale bez dodatkowej, ręcznie instalowanej zależności. Podobnie jak openapi-typescript, Orval wspiera klienty oparte o Fetch API, axios, React Query czy SWR. Naszą uwagę zwróciło eleganckie API i intuicyjna konfiguracja, która steruje zachowaniem generatorów.

W sekcji ćwiczeń przygotowaliśmy dla ciebie zadanie, które wykorzystuje właśnie to narzędzie - zobaczysz jak łatwo zintegrować je z istniejącym kodem React / TypeScript.

Generowanie dokumentacji w Node / Express

W tym module głównie skupiamy się na zagadnieniach frontendowych, więc z perspektywy komunikacji z API interesuje nas rola klienta korzystającego z istniejącej specyfikacji. Dla ciekawskich dodajemy jeszcze kilka informacji o tym, jak poradzić sobie z rolą producenta specyfikacji na backendzie.

W ekosystemie narzędzia openapi-typescript znajduje się rozwiązanie, które to zadanie ułatwia - jest to biblioteka o nazwie openapi-metadata, która umożliwia automatyczne generowanie dokumentacji na podstawie istniejącego kodu.

Jej działanie opiera się o trzy główne składowe - modele danych (klasy i pola), kontrolery (wystawiane publicznie endpointy), oraz docelowy dokument łączący wszystko w jedną całość.

Wspomniana biblioteka udostępnia zestaw dekoratorów TypeScript, które umieszczamy na wybranych elementach naszego API. Możemy dzięki nim definiować metadane endpointów oraz modeli danych, które następnie zostaną przekonwertowane na dokument zgody z OpenAPI.

import {ApiOperation, ApiResponse} from "openapi-metadata/decorators";
import {Request, Response} from "express";
import {users} from "../data/users";

export class UserController {

  @ApiOperation({
    operationId: "listUsers",
    methods: ["get"],
    path: "/users",
    summary: "List users",
    description: "List all users",
  })
  @ApiResponse({
    type: [User],
  })
  list(req: Request, res: Response) {
    res.json(users);
  }

}

Utworzony w ten sposób kod niestety traci na czytelności, ale w przypadku stabilnych, wolniej rozwijanych API, może to być akceptowalny koszt na rzecz automatycznego generowania specyfikacji dla wielu różnych klientów (np. frontendu, mobile czy innych). Warto również zadbać o odpowiednią separację logiki serwisu od samych endpointów tak, aby metadane i dekoratory dotyczyły najbardziej zewnętrznej warstwy naszej aplikacji.

Podsumowanie

Przedstawione w tej lekcji rozwiązania oparte o specyfikację OpenAPI pozwolą ci generować zarówno klienty HTTP, jak i dokumentację API w ujednoliconym formacie. Co dzięki temu zyskujesz i jakie są koszta takich rozwiązań?

Przede wszystkim, automatyzacja eliminuje potrzebę ręcznego pisania kodu odpowiedzialnego za komunikację z API. To z kolei zmniejsza ryzyko błędów związanych z niepoprawną interpretacją dokumentacji i dostarczaniem niewłaściwej implementacji. Generatory klientów zapewniają także lepszą spójność kodu i kompatybilność z publiczną specyfikacją API, co może ułatwiać utrzymanie i rozwój projektu. Generatory mogą również działać jak testy integracji - kiedy tworzony na ich podstawie kod będzie częścią scenariusza CI/CD, poprawny build całego projektu (lub jego brak) podpowie nam, czy spójność pomiędzy naszą aplikacją a ostatnią wersją specyfikacji jest nadal zachowana.

Nie można jednak zapominać, że automatyczne generowanie klientów HTTP ma swoje wady.

Aby umożliwić powtarzalne odtwarzanie klientów z istniejącej specyfikacji, wygenerowany moduł powinien być “zamrożony”. Tylko wtedy mamy pewność, że powstałe metody i modele odpowiadają temu, co deklarują twórcy danego API. Niestety, czytelność takiego kodu nie zawsze jest na akceptowalnym poziomie, a generatory mają tendencję do tworzenia nadmiarowych klas czy metod.

Dodatkowo, trzeba pamiętać, że deklarowana specyfikacja nie zawsze musi odpowiadać aktualnie opublikowanej wersji API. Może się zdarzyć, że błędy w integracji wynikają nie z tego, że docelowe API zachowuje się niepoprawnie, a z interpretacji nieaktualnej wersji specyfikacji. Zostajemy przez to obciążeni dodatkowym elementem do debugowania, co może opóźniać dostarczanie naszego projektu.

No i na koniec - techniczna kwestia związana z ekosystemem TypeScript. Duże specyfikacje mogą generować nadmiarowe typy, a ich interpretacja może z kolei obciążać sam kompilator. Warto więc podejść do takiego rozwiązania jak do eksperymentu, którego korzyści i koszta ocenimy z wielu stron, zanim uznamy go za obowiązujący standard w projekcie.
