Wprowadzenie

W ostatnich latach uwaga frontend developerów zaczęła się kierować w stronę serwera. Nowe środowisko dało więcej możliwości zarówno od strony czysto funkcjonalnej (np. łatwy kontakt z wieloma źródłami danych) jak i niefunkcjonalnej (np. optymalizacja renderowania). Frameworki takie jak Next.js, Remix czy OpenNext są odpowiedzią na te potrzeby, oferując narzędzia do skutecznego zarządzania danymi, renderowania oraz poprawy doświadczeń użytkownika.

Obecnie, w ekosystemie Reacta najwięcej uwagi skupiają na sobie:

  • Next.js - to obecnie najpopularniejszy full-stackowy framework w ekosystemie Reacta. Wzbogaca Reacta funkcjami takimi jak routing server-side, możliwość generowania stron statycznych, częściowe i przerywane renderowanie fragmentów stron, a także cache’owanie danych. Agresywna taktyka wydawnicza sprawa, że często wykorzystuje nie do końca sprawdzone funkcje Reacta lub wprowadza tzw. breaking changes między kolejnymi wersjami. Mocno związany z chmurą Vercela.

  • OpenNext - to otwarta inicjatywa wspierająca deployment Nexta na chmury inne niż Vercel. Kontrybutorzy starają się zachować ten sam zestaw funkcjonalności, które znajdziesz w oryginalnym imienniku, jednak z możliwością bardziej wszechstronnego wdrażania na dowolną infrastrukturę. Od niedawna z oficjalnym wsparciem Netlify i Cloudflare.

  • Remix / React Router v7 - to alternatywa Nexta budowana jako mało inwazyjny wrapper wokół React Routera. Remix ułatwia ładowanie i wstrzykiwanie danych po stronie serwera, prefetching, tzw. “optimistic UI” oraz lokalną obsługę błędów (error boundaries). Poza samym hostowaniem frontendu, podobnie jak w przypadku Nexta możesz tutaj wystawiać publiczne API dla twojej aplikacji oraz serwerowe metody do obsługi formularzy czy zmian na UI.

  • TanStack Start - to lekkie podejście do budowy aplikacji z wykorzystaniem Reacta, które kładzie nacisk na minimalizm i modularność. Framework bazuje na ekosystemie TanStack (np. React Query, TanStack Router i inne), pozwalając tworzyć wydajne aplikacje dzięki prostocie konfiguracji i pełnej kontroli nad narzędziami. Na dzisiaj na dość wczesnym etapie rozwoju.

  • Astro - to framework, który wyróżnia się możliwością budowy stron statycznych i hybrydowych z użyciem dowolnych bibliotek frontendowych, w tym Reacta. Astro stosuje technikę tzw. „zero JS by default”, co oznacza, że kod JavaScript dla komponentów Reacta jest ładowany tylko wtedy, gdy jest to konieczne. Dzięki temu aplikacje są wyjątkowo szybkie i zoptymalizowane.

Tworząc tę lekcję, zastanawialiśmy się który punkt z tej listy będzie dla was najbardziej interesujący. W sieci znajdziecie mnóstwo materiałów na temat Nexta czy Remixa (ostatnio, szczególnie w przypadku Nexta 15, nie zawsze pozytywnych), więc postanowiliśmy poszerzyć waszą wiedzę o tym, jak React odnajduje się w środowisku Astro.

Przed wami lekcja o fundamencie stacku Przeprogramowanych. Astro to framework, który w znakomity sposób balansuje możliwości i złożoność, oferuje użyteczne CLI i przede wszystkim nie wiąże rąk do jednej słusznej technologii. Jeśli pracujecie z Astro, będzie to dla was omówienie najnowszej wersji frameworka z premierą w grudniu 2024. Jeśli natomiast nie pracujecie, to będziecie mogli porównać jego możliwości z jego alternatywami (czekamy na wasz feedback!).

Pierwsze kroki z Astro 5

Nowy projekt w Astro możesz utworzyć w oparciu o wbudowane CLI - zacznijmy od komendy:

npm create astro@latest

Po jej wywołaniu pracę rozpocznie interaktywny przewodnik, który zapyta cię o konfigurację nowego projektu. W moim przypadku wybieram podstawowy starter, instalację zależności i nowe repozytorium GITa:

Struktura projektu sprowadza się do kilku plików statycznych:

/
├── .vscode/
│   ├── extensions.json
│   └── launch.json
├── public/
│   └── favicon.svg
├── src/
│   ├── assets/
│   │   ├── astro.svg
│   │   └── background.svg
│   ├── components/
│   │   └── Welcome.astro
│   ├── layouts/

├── .gitignore

├── package.json
├── README.md

Zgodnie z konwencją, praca będzie głównie dotyczyć tych folderów:

  • public - służy do wystawiania zasobów publicznych (pliki, obrazki, raporty, etc.)

  • src - kod źródłowy aplikacji

    • assets - statyczne pliki do importowania w komponentach

    • components - komponenty server-side i client-side

    • layout - layouty aplikacji

    • pages - strony wpływające na globalny router (file-based) oraz endpointy API

  • astro.config.mjs - globalna konfiguracja projektu

Do tego pełne wsparcie TypeScriptu z podstawowym configiem w tsconfig.json (zwróć uwagę na dziedziczenie konfiguracji z Astro).

Po utworzeniu projektu możesz podejrzeć projekt poprzez komendę:

npm run dev

W obecnej wersji (Astro 5) przywita nas prosta strona startowa oraz tzw. “Dev Toolbar”, czyli dodatkowe narzędzie do inspekcji zawartości strony. Będziemy z niego korzystać w dalszej części lekcji.

Astro CLI

Integracje i konfiguracja narzędzi często spędzają sen z powiek developerów, którzy muszą pracować w niekompatybilnym ekosystemie. Na szczęście Astro opiera się o system pluginów, które w łatwy sposób dodamy do projektu poprzez CLI.

Zobaczmy, jak łatwo dodać do projektu Reacta i Tailwinda. Wykonajmy następną komendę:

npx astro add tailwind react

W jej efekcie, Astro zapyta, czy zgodzimy się na instalację następujących bibliotek:

npm install @astrojs/tailwind@^5.1.3 tailwindcss@^3.4.16 @astrojs/react@^4.1.0 react@^19.0.0 react-dom@^19.0.0 @types/react@^19.0.1 @types/react-dom@^19.0.2

Po zaakceptowaniu tego polecenia, narzędzie wykona dodatkową pracę na poziomie projektu - zostaną utworzone dedykowane pliki konfiguracyjne, które zminimalizują czas i energię potrzebną do rozpoczęcia pracy z tymi narzędziami:

Zaktualizowany zostanie m.in. tsconfig.json (wsparcie dla składni TSX), a w projekcie dodatkowo powstaną dodatkowe konfiguracje (np. Tailwinda):

/** @type {import('tailwindcss').Config} */
export default {
	content: ['./src/**/*.{astro,html,js,jsx,md,mdx,svelte,ts,tsx,vue}'],
	theme: {
		extend: {},
	},
	plugins: [],
}

Pluginy zostaną również dodane do głównego pliku astro.config.mjs (sekcja integrations) - tutaj, w zależności od preferencji, Reacta może zastąpić np. Vue, Svelte czy inne wspierane frameworki:

// @ts-check
import { defineConfig } from 'astro/config';

import tailwind from '@astrojs/tailwind';
import react from '@astrojs/react';

// https://astro.build/config
export default defineConfig({
  integrations: [tailwind(), react()]
});

Jeśli zajdzie potrzeba dalszej konfiguracji Astro, w samym obiekcie możesz się opierać o konwencje znane z bundlera Vite - możesz korzystać ze znanych pól “server”, oraz “vite” do nadpisania ukrytej w Astro konfiguracji:

export default defineConfig({
  integrations: [tailwind(), react()],
  server: {
    port: 3000,
  },
  vite: {
    // ...
  },
});

Po dodaniu Reacta, zobaczmy jak teraz jak zachowa się pierwszy dodany komponent.

💡 Obecny stan aplikacji pokazuje podstawowe założenia Astro - to framework, którego priorytetem jest minimalizowanie dostarczanego do klienta JavaScriptu. Dopóki wprost nie zdecydujemy, że wybrany komponent powinien być renderowany w bardziej interaktywny sposób, Astro usunie z projektu wszystko to, co zbędne.

Kiedy może się to również okazać przydatne? W Astro, w naprawdę prosty sposób możesz tworzyć kod komponentu renderowany jednokrotnie, po stronie serwera - w momencie, kiedy utworzony zostanie sam HTML (np. zawierający informacje pobrane z bazy), Astro wyrenderuje go w sposób statyczny. Dzięki temu unikniemy zaśmiecania aplikacji np. runtime Reacta, który w niektórych typach aplikacji będzie zbędny.

W przypadku naszego Countera oczywiście tak nie jest - ten konkretny element powinien być interaktywny, więc zobaczmy, jak to osiągnąć.

Dyrektywy komponentów

Sposobem na to, aby w Astro poszczególne komponenty “wyciągać” ze świata serwera do przeglądarki, są tzw. dyrektywy client-side. Jedną z nich jest client:load - rozszerz plik index.astro o ten fragment:

<Layout>
  <Counter client:load />
</Layout>

Pisząc dyrektywę zauważysz, że strategii “ożywiania” komponentu jest jeszcze kilka. Szczegóły wszystkich dyrektyw znajdziesz pod tym linkiem - każda z nich wpływa na sposób ładowania zasobów do przeglądarki lub ogranicza je do momentu spełnienia konkretnych warunków.

A jak działa aplikacja po naszej zmianie? Counter zaczął być interaktywny, a wbudowany Dev Toolbar zaczął go zauważać i oznaczać. Dzięki dyrektywie rozbudowaliśmy aplikację w bardziej świadomy sposób - ten wymóg jawnych deklaracji dot. poszczególnych komponentów sprawia, że aplikacje Astro mogą działać naprawdę szybko.

W efekcie twojej zmiany, pliki tworzone w trakcie komendy npm run build również się zmienią - możesz porównać folder dist z tym, co widziałeś tam kilka chwil temu (więcej Reacta w przeglądarce).

Wyspy client-side i server-side

Astro określa swoje podejście do struktury aplikacji jako “island architecture” - to mechanizm separowania komponentów tworzonych w różnych technologiach, które mogą pochodzić z różnych źródeł i być integrowane niezależnie od pozostałych.

Jedną z wysp utworzyliśmy chwilę wcześniej - komponent Counter to wyspa izolująca fragment aplikacji Reactowej - w naszym przypadku to jeden prosty element interfejsu, ale mogłaby to być bardziej złożona część aplikacji (np. wykres).

Od wersji 5, Astro wspiera też tzw. “server islands” - są to dynamicznie przygotowywane fragmenty interfejsu, gdzie pracę delegujemy na serwer.

Zobaczmy, jak to działa.

Pierwsza wyspa serwerowa

Wyspą serwerową będzie komponent Astro (*.astro) działający i renderujący swoją zawartość po stronie serwera, oznaczony dodatkową dyrektywą server:defer.

Sprawdźmy ten mechanizm w akcji - najpierw utwórz komponent src/components/Episodes.astro.

Będzie on służył do wyświetlenia określonej liczby odcinków podcastu Opanuj.AI:

---
const {count} = Astro.props as {count: number}; // Propsy + TypeScript!

const PODCAST_URL = "https://anchor.fm/s/e2cb03d0/podcast/rss";
const response = await fetch(PODCAST_URL); // Top-level Await!
const feedXml = await response.text();

const itemRegex = /<item>[\s\S]*?<\/item>/g;
const items = [...feedXml.matchAll(itemRegex)];

const titles = items.map((item) => {
  const titleMatch = item[0].match(/<title><!\[CDATA\[(.*?)\]\]><\/title>/);
  return titleMatch ? titleMatch[1] : "";
});

const links = items.map((item) => {
  const linkMatch = item[0].match(/<link>(.*?)<\/link>/);
  return linkMatch ? linkMatch[1] : "";
});

const episodes = titles
  .map((title, i) => ({
    title,
    link: links[i],
  }))
  .slice(0, count);
---

<ul class='list-disc p-4'>
  {
    episodes.map((e) => (
      <li class='hover:underline hover:text-blue-400'>
        <a href={e.link}>{e.title}</a>

</ul>

Nasz nowy komponent wykona zapytanie z serwera po feed RSS, wybierze odpowiednie dane i przygotuje fragment widoku.

Dodajmy go do strony głównej, obok Countera - przy okazji, możesz od razu dodać tzw. fallback renderowany w momencie oczekiwania na dane (a’la Suspense w React):

---
import Counter from "../components/Counter";
import Episodes from "../components/Episodes.astro";
import Layout from "../layouts/Layout.astro";
---

<Layout>
  <Counter client:load />
  <Episodes count={5} server:defer>
    <div slot='fallback'>Loading episodes...</div>
  </Episodes>
</Layout>

Do uruchomienia naszej wyspy będziemy potrzebować jeszcze jednego mechanizmu - docelowego runtime backendowego, który wykona nasz kod.

W Astro zostało to uogólnione do tzw. “adapterów”, czyli modułów umożliwiających wykonywanie kodu backendowego na konkretnej infrastrukturze (więcej tutaj). W naszym przypadku mówimy o standardowym kodzie backendowym, więc wystarczy nam backend oparty o Node’a (alternatywy to np. Vercel, Cloudflare czy Netlify):

npx astro add node

Akceptujemy nową konfigurację, po czym możemy przeładować aplikację i przetestować nowy kod:

Po dodaniu adaptera, zaktualizujmy konfigurację naszej aplikacji:

import {defineConfig} from "astro/config";

import tailwind from "@astrojs/tailwind";
import react from "@astrojs/react";
import node from "@astrojs/node";

// https://astro.build/config
export default defineConfig({
  integrations: [tailwind(), react()],
  adapter: node({
    mode: "standalone",
  }),
  server: {
    port: 3000,
  },
});

Efekt? Aplikacja z dwoma komponentami - jednym uruchamianym w przeglądarce (Counter), z dodatkowymi elementami interaktywnymi, a drugi (Episodes) przygotowywany i renderowany po stronie serwera (zwróć uwagę na zawartość odpowiedzi w Dev Toolsach):

Połączenie komponentów przeglądarkowych i serwerowych sprawia, że programista zyskuje potężne narzędzie do sterowania tym, gdzie i kiedy mają się wykonać poszczególne operacje. Te interaktywne będą zapewne delegowane do przeglądarki, a te złożone i wymagające obliczeń - na serwer.

💡 W ramach treningu możesz sprawdzić, jak zachowa się serwerowy komponent zawierający komponent kliencki tworzony np. w oparciu o Svelte czy Astro - to dopiero różnorodny stack technologiczny 😎

Kolekcje treści

Chociaż nasz serwerowy komponent spełnia swoje zadanie i wyświetla listę odcinków, w Astro tego typu zestawy danych jak odcinki zasługują na przeniesienie do tzw. kolekcji treści (content collection). Dzięki temu można ich używać w wielu miejscach - mogą chociażby służyć do dynamicznego budowania routingu czy pobierania w komponentach serwerowych.

Zacznijmy od utworzenia nowego pliku src/loaders/episode-loader.ts:

export interface Episode {
  id: string;
  title: string;
  link: string;
}

export async function episodeLoader(): Promise<Episode[]> {
  const PODCAST_URL = "https://anchor.fm/s/e2cb03d0/podcast/rss";
  const response = await fetch(PODCAST_URL);
  const feedXml = await response.text();

  const itemRegex = /<item>[\s\S]*?<\/item>/g;
  const items = [...feedXml.matchAll(itemRegex)];

  const titles = items.map((item) => {
    const titleMatch = item[0].match(/<title><!\[CDATA\[(.*?)\]\]><\/title>/);
    return titleMatch ? titleMatch[1] : "";
  });

  const links = items.map((item) => {
    const linkMatch = item[0].match(/<link>(.*?)<\/link>/);
    return linkMatch ? linkMatch[1] : "";
  });

  return titles.map((title, i) => ({
    id: `ep-${i}`,
    title,
    link: links[i],
  }));
}

W kolejnym kroku utwórz główny katalog wszystkich kolekcji, czyli src/content.config.ts:

import {defineCollection, z} from "astro:content";
import {episodeLoader} from "./loaders/episode-loader";

const episodes = defineCollection({
  loader: episodeLoader,
  schema: z.object({
    title: z.string(),
    link: z.string(),
  }),
});

export const collections = {episodes};

Możesz teraz przebudować aplikację (npm run build) i sprawdzić, czy w projekcie pojawił się nowy folder o nazwie “.astro” - w środku powinny znajdować się typy i definicje nowej kolekcji.

Jeśli tak jest, to pozostało nam przeprowadzić refaktoryzację komponentu Episodes i skorzystać z wbudowanej funkcji “getCollection” do pobrania naszych danych. Zauważ, że całość działa tak jak wcześniej - propsy są akceptowane przez komponent, a dane zwracane do UI - zmienia się tylko źródło danych, które można teraz wykorzystywać w wielu innych miejscach aplikacji:

---
import {getCollection} from "astro:content";

const {count} = Astro.props as {count: number};
const episodes = await getCollection("episodes");

const selectedEpisodes = episodes.slice(0, count);
---

<ul class='list-disc p-4'>
  {
    selectedEpisodes.map((e) => (
      <li class='hover:underline hover:text-blue-400'>
        <a href={e.data.link}>{e.data.title}</a>
      </li>
    ))
  }
</ul>

Ogromną zaletą kolekcji treści jest wsparcie dla systemu typów TypeScript. W momencie, kiedy zarejestrujemy nową kolekcję, Astro tworzy dla nas nowe typy, dzięki czemu pobieranie danych jest dodatkowo weryfikowane przez TypeScript:

Pojedynczy element kolekcji jest dodatkowo definiowany przez Zoda, więc jeśli dostarczone dane nie pasują do oczekiwanego pola schema w loaderze, Astro nie zbuduje aplikacji!

Kolekcje treści to prawdziwy fundament nowoczesnego Astro - wczytamy je w oparciu o dwa domyślnie dostarczane loadery oparte o glob i system plików, ale nie ma przeszkód aby zbudować własne, pobierające dane z API, feedu RSS, endpointu GraphQL czy dowolnego innego źródła, do którego możemy się podłączyć z poziomu Node.js. Dzięki temu ładowanie bazy treści z silnym typowaniem jest intuicyjne i błyskawiczne.

Ścieżki dynamiczne

Nasza nowa kolekcja treści może zostać wykorzystana do generowania statycznych stron zasilanych danymi z serwera. To coś, co sprawia, że Astro znakomicie nada się do prezentowania wszelkiego rodzaju danych z CMSów, feedów czy publicznych API.

Aby przetestować ten element Astro utwórz nowy komponent: src/pages/blog/[...slug].astro

Ta nietypowa nazwa komponentu oznacza, że tworzymy “szablon”, a nie jedną konkretną ścieżkę. Na podstawie konfiguracji, strony będą dostępne pod adresem:

  • /blog/{id-1}

  • /blog/{id-2}

  • /blog/{id-3}

  • etc…

Co zadecyduje o tym, ile stron zostanie utworzonych? Dane zwrócone z funkcji getStaticPaths:

---
// Zawartość strony:
import Layout from "../../layouts/Layout.astro";
import type {GetStaticPaths} from "astro";
import type {Episode} from "../../loaders/episode-loader";
import {getCollection} from "astro:content";

const {episode} = Astro.props as {episode: Episode};

// Szablon danych:
export const getStaticPaths = (async () => {
  const episodes = await getCollection("episodes");
  return episodes.map((episode) => ({
    params: {slug: episode.id},
    props: {episode: episode.data},
  }));
}) satisfies GetStaticPaths;

// Layout strony:
---

<Layout>
  <div class='flex flex-col gap-4 p-4'>
    <h1 class='text-2xl font-bold'>{episode.title}</h1>
    <a class='text-blue-500' href={episode.link}>{episode.link}</a>
  </div>
</Layout>

Funkcja getStaticPaths wskazuje Astro, ile stron ma zostać utworzonych wg szablonu /blog/[…slug]. Jak widzisz, wykorzystujemy w niej utworzoną wcześniej kolekcję odcinków, dzięki czemu Astro utworzy tyle stron, ile danych pobraliśmy z naszego feedu RSS.

Po uruchomieniu aplikacji (npm run dev) możesz wejść na http://localhost:3000/blog/ep-1 i zobaczyć, jak to działa.

Co ciekawe, generowanie ścieżek dynamicznych działa niemal identycznie w Next.js, tyle, że zamiast komponentów Astro mamy tam strony z rozszerzeniem .tsx (to kolejny dowód na to, żeby w nauce skupiać się na ogólnych konceptach, a nie nazwach czy detalach jak funkcje w Astro vs Next):

export const getStaticPaths: GetStaticPaths = async () => {
  const episodes = await getAllEpisodes("episodes");
  return {
    paths: episodes.map((episode) => ({
      params: {slug: episode.id},
    })),
    fallback: false,
  };
};
 
export default function Page() {
  return <h1>Hello, Blog Post Page!</h1>
}

Podsumowanie

Ta próbka możliwości Astro pokazuje, jak bardzo elastyczny i użyteczny jest to framework. W połączeniu z Reactem, otrzymujemy do ręki znakomite narzędzie do poruszania się w warstwie backendu, które dodatkowo może być napędzane interaktywnymi, renderowanymi na żądanie komponentami.

Poza prostym routingem i wyspami, znajdziesz tam m.in. natywne wsparcie dla renderowania Markdown, możliwość wystawiania endpointów w formie publicznego API a także ekosystem pluginów ułatwiających chociażby generowanie sitemap czy wsparcie dla podświetlania kodu na stronie. Wykorzystanie Vite jako domyślnego bundlera powoduje, że możemy korzystać z najnowszych osiągnięć frontendu i naprawdę szybkiego budowania złożonych projektów.
