Wprowadzenie

Wymiana danych pomiędzy wieloma warstwami aplikacji nie jest łatwym zadaniem. Niezależnie od tego, czy korzystasz z automatycznych albo ręcznie tworzonych klientów HTTP, nie dają one pełnej gwarancji co do formatu przesyłanych danych w runtime. Tym problemem zajmiemy się w tej lekcji, poznając dwa popularne elementy ekosystemu TypeScripta, czyli biblioteki Zod i tRPC.

Pierwsza zapewni nam dodatkową warstwę walidacji danych przy jednoczesnym wsparciu systemu typów TypeScript, a druga pokaże jak pisać spójne aplikacje full-stackowe, gdzie kontrakty między backendem i frontendem muszą być określane w precyzyjny sposób.

Walidacja w runtime - Zod (https://zod.dev/)

Aby zapewniać spójność i stabilność rozwijanego systemu, w złożonych projektach webowych często implementujemy walidację danych pochodzących z zewnętrznych źródeł - może to być zewnętrzne API, dane użytkownika z formularza, czy pliki konfiguracyjne. Mimo iż TypeScript dostarcza statyczną kontrolę typów podczas kompilacji, to po stronie runtime dane nadal mogą być niepoprawne. Wyobraźmy sobie scenariusz:

  • Frontend i backend developer rozwijają moduł do obsługi produktów w sklepie e-commerce,

  • Wspólnie określają typ danych, który akceptują obie warstwy aplikacji (kontrakt),

  • Backend developer rozwija endpoint zaczynając od akceptacji parametru o umówionym typie,

  • Frontend developer deklaruje typ w swojej aplikacji, określa wymagane pola, ale zapomina o poprawnej konfiguracji formularza - chociaż na poziomie TypeScriptu wszystko jest “required”, to jednak z punktu widzenia użytkownika część pól jest “optional”,

  • Po wdrożeniu formularza system zaczyna zwracać błędy - użytkownicy nie wypełniają pól, które powinny być oznaczone jako wymagane.

Rozwiązaniem problemu jest zaimplementowanie dodatkowej warstwy walidacji, która oceni poprawność wpisywanych przez użytkownika danych. W tej lekcji pokażemy ci bibliotekę Zod, która rozwiązuje ten problem i która w nowoczesnych projektach TypeScriptowych błyskawicznie zyskuje na popularności.

Pierwsze kroki z biblioteką Zod

Zacznijmy od instalacji korzystając z preferowanego managera paczek:

npm install zod 
# yarn install zod
# pnpm install zod

Jak wskazuje dokumentacja, najlepsze efekty z Zodem uzyskasz przy włączonym trybie strict:

// tsconfig.json
{
  // ...
  "compilerOptions": {
    // ...
    "strict": true
  }
}

Po zainstalowaniu Zoda, przejdźmy teraz do zbudowania pierwszego modelu i określenia reguł walidacji.

Modele i reguły walidacji

Budowanie modeli w oparciu o Zoda jest naprawdę intuicyjne - z modułu “zod” importujemy pomocniczy obiekt o nazwie “z”, który wystawia dla nas funkcje do budowania obiektów - zarówno typu prostego jak i złożonego.

W zależności od zadeklarowanego typu danego pola, zostają nam udostępnione rozmaite metody walidacji w oparciu o tzw. “fluent API”, gdzie przez łańcuch wywołań określamy nasze oczekiwania względem danych:

import {z} from "zod";

export const userSchema = z.object({
  firstName: z
    .string()
    .min(2, "Imię musi mieć co najmniej 2 znaki")
    .max(50, "Imię musi mieć mniej niż 50 znaków"),
  lastName: z
    .string()
    .min(2, "Nazwisko musi mieć co najmniej 2 znaki")
    .max(50, "Nazwisko musi mieć mniej niż 50 znaków"),
  email: z.string().email("Nieprawidłowy adres email"),
  age: z
    .number()
    .min(18, "Musisz mieć co najmniej 18 lat")
    .max(120, "Nieprawidłowy wiek"),
  password: z
    .string()
    .min(8, "Hasło musi mieć co najmniej 8 znaków")
    .regex(/[A-Z]/, "Hasło musi zawierać co najmniej jedną wielką literę")
    .regex(/[a-z]/, "Hasło musi zawierać co najmniej jedną małą literę")
    .regex(/[0-9]/, "Hasło musi zawierać co najmniej jedną cyfrę"),
});

Kiedy zdefiniujemy nasz szablon oczekiwanych danych, możemy do niego porównywać dane pochodzące z różnych źródeł (więcej o tym podejściu).

W zależności od preferencji, mamy do wyboru metodę “parse” która potwierdzi zgodność lub rzuci wyjątek, albo metodę “safeParse”, która przekaże nam wyniki walidacji bez rzucania wyjątku:

try {
  const user1 = userSchema.parse({ ... }); // ✅ { ... } 
  const user2 = userSchema.parse('Jacek Placek'); // ❌ => wyjątek typu ZodError
} catch (error) {
  if (error instanceof ZodError) { ... }
}

const user3 = userSchema.safeParse({ ... }); // ✅ { success: true; data: { ... } }
const user4 = userSchema.safeParse('Jacek Placek'); // ❌ => { success: false; error: ZodError }

Już na tym etapie zalety tej biblioteki są jasne i zrozumiałe, jednak Zod, jako narzędzie w pełni zintegrowane z ekosystemem TypeScriptu, idzie o krok dalej.

Zod wystawia dla nas generyczny typ pomocniczy “infer”, dzięki któremu nasz schemat danych przekonwertujemy w rzeczywisty typ, którym możemy się posługiwać w całej aplikacji:

type UserType = z.infer<typeof userSchema>;

Mechanizm dynamicznego tworzenia typów pozwala nam unikać sytuacji, w których wewnątrz aplikacji powstają dwie odrębne definicje obiektów - ta na potrzeby walidacji i ta wykorzystywana w ramach logiki biznesowej.

Nowe reguły walidacji

Jeśli okaże się, że standardowe metody walidacji nie są dla ciebie wystarczające, do rozszerzenia możliwości Zoda możesz wykorzystać wbudowaną funkcję refine. Wspiera ona dwa rodzaje argumentów, synchroniczne i asynchroniczne:

export const workshopsRequest = z.object({
  companyName: z.string().refine(async (data) => {
    const blacklisted = await fetchBlacklistedCompanies();
    return !blacklisted.includes(data)
  }, { message: 'Ta firma jest na liście dłużników' }),
  members: z.number().refine((data) => data > 0 && data % 2 === 0, {
    message: "Liczba uczestników warsztatów musi być parzysta",
  }),
});

export type WorkshopsRequest = z.infer<typeof workshopsRequest>;

Ważna uwaga - korzystając z asynchronicznych reguł walidacji, parsowanie musimy wykonać metodą parseAsync (inaczej Zod rzuci wyjątek):

const result = await workshopsRequest.parseAsync({
  companyName: "Google",
  members: 10,
});

Funkcje pomocnicze

Elastyczny interfejs Zoda jest czymś, co zasługuje na uznanie. Narzędzie udostępnia nam nie tylko mechanizmy walidacji, ale również rozmaite funkcje pomocnicze służące do wykonywania operacji na schematach.

Definicje obiektów można łączyć i modyfikować w dowolny sposób - chociażby tak jak poniżej, gdzie z istniejącej definicji modelu użytkownika wybieramy wyłącznie reguły dotyczące wieku:

const ageSchema = userSchema.pick({age: true});
type JustAge = z.infer<typeof ageSchema>;
const age: JustAge = {age: 18}; // ✅ OK

Takim precyzyjnym schematem możemy się posługiwać w mniejszych formularzach, gdzie użytkownik ma dostęp do edycji fragmentu obiektu - dzięki temu unikamy powielania walidacji i utrzymujemy jedno źródło prawdy o naszych danych.

Zod na produkcji

Zod zbudował już na tyle mocną markę w środowisku, że obecnie można go spotkać nie tylko w kodzie aplikacji, ale również w narzędziach i frameworkach, z których te aplikacje budujemy. Poniżej prezentujemy przykłady jego użycia.

react-hook-form - Walidacja formularzy

Wspomniana na początku walidacja formularzy może być udręką, ale w ekosystemie Reacta pojawiają się coraz bardziej dojrzałe rozwiązania pozwalające unikać wymyślania koła na nowo. Jedno z nich to react-hook-form (https://react-hook-form.com/), którego rozszerzenia pozwalają na integrację z Zodem.

Formularze, które wykorzystują istniejące schematy Zoda, opierają się na hooku “useForm” - przekazujemy do nich resolver ze schematem danych oraz wartości domyślne. W odpowiedzi uzyskujemy dwie funkcje (register, handleSubmit) oraz obiekt z bieżącym stanem formularza:

import {useForm} from "react-hook-form";
import {zodResolver} from "@hookform/resolvers/zod";
import {userSchema, type UserType} from "../schemas/userSchema";

export default function UserForm() {

  const {
    register,
    handleSubmit,
    formState: {errors},
  } = useForm<UserType>({
    resolver: zodResolver(userSchema),
    defaultValues: {
      firstName: "",
      lastName: "",
      email: "",
      age: undefined,
      password: "",
    },
  });

  // ...

}

Dzięki wykorzystaniu Zoda, w komponencie nie musimy utrzymywać dodatkowej logiki walidacyjnej, co mogłoby wprowadzać dodatkową złożoność i problemy ze skalowaniem takiego komponentu. Intuicyjne mechanizmy do rejestrowania kontrolek (funkcja register) i obsługi gotowego formularza (funkcja handleSubmit) pozwalają na rozwijanie formularzy w naprawdę zrozumiały sposób.

Astro - Content collections

W popularnym frameworku Astro (https://astro.build/) Zoda można wykorzystać do definiowania tzw. content collections - są to “kolekcje treści”, które zasilają statycznie renderowane strony w naszej aplikacji.

import { defineCollection, z } from 'astro:content';
import { glob } from 'astro/loaders';

const post = defineCollection({
  loader: glob({ pattern: "**/*.md", base: "./src/data/blog" }),
  schema: z.object({
    id: z.number(),
    title: z.string(),
    description: z.string(),
    tags: z.array(z.string())
  })
});

export const collections = { post }

Tak zdefiniowany schemat będzie weryfikował to, czy dostarczane przez nas pliki Markdown zawierają wszystkie niezbędne informacje. Jeśli reguły złamiemy, to aplikacja nie przejdzie kompilacji.

---
id: 0
title: Hello world
description: Lorem ipsum dolor sit amet
tags:
 - one
 - two

---

...

Dodatkowo, definicje Zoda pozwalają nam nie tylko weryfikować schemat pobieranych danych, ale otrzymywać do nich dostęp w mocno typowanej formie w reszcie aplikacji: 

import { getCollection } from 'astro:content';

const posts = await getCollection('post');

posts[0]. // Podpowiadanie składnie - id, title, description, tags...

OpenAPI - Structured outputs

Co ciekawe, opisywana przez nas biblioteka jest nawet w stanie kontrolować Sztuczną Inteligencję 😎

Twórcy oficjalnego SDK OpenAI od niedawna wspierają funkcję pomocniczą, która dopasowuje odpowiedzi modelu językowego do schematu definiowanego przez Zoda:

import OpenAI from "openai";
import { zodResponseFormat } from "openai/helpers/zod";
import { z } from "zod";

const openai = new OpenAI();

const CalendarEvent = z.object({
  name: z.string(),
  date: z.string(),
  participants: z.array(z.string()),
});

const completion = await openai.beta.chat.completions.parse({
  model: "gpt-4o-2024-08-06",
  messages: [
    { role: "system", content: "Extract the event information." },
    { role: "user", content: "Alice and Bob are going to a science fair on Friday." },
  ],
  response_format: zodResponseFormat(CalendarEvent, "event"),
});

const event = completion.choices[0].message.parsed;

Ten przykład pochodzi ze strony: https://platform.openai.com/docs/guides/structured-outputs

tRPC – spójność kontraktów w całej aplikacji

Na tym etapie wiesz już, jak biblioteka Zod zabezpiecza nas przed nieprawidłowymi danymi przekazywanymi w runtime. Zyskaliśmy możliwość walidacji danych z zewnętrznych źródeł, a także bezpośrednie mapowanie schematów na typy - dzięki temu unikamy duplikowanych definicji obiektów.

Jednak Zod to tylko część układanki o nazwie “bezpieczne kontrakty i wymiana danych”.

W dalszej części lekcji poznasz rozwiązanie, które pozwala jeszcze bardziej zbliżyć do siebie warstwy aplikacji full-stackowej. Oto właśnie pole do popisu dla tRPC.

Czym jest tRPC? (https://trpc.io/)

tRPC to biblioteka, która upraszcza komunikację między frontendem i backendem w pełni typowany sposób, bez konieczności generowania klientów w stylu OpenAPI/Swagger czy GraphQL. Dzięki ścisłej integracji z TypeScriptem i Zodem, tRPC pozwala:

  • Deklarować endpointy backendowe ze schematami walidacji (Zod), które precyzyjnie definiują kształt danych wejściowych i wyjściowych,

  • Automatycznie generować po stronie frontendu typowane funkcje klienta, które w czasie kompilacji i działania programu uniemożliwią wykonanie zapytań niezgodnych z kontraktem,

  • Zachować pełną spójność kontraktów: jeśli zmienimy schemat endpointu po stronie backendu, frontend natychmiast przestanie się kompilować - błąd będzie istniał dopóki nie zaktualizujemy kodu do nowych wytycznych.

W efekcie, zwiększone bezpieczeństwo dostarczamy zarówno w runtime (dzięki bibliotece Zod) jak i w buildtime (dzięki TypeScriptowi i ścisłej integracji narzędzi). Zalety mają jednak swoją cenę - wymaganiem tRPC jest to, aby obie warstwy aplikacji były dla siebie widoczne już na etapie rozwoju kodu (np. monorepo / monolit).

Filozofia RPC

Biblioteka tRPC jest inspirowana podejściem o nazwie “Remote Procedure Call” - to rozwiązanie, w którym dwa niezależne systemy (np. frontend i backend) stają się dla siebie widoczne, a jeden z nich może korzystać z publicznych funkcji drugiego.

Wprowadza to fundamentalną zmianę sposobu komunikacji - zamiast budowania klientów HTTP i odpytywania dostępnych endpointów, tRPC integruje ze sobą warstwy aplikacji poprzez definiowanie i wywoływanie standardowych funkcji.

// Podejście klasyczne
const response = await fetch('/api/movies/drama/5');
const movie = await res.json();

// Podejście "RPC"
const movie = await api.movies.getById({ category: 'drama', id: 5 });

W rzeczywistości, komunikacja między przeglądarką i serwerem nadal odbywa się w ten sam sposób, jednak z punktu widzenia programisty zyskujemy nową warstwę funkcji, które służą do wymiany danych.

W tRPC istnieje ustandaryzowane nazewnictwo na poszczególne operacje:

  • Procedury - ogólna nazwa funkcji służących do komunikacji (read / write)

  • Query - funkcje służące do pobierania danych (read)

  • Mutacje - funkcje służące do modyfikacji stanu systemu (write)

  • Subskrypcje - funkcje umożliwiające nasłuchiwanie na zmiany stanu serwera

  • Router - zestaw Procedur

Zobaczmy teraz jak zbudować podstawową integrację dwóch warstw aplikacji i jakie korzyści wnoszą do niej TypeScript oraz Zod.

Konfiguracja backendu

Załóżmy, że tworzymy prostą aplikację z listą produktów i chcemy zdefiniować endpoint, który zwróci nam dane o produkcie na podstawie jego identyfikatora. W tRPC wystawienie takiej procedury wygląda następująco:

// server/router.ts
import {initTRPC} from "@trpc/server";
import {z} from "zod";

const t = initTRPC.create();

const productQueryParam = z.object({
  id: z.number().min(1),
});

export const appRouter = t.router({
  product: t.procedure.input(productQueryParam).query(({input}) => {
    return {
      id: input.id,
      name: "Pineapple Vision 4",
      price: 1899.99,
      image: "/vision.png",
    };
  }),
});

export type AppRouter = typeof appRouter;

Na tym przykładzie:

  • Zaczynamy od definicji router tRPC (appRouter)

  • Rejestrujemy pierwszą procedurę o nazwie “product”, która służy do odczytu danych (.query()).

  • Wskazujemy oczekiwane parametry za pomocą Zoda

Eksportowany na końcu pliku AppRouter pozwoli wywoływać tylko te metody, na które wskazuje nasza definicja routera. TypeScript i Zod zadbają o to, aby po stronie frontendu procedurę wywoływać tylko z obiektem żądanego kształtu.

Całość hostujemy na prostym serwerze, którego rozszerzymy o wsparcie dla CORS:

// server/index.ts
import cors from "cors";
import {createHTTPServer} from "@trpc/server/adapters/standalone";
import {appRouter} from "./router";

const server = createHTTPServer({
  router: appRouter,
  middleware: cors(),
});

server.listen(3100);

Konfiguracja frontendu

Na frontendzie korzystamy z biblioteki @trpc/client oraz definicji wcześniej wystawionego AppRoutera.

// client/api/api-config.ts
import {createTRPCProxyClient, httpBatchLink} from "@trpc/client";
import type {AppRouter} from "../../server/router";

export const trpc = createTRPCProxyClient<AppRouter>({
  links: [httpBatchLink({url: "http://localhost:3100"})],
});

Użycie klienta tRPC sprowadza się do wywołania dostępnych funkcji z określonymi parametrami:

import {useEffect, useState} from "react";
import {trpc} from "./api/api-config";

export default function App() {
  
  const [product, setProduct] =
    useState<Awaited<ReturnType<typeof trpc.product.query>>>();

  useEffect(() => {
    const fetchProduct = async () => {
      const result = await trpc.product.query({id: 1});
      setProduct(result);
    };

    fetchProduct();
  }, []);

// ...

Kiedy spróbujemy przekazać do .query() obiekt niewłaściwego typu, to TypeScript poinformuje nas o błędzie typowania jeszcze przed uruchomieniem aplikacji. Ten fakt eliminuje całą klasę problemów, które mogłyby pojawić się dopiero w runtime, a które przeoczymy na etapie budowania integracji.

Dodatkowe ułatwienia w integrowaniu tRPC z Reactem i react-query dostarczy ci @trpc/react-query.

Kiedy korzystać z tRPC

Jeśli budujesz aplikacje full-stackowe oparte o TypeScripta, gdzie zachowanie spójności kontraktów jest kluczowe (lub same dane zmieniają się często), to tRPC może się okazać znakomitym wyborem. Połączenie systemu typów z TypeScripta oraz reguł walidacji Zoda pozwala budować spójne i przewidywalne projekty, gdzie błędy kontraktów praktycznie nie mogą wystąpić.

Skoro jednak zalety tRPC są tak znaczące, to dlaczego nie jest ona standardowym wyborem web developerów?

Na jej niekorzyść gra z pewnością fakt, że nie wszyscy mogą sobie pozwolić na budowanie wszystkich warstw w oparciu o ten sam stack. Liczby pobrań nie są tutaj tak imponujące jak w przypadku Zoda, jednak zdecydowanie widać trend rosnący i obiecującą przyszłość. 35k gwiazdek na GitHubie i majorowa wersja “10.x” wskazują na to, że z pewnością nie będziesz jej beta-testerem.

Podsumowanie

W tej lekcji poznałeś dwa rozwiązania, które mogą zwiększyć bezpieczeństwo twoich aplikacji opartych o Reacta i TypeScript.

Pierwszy poziom to biblioteka Zod - umożliwia ona tworzenie schematów danych wraz z regułami walidacji, które są ściśle powiązane z typami poszczególnych obiektów. Zoda można stosować niezależnie od frameworka frontendowego, jednak co pokazała ta lekcja, ekosystem Reacta jest już świadomy jego zalet (np. react-form-hook).

Drugi poziom to biblioteka tRPC. W porównaniu do samej walidacji Zodem, tRPC daje nam:

  • Automatyczne, pełne typowanie komunikacji między warstwami.

  • Wykrywanie niezgodności pomiędzy warstwami jeszcze przed uruchomieniem aplikacji.

  • Silne zabezpieczenie zarówno w runtime, jak i w buildtime.

Warto pamiętać, że tRPC jest narzędziem ukierunkowanym na spójne wewnętrznie projekty full-stackowe, oparte o stack TypeScriptowy. W sytuacjach wymagających kompatybilności z innymi platformami lub klientami musimy pozostać przy innych rozwiązaniach (np. jawne deklaracje typów, standard OpenAPI, generowane klienty i dodatkowa walidacja danych).
