Wersjonowanie pakietów zgodnie z semantic versioning (SemVer)

Wersjonowanie semantyczne, znane również jako Semantic Versioning lub SemVer, to konwencja nazewnictwa wersji oprogramowania, która ma na celu ułatwienie zarządzania zależnościami oraz umożliwienie użytkownikom łatwiejszego zrozumienia potencjalnego wpływu aktualizacji oprogramowania na ich projekty. Obecnie wykorzystywana jest specyfikacja Semantic Versioning 2.0.0, dostępna na stronie semver.org.

Zasady SemVer opierają się na trzech głównych liczbach: major, minor oraz patch, które są rozdzielone kropkami (np. 1.2.3), gdzie:

  • Major wersja (pierwsza liczba) wskazuje na zmiany, które wprowadzają niekompatybilne modyfikacje API, wymagające potencjalnych zmian w kodzie projektów, które z nich korzystają.

  • Minor wersja (druga liczba) oznacza dodanie nowej funkcjonalności w sposób kompatybilny z poprzednimi wersjami, co oznacza, że nie powinno być konieczności modyfikacji kodu projektu w celu korzystania z tej aktualizacji, ale funkcjonalność pakietu zostaje rozszerzona.

  • Patch wersja (trzecia liczba) dotyczy poprawek błędów, które nie wpływają na kompatybilność API z wcześniejszymi wersjami, co oznacza, że aktualizacje te naprawiają istniejące problemy bez wprowadzania nowych funkcji czy zmian w API.

Do czego służy SemVer?

Wersjonowanie semantyczne służy do przekazywania znaczenia zawartych w danej wersji zmian. Umożliwia nam szybką ocenę ryzyka związanego z aktualizacją pakietu oraz decyzję o tym, kiedy i jakie zmiany powinni wprowadzić w swoich projektach.

Odwoływanie się do konkretnych wersji za pomocą npm i package.json

W pliku package.json w projekcie Node.js można określić, z jakich wersji pakietów projekt powinien korzystać. Można to zrobić na kilka sposobów, np.:

  • Podając konkretną wersję: "pakiet": "1.2.3"

  • Używając zakresów wersji:

    • ^1.2.3 oznacza dowolną wersję większą lub równą 1.2.3, ale mniejszą niż 2.0.0 (minor i patch updates są dozwolone)

    • ~1.2.3 oznacza dowolną wersję większą lub równą 1.2.3, ale mniejszą niż 1.3.0 (tylko patch updates są dozwolone)

    • >, <, >=, <= do definiowania bardziej szczegółowych zakresów

Podczas instalacji pakietu można określić zakres wersji, który ma zostać zainstalowany, używając symbolu @, po którym następuje wersja lub zakres (np. npm install react@16.13.1 lub npm install react@^16.13.1).

Przykłady zmian major, minor, patch na przykładzie pakietu zawierającego design tokens:

  • Major (1.0.0 -> 2.0.0): Zmiana nazewnictwa kluczowych design tokens, co wymaga zmian w projektach korzystających z tego pakietu, aby mogły one nadal poprawnie funkcjonować.

  • Minor (1.0.0 -> 1.1.0): Dodanie nowych design tokens, np. dodatkowych kolorów, które nie wymagają żadnych zmian w istniejących projektach, ale oferują więcej opcji.

  • Patch (1.0.0 -> 1.0.1): Korekta wartości istniejącego design token, np. poprawa błędnej wartości koloru, która nie zmienia API, ale naprawia istniejący problem.

Zarządzanie zależnościami i decyzje dotyczące aktualizacji mogą znacząco wpłynąć na stabilność i bezpieczeństwo projektu, dlatego ważne jest świadome korzystanie z wersjonowania semantycznego.

Zaczynamy proces publikacji @przeprogramowani/design-tokens. Pierwszą wersję opublikujemy z wykorzystaniem npm publish, a następnie przejdziemy na wykorzystanie narzędzia np, które poprawi nasze developer experience.

Aby móc publikować pakiety w rejestrze npm, musisz najpierw założyć konto na npmjs.com i być zalogowanym za pomocą komeny npm login. Możesz sprawdzić czy jesteś poprawnie zalogowany za pomocą komendy npm whoami, która powinna zwrócić Twoją nazwę użytkownika (np. przeprogramowani).

Aby uniknąć konfliktu nazw z istniejącą biblioteką design-tokens, opublikowaliśmy “scoped package”, powiązane z moją nazwą użytkownika: przeprogramowani. Nazwę scoped package poprzedzamy @, czyli “@przeprogramowani”.

W filmie przewinęło się zagadnienie “Git tagów” oraz “Github Release”, o to więcej informacji na ich temat:

Git tagi to mechanizm, który służy do oznaczania konkretnych punktów w historii repozytorium. Są to swoiste "milestones", które pozwalają zidentyfikować ważne momenty w życiu projektu, takie jak wydania (releases), wersje stabilne aplikacji czy ważne zmiany. Tagi mogą być dwóch rodzajów: lekkie (lightweight) i anotowane. Lekkie tagi są prostym wskaźnikiem do konkretnego commita, natomiast anotowane tagi są przechowywane jako pełne obiekty w bazie danych Git, co oznacza, że mogą zawierać autora, datę, wiadomość i mogą być podpisane i weryfikowane za pomocą GPG.

Łączenie git tagów z release'ami na GitHub

GitHub poszerza funkcjonalność git tagów, integrując je z konceptem release'ów. Release na GitHub to zasadniczo "ubrany" tag, który dołącza binarne pliki, dłuższy opis, oraz linkuje zmiany do konkretnych commitów i pull requestów, ułatwiając w ten sposób developerom śledzenie zmian oraz pobieranie konkretnych wersji aplikacji. Po utworzeniu taga w repozytorium git (lokalnie), można go wypchnąć na GitHub i tam stworzyć na jego podstawie release, dodając dodatkowe informacje i pliki, które będą pomocne dla użytkowników.

Praktyczne użycie git tagów

  1. Wersjonowanie oprogramowania: Tagi są idealnym narzędziem do oznaczania wydań oprogramowania. Korzystając ze standardów wersjonowania semantycznego (SemVer), można łatwo zarządzać cyklem życia oprogramowania.

  2. Łatwe odnajdywanie: Tagi pozwalają na szybkie odnajdywanie ważnych momentów w historii projektu bez konieczności przeglądania całej historii commitów.

  3. Budowanie i deployment: W połączeniu z narzędziami CI/CD, tagi mogą służyć jako triggery do budowania i deployowania określonych wersji aplikacji.

  4. Rollback do stabilnych wersji: W przypadku wykrycia błędów w nowo wydanej wersji, tagi ułatwiają szybki rollback do wcześniejszej, stabilnej wersji oprogramowania.

Jak korzystać z tagów w Git

  • Tworzenie anotowanego taga: git tag -a v1.0 -m "wersja 1.0"

  • Wypchnięcie tagów do zdalnego repozytorium: git push origin v1.0

  • Wylistowanie tagów: git tag

  • Przejście do wersji oznaczonej tagiem: git checkout tags/v1.0


Przygotowujemy produkcyjny bundle z rollup

Aby nasza paczka była gotowa na produkcyjny release 1.0.0, musimy zadbać o odpowiedni komfort jej wykorzystywania. W tym celu wykorzystamy rollup do wygenerowania plików wejściowych index.js, które pozwolą na wygodny import zawartości biblioteki.

Podczas lekcji rozbudowaliśmy package.json, aby spełniał oczekiwania niezbędne w bibliotece open source. Poniżej opis poszczególnych właściwości:

name - unikalna nazwa pakietu. W tym przypadku @przeprogramowani/design-tokens wskazuje, że jest to pakiet o nazwie design-tokens, należący do przestrzeni nazw @przeprogramowani. Przestrzenie nazw są często używane w organizacjach do grupowania powiązanych pakietów. W tym wypadku wykorzystujemy “user scope”, aby zapewnić unikalną nazwę paczki.

version - określa aktualną wersję pakietu zgodnie z semantycznym wersjonowaniem (SemVer), co w tym przypadku to 2.0.1. SemVer jest powszechnie stosowaną konwencją, gdzie zmiany w numerach wersji wskazują na rodzaj zmian dokonanych w pakiecie (major.minor.patch).

description - krótki opis pakietu, który pomaga zrozumieć, co dany pakiet robi i jest wykorzystywany przez narzędzia wyszukiwania pakietów.

license - wskazuje na licencję, na której pakiet jest udostępniony, co w tym przypadku to MIT. Wybór licencji ma znaczenie prawne i określa, jak inni mogą używać, modyfikować i dystrybuować pakiet.

keywords - tablica słów kluczowych, która pomaga w kategoryzacji i wyszukiwaniu pakietu.

files - lista plików i katalogów, które zostaną dołączone do pakietu przy publikacji.

exports - określa punkty wejścia pakietu dla różnych formatów modułów (np. ESM, CommonJS) i wariantów (np. typy TypeScript). Umożliwia to użytkownikom pakietu importowanie określonych wersji plików.

engines - definiuje, z którymi wersjami środowiska wykonawczego pakiet jest kompatybilny. Przykład wskazuje, że pakiet wymaga Node.js w wersji >=20.

repository - Informacje o repozytorium kodu źródłowego, w tym typie repozytorium (np. git), URL do repozytorium, i opcjonalnie ścieżkę do katalogu projektu w obrębie repozytorium.

publishConfig - Konfiguracja dotycząca publikacji pakietu, na przykład access: "public" oznacza, że pakiet jest publicznie dostępny na npm.

Więcej właściwości package.json znajdziesz w dokumentacji npmjs.

Licencjonowanie bibliotek

Licencjonowanie bibliotek open source to kluczowy aspekt zarządzania i dystrybucji kodu w otwartym środowisku. Decyzja o wyborze odpowiedniej licencji ma istotne znaczenie dla sposobu, w jaki inny użytkownicy mogą używać, modyfikować i dystrybuować kod. Oto przegląd najpopularniejszych licencji open source oraz informacje o tym, gdzie należy umieszczać informacje o licencji w projekcie.

Najpopularniejsze licencje open source

MIT License: Jest to jedna z najbardziej liberalnych licencji open source. Umożliwia użytkownikom niemal nieograniczone prawo do kopiowania, modyfikowania, łączenia, publikowania, dystrybuowania, sublicencjonowania i/lub sprzedaży kopii oprogramowania. Wymaga jedynie, aby w każdej znaczącej części zmodyfikowanego kodu zawarte było powiadomienie o prawach autorskich i warunkach licencji.

GPL (General Public License): GPL jest znacznie bardziej restrykcyjna niż licencja MIT. Wymaga, aby każdy, kto dystrybuuje twój kod z modyfikacjami, udostępnił kod źródłowy zmian pod tą samą licencją. GPL ma na celu zapewnienie, że oprogramowanie pozostanie wolne i otwarte.

Apache License 2.0: Podobnie jak licencja MIT, Apache 2.0 jest liberalna i pozwala na szerokie wykorzystanie oprogramowania. Dodatkowo, zapewnia ochronę przed patentowymi roszczeniami osób trzecich, co jest ważne w korporacyjnym użyciu oprogramowania.

BSD Licenses: Istnieje kilka wariantów licencji BSD, ale ogólnie są one dość liberalne. Pozwalają na niemal nieograniczone wykorzystanie i dystrybucję, pod warunkiem zachowania informacji o prawach autorskich i warunkach licencji.

Pełen przegląd licencji wykorzystywanych w projektach open source znajdziesz na stronie Open Source Initative.

Umieszczanie informacji o licencji

Plik LICENSE: Standardową praktyką jest umieszczanie pełnego tekstu licencji w root’cie projektu, w pliku o nazwie LICENSE lub LICENSE.txt. To pierwsze miejsce, gdzie użytkownicy i kontrybutorzy będą szukać informacji o licencji.

package.json: Dla projektów JavaScript, takich jak te wykorzystujące npm jako menedżera pakietów, informacje o licencji powinny być również zawarte w pliku package.json. Pole license w tym pliku powinno zawierać skrót nazwy licencji, np. "MIT", "GPL-3.0", czy "Apache-2.0". Jest to ważne dla automatycznego zarządzania zależnościami i spełniania wymogów licencyjnych.

Najlepsze praktyki open source

Jak komunikować się w projektach open-source?

Wykorzystywanie Issues na GitHub jest kluczowym elementem pracy nad projektami open source, umożliwiając efektywną komunikację między twórcami i użytkownikami. Issues służą nie tylko do zgłaszania błędów, ale również do proponowania nowych funkcji, dyskutowania o zmianach w projekcie oraz zadawania pytań dotyczących jego używania. Poniżej przedstawiamy, jak można wykorzystać ten mechanizm w praktyce.

1. Zgłaszanie błędów

Gdy użytkownik napotka błąd w projekcie, może utworzyć nowe Issue, opisując problem. Ważne jest, aby zgłoszenie było jak najbardziej szczegółowe. Dobre zgłoszenie błędu powinno zawierać:

  • Opis problemu: Co dokładnie jest problemem.

  • Kroki do reprodukcji: Sekwencja kroków, która prowadzi do wystąpienia błędu.

  • Oczekiwane zachowanie: Jak powinna zachować się aplikacja w danej sytuacji.

  • Rzeczywiste zachowanie: Jak aplikacja zachowuje się w praktyce.

  • Informacje o środowisku: Wersja projektu, przeglądarka, system operacyjny itp.

2. Proponowanie nowych funkcji

Jeśli ktoś ma pomysł na nową funkcję, która mogłaby zostać dodana do projektu, również może utworzyć Issue. W takim zgłoszeniu warto zawrzeć:

  • Opis propozycji: Szczegółowe wyjaśnienie proponowanej funkcji i argumenty przemawiające za jej dodaniem.

  • Potencjalne korzyści: Jak nowa funkcja może poprawić projekt lub rozwiązać istniejący problem.

  • Możliwe wyzwania: Przewidywane trudności związane z implementacją.

3. Dyskusje i pytania

Issues mogą być również miejscem na dyskusje na temat rozwoju projektu czy zadawania pytań związanych z jego użytkowaniem. W takim przypadku ważne jest, aby utrzymać kulturę otwartości i szacunku, zachęcając do konstruktywnego dialogu.

4. Dobre praktyki

  • Szukaj istniejących Issues: Zanim utworzysz nowe Issue, sprawdź, czy ktoś inny nie zgłosił już podobnego problemu lub propozycji. Pamiętaj o odznaczeniu opcji is:open, która jest domyślnie dodawana do wyszukiwarki, aby przejrzeć całą historię issues.

  • Użyj etykiet: Wiele projektów wykorzystuje etykiety (labels) do kategoryzacji Issues, co ułatwia zarządzanie nimi i szybsze odnalezienie potrzebnych informacji.

  • Bądź uprzejmy i konkretny: Pamiętaj, że za projektem stoją ludzie, którzy poświęcają swój czas na rozwój oprogramowania. Zawsze bądź uprzejmy i staraj się być jak najbardziej konkretny w swoich zgłoszeniach.

Wykorzystywanie Issues na GitHub to doskonały sposób na uczestnictwo w życiu projektów open source, niezależnie od tego, czy jesteś doświadczonym programistą, czy nowym użytkownikiem projektu. To narzędzie umożliwia nie tylko zgłaszanie problemów, ale również aktywny udział w tworzeniu i doskonaleniu oprogramowania.

Jak ocenić jakość pakietu w rejestrze npm?

Zanim wciśniesz komendę npm install, warto rozważyć kilka rzeczy:

  1. Czy projekt jest nadal utrzymywany? Sprawdź, kiedy ostatnio był aktualizowany. Ostatnią rzeczą, jakiej chcesz, jest uczynienie pakietu podstawową częścią projektu, tylko po to, by zdać sobie sprawę, że nie jest on już utrzymywany. Może to prowadzić do naruszenia bezpieczeństwa lub błędów.

  2. Ile pobrań tygodniowo? Nie musi to być konkurs popularności, ale jeśli używasz pakietu w środowisku biznesowym, chcesz mieć pewność, że jest on renomowany. Więcej osób korzystających z niego oznacza więcej oczu na kod, co oznacza większe szanse, że najpoważniejsze błędy zostały już rozwiązane.

  3. Jak wygląda dokumentacja? Kiedy utkniesz, bardzo pomocne jest, jeśli pakiet ma dobrą dokumentację, w przeciwnym razie musisz radzić sobie sam.

  4. Czy pakiet ma społeczność? Sprawdź issues na Githubie, czy są aktywne i mają odpowiedzi od maintenerów? Dobrze jest mieć gdzie szukać pomocy w razie potrzeby.
