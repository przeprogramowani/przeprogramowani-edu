Wprowadzenie

W poprzedniej lekcji zrealizowałeś pierwsze wdrożenie twojej aplikacji na frontendową chmurę z wykorzystaniem Github Actions. W przykładach z lekcji korzystaliśmy z Netlify, a ćwiczeniem dodatkowym było wdrożenie na Vercela, dodanie nowego inputu oraz dodanie testów jednostkowych.

Jeśli udało ci się to ćwiczenie zrealizować, to twój workflow powinien przypominać ten poniżej:

name: Deploy

on:
  workflow_dispatch:
    inputs:
      deploy-netlify:
        type: boolean
        description: Enable Netlify deployment
        default: true
      deploy-vercel:
        type: boolean
        description: Enable Vercel deployment
        default: true

jobs:
  deploy:
    runs-on: ubuntu-latest
    env:
      VERCEL_ORG_ID: ${{ secrets.VERCEL_ORG_ID }}
      VERCEL_PROJECT_ID: ${{ secrets.VERCEL_PROJECT_ID }}
      NETLIFY_AUTH_TOKEN: ${{ secrets.NETLIFY_AUTH_TOKEN }}
      NETLIFY_SITE_ID: ${{ secrets.NETLIFY_SITE_ID }}
    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Configure node
        uses: actions/setup-node@v4
        with:
          node-version-file: '.nvmrc'

      - name: Install dependencies
        run: npm ci

      - name: Run unit tests
        run: npm run test

      - name: Build app
        run: npm run build

      - name: Deploy to Netlify
        uses: nwtgck/actions-netlify@v3.0
        if: ${{ inputs.deploy-netlify }}
        with:
          publish-dir: './dist'
          production-branch: master
          github-token: ${{ secrets.GITHUB_TOKEN }}
          deploy-message: 'Deployed from GitHub Actions'

      - name: Install Vercel CLI
        if: ${{ inputs.deploy-vercel }}
        run: npm install --global vercel@canary

      - name: Pull Vercel Environment Information
        if: ${{ inputs.deploy-vercel }}
        run: vercel pull --yes --environment=production --token=${{ secrets.VERCEL_TOKEN }}

      - name: Build for Vercel
        if: ${{ inputs.deploy-vercel }}
        run: vercel build --prod --token=${{ secrets.VERCEL_TOKEN }}

      - name: Deploy to Vercel
        if: ${{ inputs.deploy-vercel }}
        run: vercel deploy --prebuilt --prod --token=${{ secrets.VERCEL_TOKEN }}

📝 Ten workflow znajdziesz w folderze: examples/module3/lesson1/_solutions/deploy.yml

Scenariusz realizuje swoje założenia, ale prawdę mówiąc - nie wygląda najlepiej. Widzimy co najmniej trzy obszary, w których możemy go optymalizować:

  • nie pozwala na pełną automatyzację wdrożenia - korzystamy z manualnego triggera

  • staje się co raz bardziej złożony i trudny w utrzymaniu

  • Vercel i Netlify odgrywają dla nas podobną rolę - nie widać korzyści z utrzymywania dwóch środowisk

W tej lekcji postaramy się wprowadzić kilka zmian, z których najważniejszymi będą:

  • automatyczna weryfikacja tworzonych w repozytorium Pull Requestów

  • kompozycja akcji i łatwiejsze zarządzanie fragmentami scenariuszy automatyzacji

  • praca z dwoma niezależnymi środowiskami - dev i staging

Zanim skupimy się na samym scenariuszu i jego kolejnych ulepszeniach, zacznijmy od nieco szerszego kontekstu i tematyki CI/CD, czyli fundamentu zwinnego tworzenia oprogramowania.

Let’s go!

Balansowanie między szybkością i jakością

Ponad 20 lat temu, na początku pewnego legendarnego już manifestu, umieszczono takie zdanie:

Our highest priority is to satisfy the customer
through early and continuous delivery
of valuable software.

Ten fragment pochodzi oczywiście z manifestu Agile, czyli fundamentu nowoczesnego podejścia do rozwoju oprogramowania. Twórcy Agile próbowali ponad dwie dekady temu zaproponować alternatywę do projektów waterfallowych (jednokierunkowych, ze sztywno określonymi wymaganiami), które najczęściej kończyły się rozjazdem oczekiwań klienta względem tego, co wytworzyli programiści.

Rozwój oprogramowania w duchu Agile promuje podejście iteracyjne. Takie, w którym oprogramowanie dowozimy w krótszych odstępach czasowych, często je integrujemy, sprawdzamy jego stabilność a także unikamy znanego motto “u mnie działa”.

Deliver working software frequently, from a
couple of weeks to a couple of months, with a
preference to the shorter timescale.

Teoretyczne założenia Agile brzmią naprawdę ekscytująco, ale wszyscy wiemy, jak trudno je realizować w środowisku rozproszonym, składającym się z wielu zespołów i programistów. Niestety, w wielu firmach filozofia Agile często zostaje na poziomie ulotnej motywacji, luźnego pomysłu, albo grupy konsultantów, którzy wprowadzają ceremonie i certyfikaty dla programistów.

Praktyczne Agile musi wyglądać inaczej. Nasze środowisko (narzędzia i techniki pracy) musi wspierać ciągłe i częste integrowanie, a także wdrażanie oprogramowania. Skracanie cykli pomaga nie tylko w szybszym wychodzeniu na produkcję, co może budować przewagę konkurencyjną, ale również w szybszym wdrażaniu bugfixów i łatwiejszym eksperymentowaniu. Finalnie, skutkuje szybszym uzyskiwaniem feedbacku od klienta, co jest podstawą do dalszej pracy i kolejnych iteracji.

Tutaj pojawia się pytanie - czy oprogramowanie można wdrażać za szybko i za często?

Okazuje się, że tak. Bez odpowiedniego przygotowania, jednoznaczna optymalizacja na szybkość dowożenia kodu skutkuje spadkiem jego jakości, a także rosnącą liczbą incydentów na produkcji. Jako profesjonalni programiści musimy więc odnaleźć balans pomiędzy światem szybkości i jakości. Jak to zrobić?

Continuous Integration, Delivery & Deployments

Efektem ubocznym filozofii Agile jest popularyzacja technik CI/CD, które starają się formalizować balansowanie pomiędzy szybkością i jakością. Martin Fowler, jeden z twórców manifestu Agile, definiuje je w ten sposób:

Continuous Deployment - Proces, w którym aplikacja jest wdrażana na produkcję w sposób ciągły

Continuous Delivery - Proces, w którym aplikacja może być wdrożona na żądanie, ale nie musi

Continuous Integration - Proces, w którym integrujemy zmiany pochodzące z różnych źródeł i kierunków

To, na czym zależy nam najbardziej, to zarządzanie zmianą na poziomie projektu. Każda wartościowa zmiana zaproponowana przez dowolnego programistę powinna wyjść na produkcję tak szybko, jak to możliwe, przy jednoczesnym zachowaniu akceptowalnych standardów jakości. Stąd nacisk na automatyzację.

Podstawą automatyzacji jest tzw. Deployment Pipeline, czyli zestaw kroków weryfikujących naszą zmianę.

W trakcie tego procesu możemy się upewnić, że zmiana zmierzająca na produkcję jest obiektywnie dobra, a produkcja dzięki niej nie wybuchnie.

Frontendowy pipeline, uruchamiany na jednym z zewnętrznych środowisk, może weryfikować m.in.:

  • czy aplikacja buduje się poprawnie i bez błędów?

  • czy kod jest sformatowany poprawnie?

  • czy przechodzą testy jednostkowe?

  • czy przechodzą testy integracyjne?

  • czy pokrycie testami jest na akceptowalnym poziomie?

  • czy wydajność jest na akceptowalnym poziomie?

  • czy dostępność jest na akceptowalnym poziomie?

  • czy sam rozmiar aplikacji jest nadal w określonym limicie?

Jeśli dana zmiana przejdzie taki zestaw testów, to uznajemy ją za gotową do wdrożenia. W duchu CI/CD chcemy ją umieścić maksymalnie blisko produkcji, albo - o czym przekonamy się w kolejnych lekcjach - nawet na samej produkcji, kontrolując kto i kiedy uzyska do niej dostęp.

Stosując w naszym projekcie podejście CI/CD, dążymy do pełnej swobody wdrażania i wycofywania poszczególnych wersji projektu nie tylko na poziomie kodu, ale również konfiguracji czy infrastruktury, co również przetestujemy w następnej lekcji.

Dzięki temu uzyskujemy więcej bezpieczeństwa w kontekście integrowania wszystkiego, co zmierza na mastera z wielu kierunków, o dowolnej porze dnia.

Kompozycja akcji

Nowoczesne systemy CI/CD radzą sobie z problemem duplikacji i złożoności poprzez wsparcie dla komponowania scenariuszy z mniejszych składowych. Github Actions nie jest tutaj wyjątkiem, a rozbudowany Marketplace akcji sprawia, że z GHA naprawdę trudno konkurować na poziomie integracji.

Korzystając z GHA mamy do wyboru trzy lokalizacje akcji - standardowy Marketplace, akcje wewnątrz repozytorium projektowego, a także wskazania na akcje umieszczone w innych repozytoriach. Daje to ogromną pulę możliwości względem innych systemów CI/CD.

Korzystając z dedykowanych akcji, zamiast polecenia run wykorzystujemy uses, podając przy okazji wskazanie na lokalizację akcji:

steps:
  - name: Akcja z GitHub Marketplace
    uses: actions/checkout@v4

  - name: Akcja z tego samego repozytorium
    uses: ./.github/actions/action-name

  - name: Akcja z publicznego repozytorium
    uses: user/repo/.github/actions/action-name@ref

Parametrem ref może być:

  • SHA commitu

  • nazwa brancha

  • tag releasu

W poniższym przykładzie wykorzystamy właśnie ten mechanizm, aby scenariusz deploy.yml rozbić na łatwiejsze do zarządzania składowe:

Podobnie jak w przypadku akcji, nasze scenariusze mogą pochodzić z tego samego, lub zewnętrznego repozytorium na GitHubie:

jobs:
  local-job:
    uses: ./.github/workflows/workflow.yml
    secrets: inherit
  remote-job:
    uses: user/repo/.github/workflows/workflow.yml@ref
    secrets: inherit

W kwestii bezpieczeństwa GitHub zachowuje się tak, jak można tego oczekiwać - kontekst zadania zostaje ustawiony względem scenariusza rozpoczynającego cały proces, a z zewnątrz pobieramy jedynie przepisy na akcje, a nie dane czy sekrety.

👉 Aby ujednolicić sposób zarządzania sekretami, tworząc workflow nadrzędny stosuj spójną konfigurację - wykorzystaj parametr inherit lub podaj sekrety jawnie dla wszystkich zadań podrzędnych.
