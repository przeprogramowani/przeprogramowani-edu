Przechwytywanie błędów w przeglądarce

Aby dowiedzieć się jak popularne usługi do monitorowania aplikacji wykrywają błędy na produkcji, zróbmy mały reverse engineering biblioteki Sentry.

Po wykonaniu krótkiego rozpoznania jej metod inicjalizacyjnych, odnajdujemy w środku dwie kluczowe funkcje dostępne na obiekcie window:

https://github.com/getsentry/sentry/blob/8bc643a80566012510109eeb5fbc04b6a20a6631/src/sentry/templates/sentry/js-sdk-loader.ts#L233-L234

Sentry, popularna biblioteka do monitoringu i alertingu, nadpisuje dwie dostępne w przeglądarce funkcje do obsługi błędów:

  • window.onerror - wywoływany w momencie wystąpienia błędu w kodzie client-side

  • window.onunhandledrejection - wywoływany w momencie odrzucenia Promise’a

Dzięki temu Sentry (jak i wszystkie popularne alternatywy, np. NewRelic czy TrackJS) może przechwytywać występujące w przeglądarce błędy, zapisywać je w bazie danych, agregować i prezentować w docelowym interfejsie użytkownika.

Nic nie stoi na przeszkodzie, żebyś te funkcje wykorzystywał również ty. Jeśli w twojej firmie masz dostęp do chmury, ale trudno będzie wynegocjować dodatkowe subskrypcje, to usługi takie jak Lambda czy CloudWatch mogą służyć za fundamenty systemu do monitoringu aplikacji.

Poniższy przykład prezentuje mechanizm nadpisywania funkcji window.onerror tak, aby możliwe było przekazanie błędów do dowolnej usługi zewnętrznej. (Ważne: w przypadku nadpisywania wspomnianej funkcji, błąd nie może pochodzić z konsoli DevTools, a samego kodu aplikacji).

<!DOCTYPE html>
<html lang="en">
  <head>
    <meta charset="UTF-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1.0" />
    <title>Custom error processor</title>
  </head>
  <body>
    <h1>Otwórz konsolę i zobacz niestandardowe powiadomienie o błędzie</h1>
    <script>
      // Nadpisanie domyślnego mechanizmu obsługi błędów kodem użytkownika
      (() => {
        const originalOnError = window.onerror;

        window.onerror = function (message, source, lineno, colno, error) {
          const errorData = {
            message: typeof message === "string" ? message : message.toString(),
            url: source,
            lineNumber: lineno,
            columnNumber: colno,
            errorStack:
              error && error.stack ? error.stack : "Stack trace not available",
          };

          console.warn("🚨 Widać mnie? 👀");
          // Wyślij informacje o błędzie do usługi agregującej powiadomienia
          // fetch(..., { method: 'POST', body: JSON.stringify(errorData) })

          if (originalOnError) {
            return originalOnError.apply(this, [
              message,
              source,
              lineno,
              colno,
              error,
            ]);
          }

          return true;
        };
      })();

      throw new Error("Testing custom error processor");
    </script>
  </body>
</html>

Pierwsze kroki z Sentry

Instalacja Sentry na darmowym planie rozpoczyna się na stronie https://sentry.io/signup/. Po wprowadzeniu danych wybieramy platformę, której dotyczy integracja:

Wybierając konkretny framework, Sentry podpowiada jak zintegrować go z systemem do monitoringu:

Po wyborze Reacta lub innego preferowanego frameworka, Sentry ma dla nas gotową paczkę, która powinna dostarczyć niezbędne narzędzia do instrumentacji “out of the box”:

npm install --save @sentry/react

Sama paczka zawiera w sobie trzy składowe - monitoring błędów, wydajności oraz sesji użytkownika. Dla przetestowania możliwości samej biblioteki wystarczy nam podstawowy monitoring:

Instalacja biblioteki powinna odbyć się jeszcze przed załadowaniem właściwej części aplikacji, więc dobrym miejscem na instalację będzie plik main.tsx lub inny, od którego zaczyna działanie kod client-side.

Na potrzeby integracji produkcyjnej wprowadzam dodatkowy moduł (sentry.ts), który zawiera funkcję do inicjalizacji biblioteki, a także weryfikację środowiska produkcyjnego i mój unikalny identyfikator przekazywany jako zmienna środowiskowa (SENTRY_DSN):

// sentry.ts
import * as Sentry from '@sentry/react';

export function initSentry() {
  if (import.meta.env.VITE_ENV_NAME !== 'production') {
    return;
  }
  Sentry.init({
    dsn: import.meta.env.VITE_SENTRY_DSN,
    integrations: [],
  });
}
// -------------------------------------
// main.tsx
import { initSentry } from './sentry';

...

initSentry();

Wdrażając Sentry na produkcję, domyślnie biblioteka nie będzie mieć dostępu do pełnej wersji naszego kodu, w związku z czym analiza błędów i stack trace’u może być utrudniona. Na szczęście na npm znajdziesz oficjalny plugin do Vite i innych popularnych bundlerów, dzięki którym zbudujesz mapowanie pomiędzy kodem źródłowym, a minifikowanymi plikami na produkcji:

npm install @sentry/vite-plugin --save-dev

Dodanie pluginu uwarunkujemy od konkretnego środowiska:

# vite.config.ts
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';
import { sentryVitePlugin } from '@sentry/vite-plugin';

export default defineConfig({
  build: {
    sourcemap: true,
  },
  server: {
    port: 3000,
  },
  plugins: [
    react(),
    process.env.VITE_ENV_NAME === 'production' &&
      sentryVitePlugin({
        org: 'przeprogramowani-34',
        project: 'rm-catalog',
        authToken: process.env.SENTRY_AUTH_TOKEN,
      }),
  ],
});

Token do API wygenerujesz na tej stronie.

Podobnie jak w przypadku monitoringu, wrażliwe dane zostaną przekazane w scenariuszu CI/CD poprzez zmienne środowiskowe i sekrety:

# deploy-aws.yml

- name: Build app
  run: npm run build
  env:
    VITE_AWS_MONITORING_API: ${{ secrets.AWS_MONITORING_API }}
    SENTRY_AUTH_TOKEN: ${{ secrets.SENTRY_AUTH_TOKEN }}
    VITE_SENTRY_DSN: ${{ secrets.SENTRY_DSN }}
    VITE_ENV_NAME: 'production'

Jeśli konfiguracja przebiegnie poprawnie, to już na etapie produkcyjnego budowania aplikacji zobaczysz na GitHub Actions odpowiedni fragment logów:

[sentry-vite-plugin] Info: Successfully uploaded source maps to Sentry

Po wywołaniu błędu na stronie, chociażby poprzez rzucenie wyjątku, dashboard Sentry przedstawi ci pełne informacje o zaistniałym wydarzeniu wraz z miejscem w kodzie, gdzie ten błąd wystąpił:

Od tego momentu podstawową instalację monitoringu poprzez Sentry możesz uznać za zakończoną. Dalsza konfiguracja tego narzędzia zależy tak naprawdę od twoich potrzeb. Jeśli chcesz, możesz do projektu wprowadzić nowych członków zespołu, zbudować alerty, wprowadzić śledzenie na innych środowiskach albo określić reguły grupowania błędów (szczególnie istotne w dużej skali).

Monitoring a prywatność

W lekcji o monitoringu z wykorzystaniem zewnętrznych usług nie możemy pominąć tematu bezpiecznego przetwarzania danych i prywatności naszych użytkowników.

Zanim zintegrujesz się z jakimkolwiek zewnętrznym serwisem koniecznie zweryfikuj jego politykę prywatności i warunki użytkowania. Sprawdź, w jakim regionie przechowywane będą wysyłane przez ciebie dane oraz co planuje z nimi robić właściciel usługi.

Aby zminimalizować ryzyko naruszenia prywatności nie zbieraj ich ponad miarę. Zadbaj o to, żeby przekazywane dane były anonimizowane - szczególnie przy zbieraniu danych wrażliwych lub identyfikujących, które mogą przypadkowo znaleźć się chociażby w logach aplikacji.

Zapoznaj się z funkcjami Sentry, które mogą ci pomóc przetwarzać dane w bardziej bezpieczny sposób.

W materiałach dodatkowych znajdziesz też opracowanie ról i odpowiedzialności podmiotów, które wg RODO biorą udział w przetwarzaniu danych.
