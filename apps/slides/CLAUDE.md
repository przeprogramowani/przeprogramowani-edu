# CLAUDE.md

This file guides agents working in the `projects/slides/` workspace.

## What This Is

A reusable React + Tailwind slide framework for Polish-language developer presentations by Przeprogramowani. The framework provides a dark editorial design system with 15+ reusable slide components that can be assembled into consistent presentation decks from structured data.

Each webinar or presentation gets its own subfolder under `webinars/<slug>/` containing pipeline output (research, talking points, demo ideas) and a generated `slides.jsx` file.

## Per-Webinar Content Structure

```
webinars/
  <YYYY-MM-DD>-<slug>/
    research.md          # Written by 10x-webinar-planner
    talking-points.md    # Written by 10x-webinar-material
    demo-ideas.md        # Written by 10x-webinar-material
    slides.jsx           # Generated by 10x-webinar-presentation
    background.png       # Optional per-webinar background override
```

## Active Deck Switching

The file `src/deck/slides.jsx` is a thin re-export that points Vite at the active webinar's slides:

```jsx
export { slides } from "../../webinars/<slug>/slides.jsx";
```

To switch which presentation is active, update this single import path. Vite HMR picks up the change instantly. The git diff clearly shows which deck is active.

## Reference Example

`src/deck/slides.example.jsx` is a complete reference deck (60+ slides) showing how to compose slides from `system.jsx` components, including custom per-presentation sub-components for data-heavy slides.

## Presentation Builder Skill

The `/10x-webinar-presentation` skill reads `talking-points.md` from a webinar folder, maps each slide to a `system.jsx` component using heuristic matching, and generates a `slides.jsx` file after user approval.

## Background Override

Each webinar can provide a custom `background.png` in its folder. To use it, copy it to `assets/background.png` (the CSS references `/assets/background.png`). The default background is a dark space-themed image.

## Development

Run commands from `projects/slides/`:

```bash
npm run dev
npm run build
```

Or from the monorepo root:

```bash
npm run dev --workspace=@przeprogramowani/slides
npm run build --workspace=@przeprogramowani/slides
```

The Vite dev server serves the deck locally. Navigate to a slide with the URL hash, for example `/#15`.

## Architecture

- `index.html` - minimal Vite host only. Do not put slide markup or navigation logic here.
- `src/main.jsx` - React entrypoint.
- `src/App.jsx` - deck runtime: keyboard navigation, touch/click navigation, fullscreen, hash tracking, progress, counter, cursor visibility.
- `src/deck/system.jsx` - reusable slide components and primitives. Add new slide templates here when a pattern will be reused.
- `src/deck/slides.jsx` - re-export pointing at the active webinar's slides file.
- `src/deck/slides.example.jsx` - reference example deck showing all component usage patterns.
- `styles.css` - Tailwind entrypoint plus `@layer base/components/utilities` definitions for the presentation design system.
- `tailwind.config.js` - design tokens: colors, fonts, type scale, shadows.
- `assets/` - static images used by slides (including `background.png`).
- `webinars/` - per-webinar content library (pipeline output + generated slides).

## Design System Rules

Prefer existing slide components before creating new ones:

| Component | Purpose |
|---|---|
| `TitleSlide` | Opening/CTA slides with large title and optional meta |
| `StatementSlide` | One big idea, centered |
| `SectionSlide` | Act or chapter break |
| `SplitShowcaseSlide` | Two-column text + screenshot stack comparison |
| `ImageSlide` | Screenshot/image-led slide with optional title/caption |
| `FullImageSlide` | Full-viewport image |
| `CodeSlide` | Prompt, terminal, or code-like block |
| `ListSlide` | Bullets and checklists |
| `QuoteSlide` | Large quote with attribution |
| `NumberSlide` | Large numeric callout |
| `InsightSlide` | Numbered insight with tag, headline, body, optional image |
| `SkillTheorySlide` | Skill demo card with theory points and launch command |
| `Compare`, `CompareCol`, `Stat` | Reusable comparison/stat primitives |
| `Em` | Inline emphasis with tone/gradient coloring |

When a new layout is needed:

1. Add a generic component to `src/deck/system.jsx` if it can be reused.
2. Add only the slide data/composition to the webinar's `slides.jsx`.
3. Add styling through Tailwind component classes in `styles.css` or tokens in `tailwind.config.js`.
4. Avoid one-off inline styles and avoid embedding CSS in JSX.

## Styling Rules

- Use Tailwind utilities and the existing component classes from `styles.css`.
- Keep colors and typography driven by tokens in `tailwind.config.js` and CSS variables in `styles.css`.
- Preserve the dark editorial presentation style: large balanced type, restrained cards, strong screenshots, minimal chrome.
- Keep cards at `rounded-lg` / 8px radius unless there is a clear established reason.
- Do not use negative letter spacing; this deck uses `letter-spacing: 0`.
- Do not add decorative gradient orbs, bokeh, or unrelated background effects.
- Avoid visible explanatory UI text about how the deck works. Controls are already implicit via keyboard/click/touch.

## Slide Content Rules

- Slide content is in Polish. Continue in Polish when adding or editing slides.
- Prefer concise headlines and short supporting text; these slides are meant to be spoken over.
- Use real screenshots/assets from `assets/` when a slide refers to a product, ranking, model output, or benchmark result.
- Add images as `/assets/name.png` or through the `assets(name)` helper pattern in slides files.
- Keep `act` values meaningful for narrative grouping.

## Navigation Behavior

Navigation lives in `src/App.jsx`:

- ArrowRight, Space, Enter: next slide
- ArrowLeft, Backspace: previous slide
- Home / End: first / last slide
- `F`: fullscreen
- click left/right third of screen: previous/next
- touch swipe: previous/next
- URL hash tracks slide number
- cursor auto-hides after inactivity

Do not duplicate navigation behavior inside slide components.
